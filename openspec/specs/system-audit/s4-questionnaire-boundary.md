## 一、现状诊断

### 当前流程

```
┌─────────────┐    ┌──────────────┐    ┌───────────────┐    ┌──────────────┐
│ 网页轻问卷    │ →  │ 提交成功页    │ →  │ 微信人工交付   │ →  │ 精调（如有）  │
│ 5题+微信号   │    │ 引导加微信    │    │ 付费后追问     │    │ 结构化修改    │
└─────────────┘    └──────────────┘    └───────────────┘    └──────────────┘
```

### 当前5题问卷

| # | 问题 | 字段 | 类型 |
|---|---|---|---|
| 1 | 去哪里 | destination_cities | 必填 |
| 2 | 去几天 | travel_days | 必填 |
| 3 | 和谁去 | party_type | 必填 |
| 4 | 风格偏好 | preferences | 必填 |
| 5 | 微信号 | wechat_id | 必填 |

### user-intake/spec.md 中定义的完整字段

trip_profile 表有 **25+ 字段**，包括出发城市、出行月份、精确日期、人数、预算区间、总预算、
第几次去、节奏偏好、是否接受换乘、是否需要酒店/餐厅推荐、雨天备案、航班偏好、特殊需求等。

### 核心矛盾

| 矛盾 | 说明 |
|---|---|
| **前台要简单** | 5题已经是上限，超过6题用户流失率暴增（尤其移动端） |
| **后台要充分** | 生成个性化方案至少需要12-15个维度信息 |
| **精调不能拖死** | 用户"改完还想改"，后台处理成本不可控 |
| **信息不能白采** | 付费前问太细→用户嫌麻烦；太粗→方案不准需要频繁精调 |

## 二、信息采集三阶段架构

> 核心原则：**前台越简单越好，后台可以复杂。缺的信息用合理默认值填补，而非问用户。**

### 三阶段定义

| 阶段 | 时机 | 采集方式 | 目标 |
|---|---|---|---|
| **L1 轻问卷** | 付费前（网页） | 5题结构化 | 够生成初版方案 |
| **L2 补充确认** | 付费后（微信） | 3-5题结构化 + 客服引导 | 方案精准度从60%→90% |
| **L3 精调修改** | 交付后（交付页） | 按钮/选项交互 | 90%→98%，个性化微调 |

### 信息采集量 vs 用户体验曲线

```
  方案准确度
  100% ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ✕ L3精调2次
   90% ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ✕ L2补充确认
   60% ─ ─ ─ ─ ─ ✕ L1轻问卷（合理默认值填补）
                  │
   用户烦躁度      ■■■ 低      ■■■ 中     ■ 低（主动发起）
```

## 三、L1 轻问卷审查（付费前 · 网页）

### 设计原则
1. **不超过5题**（含微信号）= 4个业务题 + 1个联系方式
2. **每题2秒可答**（选项式，非输入式）
3. **不问用户不确定的事**（如精确日期、精确预算）
4. **不问可以有合理默认值的事**（如节奏、是否需要雨天备案）

### 当前5题评审

| # | 当前问题 | 评审结论 | 理由 |
|---|---|---|---|
| 1 | 去哪里 | ✅ **保留** | 核心变量，决定路线库 |
| 2 | 去几天 | ✅ **保留** | 核心变量，决定模板选择 |
| 3 | 和谁去 | ✅ **保留** | 决定节奏/景点筛选/酒店类型 |
| 4 | 风格偏好 | ⚠️ **优化** | 当前太泛，需要收敛选项 |
| 5 | 微信号 | ✅ **保留** | 交付渠道，必须 |

### 第4题优化方案

**现状问题：** "风格偏好"选项太多（11个标签），用户选择困难。

**优化方案：** 改为"这次旅行最想要什么？"，4选1-2：

| 选项 | 映射的 theme_weights | 后台效果 |
|---|---|---|
| 🍜 吃遍日本 | food: 0.4, shopping: 0.2 | 餐厅密度↑，Tabelog 高分优先 |
| 📸 拍出好照片 | photo: 0.3, culture: 0.3 | 出片景点优先，标注机位 |
| 🏯 文化深度游 | culture: 0.4, nature: 0.2 | 寺庙/博物馆/传统体验密度↑ |
| 🛍️ 买买买+打卡 | shopping: 0.3, food: 0.2, anime: 0.1 | 商圈/购物区优先 |

**交互方式：** 4张卡片，点击选1-2个，有选中态动画。不叫"风格偏好"，叫"这次最想："。

### 应该问但没问的

| 字段 | 是否加入L1 | 理由 |
|---|---|---|
| 出发城市 | ❌ 不加 | L1阶段不影响方案生成（机票信息只有888档需要），L2补充 |
| 出行月份 | ❌ 不加 | 用合理默认值（假设1-2个月后），L2补充精确日期 |
| 精确日期 | ❌ 不加 | 很多用户付费时还没定日期，付费后再确认 |
| 预算区间 | ❌ 不加 | 用档位价格反推（买248的≈中档预算），L2微调 |
| 人数 | ❌ 不加 | 从 party_type 推导（couple=2, family=3-4），L2确认 |
| 第几次去 | ❌ 不加 | 可从风格偏好推导（选"文化深度"多半是repeat），L2补充 |
| 节奏偏好 | ❌ 不加 | 从 party_type 推导默认值（senior→relaxed, solo→intensive） |
| 必去景点 | ❌ 不加 | 太自由=太复杂，留到L3精调 |
| 不去的地方 | ❌ 不加 | 负向问题影响转化，留到L3精调 |

### L1 默认值推导规则

```python
# party_type → 默认节奏
PACE_DEFAULTS = {
    "solo": "balanced",
    "couple": "balanced",
    "friends": "intensive",
    "family_child": "relaxed",
    "family_elder": "relaxed",
    "group": "balanced",
}

# party_type → 默认人数
SIZE_DEFAULTS = {
    "solo": 1,
    "couple": 2,
    "friends": 3,
    "family_child": 4,
    "family_elder": 3,
    "group": 6,
}

# 购买档位 → 默认预算
BUDGET_DEFAULTS = {
    "248": "mid_range",    # ¥500-1000/天/人
    "888": "comfortable",  # ¥1000-1500/天/人
}

# 出行月份：默认为下单日 +45天所在月份
def default_travel_month(order_date):
    return (order_date + timedelta(days=45)).strftime("%Y-%m")
```

## 四、L2 补充确认（付费后 · 微信）

### 设计原则
1. **只在付费后问**（用户已有commitment，耐心更高）
2. **不超过5个问题**
3. **以确认为主**（"我们默认你是2人出行，对吗？"而非"你几个人？"）
4. **客服话术包装**（不像问卷，像朋友在聊天）

### L2 补充问题清单

| # | 问题 | 话术包装 | 什么时候问 | 不问会怎样 |
|---|---|---|---|---|
| 1 | **精确出行日期** | "大概哪几天出发呀？确定了我好给你算樱花/红叶/节庆时间" | 必问 | 无法匹配季节性内容、交通卡建议不准 |
| 2 | **出发城市** | "从哪个城市飞？我帮你看看机场到市区的最优走法" | 必问 | 交通指南第一天写不准 |
| 3 | **精确人数 + 有无老人/小孩** | "一共几位？有没有带老人或小朋友？我调一下行程节奏" | 必问 | 酒店房型不准、节奏可能过紧 |
| 4 | **预算倾向确认** | "酒店方面，你更偏省钱还是住好点？（我帮你平衡）" | 248档问 | 酒店推荐可能偏贵或偏简 |
| 5 | **有没有特别想去/不想去的** | "有没有一定要去的地方？或者已经去过不想再去的？" | 可选问 | 方案可能不合预期 |

### L2 问题的档位差异

| 问题 | 248档 | 888档 |
|---|---|---|
| 精确日期 | ✅ 必问 | ✅ 必问 |
| 出发城市 | ✅ 必问 | ✅ 必问（888含机票比价，更重要） |
| 精确人数 | ✅ 必问 | ✅ 必问 |
| 预算倾向 | ✅ 问 | ✅ 更细致地问（酒店/餐厅/体验分开） |
| 特别想去的 | ✅ 可选 | ✅ 必问（深度定制需要锚点） |
| 第几次去日本 | ❌ 不问 | ✅ 问（影响路线深度） |
| 航班偏好 | ❌ 不问 | ✅ 问（含机票比价，需要知道时间/中转偏好） |
| 酒店特殊需求 | ❌ 不问 | ✅ 问（温泉？和室？连住？） |

### L2 客服话术模板

**开场白：**
> 嗨～你的日本行程定制已经开始啦！🎌
> 我先跟你确认几个小细节，好给你安排最合适的路线～

**问日期：**
> 大概哪几天出发呀？如果还没完全确定也没关系，给个大概范围就行～
> （我会帮你避开当地的拥挤日和休息日）

**问人数：**
> 确认一下，这次是 {party_type_zh} 出行对吧？一共几位？
> 有带老人或小朋友的话我会特别注意节奏和路线～

**问预算（248档）：**
> 酒店方面，你更想：
> A. 能省则省，住干净舒适就行 🏨
> B. 位置好+体验好，贵一点也OK 🌟
> （直接回 A 或 B 就行～）

**问想去的地方：**
> 有没有一定要去的地方？比如浅草寺、伏见稻荷这种～
> 或者有什么已经去过不想再去的？
> 没有的话也完全OK，我来帮你安排！

**结尾：**
> 好的！信息够了～ 我去给你安排，预计 {deadline} 前交付给你。
> 有问题随时来问哈 😊

## 五、L3 精调修改（交付后 · 交付页）

> 详细设计见 DELIVERY_PAGE_DESIGN.md A32-A33，这里聚焦"哪些需求该进精调阶段"

### 应该在 L3 而非 L1/L2 处理的需求

| 需求类型 | 为什么不该前置 | L3 如何处理 |
|---|---|---|
| **换具体景点** | 用户没看到方案前不知道要换哪个 | 看到卡片后点击"换一个"→ 3个备选 |
| **换具体餐厅** | 同上 | 点击餐厅卡片 →"换一家" |
| **调某天节奏** | 需要看到具体安排才知道松紧 | 选择"更悠闲/更紧凑" |
| **调城市天数分配** | "京都多待一天"要看到方案才有概念 | 选择"XX多待1天" |
| **跳过某天** | "这天我自己安排"要看到方案才有概念 | 点击某天→"这天我自己安排" |
| **必去/不去** | L2问了但用户当时没想到的 | 精调时补充 |

### 不应该进精调的需求（会拖死后台）

| 需求类型 | 为什么不让改 | 怎么处理 |
|---|---|---|
| **改目的地** | 完全不同的产品 | 不展示修改入口 |
| **改总天数** | 影响路线架构，相当于重做 | "如需调整天数请联系定制师" |
| **改出行日期** | 影响季节性/机票/酒店，牵一发动全身 | 同上 |
| **改同行人数** | 影响预算/房型/路线 | 同上 |
| **要求加入不在库中的景点** | 自由输入=不可控 | 不提供自由输入框 |
| **要求生成全新方案** | ≠精调，=重做 | 引导加价¥69/次 |

## 六、前台不麻烦的设计要点

### 6.1 问卷层面

| 策略 | 实现 |
|---|---|
| **题目少** | 4题业务+1题联系方式=5题 |
| **选项化** | 全部用卡片/按钮，零输入框（微信号除外） |
| **视觉轻** | 每屏1题，上滑切换，进度条在顶部 |
| **可跳过** | 第4题（风格）可不选，用默认值 |
| **即时反馈** | 每选一项，下方预览"你的行程大概长这样" |
| **2分钟完成** | 从打开到提交控制在2分钟内 |

### 6.2 补充问题层面

| 策略 | 实现 |
|---|---|
| **用确认替代提问** | "你是2人情侣出行对吧？" 而不是 "几个人？" |
| **给选项不给输入** | "A省钱 B体验好" 而不是 "你预算多少？" |
| **允许不答** | "没有特别想去的也OK～" |
| **像聊天** | 语气 = 朋友帮你规划，不是表单 |
| **分散到自然对话** | 不是连续5个问题，穿插在交流中 |

### 6.3 精调层面

| 策略 | 实现 |
|---|---|
| **结构化操作** | 按钮/选项/滑块，无自由输入 |
| **展示备选** | 不让用户想，给3个选 |
| **限定范围** | 每次1-3个修改项（248档）/ 1-5个（888档） |
| **预估影响** | "换这个景点后，步行距离不变" |
| **次数可见** | "你还有N次精调机会"（248档剩余上限2次，888档5次）|

## 七、后台不被拖死的防御机制

### 7.1 精调次数硬上限

| 档位 | 精调次数 | 每次修改项 | 超出处理 |
|---|---|---|---|
| 248 | **2次** | 1-3项 | 引导升级888 或 加购¥69/次 |
| 888 | **5次** | 1-5项 | 联系定制师，加购¥69/次 |

### 7.2 "大改"识别与拦截

```python
# 自动判断是否为"大改"（不消耗精调次数，引导加价）
def is_major_change(modifications: list) -> bool:
    MAJOR_TYPES = {"change_destination", "change_total_days", "change_dates",
                   "change_party_size", "regenerate_full"}
    return any(m["type"] in MAJOR_TYPES for m in modifications)

def exceeds_scope(modifications: list, tier: str) -> bool:
    MAX_ITEMS = {"248": 3, "888": 5}
    return len(modifications) > MAX_ITEMS.get(tier, 3)
```

### 7.3 纠错不扣次数

| 情况 | 是否扣精调 | 处理 |
|---|---|---|
| 景点已关闭/搬迁 | ❌ 不扣 | 免费修正 |
| 营业时间错误 | ❌ 不扣 | 免费修正 |
| 交通信息过期 | ❌ 不扣 | 免费修正 |
| 用户想换景点 | ✅ 扣 | 正常精调 |
| 用户嫌某天太紧 | ✅ 扣 | 正常精调 |

### 7.4 SLA 与处理时间

| 请求类型 | SLA | 实际目标 |
|---|---|---|
| 首次交付（248） | 48h | 24h |
| 首次交付（888） | 72h | 48h |
| 精调请求 | 48h | 24h |
| 纠错请求 | 24h | 12h |
| 大改（加价后） | 72h | 48h |

### 7.5 后台处理效率优化

| 策略 | 说明 |
|---|---|
| **精调请求结构化** | 用户提交的是 `{type, target_day, target_item, new_value}`，不是自由文本 |
| **工作台一键操作** | 点击"替换"→ 备选自动弹出，不用手动搜索 |
| **自动连锁计算** | 换景点后，交通时间自动重算 |
| **推荐理由模板** | 70%的推荐理由用模板+变量填充，仅30%需AI润色 |
| **批量处理** | 同一订单的多个修改项一次性处理 |

## 八、轻确认页设计

> 问卷任务要求评估：免费预览和正式问卷之间需不需要"轻确认页"？

### 结论：**需要，但极简**

### 轻确认页定位
- 出现在：问卷提交后 → 付费前
- 目的：让用户确认信息+展示价值预期+引导付费
- 不是再次编辑问卷，是**确认+加购转化**

### 轻确认页内容

```
┌──────────────────────────────────────────┐
│                                          │
│  ✅ 收到！你的行程正在准备中               │
│                                          │
│  ┌────────────────────────────────────┐  │
│  │  📍 东京+京都 · 7天                 │  │
│  │  👫 情侣出行                        │  │
│  │  🍜 美食+文化                       │  │
│  │  ─────────────────────────          │  │
│  │  预计包含：                          │  │
│  │  · 7天每日行程（含景点+交通）         │  │
│  │  · 14家精选餐厅推荐                  │  │
│  │  · 5家精选酒店推荐                   │  │
│  │  · 出行准备全攻略                    │  │
│  │  · 避坑指南                         │  │
│  │  · 2次专属精调                      │  │
│  └────────────────────────────────────┘  │
│                                          │
│  💰 定制费：¥248                         │
│                                          │
│  [微信支付 ¥248]                         │
│                                          │
│  想要机票酒店比价+5次精调？               │
│  [了解 ¥888 深度定制版 →]                │
│                                          │
│  ──────────────────                      │
│  信息有误？[返回修改]                     │
└──────────────────────────────────────────┘
```

### 轻确认页的关键设计点

| 设计点 | 说明 |
|---|---|
| **不是问卷第二页** | 不增加新问题，只确认已答 |
| **展示价值** | 列出具体交付物（"14家餐厅""5家酒店"），让用户觉得值 |
| **升级入口** | 自然展示888档差异，不强推 |
| **可返回修改** | 小字链接，不突出 |
| **付费按钮大且明确** | 微信支付，一键操作 |


### 关键决策说明

| 决策 | 理由 |
|---|---|
| **special_requirements 全流程不提供自由输入** | 自由输入=不可控的后台负担。"特殊需求"用结构化选项覆盖（如"有轮椅需求"→勾选项） |
| **精确日期付费后才问** | 很多人付费时日期没定；提前问增加流失；有了日期后季节性内容更准 |
| **预算不直接问数字** | 问数字=社交压力+决策困难。用"A/B"选择替代 |
| **必去景点放L2可选** | 有的用户有强烈必去，有的没有。L2自然对话中询问，压力最小 |
| **不去的地方放L2可选** | 负面问题（"不想去哪"）影响情绪，不适合付费前问卷 |
