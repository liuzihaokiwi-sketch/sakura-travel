# S1. 系统完整性审查 — 缺口清单

> **前提**：方案已能生成（route_templates + assembler + scorer + copywriter + magazine renderer 全链路跑通）。

## 逐项详解

### G01. 问卷提交 → 后台通知链路不通

**优先级**：P0  
**影响**：成交 + 履约

**现状**：
- 前端 `/quiz` 页面已有 UI 骨架
- 后端 `POST /api/quiz` 尚未实现（E1 任务未完成）
- 用户提交后，数据无法写入 `trip_requests` 表
- 没有通知机制告知运营人员"有新问卷进来了"

**不做会怎样**：
- 用户填完问卷，石沉大海
- 运营不知道有新订单进来，无法在 2 小时内响应
- 免费预览无法交付，核心转化链路断裂

**完整要做的事**：
1. 实现 `POST /api/quiz` → 创建 `trip_request`（status=pending）
2. 问卷数据写入 `trip_requests.raw_input`（JSON）
3. 提交成功后触发通知（最简方案：发一条企业微信/飞书消息）
4. 前端 `/quiz` 提交成功后跳转 `/submitted`

---

### G02. 订单状态机 + 微信交付工作流缺失

**优先级**：P0  
**影响**：履约

**现状**：
- `orders` 表存在，有 status 字段（pending/paid/processing/delivered/refunded）
- 但没有状态流转逻辑：谁把 pending 改成 paid？谁把 processing 改成 delivered？
- 微信交付全靠人脑记忆，没有系统追踪

**不做会怎样**：
- 运营人员收到微信转账后，不知道对应哪个 trip_request
- 多单并行时容易搞混
- 无法统计"平均交付时长""超时未交付"等关键指标
- 用户付了钱但方案迟迟不出，无人追踪

**完整要做的事**：
1. 实现状态机：`quiz_submitted → preview_sent → paid → generating → review → delivered`
2. 运营可通过 admin 工作台手动推进状态
3. 微信号与 trip_request_id 绑定（在 raw_input 中记录微信号字段）
4. 超时未处理（>2h 未发预览 / >24h 未交付）自动提醒

---

### G03. 引流首页 + 轻问卷页未按最终文案重构

**优先级**：P0  
**影响**：成交

**现状**：
- 首页 `/` 当前是"技术展示型"（有樱花排行榜、定制服务介绍等）
- 未包含 master-product-strategy.md 中定义的 10 个模块（Hero → 痛点 → 解决方案 → 免费预览介绍 → 主推248 → 888锚点 → 交付展示 → 信任 → FAQ → 最终CTA）
- 轻问卷 `/quiz` 需要增加第 5 步「微信号收集」
- 价格页 `/pricing` CTA 仍指向付款，应改为指向 `/quiz`

**不做会怎样**：
- 用户从小红书点进来，看到的页面和产品定位不匹配
- 无法建立"痛点共鸣 → 方案说服 → 免费试用 → 付费"的说服链
- 转化率极低

**完整要做的事**：
1. 首页完全重写，按 10 模块结构实现
2. 问卷增加 Step 5 微信号收集
3. 价格页 CTA 统一改为"先免费看一天"→ `/quiz`
4. 全站文案按品牌语气规范统一审查

### G04. 提交成功页 `/submitted` 不存在

**优先级**：P0  
**影响**：成交

**现状**：
- 设计文档已详细定义（加微信二维码 + 期望设定 + 信任建立）
- 页面尚未创建
- 这是网页端到微信端的关键衔接点

**不做会怎样**：
- 用户提交问卷后看到空白页或通用 404
- 不知道下一步做什么，不会主动加微信
- 微信联系断裂 = 后续所有转化归零

**完整要做的事**：
1. 创建 `/submitted` 页面
2. 内容：确认收到 + 规划师微信二维码 + "2小时内联系你" + "不会群发广告"
3. 微信号/二维码可配置（不硬编码）

### G05. 免费1天预览的生成+审核+交付流程不完整

**优先级**：P0  
**影响**：成交

**现状**：
- 方案生成引擎已能生成完整 N 天方案
- 但"只生成第1天 → 渲染为预览版 H5 → 审核 → 微信发给用户"这条链路没有串通
- 没有"预览版"渲染模式（只展示 Day 1，其余天数不可见）
- 没有"审核后一键拿到可发送的链接"的快捷操作

**不做会怎样**：
- 运营拿到方案后，需要手动截图/手动拼接 → 效率极低
- 2小时内完成响应变得不现实
- 预览质量不稳定，影响付费转化

**完整要做的事**：
1. 方案生成增加 `preview_only=true` 参数（只生成/渲染 Day 1）
2. 交付页增加 preview 模式：`/plan/[id]?mode=preview`（只显示 Day 1 完整内容）
3. admin 工作台增加"发送预览"一键操作（生成短链 → 复制到剪贴板）
4. 预览版页面底部加"想看完整方案？联系你的规划师"引导

### G06. 图片资产不可控

**优先级**：P0  
**影响**：履约 + 成交

**现状**：
- `entity_media` 表存在，结构合理（has `is_cover`, `sort_order`, `source`）
- 但实际入库的图片数据极少——爬虫入库时主要抓文字信息，图片字段经常为空
- 交付页渲染器有 `_get_entity_image` 函数，但缺乏 fallback 策略
- 没有图片可用性检查（URL 可能已失效）
- 没有 city_defaults 兜底图系统

**不做会怎样**：
- 交付页出现大量空白图位置 → 用户觉得粗糙、不值钱
- Landing 页的"攻略展示"模块无图可用
- 引用第三方图片 URL 可能随时 404

**完整要做的事**：
1. 每个城市建立 8-12 张通用高质量图片（自有/可商用）作为 fallback
2. 渲染器 fallback 链：实体精选图 → 实体候选图 → 区域默认图 → 城市默认图
3. 首批 S 级实体（每城 top 30 景点/餐厅/酒店）人工确认至少 1 张封面图
4. 定期检查 URL 可达性（每周 cron job，标记失效图片）

### G07. 数据新鲜度无过期机制

**优先级**：P1  
**影响**：履约

**现状**：
- `source_snapshots` 表有 `expires_at` 字段，`record_snapshot` 函数支持 `expires_in_days`
- 但没有消费这个过期标记的逻辑——过期了不会触发重新爬取
- 餐厅/酒店数据可能已经过时（价格变动、歇业、装修）
- `entity_editor_notes` 有 `valid_from`/`valid_until`，但没有到期检查

**不做会怎样**：
- 推荐已关店的餐厅、已涨价的酒店 → 用户投诉
- 季节性信息过期（如樱花花期数据在5月还在展示）
- 长期运营后数据质量持续下降

**完整要做的事**：
1. entity_base 增加 `data_freshness_at` 字段（最后一次数据确认时间）
2. 建立过期规则：酒店价格 30 天、餐厅 60 天、景点 90 天、花期数据 7 天
3. 过期实体自动降级：`data_tier` 从 S→A 或 A→B，不再作为首选推荐
4. 周报告：列出过期即将到达的 S 级实体，提醒人工确认

---

### G08. 方案文案 token 浪费严重

**优先级**：P1  
**影响**：成本

**现状**：
- `copywriter.py` 对每个实体都调用 GPT 生成推荐理由
- 一个 7 天方案约 25-35 个实体，每个实体 1 次 API 调用
- 交通说明、出行准备、安全须知等固定内容也每次重新生成
- 没有文案缓存/复用机制

**不做会怎样**：
- 每单 AI 成本约 ¥3-8（取决于模型），100 单/月 = ¥300-800 纯 token 成本
- 规模化后成本线性增长，吃掉利润
- 等待时间也因大量 API 调用而增加

**完整要做的事**：
1. S 级实体的推荐理由预生成并缓存（每个实体 × 3 种场景 = 缓存文案）
2. 交通说明/出行准备/安全须知改为模板化静态内容块
3. AI 只用于：用户特殊需求的个性化润色、拼接后的整体通顺性检查
4. 预估优化后每单 token 消耗降低 60-70%

### G09. 内部审核工作台不存在

**优先级**：P1  
**影响**：履约

**现状**：
- 详细 spec 已写好（三列 Kanban、单订单审核页、5 分钟流程）
- 前端 `/admin` 路由不存在
- 后端审核 API 不存在（E4 任务未完成）

**不做会怎样**：
- 审核全靠人工打开 H5 链接肉眼看 + 微信聊天人肉操作
- 无法追踪审核进度
- 5 分钟/单的目标无法实现（实际可能 20-30 分钟）
- 多人协作时无法分工

**完整要做的事**：
1. 实现 `/admin` 订单看板页面
2. 实现 `/admin/order/[id]` 审核详情页（左预览 + 右编辑）
3. 实现审核 API：获取待审核列表、更新状态、触发重新生成
4. 实现简单密码保护中间件

### G10. 修改/精调后端 API 不存在

**优先级**：P1  
**影响**：履约

**现状**：
- 修改页 `/plan/[id]/edit` 前端已有骨架
- 修改 spec 已详细定义（替换景点/餐厅、调节奏、调侧重等）
- 但后端 `POST /api/trips/{id}/modify` 不存在
- 修改请求 → 重新生成 → 审核队列的链路未实现

**不做会怎样**：
- 用户在修改页提交后，什么都不会发生
- 精调只能通过微信沟通 → 人工修改 → 手动发新版本
- 2 次精调的承诺无法系统化履行

**完整要做的事**：
1. 实现修改请求 API（接收结构化修改指令）
2. 修改指令 → 部分重新生成（只重新生成受影响的天/实体）
3. 生成结果进入审核队列
4. 版本管理：`trip_versions` 表记录变更历史
5. 精调次数扣减 + 剩余次数查询 API

### G11. 交付页缺少"让用户觉得值"的增值模块

**优先级**：P1  
**影响**：成交（口碑 + 复购推荐）

**现状**：
- 交付页已实现基本模块（封面、总览、每日详情、住宿、准备清单等）
- 但缺少拉开"我们 vs 免费攻略"差距的内容

**缺少的增值模块**：
1. **方案设计说明页**（1页）— "为什么这样排"，展示规划逻辑，让用户觉得有专业判断
2. **预订优先级页** — 哪些要提前 1 个月预约、哪些提前 1 周即可、哪些当天排队
3. **"最容易后悔的错误"页** — 3-5 条"去日本最常犯的错"，展示我们帮你避开了
4. **预算感知小结** — 每天大概花多少、全程总预算参考，让用户心里有数

**不做会怎样**：
- 用户看完觉得"好像小红书上拼拼也能做出来"
- 分享意愿低，口碑传播弱
- 难以支撑 248 的价格感知

### G13. 方案版本管理 + 回滚

**优先级**：P1  
**影响**：履约

**现状**：
- `trip_versions` 表存在，有 version_number、change_reason 字段
- 但没有实际写入逻辑——生成和修改都没有创建版本记录
- 无法回滚到上一版

**不做会怎样**：
- 精调后如果效果更差，无法恢复
- 运营误操作无法撤回
- 用户看到的始终是最新版，无法对比

**完整要做的事**：
1. 每次生成/修改后自动创建 trip_version 记录
2. admin 工作台版本 Tab 展示历史，支持一键回滚
3. 回滚 = 将 plan_id 指向旧版本的 itinerary_plan

### G16. 用户反馈闭环

**优先级**：P2

**不做会怎样**：

**完整要做的事**：
1. 交付 7 天后微信发送满意度调查（3 个问题：整体满意度/最满意的部分/建议）
2. 结构化记录到 feedback 表
3. 负面反馈自动标记对应实体（供编辑审核）
