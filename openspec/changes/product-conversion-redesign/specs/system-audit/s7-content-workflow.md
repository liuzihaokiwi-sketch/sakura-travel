# S7. 内容生成与审核工作流

> 从"已有数据"到"发布内容"的完整 10 步工作流。
> 目标：最大化复用已有数据，最小化人工介入，确保内容质量和平台安全。

---

## 工作流总览

```
①选题生成 → ②素材召回 → ③草稿生成 → ④变体生成
     ↑                                    ↓
⑩反馈选题 ←── ⑨表现回收 ←── ⑧排期发布 ←── ⑤风格检查
                                    ↑          ↓
                                  ⑦人工审核 ←── ⑥导流风险检查
```

---

## Step 1：选题生成

| 维度 | 内容 |
|---|---|
| **输入** | 内容排期表（本周应发的支柱+类型组合）+ 近期已发列表 + 选题补充池（评论/私信/修改日志） |
| **输出** | 本周 5-7 个选题卡，每个包含：支柱编号 / 类型编号 / 主题关键词 / 主角实体(如有) / 角度 |
| **是否需要 AI** | 部分。排期表按规则生成；角度和标题可用 AI 建议 |
| **是否需要人工** | 需要。最终选题需人工确认（1分钟/周，扫一眼确认或微调） |
| **是否适合定时任务** | ✅ 每周一自动生成本周选题草案 |
| **是否适合缓存** | ✅ 选题库长期累积，已用选题标记为 used |

### 选题生成规则
1. 从排期表取本周的 [支柱×类型] 组合
2. 检查去重约束（近 60 天标题 / 近 30 天主角实体 / 近 7 天类型）
3. 从选题补充池取高优先级选题（评论高频问题 > 热点 > 常规轮换）
4. 自动为每个选题推荐 3 个标题变体
5. 输出选题卡列表 → 等人工确认

## Step 2：素材召回

| 维度 | 内容 |
|---|---|
| **输入** | 确认后的选题卡（含主题/主角实体/角度） |
| **输出** | 每个选题对应的素材包：实体数据 / 推荐理由 / 图片 / 数据点 / 关联信息 |
| **是否需要 AI** | 否。纯数据库查询 + 规则筛选 |
| **是否需要人工** | 否 |
| **是否适合定时任务** | ✅ 选题确认后自动触发 |
| **是否适合缓存** | ✅ 实体数据和图片可缓存，花期数据需实时查 |

### 召回逻辑
```
选题卡.主角实体
  → entity_base 查基础信息
  → entity_media 查图片（优先精选图 > 候选图）
  → entity_editor_notes 查推荐理由
  → entity_tags 查标签

选题卡.支柱 = P1(时效)
  → events_raw 查最新花期/活动数据
  → sakura 查樱花预测

选题卡.支柱 = P3(预算)
  → hotels 查价位区间
  → restaurants 查人均
  → route_matrix 查交通成本

选题卡.类型 = T10(样张)
  → 已生成方案库 查最新 Day1 渲染结果
```

## Step 3：草稿生成

| 维度 | 内容 |
|---|---|
| **输入** | 素材包 + 内容类型模板 + 口吻指令 |
| **输出** | 完整草稿（标题 + 正文 + 图片排序 + CTA建议） |
| **是否需要 AI** | 分层（见下表） |
| **是否需要人工** | 否（草稿阶段不需要） |
| **是否适合定时任务** | ✅ 素材召回完成后自动触发 |
| **是否适合缓存** | ✅ 草稿存储，后续步骤复用 |

### 分层生成

| 内容类型 | 生成方式 | AI token 消耗 |
|---|---|---|
| T1 榜单图 | **规则拼接**：按模板填入实体名+理由+图片 | 0 |
| T2 地图图文 | **规则拼接**：标注坐标+生成步行距离文案 | 0 |
| T3 时间轴 | **AI 润色**：时间轴数据 → AI 改写成口语 | 低（~500 token） |
| T4 对比图 | **规则拼接**：左右列数据填入模板 | 0 |
| T5 清单图 | **纯模板**：替换城市/季节变量 | 0 |
| T6 纠偏帖 | **AI 创作**：给主题+素材，写 300 字观点文 | 中（~1500 token） |
| T7 问答 | **AI 润色**：FAQ 答案改写成口语化 | 低（~500 token） |
| T8 后悔清单 | **AI 创作**：给踩坑数据，写成"后悔"口吻 | 中（~1500 token） |
| T9 拆解帖 | **AI 创作**：给路线逻辑，写成决策过程 | 中（~2000 token） |
| T10 样张展示 | **AI 润色**：方案数据 → 写解说文案 | 低（~800 token） |
| T11 季节提醒 | **规则拼接** + AI 润色标题 | 极低（~200 token） |
| T12 预算方案 | **规则拼接**：三档数据填入模板 | 0 |
| T13 评论回答 | **人工主导** + AI 辅助查资料 | 低（~500 token） |
| T14 热点跟进 | **人工主导** + AI 辅助写稿 | 中（~1000 token） |

### AI Prompt 模板（以 T6 纠偏帖为例）

```
你是一个去过日本很多次的旅行博主，口吻是朋友聊天、偶尔吐槽。

主题：{topic}
核心观点：{viewpoint}
数据支撑：{data_points}
相关提示：{tips}

请写一篇小红书帖子：
- 标题要有冲击力，让人想点
- 正文 200-400 字
- 口语化，不要书面语
- 不要用"AI""算法""系统"等词
- 结尾自然地提一句"有问题可以问我"（不要太刻意）
- 给出 3 个标题变体
```

## Step 4：变体生成

| 维度 | 内容 |
|---|---|
| **输入** | 草稿 |
| **输出** | 同一主题的 2-3 个表达变体（标题变体 + 开头变体 + CTA变体） |
| **是否需要 AI** | 是（标题和开头变体用 AI 生成，~300 token/篇） |
| **是否需要人工** | 否 |
| **是否适合定时任务** | ✅ 草稿生成后自动触发 |
| **是否适合缓存** | ✅ 变体存储，人工选一个发 |

### 变体维度

| 维度 | 变体数 | 示例 |
|---|---|---|
| 标题 | 3 个 | "别再去这家网红店了" / "踩了这个雷你整天行程都毁了" / "本地人根本不去的5家餐厅" |
| 开头 | 2 个 | 反问式："你是不是也排了2小时队？" / 数据式："Tabelog 3.2分，小红书却吹上天" |
| CTA | 2 个 | "有问题可以问我" / "你们去的时候踩过什么雷？评论区聊聊" |

## Step 5：风格检查

| 维度 | 内容 |
|---|---|
| **输入** | 草稿/变体 |
| **输出** | 通过/不通过 + 不通过原因 |
| **是否需要 AI** | 是（用 AI 做风格审核，~200 token） |
| **是否需要人工** | 否（自动检查） |
| **是否适合定时任务** | ✅ 变体生成后自动触发 |
| **是否适合缓存** | 否 |

### 检查清单

| 检查项 | 规则 | 不通过处理 |
|---|---|---|
| AI 痕迹 | 扫描禁用词：AI/算法/评分系统/模型/推荐引擎/数据分析 | 自动替换或打回重写 |
| 书面语残留 | 检测"综上所述""因此""鉴于"等书面表达 | 自动改写 |
| 虚假信息 | 检测"官方合作""独家""业内领先"等 | 阻断，标记为需人工 |
| 口吻一致性 | 检测是否匹配当前设定口吻（朋友/科普/吐槽） | 标记建议 |
| 长度检查 | 正文 150-500 字，标题 10-25 字 | 标记过长/过短 |
| Emoji 密度 | 每 100 字不超过 3 个 emoji | 自动削减 |

## Step 6：导流风险检查

| 维度 | 内容 |
|---|---|
| **输入** | 通过风格检查的草稿 |
| **输出** | 安全/风险 + 风险点标注 |
| **是否需要 AI** | 否（纯规则匹配） |
| **是否需要人工** | 否 |
| **是否适合定时任务** | ✅ 风格检查通过后自动触发 |
| **是否适合缓存** | 否 |

### 风险规则

| 规则 | 检测方式 | 处理 |
|---|---|---|
| 微信号/二维码 | 正则匹配 | 阻断 |
| 站外链接 | URL 正则 | 阻断 |
| "私信我"过频 | 统计近 7 天帖子中出现次数 | 如超 2 次则删除本篇 CTA |
| "关注""点赞"诱导 | 关键词匹配 | 自动删除相关句子 |
| 价格信息 | 匹配 ¥ + 数字 | 标记提醒（不阻断，但确认价格准确） |
| 竞品名称 | 竞品列表匹配 | 标记提醒 |

## Step 7：人工审核

| 维度 | 内容 |
|---|---|
| **输入** | 通过自动检查的草稿 + 变体 + 素材包 + 图片 |
| **输出** | 批准发布 / 打回修改 / 选择变体 / 微调文案 |
| **是否需要 AI** | 否 |
| **是否需要人工** | ✅ 必须 |
| **是否适合定时任务** | 否（人工触发） |
| **是否适合缓存** | 否 |

### 审核界面需求

```
┌─────────────────────────────────────────────────┐
│ 本周待审核内容（5/7 篇）                          │
├─────────────────────────────────────────────────┤
│ 周一 · T11 季节提醒 · P1 时效                     │
│                                                 │
│ 标题变体：                                       │
│ ○ "3月底去东京的注意了，这几个要提前订"            │
│ ● "下周去东京？这5件事现在就要做" ← 已选          │
│ ○ "樱花季去东京，80%的人忘了做这件事"              │
│                                                 │
│ [预览正文]  [预览图片]  [编辑文案]                 │
│                                                 │
│ ✅ 自动检查：风格通过 · 导流安全                    │
│                                                 │
│ [批准发布] [打回修改] [跳过]                       │
└─────────────────────────────────────────────────┘
```

### 审核耗时目标
- 纯模板/规则内容（T1/T2/T4/T5/T12）：30 秒/篇（扫一眼+选标题+批准）
- AI 润色内容（T3/T7/T10/T11）：1 分钟/篇（读一遍+微调）
- AI 创作内容（T6/T8/T9）：2-3 分钟/篇（仔细读+可能改几句）
- 人工主导内容（T13/T14）：审核环节跳过（本身就是人写的）
- **全周审核总耗时：约 10-15 分钟**

## Step 8：排期发布

| 维度 | 内容 |
|---|---|
| **输入** | 审核通过的内容 + 排期表 |
| **输出** | 按日期排好的发布队列 |
| **是否需要 AI** | 否 |
| **是否需要人工** | 最小化。排期自动，发布可手动或半自动 |
| **是否适合定时任务** | ✅ 每天定时提醒发布 / 未来可接 API 自动发 |
| **是否适合缓存** | ✅ 发布队列持久化 |

### 发布策略

| 项目 | 规则 |
|---|---|
| 发布时间 | 工作日 12:00-13:00 或 20:00-21:00（午休/晚间高峰） |
| 周末时间 | 10:00-11:00 或 15:00-16:00（周末节奏不同） |
| 发布方式 | 第一阶段：人工复制到小红书发布；后续可接三方工具 |
| 发布检查 | 发布前最后确认：图片顺序、标题、正文、话题标签 |
| 话题标签 | 每篇 3-5 个话题：1-2个大话题（#日本旅行）+ 2-3个小话题（#东京美食 #樱花季） |

## Step 9：表现回收

| 维度 | 内容 |
|---|---|
| **输入** | 已发布内容的小红书数据（点赞/收藏/评论/分享/私信） |
| **输出** | 每篇内容的表现评分 + 支柱/类型维度的汇总分析 |
| **是否需要 AI** | 否（纯数据统计） |
| **是否需要人工** | 部分。数据自动采集，分析报告需人工看 |
| **是否适合定时任务** | ✅ 发布后 24h / 48h / 7d 三次采集 |
| **是否适合缓存** | ✅ 表现数据持久化 |

### 采集指标

| 指标 | 权重 | 说明 |
|---|---|---|
| 收藏数 | 高 | 说明内容有用（信息密度型核心指标） |
| 评论数 | 高 | 说明内容引发讨论（观点型核心指标） |
| 私信数 | 最高 | 直接转化信号 |
| 点赞数 | 中 | 基础互动 |
| 分享数 | 中 | 传播力 |
| 负面评论率 | 反向 | 内容可能有问题 |

### 表现分级

| 等级 | 标准 | 后续动作 |
|---|---|---|
| S 级 | 私信 > 10 或 收藏 > 500 | 同主题追加变体、同支柱增加频率 |
| A 级 | 收藏 > 200 或 评论 > 50 | 正常轮换 |
| B 级 | 收藏 50-200 | 正常轮换 |
| C 级 | 收藏 < 50 且 评论 < 10 | 分析原因，同类型/角度进入观察 |
| D 级 | 负面评论 > 20% | 该主题/角度进入黑名单 |

## Step 10：反馈到下次选题

| 维度 | 内容 |
|---|---|
| **输入** | 表现数据 + 评论内容 + 私信问题 + 黑名单/白名单 |
| **输出** | 下周选题补充池更新 + 支柱比例调整建议 + 新选题卡 |
| **是否需要 AI** | 部分。评论/私信中提取选题可用 AI 辅助分类 |
| **是否需要人工** | 部分。最终选题确认需人工 |
| **是否适合定时任务** | ✅ 每周日自动汇总本周数据 |
| **是否适合缓存** | ✅ 反馈数据持久化 |

### 反馈闭环规则

| 信号 | 动作 |
|---|---|
| 某支柱连续 2 周 S 级 | 下月增加该支柱频率（+1 篇/周） |
| 某支柱连续 2 周 C 级 | 下月减少该支柱频率（-1 篇/周），换角度 |
| 某内容类型 D 级 | 该类型进入 60 天冷却期 |
| 评论高频出现某问题 | 自动生成选题卡：T13 评论回答 |
| 私信出现新需求模式 | 标记为潜在新支柱，累积到 5 次以上则建议新增 |
| 竞品某帖爆了 | 提取主题 → 差异化角度 → 加入下周选题池 |

## 工作流耗时汇总

| 步骤 | 执行方式 | 耗时 |
|---|---|---|
| ①选题生成 | 自动+人工确认 | 自动 + 1分钟/周 |
| ②素材召回 | 全自动 | 0 |
| ③草稿生成 | 自动（模板/规则/AI） | 0（AI ~5000 token/周） |
| ④变体生成 | AI 自动 | 0（~2000 token/周） |
| ⑤风格检查 | 全自动 | 0 |
| ⑥导流风险 | 全自动 | 0 |
| ⑦人工审核 | 人工 | 10-15 分钟/周 |
| ⑧排期发布 | 半自动 | 5 分钟/天（复制发布） |
| ⑨表现回收 | 全自动 | 0 |
| ⑩反馈选题 | 自动+人工审核 | 5 分钟/周 |

**总人工耗时：每周约 50-60 分钟**（审核15分 + 发布25分 + 选题确认5分 + 复盘5分）

**总 AI token：每周约 7000-10000 token**（低成本）
