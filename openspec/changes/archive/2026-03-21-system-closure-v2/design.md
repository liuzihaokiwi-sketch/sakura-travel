## Context

当前项目处于"冷启动成交期（Pre-PMF）"——技术栈已搭建（三层评分 + 行程装配 + 杂志渲染 + 管理后台 + 前端漏斗），但零真实付费用户完成过完整闭环。本次设计目标是补全系统关键缺口，形成稳定生产闭环。

## Goals / Non-Goals

**Goals:**

- 建立单一产品真相源配置模型，消除所有定义冲突
- 把免费预览升级为独立成交引擎（选天 + 模块控制 + CTA + 校验）
- 把自助微调做成不消耗正式修改次数的产品力模块
- 建立城市上下文数据核心表，让推荐理由可信
- 实现多模型评审流水线，减少人工审核负担
- 把前端从展示层升级为成交引擎 + 传播引擎
- 建立最小可用的埋点 + 评测体系

**Non-Goals:**

- 不做多语言（v1 只面向中国用户）
- 不做 APP（v1 只做 H5）
- 不做支付接口对接（v1 微信手动收款）
- 不做实时个性化推荐引擎（v1 用静态权重包 + 批量预计算）
- 不做 19.9 引流款 SKU（v1 只有免费 + 248 + 888 三档）

## Decisions

### D1: 自助微调 ≠ 正式修改，这是产品力

**选择**：自助微调（替换景点/餐厅/调节奏/切夜间方案）永远不消耗正式修改次数。只有在用户经过自助微调后仍不满意、需要人工介入时，才消耗正式精调次数。

**为什么**：
- 自助微调是系统能力的体现（"我们给你选择"），不是用户在"改作业"
- 不消耗次数 → 用户更敢改 → 改完更满意 → 人工介入更少
- 消耗次数的设计会让用户"省着用"→ 不敢改 → 不满意 → 反而找人工

**边界**：
- 自助微调：替换景点（从 3-5 个候选中选）/ 替换餐厅 / 调节奏（轻松/充实）/ 删景点 / 调出发时间 / 切夜间方案
- 正式精调：改城市 / 改日期 / 改整体路线骨架 / 自由文字需求 / 任何需要人工判断的修改

### D2: 预览页回归 H5 独立页面，不再全走微信

**选择**：恢复 `/preview/[id]` 作为独立 H5 成交引擎。

**为什么之前砍掉了**：为了简化 MVP，全程微信人工交付。
**为什么现在要恢复**：微信交付的问题是① 响应延迟（2 小时 vs 即时）② 无法做 A/B test ③ 无法埋点 ④ 无法规模化。预览页必须是产品而非人工。

**妥协**：预览页底部保留"加微信解锁"入口作为备选 CTA（对于犹豫的用户），但主 CTA 是"¥248 在线解锁"。

### D3: 多模型评审用 LLM 链式调用，不用 agent framework

**选择**：6 个评审角色用 6 次 LLM 调用链式执行，不用 LangGraph/AutoGen 等 agent 框架。

**为什么**：
- Agent 框架增加复杂度和不可预测性
- 链式调用更可控：每步输入/输出确定，可独立缓存和重试
- Token 成本可精确控制：每步有独立的 max_tokens 限制
- 失败恢复更简单：某步失败可以从该步重试，不需要重跑整个 agent loop

### D4: 城市上下文数据优先做 4 张表，不是 10 张

**选择**：v1 只做最高 ROI 的 4 张表：
1. `city_seasonal_events` — 季节活动/花期/红叶（让"现在去能看到什么"有据可查）
2. `area_crowd_patterns` — 区域 × 时段 × 平日周末人流密度（让"这个时段去人少"有据可查）
3. `entity_operational_context` — 实体营业日历/定休日/预约必要性（让"周一闭馆"不再踩雷）
4. `transport_pass_recommendations` — 交通卡券推荐矩阵（让"买什么卡最划算"有据可查）

**为什么不做更多**：先验证这 4 张表对推荐理由质量的提升是否显著，再扩展。

### D5: 分享卡用 Satori + Vercel OG 生成，不用 Canvas/Puppeteer

**选择**：分享卡（朋友圈长图/小红书封面/微信卡片）用 React → SVG → PNG 的 Satori 方案。

**为什么**：
- 已有 Next.js 生态，Satori 原生集成
- 服务端渲染，不依赖浏览器
- 速度快（<500ms），可批量生成
- 组件复用前端样式

## Risks / Trade-offs

### [风险] 预览页回归 H5 可能导致"白嫖"

→ **缓解**：预览只给 1 天（选天引擎选的），餐厅名故意模糊，其他天标题可见但不可点，核心价值在完整版。实测白嫖率可通过 preview_bounced / preview_converted 比值监控。

### [风险] 自助微调不限次可能被滥用

→ **缓解**：自助微调只提供系统筛选的 3-5 个候选，不是自由替换；每次替换有防改崩检查；极端情况下（同一实体 24 小时内替换 > 5 次）触发冷却提示。

### [风险] 多模型评审增加生成时间和成本

→ **缓解**：6 步评审中 QA + Ops Proxy 用 gpt-4o-mini（便宜快），User Proxy + Final Judge 用 gpt-4o（质量优先）。总增加成本约 $0.03-0.05/单，时间增加约 30-60 秒（可接受，用生成等待页覆盖）。

### [风险] 单一配置模型可能过度中心化

→ **缓解**：product_config 只存"产品定义"（权益/价格/修改次数），不存"UI 文案"（文案仍在前端组件中）。配置模型是 JSON 文件 + API 读取，不是数据库表（避免运行时被误改）。
