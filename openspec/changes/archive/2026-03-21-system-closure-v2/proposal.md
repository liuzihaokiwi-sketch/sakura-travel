## Why

### A. 一句话结论

**最大机会**：系统已有三层评分 + 行程装配 + 杂志渲染 + 微信私域闭环，但这些能力没有被串联成一条"免费预览自动成交 → 攻略自动优质 → 用户自助满意 → 自传播回流"的闭环。**最大缺口不是缺功能，而是缺一条从展示到成交到传播的完整生产闭环**。

### B. 第一性原理判断

**用户为什么付费？**
不是因为"信息量大"。用户自己刷小红书 3 天也能拼出一版攻略。用户付费是因为：① 看到免费一天觉得"真的懂我、真的顺"（信任建立）→ ② 意识到自己做不出这个质量（能力差）→ ③ 解锁全部的成本远低于自己花的时间（性价比）。核心触发点是**免费预览的质量所制造的"落差感"**——"这个比我自己做的好太多了，但我只看到一天"。

**预览为什么成交？**
预览成交的本质不是"展示信息"，而是**制造解锁欲望**。最强触发点是：① 餐厅名故意模糊但存在感很强（"中午推荐了一家步行 3 分钟的拉面店..."）→ ② 专业判断感外溢（"选这个时段是因为光线最好"）→ ③ 其他天的标题可见但不可点（"Day 3 京都寺庙深度游"）→ ④ 一个明确的"只要 ¥248 就能解锁全部"的 CTA。

**用户为什么喜欢、复购、转介绍？**
喜欢 = 用的时候真的顺 + 真的有惊喜 + 没有踩雷。复购 = 这次体验好 + 下次还有新需求。转介绍 = 攻略本身长得像可分享的内容（朋友圈截图好看 + 转发有面子 + 朋友看了也想要）。

**真正的壁垒是什么？**
不是 prompt（可复制），不是规则（可抄），不是人工经验（不可规模化）。真正的壁垒是：**城市上下文数据 × 排序引擎 × 多模型评审 × 用户反馈学习回路 × 前端产品化**的飞轮。每多一个真实用户，数据更好 → 排序更准 → 攻略更好 → 用户更满意 → 更多传播 → 更多用户。

**为什么前端必须是增长引擎？**
前端不是"展示攻略的壳"，而是 4 个引擎：① 成交引擎（预览页制造解锁欲望）→ ② 信任引擎（交付页体现专业判断）→ ③ 传播引擎（分享卡让攻略变成社交货币）→ ④ 协作引擎（同行人投票、标记让多人参与）。如果前端做不好，后端再强也是自嗨。

### C. 当前系统缺口总表

| 模块 | 已有 | 缺失 | 为什么是关键缺口 | 不补会怎样 |
|------|------|------|----------------|-----------|
| **产品真相源** | 各处散落定义（product-conversion spec / pricing-page spec / modification-system spec 各有一套定义） | 无单一配置模型，免费预览深度、标准版/尊享版权益、修改次数在页面/客服/后台逻辑间不一致 | 改一处忘一处，用户被告知的和系统执行的不一样 | 客诉、退款、信任崩塌 |
| **免费预览引擎** | 有 `/preview/[id]` 路由设计（已砍掉改微信交付），有"展示 1 天"的产品决定 | 无独立成交引擎——不会选"最能成交的天"、不会做模块露出/锁定、不会插 CTA、不会做预览质量校验 | 预览是全漏斗最关键转化节点 | Day 1 付费率低 → 获客成本打不平 |
| **自助微调闭环** | 有结构化修改 spec（A32-A33），有替换景点/餐厅/节奏的交互设计 | 无"替换 → 局部重排 → 自动重检"闭环，无候选排序引擎，无防改崩机制，自助微调与正式修改次数边界未系统化 | 每次修改都要人工介入 → 人力瓶颈 | 1 人 1 天最多处理 5-8 单修改 |
| **城市上下文数据** | entity_base / entity_tags / entity_scores / entity_media | 缺 city_context_calendar / area_dining_density / crowd_level / seasonal_events / transport_passes 等城市级上下文表 | 没有城市上下文，"这个时段去人少""今天适合去这里"全靠 AI 瞎编 | 推荐理由不可信 → 专业感崩塌 |
| **软规则系统** | soft-rule-system change 已设计（12 维度 + 7 客群 + 4 阶段权重包） | 还未实现，无实体级软规则分，无预览/微调专用引擎 | 没有软规则，所有实体按硬分排序，"懂我感""分享感"为零 | 攻略变成信息堆砌，无差异化 |
| **多模型评审** | guardrails.py 有设计（hard_fail / soft_fail），但只有实体数量/重复检查 | 无 QA / User Proxy / Ops Proxy / Tuning Guard / Final Judge 多角色评审 | 全靠单次 LLM 生成，无纠错能力 | 低级错误率高 → 人工审核负担大 |
| **学习回路** | soft_rule_feedback_log 表已设计 | 无埋点实现、无反馈回写、无 A/B test 基础 | 系统无法从用户行为中学习 | 永远用拍脑袋的种子值 |
| **离线评测** | 无 | 无评测集、无验收机制、无回归测试 | 改了排序/权重不知道是变好还是变差 | 模型迭代靠感觉 |
| **前端增长系统** | Landing/Quiz/Pricing/Plan 四大页面已实现 | 预览页不是成交引擎、无分享卡生成、无旅中模式、无微信承接产品化、无同行人协作 | 前端只是展示层，不是增长引擎 | 获客效率低、传播系数 < 1 |
| **后台运营** | Admin 看板 + 单订单审核页 + 三列 Kanban | 无风险标红、无候选替换影响提示、无预览成交表现看板、无用户行为回放 | 审核靠直觉，不靠数据 | 审核质量不稳定 |

### 首发最优客群判断

基于产品适配度分析：
- 当前系统**天然更适合 2-4 人小团 FIT**（自由行散客），因为路线模板 + 评分排序 + 结构化微调的设计就是为这类用户做的
- **首次赴日情侣 / 闺蜜**是最优首发客群，因为：① 决策链短（2 人商量比家庭快）② 付费意愿高（愿意为体验付费）③ 分享意愿最强（朋友圈 / 小红书高频晒图）④ 对"懂我感"和"出片回报"最敏感
- **不应首发带孩子 / 带父母客群**，因为他们对"确定性"要求极高，当前数据覆盖度不够支撑
- 建议 v1 默认权重偏向 couple + besties + first_time_fit 三个客群

## What Changes

本次变更目标是**补全系统闭环**，形成"预览自动成交 → 攻略自动优质 → 用户自助满意 → 自传播回流"的完整飞轮。

### 核心变更

1. **统一产品真相源**：建立 product_config 单一配置模型，所有页面/客服/后台从此读取
2. **免费预览引擎**：从"展示 Day 1"升级为独立成交引擎——选天 + 模块露出/锁定 + CTA + 校验
3. **自助微调闭环**：替换 → 局部重排 → 自动重检 → 保底，大幅减少人工修改介入
4. **城市上下文数据系统**：6-8 张核心表，让推荐理由从"AI 瞎编"变成"有据可查"
5. **软规则系统落地**：12 维度评分 + 权重包 + pipeline 实际实现
6. **多模型评审流水线**：6 角色 pipeline 替代单次 LLM 生成
7. **学习回路与埋点**：从预览到微调到复购的全链路埋点 + 反馈回写
8. **离线评测集**：最小可用评测集 + 发布前验收 + 回归测试
9. **前端增长系统**：预览页成交引擎 + 分享卡 + 旅中模式 + 微信承接
10. **后台运营补全**：风险标红 + 成交看板 + 行为回放

## Capabilities

### New Capabilities

- `product-config-source`: 统一产品真相源配置模型——免费预览/标准版/尊享版权益、修改次数、文案的单一来源
- `preview-engine`: 免费预览独立成交引擎——选天/模块控制/CTA/校验/失败重做
- `tuning-loop`: 自助微调完整闭环——替换排序/局部重排/自动重检/保底/边界定义
- `city-context-data`: 城市上下文数据系统——日历/区域/人流/季节/交通/餐饮密度核心表
- `multi-agent-review`: 多模型评审流水线——6 角色 pipeline
- `feedback-learning`: 学习回路与埋点体系——全链路事件 + 反馈回写 + A/B
- `offline-eval`: 离线评测集与验收机制
- `frontend-growth`: 前端增长系统——落地页/预览页/详情页/微调UX/分享协作/旅中/微信承接
- `admin-ops-v2`: 后台运营能力补全

### Modified Capabilities

- `scoring-engine`: 接入 soft_rule_score 三维公式 + 城市上下文分
- `itinerary-planner`: 接入软规则约束 + 多模型评审 + 自助微调重排
- `review-gate`: 从简单护栏升级为多角色评审门控
- `template-renderer`: 增加预览模式渲染 + 分享卡渲染
- `conversion-funnel`: 预览页从微信交付回归为独立 H5 成交引擎

## Impact

### 产品价位梯度作用

- **免费预览**：preview-engine 直接决定 Day 1 → 付费的转化率（最核心杠杆）
- **标准版 ¥248**：tuning-loop + soft-rules + multi-agent-review 决定攻略质量（留存 / 复购 / 转介绍）
- **尊享版 ¥888**：city-context-data + admin-ops-v2 决定人工审核效率和高客单体验

### MVP 优先级

| 优先级 | 模块 | 理由 |
|--------|------|------|
| P0 | product-config-source | 不统一真相源，后续所有模块都会出现定义冲突 |
| P0 | preview-engine | 免费→付费是最关键转化节点 |
| P0 | city-context-data（核心 4 张表） | 没有城市上下文，推荐理由不可信 |
| P0 | frontend-growth（预览页 + 分享卡） | 前端不改，后端再强也无法成交 |

| P1 | multi-agent-review | 减少人工审核压力，提升出品一致性 |
