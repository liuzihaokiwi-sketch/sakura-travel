## Context

/rush 页面当前状态：Next.js 14 Server Component + Client Component 混合渲染。数据从 `sakura_rush_scores.json` + `weathernews_all_spots.json` 合并读取（ISR 30min）。页面结构为 Hero → WeeklyRush → BloomTimeline → CityTabs → SpotCards。缺失：地图、景点详情、数据源信任、转化层、SEO 文本层、埋点。

旧版 HTML 页面（`archive/legacy-html/`）有完整的 Leaflet 地图 + 左侧排行 + 右侧详情三栏布局，6 大数据源展示，但和主站完全割裂。

当前 Vercel 部署地址：`web-rust-theta-98.vercel.app`。后端通过 Cloudflare Tunnel 穿透本地。

## Goals / Non-Goals

**Goals:**
1. /rush 页面具备独立用户价值——用户来了能真正做花期决策，不是看两眼就走
2. 自然转化——每个有价值的交互点附近都有通向定制服务的路径，但不打断用户体验
3. 搜索可发现——SEO 文本层让搜索引擎和 AI 摘要能抓取核心内容
4. 可分享——用户看到好景点能生成卡片分享给同行人
5. 品牌统一——和主站同一套视觉语言
6. 性能达标——首屏 LCP < 2s，地图延迟加载不阻塞首屏

**Non-Goals:**
- 不做实时推送（WebSocket / SSE）——ISR 30min 够用
- 不做用户账号体系——「加入行程」先存 localStorage，登录后再关联
- 不做独立的樱花数据后台——数据通过爬虫脚本更新 JSON 文件
- 不做樱花预测模型——直接用爬虫抓取的数据源融合结果
- 不做小红书内容生成——单独截图交付

## Decisions

### D1: 页面信息架构（6 屏）

**决定**：保留 6 屏结构，但调整顺序和权重。

```
Screen 1 — Hero + 实时状态摘要       → 「我来对了」的确认感
Screen 2 — 城市景点排行（核心功能区） → 「现在该冲哪」的决策
Screen 3 — 花期时间轴（城市对比）     → 「什么时候去最好」的规划
Screen 4 — Leaflet 实时地图          → 「景点在哪 / 怎么走」的深入探索
Screen 5 — 数据来源信任条            → 「这数据靠谱吗」的信任建设
Screen 6 — 转化层                    → 「帮我安排进行程」的行动
```

**为什么把排行榜放第二屏而不是时间轴**：用户来 /rush 最核心的问题是「现在去哪」而不是「什么时候去」。排行榜直接回答第一个问题，时间轴回答第二个。大多数用户已经买了机票、时间已定，所以「去哪个景点」比「什么时候去」更紧迫。

**替代方案**：时间轴放第二屏 → 被否决，因为时间轴是规划型信息，不是决策型信息，放太前面会让用户觉得「信息太多还没看到重点」。

### D2: 景点详情用 Slideover 抽屉而不是固定侧边栏

**决定**：点击景点卡片 → 从右侧滑出 Slideover 抽屉，覆盖在页面上方。

**为什么**：
- 移动端友好——旧版三栏布局在手机上完全不可用
- 不跳页——用户不会丢失当前位置
- 可以放更丰富的内容（比固定侧边栏空间大）

**抽屉必须包含**：景点照片（大图）、花期日历条、能冲指数 + 简要理由、树木数量、夜樱/祭典信息、地图小卡片（单点定位）、「帮我安排进行程」CTA
**抽屉不包含**：评论、UGC 内容、历史花期对比（过于深度）

### D3: Leaflet 地图放第四屏 + 延迟加载

**决定**：地图不在首屏，放在排行榜和时间轴之后。使用 `dynamic import` + `IntersectionObserver` 延迟加载。

**为什么**：
- Leaflet JS + 瓦片加载约 200KB+，放首屏会严重拖慢 LCP
- 80% 用户只需要排行榜就够了，地图是 20% 深度用户的工具
- 延迟加载后，滚动到地图区域时再初始化

**地图交互**：
- 标记按花期状态着色（绿=未开、浅粉=咲き始め、粉=五分咲、红=満開、灰=散り）
- 点击标记 → 弹出 Popup（名称 + 分数 + 状态 + 「查看详情」按钮 → 打开 Slideover）
- 城市 Tab 切换时地图自动 flyTo 对应城市中心

### D4: 转化触点分层——不打断体验

**决定**：3 层 CTA 策略。

| 层级 | 位置 | 形式 | 文案方向 |
|------|------|------|---------|
| **Ambient** | Hero 次 CTA、每张景点卡片角标 | 小文字/图标 | 「加入行程」「安排进行程」 |
| **Contextual** | 排行榜底部、地图下方 | 行内横幅 | 「这些景点帮你排好一天的路线」 |
| **Dedicated** | 第六屏转化层 | 完整三卡片 | 「安排赏樱行程 / 看样例 / 加微信」 |

**为什么不用 Floating CTA**：/rush 是内容型页面，不是成交页。浮动 CTA 会让用户觉得「这个页面就是想卖我东西」，破坏工具型页面的信任感。在 /preview 页面用浮动 CTA 合理（用户已经看到了自己的行程），在 /rush 不合理。

### D5: 数据源信任模块——精简但有力

**决定**：不再用旧版的 6 张大卡片占满一屏。改为：
- 一行 Logo Strip（JMA / JMC / Weathernews / 地方官方 / 历史数据 / AI 融合）
- 下方一句话：「融合日本气象厅等 6 大权威数据源，每天更新 3 次」
- 可展开折叠：「了解我们的数据有多硬 ↓」→ 展开后显示简要描述

**为什么**：信任不需要长篇大论。用户看到「日本气象厅」就够了，不需要知道 JMA 有 58 个观测点。但对于高意图用户，折叠展开让他们可以深入了解。

### D6: SEO 文本层设计

**决定**：每个城市 Tab 下方有一段**不可见但可索引的文本摘要**（`<details>` 或 `sr-only` + 可展开）。

结构：
```
H1: 2026 日本樱花实时追踪 — 240+ 景点花期数据
H2: {城市名} 赏樱景点排行
  → 可读文本摘要（200-300字，包含景点名、花期、特色）
H2: 花期时间轴 — 什么时候去最好
H2: 实时樱花地图
H2: 常见问题（FAQ schema）
  → 5 个 Q&A，覆盖「现在去日本能看到樱花吗」「最佳赏樱时间」等
```

**为什么**：纯交互型页面（地图+卡片）搜索引擎看不到有价值内容。加了文本层后，Google 和百度能索引到「2026 东京樱花满开时间 3 月 29 日」这样的高价值关键词。

### D7: 分享机制

**决定**：
- 景点卡片右上角 → 「分享」按钮 → 调用 `navigator.share` 或生成分享链接
- 分享链接格式：`/rush?city=tokyo&spot=ueno` → 落地后直接滚动到对应城市并高亮该景点
- 后续（P1）：Satori 生成分享卡图片

### D8: 移动端适配

**决定**：
- **桌面**：排行榜 4 列网格 + 地图全宽
- **平板**：3 列网格
- **手机**：2 列网格 + 地图占满屏高 + Slideover 从底部滑出（而不是右侧）
- 城市 Tab 在手机上改为可左右滚动的 ScrollArea

## Risks / Trade-offs

| 风险 | 影响 | 缓解 |
|------|------|------|
| Leaflet 在 SSR 环境报错 | 构建失败 | `dynamic(() => import(...), { ssr: false })` 关闭 SSR |
| weathernews 图片 CDN 被墙 | 国内用户看不到照片 | 配置 next/image 代理 + 降级 placeholder |
| 城市/景点数据量大，首屏慢 | LCP > 2s | 排行榜只加载当前城市 Top 12，其他懒加载 |
| SEO 文本层过于人工 | 被 Google 判为门页 | 文本必须有真实信息价值，不是关键词堆砌 |
| 分享链接打开后需要滚动定位 | 用户困惑 | URL 参数 → `scrollIntoView` + 高亮动画 |
