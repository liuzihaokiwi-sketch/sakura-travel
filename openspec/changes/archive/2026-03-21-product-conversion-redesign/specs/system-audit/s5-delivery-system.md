### 1.3 核心问题

当前交付物 = 封面 + N天时间轴 + 一页 tips。

用户收到的感受：**"就是个行程表"**。缺少"替我想了很多"的层次感。

具体缺失：

| 缺失维度 | 影响 | 严重度 |
|---|---|---|
| 没有行程总纲（为什么这么安排） | 用户不知道路线逻辑，觉得是随便排的 | 🔴 高 |
| 没有节奏说明 | 用户不知道哪天松哪天紧，到了当地容易乱 | 🔴 高 |
| 没有"开篇一封信" | 缺少个人感/温度，像份冷冰冰的报告 | 🔴 高 |
| 没有预算感知 | 花了 ¥248 但不知道这趟大概花多少，心里没底 | 🟡 中 |
| 没有预订优先级 | 到了日本才发现该提前订的没订 | 🟡 中 |
| 没有 PlanB 总览 | 遇到变化时不知道怎么临时调整 | 🟡 中 |
| 避坑页没有接入 | 模板写好了但没展示，浪费 | 🟡 中 |
| 餐厅/住宿/拍照总览没有接入 | 分散在每日中看不到全貌 | 🟡 中 |
| 没有"最容易后悔的错误" | 少了一层"替你踩过坑"的贴心感 | 🟢 低（可合并到避坑页） |

## 二、目标交付物结构：30-40 页完整手册

### 2.1 总体页数分配

一份 7 天行程的完整手册目标 **32-38 页**，结构如下：

| 区域 | 页面 | 页数 | 类型 |
|---|---|---|---|
| **开篇** | 封面 | 1 | 固定 |
| | 给你的一封信（开篇寄语） | 1 | 固定 |
| | 行程总览+节奏图 | 1 | 固定 |
| **每日详情** | Day 1-7 × 每天 3-4 页 | 21-28 | 固定+条件 |
| **专题汇总** | 全行程餐厅地图 | 1 | 固定 |
| | 住宿区域指南 | 1 | 固定 |
| | PlanB 应变方案 | 1 | 条件 |
| | 预订优先级清单 | 1 | 固定 |
| **出发前** | 行前准备清单 | 1 | 固定 |
| | 避坑指南 | 1 | 固定 |
| | 预算参考 | 1 | 条件 |
| **尾页** | 封底（精调入口+升级提示） | 1 | 固定 |
| **合计** | | **32-38** | |

### 2.2 每天固定页 + 条件页

每天由 **3 页固定 + 0-1 页条件** 组成：

**固定页（每天必出）：**

| 页 | 内容 | 数据来源 |
|---|---|---|
| **日主题页** | Day X 标题 + 一句场景描述 + 当日关键词标签 + 区域示意 | 模板 + 规则 |
| **时间轴页** | 完整时间轴（景点+交通+餐厅，含推荐理由） | 数据库 + 文案润色 |
| **当日小结页** | 交通总结 + 花费参考 + 一句收尾话 + 次日预告 | 规则计算 + 模板 |

**条件页（满足条件时追加）：**

| 条件 | 追加页 | 触发规则 |
|---|---|---|
| 当日有 ≥2 个拍照热点 | 📸 当日出片指南 | 实体标签含 `photo_spot` |
| 当日有大型市场/商圈 | 🛍️ 购物攻略页 | 实体标签含 `shopping` |
| 当日有季节限定体验 | 🌸 限定体验说明 | `scene` 字段 = sakura/momiji/matsuri |
| 当日景点有闭馆风险 | ⚠️ 当日应变提示 | 实体有定休日且与行程日匹配 |

## 三、新增页面详细设计

### 3.1 给你的一封信（开篇寄语）

**位置**：封面之后第一页
**作用**：让用户一打开就觉得"这是为我准备的"，不是批量生产的模板

**内容结构**：
```
亲爱的 [用户称呼]，

这份行程是根据你说的"[核心偏好摘要]"来安排的。
[城市] [天数] 天，我帮你把路线理了一遍——
哪天松、哪天紧、哪顿饭值得专门跑一趟、
哪个景点几点去人最少，都写在里面了。

几个重要提醒先说：
· [最需要提前预订的事项]
· [最容易踩的坑]
· [最值得期待的亮点]

翻开看看吧，祝你旅行愉快。
```

**生成方式**：模板骨架 + 动态填充（用户偏好 + 行程亮点提取）
**token 消耗**：低（80%模板，20%动态填充字段）

---

**位置**：开篇寄语之后
**作用**：让用户一眼看清 7 天全貌，知道每天的松紧度

**内容结构**：
```
┌──────────────────────────────────────────┐
│  你的 7 天，是这样安排的                    │
│                                          │
│  Day 1  ▓▓▓░░░░░  轻松      浅草·上野     │
│  Day 2  ▓▓▓▓▓░░░  适中      秋叶原·银座   │
│  Day 3  ▓▓▓▓▓▓▓░  充实      新宿·涩谷     │
│  Day 4  ▓▓▓▓░░░░  适中      �的�的根      │
│  Day 5  ▓▓░░░░░░  休息日    自由时间       │
│  Day 6  ▓▓▓▓▓▓░░  充实      京都（日归）   │
│  Day 7  ▓▓▓░░░░░  轻松      购物+机场      │
│                                          │
│  这样安排的原因：                           │
│  · 前两天适应时差，从轻松的区域开始          │
│  · 第3天精力最好，安排最丰富的路线           │
│  · 第5天故意留白——旅行到中段需要喘口气       │
│  · 最后一天不赶景点，留够时间买伴手礼和去机场 │
└──────────────────────────────────────────┘
```

**生成方式**：纯规则计算（每日实体数量 → 强度等级 → 节奏条 + 说明句）
**token 消耗**：零（规则模板）

### 3.3 预订优先级清单

**位置**：专题汇总区
**作用**：告诉用户"现在就该订什么、到了再订什么"

**内容结构**：
```
┌──────────────────────────────────────────┐
│  🔔 预订优先级                             │
│                                          │
│  ❗ 现在就订（出发前 2 周以上）              │

│  · [热门餐厅A]（Day 3 午餐，需提前预约）     │
│  · [特定体验B]（如茶道、和服，热门日期抢手）  │
│                                          │
│  📋 出发前 3 天确认                        │
│  · 酒店（确认地址、check-in 时间）          │
│  · 交通卡（建议在国内提前买好）              │
│  · Wi-Fi / eSIM                          │
│                                          │
│  🏃 到了再说就行                           │
│  · 大部分景点门票（现场买即可）              │
│  · 便利店/药妆店（随到随逛）                │
│  · 地铁一日券（当天在车站买）                │
└──────────────────────────────────────────┘
```

**生成方式**：规则引擎（扫描行程中的实体标签，标记 `reservation_required`/`advance_booking` 的提取出来）
**token 消耗**：零（规则模板）

## 四、内容生成分配矩阵

### 4.1 三种生成方式定义

| 方式 | 定义 | token 成本 | 适用场景 |
|---|---|---|---|
| **📋 模板** | 固定 HTML 骨架 + 变量填充，零 LLM 调用 | 0 | 结构固定、数据驱动的内容 |
| **⚙️ 规则** | 代码逻辑计算 + 模板句组装，零 LLM 调用 | 0 | 可枚举的条件判断、数值计算 |
| **✍️ 动态润色** | LLM 基于数据写 1-3 句话 | 低-中 | 需要自然语言表达的个性化内容 |

### 4.2 完整分配表

| 页面 | 模板 📋 | 规则 ⚙️ | 动态润色 ✍️ | 说明 |
|---|---|---|---|---|
| **封面** | 90% | 10% | 0% | 城市名+日期+标签填入模板 |
| **开篇寄语** | 60% | 10% | 30% | 骨架固定，3个动态要点需润色 |
| **行程总览+节奏** | 50% | 50% | 0% | 强度条由实体数计算，说明句由规则匹配 |
| **每日主题页** | 70% | 10% | 20% | 标题+标签模板化，场景描述一句话润色 |
| **每日时间轴** | 40% | 20% | 40% | 骨架模板，推荐理由需动态写 |
| **每日小结页** | 60% | 35% | 5% | 数据汇总为规则，"明天预告"一句话润色 |
| **条件页：出片指南** | 70% | 20% | 10% | 机位信息模板化，光线建议半规则半润色 |
| **条件页：购物攻略** | 80% | 15% | 5% | 店铺列表+营业时间，纯数据 |
| **条件页：限定体验** | 50% | 10% | 40% | 季节内容需要生动描述 |
| **条件页：当日应变** | 60% | 30% | 10% | 规则匹配定休日+备选实体 |

| **住宿区域指南** | 60% | 20% | 20% | 区域比较需要自然语言解释 |
| **PlanB 总览** | 50% | 30% | 20% | 触发条件规则判断，应变建议需润色 |
| **预订优先级** | 70% | 30% | 0% | 纯规则：扫描标签，按优先级排列 |
| **行前准备清单** | 90% | 10% | 0% | 几乎全模板，按城市/季节微调 |
| **避坑指南** | 60% | 10% | 30% | 通用坑模板化，城市特有坑需润色 |
| **预算参考** | 40% | 55% | 5% | 费用计算为规则，省钱提示模板化 |

## 七、回答六个核心问题

### Q1: 还缺哪些让用户觉得值的层？

见上方第一章和第二章。核心缺三层：
1. **情感层**：开篇寄语、节奏说明——让用户觉得"这是为我做的"
2. **实用层**：预订清单、预算参考、PlanB 总览——让用户觉得"什么都替我想了"
3. **完整层**：每日小结、专题汇总——让手册有头有尾、有总有分

### Q2: 开头总纲是否足够？

**不够。** 当前只有封面，缺少两个关键页面：
- **开篇寄语**：建立信任感和个人连接
- **行程总览+节奏图**：让用户一眼理解"为什么这么排"

### Q3: 每天是否应固定 3-4 页 + 条件页？

**是的，建议每天 3 页固定 + 0-1 页条件：**
- 固定：日主题页 → 时间轴页 → 当日小结页
- 条件：出片指南/购物攻略/限定体验/当日应变（按标签触发）
- 这样 7 天 = 21-28 页，加上开篇+专题+尾页 = 32-38 页，恰好 30-40 页

### Q4: 哪些适合模板化/规则/动态写？

见第四章分配矩阵。核心原则：
- **结构和框架 → 模板**（封面、清单、骨架）
- **可计算的内容 → 规则**（预算、强度条、预订优先级、交通汇总）
- **需要自然语言的个性化表达 → 动态润色**（推荐理由、场景描述、应变建议）
- 整本手册只需约 5,000 token 润色量

### Q5: 怎样厚、细、完整但不浪费 token？

**关键策略**：
1. 80% 的页面由模板+规则生成，零 token
2. 推荐理由是 token 大头（2,800/5,000），但也是用户感知价值最高的部分，值得投入
3. 条件页机制避免"所有人看到一样的东西"，也避免为不需要的页面浪费 token
4. 每日小结页几乎纯规则（步行距离、交通段、花费），但给用户"每天都有收尾"的完整感

### Q6: 是否需要预订优先级页/PlanB页/预算感知页/避坑页/节奏说明页？

| 页面 | 结论 | 理由 |
|---|---|---|
| **预订优先级页** | ✅ 需要 | 用户最怕"该订的没订"，这一页实用价值极高 |
| **PlanB 总览页** | ✅ 需要（条件） | ≥3个应变日时生成，少于3个则内嵌到每日即可 |
| **预算参考页** | ✅ 需要（条件） | 非管家版用户或预算敏感用户展示 |
| **避坑指南页** | ✅ 需要 | 模板已有，只需接入，成本为零 |
| **节奏说明页** | ✅ 需要 | 合并到行程总览页，不单独成页 |
| **"最容易后悔的错误"页** | ❌ 不单独做 | 合并到避坑指南中作为一个小节，避免页面过多 |
