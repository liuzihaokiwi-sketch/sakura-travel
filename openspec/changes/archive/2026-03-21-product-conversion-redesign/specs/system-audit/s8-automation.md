## 二、每个任务详细说明

### A1. 数据快照刷新

**做什么**：对 entity_base / pois / hotels / restaurants / entity_tags 等核心表做增量快照，记录变更历史。

**为什么需要**：
- 数据会随爬虫更新而变化（价格、营业状态、评价）
- 已交付的行程引用的是交付时的数据，但样张和新订单需要最新数据
- 快照用于数据回溯和问题排查

**执行细节**：

| 项目 | 说明 |
|---|---|
| 频率 | 每周一次（周一凌晨 3:00） |
| 触发方式 | Cron 定时任务 |
| 输入 | 当前数据库表 |
| 输出 | `snapshots` 表新增记录 + 变更 diff 日志 |
| 耗时 | 约 5-15 分钟 |
| 人工兜底 | **不需要** |
| 可全自动 | **✅ 是** |
| 失败处理 | 记录日志 + 发消息通知，下次重跑 |

**实现参考**：已有 `app/core/snapshots.py`，可复用。

---

### A2. 图片可用性检查

**做什么**：遍历 `entity_media` 表中所有图片 URL，逐一 HEAD 请求检查是否仍可访问（200），标记失效图片。

**为什么需要**：
- 外链图片（Unsplash、Google Places）会失效
- 失效图片出现在交付页上会让用户觉得不靠谱
- 及时发现 → 替换为备选图或城市默认图

**执行细节**：

| 项目 | 说明 |
|---|---|
| 频率 | 每周一次（周三凌晨 4:00） |
| 触发方式 | Cron 定时任务 |
| 输入 | `entity_media` 表全量 URL |
| 输出 | 失效图片列表 → 标记 `status = broken` |
| 耗时 | 约 10-30 分钟（取决于图片总量，可并发 50 请求） |
| 人工兜底 | **需要** — 失效图片需人工决定替换为什么 |
| 可全自动 | **部分** — 检测自动，替换决策需人工（或降级为城市默认图） |
| 失败处理 | 超时/网络错误的 URL 重试 1 次，仍失败则标记 `status = check_failed` |

**降级策略**：
- 如果某实体所有图片都失效 → 自动降级为城市默认图（`data/city_defaults/`）
- 降级后发通知，人工在下一个工作日补充新图

### A3. 热门城市内容刷新

**做什么**：对热门城市（东京、大阪、京都、北海道、冲绳）重新跑爬虫管线，更新餐厅/景点/酒店/活动数据。

**为什么需要**：
- 餐厅会关店、改菜单、调价格
- 景点会改营业时间、整修
- 新开的店/新展览需要入库
- 保持数据新鲜度是交付质量的基础

**执行细节**：

| 项目 | 说明 |
|---|---|
| 频率 | 每 2 周一次（每月 1 日和 15 日凌晨 2:00） |
| 触发方式 | Cron 定时任务 |
| 输入 | 热门城市 city_code 列表 |
| 输出 | `entity_base` + 子表更新记录、新增实体入库 |
| 耗时 | 约 1-3 小时（5 个城市，每城市 30-40 分钟） |
| 人工兜底 | **需要** — 新增实体需要人工审核标签和分类 |
| 可全自动 | **部分** — 爬取+入库自动，新增实体的标签/分类/图片精选需人工 |
| 失败处理 | 单个爬虫失败不影响其他城市，记录失败原因 + 通知 |

**已有实现**：
- `scripts/crawlers/tabelog.py` — 餐厅
- `scripts/crawlers/hotels.py` — 酒店
- `scripts/crawlers/experiences.py` — 体验
- `scripts/crawlers/events.py` — 活动
- 需新增：统一调度脚本，按城市批量调用

### A4. 花期/活动/季节变动监测

**做什么**：监控樱花花期、红叶时间、祭典日期、特殊活动变更，及时更新数据和行程模板。

**为什么需要**：
- 花期预测会随天气变化频繁调整（特别是开花前 2 周）
- 祭典/活动可能临时取消或改期
- 交付的行程如果花期信息过时，用户体验大打折扣

**执行细节**：

| 项目 | 说明 |
|---|---|
| 频率 | **季节期**：每日一次（凌晨 6:00）；**非季节期**：每周一次 |
| 季节期定义 | 樱花季 3/1-4/30、红叶季 10/1-12/15、年末年始 12/20-1/10 |
| 触发方式 | Cron 定时任务 |
| 输入 | 气象厅/weathernews 花期数据、活动网站 |
| 输出 | `data/sakura/` 数据更新 + 行程模板中花期日期字段更新 |
| 耗时 | 约 5-10 分钟 |
| 人工兜底 | **需要** — 花期大幅变动时需判断是否需要通知已购买用户 |
| 可全自动 | **部分** — 数据抓取自动，"是否需要通知用户"的决策需人工 |
| 失败处理 | 抓取失败时保留上次数据，不覆盖 |

**已有实现**：
- `scripts/crawlers/sakura_pipeline/` — 花期爬虫
- `data/sakura/` — 花期数据存储

### A5. 小红书内容预生成

**做什么**：基于内容支柱和排期表，预先生成下一周的小红书图文草稿（标题 + 正文 + 配图建议）。

**为什么需要**：
- 内容引流是获客主渠道
- 提前预生成可以批量审核，减少日常运营压力
- 避免临时生成导致质量不稳定

**执行细节**：

| 项目 | 说明 |
|---|---|
| 频率 | 每周一次（周日晚 21:00） |
| 触发方式 | Cron 定时任务 |
| 输入 | 内容排期表 + 内容支柱模板 + 当前热点事件 |
| 输出 | 5-7 篇图文草稿（存入 `content_drafts` 表） |
| 耗时 | 约 10-20 分钟 |
| 人工兜底 | **需要** — 所有草稿必须人工审核后才能发布 |
| 可全自动 | **部分** — 生成自动，审核+发布人工 |
| 失败处理 | 生成失败的草稿标记 `status = failed`，人工补写 |

**工作流**：
```
周日晚自动生成 → 周一上午人工审核 → 审核通过的进入排期队列 → 按排期自动提醒发布
```

### A6. 内容排期与发布提醒

**做什么**：按排期表在指定时间提醒运营发布内容（不自动发布到平台，因小红书无 API）。

**为什么需要**：
- 固定发布节奏比随机发布效果好
- 人工发布容易忘记或错过最佳时间
- 排期提醒确保内容不断档

**执行细节**：

| 项目 | 说明 |
|---|---|
| 频率 | 每日（按排期表，通常 10:00 / 18:00 / 21:00 三个时段） |
| 触发方式 | Cron 定时任务 |
| 输入 | `content_schedule` 表 |
| 输出 | 微信/飞书消息提醒 + 待发布内容链接 |
| 耗时 | 秒级 |
| 人工兜底 | **需要** — 必须人工手动发布到小红书 |
| 可全自动 | **部分** — 提醒自动，发布人工 |
| 失败处理 | 提醒失败则 30 分钟后重发 |

### A7. 高意向评论/私信提取

**做什么**：从小红书评论区和私信中提取高意向用户关键词（"怎么定制""多少钱""有链接吗"等）。

**为什么需要**：
- 评论区是主要转化入口
- 快速回复高意向评论可显著提升转化率
- 漏回复 = 丢客户

**执行细节**：

| 项目 | 说明 |
|---|---|
| 频率 | 每日 3 次（10:00 / 14:00 / 20:00） |
| 触发方式 | Cron 定时任务 |
| 输入 | 小红书评论/私信数据（手动导出或浏览器插件辅助） |
| 输出 | 高意向评论列表 + 建议回复话术 → 推送到运营微信 |
| 耗时 | 约 2-5 分钟 |
| 人工兜底 | **需要** — 回复必须人工操作 |
| 可全自动 | **部分** — 提取+分类自动，回复人工 |
| 失败处理 | 无法提取时提醒人工手动检查 |

**注意**：小红书无官方 API，数据提取需浏览器插件或手动导出。初期可完全人工，后期再考虑半自动化。

### A8. FAQ 内容更新

**做什么**：根据最近 30 天的用户问题、评论、私信，识别高频新问题，更新 FAQ 页面。

**为什么需要**：
- 用户问题会随季节和热点变化（签证政策变更、新航线开通、汇率波动等）
- FAQ 过时会增加客服压力
- 及时更新 FAQ 可减少重复问答

**执行细节**：

| 项目 | 说明 |
|---|---|
| 频率 | 每月一次（每月 1 日） |
| 触发方式 | Cron 定时任务 |
| 输入 | 近 30 天用户问题记录 + 当前 FAQ 列表 |
| 输出 | 新增/修改 FAQ 条目草稿 → 推送给运营审核 |
| 耗时 | 约 5-10 分钟 |
| 人工兜底 | **需要** — FAQ 涉及准确性，必须人工审核 |
| 可全自动 | **部分** — 高频问题识别自动，FAQ 编写+审核人工 |
| 失败处理 | 生成失败则提醒人工手动更新 |

### A9. 热门路线样张更新

**做什么**：重新生成热门路线（东京经典 5 日、关西经典 7 日等）的展示样张，用于官网/小红书/免费预览。

**为什么需要**：
- 样张是用户第一次看到交付物质量的窗口
- 数据更新后样张应同步更新（餐厅变了、景点变了）
- 季节变化时样张需体现时令内容（樱花季 vs 红叶季 vs 常规）

**执行细节**：

| 项目 | 说明 |
|---|---|
| 频率 | 每月一次（每月 5 日凌晨） + 季节切换时额外触发 |
| 触发方式 | Cron 定时任务 + 手动触发 |
| 输入 | `route_templates` + 最新实体数据 |
| 输出 | 更新的样张 HTML/PDF → `exports/` 目录 |
| 耗时 | 约 20-40 分钟（6-8 个热门路线） |
| 人工兜底 | **需要** — 样张是面向用户的，需审核排版和内容 |
| 可全自动 | **部分** — 生成自动，审核人工 |
| 失败处理 | 单个路线生成失败不影响其他，保留上版样张 |

**已有实现**：
- `data/route_templates/` — 路线模板数据
- `exports/` — 样张输出目录
- 渲染管线已可用

### A10. 餐厅/景点营业状态校验

**做什么**：校验行程中引用的实体是否仍在营业（餐厅关门、景点装修、临时休业等）。

**为什么需要**：
- 交付行程推荐了一家已关门的餐厅 = 最大的信任崩塌
- 日本餐厅关店率不低，特别是小店
- 是影响履约质量的核心风险

**执行细节**：

| 项目 | 说明 |
|---|---|
| 频率 | 每 2 周一次（与 A3 配合） |
| 触发方式 | Cron 定时任务 |
| 输入 | 近 30 天内被引用过的实体列表 |
| 输出 | 异常实体列表（可能关店/改时间/装修）→ 标记 `status = needs_review` |
| 耗时 | 约 15-30 分钟 |
| 人工兜底 | **需要** — 确认是否真的关了，以及替换方案 |
| 可全自动 | **部分** — 检测自动（Google Maps 状态+Tabelog 更新时间），确认和替换人工 |
| 失败处理 | 检测失败的实体不变更状态，仅记录日志 |

**检测方法**：
1. Google Places API 查询 `business_status`（OPERATIONAL / CLOSED_TEMPORARILY / CLOSED_PERMANENTLY）
2. Tabelog 页面是否仍可访问
3. 实体最后更新时间是否超过 90 天（标记为 `stale`）

### A11. 交付物模板预热

**做什么**：预编译 Jinja2 模板、预加载通用资源（城市默认图、通用文案块），减少交付时的冷启动时间。

**为什么需要**：
- 用户付款后等待时间越短，满意度越高
- 模板编译和资源加载在首次请求时较慢
- 预热后交付速度从 8-12 秒降到 2-4 秒

**执行细节**：

| 项目 | 说明 |
|---|---|
| 频率 | 每日凌晨 5:00 |
| 触发方式 | Cron 定时任务 |
| 输入 | `templates/magazine/` 全部模板文件 |
| 输出 | 编译缓存 + 预热日志 |
| 耗时 | 约 1-2 分钟 |
| 人工兜底 | **不需要** |
| 可全自动 | **✅ 是** |
| 失败处理 | 预热失败不影响运行（降级为首次请求时编译） |

### A12. 爬虫健康巡检

**做什么**：检查所有爬虫（Tabelog、Hotels、Events、Experiences、Google Flights、Weathernews）是否能正常运行，发现目标网站结构变化或反爬策略升级。

**为什么需要**：
- 爬虫是数据管线的源头，爬虫挂了 = 数据不更新
- 目标网站经常改版，爬虫 selector 会失效
- 早发现早修复，避免数据断供

**执行细节**：

| 项目 | 说明 |
|---|---|
| 频率 | 每日凌晨 4:00 |
| 触发方式 | Cron 定时任务 |
| 输入 | 每个爬虫跑一个最小样本（1-2 条数据） |
| 输出 | 健康状态报告：正常/异常/降级 → 推送通知 |
| 耗时 | 约 3-8 分钟（所有爬虫串行跑一遍） |
| 人工兜底 | **需要** — 爬虫异常时需开发人员修复 |
| 可全自动 | **部分** — 检测自动，修复需人工 |
| 失败处理 | 连续 2 天异常则升级为紧急通知 |

**已有爬虫清单**：
| 爬虫 | 文件 | 依赖 |
|---|---|---|
| Tabelog | `scripts/crawlers/tabelog.py` | Playwright |
| Hotels | `scripts/crawlers/hotels.py` | HTTP API |
| Events | `scripts/crawlers/events.py` | HTTP API |
| Experiences | `scripts/crawlers/experiences.py` | HTTP API |
| Google Flights | `scripts/crawlers/google_flights.py` | Playwright |
| Sakura | `scripts/crawlers/sakura_pipeline/` | HTTP |
| JNTO | `scripts/crawlers/jnto.py` | HTTP |

### A13. 过期行程清理

**做什么**：清理旅行日期已过 30 天以上的行程数据，释放存储空间，保留统计摘要。

**为什么需要**：
- 过期行程不再被用户访问
- 但行程中的选择数据（哪些实体被选中、用户偏好）有统计价值
- 需要"清理详情，保留摘要"的策略

**执行细节**：

| 项目 | 说明 |
|---|---|
| 频率 | 每周一次（周日凌晨 3:00） |
| 触发方式 | Cron 定时任务 |
| 输入 | `itinerary_plan` 表中 `travel_end_date < now() - 30天` 的记录 |
| 输出 | 详情数据归档/删除 + 统计摘要保留 |
| 耗时 | 约 2-5 分钟 |
| 人工兜底 | **不需要** |
| 可全自动 | **✅ 是** |
| 失败处理 | 清理失败不影响业务，下次重跑 |

**清理策略**：
- `itinerary_item` 详情 → 删除
- `itinerary_day` → 删除
- `itinerary_plan` → 保留，`status` 改为 `archived`
- 生成的 HTML/PDF 文件 → 删除（`exports/` 目录）

### A14. 用户反馈汇总

**做什么**：汇总最近一周的用户反馈（微信聊天记录中的评价、修改请求类型、常见不满），生成周报。

**为什么需要**：
- 用户反馈是产品迭代的核心输入
- 分散在微信聊天中的反馈容易遗漏
- 周报帮助发现系统性问题（比如"连续 5 个用户都觉得交通说明不够细"）

**执行细节**：

| 项目 | 说明 |
|---|---|
| 频率 | 每周一次（周一上午 9:00） |
| 触发方式 | Cron 定时任务 + 人工录入原始数据 |
| 输入 | 人工录入的用户反馈标签（结构化） |
| 输出 | 周报：高频问题 top 5 + 趋势变化 + 建议行动项 |
| 耗时 | 生成约 5 分钟，录入约 15 分钟/周 |
| 人工兜底 | **需要** — 原始数据录入需人工 |
| 可全自动 | **部分** — 汇总分析自动，数据采集人工 |
| 失败处理 | 无数据时跳过，不生成空报告 |

## 三、自动化程度分级总结

### 3.1 完全自动化（零人工）

| 任务 | 说明 |
|---|---|
| A1 数据快照刷新 | 纯数据操作，无判断需求 |
| A11 交付物模板预热 | 纯技术操作 |
| A13 过期行程清理 | 规则明确，无二义性 |

### 3.2 自动执行 + 人工审核

## 五、技术实现建议

### 5.1 任务调度方案

| 方案 | 适用阶段 | 优势 | 劣势 |
|---|---|---|---|
| **系统 Cron + Python 脚本** | 早期（当前） | 简单、零依赖 | 无重试、无监控面板 |
| **APScheduler** | 中期 | Python 原生、支持重试 | 单进程，不支持分布式 |
| **Celery Beat** | 后期 | 分布式、重试、监控 | 依赖 Redis/RabbitMQ |

**建议**：当前阶段用 Cron + Python 脚本即可，配合已有的 `app/workers/` 框架。等任务量超过 10 个且需要重试机制时再迁移到 Celery Beat。

### 5.2 通知渠道

| 通知类型 | 渠道 | 说明 |
|---|---|---|
| 任务成功 | 日志文件 | 不主动通知，查看即可 |
| 任务失败 | 企业微信 webhook | 即时通知 |
| 需人工审核 | 企业微信 webhook + 看板标记 | 带链接直达审核页 |
| 紧急问题 | 企业微信 + 手机通知 | 连续失败/核心数据异常 |

### 5.3 Workers 框架集成

已有 `app/workers/__main__.py` 和 `app/workers/jobs/`，建议新增任务目录结构：
