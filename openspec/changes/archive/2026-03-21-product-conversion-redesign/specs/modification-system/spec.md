## ADDED Requirements

### Requirement: 用户端结构化修改
修改页 `/plan/[id]/edit` SHALL 只提供结构化修改选项：

| 可修改项 | 交互方式 | 说明 |
|---|---|---|
| 某天节奏太紧/太松 | 滑杆（紧凑↔️宽松） | 调节后重新编排当天 |
| 替换某个景点 | 点击景点→弹出3个备选 | 备选从同区域同类型中选 |
| 替换某家餐厅 | 点击餐厅→弹出3个备选 | 备选从同区域中选 |
| 调整某天侧重 | 标签切换（购物/文化/出片/美食/亲子）| 影响当天景点优先级 |
| 调整出发时间 | 时间选择器（8:00-11:00，步进30分钟） | 影响当天时间轴 |
| 删除某个景点 | 点击景点→"不想去这里"按钮 | 删除后系统自动填充或留空闲 |

### Requirement: 用户端不可修改项
- ❌ 不能自由输入文字修改要求
- ❌ 不能改整体路线骨架和城市
- ❌ 不能改旅行日期和同行人数
- ❌ 不能拖拽调整景点顺序
- ❌ 不能无限修改

**设计原则**：不可修改的项目直接不展示修改按钮。用户不知道不能改 → 没有挫败感。

### Requirement: 修改次数策略
> ⚠️ B2 裁决（strategic-upgrade/specs/hard-rules-system/spec.md §B2）— 以本节为唯一真相源

- 免费预览：**0次修改**（Day 1 完整展示，Day 2-7 锁定标题）
- 行程优化版（¥248）：**包含 2 次行程精调**
- 深度管家版（¥888）：**包含 5 次行程精调**
- 文案包装："包含 X 次行程精调" 而非 "限制 X 次修改"
- 超出次数：提示"需要更多调整？升级管家版可享更多精调"
- 校准理由：2次覆盖95%用户真实诉求，5次覆盖高需求用户，梯度设计强化升级动力

### Requirement: 次数计算规则
- 每次提交修改页面 = 消耗1次（不管改了几个项目）
- 未提交的修改不计次
- 同一次提交内可以改多个项目（鼓励一次想清楚再提交）

### Requirement: 修改文案包装（全程正向表达）

| 场景 | 文案 |
|---|---|
| 修改页顶部 | "你还有 1 次行程精调机会" |
| 提交修改时 | "提交后将消耗 1 次精调，确认提交？" |
| 剩余1次 | "最后 1 次精调机会，建议仔细看看再提交" |
| 次数用完 | "精调次数已用完。需要更多调整空间？" + [了解管家版] |
| 升级后 | "已升级！你现在有 2 次额外精调机会" |

禁用词："限制"、"上限"、"不能"、"禁止"、"权限不足"

### Requirement: 内部修改处理
内部审核端收到修改请求后：
1. 系统自动根据用户选择重新生成受影响的部分（AI做80%）
2. 审核人员检查生成结果是否合理
3. 如需微调：使用结构化切换（替换景点/餐厅）或提示词补充
4. 确认后发布新版本

### Requirement: 修改入口文案
默认推荐："想微调几个地方？换景点、调节奏，都可以 →"
- 位置：交付页每日详情之后、住宿建议之前
- 样式：圆角卡片，浅暖色背景（sakura-50）
- 展示剩余次数：卡片右上角小字 "还剩 X 次"
- 精调次数为0时：文案变为"想做更多调整？了解管家版 →"，跳转 upgrade 页
