## 14 坑详细（按问题/原因/防坑写）

### 一、Phase 1 发现池采集

#### 坑 1：合并脚本丢强信号

- **现象**：多源 CSV 合并时，强信号字段（OTA 评分、原始价格、米其林 Keys、Forbes 星级）被排除在 fieldnames 外，合并后只剩基础列。
- **原因**：合并脚本只定义了最小公共 schema。
- **防坑**：合并脚本 fieldnames 取所有来源列名的**并集**·来源特有字段宁可多带不可丢·后续归一化再处理。

#### 坑 2：菜系/类型分类混乱

- **现象**：多源各用不同语言/粒度的分类（日文「ラーメン」/英文「ramen」/中文「拉面」混用），合并后类型枚举膨胀到 70+ 种。
- **原因**：采集时直接用了原始平台的分类标签·没在入库前归一化。
- **防坑**：**采集时**就映射到标准码·维护 `cuisine_mapping.json`·标准枚举定义在 SCHEMA·新城市圈先审枚举·缺的先扩。

#### 坑 3：酒店 price_level 被默认值覆盖

- **现象**：80%+ 酒店标记为 `moderate`·几乎没有分层。
- **原因**：合并脚本对缺 price_level 的酒店统一填了默认值。
- **防坑**：缺字段时标 null·不填默认值·归一化步骤用回溯的原始价格重新分层·原始来源价格嵌在文本里的合并时正则提取。

#### 坑 4：Agent 输出落到错误目录

- **现象**：Agent 输出 CSV 落在项目根目录·不是约定的子目录。
- **原因**：Agent prompt 没指定输出路径·Agent 默认写到 CWD。
- **防坑**：Agent prompt 中**明确指定输出文件完整路径**·Agent 完成后立即检查·在批次表里写明每个 Agent 的输出路径。

#### 坑 5：Windows GBK 编码与 emoji 冲突

- **现象**：Python 脚本中的 emoji 在 Windows 控制台输出时报 `UnicodeEncodeError: 'gbk' codec`。
- **原因**：Windows 默认 stdout 编码是 GBK·无法显示 emoji。
- **防坑**：脚本中不用 emoji·用 ASCII 文本替代（`[ok]` / `[warn]` / `[pending]` / `[NO]`）·或在脚本开头 `sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding="utf-8")`·CSV 文件始终 `encoding='utf-8'`。

### 二、Phase 1→2 衔接（数据归一化）

#### 坑 6：跳过归一化直接进 Phase 2

- **现象**：Phase 1 完成后直接试图跑 selection model·发现分类混乱、price_level 没分层、缺 corridor 映射·无法分组排序。
- **原因**：文档中 Phase 1→2 之间没有明确的归一化步骤·或被认为可选。
- **防坑**：**N1-N5 归一化是文档化的必经步骤不是可选的**·归一化后检查：cuisine 唯一值 < 40 / price_level 分布合理 / corridor 非空率 > 80%。

#### 坑 7：budget_tier 用评分推断

- **现象**：用「评分 ≥3.8 + 非快餐 → premium」推价格层。
- **问题**：评分反映**口碑强弱不反映消费层级**。3.8 的拉面店可能 mid·3.5 的怀石可能 premium。
- **防坑**：**budget_tier 是价格层不是质量层**·推断优先级：已知价格 > 米其林非空→luxury > 菜系场景弱推断 > 保留原值标 `tier_confidence: inferred`。**绝不用评分推价格**。

#### 坑 8：base_quality_score 做全局 min-max 归一化

- **现象**：用 `(评分 - min) / (max - min)` 做全局线性归一化。
- **问题**：上下界数据集依赖·补一轮新数据就变·而且评分在不同品类语义不同（怀石 3.5 中等·咖啡甜品 3.5 已强）·全局抹掉品类差异。
- **防坑**：**组内 percentile 为主**·在 `城市 × 品类 × budget_tier` 分组内算 percentile rank·原始评分保留为辅助字段·不做全局归一化·不做跨平台线性换算。

#### 坑 9：酒店 base_quality 用弱代理变量

- **现象**：用「来源数量 + 字段丰富度」当主轴。
- **问题**：来源多不多、字段丰富不丰富是**弱信号**·原始数据已有 OTA 星级 / 米其林 Keys / Forbes 星·只是合并时被丢（即坑 1）。
- **防坑**：先回溯原始来源文件提取强信号·主轴用 OTA/Keys 评分·hotel_type/features 只做修正·无评分才回退到组内 median。

### 三、多源采集效率

#### 坑 10：单 Agent 跑了 6+ 小时

- **现象**：某抓爆款源 Agent 从 10+ 类目 × N 城市逐页抓取·耗时 6+ 小时。
- **原因**：Agent 没超时·逐条处理太慢·遇反爬不会自动跳过。
- **防坑**：Agent 设 2 小时超时·超时后有 fallback plan（用已有数据继续 / 换小规模抓取）·成熟类目只取 top N 页不做穷举。

#### 坑 11：等一个 Agent 阻塞全流程

- **现象**：某 Agent 长时间未完成·其他工作被阻塞。
- **防坑**：Phase 1 各 Agent 独立运行不互相依赖·每个完成后立即合并到候选池·合并后立即跑 go/no-go 检查·某 Agent 超时则用已有数据跑 Phase 1.5 决策框架。

#### 坑 12：多源重叠率低于预期

- **现象**：6 个来源中只有 14% 餐厅出现在 2+ 个来源中（目标 40%）。
- **原因**：不同来源覆盖不同品类（高端/游客热门/本地）·交叉少是正常的。
- **防坑**：多源置信度目标定 20-25% 更现实·indie_support_score 价值不在重叠在补充视角·Phase 2 编辑判断比机械重叠靠谱。

### 四、数据质量

#### 坑 13：CSV schema 不一致

- **现象**：某来源含特殊列·不在 DictWriter fieldnames 中·报 `dict contains fields not in fieldnames`。
- **原因**：不同来源 CSV 列名不同·合并脚本 fieldnames 硬编码。
- **防坑**：合并脚本先扫所有来源列名·取并集·或用 `extrasaction='ignore'`（不推荐·可能丢数据）·更好是采集时统一列名（在 Agent prompt 中指定）。

#### 坑 14：餐厅数据前后不一致

- **现象**：文档说 X 家·实际 merged_final.csv 只有 Y 行（差异显著）。
- **原因**：补充数据可能没被最终合并脚本纳入·或去重后数量下降。
- **防坑**：每次合并后立即统计并更新文档·合并脚本输出每来源新增/重复计数·**最终数字以 merged_final.csv 实际行数为准**·不以文档为准。

## 这次盘点新发现的坑

### 坑 N1：SCHEMA 枚举不全·新圈数据卡住

- **现象**（关西 entity 池暴露）：SCHEMA §2.1.3 写了 12 类 category·实际数据中出现 aquarium / observation_deck / amusement_park / shopping_street / boulevard / experience / ferris_wheel / unesco_temple 等未列值·validator FAIL。
- **原因**：开圈时 entity 类型多样·SCHEMA 起草时考虑不到·写数据时直接造新值·没回头扩 SCHEMA。
- **防坑**：
  - 数据写入时遇到 SCHEMA 没列的枚举值·**先停下来扩 SCHEMA 再写数据**（CLAUDE.md 第 4 条红线）
  - 开新圈前先扫一遍 SCHEMA 枚举·把目标圈典型类型列全
  - validator FAIL 时**不许自改 validator 放行**（feedback memory）·错的是 SCHEMA 没列全或数据用错

### 坑 N2：D36 砍字段后历史数据残留

- **现象**：SCHEMA §2.1.5 D36 已砍 `name_en` `phone` `unesco` `address_ja` `access` 等字段·但 entity JSON 里还残留。
- **原因**：D36 字段瘦身只改了 SCHEMA 没批量清数据。
- **防坑**：
  - 字段变更四步流程（SCHEMA → 写作指引 → DECISIONS → 批量改数据）的**第 4 步不能跳**
  - validator 的 LEGACY 字段集容许（不报错）但**新写数据不要再加**
  - 下个清理窗口时一次性 grep 删除

### 坑 N3：必填字段 opening_hours 大量空缺

- **现象**：很多 entity 缺 `opening_hours`（街道 / 河岸 / 商业街 / 公共步道型）。
- **原因**：街道型景点没有真实"营业时间"·写作时跳过·SCHEMA 没说"全天开放写什么"。
- **防坑**：
  - SCHEMA §2.1.1 应明确"24h 通行型"约定（如 `{regular: "24h", closed: "无"}`）
  - validator 要求必填·不能因"实际写不出"而放行·而是 SCHEMA 给约定值

### 坑 N4：文档元数据当事实写

- **现象**（CLAUDE.md 第一红线反例）：文档里写「博主站 coverage = 关西」「WebFetch 可用」「N3 数据源更新节奏每周」·没真验证。
- **原因**：写文档时图省事·上下文推断当事实。
- **防坑**：写文档前问"我这条是真搜过的还是从上下文推断"·推断的去搜或标"未核实"·**绝不把推断写成事实**·**不只数据 JSON 字段适用·文档元数据同等严格**。

### 坑 N5：validator 字段白名单和 SCHEMA 双源真相

- **现象**：写 validator 时发现 SCHEMA §2.2 餐厅字段表写的（17 字段）和实际数据（14 字段）出入大·SCHEMA 列了 `meal_type_fit` `facility_tags` `risk_flags` `last_verified` 实际数据没有·实际数据有 `editor_note` `vibe_tags` `price_lunch_jpy` `price_dinner_jpy` `meal_role` `ab_role` SCHEMA 没列。
- **原因**：D36 重构时部分字段名改了（`cuisine`→`cuisine_tag`）·但 SCHEMA 没全部对齐·或 SCHEMA 是"理想态"实际是"现实态"。
- **防坑**：
  - SCHEMA = 唯一权威源（CLAUDE.md 第 4 红线）·实际偏离时优先改 SCHEMA 对齐现实·或改数据对齐 SCHEMA·**不能两边并存**
  - validator 写完先跑一遍真实数据·暴露双源不一致

## 改进后的理想流程

```
Phase 0 城市级资产（一次性）
  taxonomy.json / corridor_definitions.json / cuisine_mapping.json
        ↓
Phase 1 发现（Agent 批量·每 Agent 有超时+指定输出+列名规范）
  每 Agent 完成立即合并 + go/no-go 检查
        ↓
Phase 1→2 归一化（N1-N5·纯 Python·必做不可跳过）
  检查点：cuisine < 40 / price_level 分布合理 / corridor 非空 > 80%
        ↓
Phase 2A Python 预处理（base_quality 组内 percentile + slot 分组）
        ↓
Phase 2B Sonnet 批量结构化短证据（仅 selected + borderline）
        ↓
Phase 2C Opus 审关键条目（S/A + 边界 + risk + slot 冲突）
        ↓
Phase 2D selection_ledger.json
        ↓
Phase 2E 各区域 md 写作层
        ↓
Phase 3 结构化 + 联调测试 + 端到端跑通
```
