# 城市圈构建流程总图

> 通用流程，地区差异在 `地区/{日本,欧洲,中国}.md`。

---

## 第一性原理

**用户拿到行程能照着走** → 倒推数据需求：
- 用户去的是**景点（entity）**，不是餐厅/酒店
- 餐厅/酒店/动线模板都是**配套**

所以：**entity 是基础设施**，餐厅/酒店/模板是并行三件套，全都挂在 entity 上。

---

## 4 阶段

### 阶段 0：定城市圈范围（人决定）

不是 AI 决定。产品判断：
- 这个圈包含哪些城市/区域
- 主场城市 vs 日归动线 vs 温泉镇 vs 单日景点

输出：`{circle}/范围.md`（一段话即可，列城市清单+定位）

参考关西：京都/大阪 主场 + 神户姬路 顺路 + 奈良/有马/城崎/高野山/吉野/熊野/天桥立 延伸

### 阶段 1：建 entity 库（基础设施）

**先做**，餐厅/酒店/模板都依赖。

详见 [02_entity构建.md](02_entity构建.md)。

判断标准（铁律）：
- 用户真会专程去（不是路过）
- 是动线锚点 / 文化地标 / 自然地标
- **不为挂店凑 entity**

### 阶段 2：三件套并行

**互不依赖**，可以三个 sonnet 窗口同时跑：

| 件 | 依赖 | SOP |
|---|---|---|
| restaurants/ + stops/ | entity 锚定 | [03_餐厅构建.md](03_餐厅构建.md) |
| hotels/ | entity 集群 | [04_酒店构建.md](04_酒店构建.md)（待写） |
| templates/ | entity 串动线 | [05_模板构建.md](05_模板构建.md)（待写） |

三件套都通过 `near_attractions.entity_id` / `area` / 动线 entity 列表挂在 entity 库上。

### 阶段 3：联调 + 回归测试

详见 [06_联调测试.md](06_联调测试.md)（待写）。

跑 `coverage_test.py`：
- 每个 template slot 能挑出餐厅/酒店吗
- 哪些 entity 孤立（没餐厅/酒店/动线挂载）
- 真孤立（用户不去）还是数据漏？人审决定

跑通才算城市圈完成。

## 防 AI 跑偏的 3 层防线

| 防线 | 工具 | 救什么 |
|---|---|---|
| 入库时 | schema 校验脚本（白名单字段，FAIL 即拦截） | 字段错 / 字段越界 / 假数据 |
| 入库后 | coverage 回归测试 | 装配挑不出 / slot 不达标 |
| 累计后 | quality audit | 同菜系扎堆 / 跨日重复 |

每层都有自动脚本，**坏数据进不来，进来了被自动报告**。

## 派工模式（多窗口）

### 第一阶段（串行）

**单窗口跑阶段 0 + 阶段 1**——entity 是基础，必须先稳。

### 第二阶段（并行 3 窗口）

| 窗口 | 任务 |
|---|---|
| AI-餐厅 | restaurants/ + stops/（按 entity 周边搜店） |
| AI-酒店 | hotels/（按 entity 集群搜酒店） |
| AI-模板 | templates/（按 entity 串动线） |

每窗口独立读：
```
读：
- docs/04_操作SOP/城市圈构建/0X_{自己 SOP}.md
- docs/04_操作SOP/城市圈构建/地区/{日本/欧洲/中国}.md
- {circle}/entities/（已建）
+ 本轮任务
```

### 第三阶段（单窗口）

联调测试，看孤立 entity + 缺口报告。
