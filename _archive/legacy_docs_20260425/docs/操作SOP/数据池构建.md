# 数据池构建 SOP

> 跨地区通用 SOP。从零搭餐厅 / stops / entity / 酒店 4 类数据池的执行流程。
> 配套规范：[japan/docs/餐厅规范.md](../../japan/docs/餐厅规范.md) / [stops规范.md](../../japan/docs/stops规范.md) / [entity规范.md](../../japan/docs/entity规范.md) / [酒店规范.md](../../japan/docs/酒店规范.md)
> 配套方法论：[数据采集.md](数据采集.md)（三轴/三层池/小众入池/score）+ [研究方法.md](研究方法.md)（MTE 七维度）
> 配套踩坑：见本文 §六

## 总图：从空圈到数据池就位

```
Phase 0 城市圈准备（一次性）
  ├─ 范围定义（哪些城市/区域归这个圈）
  ├─ area_registry.json 白名单
  ├─ entities/ 基础设施先建（必须先做·餐厅/酒店都要挂）
  └─ SCHEMA 枚举值审一遍（缺的先扩 SCHEMA）
        ↓
Phase 1 发现池（尽量不漏）
  ├─ 多语搜索（简中/繁中/日文/英文）
  ├─ P0 + P1 + P2 全部源覆盖
  └─ 数量：终选 5-10 倍
        ↓
Phase 1.5 归一化（必做·N1-N5）
  ├─ N1 cuisine/type 标准化（用 mapping.json）
  ├─ N2 area → 动线区映射
  ├─ N3 budget_tier 推断（按价格不按评分·铁律）
  ├─ N4 价格层修正（不用默认值覆盖）
  └─ N5 hotel_type / category 补全
        ↓
Phase 2 入围池（交叉验证）
  ├─ 三轴模型（两轴覆盖即入）
  ├─ 数量：终选 2-3 倍
  └─ 核验语言收窄（日文+官方+简中）
        ↓
Phase 3 终选池（编辑精选）
  ├─ 判断："朋友第一次去会专门推荐这个吗？"
  ├─ 品类内相对排名 top 10-20%
  ├─ 同决策位 ≤ 3 家
  └─ 允许稀疏（没合格的就空着）
        ↓
Phase 4 字段填充 + 校验
  ├─ 按规范字段定稿写 JSON
  ├─ 中文硬规 + 价格人民币换算
  ├─ 跑 validator 强制校验
  └─ 完整输出粘交付
```

## 一、Phase 0 城市圈准备（一次性）

### 1.1 范围定义

**人决定**（不是 AI）：
- 这个圈包含哪些城市/区域
- 主场 / 日归 / 温泉镇 / 单日景点 4 类区域归类
- 边界（这个圈到哪结束·跨圈日归衔接）

输出 `{圈}/这是什么.md`（一段话即可，列城市清单 + 定位）。

参考关西样板：京都+大阪 主场 + 神户姬路 顺路 + 奈良/有马/城崎/高野山/吉野/熊野/天桥立 延伸。

### 1.2 area_registry.json 白名单

跨 4 类数据共用——餐厅/stops/entity/酒店的 area 字段都来这里查。

```json
{
  "kansai": [
    {"area": "东山轴", "type": "主场密集", "city": "京都"},
    {"area": "心斋桥-美国村", "type": "主场密集", "city": "大阪"},
    {"area": "有马温泉", "type": "温泉宿坊", "city": "神户"},
    ...
  ]
}
```

字段：`area`（动线粒度名）/ `type`（4 类）/ `city`。

**铁律 3**（4 份规范都有）：area 必须从 registry 选，自由命名 = FAIL。

### 1.3 entities/ 基础设施先建

**先做·餐厅/stops/酒店都要挂 entity**。

按 [entity规范.md §四 数据准入](../../japan/docs/entity规范.md) 4 套递进判断（价值 → 证据 → 角色）建：
- 用户必去的 S 级（清水寺/金阁寺/通天阁等）→ 30-40 分钟/个
- 动线锚点 A 级（哲学之道/锦市场/心斋桥筋）→ 15-20 分钟/个
- 顺路 B 级（小神社/老铺）→ 10 分钟/个

**铁律 1**：entity 不为挂店服务·店挂不上就降 skeleton 不补 entity。

### 1.4 SCHEMA 枚举审一遍

开新圈前先扫 [docs/SCHEMA.md](../SCHEMA.md)：
- entity category 枚举够不够（关西曾漏 13 项：aquarium/observation_deck/ferris_wheel 等）
- 餐厅 cuisine 枚举（关西 75+ 种归一化前一片乱）
- 酒店 type 枚举

**缺的先扩 SCHEMA·再写数据**（CLAUDE.md 第 4 红线）。

## 二、Phase 1 发现池（尽量不漏）

### 2.1 多语搜索（仅新圈首次采集开五语）

成熟圈用日文主源 + 中文 traveler 源即可·不开五语。

| 语言 | 关键词模板 |
|---|---|
| 日文 | `"{城市} {菜系} 人気"` / `"{城市} ホテル ランキング"` / `"{店名} 食べログ"` |
| 简中 | `"{城市}{菜系}推荐"` / `"{店名}避雷"` / `"{城市}小众"` |
| 繁中 | `"{城市} {菜系} 必吃"` |
| 英文 | `"{city} hidden gems locals"` / `"{city} where to stay"` |

### 2.2 数据源覆盖（4 类各自）

按规范 §四数据源段：

| 4 类 | P0 主源 | P1 辅助 | P2 独立站 |
|---|---|---|---|
| **餐厅** | Tabelog / 米其林 / 百名店 / Google Maps | 携程 / 小红书 / 大众点评 / Retty / SAVOR JAPAN | 60+ 站（凯的日本食尚日记 / TISS / Inside Kyoto 等） |
| **stops** | Google Maps / 官网 | 小红书（collects > likes） / 携程 | thisismedia（古着）/ icotto / Hanako / cottrip / Recoya |
| **entity** | JNTO 官方观光 / 京都市公式 / Japan-guide / MATCHA | 小红书 / 携程 / 知乎长文 / 马蜂窝 | Inside Kyoto / Inside Osaka / 季节专题 |
| **酒店** | 一休 / 楽天 / じゃらん / Relux / 携程 / 米其林 Keys / Forbes | 小红书 / 大众点评 / 马蜂窝 | 台湾博主实住（Mimi 韩 / 波比看世界 / 阿波旅行中 / Adventurous Mark 等） |

### 2.3 发现池数量

**终选的 5-10 倍**。例：终选 100 家餐厅 → 发现池 500-1000 家。

### 2.4 停机条件（任一即停）

- 某组连续新增 10 个候选·无 1 个能进入入围池
- 某组已有 3 个高质量终选 + 2 个备选
- 新增来源只是重复提及已有候选
- 成熟类目（如京都怀石）已从主源获得充足候选

## 三、Phase 1.5 归一化（必做·跳过即坑）

> **关西教训**：跳过归一化直接进 Phase 2 → cuisine 75+ 种混乱·price_level 85% 被默认值覆盖·跑不出选择模型。
> 归一化是**必经步骤不是可选**。

### N1 cuisine / type 标准化

多源各用不同语言粒度（日文「ラーメン」/英文「ramen」/中文「拉面」），合并后类型膨胀到 70+。

**对策**：
- 维护 `cuisine_mapping.json`（圈级一次性资产）
- 标准枚举见 SCHEMA
- **采集时**就映射不留到后期
- 归一化后 cuisine 唯一值 < 40

### N2 area → 动线区映射

各源 area 字段粒度不一（有的"心斋桥"有的"难波"有的"道顿堀"），统一到 area_registry。

**对策**：用 `corridor_definitions.json` 做文本匹配 + 关键词映射。

### N3 budget_tier 推断（按价格不按评分·铁律）

> **N3 铁律**：Tabelog 3.8 拉面店可能人均 1000 円（mid）·3.5 的怀石可能 15000 円（premium）。**评分反映口碑不反映价格层**。

推断优先级：
1. 已有 JPY 价格 → 直接按阈值分层
2. 米其林非空 → luxury / showcase
3. 菜系场景弱推断（怀石/割烹/铁板烧/河豚 → premium；拉面/乌冬/丼 → mid/budget）
4. 无法判断 → 保留原值·标 `tier_confidence: inferred`

**绝不用评分推价格**。

### N4 酒店 price_level 修正

```
JPY ≥ 30000/晚 → luxury（≈ ¥1500+/晚 RMB）
JPY 15000-30000 → expensive（≈ ¥800-1500）
JPY 8000-15000 → moderate（≈ ¥400-800）
JPY < 8000 → budget（≈ ¥400 以下）
```

**铁律**：缺数据时标 null·**不用默认值覆盖**（关西曾 85% 酒店统一标 moderate）。

### N5 hotel_type / category 补全

从 `name_ja` 推断：
- 含「旅館」「ホテル」「リゾート」「ゲストハウス」→ 对应 type
- entity 同理（含「神社」「寺」「城」→ 对应 category）

## 四、Phase 2 入围池（三轴交叉验证）

### 4.1 三轴模型

每条候选必须**三轴中至少两轴有真实证据**：

| 轴 | 回答什么 | 信号源 |
|---|---|---|
| **quality** | 该类型够不够强 | 餐厅：Tabelog/米其林；entity：官方观光局/Japan-guide；酒店：Keys/Forbes/一休 |
| **traveler_fit** | 中国游客实际满意度 | 携程评分+评论 / 小红书 collects > likes |
| **execution** | 能不能去 | 营业 / 预约 / 排队 / 价格真实 |

可信度 4 档：
- 三轴全过 → `verified`
- 两轴过 → `cross_checked`
- 一轴过 → `single_source`（标注后入）
- 无证据 → `ai_generated`（**禁止上生产**）

### 4.2 入围池数量

**终选 2-3 倍**。例：终选 100 家 → 入围 200-300 家。

### 4.3 多源重叠率别死磕 40%

**关西教训**：6 个来源中只有 14% 餐厅出现在 2+ 个来源中。
不同来源覆盖不同品类（高端/游客热门/本地），交叉少是正常的。

**对策**：多源置信度目标定 20-25% 更现实·indie_support_score 价值在补充视角不在重叠。

## 五、Phase 3 终选池（编辑精选）

### 5.1 核心判断

> **"如果朋友第一次去，我会专门推荐这个吗？"**

不是"这个也不错"·是"这个比那个更值"才入。

### 5.2 数量规则

- 品类内相对排名：样本 ≥ 15 取 top 10-20%
- 小样本用编辑判断
- 同一决策位最多 3 家（首推 / 稳妥替代 / 机动备选）

### 5.3 允许稀疏（产品哲学）

> **宁可空着不给·也不给"还行"的**——保护"专家挑的"信任感。

某区域某档没合格 → 空着·不补"还行的"。

### 5.4 负向信号过滤（每类规范 §四.4 详）

| 信号 | 处理 |
|---|---|
| 公认游客陷阱 | 不收 |
| 连锁化无特色 | 不收 |
| 排队 >90 分钟同区有替代 | 降 skeleton |
| 小红书爆火但 Tabelog/Google <3.3 | 降 skeleton |
| 评价两极 | 降 skeleton |
| 价格与体验不匹配 | 降 skeleton |
| 装修中/搬迁/闭店 | 立即降 skeleton 标注 |

### 5.5 Tier 分层投入精力

| Tier | 餐厅/景点定位 | 投入精力 | 完成度 |
|---|---|---|---|
| **S 级** | 不去会遗憾 | 30-40 分钟/个 | depth=full + templates_meta 全填 |
| **A 级** | 重要搭配·值得专程 | 15-20 分钟/个 | depth=full + 至少 3 key |
| **B 级** | 顺路可以 | 10 分钟/个 | depth=verified |
| **C 级** | 时间富余考虑 | 5 分钟/个 | depth=skeleton |

## 六、Phase 4 字段填充 + 校验

### 6.1 中文硬规

**所有用户可见字段中文为主**·日文/英文括号补充：

✅ 正确：
- 店名："瓢亭 本店（ひょうてい / Hyotei）"
- 营业："平日 11:00-14:00 / 17:00-21:00；木曜定休"
- 简介："江户时代传下的怀石老铺·南禅寺寺院深处..."

❌ 错误：
- 营业："平日 11:00-14:00 / 17:00-21:00、木曜定休"（日文标点）
- 留下纯假名词不翻译："コース 11,000 円"（应译"套餐 ¥11,000"）

### 6.2 价格人民币换算

**汇率**：100 日元 ≈ 4.30 RMB（2026/4 月，央行中间价）

**精度**：取整到 50 或 100 RMB，**禁精确到个位/十位**：
- ❌ `[2327, 2327]`（¥49,500 精确换算）
- ✅ `[2000, 2500]`（合理浮动区间）

**禁 min=max**：套餐定价单一也写区间反映含税/酒水浮动。

### 6.3 4 档价格区间（餐厅·单顿统一标准）

| tier | 单顿 RMB | 典型场景 |
|---|---|---|
| economy | 0-180 | 拉面/章鱼烧/牛丼 |
| mid | 150-500 | 居酒屋/定食/必比登 |
| high | 450-1200 | 和牛/普通怀石/米其林一星午餐 |
| showcase | 1100-5000+ | 米其林一星以上/三星寿司 |

午晚通用一套尺子·一家店午 mid + 晚 high 是常态（按实际单顿价填）。

### 6.4 4 档价格区间（酒店·单晚统一标准）

| tier | 单晚 RMB | 装配 |
|---|---|---|
| 三星 | < 500 | 主力（经济档） |
| 四星 | 500-1500 | 主力（舒适·主流） |
| 五星 | 1500-4000 | 可选（高端） |
| 奢华 | ≥ 4000 | 默认 0·仅定制 |

### 6.5 跑 validator 校验

每类对应一个校验脚本：
- `scripts/validate_entity.py`
- `scripts/validate_restaurants.py`（同时校验 stops）
- `scripts/validate_hotels.py`

校验规则：
- 字段白名单 + 必填 + 枚举 → 任一违反 FAIL
- area 必须在 area_registry.json
- near_attractions.entity_id 必须在 entities/
- 价格区间合理性

**跑法**：`python scripts/validate_*.py`，**完整输出粘交付消息**（feedback memory 硬规）。

### 6.6 交付物

```
{圈}/restaurants/{city}/{动线区}.json
{圈}/stops/{city}/{动线区}.json
{圈}/entities/{city}.json
{圈}/hotels/{city}/{大区}.json
+ validator 完整输出
+ 卡点报告（A/B 类卡点·小事不报）
```

## 七、踩坑速查（开新圈前必读·14 条勾选清单）

> 抽自 [关西 ops_lessons.md](../../japan/kansai/research/ops_lessons.md) + 历史教训。

逐条勾选才能开始 Phase 1：

- [ ] **1. SCHEMA 枚举值已审过一遍**（开新圈常发现 category / hotel_type / cuisine 漏值；漏值先扩 SCHEMA 再写数据）
- [ ] **2. cuisine_mapping.json 模板已准备**（预填常见日文/中文 → 标准码映射）
- [ ] **3. area_registry.json 白名单已建**（不在 registry 的 area = FAIL）
- [ ] **4. 标准菜系码枚举已更新**（新菜系先加到 SCHEMA）
- [ ] **5. 合并脚本 schema 含所有强信号字段**（不丢 OTA 评分/原始价格/Keys/百名店标）
- [ ] **6. Agent prompt 指定输出文件完整路径和列名规范**（默认 CWD 会落错位置）
- [ ] **7. Agent 设 2 小时超时 + fallback plan**（某 Agent 卡住不阻塞其他）
- [ ] **8. 归一化 N1-N5 写入执行清单·不可跳过**
- [ ] **9. budget_tier 推断规则不依赖评分**（价格层 ≠ 质量层）
- [ ] **10. base_quality_score 用组内 percentile**（不做全局 min-max）
- [ ] **11. 酒店 price_level 不用默认值覆盖**（缺数据标 null）
- [ ] **12. Python 脚本不用 emoji**（Windows GBK 兼容；用 `[ok]` `[warn]` 替代）
- [ ] **13. validator 三件套就位**（validate_entity / validate_restaurants / validate_hotels）
- [ ] **14. 文档元数据已真验证**（CLAUDE.md 第一红线：文档事实和数据事实同等严格）

## 八、维护节奏

- **每开一个新圈跑通后**：把新发现的坑加到本文 §七 + 对应规范 §六
- **关西基线数字**当参考·下一圈跑出新值放新表对比
- **每季度扫一次终选池**：是否有新开店符合准入·是否有负向信号出现
