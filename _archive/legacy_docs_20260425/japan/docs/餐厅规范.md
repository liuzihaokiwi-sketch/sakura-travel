# 日本餐厅规范

> 日本各圈通用规范。餐厅 = 用户必去的正餐池（每天必吃饭），走 tier×cuisine×档位匹配。
> 跟停留点（stops·歇脚）/ entity（景点）/ 酒店（住宿）三类规范并列。
> 关西特定的区域基因 / 餐位需求分级 → `japan/kansai/这是什么.md`
> 停留池（咖啡/购物/手作）→ [stops规范.md](stops规范.md)

## 一、7 条铁律（违反任何一条 = 全部不通过）

### 1. 字段不准膨胀

字段定稿见 §二。**不准私自加任何字段**：
- `walk_min_estimated` ❌（已禁用）
- `drive_only` / `tags` / `vibe` 等 sonnet 自创 ❌
- 发现需要新字段 → **停下问用户**，不要先加再说

校验脚本对字段做严格白名单，未知字段 = FAIL。

### 2. 数据不准造假

**查不到 ≠ 写假的**。三种处理：
- 模糊化（"小店"替"18 席"）
- 标"待核实"或写"营业时段以官网为准"
- 字段留空或 null

**禁止行为**：
- 全部餐厅写"通常 11:00-21:00"凑营业 ❌
- 米其林星级年份猜 ❌
- 步行分钟全标 estimated ❌
- 跨店比较句"全京都最好的怀石" ❌

### 3. area 必须从白名单选

`area` 字段是全局事实（跨餐厅/酒店/模板/装配），**禁止自由命名**。

写 area 前必 grep `{圈}/area_registry.json`：
- 在白名单里 → 用 registry 标准名
- 不在 → **停下报告**，由用户决定加 registry 还是改用现有 area

校验脚本对 area 做严格白名单，不在 registry = FAIL。

### 4. entity 不为挂店服务

entity 必须**自己有意义**才加：
- 用户必去 / 模板动线锚点 / 文化地标
- 不是为了"店挂不上"而凑

**店挂不上就挂最近合理 entity，walk_min 大一点没关系**。挂不上任何合理 entity 的店 → 降 skeleton 或不进 full，**不为它加新 entity**。

### 5. walk_min 弹性

- **正餐**：用户为吃顿好的愿意走，15-20 分都接受
- **不为了"挂得近"伪造小数字**

### 6. depth 必须真实

- 地址查不到 / 营业全靠类目均值 = 自动降 skeleton
- full 必须三段式完整（系统字段 + note 全填 + 准 entity 挂载）
- 不强凑 full 撑数量

### 7. 主次分明（避免反复纠结）

- **A 类**（装配跑不动 / 字段缺失） → 停下报告
- **B 类**（用户体验硬伤：地址错 / 档位错 / 价格离谱） → 停下报告
- **小事**（Tabelog 缺 / 营业不全 / 文笔粗糙） → 标 verified 或留空，**继续做不停**

## 二、字段定稿

```json
{
  "id": "kyo_gion_hyotei",
  "area": "北区",
  "near_attractions": [{"entity_id": "nanzenji_temple", "walk_min": 3}],
  "tier": "showcase",
  "cuisine": ["怀石"],
  "recommended_meals": [{"meal": "lunch", "price_cny": [800, 1500]}],
  "reservation_difficulty": "must",
  "queue_level": "none",
  "closed_days": [3],
  "season_months": null,
  "depth": "full",
  "note": {
    "店名": "瓢亭 本店（ひょうてい / Hyotei）",
    "简介": "...",
    "亮点": ["米其林三星", "庭院茶室", "朝粥名物"],
    "招牌菜": ["朝粥（限早场 8:00-10:00）", "松花堂弁当（午）"],
    "地址": "京都市左京区南禅寺草川町 35（地铁东西线·蹴上站步行 8 分钟）",
    "营业": "11:00-14:00 / 17:00-21:00；周三定休",
    "预约": "电话预约（075-...），3 个月前 1 日 10:00 开放受理",
    "到店提醒": "建议 14:00 后到避高峰；红叶季全天无空位务必预约",
    "Tabelog 分数": 4.3
  }
}
```

### 系统字段（11 个）

| 字段 | 类型 | 必填 | 枚举值 / 格式 |
|---|---|---|---|
| `id` | string | ✅ | `{城市拼音简写}_{区域简写}_{店名拼音}` |
| `area` | string | ✅ | 动线级粒度（与 area_registry 对齐） |
| `near_attractions` | array | ✅（≥1） | `[{entity_id, walk_min}]`。entity 必须真实存在 |
| `tier` | string | ✅ | `showcase` / `high` / `mid` / `economy` |
| `cuisine` | array | ✅（≥1） | 字符串数组，主菜系第一。多菜系老铺如 `["稻荷寿司", "鳗鱼", "乌冬"]` |
| `recommended_meals` | array | ✅（≥1） | `[{meal, price_cny}]`。meal: `breakfast/lunch/dinner/cafe/bar`。**4 档单顿统一标准·午晚通用**：economy [0, 180] / mid [150, 500] / high [450, 1200] / showcase [1100, 无上限]。精度取整到 50 或 100 |
| `reservation_difficulty` | string | 选填，缺省 `none` | `none` / `recommended`（不预约要排队≥1h）/ `must`（不预约进不去） |
| `queue_level` | string | 选填，缺省 `mild` | `none` / `mild` / `high` |
| `closed_days` | array \| null | ✅ | ISO 8601 数字数组：1=周一... 7=周日。三态：`[]`=已查实无定休 / `[3]`=周三定休 / `null`=未查到 |
| `season_months` | array/null | 选填 | `null` = 常年；`[5,6,7,8,9]` = 季节限定 |
| `depth` | string | 选填，缺省 `skeleton` | `skeleton` / `verified` / `full` |

### note 字段（9 个）

| key | 必填 | 说明 |
|---|---|---|
| `店名` | ✅ | 中文（日文假名 / 英文）。如 `瓢亭 本店（ひょうてい / Hyotei）` |
| `简介` | full 必填 | 80-150 字，"懂当地的朋友讲口吻" |
| `亮点` | full 必填 | 3-5 个短词标签 |
| `招牌菜` | full 必填 | 列表，含限定提示如"朝粥（限早场）" |
| `地址` | full 必填 | 含最近交通+步行分钟 |
| `营业` | full 必填 | 自然语言一句：开门时段+定休 |
| `预约` | full 必填 | 渠道+开放窗口。`reservation_difficulty=none` 写"无需预约" |
| `到店提醒` | 选填 | 错峰 / 旺季 / 不规则定休 / 外国客提示 |
| `Tabelog 分数` | 选填 | 数字，查不到不写 |

## 三、分类标准

### 3.1 跟停留池的边界

判断进 restaurants 还是 stops：
- **能解决一顿正餐** → restaurants（即使便宜也算）
- **纯咖啡甜品/购物/手作** → stops

边界 case：
- 伏見夢百衆（咖啡+品酒+酒馒头·无正餐）→ stops
- 鳥せい本店（酒藏居酒屋·能正餐）→ restaurants
- Smart Coffee 2F 洋食午市（能正餐）→ restaurants

### 3.2 tier 4 档（用户表单对齐）

| tier | 中文 | 单顿人民币 | 典型菜系/场景 |
|---|---|---|---|
| `economy` | 街头地道 | 0-180 | 拉面/章鱼烧/牛丼/咖啡店简餐 |
| `mid` | 本地好店 | 150-500 | 居酒屋/定食/中端洋食/必比登 |
| `high` | 好好享受 | 450-1200 | 和牛专门店/普通怀石/米其林一星午餐 |
| `showcase` | 顶级体验 | 1100-5000+ | 米其林一星以上/顶级怀石/三星寿司 |

**4 档区间是单顿统一标准·午晚通用**——不是"午餐档"和"晚餐档"两套尺子。一家店午餐落 mid、晚餐落 high 是常态（如和牛/怀石午定食划算），按实际单顿价填即可。

### 3.3 cuisine 多菜系处理

老铺常多菜系（稻荷寿司+鳗鱼+乌冬）。`cuisine` 是数组，**主菜系放第一**：

```json
"cuisine": ["稻荷寿司", "鳗鱼", "乌冬"]
```

装配按 `cuisine.includes("乌冬")` 跨菜系命中。

## 四、数据准入

> **总原则**：准入靠**三轴模型 + 三层池**，不是 Tabelog 单门槛。详见 [docs/操作SOP/数据采集.md §三 §五](../../docs/操作SOP/数据采集.md)。

### 4.1 三轴模型（quality / traveler_fit / execution）

每个 Full 必须**三轴中至少两轴有真实证据**：

| 轴 | 信号源 | 通过标准 |
|---|---|---|
| **quality** | Tabelog / 米其林 / 百名店 | 见下面菜系 Tabelog 门槛（quality 轴硬指标） |
| **traveler_fit** | 携程评分+评论 / 小红书 collects > likes / 大众点评 / 60+ 独立站（§4.3 P2） | 中国游客实际去了不会后悔（不是单纯"好吃"） |
| **execution** | 营业稳定 / 预约难度可承受 / 排队 ≤90 分钟 / 定休清晰 | 用户照行程能去到 |

**三轴规则**：
- **三轴全过** → `verified` 入 full
- **两轴过**（任意组合） → `cross_checked` 入 full
- **一轴过** → `single_source`，标注后入 full
- **无任何真实证据** → `ai_generated`，禁止上生产

**Tabelog 门槛**（quality 轴硬指标，**但不是唯一通道**）：

| 菜系 | Tabelog 门槛 |
|---|---|
| 怀石 / 割烹 | ≥3.7 |
| 寿司 / 天妇罗 / 烧肉 | ≥3.6 |
| 拉面 / 乌冬 / 居酒屋 | ≥3.5 |
| 咖啡 / 甜品 / 地方小吃 | 看知名度不看分（3.3 的老铺 > 3.6 的网红） |

**Tabelog 不达门槛的店**：traveler_fit + execution 两轴覆盖足以入 full（如本地居酒屋 Tabelog 3.4 但小红书 collects 100+ + 营业稳定 + 不预约可吃 = `cross_checked` 入）。**禁止仅因为 Tabelog 卡线就剔除合格店**。

### 4.2 景点定位（near_attractions）

每家 Full 必须挂 1-2 个 entity。挂不上 → 降 skeleton 或不收。

### 4.3 数据源 P0/P1/P2

#### P0 权威源

| 源 | 域名 | 角色 |
|---|---|---|
| Tabelog | tabelog.com | 质量主骨架（菜系×价格带相对排名） |

### 4.4 负向信号（任一触发即降 skeleton 或不收）

| 信号 | 处理 |
|---|---|
| 公认游客陷阱（本地人避开） | 不收 |
| 连锁化无特色 | 不收 |
| 排队 >90 分钟且同区有替代 | 降 skeleton |
| 小红书爆火但 Tabelog <3.3 | 降 skeleton |
| 体验波动大（评价两极） | 降 skeleton |
| 价格与体验不匹配 | 降 skeleton |
| 关店/搬迁/装修中 | 立即降 skeleton 标注 |

### 4.5 使用优先级（实操）

1. 现有池（`assembly/restaurants/data/restaurants__*.json` 1030+ 家）— 首选
2. P0 源交叉核实（Tabelog + 米其林 / 百名店）
3. P1 小红书 opencli 补本地视角（collects > likes 2-3 条）
4. P2 独立站补小吃/体验型 / 区域专题

**不自己编**。

### 4.6 搜索词模板

| 语言 | 词 |
|---|---|

| 简中 | `"{城市}{菜系}推荐"` / `"{店名}避雷"` |
| 繁中 | `"{城市} {菜系} 必吃"` |

## 五、校验

校验脚本：`scripts/validate_restaurants.py`（同时校验 restaurants + stops）

校验规则：
- 系统字段白名单（11 个）+ note 字段白名单（9 个）→ 未知字段 FAIL
- area 字段必须在 `{圈}/area_registry.json` → 不在 FAIL
- near_attractions.entity_id 必须在 `{圈}/entities/*.json` → 不在 FAIL
- 必填字段缺 → FAIL
- depth=full 但 note 字段缺 → FAIL（自动建议降 skeleton）
- recommended_meals.price_cny 落在 4 档区间外 → FAIL（如 economy 标 [500, 800]）

跑法：`python scripts/validate_restaurants.py`，粘**完整输出**到交付消息。

## 六、历史教训

### 6.1 area 命名漂移（已解决）

D43 前 area 自由字符串，导致同一动线区有"三宮"/"旧居留地"/"神户三宮-旧居留地"三个桶散在多处文件。
**对策**：area_registry.json 白名单 + validator 强制校验。

### 6.2 Tabelog 单门槛漂移（已解决）

之前 sonnet 仅按 Tabelog 分卡线，导致"东山轴 mid 居酒屋池里没合格"——其实有 traveler_fit 强的本地居酒屋被 Tabelog 3.4 卡掉。
**对策**：准入改三轴模型，Tabelog 是 quality 轴硬指标但不是唯一通道。

### 6.3 价格区间漂移（已解决）

之前 sonnet 把日元换算成精确人民币（如 [2327, 2327]），档位无意义。
**对策**：4 档单顿统一区间·精度取整到 50 或 100·禁 min=max。

### 6.4 walk_min_estimated 字段膨胀（已禁用）

之前为标"步行分钟估算"加了 `walk_min_estimated: bool`，结果 sonnet 全部标 true 等于没标。
**对策**：字段删除，walk_min 一律按真实查得填。

### 6.5 营业类目均值漂移（已解决）

之前 sonnet 把不同店都写"通常 11:00-21:00"凑营业。
**对策**：营业查不到必须写"营业时段以官网为准"或具体真实时段，禁用类目均值。

### 6.6 entity 挂载粒度太粗（已解决）

之前 13 家全挂 osk_namba 一个站名 entity，装配按景点 filter 时漂移。
**对策**：必须挂具体地标级 entity（如美国村/心斋桥筋/堀江/日本桥），不够 → 停下补 entity 不将就。

### 6.7 tier 错档（注意点）

容易把"高分但价格平价"的店硬挂 high/showcase（如京都站京林泉 ¥150 lunch 标 showcase）。
**对策**：tier 必须真实反映 price 档位，不为"显得高级"挂错档。
