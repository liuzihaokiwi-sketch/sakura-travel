# 日本酒店规范

> 日本各圈通用规范。酒店 = 用户每天必住，决定动线半径（"sleep first, play second"）。
> 跟餐厅 / stops / entity 三类规范并列。
> 关西特定的酒店区域分类 → `japan/kansai/这是什么.md`
> 装配硬规（连住天数/温泉日时机）→ `japan/{圈}/assembly/hotels/装配规则.md`

---

## 一、6 条铁律（违反任何一条 = 全部不通过）

### 1. 字段不准膨胀

字段定稿见 §二。**不准私自加任何字段**。

D40 时期酒店字段超 25 个（含 ranking_info/breakfast/opened_year/renovated_year/room_count/rating_subscores/review_keywords/nearest_station 等），导致打标失败被评审推翻。

**对策**：精简到核心字段·细节内容塞 note 自由文本。

发现需要新字段 → 停下问用户。

### 2. 数据不准造假

- 价格必须 OTA 真实抓取（携程/一休/楽天），不编不推断
- 含早 / 房型细节 → 没查到写"以官网为准"，禁用类目均值
- 不写跨酒店比较句（"京都最豪华"）

### 3. area 必须从白名单选

跟餐厅/stops/entity 共用同一 `area_registry.json`。酒店 area 是**该酒店所在动线区**（粒度比餐厅可粗·城市级+大区即可）。

历史教训：D40 酒店 area 标错（神户六甲温泉酒店标"市中心"），导致装配把温泉日酒店放成市区便利型。
**对策**：area_registry 严格白名单 + validator 校验。

### 4. tier × type 双标签

- `tier`（4 档）：三星 / 四星 / 五星 / 奢华——**决定价位**
- `type`（2 类）：city（普通便利） / experience（含食宿温泉打包体验）——**决定装配角色**

不可缺一。tier 三星 + type=experience（如有马平价旅馆含 2 食 ¥1500/晚）是合理组合。

### 5. depth 必须真实

- 价格未抓 OTA 真实数据 → 降 skeleton
- 含早/房型/预约渠道未查 → 降 skeleton
- full 必须三段式完整（系统字段 + note 全填 + 准 entity 挂载）

### 6. 主次分明

- **A 类**（装配跑不动 / 字段缺失） → 停下报告
- **B 类**（用户体验硬伤：地址错 / 档位错 / 价格离谱） → 停下报告
- **小事**（房型不全 / 含早细节 / 评论摘要） → 标 verified 或留空，**继续做不停**

```json
{
  "id": "kyo_kawaramachi_kyoto_hotel_okura",
  "city": "京都",
  "area": "河原町",
  "near_attractions": [{"entity_id": "kyo_pontocho", "walk_min": 3}],
  "tier": "四星",
  "type": "city",
  "price_cny_per_night": [800, 1500],
  "season_months": null,
  "depth": "full",
  "note": {
    "店名": "京都大仓酒店（京都ホテルオークラ / Kyoto Hotel Okura）",
    "简介": "1888 年创业老牌四星，河原町御池路口最大体量，鸭川景观房型是签名。低层早餐厅望御所、高层套房俯瞰东山三十六峰。中国客人对接友好，前台日英中可用。",
    "亮点": ["河原町最大格局", "鸭川夜景客房", "1888 年创业"],
    "地址": "京都市中京区河原町御池（地铁东西线京都市役所前站直结）",
    "房型": "标准双人 / 鸭川景观双人 / 行政套房",
    "含早": "可选·¥3,500/人·和洋自助",
    "预约": "官网 / 一休 / 楽天 / 携程·提前 30 天最优惠 15-20%",
    "到店提醒": "Check-in 15:00 / Check-out 12:00·红叶季鸭川房型涨价 30%·樱花季提前 60 天约必须"
  },
  "可信度": "verified",
  "数据来源": ["https://...", "https://..."],
  "最后核实": "2026-04-25"
}
```

### 系统字段（9 个）

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | string | ✅ | `{城市拼音简写}_{区域}_{酒店名}` |
| `city` | string | ✅ | 中文城市名 |
| `area` | string | ✅ | 动线区·跟 area_registry 对齐 |
| `near_attractions` | array | ✅（≥1） | `[{entity_id, walk_min}]`·entity 必须真实存在 |
| `tier` | string | ✅ | `三星` / `四星` / `五星` / `奢华` |
| `type` | string | ✅ | `city`（普通便利型） / `experience`（含食宿温泉打包体验型） |
| `price_cny_per_night` | array | ✅ | `[min, max]` 人民币/晚区间。精度取整到 50 或 100 |
| `season_months` | array/null | 选填 | 季节限定旅馆（如蟹宿冬季）·常年=null |
| `depth` | string | 选填，缺省 `skeleton` | `skeleton` / `verified` / `full` |

### note 字段（8 个）

| key | 必填 | 说明 |
|---|---|---|
| `店名` | ✅ | 中文（日文/英文）。如 `京都大仓酒店（京都ホテルオークラ / Kyoto Hotel Okura）` |
| `简介` | full 必填 | 100-200 字，"懂当地的朋友讲口吻"。讲品牌定位+签名体验+客群匹配 |
| `亮点` | full 必填 | 3-5 个短词标签 |
| `地址` | full 必填 | 含最近交通+步行分钟（最近车站/动线 entity） |
| `房型` | full 必填 | 列主流房型（"标准双人 / 景观双人 / 套房"），不堆所有房型 |
| `含早` | 选填 | "可选·价格"或"含"或"无"。不知道写"以官网为准" |
| `预约` | full 必填 | 主推渠道 + 提前预约建议（"提前 30 天约最划算" / "樱花季提前 60 天必须"） |
| `到店提醒` | 选填 | Check-in / Check-out 时间·旺季加价·特殊政策（外国客政策/不接受外国客等） |

## 三、分类标准

### 3.1 tier 4 档（按单晚价格）

| tier | 单晚 RMB | 装配 | 例子 |
|---|---|---|---|
| `三星` | < 500 | 装配主力（经济档用户） | 东横 INN / Super Hotel / Mimaru 公寓 / 京都站皇家王子 |
| `四星` | 500-1500 | 装配主力（舒适档·主流） | 万豪系 / Cross / Granvia / 都酒店 / Hyatt Place |
| `五星` | 1500-4000 | 装配可选（高端档·用户预算充足时） | Park Hyatt（部分房型）/ 二条城悦柳 / 高级怀石旅馆 |
| `奢华` | ≥ 4000 | **默认 0 次**·仅定制启用 | Aman / Four Seasons / 丽思 / 安缦 / Bvlgari / Park Hyatt 顶级套房 |

**奢华边界 ¥4000+**：调研显示 ¥3000-4000 是中国游客"愿意为体验花"的真实上限·¥4000+ 几乎只剩国际顶奢品牌，进入定制级消费。

### 3.2 type 2 类（按角色）

| type | 判断 | 例子 |
|---|---|---|
| `city` | 住着方便+睡舒服·离开酒店玩才是行程主体 | 万豪系/凯悦/东横 INN/Mimaru/普通町家 |
| `experience` | 住宿费里打包食宿温泉·住一晚 = 核心体验·离开反而损失体验 | 怀石旅馆/温泉宿/宿坊/虹夕诺雅/含 2 食的高级 ryokan |

**判断标准**：**住宿费里有没有打包"体验"**（怀石/泡汤/精进料理/贷切）。打包了 = experience，没打包 = city。

### 3.3 边界 case

- **京都町家**：普通町家（Mimaru 连锁）→ city；高端贷切町家（虹夕诺雅 ¥5000+ 一栋一组）→ experience
- **神户万豪**（贵但纯城市便利型）→ city（高端便利·四星）
- **有马温泉旅馆**（含怀石 + 泡汤 ¥1500-3000/晚）→ experience（即使便宜·因为体验打包是核心）
- **Park Hyatt 京都**（¥4500+ 城市便利型）→ city + 奢华档（贵但没打包食宿体验）
- **一家有 ryokan 房间也有普通房间的复合体**：按主营定，不拆两条 entity

## 四、数据准入

> **核心原则**：跟餐厅/entity 一致——三轴模型 + 三层池。详见 [docs/操作SOP/数据采集.md §三 §五](../../docs/操作SOP/数据采集.md)。

### 4.1 三轴模型（quality / traveler_fit / execution）

每个 Full 必须**三轴中至少两轴有真实证据**：

| 轴 | 信号源 | 通过标准 |
|---|---|---|
| **quality** | 米其林 Keys / Forbes Stars / Relux 评分 / 一休评分 / 业内榜单（Tablet Hotels / National Geographic） | 该价位段是否够强 |
| **traveler_fit** | 携程评分+评论 / 小红书 collects > likes / 60+ 独立站（台湾博主实住评测最丰富） | 中国游客实际入住满意 |
| **execution** | OTA 真实价格抓取 / 营业稳定 / 接待外国客政策 / 交通可达 | 用户能订到能入住 |

**三轴规则**（4 档可信度）：
- **三轴全过** → `verified` 入 full
- **两轴过** → `cross_checked` 入 full
- **一轴过** → `single_source`，标注后入 full
- **无任何真实证据** → `ai_generated`，禁止上生产

### 4.2 数据源

完整 60+ 站清单见 [独立站清单.md](独立站清单.md)。酒店重点用：

- **P0 OTA + 评级权威**：一休.com（日本本地最权威）/ 楽天トラベル / じゃらん / Relux / Booking / Agoda / 携程（execution 主源·中国游客 RMB 价格）/ 米其林 Keys（2024+） / Forbes Travel Guide Stars
- **P1 中国游客辅助**：小红书（collects > likes 是 traveler_hot 信号）/ 大众点评 / 马蜂窝
- **P2 台湾博主实住评测最丰富**：Mimi 韩 / 波比看世界 / 乐活的大方 / 阿波旅行中 / 熊猫爱吃鱼 / Adventurous Mark / Vivianexplore
- **P2 区域分析**：Inside Kyoto / Inside Osaka / icotto / aumo / 関西おでかけ手帖 / HK01 旅游 / imtravelholic

### 4.3 价格采集 3 时段法

OTA 真实价格不取单日，必取 3 时段中位数：
- 淡季（1 月中-2 月 / 6 月初）
- 平季（多数月份）
- 旺季（樱花 3/底-4/初 / 红叶 11/中-12/初 / 黄金周 / 年末年始）

`price_cny_per_night` 的区间 = 平季中位数 → 旺季中位数·**不写淡季最低价**（误导）。

### 4.4 负向信号（任一触发即降 skeleton 或不收）

| 信号 | 处理 |
|---|---|
| 接待外国客政策不友好（不接受/不会英文/小费要求） | 不收 |
| 评论两极（清洁度差/服务差） | 降 skeleton |
| OTA 虚标价（标 ¥600 实际只有月初一天 ¥600） | 用真实平季价·标"OTA 虚标" |
| 长期满房无法订（婚庆专用/商务包场） | 降 skeleton |
| 装修中 / 改名 / 闭店 | 立即降 skeleton 标注 |
| Booking 评分 < 7.5 / 携程 < 4.0 / 一休 < 3.5 | 降 skeleton |

### 4.5 使用优先级（实操）

1. 现有池（`assembly/hotels/data/hotels__*.json` 387+ 家）— 首选，但 D40 打标已被评审推翻，需重新核
2. P0 OTA 三时段抓真实价格 + Keys/Forbes/Relux 抓评级
3. P1 小红书 opencli 补本地视角（collects > likes 2-3 条）
4. P2 台湾博主实住评测补 traveler_fit

**不自己编**。

## 五、校验

校验脚本：`scripts/validate_hotels.py`

校验规则：
- 系统字段白名单（10 个）+ note 字段白名单（8 个）→ 未知字段 FAIL
- area 必须在 `{圈}/area_registry.json` → 不在 FAIL
- near_attractions.entity_id 必须在 `{圈}/entities/*.json` → 不在 FAIL
- tier 枚举校验 / type 枚举校验 → 不在 FAIL
- price_cny_per_night 必须 `[min, max]` 数组·禁 min=max → FAIL
- price 跟 tier 不一致（如 tier=三星但 price=[2000, 3000]） → 警告·检查归档错误
- 必填字段缺 → FAIL
- depth=full 但 note 字段缺 → FAIL（建议降 skeleton）

跑法：`python scripts/validate_hotels.py`，粘**完整输出**到交付消息。

## 六、历史教训

### 6.1 D40 五档打标失败被推翻（核心教训）

D40 时期把 387 家酒店按 budget_tier 五档（economy/comfort/premier/luxury/ultra_luxury）批量打标，结果评审推翻：
- **套话 editor_note**（"性价比之选"/"豪华体验"通用模板）
- **area 错标**（神户温泉酒店标市中心）
- **枚举不匹配**（实际五档但部分文件四档）
- **价格用合并默认值**（不抓 OTA 真实数据）

**对策**：D43 重做，4 档 tier（三星/四星/五星/奢华）+ 双标签（type=city/experience）·价格必须 OTA 真实抓取·禁套话简介。

### 6.2 area 错标（已解决）

神户六甲温泉酒店标"市中心"，装配把温泉日酒店放成市区便利型。
**对策**：area_registry 白名单 + validator 强制校验。

### 6.3 价格虚标（已禁）

OTA 标 ¥600 但实际只是月初某天的促销价，平季实际 ¥1500。
**对策**：3 时段中位数法·区间 = 平季→旺季·不写淡季最低价。

### 6.4 房型/含早细节膨胀（已禁）

D40 字段含 room_count / rating_subscores / review_keywords / nearest_station / breakfast 详情等 8+ 字段。
**对策**：合并到 note.房型 / note.含早 / note.到店提醒 自由文本·细节归用户最终看·装配不消费。

### 6.5 type 边界混淆（注意点）

虹夕诺雅京都贵但是不是 city？高级万豪贵但不是 experience？
**对策**：判断按"住宿费打包体验吗"——打包了 experience，没打包 city。价格只决定 tier 不决定 type。
