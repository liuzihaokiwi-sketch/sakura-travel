# 旅行手账本模板制作指引（通用版）

> 本文档是通用操作指引，不绑定任何具体城市/国家。
> 关西实操案例见 TEMPLATE_CREATION_PROCESS_KANSAI.md
> 具体选择规则见 selection_rules.md

---

## 产品定位

我们卖的是**策展产品**，不是定制工具。
用户付费买的是"专家帮你做好了决策，你照着走"，不是"给你一堆选项自己拼"。

核心标准：**一个真实的旅行顾问看到方案会不会认可。**

## 体验设计原则

> 这些不是系统硬规则，是做模板的人心里要有的标准。
> 目的不是让每天都"很有设计感"，而是**防止景点堆砌——看起来很满但什么都没记住**。
> 不需要灌输给用户，而是潜移默化让用户感受到。

### 1. 一段行程2-3个情绪阶段，不是每天换主题

主题不是每天一个，而是按**城市/阶段**自然切换：

```
例：8天关西
大阪（前段）：热闹、能量、城市感
京都（中段）：安静、深度、文化沉浸
神户/收尾（后段）：放松、味觉、余韵
```

同一个城市内部不需要刻意制造情绪反差——大阪的几天都在"热闹城市"大氛围里，自然有节奏差异就够。**真正的情绪转折发生在城市切换的时候**——从大阪的喧嚣走进京都的安静，用户自然就感受到了。

### 2. 70%保底，10%才是"值了"

| 比例 | 内容 | 作用 |
|------|------|------|
| 70% | 核心S级景点 | 保证"不亏"，不为主题牺牲 |
| 20% | 次要搭配（咖啡/步行段/餐厅） | 立住当天的氛围 |
| 10% | 惊喜moment | **用户真正记住的部分** |

> 用户回忆时记住的不是70%的热门景点（大家都去过），而是那10%——摩天轮上日落的15分钟、小巷里的发现、师傅在面前切鱼的手法。所以10%虽然时间最短，选择要最用心。

### 3. B选项要真的"惊"

B不是"同类型另一家"，而是让用户眼前一亮：
- ✅ A回转寿司，B路边关东煮 → "从没想过可以这样吃"
- ✅ A串炸名店，B米其林焼鳥割烹 → "原来大阪还有这种水平"
- ❌ A拉面，B另一家拉面 → 没有惊喜

标准：**用户看完想发给朋友说"你看这个"**

### 4. 热门景点差异化靠"怎么玩"，不靠"去不去"

| 维度 | 说明 |
|------|------|
| 时间点 | 清晨/黄昏/夜晚感受完全不同 |
| 进入方式 | 从哪边切入，路上经过什么 |
| 搭配对象 | 同一核心景点+不同次要点=不同氛围 |
| 逗留方式 | 快进打卡 vs 慢慢散步 |
| 收尾方式 | 每天有个收束画面 |

### 5. 峰值要稀缺

不追求天天高潮。Peak-End Rule：人记住的是"最强烈的时刻"和"结尾"。

| 行程 | 大峰值 | 小峰值 |
|------|--------|--------|
| 6-8天 | 1个 | 1个 |
| 9-13天 | 2个 | 1-2个 |

峰值前要有铺垫（高光是一个序列不是一个点），峰值后要有余韵（不再塞同等级刺激）。

### 6. 人味——旅行不能只有空景

每2-3天至少有一个"有人在的moment"：市场烟火气、居酒屋街的笑声、寺庙的钟声、祭典的人群。不需要每天安排，但整段行程不能全是"看建筑物"。

### 7. 降档也是体验

高档用户吃一顿路边关东煮不是省钱，是被设计的本地生活moment。

### 8. 文案有温度

- ❌ "大阪城，1583年建"
- ✅ "你第一眼看到天守阁从樱花里冒出来的时候，会明白为什么大阪人管这儿叫'太阁さんのお城'"

每个推荐不是冷冰冰的信息罗列，而是一句让用户感受到"有个人在跟我说话"的文字：

- ❌ "大阪城，1583年建，世界文化遗产"
- ✅ "你第一眼看到天守阁从樱花里冒出来的时候，会明白为什么大阪人管这儿叫'太阁さんのお城'"

这个不写进规则里，但是做模板时心里要有。

## 总体架构

### 四层结构

| 层级 | 内容 | 变化频率 | 格式 |
|------|------|---------|------|
| **骨架层** | 核心基座日程 + 可插拔模块 | 低（城市不变就不变） | base_schedule.json |
| **内容层** | 餐厅/酒店/店铺推荐 | 中（店铺开关/价格变动） | meals/hotels/shops.json |
| **运营层** | 票价/营业时间/预约渠道/交通时刻 | 高（随时可能变） | live_facts（待建）|
| **规则层** | 选择逻辑/替换规则/联动规则 | 低（策略确定后很少改） | rules.json |

### 模块化思维

不要写"一篇完整的7天攻略然后让AI删"。
要写"核心基座 + 可插拔模块，系统往上拼积木"。

```
核心基座（最少天数）= 固定
+ 用户选择模块（半天决策项）
+ 人群日模块（情侣/闺蜜/亲子/默认）
+ 季节活动模块（日期匹配自动加入）
```

## 用户决策

### 用户问卷设计

**核心问题（必填，每题都会显著改变结果）：**

| # | 问题 | 选项 | 为什么必须问 |
|---|------|------|-------------|
| 1 | 这趟更像哪种出游？ | 约会感 / 出片感 / 亲子 / 经典版 | 决定内容气质和人群变体 |
| 2 | 想排得多满？ | 特种兵 / 平衡版 / 悠闲度假 | 决定每天slot数量和砍减力度 |
| 3 | 预算大概在哪个档？ | 经济 / 中档 / 高档 | 决定酒店餐厅全部推荐 |
| 4 | 半天决策项 | USJ要不要？岚山要不要？等 | 直接改变天数和行程结构 |

**可选修饰器（不强制，没选就默认）：**

| # | 问题 | 选项 | 默认 |
|---|------|------|------|
| 5 | 酒店更偏向？ | 住得更好/更有度假感/更看重性价比/帮我平衡 | 帮我平衡 |
| 6 | 有需要特别注意的吗？ | 自填文本框，可跳过 | 空 |

> 第6题覆盖老人/残障/过敏/特殊需求，不做成正式选项标签。

**其余全部系统决定：** 餐厅、店铺、晚间体验、季节活动。

**"旅行密度"影响系统行为：**
| 密度 | 系统行为 |
|------|---------|
| 特种兵 | 用满版模板，每天slot不砍 |
| 平衡版 | 适度砍P4-P5，留出自由时间 |
| 悠闲度假 | 砍到P1-P2，每天只保留核心+一个亮点 |

### 半天决策项的识别标准

一个景点/活动是否需要让用户选：**会不会因为去了它而放弃另一个区域/半天的行程。**
- 1-2小时的景点 → 写死在模板里，不问
- 半天以上的 → 让用户选yes/no

## 推荐策略

### A/B结构（适用于所有推荐品类）

| | A（安全牌） | B（惊喜牌） |
|---|---|---|
| 定位 | 系统默认推荐 | 用户可能不知道但值得试 |
| 知名度 | 高，大众接受 | 低，小众特色 |
| 体验角色 | 主流体验 | 与A体验角色不同 |
| 呈现 | 主推 | "想试试不一样的？" |

**核心理念：**
- 推荐 ≠ 让用户选。A是主推，B是加分项
- A/B是**隐性个性化**——不可能把用户所有偏好都问一遍，A/B给两个不同方向，用户偏好通过自己的选择表达
- A和B必须**体验角色不同，优先不同菜系**——有时同为寿司或同为肉类，只要氛围、形式、价格带、在地感差异够大，安全牌和惊喜牌仍然成立

### 人群差异处理

- 大部分A/B通用，不分人群
- 只有"明显更好的替代"时才换（不是"也可以"，而是"体验差异大到值得换"）
- 替换后仍保持A/B差异原则

### 兴趣标签替换

替换优先级：**兴趣标签 > 人群标签 > 默认**

## 景点规则

### S级景点分类

| 类型 | 特征 | 处理 |
|------|------|------|
| 顺路型 | 1-2h，不影响日程 | 写死在模板里 |
| 半天决策型 | 3h+，挤掉其他内容 | 用户勾选 |
| 季节升级型 | 平时A/B级，特定季节升S | 系统按日期自动处理 |
| 日期锁定型 | 祭典/花火，固定日期 | 系统按日期自动加入 |

### 每天每个slot标优先级

P1（不砍）→ P2 → P3 → P4 → P5（先砍）
时间不够从P5开始砍。

### 季节活动处理

- 不需要独立一天的 → overlay加到现有天（上午/晚上）
- 需要独立一天的 → 独立模块拼上去
- 季节活动是**加进去**，不是替换。时间不够砍常规低优先级slot

## 餐饮规则

### 预算策略

> 核心：不是"每顿都吃同档"，而是"普通餐压低，高光餐拉高"。

| 档位 | 约8顿正餐的分布 |
|------|----------------|
| 经济 | 5-6顿经济 + 2顿中档 + 0-1顿中高档 |
| 中档 | 2顿经济/低中档 + 4顿中档 + 1-2顿中高档 + 0-1顿高档 |
| 高档 | 1顿经济/低中档 + 2-3顿中档 + 3-4顿高档 + 0-1顿顶级 |

> 中档和高档也要降档：日本大量"低价但非常值得"的餐位，到达恢复餐/园区恢复餐/本地生活体验餐天然适合降档

### 日类型自动升降

| 日类型 | 调整 |
|--------|------|
| 到达日午餐 | -1（刚到不折腾） |
| 主题公园/暴走日晚餐 | -1（累了吃什么不重要） |
| 人群日午餐 | +1（开始升温） |
| 人群日晚餐 | +1~+2（全程最高点） |

### 人群修饰器

| 人群 | 修饰 | 体验重点 |
|------|------|---------|
| 情侣 | 人群日晚餐+1，可触发顶级 | 氛围+好看 |
| 闺蜜 | 人群日晚餐+1，可触发顶级 | 颜值+拍照 |
| 亲子 | 整体-0.5~-1档，上限中高档 | 互动+友好 |
| 默认 | 人群日+1 | 稳定 |

### 全程约束

- 经济档：最多1顿中高档
- 中档：最多1顿高档，至少1顿经济体验
- 高档：最多1顿顶级，至少2顿中档或以下
- 亲子：不触发顶级
- **showcase meal上限**：每城最多1顿，全程最多2顿。防止城城都顶到最满拉坏预算

### 每顿饭有角色（meal_role）

| 角色 | 含义 |
|------|------|
| arrival_recovery | 到达恢复餐，轻松简单 |
| core_local_experience | 城市核心名物体验 |
| theme_park_recovery | 主题公园后恢复餐 |
| audience_day_showcase | 人群日高光餐，全程最高 |
| everyday_good | 日常好餐，稳定输出 |
| local_life_experience | 本地生活体验（路边摊/站着吃），降档但有体验价值 |

### 升档必须选城市代表性菜系

升档花的钱要花在"这个城市才能吃到"的东西上。
不能在A城市升档吃B城市的代表菜系。

### 口味搭配

- 午餐和晚餐菜系不重复
- 一天内不吃两顿同类主食
- 连续两天同一菜系不出现两次
- 口味要多样：不能天天本地传统菜，要穿插西餐/其他风格
- 不推荐用户本国菜系（用户从中国来不推荐中华料理）
- 小吃slot不参与菜系去重

### 跨城市菜系去重

**只对A（主推）轨道去重，不对全部候选去重。**
B是备选，用户可能不选，不应该因为B的存在就误判为"已消费"。

去重逻辑：
1. 先拼装全程所有城市的A轨道
2. 扫描A轨道有没有同菜系重复
3. 重复了 → 优先城市保留，其他城市触发fallback
4. 再检查B是否与同天A完全重合（同菜系+同体验角色），重合则换B

菜系冲突处理：
- 城市独占菜系不参与去重（如只有A城市有的名物）
- 有优先城市的菜系（如和牛归神户），其他城市同菜系slot带fallback
- 通用菜系（拉面/天妇罗/居酒屋/烧鸟等）不硬排他，设全程上限1-2次，**第二次必须承担不同meal_role**（如第一次是everyday_good，第二次必须是local_life_experience）
- 每个冲突菜系slot在meals.json里带fallback字段

### 预约难度过滤

- ★★★★★（2个月+/需介绍/会员制）→ 不进入推荐池
- ★★★★（1-2个月）→ 仅高档/顶级推荐，标注提前时间
- ★★★以下 → 正常推荐

## 酒店规则

### 区域选择

按停留天数和后续行程决定住哪个区域（每个城市单独定义）。

### 搜索方法论（适用于酒店/餐厅/店铺/活动所有品类）

任何品类的搜索必须从**至少4个维度交叉**，不能只从一个方向搜：

| 维度 | 搜什么 | 能发现什么 |
|------|--------|-----------|
| 按区域 | 各主要区域逐个搜 | 常规选项 |
| 按排名/榜单 | top榜、米其林、Conde Nast、一休排行 | 头部名牌 |
| 按体验类型 | 温泉/川床/宿坊/町家/设计等 | 特色体验型 |
| 按地理扫描 | 把城市分成东/西/南/北/中心/近郊逐一搜 | 不在主流区域但值得去的 |

搜完后做**交叉检查**：别的来源反复提到但我的清单里没有的，一定是漏了。

常见漏洞：
- 只搜了"最方便的区域"，漏了"值得专程去住的近郊"
- 只搜了"常规酒店"，漏了"体验型住宿（寺庙/山里/河边）"
- 只看了中文来源，漏了日文/英文独立站的推荐

### 系统直接推荐

- 不给用户多家让他选，系统根据人群+预算+标签选最匹配的一家
- 手账本：推荐+理由+备选1家（小字）
- 备选要与推荐差异化

## 店铺规则

### 定位

不排进时间线，是**"附近逛一逛"信息附录**。

### 推荐数量

| 自由时间 | 推荐数 |
|---------|--------|
| 半天以上 | 5家 |
| 1-2h | 3家 |
| <1h | 1-2家 |
| 无 | 0 |

### 替换矩阵

- 默认5家差异化
- 人群替换（情侣/闺蜜/亲子各替换部分）
- 兴趣替换（二次元/购物/古着等）
- 替换优先级：兴趣 > 人群 > 默认
- 每家打评级（S+/S/S-/A/B），时间不够从低评级砍

---

## 跨天联动

### 餐饮联动
- 连续两天菜系不重复
- 小吃也错开

### 体力联动
- 前一天晚睡/暴走 → 第二天上午低优先级slot可砍或延后

### 店铺联动
- 跨天不推荐同区域/同类型

---

## 天气Plan B

- 所有户外景点必须有室内替代方案
- 全天户外活动（主题公园）提前确认天气，雨天建议调换
- 季节活动因天气取消 → 回退到常规模板
- 标注"雨天加分项"（温泉/博物馆/室内购物/咖啡厅）

## 信息分层

手账本不是百科全书，信息有主次：

| 层级 | 内容 | 呈现 |
|------|------|------|
| 主线（必看） | 景点+餐厅+酒店 | 大字，核心页面 |
| 次级（扫一眼） | 拍照点+咖啡+小吃 | 中字，穿插在页面里 |
| 附录（感兴趣翻） | 店铺+趣味体验+技巧 | 小字，折叠/附录页 |

| 模块 | 必须 | 说明 |
|------|------|------|
| 景点 | 是 | 当天主要活动 |
| 午餐 | 是 | A+B，按预算档 |
| 晚餐 | 是 | A+B，按预算档 |
| 小吃 | 推荐 | 当地名物零食 |
| 咖啡/休息点 | 推荐 | 逛累了坐一坐 |
| 拍照点 | 推荐 | 路过顺手拍 |
| 店铺推荐 | 可选 | 按自由时间决定数量 |
| 趣味体验 | 可选 | 手工/体验类 |

## 专业点评

所有推荐实体需要真实评价：

**餐厅：** 排队时间/推荐菜/口味/注意事项/环境/一句话总结
**酒店：** 位置/房间/特色/注意/一句话总结
**活动：** 体验亮点/最佳时段/注意/一句话总结

规则：
- 必须来自真实来源，不编造
- 负面信息圆滑表达（让用户有心理准备但不受伤）
- 所有评价内容用目标用户语言

---

## 多维评级

所有推荐实体需要多维评级（ratings.json）：

- 景点 × 人群（couple/friends/family/elderly）
- 餐厅：菜系整体评级 + 特定店铺override
- 酒店 × 人群 + 特殊标签（温泉/购物等）
- 店铺 × 人群 × 兴趣标签
- 体验活动 × 人群

评级供系统选择时参考，不直接展示给用户。

## 输出文件清单

每个城市产出：

```
templates/{city}/
├── manifest.json          # 模板入口（变体/用户选择/标签）
├── base_schedule.json     # 日程骨架（核心基座+模块）
```
