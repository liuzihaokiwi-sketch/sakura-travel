# 酒店、餐厅与店铺选择规则

> 核心原则：
> 1. 系统替用户做决策，不让用户自己拼攻略
> 2. 用户只做3个显式选择：USJ是否 / 人群类型 / 预算档位。其余全部系统定
> 3. **推荐 ≠ 让用户选。** 可以推荐多家，但必须有明确的主推（A），用户翻开就知道该去哪。B和其他是"感兴趣还有这些"，是加分项不是选择题。不能对用户造成决策困扰
> 4. **A/B是隐性个性化。** 不可能把用户所有偏好都问一遍（吃不吃河豚、爱不爱吵闹），问多了用户烦，问少了覆盖不全。A/B给两个不同方向的好选择，用户的偏好通过他自己的选择表达，而不是通过问卷

## 酒店选择规则

### 按天数选区域
- 大阪1-2天 → 住梅田（方便切京都，阪急/JR直达30-40min）
- 大阪3天+ → 住难波/心斋桥（道顿堀步行5min，深度玩大阪）
- 最后一晚 → 泉佐野（关西机场10min，赶早班飞机）

### 按预算分档（人民币/晚）
| 档位 | 淡季 | 旺季 |
|------|------|------|
| 经济 | 200-400 | 400-600 |
| 中档 | 400-800 | 600-1200 |
| 中高档 | 800-1500 | 1200-2500 |
| 高档 | 1500+ | 2500+ |

### 系统直接推荐最匹配的一家
- 不给用户2-3家让他选，系统根据人群+预算+标签自动选最合适的
- 手账本呈现：推荐酒店 + 一句话理由 + 备选1家（小字）
- 同一档位内备选要有差异化（比如推荐主打温泉，备选主打交通直结）
- 不推荐预约极难或会员制的酒店

### 人群匹配
- 每家酒店有audience评级（见ratings.json）
- 系统在用户预算档位内，选audience评级最高的那家
- 例：情侣+中档 → 系统选光芒酒店难波（顶楼Sky Spa夜景，情侣S+）

## 餐厅选择规则

### 基本结构
- 每天安排：午餐 + 晚餐 + 小吃（可选）
- 每餐有A和B两个选项：
  - **A = 安全牌**（系统默认推荐给用户的，知名度高、接受度广）
  - **B = 惊喜牌**（小众特色，用户可能不知道但值得试，体现手账本策展价值）
- A和B必须**体验角色不同，优先不同菜系**——有时同菜系但氛围/形式/价格带差异大，安全牌和惊喜牌仍然成立
- 手账本呈现：推荐A + 备选B（小字"想试试不一样的？"）

### 预算分档（人均/人民币）
| 档位 | 人均 |
|------|------|
| 经济 | ~100元以内 |
| 中档 | 100-300元 |
| 中高档 | 300-500元 |
| 高档 | 500-800元 |

### 餐饮策略：预算基线 + 日类型升降 + 人群修饰

> 核心理念：日本餐饮不是"每顿都吃同档"，而是"普通餐压低，高光餐拉高"。
> 日本¥1000左右的午餐性价比极高，没必要每顿都硬顶。
> 预算档决定整段行程的重心，日类型和人群做修饰。

**预算基线（约8顿正餐）：**
| 档位 | 分布 |
|------|------|
| 经济 | 5-6顿经济 + 2顿中档 + 0-1顿中高档 |
| 中档 | 2顿经济/低中档 + 4顿中档 + 1-2顿中高档 + 0-1顿高档 |
| 高档 | 1顿经济/低中档 + 2-3顿中档 + 3-4顿高档 + 0-1顿顶级 |

**日类型自动升降：**
| 日类型 | 调整 | 理由 |
|--------|------|------|
| 到达日午餐 | -1 | 刚到不折腾 |
| USJ/暴走日晚餐 | -1 | 累了吃什么不重要 |
| 人群日午餐 | +1 | 开始升温 |
| 人群日晚餐 | +1~+2 | 全程最高点，餐饮高潮 |

**人群修饰器：**
| 人群 | 修饰 | 体验重点 |
|------|------|---------|
| 情侣 | 人群日晚餐+1，可触发顶级 | 氛围+好看：景观/包间/吧台 |
| 闺蜜 | 人群日晚餐+1，可触发顶级 | 颜值+拍照：摆盘/装修/甜品 |
| 亲子 | 整体-0.5~-1档，人群日晚餐上限中高档 | 互动+友好：儿童餐/不嫌吵 |
| 默认 | 人群日+1 | 稳定 |

**全程约束（防规则失控）：**
- 经济档：最多1顿中高档
- 中档：最多1顿高档，至少1顿经济体验
- 高档：最多1顿顶级，至少2顿中档或以下
- 亲子：不触发顶级
- **showcase meal**：每城最多1顿，全程最多2顿

**每顿饭有角色（meal_role）：**
- arrival_recovery：到达恢复餐，轻松简单
- core_local_experience：城市核心名物体验
- theme_park_recovery：主题公园后恢复餐
- audience_day_showcase：人群日高光餐，全程最高
- everyday_good：日常好餐，稳定输出
- local_life_experience：本地生活体验（路边摊/站着吃），降档但有体验价值

**升档必选城市代表性菜系：**
| 城市 | 升档必选 |
|------|---------|
| 大阪 | 河豚 / 高端串炸 / 寿司omakase / 焼鳥割烹 / 肉割烹 |
| 神户 | 神户牛 / 和牛铁板烧 |
| 京都 | 怀石料理 / 汤豆腐 / 京料理 |
| 奈良 | 柿叶寿司 / 大和料理 |

**降档也是体验：** 高档用户吃一顿路边关东煮不是降级，是本地生活体验。

### 跨城市菜系去重（全程拼装后检查）

**只对A（主推）轨道去重，不对全部候选去重。**
B是备选用户可能不选，不应因为B存在就误判为"已消费"。

去重逻辑：
1. 拼装全程所有城市的A轨道
2. 扫描A轨道有没有同菜系重复
3. 重复了 → 优先城市保留，其他城市触发fallback
4. 再检查B是否与同天A完全重合（同菜系+同体验角色），重合则换B

**城市独占菜系（不参与去重）：**
- 大阪：章鱼烧、串炸、大阪烧、河豚、551肉包
- 京都：怀石、汤豆腐、京料理、抹茶甜品
- 神户：神户牛（和牛优先权归神户）
- 奈良：柿叶寿司

**有优先城市的菜系：**

| 菜系 | 优先城市 | 其他城市fallback方向 |
|------|---------|-------------------|
| 和牛/烤肉 | 神户 | 京都→京鸭料理/精进料理；大阪→肉割烹/焼鳥割烹 |
| 寿司omakase | 大阪 | 京都→京料理割烹 |

**通用菜系（不硬排他，设上限+不同meal_role）：**

| 菜系 | 全程上限 | 第二次必须承担不同meal_role |
|------|---------|--------------------------|
| 拉面 | 2次 | 第1次everyday_good → 第2次local_life_experience |
| 天妇罗 | 2次 | 不同城市，不同meal_role |
| 居酒屋 | 2次 | 同上 |
| 乌冬 | 2次 | 同上 |
| 洋食 | 2次 | 同上 |
| 烧鸟 | 2次 | 同上 |

**备选机制：**
每个冲突菜系slot在meals.json里带`fallback`字段，系统拼装后自动触发。

**不需要备选的情况：**
- 城市独占菜系
- 小吃slot（零食不参与去重）
- 全程只出现1次的菜系

**人群日是沉浸体验日：** 餐饮要和当天活动形成一体化：
- 情侣日：海游馆→日落摩天轮→高层餐厅→夜景（全天浪漫递进）
- 闺蜜日：洋馆咖啡→美国村→网红甜品→梅田晚餐（全天好看好拍）
- 亲子日：海游馆→商场午餐→乐高→摩天轮（全天轻松无压力）

### 口味搭配规则（AI必须遵守）
- 午餐和晚餐**菜系不重复**
- 一天内**不吃两顿粉物**（拉面、乌冬、大阪烧不同天吃）
- 章鱼烧/551肉包是零食slot，不算正餐，不受菜系限制
- **两天之间同一菜系不出现两次**（A/B总共，Day 1用了拉面，Day 2不能再有拉面）
- 优质特色店即使与前一天撞菜系也可推荐（因为用户可能前一天没选那个）
- **口味多样性**：不能天天都是传统日料，日本也有很多优秀的西餐（洋食/法餐/意大利料理）、铁板烧等，要穿插安排
- **不推荐中华料理**：用户是从中国来的，不需要在日本吃中餐（551蓬莱肉包除外，这是大阪名物不是"中华料理"）

### 人群差异处理
- 大部分A/B通用，不分人群
- 只有当某个人群有**明显更好的替代**时，才把A或B换掉
- 替换标准：不是"也可以"，而是"氛围/体验差异大到值得换"
- 例：情侣+高档晚餐，A从串炸换成寿司omakase（约会氛围完全不同）
- 替换后仍然保持A/B菜系不同的原则

### A/B推荐通用规则
- 不只是餐厅，所有需要推荐的品类（晚间体验/咖啡厅/店铺等）都用A/B结构
- A = 安全牌（知名度高、接受度广、系统默认推荐）
- B = 惊喜牌（小众特色、体验差异大、用户可能不知道但值得试）
- A和B必须有明显差异（不同类型/不同风格/不同氛围）
- 手账本上两个都展示，A为主推，B为"想试试不一样的？"
- 例：晚间体验 A=爵士酒吧（安静氛围） B=裏难波居酒屋（热闹接地气）
- 例：咖啡厅 A=网红打卡型 B=本地老喫茶店

### 预约难度过滤
- 预约难度★★★★★（2个月+/一见谢绝/会员制）→ 不进入推荐池
- 预约难度★★★★（1-2个月）→ 仅在高档/顶级推荐，标注"需提前X月预约"
- 预约难度★★★以下 → 正常推荐
- 具体排除名单：尽诚（一见谢绝）、mikami limited50（会员制）、京洛烤肉ぽめ（~1年）、本湖月（2个月+电话抢）、もめん（2个月+仅9席）

### 地理就近原则
- 餐厅推荐必须在当天活动区域内或附近
- Day 1在道顿堀/难波 → 只推荐这个区域的餐厅
- Day 2在新世界 → 下午餐推荐新世界附近
- Day 4在天保山/梅田 → 推荐对应区域
- 不让用户为了吃一顿饭跨城区坐30分钟地铁

### 数据要求
- 推荐的餐厅必须有真实数据来源（tabelog评分/米其林/多个攻略来源）
- 没有任何来源验证的店不进入推荐池
- price_tier缺失的，根据菜系和来源推断档位，标注"待验证"

## 店铺推荐规则

### 定位
- 店铺不排进行程时间线，作为**"附近逛一逛"信息附录**
- 告诉用户当天活动区域步行范围内有哪些店，感兴趣自己去
- 类似手账本里的一个小模块

### 推荐数量根据当天自由时间决定
- 自由逛街时间多（半天以上）→ 推荐5家
- 自由时间中等（1-2h）→ 推荐3家
- 自由时间少（行程紧凑）→ 推荐1-2家
- 无自由时间（USJ整天）→ 不推荐

### 每天推荐店铺要求
- 必须在当天活动区域步行范围内
- 推荐店铺之间要有差异化（不同类型/不同风格）
- 每家打评级（S+/S/S-/A/B），时间不够从低评级开始砍

### 评级标准
| 评级 | 含义 |
|------|------|
| S+ | 必去，不砍 |
| S | 强烈推荐，时间紧也尽量保留 |
| S- | 推荐，时间不够可砍 |
| A | 有兴趣去，没兴趣跳过 |
| B | 锦上添花 |

### 替换规则
- 用户有兴趣标签（二次元/购物/古着等）和人群标签（情侣/闺蜜/亲子等）
- 替换店铺继承被替换店的位置，评级独立打分
- 每个标签组合都有完整的5家推荐+评级

### 替换优先级
```
1. 兴趣标签（二次元/购物/古着） — 最优先，用户明确勾选的
2. 人群标签（情侣/闺蜜/亲子） — 其次
3. 默认5家 — 兜底
```

- 勾了兴趣+人群 → 用兴趣版本（兴趣覆盖人群）
- 只勾了人群 → 用人群版本
- 什么都没勾 → 默认5家

## 跨天联动规则

### 餐饮联动
- 连续两天不吃同一菜系
- 连续两天不吃同类粉物（拉面/乌冬/大阪烧）
- 每天的小吃slot也要错开（Day 1章鱼烧 → Day 2串炸）

### 体力联动
- 前一天晚睡（酒吧/祭典/花火）→ 第二天上午第一个slot可砍或延后
- 前一天暴走（USJ整天）→ 第二天上午安排轻松
- 按优先级砍：上午低优先级slot先砍

### 店铺联动
- 不推荐跟前一天同区域的店铺
- 每天推荐的店铺类型不重复（Day 1推了古着，Day 2推别的类型）

## 雨天Plan B规则

### 通用原则
- 所有户外景点必须有室内替代方案
- USJ等全天户外活动，需提前确认天气，雨天建议调换到其他天
- 雨天不影响室内景点（海游馆、博物馆、商场购物、温泉）

### 大阪户外景点 → 室内替代

| 户外景点 | 雨天Plan B |
|---------|-----------|
| 大阪城公园散步 | 大阪城天守阁内部（室内）+ 缩短公园时间 |
| 新世界街头逛 | 通天阁内部 + 室内串炸店 |
| 道顿堀逛街 | 有屋顶商店街覆盖，影响不大 |
| 心斋桥 | 心斋桥筋有顶棚，不受影响 |
| 天保山摩天轮 | 海游馆延长时间 |
| 箕面红叶 | 改为大阪市内室内景点（博物馆/温泉） |
| USJ | 尽量调换到晴天；如无法调换，园内有室内项目但体验打折 |
| 岸和田花车祭 | 不可打伞穿雨衣，小雨照常，暴风雨取消 |

### 雨天加分项（下雨反而适合去的）
- 温泉（御宿野乃/空庭温泉）
- 博物馆（大阪国立美术馆/京都国立博物馆）
- 室内购物（大丸百货/Grand Front/难波Parks）
- 咖啡厅巡礼

## 每天必备模块清单

每天除了景点/餐厅/店铺，还需要配备：

### 咖啡厅/休息点
- 每天推荐1-2家当天活动区域步行范围内的咖啡厅
- 不排进时间线，用户逛累了自己去
- 分人群：闺蜜要拍照好看的，情侣要有氛围的，带长辈要座位舒服的
- 数据待补采

### 拍照打卡点
- 每天列出当天路线沿途的拍照点
- 不占时间slot，路过顺手拍
- 例：Day 1 格力高看板、道顿堀运河、心斋桥筋入口等
- 数据待补采

### 趣味体验类（待补采）
- 食品模型制作体验（道具屋筋附近）
- 章鱼烧自己做体验
- 千日前厨刀店（刻名服务）
- 唐吉诃德（本身就是体验）
- 日本桥电电街（まんだらけ/Super Potato等，二次元标签用）
