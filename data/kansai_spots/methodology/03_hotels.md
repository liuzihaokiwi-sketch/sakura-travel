# 酒店采集 -- 品类特有规则

> 前置阅读：00_playbook.md
> 更新：2026-04-02 基于关西395家实战

---

## 核心思路

目标很简单：为游客找到最合适的酒店。分两种：
- **便利型** -- 住着方便，交通好，性价比。按城市组织
- **体验型** -- 值得专程去住，住宿本身就是体验。按场景组织

不按价格分，按"住的目的"分。有马温泉的花小宿TWD5320和Aman京都几十万日元都是体验型。

## 不要做什么（反面教训）

1. **不要先设计系统再采集** -- 不预设"城市x区域x档次"表格再填数据。先搜索，搜到什么再组织
2. **不要只搜大词** -- "京都住宿推荐"只能搜到便利型热门。体验型必须按场景搜（见下）
3. **不要只看搜索摘要** -- 一篇"京都住宿30选"文章里提到30家，必须逐家提取，不能只挑5-6家
4. **不要排序打分** -- AI本身知道交通信息（难波到USJ 15分钟），不需要在数据里算好。只管采集，判断交给AI
5. **不要排除低价** -- CNY200以下不收（不符合产品调性），但CNY300-500的必须充分覆盖，这是主力客群
6. **不要忽略用户盲区** -- 用户会问"泉佐野那家"，暴露的是采集时没想到的场景（赶早班机）

## 采集流程（实际执行5步）

### Step 0: 搜"住哪里方便"
先回答最基本问题。搜索词：
- 简中："京都住宿推荐 住哪里方便"
- 繁中："京都住宿推薦 飯店 住哪裡方便"
- 日文："京都 ホテル おすすめ エリア別"
- 英文："best area to stay in Kyoto for tourists blog"
- 香港："京都酒店推介 住邊區方便"

这一步的目的不是找酒店，而是找**优质独立站**。记下每个搜索结果中出现的博客站。

### Step 1: 深入独立站批量提取
用OpenCLI web read抓取各独立站的住宿推荐文章（台湾/香港站是JS渲染，WebFetch抓不到）。

操作示例：
```bash
cd opencli-main
node dist/main.js web read --url "https://mimigo.tw/kyoto-hotel/" --output "d:/tmp/webcli" --download-images false
```

每篇文章逐家提取酒店：名称/区域/价格/特色。一篇文章通常有15-30家。

### Step 2: 场景搜索（体验型的关键）
便利型Step 0-1就够了。体验型必须换搜索词：

| 场景 | 搜索词示例 | 会发现什么 |
|------|-----------|-----------|
| 红叶温泉 | "京都 紅葉 ホテル 客室から見える" | 翠岚、渡月亭、菊水、もみぢ家 |
| 料理旅馆 | "京都 料理旅館 懐石 一泊二食" | き乃ゑ、柚子屋、ゆとね、天ぷら吉川 |
| 宿坊 | "京都 宿坊 朝のお勤め 精進料理" | 智积院、仁和寺、知恩院、金光院 |
| 町屋 | "京都 町屋 一棟貸し 庭園" | 町家レジデンスイン、季楽、葵STAY |
| 蟹季温泉 | "城崎温泉 旅館 蟹 冬" | 西村屋、三木屋、銀花 |
| 城市夜景 | "大阪 夜景 高層 ホテル カップル" | Conrad、W、阪急インターナショナル |

### Step 3: 多轮补充
- 第一轮后回查搜索结果遗漏（逐条check）
- 新开业/新装修酒店：搜 "{城市} 新しいホテル 2024 2025 オープン"
- 小红书真实价格：`opencli xiaohongshu search "京都住宿 性价比"` 搜索标题含价格线索
- 用户指定补充

### Step 4: 旧数据吸收
如果有旧数据池，和新池做模糊匹配去重，有价值的保留（如城崎/高野山旧数据中新池漏的）。

## 便利型主源

| 来源 | 价值 | 局限 |
|------|------|------|
| 台湾博主实住(Mimi韩/波比/阿波等) | 最详细的优缺点、价格、照片 | 偏中高端，CNY300以下覆盖弱 |
| 香港站(HK01/imtravelholic) | 港币价格、亲子视角 | 部分被robot challenge |
| 小红书 | 真实住客体验、真实CNY价格 | note内容抓不到(2026-04)，只有搜索标题 |
| 换乘案内の案内君 | 带CNY价格的平价酒店列表 | 只覆盖大阪 |

## 体验型主源

| 来源 | 价值 | 局限 |
|------|------|------|
| 一休.com | 体验型酒店评分排名、紅葉客室特集 | WebFetch 403，用WebSearch间接 |
| MICHELIN Keys | 最高权威 | 只覆盖高端 |
| icotto/aumo | 红叶酒店/新开酒店/料理旅馆特集 | 编辑视角，非用户评分 |
| 関西おでかけ手帖 | 红叶/料亭/町屋专题 | 部分页面超时 |

**体验型不能用携程做主源** -- 很多顶级旅馆不在携程上或评论极少，产品溢价在于推荐用户发现不了的好东西。

## 文件组织

```
hotels/
  convenient/          按城市分（AI查"大阪有哪些"）
    {city}.json        一级城市各一个，二三级城市可合并
  experience/          按场景分（AI查"想泡温泉住哪"，可能跨城市）
    onsen.json         温泉旅馆（有马/城崎/白滨/市内温泉酒店等）
    design_luxury.json MICHELIN Keys/建筑师设计/顶奢
    ryokan_cuisine.json 料理旅馆（怀石/柚子锅/川床等一泊二食）
    shukubo.json       宿坊（京都寺院+高野山等）
    machiya.json       一栋贷町屋
    seasonal.json      季节限定（红叶客室/樱花/雪景）
```

discovery_pool/hotels_candidate_pool.csv 是全量平文件，JSON是分类后的消费文件。

## AI消费方式

不需要预先把所有酒店塞进context。给AI一个search_hotels工具，AI自己决定查什么：
- 收到"情侣去USJ" -- 查convenient/osaka + experience/design_luxury
- 收到"带爸妈泡温泉" -- 查experience/onsen
- AI本身就知道交通信息（难波到USJ 15分钟），不需要在数据里写死

## 规模参考（关西圈实战）

一级城市(京都/大阪)各约100家便利型，二级城市(神户/奈良)各约15-40家。
体验型总量约为便利型的一半。温泉旅馆通常最多（圈内有温泉地的话），其次是设计/奢华。
全量候选池约400家左右比较充分。

## 酒店特有教训

1. **场景搜索是关键** -- "有馬温泉 紅葉 露天風呂"才能搜到月光园，大词搜不到
2. **OpenCLI web read必须用** -- 台湾/香港博客全是JS渲染，WebFetch只拿到CSS
3. **小红书搜索有价值** -- "京都300多住万豪"（2002收藏）发现Four Points Flex，note详情抓不到但标题有线索
4. **新开业要主动搜** -- 2024-2025大阪万博前开业潮（Waldorf/Canopy/Patina/Caption/Series等），不搜就漏
5. **CNY300-500是主力区间** -- 台湾博主不写这个价位，要靠小红书和换乘案内补
6. **用户视角补盲区** -- 用户问"泉佐野那家""art大阪"暴露了采集未覆盖的需求场景
7. **旧数据要对比吸收** -- 城崎6家旅馆、高野山4家宿坊是从旧池中救回来的
