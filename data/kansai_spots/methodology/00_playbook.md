# 数据采集通用手册

> 适用范围：所有品类（餐厅/POI/酒店/精品店），所有城市圈
> 读一次，然后读具体品类文档

---

## 一、数据质量红线

1. **ai_generated = 生产事故** — 所有事实性字段（评分/价格/坐标/营业时间）必须有真实来源URL。搜不到填null，绝不编造
2. **source_url 必填** — 每条数据必须标明来自哪个页面
3. **不允许"先AI生成再补验证"** — 必须先搜真实数据源，AI只做辅助判断
4. **交叉验证是核心信号** — 被3+个独立来源提到的数据比单一来源高分更可靠
5. **质量 > 数量** — 宁可少收10条也不收1条假数据

## 二、搜索策略

### 多语言搜索
四种语言覆盖不同视角：
- **简中**："大阪美食推荐"/"京都必去景点" — 大陆游客视角
- **繁中**："大阪美食推薦"/"京都必去景點" — 台湾+香港博主（日本旅游内容最多最详细）
- **日文**："大阪 グルメ おすすめ"/"京都 観光" — 本地视角
- **英文**："best restaurants osaka"/"things to do kyoto" — 国际视角

### 不预设框架
先搜索再组织，不是先画框架再填数据。让数据告诉你该怎么分类。

### 搜索工具
- **WebSearch** — 发现文章和独立站
- **WebFetch** — 从文章页面提取具体数据
- **OpenCLI xiaohongshu search** — 小红书搜索（note详情抓取器已坏，只有标题+赞数）
- **OpenCLI ctrip search** — 携程搜索

OpenCLI是关键工具——台湾/香港博客很多是JS渲染，WebFetch抓不到，必须用OpenCLI。

### 独立站发现方法
不预先指定用哪些站。搜索"XX推荐"时，自然会发现高质量的独立站。跳过大平台（携程/小红书/马蜂窝/知乎），关注独立博客和专业站。

## 三、城市分级

每个圈的城市分三级，影响采集强度：

| 级别 | 说明 | 采集强度 |
|------|------|---------|
| 一级 | 游客主要停留地（2-4天） | 全量采集 |
| 二级 | 常见day trip或短住 | 中等采集 |
| 三级 | 半天或特定目的 | 只采精华 |

关西为例：一级=京都/大阪，二级=神戸/奈良，三级=姫路/有馬/城崎/宇治

## 四、文件格式规范

### 目录结构
```
data/{circle}_spots/
  restaurants/     按城市分文件
  poi/             按城市分文件 + 季节限定 + S级名单
  hotels/          按功能分目录（convenient/experience）
  shops/           按区域分文件
  methodology/     采集方法论
```

### JSON格式
所有品类统一字段风格：
```json
{
  "city/category": "说明",
  "count": 数量,
  "items": [
    {
      "name_ja": "日文名",
      "city_code": "kyoto",
      "area": "区域",
      "type/cuisine/hotel_type/shop_type": "类型",
      "source": "来源",
      "brief_note": "一句话说明"
    }
  ]
}
```

## 五、采集节奏

### 两阶段采集
- **Phase 1**：批量采集基础字段（店名/评分/区域/类型/来源）。快速建池，不进详情页
- **Phase 2**：对推荐池逐条补全详情（营业时间/定休日/预约/价格）。只补值得补的

### 多轮补充
- 第一轮：主源批量（Tabelog/japan-guide等）
- 第二轮：权威补漏（米其林/百名店/JNTO等）
- 第三轮：有机搜索（多语言搜索，独立站自然浮现）
- 第四轮：小红书/携程补中国游客视角

### 并行执行
最多3个agent并行，按语言或城市分工。

## 六、交叉验证

合并所有来源后，每条数据记录 source_count（被几个独立来源提到）。

- source_count ≥ 3：强信号，高置信度
- source_count = 2：有交叉，可信
- source_count = 1：单一来源，需谨慎

source_count 不直接等于评级（金閣寺可能只出现2次但绝对是S+），但是重要的参考信号。

## 七、经验教训（跨品类通用）

1. **先搜索再组织** — 不预设框架填数据，让数据告诉你结构
2. **场景搜索** — 大词搜不到的好东西，要用场景词搜（"红叶+温泉"/"情侣+京都"）
3. **逐条提取** — 搜索摘要中提到的店必须逐条check，否则大量遗漏
4. **标签要归一化** — 多语言搜索来的标签会混乱（temple/寺庙/寺院），合并时必须统一
5. **先基础后详情** — Phase 1只采6个字段，避免在详情页上浪费时间
6. **小红书搜索有用但有限** — note详情抓不到，只能用标题+赞数做热度信号
7. **新开业的要主动搜** — 新店没有评论，不会出现在任何榜单
