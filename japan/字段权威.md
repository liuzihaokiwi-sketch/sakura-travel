# 日本字段权威（city/area 命名表 + 命名规则）

> 跟全球 `docs/项目核心/字段权威.md`（定字段名+全局枚举）+ 各圈 `area_registry.json`（区圈 area 白名单）形成三层。
> 本文档定义**全日本城市圈共享的 city/area 命名规则**·任何关西/关东/北海道圈的数据池字段都遵守这里。

---

## 一、命名硬规

### 1. city 命名

- **格式**：英文 snake_case slug（小写·下划线分隔）
- **粒度**：城市级（行政市/温泉地/独立旅游目的地各算一个 city）
- **唯一性**：全日本范围唯一（kyoto 不能跟 kanto 圈再来一个 kyoto）
- **不允许**：中文（京都/大阪/...）·片假名（キョウト）·空格·驼峰

### 2. area 命名

- **格式**：英文 snake_case slug（小写·下划线分隔）
- **粒度**：区圈下子区域（一个 city 下可有多个 area）
- **唯一性**：**(city, area) tuple 在区圈内唯一**·area slug 本身可以跨 city 重复（如 `bay_area` 在 osaka 和 kobe 都可以有，因为 city 不同）
- **可读性**：要表意清晰·能让 agent 一眼知道是哪里
  - 好：`shijo_kawaramachi` / `higashiyama_axis` / `arima_onsen` / `koyasan_temple`
  - 差：`area_01` / `kyoto_a` / `north`（太泛）
- **冲突防范**：如果两个 city 都有「kita」（北区），分别命名 `kita`（city=kyoto）和 `kobe_kita`（city=kobe）·让 area slug 自带 city 前缀消歧（全日本通行做法·参考关西现有命名）

### 3. area_registry.json 字段约定

每个区圈根目录下放一个 `area_registry.json`·结构：

```json
[
  {"area": "shijo_kawaramachi", "type": "主场密集", "city": "kyoto"},
  {"area": "arashiyama", "type": "日归动线", "city": "kyoto"},
  ...
]
```

- `area`：英文 snake_case slug（见上）
- `type`：area 属性枚举·见各区圈 `这是什么.md`（关西用：主场密集 / 日归动线 / 温泉宿坊 / 景点单日）
- `city`：英文 snake_case slug

**严禁中文 city/area**·全部英文 slug。

## 三、参考 area 命名风格（关西现行·开新区圈参考）

### 主场密集型（市内核心区）
- 京都：`shijo_kawaramachi` / `higashiyama_axis` / `karasuma_oike` / `gion` / `nakagyo` / `nijo_central` / `kyoto_center`
- 大阪：`umeda_kita` / `namba` / `shinsaibashi` / `namba_dotonbori` / `honmachi` / `osaka_central`
- 神户：`sannomiya` / `kobe_central` / `motomachi_nankinmachi`

### 日归动线型（市内+周边日归）
- 京都：`arashiyama` / `kita` / `fushimi` / `ohara` / `sakyo` / `kyoto_station` / `higashiyama_north` / `kinkaku_nishiyama`
- 大阪：`tenma` / `horie` / `nakazakicho` / `nipponbashi` / `tennoji_shinsekai` / `osaka_ikeda` / `amerika_mura`
- 神户：`kobe_kitano` / `kitano_motomachi` / `kobe_waterfront` / `harborland_meriken` / `kitano_shinkobe` / `kobe_suma` / `kobe_nada`

### 温泉宿坊型
- `arima_onsen`（神户·有马）
- `kinosaki_onsen`（兵库·城崎）
- `koyasan_temple`（和歌山·高野山）
- `shirahama`（和歌山·白浜）

### 景点单日型
- `usj` / `osaka_castle` / `tempozan` / `bay_area`（大阪）
- `himeji`（兵库·姬路城）
- `nara_park` / `nara_yoshino` / `horyuji` / `nishinokyo` / `naramachi`（奈良）
- `kumano_kodo`（熊野古道）/ `amanohashidate_ine`（天桥立）
- `kobe_rokko` / `kobe_kita`（六甲山+北神户）

### 命名 anti-pattern（不要）

- `shimogyo_ku`（行政区代号·读者不知道是哪）
- `central`（无 city 前缀·歧义）
- `area_a` / `north_1`（无意义编号）
- `京都站`（中文）
- `kyoto-station`（横线·应用下划线）

## 四、跟全球字段权威的关系

```
docs/项目核心/字段权威.md          ← 全球·定字段名+字段含义+全局枚举（category/depth/可信度等）
        ↓
japan/字段权威.md（本文档）        ← 日本·定 city/area 命名规则+参考列表
        ↓
japan/{kansai,kanto,...}/area_registry.json  ← 区圈·真正的 area 白名单（agent 直接消费）
```

**字段定义** → 看全球字段权威；**city/area 怎么取名** → 看本文档；**这个圈现在有哪些 area** → 看 area_registry.json。

## 五、开新区圈步骤（如开 kanto）

1. **建目录**：`japan/kanto/`
2. **写 `这是什么.md`**：定位+区域基因+装配硬规（参考 `japan/kansai/这是什么.md`）
3. **建 `area_registry.json`**：照本文档命名规则填·开局可只填 5-10 个核心 area·后续补
4. **建数据池目录**：`entities/` `restaurants/` `stops/` `hotels/` `templates/`
5. **D49 通用化已实现**：validator 自动从 `area_registry.json` 读取 area 白名单·无需改代码

### 加新 area / city 顺序

1. **先改 area_registry.json**（加一行）
2. **再改数据池**（entity/restaurant/stop/hotel/template 引用新 area）
3. **跑 validator** 确认通过
4. **不允许反序**（先在数据里写 area·后补 registry——会让 validator 历史性 PASS 失效）
