# 酒店准入标准

> 装配层回答"**哪些酒店该进池、以什么深度呈现**"。
> 配套：[index.md](index.md) 装配规则、[写作规范.md](写作规范.md) 格式。
> 建于 2026-04-24。吸收历史酒店精选指南 v2.1 + MICHELIN Keys 五维。

---

## 核心思想：区域自然分类 + 区域内双轨

**用户选"住哪里"本身就决定了类型**：
- 选"有马" = 选温泉旅馆
- 选"高野山" = 选宿坊
- 选"祇园" = 选便利型 + 老铺旅馆
- 选"京都站" = 选便利型商务

所以**区域是天然分类单位**，不造"特色/"独立目录层。区域 md 内部按需分两段：
- **城市酒店**（便利型 / city_hotel）
- **特色住宿**（体验型 / ryokan_onsen / luxury_ryokan / shukubo）

例外：**町家**在京都多区域分布，独立成 1 份 md。

不贪多。全关西 Full 目标约 80-100 家。

## 一、类型枚举

| 类型 | 含义 | 用户场景 |
|---|---|---|
| `city_hotel` | 商务/连锁/设计/都市豪华 | 便利性主导，每天早晚出门 |
| `ryokan_onsen` | 温泉旅馆 | 为温泉去有马/城崎/岚山等 |
| `luxury_ryokan` | 顶级老铺旅馆（俵屋/柊家等） | 住宿本身是此行目的 |
| `machiya` | 京都町家改造 | 体验京都生活，一家一栋 |
| `shukubo` | 寺院宿坊（高野山/岚山等） | 精进料理 + 晨课 |

## 二、价位（4 档，通用）

| 档 | 日均（素泊） | 一泊二食倒推 | 典型 |
|---|---|---|---|
| `economy` | ¥8k-¥15k | — | 商务连锁 / 经济胶囊 |
| `mid` | ¥15k-¥35k | ¥25k-¥50k | 中档都市 / 普通温泉旅馆 |
| `high` | ¥35k-¥80k | ¥60k-¥120k | 都市豪华 / 顶级连锁 |
| `showcase` | ¥80k+ | ¥150k+ | 俵屋 / 虹夕诺雅 / MICHELIN 2-3 Keys |

含餐酒店按**素泊单价**倒推归档，不按总价。

## 三、Full 准入四道筛

### 筛 1：质量过线（分类型阈值）

| 类型 | 主源 | 阈值 |
|---|---|---|
| `city_hotel` | 一休 / 楽天 | 同区域同档位前 30%，或携程前 20% |
| `ryokan_onsen` | 一休.com | 区域内同档位前 25%，或 MICHELIN Keys |
| `luxury_ryokan` | MICHELIN Keys + 一休 | 基本都是 Keys，京都 8-10 家即止 |
| `machiya` | 独立站 + 小红书 | 运营 3 年 + 20+ 真实评价 |
| `shukubo` | 官网 + 独立站 | 稀缺，候选 ≥4 家就全收 |

### 筛 2：归一个区域 + 便利/体验归类

每家 Full 只挂 1 个区域 md。町家是例外，独立进 `kyoto/町家.md`。

**归类判断（便利型 vs 体验型）**：问一句"**用户为什么去那个区域？**"

- 用户是为了**逛市区 / 走动线**，顺带住——**便利型段**
- 用户是为了**这家酒店本身的温泉/町家/宿坊/老铺**，为此接受偏远或贵——**体验型段**

所以：
- **神户三宫的有温泉的高档城市酒店**（如神户温泉饭店）→ 便利型（用户来神户是逛港口/吃牛肉，温泉是附加）
- **有马温泉街上交通方便的旅馆**（如御所坊中央位置）→ 仍是体验型（用户来有马就是为了泡温泉，不是"顺路住有马"）
- **岚山温泉酒店翠岚** → 体验型（用户为了岚山温泉+嵯峨野氛围）
- **京都祇园便利型 + 顶级老铺旅馆混住区** → 便利型段收商务/连锁/都市豪华，体验型段收俵屋/柊家

判断尺度：**这家酒店是"住宿本身是目的"还是"便利性 + 附加体验"？** 前者体验型，后者便利型。

### 筛 3：覆盖缺口优先（每区域独立矩阵）

**城市酒店（city_hotel）每区域**：4 档 × 1 首推 + 1 备选 = 最多 8 家。已有首推+备选的档位不再升第三家。

**特色住宿每区域**：
- 温泉区（有马/城崎/岚山）：区域内温泉旅馆 3-5 家代表
- 宿坊区（高野山）：4-6 家
- 京都町家：跨区域合集 6-10 家
- 顶级老铺旅馆：归对应区域（俵屋/柊家 → 祇园.md），全京都合计 6-8 家

### 筛 4：负向信号（任一触发即降 skeleton 或不收）

| 信号 | 处理 |
|---|---|
| 隔音差（3+ 评论提及） | 降 skeleton |
| 设施老旧 + 价格不低 | 降 skeleton |
| 地址骗人（"步行 5 分钟"实际爬坡 15 分钟） | 标 access_friction，不降 |
| 房间极小（中档以上） | 降 skeleton |
| check-in 排队 30+ 分钟 | 标 practical_tip，不降 |
| 旺季价 >3x 平季（无特殊理由） | 标注，建议替代时段 |
| 中国游客近 1 年差评集中 | 不收 |

## 四、MICHELIN Keys 接入

五维标准：
1. 是目的地的入口
2. 建筑/室内设计
3. 服务质量一致性
4. 价格与体验匹配
5. 个性与真实性

- Three Keys → 自动 showcase + "此行目的"级
- Two Keys → 自动 high+/showcase
- One Key → 至少 high 备选

装配层可优先推 Keys 酒店。

## 五、区域 md 叙事要求

### 城市酒店为主的区域（四条/京都站/梅田/难波/祇园/三宫等）

头部必写四段：
- **住这里的人是谁**（第一次来 / 多刷 / 情侣 / 亲子 / 商务）
- **区域优势**（交通 / 动线 / 氛围 / 餐饮配套）
- **不该住这里的人**
- **典型行程建议**（如拆分住宿）

### 特色住宿为主的区域（有马/城崎/高野山/岚山等）

头部改写三段（不写便利性）：
- **来这里的理由**（泡温泉 / 宿坊体验 / 嵯峨野清晨）
- **怎么选**（一泊二食规矩 / 金泉银泉区别 / 宿坊起床时间）
- **什么人不适合**（赶行程 / 老人爬坡 / 挑食等）

## 六、价格采集硬规

**价格必须 OTA 真实数据，绝不 AI 估算。**

- 一休 / 楽天 / 携程各查 3 时段：淡季（1 月平日）/ 平季（5 月平日）/ 旺季（11 月周末 or 樱花季）
- 记录：最低-最高、是否含餐、是否含税
- 多平台取中位数

字段：
```
pricing.off_season_jpy: [min, max]
pricing.regular_season_jpy: [min, max]
pricing.peak_season_jpy: [min, max]
pricing.note: "含一泊二食 / 旺季翻 3 倍 / 素泊单价"
```
