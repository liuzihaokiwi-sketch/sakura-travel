# 旅行手账本产品原则（关西 348 元）

> 本文档是手账本产品的**唯一原则文档**，合并了此前分散的四份文件。
> AI 生成行程时只需读这一份。

---

## 1. 产品定位

**我们卖的是策展产品，不是定制工具。**

用户付费买的是"专家替你做好了决策的确定性"——照着走就对了，不需要自己比较、纠结、拼凑。

核心模式：**一套主线模板 x 微调参数 = 覆盖所有用户**。不是给每个用户从零生成，而是在一个经过反复打磨的主线上，根据用户输入做有限调整。

质量标准：**十个日本旅游专家联合做出的攻略。** 一个真实的旅行顾问看到方案会认可，而不是"AI 生成的看起来还行"。

---

## 2. 用户输入分类

用户输入分三层，系统处理方式完全不同：

### 硬约束（必须满足，不满足 = 产品失败）

- 出发/返回日期、总天数
- 明确勾选的模块（USJ、岚山等半天决策项）
- 季节活动的固定日期（祭典、花火大会）
- 已购票/已预约的项目

### 强偏好（尽力满足，影响氛围和选择方向）

- **人群类型**：情侣 → 氛围感；闺蜜 → 颜值出片；亲子 → 安全友好；默认 → 稳定经典
- **预算档位**：影响酒店星级、餐厅档次分布、体验活动选择
- **旅行密度**：影响每天 slot 数量和砍减策略

### 编辑默认（系统决定，用户不需要操心）

- 具体推哪家餐厅的 A/B
- 店铺、咖啡厅、拍照点的选择
- 每天的节奏编排和时间分配
- 季节活动的嵌入位置

> **判断标准**：一个半天决策项（3h+，会挤掉其他内容）才让用户选；1-2h 的景点直接写进模板。

## 3. 体验设计原则

### 3.1 情绪阶段按城市切换

一段行程 2-3 个情绪阶段，不是每天换主题。主题按**城市/阶段**自然切换：

```
大阪（前段）：热闹、能量、城市感
京都（中段）：安静、深度、文化沉浸
神户/收尾（后段）：放松、味觉、余韵
```

真正的情绪转折发生在城市切换时，同一城市内部不需要刻意制造反差。

### 3.2 70/20/10 内容比例

| 比例 | 内容 | 作用 |
|------|------|------|
| 70% | 核心 S 级景点 | 保证"不亏"，不为主题牺牲 |
| 20% | 搭配内容（咖啡/步行段/餐厅） | 立住当天氛围 |
| 10% | 惊喜 moment | **用户真正记住的部分** |

用户回忆时记住的不是 70% 的热门景点，而是那 10%——摩天轮上的日落、小巷里的发现、师傅在面前切鱼的手法。10% 时间最短，选择要最用心。

### 3.3 B 选项要真的"惊"

B 不是"同类型另一家"，而是让用户眼前一亮，标准：**用户看完想发给朋友说"你看这个"**。

- 正确：A 回转寿司，B 路边关东煮 → "从没想过可以这样吃"
- 正确：A 串炸名店，B 米其林烧鸟割烹 → "原来大阪还有这种水平"
- 错误：A 拉面，B 另一家拉面 → 没有惊喜

### 3.4 热门景点差异化靠"怎么玩"

不靠"去不去"做差异化，靠以下维度：

- **时间点**：清晨/黄昏/夜晚感受完全不同
- **进入方式**：从哪边切入，路上经过什么
- **逗留方式**：快进打卡 vs 慢慢散步
- **搭配对象**：同一核心景点 + 不同次要点 = 不同氛围
- **收尾方式**：每天有个收束画面

### 3.5 峰值稀缺

不追求天天高潮。Peak-End Rule：人记住的是"最强烈的时刻"和"结尾"。

| 行程长度 | 大峰值 | 小峰值 |
|---------|--------|--------|
| 6-8 天 | 1 个 | 1 个 |
| 9-13 天 | 2 个 | 1-2 个 |

峰值前要有铺垫（高光是一个序列不是一个点），峰值后要有余韵（不再塞同等级刺激）。

### 3.6 人味

每 2-3 天至少有一个"有人在的 moment"：市场烟火气、居酒屋街的笑声、寺庙的钟声、祭典的人群。整段行程不能全是"看建筑物"。

### 3.7 降档也是体验

高档用户吃一顿路边关东煮不是省钱，是**被设计的平价美食体验**。

### 3.8 文案有温度

每个推荐不是冷冰冰的信息罗列，而是一句让用户感受到"有个人在跟我说话"的文字。

- 错误："大阪城，1583年建，世界文化遗产"
- 正确："你第一眼看到天守阁从樱花里冒出来的时候，会明白为什么大阪人管这儿叫'太阁さんのお城'"

### 3.9 给线索，不给结论

手账本的描述是"吸引和勾起兴趣"，不是吹捧。给用户发现的方向，让用户自己走到那里得出结论。

- 错误："这是大阪最好吃的章鱼烧，入口即化让你难忘"（结论 + 承诺，容易让人失望）
- 正确："道顿堀旁边一条小巷，排队的全是本地人"（线索 + 悬念，用户自己验证）

专业感通过"我们考虑了你没考虑的维度"来传达，而不是形容词堆砌。decisions字段里解释为什么这样排，比说"我们为你精心安排"更有说服力。

## 4. A/B 推荐策略

适用于所有推荐品类：餐厅、晚间体验、咖啡厅、店铺。

| | A（安全牌） | B（惊喜牌） |
|---|---|---|
| 定位 | 系统默认推荐 | 用户可能不知道但值得试 |
| 知名度 | 高，大众接受度高 | 低，小众特色 |
| 体验角色 | 主流体验 | 与 A 体验角色不同 |
| 呈现 | 主推 | "想试试不一样的？" |

**核心理念：**

1. **推荐 ≠ 让用户选。** A 是默认，B 是"想试试不一样的？"——用户不选 B 也完全没问题
2. **A/B 是隐性个性化。** 不可能把用户所有偏好都问一遍，A/B 给两个不同方向，用户偏好通过自己的选择表达
3. **A 和 B 必须体验角色不同，优先不同菜系。** 同为寿司也可以，但氛围、形式、价格带、在地感差异必须够大

## 5. 餐饮策略

### 5.1 核心理念

不是"每顿都吃同档"，而是**"普通餐压低，高光餐拉高"**。中档和高档也要有降档餐——日本大量"低价但非常值得"的餐位，到达恢复餐/园区恢复餐/平价美食体验天然适合降档。

### 5.2 预算基线（约 8 顿正餐）

| 档位 | 分布 |
|------|------|
| 经济 | 5-6 经济 + 2 中档 + 0-1 中高档 |
| 中档 | 2 经济 + 4 中档 + 1-2 中高档 + 0-1 高档 |
| 高档 | 1 经济 + 2-3 中档 + 3-4 高档 + 0-1 顶级 |

### 5.3 日类型升降

| 日类型 | 调整 |
|--------|------|
| 到达日午餐 | -1（刚到不折腾） |
| USJ/暴走日晚餐 | -1（累了吃简单） |
| 人群日午餐 | +1（开始升温） |
| 人群日晚餐 | +1 ~ +2（全程最高点） |

### 5.4 人群修饰器

| 人群 | 修饰 |
|------|------|
| 情侣/闺蜜 | 人群日晚餐 +1，可触发顶级 |
| 亲子 | 整体 -0.5 ~ -1 档，上限中高档 |
| 默认 | 人群日 +1 |

### 5.5 全程约束

- **经济档**：最多 1 顿中高档
- **中档**：最多 1 顿高档，至少 1 顿经济体验
- **高档**：最多 1 顿顶级，至少 2 顿中档或以下
- **showcase meal 上限**：每城最多 1 顿，全程最多 2 顿

### 5.6 meal_role（每顿饭的角色）

| 角色 | 含义 |
|------|------|
| `arrival_recovery` | 到达恢复餐，轻松简单 |
| `core_local_experience` | 城市核心名物体验 |
| `theme_park_recovery` | 主题公园后恢复餐 |
| `audience_day_showcase` | 人群日高光餐，全程最高 |
| `everyday_good` | 日常好餐，稳定输出 |
| `affordable_local` | 平价美食（路边摊/站着吃/街头小店），降档但有体验价值 |

### 5.7 升档必须选城市代表性菜系

升档花的钱要花在"这个城市才能吃到"的东西上：

- 大阪：河豚/串炸
- 神户：和牛
- 京都：怀石/割烹

### 5.8 口味多样性

- 同天午餐和晚餐菜系不重复
- 一天内不吃两顿面食类主食
- 穿插西餐/其他风格，不能天天本地传统菜
- **不推荐中华料理**（用户从中国来，例外：551 蓬莱是大阪名物可推）

### 5.9 跨城市菜系去重

**只对 A（主推）轨道去重，B 不算已消费。**

规则：
1. 先拼装全程所有城市的 A 轨道
2. 扫描 A 轨道有无同菜系重复
3. 重复 → 优先城市保留（和牛归神户、怀石归京都），其他城市触发 fallback
4. 再检查 B 是否与同天 A 完全重合（同菜系 + 同体验角色），重合则换 B
5. 通用菜系（拉面/天妇罗/居酒屋/烧鸟等）全程上限 2 次，第二次必须承担不同 meal_role

## 6. 酒店策略

**系统直接推荐一家最匹配的，不给用户多家让他选。**

### 呈现方式

手账本中：
- 推荐酒店 + 推荐理由（主要篇幅）
- 备选 1 家（小字，与推荐差异化——如推荐选了便利型，备选给体验型）

### 选择逻辑

系统根据以下因素综合匹配：
- 人群类型（亲子需要房间大/安全；情侣需要氛围）
- 预算档位
- 当天活动区域（通勤时间合理）
- 后续行程衔接（换城日住车站附近）

### 酒店偏好修饰器

用户可选，不选默认"帮我平衡"：

| 修饰器 | 倾向 |
|--------|------|
| `quality_first` | 住得好，星级优先 |
| `resort_vibe` | 度假感，温泉/景观优先 |
| `value_for_money` | 性价比优先 |
| `balanced` | 系统综合判断（默认） |

---

## 7. 店铺策略

**不排进时间线，是"附近逛一逛"信息附录。**

### 推荐数量（按自由时间）

| 自由时间 | 推荐数 |
|---------|--------|
| 3h+ | 5 家 |
| 1-2h | 3 家 |
| <1h | 1-2 家 |
| 无 | 0 |

### 选择逻辑

AI 根据实体标签匹配用户的 party + interest 自动选择，不使用人工替换矩阵。

选择优先级：**兴趣标签 > 人群标签 > 默认**

示例：
- 用户标注"二次元" → 优先推动漫周边店、扭蛋专门店
- 用户标注"古着" → 优先推中古店、vintage shop
- 无兴趣标签的情侣 → 推文具/杂货/设计品牌
- 无兴趣标签的亲子 → 推玩具/零食/互动体验店
- 全部默认 → 推当地特色差异化 5 家

## 8. 模块拼装原则

### 8.1 基本结构

每个城市一条主线路线，模块往上叠加：

```
核心基座（最少天数）= 固定不变
+ 用户勾选模块（半天决策项：USJ/岚山/奈良等）
+ 人群日模块（情侣/闺蜜/亲子/默认 各有不同的高光安排）
+ 季节活动模块（按日期自动嵌入）
```

用户多余的天数用于 deep_dive（某个区域多花半天深度逛）或 recovery（睡到自然醒的休息日），不开新路线。

### 8.2 排序原则

**先定住点再排内容（sleep first, play second）。**

酒店位置决定当天的活动半径和交通效率。排序流程：
1. 确定每晚住哪个区域
2. 根据住点确定当天可覆盖的活动范围
3. 在范围内安排景点和餐厅
4. 最后填充店铺/咖啡/拍照点

### 8.3 换城日规则

换城日是显性的章节切换，不是"多塞一个景点"：

- 上午 = 出发城市内容，下午 = 目的地城市内容
- 强度必须低，下午不能安排 P1 重量级景点
- 行李策略必须明确定义（寄存/直送/酒店存放）
- 跨城 = 情绪切换（从大阪的喧嚣走进京都的安静），利用这个转折

### 8.4 模块冲突处理

用户勾选了太多模块 → 先满足明确的强模块，冲突时排序标准：

1. **时间稀缺性**：季节限定 > 全年可去
2. **不可替代性**：只有这个城市有 > 别的城市也有
3. **用户显式选择顺序**：先选的优先

被挤压的模块在手账本中给出解释文字（"如果多一天可以加上 XX"），而不是静默丢弃。

## 9. 密度策略

| 密度 | 系统行为 |
|------|---------|
| packed（特种兵） | 满版模板，每天 slot 不砍 |
| balanced（平衡版） | 砍 P4-P5，留自由时间 |
| relaxed（悠闲版） | 砍 P3-P5，只保留 P1-P2 核心 + 一个亮点 |

**特殊日保护：**
- USJ 日：任何密度都不砍，USJ 本身就是完整一天的体验
- 人群日：核心序列不砍，只砍外围补充内容

**砍减顺序**（每天每个 slot 标有优先级 P1-P5）：
- 时间不够从 P5 开始砍
- P1（不砍）→ P2（极端情况才砍）→ P3 → P4 → P5（先砍）
- 砍掉的内容在手账本中不出现（不是标灰，是直接不写）

## 10. 天气 Plan B

### 基本原则

- 所有户外景点必须有对应的室内替代方案
- 替代方案不是随便塞一个室内点，要与当天氛围匹配

### USJ 雨天

- 优先尝试与其他天调换，把 USJ 排到晴天
- 无法调换时：约 40% 的项目为室内，仍然可玩半天以上
- 手账本中标注"雨天推荐动线"

### 季节活动

- 季节活动因天气取消 → 回退到常规模板，不留空白
- 手账本中预写回退方案文字

### 雨天加分项

以下内容在雨天体验反而更好，优先安排：
- 温泉（雨中露天风吕）
- 博物馆/美术馆
- 室内购物（地下街/商业设施）
- 咖啡厅巡礼
- 市场/食堂类餐饮体验

## 11. 退化路径

当理想方案无法满足时，按以下优先级逐级退化：

### 11.1 天数不够

1. 保留核心锚点景点
2. 保留 1 个亮点模块
3. 保留 1 个人群记忆点
4. 砍外围内容
5. 砍店铺/咖啡/购物
6. 最后手段：砍主线内容

### 11.2 模块过多

1. 先满足用户明确勾选的强模块
2. 冲突 → 按时间稀缺性/不可替代性排序
3. 被挤压模块在手账本中加解释文字

### 11.3 餐厅池不足

1. 扩大步行半径
2. 放宽预算半档
3. 放宽 B 选项要求
4. 允许只有 A 没有 B
5. 变更 slot 角色（不动主线）

### 11.4 人群 + 预算冲突

安全/适配性上限永远优先于预算升级。

具体规则：
- 亲子 + 高档 → 安全友好优先，永远不触发顶级餐厅，酒店选亲子友好型而非最贵的
- 老人 + 特种兵 → 密度必须降级，体力约束优先于用户选的密度
- 情侣 + 经济 → 氛围优先于省钱，宁可少吃一顿好的也要保证人群日晚餐的质量

## 12. 信息分层

手账本不是百科全书，信息有主次：

| 层级 | 内容 | 呈现 |
|------|------|------|
| **主线（必看）** | 景点 + 餐厅 + 酒店 | 大字，核心页面 |
| **次级（扫一眼）** | 拍照点 + 咖啡厅 + 小吃 | 中字，穿插页面 |
| **附录（感兴趣翻）** | 店铺 + 趣味体验 + 实用技巧 | 小字，附录页 |

原则：用户拿到手账本，只看主线就能完成一趟好旅行。次级和附录是锦上添花，不看也不影响体验完整性。
