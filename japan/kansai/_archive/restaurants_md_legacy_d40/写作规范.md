# 餐厅区域策展 md 写作规范

> 适用于 `assembly/restaurants/{kyoto,osaka,other}/*.md` 的所有条目。
> 每条餐厅走 A 视图（字段栈，给装配 AI）+ B 视图（自然语言引用段，给用户）双层结构。

---

## 〇、核实范围（2026-04-24 修订）

**必核事实**（查不到就标"待核实"）：
- 营业时间 / 定休日
- 预约方式（渠道 + 开放窗口）
- 价格区间（午/晚）
- 米其林星级 + 当年年份
- 地址（核 area 是否跟源 JSON 一致）

**不必核、也不必写进条目**：
- 席数 / 创业年份 / 店主入行年数 / 道数
- 服务费比例 / 取消违约金细则 / 付款方式细节
- "曾在某店修行"这类师承（除非是主打卖点）

**描述性内容（B 视图）**：
- 流派、氛围、招牌菜、路线衔接按常识写即可
- 不要为了"显得专业"堆砌可核未核的数据
- 目标是"懂当地的朋友讲给你听"，不是百科页

**加一条信息前问自己**：用户真能用到吗？用不到就砍。

## 一、事实性字段硬规

### 1.1 营业/定休

- 以官网为准；官网无则以 Tabelog 当期页为准；两者皆无标「待核实」
- 写法：`营业：平日 10:00-16:00（周三至 15:30）/ 周末祝 10:00-17:00；木曜定休`
- 禁止写"大约 10 点开门"这种模糊推断
- 季节性营业（如贵船川床）必须标窗口月份

### 1.2 预约

- 区分三态：不接受预约 / 欢迎预约 / 必须预约
- 必须预约的要写清渠道（电话/Tabelog/一休/TableCheck）+ 开放时间（如"月初第一营业日 10:00 开始受理 2 个月后末日分"）
- 不能简写成"需预约"就完事——用户需要知道怎么约

### 1.3 排队

- A 档高热门（Tabelog 3.8+ / 米其林）必须有排队描述：旺季等多久、什么时段最松、有无取号机制
- 查不到具体数据就写"旺季基本无空位，建议预约"不要编分钟数
- 预约制餐厅排队段写「预约制，无排队」+ 退订窗口信息

### 1.4 价格

- 写午餐/晚餐区间，取官网或 Tabelog 标示值
- 套餐价格变化快，统一写「约¥X,XXX」不写精确到百位
- 季节性调价（如松茸/河豚）标「含食材时价浮动」

### 1.5 米其林

- 只写**当前星级**和**连续年份**（如"三星连续 3 年 2023-25"）
- 不写"连续 5 年"这种无法直接引 MICHELIN Guide 官网当年号的表述
- 星级改动以每年 11-12 月 MICHELIN Kyoto·Osaka 发布会为准

### 1.6 Tabelog 分数

- 写该页当前分数即可（如 `Tabelog：3.9`）
- 百名店年份可附（如 `3.9（2024 百名店 Tabelog 100 WEST 乌冬类）`）
- 不写"常年百名店"这种无法锁定年份的话

### 1.7 席数/创业年/道数

- 官网或正规媒体（東洋経済/ぴあ/Hanako）明示才写具体数字
- 否则模糊化：「小店，座位紧张」「江户时代传下的老铺」「多道怀石，随主厨组合」

### 1.8 地址与 area 核查

- 源 JSON `area` 字段**可能有错**
- 以店铺官网住所为准。源 JSON 区域错则：
  - 所在字段照官网写
  - 追加 `⚠️ 注：源 JSON 区域标注为 {旧 area}，实际位于 {真实区域}，待修正`
  - 在 `docs/04_操作SOP/D37_阶段2_候选变体清单.md` 或单独 area 修正清单留一笔

## 二、B 视图（引用段）写作约束

### 2.1 禁止项

- 禁止写源网站没有的比较句，如「和晚市同等料理序列」「比X店更好吃」「关西最好的X」
- 禁止推断的数值，如「人均消费 ¥X,XXX」（除非源明示）
- 禁止"网红"「爆款」等营销黑话（除非上下文需要反讽）
- 禁止臆造主厨背景（「曾在东京某三星修行」需有来源）

### 2.2 可写项

- 店主/主厨**公开信息**：年数、流派、师承（需 WebSearch 确认）
- 用户动作：怎么预约、点哪道招牌、适合几人、现金/刷卡
- 路线衔接：吃完步行到 X 景点 N 分钟、附近备选店
- 旺季操作：开门前到、取号、Zen Cafe 备选等
- 氛围描述：木格子、台面、暖帘这类**现场可见的物理特征**

### 2.3 字数

- 150-250 字为佳
- 最多一个 `**加粗**` 重点，用于招牌菜或关键操作提示
- 不用 emoji

### 2.4 档位标签

- `A-B 档：A档·稳妥首选` 这类**只在 A 视图**写
- B 视图引用段**不重复**档位标签

### 2.4.1 中日双语硬规（2026-04-24 增）

**所有文案以中文为主**，日文菜名/店名元素按以下规则处理：

- **餐厅名**：保留汉字/日文写法，首次出现用「中文（假名/原名）」格式，如「酒处 寺山（てらやま）」「法兰索瓦喫茶室（フランソア）」
- **菜名/术语**：中文翻译在前、日文原名在后括号内，如「金丝鳗鱼丼（きんし丼）」「厚松饼（ホットケーキ）」「撒酱拌饭（まぶしご飯）」「昼套餐（昼コース）」
- **人名/专名**：首次出现用「中文/译名（日文原名）」，如「美空云雀（美空ひばり）」「食堂小川（食堂おがわ）」
- **地名**：约定俗成的保留日文汉字（木屋町/先斗町/祇园）；假名混写的翻译，如「鱼介」「和牛」「鰹节」等单汉字词照写
- **禁止**：不要留下纯假名词不翻译（如「コース」「ホットケーキ」裸写），也不要用西里尔字母/乱码替代

### 2.5 Full 态扩展字段（两行，不加第三行）

Full 条目在 A 视图字段栈尾部加两行扩展字段：

- **体验维度**：窗景 / 町家 / 柜台独享 / 庭院 / 老铺 / 夜食 / 创意割烹 等，装配看这条挑氛围
- **午市**：有 / 无（标注是否有正餐午市；午茶/轻食另注）

**不加景点簇挂载行**——景点簇挂载统一在区域 md 顶部「景点簇索引」段写，装配 AI 读顶部表挑餐厅，单条不重复。

其他维度（素食 / 单人友好 / 儿童 / 语言 / 支付）按需写进 B 视图段落，不扩字段。

## 三、完成度标记

每条餐厅条目尾部加 HTML 注释标记当前深度：

```markdown
<!-- depth: skeleton -->
```

三态定义：

| 标记 | 含义 | 何时用 |
|---|---|---|
| `skeleton` | 旧扁平格式，只有 类型/预算/所在/特点/适配 | D36 前遗留，未改 |
| `verified` | 已核营业/预约/排队，字段栈完整 | 字段核完但 B 视图未改 |
| `full` | verified + 完整三段式 B 视图 | 最终态 |

**推进优先级**：A 档 + 预算 ≥¥10,000 优先推 `full`；B 档低价保留 `skeleton` 不动。

## 四、纪律

### 4.1 开工前

- 读 `japan/kansai/当前状态.md` 底部「进行中」段
- 追加一行「我在改 restaurants/kyoto/{file}.md」
- 避免两个 AI 抢同一个文件
- **必读 [assembly/restaurants/index.md](index.md) §1.4-1.5 档位配比**：用户表单 4 档（街头地道/本地好店/好好享受/顶级体验），绝不能一股脑推 showcase/high 忽略 mid/economy

### 4.1.1 每区域 full 名额（硬规）

**每区域 md 精选约 9 家做 full**，其余保留 skeleton 给装配 AI 做候选池。

按 high=N 的基准比例：
- **showcase**（顶级体验）= N/3
- **high**（好好享受）= N（基准）
- **mid**（本地好店）= N
- **economy**（街头地道）= 0.7N

N=3 时：**showcase 1 + high 3 + mid 3 + economy 2 = 9 家**

**为什么是倒金字塔**：用户表单里中档用户多、街头地道用户也多，顶级属少数人专程赴约。描述详尽的名额要反映实际消费分布，不是只给高端加戏。

**showcase 稀缺性**：index.md §1.8 硬顶"每城最多 1 顿 showcase，全程最多 2 顿"。祇园 showcase 已有 2 家（笹木、未在），按新规只应保留 1 家，另 1 家可考虑降为 skeleton 或合并为候选池。**执行时以"祇园 full 总 ≈ 9 家、showcase ≤ 1 家"为准绳**。

**反例**：本窗口第一次圈 35 家全是 A档+≥¥10,000 的 high/showcase，零个 mid/economy，结构性失衡。纠正：倒金字塔配比 + 按用户档位分布倒推入池，mid/economy 是主力。

**基因优先于菱形（2026-04-24 增）**：菱形 1/3/3/2 是**基因中性区**的基准（东山轴/河原町这类全场景区），不是铁律。中转快食区（京都站）可偏 economy 60% 省 high 档、景区单菜系区（岚山豆腐）可偏 high/mid 省 showcase、纯稻荷簇（伏见）可偏 economy 省 mid。**每份区域 md 在顶部「Full 目标」旁加一行说明**：「基因决定配比偏 X，菱形在此变形」，装配 AI 读到时不要质疑变形合理性。

### 4.1.2 报数字硬规

汇报"已完成 x / 待推 y"前必须跑脚本核验 depth 标记 count，不手算不估算。

### 4.2 核实时间盒

- 每条核实 3-5 分钟封顶
- 5 分钟没查到就标「待核实」，不要编数据
- WebSearch 官网 + Tabelog 两条为基线，查不到小红书找 3 条爆款补

### 4.3 SCHEMA 变更

- 发现需要新字段才能写清的情况，**先停**
- 去 `docs/03_数据契约/SCHEMA.md` 讨论增字段
- 不在条目里私自扩字段

### 4.4 数据真实性

- 所有数字、年份、地址、流派必须有来源
- 推断内容必须在 B 视图里用自然语言包装（如「据说」「网上反映」）或干脆不写
- 保持「真实日本旅行顾问看到会点头」的标准

## 六、中文硬规（用户可见字段一律中文）

所有用户可见字段一律中文。日文仅允许：
- 店名 / 菜名 / 站名首次出现用「中文（日文）」格式，后续直接中文
- entity 独立字段 `name_ja` / `name_en`（数据字段，不占文案空间）

### 字段级对照

✅ 正确：
```
特点：江户时代传下的老铺，用吉野本葛现做葛切
营业：平日 10:00-16:00（周三至 15:30），周末祝 10:00-17:00，木曜定休
预约：当日电话预约（9:00 开放）或 TableCheck 提前 3 日 0:00
B 视图：踏进来就是别处没有的京都质感，名物葛切用吉野本葛现制
```

❌ 错误（日文直接出现在文案）：
```
特点：江戸時代から続く老舗、吉野本葛の葛切が名物
营业：平日 10:00-16:00（水曜 15:30 まで）、土日祝 10:00-17:00、木曜定休
预约：当日電話予約（9:00 開放）または TableCheck
B 视图：お座敷に入った瞬間、京都らしい質感
```

### 专有名词保留白名单（无需翻译）

这些词中文餐饮界已通用，保留原样不加日文括号：
- 怀石 / 割烹 / 茶寮 / 一番出汁 / 锦市场 / 町家
- 葛切 / 抹茶 / 和菓子 / 鳗丼 / 炸豚排
- Tabelog / 一休

### 首次出现加括号的场景

用户会在 Google Maps / 店门口 / 官网看到日文时，首次括号：
- 「岚山小火车嵯峨站（トロッコ嵯峨站）」
- 「桌面红叶（テーブル紅葉）」—— 帮用户搜店
- 祭典正式名直接用中文译名（御手洗祭 / 祇园祭）

### 自检一句话

**写完每段读一遍：中国游客第一眼看懂吗？看不懂且没括号补说明 = 改。**

## 七、不编造数据（事实必有源）

所有数字 / 年份 / 地址 / 星级 / 师承 / 席数 / 营业 / 价格**必须有可追溯的源**。查不到 → 模糊化 / 标「待核实」 / 不写，三选一，**绝不编**。

### 字段级对照

✅ 正确：
```
Tabelog：3.9（查得到）
米其林：三星连续 3 年（2023-25）（米其林官网可验）
所在：小店座位紧张（席数无官方源就模糊化）
B 视图：江户时代传下的老铺（创业年无明示就模糊化）
B 视图：店主在食堂おがわ修行后独立（师承官网有写）
```

❌ 错误（编数字 / 编归属 / 推断比较）：
```
米其林：三星连续 5 年（官方是 3 年，合并历史星级错）
所在：18 席小店（官网/Tabelog 都没写席数）
B 视图：午市和晚市同等料理序列（官网无背书的推断）
B 视图：全京都最好的怀石（无法背书的评价）
B 视图：店主曾在东京三星修行（查不到的师承）
```

### 查不到的三种处理

1. **模糊化**：「小店」替「18 席」／「江户时代老铺」替「享保二年创业」／「多道怀石」替「全 8 品」
2. **标「待核实」**：营业时间 / 价格 / 米其林年份这类必核项
3. **不写**：师承 / 主厨背景查不到直接省略，不编

### B 视图禁推断

- ❌ 跨店比较句（"比 X 店好"/"和 X 同等"/"全 X 最好"）
- ❌ 无源数字（"人均 ¥X"/"排队 X 分钟"/"X 年历史"）
- ❌ 臆造师承 / 获奖
- ✅ 可写：氛围描述（木格子 / 柜台 / 暖帘）/ 官方公开信息 / 用户动作建议

### 自检一句话

**每处数字 / 归属 / 比较句都能指出源吗？指不出 = 改。**
