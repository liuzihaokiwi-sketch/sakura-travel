## 1. 用户输入三层分类(关西版)

### 硬约束(必须满足,否则产品失败)

- 出发/返回日期、总天数
- 明确勾选的模块(USJ/岚山/奈良等半天决策项)
- 季节活动固定日期(祇园祭 7/14-17、天神祭 7/24-25、五山送火 8/16、红叶/樱花窗口 ±5 天)
- 已购票/已预约项目

### 强偏好(尽力满足,影响氛围和方向)

- **人群**:情侣 → 氛围;闺蜜 → 颜值出片;亲子 → 安全友好;默认 → 稳定经典
- **预算档**:影响酒店星级、餐厅档次分布、体验选择
- **密度**:影响每天 slot 数量和砍减策略

### 编辑默认(系统决定,用户不操心)

- 餐厅 A/B
- 店铺、咖啡厅、拍照点
- 每天节奏编排
- 季节活动嵌入位置

**判断标准**:一个**半天决策项(3h+)**才让用户选;1-2h 景点直接写进模板。

## 2. 体验设计 9 原则(关西版细化)

详细原则见 `docs/01_项目定位/`。关西具体落地:

### 2.1 节奏:慢 - 快 - 慢 / 慢 - 慢 - 快

```
经典版(京都 + 大阪):
大阪(前段):热闹、能量、城市感 — 2 天以内
京都(中段):安静、深度、文化沉浸 — 3-5 天
神户/收尾(后段):放松、味觉、余韵 — 1-2 天
```

**铁律**:先京都后大阪**不对**——状态好时沉浸京都,末尾切换大阪;但如果是特种兵档(体力好+容错率高),反而**先大阪后京都**更合理(大阪适合时差恢复)。

### 2.2 70/20/10 内容比例(关西示例)

| 比例 | 关西具体 |
|---|---|
| 70% S 级 | 清水寺/岚山/伏见稻荷/金阁/道顿堀/大阪城/奈良鹿/有马金泉/东大寺 |
| 20% 搭配 | 锦市场步行/白川南通/鸭川畔/法善寺横丁/长田大佛/神户北野异人馆 |
| 10% 惊喜 | 黄昏渡月桥 15 分钟/清水寺子安塔背后视角/中崎町咖啡店/有马竹饰小径/奈良春日山原始林 |

### 2.3 B 要真的"惊"(关西示例)

- ✅ A 回转寿司(大起水产) + B 法善寺横丁站着吃关东煮 → "从没想过"
- ✅ A 蛸屋八 + B 米其林焼鳥「喜多川」 → "原来大阪还有这种"
- ❌ A 一兰拉面 + B 金久右卫门拉面 → 都是拉面,没惊喜

### 2.4 峰值稀缺(关西常见峰值)

| 行程 | 大峰值候选 | 小峰值 |
|---|---|---|
| 6-8 天 | 祇园黄昏 / 岚山黄昏渡月桥 / 有马金泉夜泡 | 伏见稻荷千本鸟居清晨 / 鸭川床纳凉 |
| 9-13 天 | 加 + 高野山宿坊 / 城崎蟹会席 / 祇园祭宵山 | 多一个春日山原始林 / 京都小夜秀 |

## 3. A/B 推荐策略(关西应用)

| | A(安全牌) | B(惊喜牌) |
|---|---|---|
| 定位 | 系统默认推荐 | 用户可能不知道但值得试 |
| 知名度 | 高,大众接受度高 | 低,小众特色 |
| 体验角色 | 主流体验 | 与 A 体验角色不同 |

**核心**:
1. **推荐 ≠ 让用户选**。A 是默认,B 是"想试试不一样的?"——用户不选 B 也完全没问题
2. **A/B 是隐性个性化**。通过用户选择表达偏好
3. **A 和 B 必须体验角色不同,优先不同菜系**。同为寿司也可以,但氛围、形式、价格带、在地感差异必须够大

## 6. 店铺策略(关西版)

**不排进时间线,是"附近逛一逛"信息附录。**

### 6.1 推荐数量(按自由时间)

| 自由时间 | 推荐数 |
|---|---|
| 3h+ | 5 家 |
| 1-2h | 3 家 |
| <1h | 1-2 家 |
| 无 | 0 |

### 6.2 选择优先级:兴趣标签 > 人群标签 > 默认

关西常见示例:
- 用户标注"二次元" → 优先推大阪日本桥电气街/京都漫画博物馆/扭蛋店
- 用户标注"古着" → 优先推下北泽风格的中古/vintage(大阪中崎町 / 京都三条小路)
- 无兴趣标签的情侣 → 推京都设计杂货 / 大阪堀江设计品牌
- 无兴趣标签的亲子 → 推梅田 Kids Plaza / 京都手作工坊
- 全部默认 → 推当地特色差异化 5 家(锦市场老铺 / 心斋桥筋 / 元町商店街)

## 7. 模块拼装原则

### 7.1 基本结构

```
核心基座(最少天数)= 固定不变(京都 2-3 + 大阪 2)
+ 用户勾选模块(半天决策项:USJ/岚山/奈良/姬路神户等)
+ 人群日模块(情侣/闺蜜/亲子/默认 各有不同高光)
+ 季节活动模块(按日期自动嵌入:祇园祭/樱花季/红叶季/大文字)
```

用户多余天数用于 **deep_dive**(某区域多花半天深度逛)或 **recovery**(睡到自然醒),不开新路线。

### 7.2 排序:Sleep first, play second

酒店位置决定当天活动半径。排序流程:
1. 确定每晚住哪个区域
2. 根据住点确定可覆盖活动范围
3. 在范围内安排景点和餐厅
4. 最后填充店铺/咖啡/拍照点

### 7.3 换城日规则

换城日是**显性的章节切换**,不是"多塞一个景点":
- 上午 = 出发城市,下午 = 目的地
- 强度低,下午不安排 P1 重量级景点
- 行李策略明确(寄存/直送/酒店存放)
- 跨城 = 情绪切换(大阪喧嚣 → 京都安静),利用转折

**关西特有**:京都-大阪 15 分钟新干线,**不需要过渡缓冲**——切换速度快反而产生强烈对比感,是关西魅力。例外:从沉浸式住宿(町家)切换到大阪商业区,可以在京都吃完晚饭再过去。

### 7.4 模块冲突处理

用户勾选太多模块时排序:
1. **时间稀缺性**:季节限定 > 全年可去
2. **不可替代性**:只有关西有 > 别的圈也有
3. **用户显式选择顺序**:先选的优先

被挤压的模块在手账本中给解释("如果多一天可以加上 XX"),不静默丢弃。

## 8. 密度策略

| 密度 | 系统行为 |
|---|---|
| packed(特种兵) | 满版模板,每天 slot 不砍 |
| balanced(平衡版) | 砍 P4-P5,留自由时间 |
| relaxed(悠闲版) | 砍 P3-P5,只保留 P1-P2 核心 + 一个亮点 |

**特殊日保护**:
- USJ 日:任何密度都不砍
- 人群日:核心序列不砍,只砍外围

**砍减顺序**:P5 → P4 → P3 → P2 → P1(不砍)。砍掉的内容在手账本中不出现(不标灰,直接不写)。

## 9. 天气 Plan B

### 基本原则

- 户外景点必须有对应的室内替代方案
- 替代方案要与当天氛围匹配,不是随便塞个室内

### USJ 雨天

- 优先尝试与其他天调换,排到晴天
- 无法调换:USJ 约 40% 项目室内,仍可玩半天以上
- 手账本标注"雨天推荐动线"

### 季节活动天气取消

- 回退到常规模板,不留空白
- 手账本预写回退方案文字

### 雨天加分项(优先安排)

- 温泉(雨中露天风吕)—— 关西有马/城崎强项
- 博物馆/美术馆(京都/大阪市中心)
- 室内购物(梅田地下街/京都大丸)
- 咖啡厅巡礼(京都町家咖啡)
- 市场/食堂餐饮体验(锦市场/黑门市场)

## 10. 退化路径

### 10.1 天数不够

1. 保留核心锚点
2. 保留 1 个亮点模块
3. 保留 1 个人群记忆点
4. 砍外围
5. 砍店铺/咖啡/购物
6. 最后:砍主线

### 10.2 模块过多

1. 先满足明确勾选的强模块
2. 冲突按时间稀缺性/不可替代性排序
3. 被挤压加解释文字

### 10.3 餐厅池不足

1. 扩大步行半径
2. 放宽预算半档
3. 放宽 B 选项要求
4. 允许只有 A 没有 B
5. 变更 slot 角色(不动主线)

### 10.4 人群 + 预算冲突(硬规)

**安全/适配性上限永远优先于预算升级。**

- 亲子 + 高档 → 安全友好优先,永不触发顶级餐厅,酒店选亲子友好而非最贵
- 老人 + 特种兵 → 密度必须降级,体力约束优先
- 情侣 + 经济 → 氛围优先于省钱,宁可少吃一顿也保证人群日晚餐质量

## 11. 信息分层(手账本呈现)

| 层级 | 内容 | 呈现 |
|---|---|---|
| **主线(必看)** | 景点 + 餐厅 + 酒店 | 大字,核心页面 |
| **次级(扫一眼)** | 拍照点 + 咖啡厅 + 小吃 | 中字,穿插 |
| **附录(感兴趣翻)** | 店铺 + 趣味体验 + 实用技巧 | 小字,附录页 |

**原则**:用户拿手账本,只看主线就能完成一趟好旅行。次级和附录是锦上添花,不看也不影响体验完整性。
