## 〇·〇、铁律（违反任何一条 = 全部不通过）

### 1. 字段不准膨胀

字段定稿就是这些（见 §二 + §二·五）。**不准私自加任何字段**，包括但不限于：
- `walk_min_estimated` ❌（已禁用）
- `drive_only` / `tags` / `vibe` 等 sonnet 自创 ❌
- 发现需要新字段 → **停下问用户**，不要先加再说

校验脚本对字段做严格白名单，未知字段 = FAIL。

### 2. 数据不准造假

**查不到 ≠ 写假的**。三种处理：
- 模糊化（"小店"替"18 席"）
- 标"待核实"或写"营业时段以官网为准"
- 字段留空或 null

**禁止行为**：
- 全部停留点写"通常 11:00-21:00"凑营业 ❌
- 米其林星级年份猜 ❌
- 步行分钟全标 estimated ❌
- 跨店比较句"全京都最好的怀石" ❌

### 3. entity 不为挂店服务

entity 必须**自己有意义**才加：
- 用户必去 / 模板动线锚点 / 文化地标
- 不是为了"店挂不上"而凑

**店挂不上就挂最近合理 entity，walk_min 大一点没关系**。挂不上任何合理 entity 的店 → 降 skeleton 或不进 full，**不为它加新 entity**。

### 4. walk_min 弹性

- **正餐池**：用户为吃顿好的愿意走，15-20 分都接受
- **停留池**：≤8 分软建议，散步小镇（城崎/伏见/高野山）/ 酒店附近场景可放宽
- **不为了"挂得近"伪造小数字**

### 5. depth 必须真实

- 地址查不到 / 营业全靠类目均值 = 自动降 skeleton
- full 必须三段式完整（字段栈 + B 视图 + 准 entity 挂载）
- 不强凑 full 撑数量

### 6. 主次分明（避免反复纠结）

- **A 类**（装配跑不动 / 字段缺失） → 停下报告
- **B 类**（用户体验硬伤：地址错 / 档位错 / 价格离谱） → 停下报告
- **小事**（Tabelog 缺 / 营业不全 / 文笔粗糙） → 标 verified 或留空，**继续做不停**

## 〇、池子拆两类

**正餐池**（restaurants/）：用户必去（每天必吃饭），走 tier×cuisine×档位匹配
**停留池**（stops/）：动线插件（顺路歇脚 / 顺路逛），不强制，走 near_attractions 就近匹配

两池字段不同、装配逻辑不同、目录平行：

```
japan/kansai/
├── restaurants/kyoto/      ← 正餐池
│   ├── 伏见.json
│   ├── 岚山.json
│   └── ...
└── stops/kyoto/            ← 停留池（咖啡/甜品/古着/书店/杂货/古道具/工艺/唱片等）
    ├── 伏见.json
    ├── 岚山.json
    └── ...
```

**进 stops 的店类型（开放枚举）**：
- 吃喝类：咖啡 / 甜品 / 抹茶 / 和菓子 / 茶寮 / 喫茶店 / 刨冰
- 购物类：古着 / 古书 / 书店 / 书店咖啡 / 杂货 / 设计杂货 / 文具
- 体验类：古道具 / 工艺品 / 手工艺 / 唱片 / 御宅店 / 当地土特产

**进 restaurants 的判断**：能解决一顿正餐的店（即使便宜也算）

**边界 case**：
- 伏見夢百衆（咖啡+品酒+酒馒头·无正餐）→ stops
- 鳥せい本店（酒藏居酒屋·能正餐）→ restaurants
- STANDARD BOOKSTORE（书+咖啡+杂货复合）→ stops（type=书店咖啡）
- 大阪现成 shops.json（23 家）→ 直接迁 stops/osaka/

## 二、字段结构（最终定稿，必须严格遵守）

```json
{
  "id": "kyo_gion_hyotei",
  "area": "北区",
  "near_attractions": [
    {"entity_id": "nanzenji_temple", "walk_min": 3},
    {"entity_id": "okazaki_park", "walk_min": 8}
  ],
  "tier": "showcase",
  "cuisine": ["怀石"],
  "recommended_meals": [
    {"meal": "lunch", "price_cny": [800, 1500]}
  ],
  "reservation_difficulty": "must",
  "queue_level": "none",
  "closed_days": [3],
  "season_months": null,
  "depth": "full",
  "note": {
    "店名": "瓢亭 本店（ひょうてい / Hyotei）",
    "简介": "...",
    "亮点": ["米其林三星", "庭院茶室", "朝粥名物"],
    "招牌菜": ["朝粥（限早场 8:00-10:00）", "松花堂弁当（午）"],
    "地址": "京都市左京区南禅寺草川町 35（地铁东西线·蹴上站步行 8 分钟）",
    "营业": "11:00-14:00 / 17:00-21:00；周三定休",
    "预约": "电话预约（075-...），3 个月前 1 日 10:00 开放受理",
    "到店提醒": "建议 14:00 后到避高峰；红叶季全天无空位务必预约",
    "Tabelog 分数": 4.3
  }
}
```

### 系统字段（装配消费）

| 字段 | 类型 | 必填 | 枚举值 / 格式 |
|---|---|---|---|
| `id` | string | ✅ | `{城市拼音简写}_{区域简写}_{店名拼音}`，如 `kyo_gion_hyotei` |
| `area` | string | ✅ | 中文区域名：东山轴 / 河原町 / 烏丸御池 / 京都站 / 岚山 / 北区 / 伏见 / 大阪难波 等 |
| `near_attractions` | array | ✅（≥1） | `[{entity_id, walk_min, walk_min_estimated?}]`。entity_id 必须在 `japan/kansai/entities/` 已存在；步行分钟估算的标 `walk_min_estimated: true`（默认 false 可省） |
| `tier` | string | ✅ | `showcase` / `high` / `mid` / `economy` |
| `cuisine` | array | ✅（≥1） | 字符串数组，主菜系放第一个。枚举：怀石 / 寿司 / 天妇罗 / 烧肉 / 拉面 / 乌冬 / 荞麦 / 居酒屋 / 洋食 / 咖啡 / 甜品 / 川床 / 精进料理 / 鳗鱼 / 烧鸟 / 河豚 / 中华 / 法餐 / 稻荷寿司 / 日本酒 / 自由文本。多菜系老铺如 `["稻荷寿司", "鳗鱼", "乌冬"]` |
| `recommended_meals` | array | ✅（≥1） | `[{meal, price_cny}]`。meal: `breakfast/lunch/dinner/cafe/bar`，price_cny: `[min, max]` 人民币区间。**精度取整到 50 或 100**（装配只分 4 档，差 50 块无意义）。**4 档区间是单顿统一标准·午晚通用**（不是"午餐档"和"晚餐档"两套尺子）：economy [0, 180] / mid [150, 500] / high [450, 1200] / showcase [1100, 无上限可达 5000+]。一家店午餐落 mid、晚餐落 high 是常态（如和牛/怀石午定食划算），按实际单顿价填即可，价格本身告诉装配落哪档·不需要 tier 区分午晚 |
| `reservation_difficulty` | string | 选填，缺省 `none` | `none`（不需）/ `recommended`（不预约要排队≥1h）/ `must`（不预约进不去） |
| `queue_level` | string | 选填，缺省 `mild` | `none` / `mild` / `high`（旺季常 60+ 分钟） |
| `closed_days` | array \| null | ✅（字段必须存在） | ISO 8601 数字数组：1=周一... 7=周日。三态：`[]`=已查实无定休 / `[3]`=周三定休 / `null`=未查到（装配看到 null 给用户"建议出发前确认"提示） |
| `season_months` | array/null | 选填 | `null` = 常年；`[5,6,7,8,9]` = 川床类季节限定 |
| `depth` | string | 选填，缺省 `skeleton` | `skeleton` / `verified` / `full` |

### note 字段（用户看，手账本渲染）

| key | 必填 | 说明 |
|---|---|---|
| `店名` | ✅ | 中文（日文假名 / 英文）。中日一致省括号；缺英文省 `/ Hyotei` |
| `简介` | full 必填 | 80-150 字，"懂当地的朋友讲给你听"口吻。讲为什么推荐 + 核心体验，不堆事实 |
| `亮点` | full 必填 | 3-5 个短词标签：米其林星 / 庭院 / 老铺 / 柜台独享 / 夜食 / 创意割烹 / 川床 等 |
| `招牌菜` | full 必填 | 列表，含限定提示如"朝粥（限早场）""松花堂弁当（午）" |
| `地址` | full 必填 | 「日本地址（最近交通+步行 X 分钟）」 |
| `营业` | full 必填 | 自然语言一句：开门时段 + 定休（系统字段 closed_days 也存一份数字，**作者两份都写**） |
| `预约` | full 必填 | 渠道+开放窗口。`reservation_difficulty=none` 的店写"无需预约" |
| `到店提醒` | 选填 | 错峰建议 / 旺季提醒 / 不规则定休（如"每月第 2 周二额外休"）/ 外国客提示 |
| `Tabelog 分数` | 选填 | 数字，如 `4.3`。查不到不写 |

## 二·五、停留池字段（stops/）极简版

**设计原则**：极简。停留点不是用户决策核心（用户不为它选行程），数据要少而准，**避免 sonnet 浪费 token 搜不必要信息**。

```json
{
  "id": "kyo_arashiyama_arabica",
  "area": "岚山",
  "near_attractions": [
    {"entity_id": "togetsukyo_bridge", "walk_min": 2}
  ],
  "type": "咖啡",
  "depth": "full",
  "note": {
    "店名": "% Arabica 京都 嵐山",
    "简介": "渡月桥北詰的爆款拍照咖啡店，世界各地分店里中国游客密度最高的一家。京都拿铁是招牌——浓缩奶泡上点抹茶粉，杯子上是樱花季限定花纹。20 平米小店，外带为主，店外渡月桥背景天然出片。",
    "亮点": ["京都拿铁", "渡月桥拍照机位"],
    "地址": "京都市右京区嵯峨天龙寺芒之马场町 3-47（渡月桥北詰即到）",
    "营业": "9:00-18:00；无定休；周末高峰排队 30-40 分钟"
  }
}
```

### 系统字段（5 个）

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | string | ✅ | 同正餐池规则 |
| `area` | string | ✅ | 动线级粒度，跟 templates/ 动线对齐。如：心斋桥-美国村 / 道顿堀-难波 / 堀江 / 日本桥 / 梅田 / 中崎町 / 福岛。**禁用城市级粗粒度**（如"大阪难波"覆盖 4 个动线区是错的） |
| `near_attractions` | array | ✅（≥1） | `[{entity_id, walk_min}]`。**禁用 walk_min_estimated 字段**（停留点本来就近似，不需要这个标记）。entity_id 必须挂到具体地标级 entity，不能 15 家全挂同一个站名 entity。entity 不存在 → 停下报告，不要将就挂远的 |
| `type` | string | ✅ | 开放枚举：咖啡/甜品/抹茶/和菓子/茶寮/喫茶店/刨冰/古着/古书/书店/书店咖啡/杂货/设计杂货/文具/古道具/工艺品/手工艺/唱片/御宅店/当地土特产 |
| `depth` | string | 选填，缺省 `skeleton` | `skeleton` / `verified` / `full`。**地址查不到 / 营业时间全靠类目均值 = 自动降 skeleton**，不能挂 full |

**全砍掉的字段**（对照正餐池）：
- `tier` — 停留点不分档
- `cuisine` — 用 type 替代
- `recommended_meals` — 不区分餐时段
- `reservation_difficulty` — 99% 不需预约
- `queue_level` — 排队进 note.营业 自由文本即可
- `closed_days` — 进 note.营业 自由文本即可
- `season_months` — 极少数有，进 note.营业 自由文本即可
- `price_cny` — 停留点价格无装配意义（用户不为价格选）

### note 字段（5 个）

| key | 必填 | 说明 |
|---|---|---|
| `店名` | ✅ | 中文（日文/英文）。简单直白，不强求注音 |
| `简介` | full 必填 | 60-120 字。"懂当地的朋友讲口吻"，含 1-2 个具体细节（招牌品/位置/氛围）即可 |
| `亮点` | full 必填 | 2-4 个短词，最招牌的特征 |
| `地址` | full 必填 | 含最近交通+步行 X 分钟。**地址查不到不强填——降 depth=skeleton** |
| `营业` | full 必填 | 一句话含开门时段+定休+排队/季节/特殊提醒。**查不到具体时段写"营业时段以官网为准"，禁止用类目均值填假数据**（如全部停留点写"通常 11:00-21:00"=数据漂移）。所有时间敏感信息都在这一句，不再拆字段 |

**全砍掉的字段**（对照正餐池 note）：
- `招牌菜` — 招牌进 `亮点` 字段
- `预约` — 不需要
- `到店提醒` — 合并进 `营业` 字段
- `Tabelog 分数` — 停留点不看 Tabelog

### 装配逻辑

- **触发位置**：每天动线 2-3 个 slot（上午景点结束 / 午后 / 傍晚回酒店前）
- **匹配规则**：按 `near_attractions.entity_id` 找，步行 ≤8 分钟，type 匹配用户偏好
- **不强制**：用户表单选偏好（"想喝咖啡 / 想逛古着 / 想看书店 / 都不需要"），可空
- **停留时长按 type 默认**（装配端硬编码）：咖啡甜品 20-30 分 / 古着杂货 30-45 分 / 书店 45-60 分

### 数据迁移来源

**大阪 stops 直接迁现成数据**：`japan/kansai/_archive/legacy_pre_d40/v1_content/osaka/shops.json`（23 家）。
- 字段映射：`shop_type_tag` → `type`（vintage_clothing→古着 / bookstore_cafe→书店咖啡 / kitchenware→道具屋 / tax_free_drugstore→当地土特产 等）
- `editor_note` → `note.简介`（不够 60 字补一句即可）
- `vibe_tags` → `note.亮点`
- 不要重新搜索补充——直接迁，省 token
