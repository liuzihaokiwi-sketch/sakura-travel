# 行程方案写作规范（D40·2026-04-24）

> 方案作者读这份 md 写 N 日预制行程。
> 消费方：装配引擎（读方案执行）+ 方案作者自律（跨日组合硬规）。
> 关联：[assembly/engine.md](../assembly/engine.md)（装配引擎行为）+ 各动线 [动线说明.md](../templates/)（查动线定位）。

## 一、方案是什么

**方案 = N 日骨架 + 每天的槽位（动线级或模板级）**。

用户输入（日期+天数+人群+偏好）→ 匹配一套预制方案 → 方案按天给装配引擎指令：

- **写死型槽位**：`D3: kyoto_arashiyama_5`（直接指定模板 ID）
- **动线级槽位**：`D3: 岚山动线（按季节自动挑变体）`
- **候选列表槽位**：`D3: [岚山动线, 东山动线, 北山动线]·情侣优先岚山·家庭优先东山`

**装配粒度由方案作者决定**——预制到动线层还是模板层都行。

---

## 二、装配两层选择

### 第 1 层：选动线
- 动线 = `templates/{city}/{动线}/` 目录
- 装配层从候选列表挑一条动线（按季节/人群/偏好）

### 第 2 层：选模板
- 选中动线后·从目录下的 JSON 变体挑一个（按季节/pace_type/time_sensitivity）

**预制可以停在任一层**：
- 停在第 1 层 = 方案写「岚山动线」·引擎按季节挑变体
- 停在第 2 层 = 方案写「kyoto_arashiyama_5」·引擎直接装

## 三、关西 7 条化学反应（写方案必守）

> 复刻自 [产品原则.md](../产品原则.md) §12。方案作者跨日组合时必须遵守。

1. **先京都后大阪** —— 状态好时沉浸京都·末尾切换大阪。例外：特种兵档反向（大阪恢复时差）
2. **奈良永远作为「桥」** —— 奈良→大阪 JR 直通 50 分钟·不独立过夜·顺路塞
3. **姬路+神户一天组合** —— 姬路上午、神户下午·铁路顺路
4. **宇治不能和姬路/奈良同一天** —— 宇治要慢·跟大景点组合变走马观花
5. **京都-大阪不需要过渡缓冲** —— 15 分钟新干线·切换快反而产生对比。例外：沉浸式町家切换大阪商业区·京都吃完晚饭再过去
6. **京都超过 5 天切日归模式** —— 第 4-5 天安排奈良/宇治/嵯峨保持新鲜
7. **神户 2 晚 + 有马温泉** —— 第 1 天港口/牛肉/灘酒造·第 2 天下午有马

## 四、pace_type 候选库（手工维护·装配引擎权威源）

> **D42 硬规（2026-04-25）**：下面 §4.1 + §4.2 两张白名单表是**装配引擎读取的唯一权威源**。
> 模板 JSON 不再自报 `pace_type` / `pace_type_sub` / `time_sensitivity` 等节奏标签（D42 弃用）。
> 装配引擎读方案层白名单 + 目录位置 + slots 第一段时间三个信号推断模板节奏——节奏归属是**全局决策**·不是模板可以单方面自报的属性。
> 维护方式：方案作者手工维护本节两表·增删候选必须在此修改·不由模板 JSON 决定。

### 4.1 fixed_early 候选（全关西 5 个）

**四要素硬规**（缺一不做）：
- **A 差异化灵魂**：早晚体验到「灵魂不同」级·不是「人少点」
- **B 晨光独家**：6:30-8:00 晨光窗口不可替代
- **C 起得来**：用户 6:30-7:00 起床可达
- **D 市区短交通**：住酒店 15-30 分钟内·拍完能回酒店睡回笼

| kyoto_higashiyama_X 清水独享·樱花版 | 3/28-4/5 | 7:00 前清水舞台樱花无人+石塀小路空镜 | ⏳ 待建 |
| kyoto_higashiyama_X 清水独享·红叶版 | 11/15-12/5 | 7:00 前清水红叶+子安塔背景无人 | ⏳ 待建 |

**方案作者写作硬规**：
- **一次行程 ≤ 1 个 fixed_early**（用户必须明确勾选「愿意为出片早起一次」才入池）
- **装配优先级**：deep_stay > fixed_early > adaptive（两日深度覆盖出片体验的不另叠 fixed_early）
- **远郊永不合格**（D39 硬规）：岚山（除 5 号例外）/ 高雄 / 吉野山 / 高野山 / 金阁寺 / 天龙寺 / 多数塔头寺院

### 4.3 adaptive（默认）

除上面 12 个模板外·全关西所有变体归 **adaptive 默认档**（按用户 8/9/10 出门档自动平移 slots）。这包括：
- 所有带时间约束的模板（小火车班次 / 抽签时刻 / 祭典 / 夜樱 / 日出机位）→ 用 `time_sensitivity=soft/hard` 字段表达
- 所有远郊动线（鞍马贵船 / 宇治 / 高雄 / 吉野山 / 奈良 / 神户等）
- 所有半日 / 特殊事件 / 到达离开日

## 五、槽位优先级设计思路

方案每个槽位的候选列表·作者**手工排优先级**（候选列表顺序 = 优先级）。排序维度：

1. **季节命中度**（樱花季优先装樱花变体）
2. **人群匹配度**（情侣优先氛围版·家庭优先互动度高的）
3. **执行风险**（旅馆预约难 / 班次稀缺 / 天气敏感的往后排）
4. **用户勾选（selectable_tag 等效）**：用户勾了「想泡温泉」→ 温泉变体上位

**不用数字打分公式**——作者在候选列表顺序里直接体现·自然语言注释补充。

## 六、跨日组合内容防重复

方案作者一次性看到 D1-DN 所有天·**在写方案时就避免内容重复**：

- 例：有马+神户 2day 组合·D1 神户走「港口线」（港口+元町+神户牛）·D3 再装神户不重复·或改装姬路
- 例：樱花季用户装「岚山 10 两日深度」·D2 清晨已有出片段·**不再额外塞 fixed_early 岚山 5**
- 例：京都 5 日用户·东山日+岚山日分开·但不同变体（东山和服日 + 岚山氛围日）避免"两天都在拍照"审美疲劳

**装配引擎不检查跨日重复**——作者写方案时保证。

## 七、命名约定

- 方案文件名：`{天数}日_{主题}.md`·例：`7日_京都深度情侣.md` / `9日_京都大阪均衡家庭.md`
- 方案内部引用模板：用 `template_id`（如 `kyoto_arashiyama_5`）
- 方案内部引用动线：用目录相对路径（如 `kyoto/arashiyama/`）

### D5 （承 D4）

### D6 大阪核心
- 装 `osk_namba_1`

### D7 离开日
- 装 `departures/1.json`

## 反射+应急
- D3 fixed_early 触发失败（起晚/下雨）→ contingencies.late_start·当天退 `kyoto_arashiyama_1` 标准日
- D4-D5 温泉旅馆预约失败 → 整行程切换「6 日纯京都」备用方案

## 九、常见错误

1. ❌ 一次行程塞 2 个 fixed_early（D39 硬规：≤ 1 个）
2. ❌ 在 fixed_early 远郊候选里塞岚山 9/10（岚山 9/10 是 deep_stay·不是 fixed_early）
3. ❌ 方案直接装 arm_arima_4（已删·跨城组合由方案骨架拼）
4. ❌ 宇治和姬路同一天（关西 7 条 §4）
5. ❌ 候选列表顺序随便排（作者必须按季节+人群+风险手工排优先级）
