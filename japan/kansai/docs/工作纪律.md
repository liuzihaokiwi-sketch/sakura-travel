# 迭代纪律

> 多窗口并行做事时怎么同步规则。避免"每个窗口都要单独叮嘱一次新规则"。

---

## 一、唯一权威源是 plan 文件

所有规则、字段定稿、铁律 → 写进 `plans/餐厅重构.md` 或对应 plan。
**不要在提示词里复述规则**——提示词只说"读 plan"+ 本轮具体任务。

新窗口开工提示词模板：
```
读 plans/餐厅重构.md（特别看 §〇·〇 铁律）。
本轮任务：{具体做什么}。
```

---

## 二、规则更新流程

发现新铁律 / 字段调整 / 边界 case → **先更新 plan，再继续派窗口**。

顺序：
1. 当前窗口反馈问题
2. 我（审核员）判断是不是普适规则
3. 是 → 改 plan 顶部铁律段（或对应字段表）
4. 改完 → 通知所有进行中的窗口"plan 已更新到 X，下轮自动读到"
5. 不是普适规则 → 只在当轮提示词里处理，plan 不动

---

## 三、不在提示词里堆历史

❌ 反例：
```
读 plan。注意：
- 不要 walk_min_estimated（之前大阪犯过）
- 营业不要类目均值（之前难波犯过）
- entity 不为挂店凑（之前高野山犯过）
- ... 列 10 条 ...
```

✅ 正例：
```
读 plan §〇·〇 铁律。本轮做河原町。
```

历史教训 → 沉淀到 plan 铁律。提示词保持最简。

---

## 四、判断"是否该进 plan 铁律"

进 plan 铁律的标准：
- 多窗口都可能犯
- 一犯就是 A/B 类（装配跑不动 / 用户体验硬伤）
- 不是单店的特殊情况

只在当轮提示词处理：
- 单店具体怎么挂 entity
- 某个数据来源用哪个
- 当轮特定区域的拆分方案

---

## 五、窗口之间的"基础设施"同步

某个窗口加了 entity / 改了 schema / 升级了校验脚本——这些**全局变更**要让其他窗口知道：

- entity 加在 entities/ → 其他窗口下轮 grep 自然能找到，不用通知
- validate_restaurants.py 升级 → 其他窗口跑校验时自然适用，不用通知
- plan 改了 → 通知"plan 已更新到 X 段，下轮读"

**不需要的同步**：进度（哪个窗口做到哪了），各窗口独立。

---

## 六、收尾

每轮通过后，审核员（我）检查：
- 本轮反馈的"教训"是否需要进 plan 铁律？
- 如果是 → 改完再派下一轮
- 如果否 → 继续

这样 plan 不会膨胀（只放"反复犯的"），提示词不会臃肿（只说本轮任务）。

完。
