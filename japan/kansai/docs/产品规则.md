# 关西产品规则

> 从设计讨论中沉淀的产品决策规则，供装配系统和内容创作参考。
> 最后更新：2026-04-20

---

## 一、行程天数规则

### 有效游览天数
**有效全天 = 总天数 - 3**
- 到达日：下午才能开始，半天
- 离开日：上午赶机场，半天
- 换城日（京都↔大阪）：换城消耗半天，另外半天轻量
- 三天合计扣除，剩余才是真正能全力游览的天数

| 总天数 | 有效全天 |
|---|---|
| 5天 | 2天 |
| 6天 | 3天 |
| 7天 | 4天 |
| 8天 | 5天 |
| 9天 | 6天 |
| 10天 | 7天 |
| 11天 | 8天 |
| 12天 | 9天 |

### 城市分配原则
- 京都+大阪是默认，不让用户选"去不去"
- 系统根据总天数自动分配京都/大阪比例
- 插件城市（奈良/神户/有马等）根据天数和用户选择自动插入
- 纯大阪不去京都的用户走客服定制线，不在产品范围内

## 二、用户可选体验分类

### A类：核心型（用户可能为它而来）
用户主动选择即开放，提示取舍，不设严格天数门槛。

| 标签 | 模板 | 最早触发 | 提示语 |
|---|---|---|---|
| 环球影城 | USJ | 6天总天数 | 选了USJ会占大阪1天，京都相应减少 |
| 温泉一泊 | 有马/岚山温泉 | 6天总天数 | 选了温泉占2天，需取舍京都主线 |
| 有马日归 | 有马温泉半日 | 6天总天数 | 接在某天下午，不独占全天 |

### B类：随时可加型（不占白天/替换型）
任何天数都可开放。

| 标签 | 模板 | 说明 |
|---|---|---|
| 和服体验 | 东山出片日 | 替换东山精华日，不额外占天 |
| 亲子活动 | 铁道博物馆/海游馆 | 替换大阪某天 |
| 酒吧/夜パフェ/鸭川夜饮 | 餐厅池夜间类 | 餐后顺路，不占白天 |

### C类：锦上添花型（需要行程有真正余裕）
按天数严格控制。

| 标签 | 模板 | 最早触发 | 有效全天 |
|---|---|---|---|
| 夜景体验 | 生驹山游乐园 | 8天 | 5天 |
| 手作体验 | 京都工艺 | 9天 | 6天 |
| 数字艺术 | teamLab | 9天 | 6天 |
| 任天堂博物馆 | 任天堂 | 10天 | 7天 |
| GEAR齿轮秀 | GEAR | 10天 | 7天 |
| 舞妓お座敷 | 舞妓 | 10天 | 7天 |
| 太阳塔 | 万博纪念公园 | 12天 | 9天 |
| 高野山 | 高野山一泊 | 13天 | 10天 |

## 三、模板分类规则

### 装配方式
- **自动装配**：主线动线（东山/岚山/金阁/伏见/冈崎/宇治/大阪精华等），装配系统根据天数和城市自动选择
- **用户勾选触发**：selectable_tag 不为 null 的模板，需要用户在表单主动选择
- **季节触发**：season_trigger 字段限制，只在特定季节装配

### 模板不处理的事（由装配系统上层决定）
- **航班/换城交通** — 精华日模板不考虑航班，装配系统保证精华日不会和到达日/离开日排在同一天。到达日/离开日有专属模板处理这件事
- **assembly_hints.not_for** — 精华日模板不写"早航班不适合"之类的表述，那是装配层的判断

### 变种规则
同一动线视为"一族"（基础版 + 季节版 + 人群变种 + 季节×人群变种），装配时从这一族里选最高优先级的一个分支，各分支之间是条件替换关系，不比分数。

**装配优先级（一族内部）：**
```
季节×人群变种 > 季节版 > 人群变种 > 基础版
```
例：用户是情侣 + 樱花季到岚山：
- 有`kyo_arashiyama_sakura_atmosphere_full` → 用它
- 只有`kyo_arashiyama_sakura_full` → 用季节版
- 没有季节版 → 检查有无`kyo_arashiyama_atmosphere_full`
- 都没有 → 用`kyo_arashiyama_core_full`基础版

**变种类型（3种）核心定义：**

**`atmosphere`（氛围型）**
- 本质：体验感受优先于视觉记录——重在"被那个场景包围的感觉"，不是"拍出来好看"
- 典型行为：夕阳下坐着不做什么、特定仪式感餐厅、温泉留白、慢节奏散步
- 触发条件：情侣人群，或用户表单选"注重氛围/仪式感"
- 朋友做会尴尬，亲子做会无聊——是真正的人群区隔
- 节奏特征：比主模板慢，有意留白，不赶景点数量

**`photo`（出片型）**
- 本质：视觉/拍照价值是核心，体验价值次要——"想要那张照片"是驱动力
- 两种情况：
  1. 景点本身以视觉为核心，肉眼体验平淡（蹴上インクライン/某构图点），不出片的人去了会觉得"就这？"
  2. 景点有出片专属时间窗口，需要比普通游览早1-2小时（如渡月桥6:30 vs 8:00），动线顺序也完全不同
- 触发条件：用户表单选"出片/拍照"，或带相机/有明显拍照需求
- 不拍照的用户：快速过/缩短停留/跳过

**何时做变种 vs 只加字段：**
- 出片只需要换几个tips → 直接加`photo_spots`字段到主模板，不做变种
- 出片需要完全不同出发时间/动线顺序 → 做`photo`变种
- 氛围影响>30%动线（节奏/景点都换）→ 做`atmosphere`变种，否则加`couple_note`字段
- 亲子有>2个slot需要替换 → 做`family`变种

**`family`（亲子型）**
- 避开爬坡/长队/高门票/成人氛围场所
- 增加儿童友好元素（草坪/动物/互动体验）
- 触发条件：用户人群含儿童

**命名规则：** `{城市}_{地点}_{变种}_{时长}`
- 时长：`full`/`half`/`2day`/`night`/`special`
- 变种按体验诉求命名，不按人群（不用couple/friends）
- 密度（packed/balanced/relaxed）控制景点数量，不做 active 变种

### 不做模板的情况
- 二刷用户 → 客服定制线
- 纯大阪不去京都 → 客服定制线
- Culture/Shopping/Night专题日 → 表单选项插入，不做专属模板

## 四、季节规则

### 温泉季节限制
温泉体验只在以下季节推荐：
- `early_spring`（早春梅花季）
- `sakura`（樱花季）
- `koyo`（红叶季）
- `deep_winter`（深冬）

夏天/梅雨季不触发（高温泡汤体验差）。

字段：`"season_trigger": ["early_spring", "sakura", "koyo", "deep_winter"]`

### 季节模板与 common 的关系
- 季节对动线影响 < 30% → 放 common，加 season_notes 字段
- 季节对动线影响 > 30% → 独立季节模板
- 季节独特但仅1-3天触发 → `template_kind: special_date`

## 五、评分规则

### core_experience（0-60分）
反映通用体验质量，不含人群加成。这是基准分，人群乘数在此基础上浮动。

| 分段 | 含义 | 典型模板 |
|---|---|---|
| 50-60 | 关西最标志性，不可替代 | 东山精华/岚山sakura/祇园祭/金阁雪景 |
| 40-49 | 主线必选，高频装配 | 岚山core/金阁/伏见/宇治/奈良/USJ/有马 |
| 30-39 | 次主线，特定偏好触发 | 中之岛/新世界/住吉/工艺/teamLab |
| 20-29 | 功能型/深度小众 | 到达日/离开日/大原/铁道博物馆 |

### final_score 计算
```
final_score(人群) = core_experience × (1 + audience_bonus[人群] / 100)
```

- audience_bonus 范围：-15 到 +20
- +15~+20：专为该人群设计的模板（atmosphere之于情侣/舞妓之于情侣）
- +5~+10：该人群显著受益
- +1~+4：轻微偏好
- 0：中性
- -1~-5：轻微不适合
- -10~-15：明显不适合（大原之于亲子/舞妓之于亲子）

**score 参与排名的场景（只有这两种）：**
1. 同一季节内不同动线竞争（岚山 vs 东山 vs 金阁，选哪天去哪）
2. 同一区域有多个模板时选哪个（okazaki_koyo vs nanzenji_koyo）

**score 不参与排名的场景：**
- 变种 vs 主模板 → 条件替换，不比分（用户选了atmosphere就用氛围版，主模板不进候选）
- selectable 模板 vs 普通模板 → 用户选了才进候选，不与普通模板竞争

**装配优先级：**
1. 季节模板 > common 模板（有樱花版用樱花版，没有才回退common）
2. 变种替换主模板（用户选了诉求就替换，不并列）
3. selectable 独立筛入（用户选了才加入候选池）

### 打分新规则（两层分离）

**第一层：selectable 模板独立处理**
- `selectable_tag != null` 的模板（USJ/温泉/和服等）：用户选了才进候选池，不靠加分竞争
- 这类模板的 audience_bonus 仅用于同类模板间排序（如两个温泉模板互相比较），不与普通模板竞争
- final_score 计算对 selectable 模板意义不大，装配系统直接按 selectable_tag 筛入

**第二层：普通模板用乘数控制人群影响**
- `final_score = core_experience × (1 + audience_multiplier)`
- `audience_multiplier` 范围：-0.15 到 +0.20
- 换算关系：audience_bonus 字段值 ÷ 100 = multiplier（保持字段格式不变，理解方式改变）
- **效果**：core 高的模板人群加成绝对值更大，差距被保留
  - core 50 × 1.20 = 60（情侣强偏好）
  - core 30 × 1.20 = 36（情侣强偏好）
  - 差距从 20 保持到 24，不会被加分拉平

**audience_bonus 字段新含义（-15 到 +20）**
- +15~+20：这类模板专门为该人群设计（氛围日之于情侣）
- +5~+10：该人群会显著受益
- +1~+4：轻微偏好
- 0：中性
- -1~-5：轻微不适合
- -10~-15：明显不适合该人群（儿童去大原深度游）

### execution_risk（负数）
- 0：无风险
- -1：需提前预约或有轻微不确定性
- -2：强依赖特定条件（天气/日期/人流）
- -3：极高不确定性（降雪触发/限定日期）

## 六、二刷与首次用户

**产品只服务首次来关西的用户。**

- 二刷用户 → 联系客服安排定制行程
- 模板设计不考虑"已经来过"的场景
- 看似"二刷才会去"的模板（如大原/大德寺），定位为"7天以上首次用户"或"对X有特定兴趣的用户"，不以二刷为前提
