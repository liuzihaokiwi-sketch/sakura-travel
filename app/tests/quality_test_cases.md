# 行程质量测试用例

> 核心问题：我们系统生成的行程，是否比用户直接问 ChatGPT 要好？
> 如果不好，系统就没有存在的价值。
>
> 这些测试用例是持久复用的——每次改动后都要重跑，确认质量没退步。

---

## 测试方法

对同一个需求，生成两个行程：
- **A组**：我们系统生成的（generate_trip 或模板）
- **B组**：直接把相同需求发给 ChatGPT/Claude 生成的

然后按以下维度逐项打分（1-5分），看哪个更好。

---

## 评分维度

### 1. 信息准确性（权重最高 — 手账本的命根子）
```
1.1 地点是否真实存在（Google Maps 验证）
1.2 地址/坐标是否正确
1.3 营业时间是否正确
1.4 价格是否大致准确（误差 <30%）
1.5 定休日是否标注
1.6 交通方式和时间是否正确
1.7 评分是否来自权威源（不是编的）
```

### 2. 路线合理性
```
2.1 地理顺序是否合理（不来回折腾）
2.2 交通衔接是否可行（末班车、换乘）
2.3 时间分配是否合理（不赶不闲）
2.4 用餐时间是否正常（12:00 午餐、18:00 晚餐左右）
2.5 第一天/最后一天是否考虑了到达/离开时间
```

### 3. 内容丰富度
```
3.1 每天活动数量是否充实（4-6 个活动+餐厅）
3.2 是否有具体的实用提醒（"只收现金"、"需预约"）
3.3 是否有招牌菜/核心体验推荐（"味噌拉面必点"）
3.4 是否有价格信息
3.5 是否覆盖了该城市的特色（去札幌没拉面？）
```

### 4. 个性化匹配
```
4.1 是否符合用户的人群类型（情侣/家庭/独行）
4.2 是否符合预算级别
4.3 是否符合兴趣标签（美食/自然/温泉）
4.4 是否考虑了季节因素
4.5 是否避开了用户不要的东西
```

### 5. 惊喜感
```
5.1 有没有"攻略上找不到"的本地推荐
5.2 有没有顺路的特色小店推荐
5.3 有没有超出预期的细节（"这个角度拍照最好"）
5.4 整体是否让人想"这个值298元"
```

## 测试用例

### 用例 TC-01：札幌情侣 5 天（基础场景）
```yaml
场景: 最常见的用户画像
输入:
  destination: 北海道
  cities: [sapporo, otaru, hakodate]
  start_date: 2026-05-10
  end_date: 2026-05-14
  party_type: couple
  party_size: 2
  budget_level: mid
  interests: [food, nature, onsen]
  pace: moderate

期望:
  - 5 天行程，每天 4-6 个活动
  - 必须包含：札幌拉面、小樽寿司、函馆夜景、温泉
  - 城市间交通用 JR
  - 每天有午餐和晚餐安排
  - 酒店在 mid 档位
  - 5月天气舒适，以户外活动为主

检查重点:
  - 札幌的特色美食（味噌拉面/成吉思汗/汤咖喱/海鲜丼）是否至少覆盖 2 种
  - 小樽是否安排了运河+寿司
  - 函馆夜景是否在 Day 1 或 Day 5（到达/离开日通常在函馆）
  - 定山溪温泉或登别温泉是否包含
```

### 用例 TC-02：家庭带幼儿 5 天（约束场景）
```yaml
场景: 有特殊约束的用户
输入:
  destination: 北海道
  cities: [sapporo, otaru]
  start_date: 2026-07-20
  end_date: 2026-07-24
  party_type: family_child
  party_size: 4
  budget_level: mid
  interests: [nature, food]
  pace: relaxed
  special: 有3岁小孩，不能走太多路

期望:
  - 节奏慢，每天 3-4 个活动
  - 必须排除体力要求高的景点（登山、长距离步道）
  - 必须包含亲子景点（旭山动物园/白色恋人工厂/圆山动物园）
  - 7月是夏季，薰衣草季节应包含富良野
  - 餐厅要 family_friendly
  - 酒店要 family_friendly（有家庭房）

检查重点:
  - 有没有安排不适合幼儿的活动（登山、酒吧）
  - 每天步行量是否合理
  - 是否有亲子活动
  - 用餐地点是否适合带小孩
```

### 用例 TC-03：冬季滑雪+温泉 4 天（季节场景）
```yaml
场景: 季节性需求
输入:
  destination: 北海道
  cities: [sapporo, niseko]
  start_date: 2026-02-05
  end_date: 2026-02-08
  party_type: couple
  party_size: 2
  budget_level: premium
  interests: [ski, onsen, food]
  pace: moderate

期望:
  - 包含滑雪（二世谷或札幌周边雪场）
  - 包含温泉（登别/定山溪/二世谷温泉）
  - 2月初可能撞上札幌雪祭 → 应该提及
  - 冬季路面积雪 → 交通建议考虑
  - premium 预算 → 酒店推荐温泉旅馆
  - 冬季日照短（16:00 天黑）→ 户外活动集中在上午

检查重点:
  - 滑雪和温泉是否都有
  - 是否提到雪祭（2月初）
  - 户外活动是否集中在日照时段
  - 是否有冬季交通提醒（积雪/减班）
```

### 用例 TC-04：周一出发（定休日场景）
```yaml
场景: 测试定休日处理能力
输入:
  destination: 北海道
  cities: [sapporo, otaru]
  start_date: 2026-05-11  # 周一
  end_date: 2026-05-13
  party_type: solo
  party_size: 1
  budget_level: budget
  interests: [food, culture]
  pace: packed

期望:
  - 不安排周一定休的餐厅
  - 不安排周二定休的景点（部分博物馆周二闭馆）
  - budget 预算 → 推荐经济型酒店和平价美食
  - solo + packed → 每天 5-6 个活动
  - 短行程 3 天 → 紧凑但不遗漏核心

检查重点:
  - 行程中的餐厅是否在对应日期营业
  - 是否标注了定休日提醒
  - budget 预算是否体现在酒店和餐厅选择上
```

### 用例 TC-05：纯美食深度游 3 天（主题场景）
```yaml
场景: 单一主题深度
输入:
  destination: 北海道
  cities: [sapporo]
  start_date: 2026-09-15
  end_date: 2026-09-17
  party_type: group
  party_size: 3
  budget_level: premium
  interests: [food]
  pace: relaxed
  special: 就是来吃的，景点随便看看

期望:
  - 每天 3-4 家餐厅（不是景点为主）
  - 覆盖札幌特色：味噌拉面、成吉思汗、汤咖喱、海鲜丼、寿司
  - 9月是秋季 → 秋季限定美食（秋�的路、新米）
  - premium → 至少 1 家高档寿司或怀石
  - 景点只安排 1-2 个"顺路的"
  - Tabelog 3.5+ 的餐厅为主

检查重点:
  - 餐厅数量是否多于景点
  - 菜系是否多样（不全是拉面）
  - 是否有 Tabelog 高分餐厅
  - 是否提到招牌菜和推荐点单
```

### 用例 TC-06：对比测试 — 同一需求 AI 直接生成
```yaml
场景: 基准对比
方法:
  1. 取 TC-01 的需求
  2. 直接发给 ChatGPT："帮我规划一个北海道5天情侣旅行，5月10-14日，预算中档，喜欢美食/自然/温泉"
  3. 对比两个行程

对比维度:
  - ChatGPT 会给出真实的 Tabelog 评分吗？（大概率不会或编的）
  - ChatGPT 会标注定休日吗？（不会）
  - ChatGPT 会给出具体价格吗？（可能编）
  - ChatGPT 会考虑地理顺序吗？（粗略）
  - ChatGPT 会给出交通方式和时间吗？（粗略或编的）
  - ChatGPT 会推荐"本地人才知道的店"吗？（大概率给热门景点）

这个对比是核心——如果我们在这些维度上没有明显优势，系统就没有价值。
```
