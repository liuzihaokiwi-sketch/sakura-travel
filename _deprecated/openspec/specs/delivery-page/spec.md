## 系统需求

### 需求：交付页模块顺序
交付页 `/plan/[id]` SHALL 按以下顺序展示行程：

1. **封面卡片** — 城市名+日期+同行人+旅行风格标签，杂志封面感
2. **行程总览** — 7天概要时间轴，每天一个关键词+图标
3. **每日详情**（核心） — 每天一个 section：
   - 日期+主题标题（如"Day 3 · 新宿御苑 — 樱花海里的慢时光"）
   - 时间轴式景点列表（时间+地点+为什么去+停留建议）
   - 餐厅推荐（店名+类型+人均+一句推荐理由，不提"评分系统"）
   - 当日交通小结
   - 侧边栏："今日拍照提示""今日避坑提醒"
4. **住宿建议** — 推荐区域+为什么选这里+预算参考
5. **出行准备清单** — 可勾选的 checklist
6. **行程精调入口** — "想调整？点这里"
7. **分享模块** — "觉得有用？分享给一起去的朋友"

### 需求：推荐理由写法
所有推荐理由 SHALL 用"专业判断"口吻，不用"AI拼接"口吻：
- ✅ "选这家是因为它在上野公园步行5分钟内，午餐高峰前去不用排队"
- ✅ "这个时段去新宿御苑，光线最适合拍樱花隧道"
- ❌ "根据评分系统推荐" "AI分析后推荐"
- ❌ "该餐厅在Tabelog评分4.2" → 改为 "本地人评价很高的XX料理"

### 需求：交付页品质标准
- 视觉风格：旅行杂志感（大图+优雅排版+暖色调）
- 每个景点至少配一张高品质图
- 移动端优先，可长图分享
- 有"导出为图片"按钮（用于分享朋友圈）

## A29. 默认展示 vs 折叠模块划分

### 默认展开（进入页面立即可见）

| 模块 | 原因 |
|---|---|
| 封面卡片 | 仪式感第一印象 |
| 行程总览条 | 快速导航 |
| 每日详情 — 当天主题标题 | 让用户知道每天在做什么 |
| 每日详情 — 时间轴景点列表 | 核心交付物，必须直接看到 |
| 每日详情 — 餐厅推荐 | 高感知价值，直接展示 |
| 行程精调入口 | 不能让用户找不到 |
| 分享模块 | 底部自然露出 |

### 默认折叠（点击展开）

| 模块 | 折叠理由 | 展开触发 |
|---|---|---|
| 每日拍照提示 | 非核心信息，感兴趣的人点开 | "📸 查看拍照提示" 按钮 |
| 每日避坑提醒 | 辅助信息 | "⚠️ 查看今日注意事项" 按钮 |
| 每日交通小结 | 细节信息，需要时再看 | "🚃 查看交通指引" 按钮 |
| 住宿建议 | 完整区块较长 | 标题可见，内容折叠 |
| 出行准备清单 | 工具性质，非阅读主线 | 标题可见，内容折叠 |
| PlanB 备选方案 | 非主线，备用 | "查看备选方案" 按钮 |
| 升级提示条 | 轻量展示即可 | 始终展示为一行条幅，不折叠 |

### 折叠交互规范
- 折叠按钮统一样式：圆角胶囊按钮，浅石色背景，带小箭头 `▾`
- 展开动画：Framer Motion `AnimatePresence` + 高度过渡（200ms ease-out）
- 折叠状态下仍保留模块标题（让用户知道有这个内容）

## A30. PlanB 设计逻辑 + 交付页内升级入口（888）

### PlanB 设计逻辑

**定义**：PlanB 不是"第二套方案"，是"主方案的弹性补充"。

**触发场景 → 备选内容**：

| 场景 | PlanB 内容 | 展示方式 |
|---|---|---|
| 下雨天 | 室内替代景点（博物馆、商场、室内体验） | "🌧️ 雨天备选" 标签 |
| 景点临时关闭 | 同区域步行可达的替代 | "如果XX关了 → 去YY" |
| 人太多/排队太长 | 错峰建议 or 周边替代 | "人多时 → 先去ZZ，下午再回来" |
| 体力不够 | 当天可砍掉的景点 + 休息推荐 | "累了就跳过XX，在YY咖啡店休息" |

**PlanB 数量**：
- 整个行程提供 5-8 个 PlanB 条目
- 不是每天都有，只在高风险日/热门景点日提供
- 每个 PlanB 条目必须包含：触发条件 + 替代方案 + 怎么去

**PlanB 卡片设计**：
```
┌─────────────────────────────────────┐
│  🌧️ Day 3 雨天备选                   │
│                                     │
│  如果新宿御苑因雨不想去：              │
│  → 新宿 NEWoMan 商场（步行5分钟）     │
│    可以逛2小时，地下一层有很好的面包店   │
│                                     │
│  → 东京都厅展望台（免费，步行10分钟）   │
│    45层看东京全景，雨天反而有城市朦胧感  │
└─────────────────────────────────────┘
```

### 升级入口（888）设计

**原则**：升级入口是"自然存在"，不是"弹窗逼单"。

**入口位置与形式**：

| 位置 | 形式 | 文案 |
|---|---|---|
| 交付页底部（模块8） | 浅色条幅，一行文字+箭头 | "需要更深度的服务？了解管家版 →" |
| 精调次数用完时 | 页内提示卡 | "精调次数已用完，升级管家版可再享额外精调 + 专属顾问" |
| PlanB 模块底部 | 小字链接 | "想要更详细的备选方案？管家版包含完整应急预案" |

**绝对不做的事**：
- ❌ 不弹窗
- ❌ 不在阅读主线中间插入大面积升级广告
- ❌ 不用"您的方案不完整"这种恐吓式文案
- ❌ 不在首次打开交付页时就强调升级

**升级页跳转**：所有入口统一跳转到 `/plan/[id]/upgrade`，该页面已实现。

## A31. 交付页视觉风格规范

### 整体调性：旅行杂志感

| 维度 | 规范 | 示例 |
|---|---|---|
| **色调** | 暖色主调：米白底(#FDFBF7) + 暖橙点缀(warm-300) + 樱粉高亮(sakura-400) | 背景不用纯白，用暖白 |
| **字体** | 标题：Playfair Display + Noto Serif SC；正文：system-ui + Noto Sans SC | 中英混排时英文用衬线体 |
| **图片** | 每个景点配 1 张高质量图（16:9 或 4:3），圆角 12px，带微阴影 | Unsplash 日本旅行类图片 |
| **间距** | 模块间距 32px，卡片内边距 24px，行距 1.75 | 留白充足，不拥挤 |
| **卡片** | 白色卡片，圆角 16px，shadow-sm，不用边框 | 浮在暖白底色上 |
| **图标** | Emoji 为主（🌸⛩️🍜🚃），不用 icon 库 | 亲切感 > 专业感 |

### 移动端优先规范

| 规则 | 具体要求 |
|---|---|
| 设计基准 | 375px 宽度为基准设计 |
| 图片 | 全宽显示（`w-full`），高度固定比例 |
| 卡片 | 左右留 16px padding，不用左右分栏 |
| 字号 | 标题 20-24px，正文 14-15px，辅助文字 12px |
| 点击区域 | 所有可点击元素最小 44×44px |
| 滚动 | 纵向自然滚动，行程总览条横向滑动 |
| 长图导出 | 支持整页截图为 1080×N 长图（小红书/朋友圈分享） |

### 动效规范

| 元素 | 动效 | 参数 |
|---|---|---|
| 页面进入 | 卡片从下方淡入 stagger | delay: index × 0.08s, duration: 0.5s |
| 折叠展开 | 高度过渡 + 透明度 | duration: 0.2s, ease-out |
| 行程总览卡片 | 横向滑动惯性 | `overflow-x: auto` + `scroll-snap` |
| 按钮 hover | 微缩放 | scale: 1.02, duration: 0.15s |
| 页面切换 | 不用整页转场 | 行内跳转用锚点滚动 |

### 推荐理由写法规范（A21 补充细化）

**语气**：像一个去过很多次日本的朋友在跟你说话。

| ✅ 正确写法 | ❌ 错误写法 |
|---|---|
| "选这家是因为它离上野公园走路5分钟，吃货 aficionado去" | "根据评分系统推荐" |
| "这个时间去人少，光线拍樱花最好看" | "AI分析最佳时间段为9:00-11:00" |
| "本地上班族常去的拉面店，性价比很高" | "Tabelog评分4.2的高分餐厅" |
| "下午留了自由时间，可以逛逛附近的小巷" | "系统检测到疲劳度较高，安排了休息" |
| "这条路线走下来不累，坡道很少" | "根据12维度体力模型计算" |

**绝对禁用词**：AI、算法、评分系统、模型、打分、维度、推荐引擎、数据分析

## A32. 用户端可修改项清单（结构化交互）

修改页 `/plan/[id]/edit` 提供以下结构化修改选项，用户**不能自由输入文字**。

### 可修改项完整清单

| # | 修改项 | 交互方式 | 修改粒度 | 说明 |
|---|---|---|---|---|
| 1 | **替换景点** | 点击景点 → 弹出3个备选卡片 → 选一个 | 单个景点 | 备选从同区域、同类型中选取 |

```

    ↓
看到当天所有景点/餐厅列表
    ↓

    ↓

  · 替换为：[备选A] [备选B] [备选C]
  · 或：[不想去这里]

    ↓
扣减1次精调额度
    ↓
提交 → 系统重新生成 → 审核 → 推送新版本
```

### 不提供的交互
- ❌ 没有文本输入框
- ❌ 没有"其他需求"自由填写
- ❌ 不能改城市、不能改日期、不能改同行人数
- ❌ 不能拖拽调整景点顺序（太复杂，且用户不擅长）

## A33. 不可修改项 + 修改次数限制 + 文案包装

### 不可修改项

| 不可修改项 | 原因 | 用户看到的说明 |
|---|---|---|
| 城市/目的地 | 改了等于重做 | 不展示修改入口 |
| 旅行日期 | 改了等于重做 | 不展示修改入口 |
| 同行人数/类型 | 改了影响全局 | 不展示修改入口 |
| 整体路线骨架 | 改了等于重做 | 不展示修改入口 |
| 推荐理由文案 | 用户不需要改这个 | 不展示修改入口 |

**设计原则**：不可修改的项目直接不展示修改按钮，而非展示后标"不可改"。用户不知道不能改 → 没有挫败感。

### 修改次数策略

| 套餐 | 精调次数 | 文案 |
|---|---|---|
| 行程优化版（¥248） | 2次 | "包含 2 次行程精调" |
| 深度管家版（¥888） | 5次 | "包含 5 次行程精调 + 专属顾问" |

### 次数计算规则
- 每次提交修改页面 = 消耗1次（不管改了几个项目）
- 未提交的修改不计次
- 同一次提交内可以改多个项目（鼓励一次想清楚再提交）

### 文案包装（全程正向表达）

| 场景 | 文案 | 语气 |
|---|---|---|
| 修改页顶部 | "你还有 2 次行程精调机会" | 中性告知 |
| 提交修改时 | "提交后将消耗 1 次精调，确认提交？" | 温和确认 |
| 剩余1次 | "最后 1 次精调机会，建议仔细看看再提交" | 友好提醒 |
| 次数用完 | "精调次数已用完。需要更多调整空间？" + [了解管家版] 按钮 | 自然引导升级 |
| 升级后 | "已升级！你现在有 3 次额外精调机会" | 正向反馈 |

**绝对不用的表达**：
- ❌ "修改次数已达上限"
- ❌ "您的修改权限不足"
- ❌ "免费修改次数已用完，请付费"
- ❌ "限制"、"上限"、"不能"、"禁止"

## A34. 内部审核工作台模块设计（目标5分钟/单）

### 工作台入口

### 单订单审核页 `/admin/order/[id]`

**布局**：桌面端左右分栏（左60% 右40%），移动端上下堆叠。

#### 顶部：订单信息条

```
[待审核 ▾] → [审核通过] → [已发布]    ← 状态下拉切换
```

#### 左侧：方案预览（60%）
- 与用户看到的交付页**完全一致**的渲染
- 只读模式，不可编辑
- 用于快速浏览检查

#### 右侧：编辑面板（40%）

**Tab 1：结构化修改**（主操作区）

| 操作 | 交互 | 说明 |
|---|---|---|
| 选择天数 | Day 1-7 标签 | 选中后显示当天内容 |
| 替换景点 | 当前景点 → 下拉选备选（5个） | 比用户端多2个选项 |
| 替换餐厅 | 当前餐厅 → 下拉选备选（5个） | 比用户端多2个选项 |
| 替换酒店/住宿区域 | 当前推荐 → 下拉选备选 | 住宿是全局的 |
| 调整时间安排 | 拖动时间轴节点 | 微调某景点的开始/结束时间 |
| 删除/添加景点 | 删除按钮 / "添加景点"按钮 | 从备选库选取 |

**Tab 2：提示词补充**（辅助区，非必须）

```
┌──────────────────────────────────────┐
│ 补充指令（可选，用于微调AI生成内容）：    │
│ ┌──────────────────────────────────┐ │
│ │ 例：第3天下午改为购物为主，推荐新宿│ │
│ │ 附近的商场                        │ │
│ └──────────────────────────────────┘ │
│ [重新生成此天]  [重新生成全部]          │
└──────────────────────────────────────┘
```

- 这是给审核人员用的 prompt 输入框
- 用户永远看不到这个区域
- 用于处理结构化按钮搞不定的特殊情况

**Tab 3：历史版本**

| 版本 | 时间 | 操作人 | 变更摘要 |
|---|---|---|---|
| v2 | 3分钟前 | admin | Day 3 替换景点：浅草寺→上野公园 |
| v1 | 2小时前 | system | 初始生成 |

- 可以点击回退到任意版本
- 每次操作自动保存版本

#### 底部：操作按钮栏
```
[保存草稿]  [预览效果]  [发布给用户 ✓]
```

- **保存草稿**：保存修改但不发布
- **预览效果**：在新标签页打开用户视角的交付页预览
- **发布给用户**：状态切换为"已发布"，用户可以看到新版本

### 5分钟审核流程设计

```
0:00  打开订单（从看板点击进入）
      ↓
0:00-1:00  左侧快速浏览方案
      · 扫一眼每天主题是否合理
      · 看看时间轴是否紧凑/宽松适度
      ↓
1:00-3:00  检查关键点
      · 餐厅是否还在营业（系统标注"⚠️ 未验证"的重点查）
      · 景点是否有临时关闭信息
      · 交通衔接是否合理（不要出现30分钟以上的无意义通勤）
      · 推荐理由是否有"AI味"残留
      ↓
3:00-4:00  如需调整
      · 用右侧结构化按钮替换1-2个问题点位
      · 特殊情况用提示词补充框微调
      ↓
4:00-4:30  预览效果（点击"预览效果"快速检查）
      ↓
4:30-5:00  点击"发布给用户"
```

**效率优化设计**：
- 系统自动标注可能有问题的内容（黄色高亮）：新开/已关闭的店、过长通勤、时间冲突
- 一键"看起来没问题"快捷按钮：跳过检查直接发布（适用于重复性高的订单）
- 键盘快捷键：`Cmd+S` 保存，`Cmd+Enter` 发布
