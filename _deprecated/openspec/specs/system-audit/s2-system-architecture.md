# S2. 产品系统结构 — 10层架构

> 产品：日本旅行定制规划（30-40页杂志级攻略）
> 更新：2026-03-21
> 原则：每层职责单一，上层只读下层、不跨层写入

---

## 架构总览

```
L10 数据反馈层        ← 用户行为/NPS/退款原因
L9  发布与排期层      ← 小红书/社群内容日历
L8  内容引流素材层    ← 从方案家族衍生的营销素材
L7  渲染缓存层        ← HTML/PDF/H5 产物缓存
L6  图片资产层        ← 候选图池/精选图集/CDN
L5  用户微调补丁层    ← 用户个性化修改差量
L4  天数扩展层        ← 模板天数裁剪/扩展规则
L3  方案家族层        ← route_templates + scene_variants
L2  快照数据层        ← 酒店报价/航班/天气/开放状态
L1  主档数据层        ← entity_base/pois/hotels/restaurants
```

---

## L1. 主档数据层

| 维度 | 说明 |
|---|---|
| **作用** | 全局唯一事实源：景点/酒店/餐厅的结构化基础信息、标签、编辑标注 |
| **核心数据** | `entity_base`（中心表）+ `pois` / `hotels` / `restaurants`（CTI扩展）+ `entity_tags` + `entity_editor_notes` + `entity_media` + `hotel_area_guide` |
| **已有实现** | ✅ 完整ORM（`app/db/models/catalog.py`），含Google Places / Tabelog外部ID映射 |
| **更新频率** | 低频：新实体入库=爬虫触发；字段修正=人工/定时校验（周级） |
| **长期保存** | ✅ 永久保存，是所有上层的基石 |
| **是否过期** | 部分字段会过期：`opening_hours_json`（景点改营业时间）、`admission_fee_jpy`（涨价）、`typical_price_min_jpy`（酒店基准价波动）。需L2快照层校正 |
| **人工审核** | S级实体100%审核，A级抽检40%，B级自动入库。审核通过`review_jobs`表 |
| **适合模板化** | ✅ 数据结构高度模板化（CTI模式已实现） |
| **适合AI生成** | ❌ 事实数据不应AI生成。唯一例外：`entity_editor_notes.content_zh`的insider_tip类型可AI起草→人工审核 |

### 当前缺口
- `entity_media` 缺图片质量评分字段（横竖比/分辨率/模糊度）
- `entity_base` 缺 `last_verified_at` 字段（数据新鲜度追踪）
- 缺城市级元数据表（`city_meta`：时区/货币/紧急电话/交通卡类型）

## L2. 快照数据层

| 维度 | 说明 |
|---|---|
| **作用** | 捕获随时间变化的动态事实：实时价格、天气、开放状态、航班报价 |
| **核心数据** | `source_snapshots`（原始API存档）+ `hotel_offer_snapshots` / `hotel_offer_lines`（酒店报价）+ `flight_offer_snapshots`（航班）+ `poi_opening_snapshots`（景点开放）+ `weather_snapshots`（天气） |
| **已有实现** | ✅ 完整ORM（`app/db/models/snapshots.py`），含expires_at过期机制 |
| **更新频率** | 高频：天气=每日，酒店报价=每周（热门城市每3天），航班=每周，景点开放=临近出行日期时每日 |
| **长期保存** | ❌ 短期保存。过期快照可归档到冷存储（S3 Glacier），保留最近3个月热数据 |
| **是否过期** | ✅ 必然过期。每条记录含`expires_at`，过期后不再用于方案生成 |
| **人工审核** | ❌ 不需要。数据来自API，异常值由规则检测（如酒店价格波动>50%触发告警） |
| **适合模板化** | ✅ 采集逻辑模板化（爬虫已实现：tabelog/hotel/flight/event/experience） |
| **适合AI生成** | ❌ 绝对不能。事实数据必须来自外部源 |

### 当前缺口
- 缺快照过期自动清理的定时任务
- `hotel_offer_snapshots`缺多平台比价聚合视图
- 缺"快照→主档回填"管道（如景点营业时间变化时自动更新L1）

## L3. 方案家族层

| 维度 | 说明 |
|---|---|
| **作用** | 路线骨架模板库：定义"东京经典5日"等标准路线的日程结构、时间块、区域分配、场景变体 |
| **核心数据** | `route_templates`表（`template_data` JSONB）+ `data/route_templates/*.json`（种子文件）+ `scene_variants`（情侣/家庭/独行变体） |
| **已有实现** | ✅ 8个模板（tokyo_classic_3d/5d/7d, tokyo_sakura_7d, kansai_classic_4d/6d/7d, tokyo_kansai_8d）+ assembler装配引擎 |
| **更新频率** | 低频：新路线上架=产品决策（月级），场景变体调优=运营反馈（周级） |
| **长期保存** | ✅ 永久保存。模板是核心资产，需版本控制 |
| **是否过期** | 不会过期，但需要迭代。season_variants（如樱花季模板）有季节有效期 |
| **人工审核** | ✅ 新模板100%人工审核。已有模板的slot微调可自动化 |
| **适合模板化** | ✅ 本身就是模板（这是"模板的模板"）。结构：meta → days[] → time_blocks[] → slot定义 |
| **适合AI生成** | 部分适合：AI可根据城市POI分布+约束条件起草新路线骨架，但必须人工审核后上架 |

### 方案家族结构

```
route_template (东京经典5日)
├── meta: {city_code, total_days, tagline_zh}
├── scene_variants:
│   ├── couple: {slot_overrides, tag_weight_overrides}
│   ├── family: {filter_exclude_tags: {nightlife: 3}}
│   └── solo: {slot_overrides}
└── days[]:
    └── day: {day_num, theme_zh, city_code, time_blocks[]}
        └── time_block: {slot_id, time, duration_min, tags_required, area_hint}
```

### 当前缺口
- 缺模板版本控制（修改历史）
- 缺"方案家族"间的关系定义（东京5日 是 东京7日 的子集）
- 缺季节性slot覆盖（樱花季某些slot应替换为赏樱点）

## L4. 天数扩展层

| 维度 | 说明 |
|---|---|
| **作用** | 将固定天数模板适配用户实际天数需求：裁剪（7日→5日）或扩展（5日→8日） |
| **核心数据** | `assembler.trim_to_days()` 函数逻辑 + 扩展日模板结构 |
| **已有实现** | ✅ `trim_to_days()`已实现裁剪（保留前N-1天+最后一天收尾）和扩展（追加自由活动日） |
| **更新频率** | 极低：算法逻辑变更=产品决策（季度级） |
| **长期保存** | ✅ 算法逻辑长期保存。扩展日产出为临时数据，随方案生命周期 |
| **是否过期** | 不过期。逻辑是确定性的 |
| **人工审核** | ❌ 不需要逐次审核。算法本身需要验证（QA级别） |
| **适合模板化** | ✅ 高度模板化。扩展日=固定结构（morning + lunch + afternoon三个轻量slot） |
| **适合AI生成** | 部分适合：当前扩展日过于简单（"自由活动日"），AI可根据用户偏好生成更有针对性的扩展日slot |

### 当前缺口
- 扩展日质量低：目前只有通用的"自由活动日"，缺乏城市特色
- 裁剪策略粗糙：直接砍天，未考虑"哪天最重要"
- 缺"天数→模板"智能匹配：用户要6天但只有5日和7日模板时的最优选择

## L5. 用户微调补丁层

| 维度 | 说明 |
|---|---|
| **作用** | 存储用户对标准方案的个性化修改，以差量补丁形式叠加在方案之上，不修改原始模板 |
| **核心数据** | `trip_profiles`（用户画像：party_type/budget/must_have_tags/avoid_tags）+ `trip_versions`（方案版本链）+ 未来新增：`plan_patches`（结构化修改指令） |
| **已有实现** | ⚠️ 部分实现。`trip_profiles`表完整（含must_have_tags/avoid_tags/special_requirements），`trip_versions`可追踪版本。但缺少结构化补丁格式 |
| **更新频率** | 按用户操作触发：提交问卷=1次，付费后追加问卷=1次，修改请求=按SKU限制（免费版0次，标准版2次，深度版5次） |
| **长期保存** | ✅ 保存至订单生命周期结束（交付后90天）。聚合脱敏后用于L10反馈层 |
| **是否过期** | 绑定订单。订单完成后补丁不再被执行，但保留用于分析 |
| **人工审核** | ❌ 轻量修改（换景点/调顺序）不需审核。重大修改（换城市/加天数）触发人工确认 |
| **适合模板化** | ✅ 补丁应为结构化指令，不是自由文本 |
| **适合AI生成** | ❌ 补丁来自用户输入，不由AI生成。AI的角色是执行补丁（重新组装受影响的天） |

### 补丁指令格式设计（建议新增）

```json
// plan_patches 表设计
{
  "patch_id": "uuid",
  "plan_id": "uuid",
  "version": 2,
  "operations": [
    {
      "op": "replace_entity",
      "day": 3,
      "slot_id": "d3_morning",
      "old_entity_id": "xxx",
      "new_entity_id": "yyy",
      "reason": "用户指定想去TeamLab"
    },
    {
      "op": "swap_days",
      "day_a": 2,
      "day_b": 4,
      "reason": "用户想先去京都"
    },
    {
      "op": "add_constraint",
      "type": "avoid_tag",
      "value": "intensive_walking",
      "reason": "用户反馈体力有限"
    }
  ]
}
```

### 当前缺口
- 缺结构化补丁格式定义（目前修改=重新生成整个方案，浪费算力）
- 缺"部分重新装配"能力：只重装被补丁影响的天，其余天不变
- 缺修改次数计数和限制逻辑

## L6. 图片资产层

| 维度 | 说明 |
|---|---|
| **作用** | 管理所有实体的图片资产：从候选图池筛选精选图、CDN分发、适配不同场景尺寸 |
| **核心数据** | `entity_media`表（已有：url/caption/sort_order/is_cover/source）+ 需新增：`image_quality_score` / `aspect_ratio` / `resolution` / `cdn_url` / `usage_context` |
| **已有实现** | ⚠️ 基础结构已有。`entity_media`表可存图URL，渲染器有`_get_entity_image`。但缺质量筛选和CDN |
| **更新频率** | 中频：新实体入库时批量采集→周级质量审核→月级失效检查 |
| **长期保存** | ✅ 精选图永久保存。候选图池可定期清理（保留最近6个月） |
| **是否过期** | 图片URL可能失效（外部源404）。需定时健康检查 |
| **人工审核** | 封面图100%人工审核。详情图自动选（按质量分+多样性排序）+ 10%抽检 |
| **适合模板化** | ✅ 图片选择规则可模板化：每个实体=1封面(横)+2详情(横)+1竖图 |
| **适合AI生成** | ❌ 图片本身不能AI生成（真实场景照片）。但AI可辅助：图片质量评分、场景分类、caption生成 |

### 图片资产管道设计

```
爬虫采集(Google Places/Tabelog/Booking)
    ↓
候选图池(entity_media, is_curated=false)
    ↓ 自动筛选: 分辨率≥800px, 非重复, 非水印
精选图集(entity_media, is_curated=true)
    ↓ 人工抽检(S/A级实体)
CDN上传(cdn_url字段, 多尺寸裁剪)
    ↓
渲染引用: cover(1200x800) / detail(800x600) / thumb(400x300) / card(600x800竖)
```

### 当前缺口
- 图片全部引用外部URL，无CDN，加载速度不可控
- 缺图片质量自动评分
- 缺图片失效检测
- 缺多尺寸裁剪

## L7. 渲染缓存层

| 维度 | 说明 |
|---|---|
| **作用** | 缓存已渲染的HTML/PDF/H5产物，避免重复渲染。同一方案+同一模板=同一产物 |
| **核心数据** | `export_jobs`（渲染任务）+ `export_assets`（产物：PDF URL/H5 URL/封面图）+ `plan_artifacts`（最终交付物）+ Redis缓存（HTML片段） |
| **已有实现** | ✅ 完整流程：`renderer.py`(Jinja2→HTML) → `render_pdf`(WeasyPrint→PDF) → `export_jobs/assets`表记录 → `plan_artifacts`交付 |
| **更新频率** | 按需触发：方案生成→渲染，方案修改→重新渲染，模板升级→批量重渲染 |
| **长期保存** | PDF文件长期保存（用户可能多次下载）。HTML缓存短期保存（7天TTL） |
| **是否过期** | ✅ 方案数据变更后缓存失效。模板CSS更新后批量失效 |
| **人工审核** | ¥298+的渲染产物100%人工校对（通过`review_jobs.job_type='render_check'`）。基础版自动通过 |
| **适合模板化** | ✅ 渲染本身就是模板驱动（Jinja2 HTML模板 + CSS）。已有magazine风格模板体系 |
| **适合AI生成** | ❌ 渲染不需AI。模板+数据=确定性输出 |

### 渲染管道

```
方案数据(itinerary_plans/days/items)
    + 实体数据(entity_base + 扩展表)
    + 文案(copywriter生成, Redis缓存)
    + 图片(entity_media → CDN URL)
        ↓
    Jinja2 模板引擎
        ↓
    HTML 字符串
        ↓ WeasyPrint
    PDF bytes → S3/OSS → delivery_url
        ↓
    export_assets 记录
        ↓
    plan_artifacts → 交付给用户
```

### 当前缺口
- 缺分页模板：每天应独立渲染为一个section，便于免费预览"只展示Day1"
- 缺渲染版本控制：CSS更新后如何批量重渲染已交付的方案
- HTML缓存粒度太粗（整个方案），应拆为section级缓存

## L8. 内容引流素材层

| 维度 | 说明 |
|---|---|
| **作用** | 从方案家族（L3）和主档数据（L1）中衍生小红书/社群营销素材，不依赖用户个性化数据 |
| **核心数据** | 需新增表：`content_pieces`（素材库）、`content_templates`（素材模板）。数据来源：route_templates的亮点提取、entity_editor_notes的insider_tip、季节性events、L10反馈层的热门问题 |
| **已有实现** | ⚠️ 无专用系统。`sakura-rush-2026/` 目录有手工制作的樱花季内容，但没有系统化的素材生成管道 |
| **更新频率** | 周级：每周预生成下周素材→人工审核→排期发布 |
| **长期保存** | ✅ 素材库长期保存。发布记录永久保存（效果追踪） |
| **是否过期** | 部分过期：季节性内容（樱花/红叶）有有效期，通用内容（避坑/交通）不过期 |
| **人工审核** | ✅ 100%人工审核（小红书内容质量直接影响品牌） |
| **适合模板化** | ✅ 高度模板化。内容类型固定（榜单/地图/时间轴/问答/纠偏），结构可复用 |
| **适合AI生成** | ✅ 素材草稿适合AI生成：从结构化数据→营销文案。但需人工审核+润色 |

### 素材类型矩阵

| 素材类型 | 数据源 | 生成方式 | 示例 |
|---|---|---|---|
| Top-N榜单 | L1 entity_scores + L1 entity_base | 规则排序 + AI起标题 | "东京Top5免费景点" |
| 区域攻略地图 | L3 route_templates 的 day分区 | 模板 + 手工标注 | "新宿周边步行地图" |
| 避坑帖 | L1 entity_editor_notes (avoid_warning) | 模板拼接 + AI润色 | "别去这5家网红餐厅" |
| 花期/红叶播报 | L2 weather_snapshots + 外部花期数据 | 规则 + 模板 | "2026京都樱花实时" |
| 行程时间轴截图 | L3 route_templates → L7渲染 | 渲染截图 | "东京5日怎么排" |
| 用户FAQ | L10 反馈层热门问题 | AI回答 + 人工审核 | "JR Pass到底值不值" |
| 对比帖 | L1 多实体数据 | 模板 + AI文案 | "箱根vs有马温泉" |

### 当前缺口
- 无内容素材管理系统
- 无素材→发布平台的自动化对接
- 素材与方案家族的关联关系未建模

## L9. 发布与排期层

| 维度 | 说明 |
|---|---|
| **作用** | 管理内容发布日历：什么内容、什么时间、什么平台发布。协调内容节奏，避免重复/空窗 |
| **核心数据** | 需新增表：`publish_schedule`（发布排期）、`publish_records`（发布记录+效果数据） |
| **已有实现** | ❌ 完全缺失。当前靠人工在小红书后台操作 |
| **更新频率** | 周级：每周规划下周排期。热点事件实时插入 |
| **长期保存** | ✅ 排期历史永久保存（用于分析内容节奏效果） |
| **是否过期** | 排期条目按日期自动过期（已发布/已跳过） |
| **人工审核** | ✅ 排期表需人工确认（每周一次）。热点内容需快速审批通道 |
| **适合模板化** | ✅ 排期规则可模板化（每周固定N篇，类型轮换规则） |
| **适合AI生成** | 部分适合：AI可根据历史效果数据建议最佳发布时间和内容类型组合 |

### 排期规则示例

```
每周内容节奏（7天）:
周一: 榜单帖 (高互动率)
周二: 行程时间轴 (转化导向)
周三: 避坑/纠偏 (评论引流)
周四: 季节播报 (时效性)
周五: 对比帖 (收藏率高)
周六: 用户FAQ (评论互动)
周日: 休息 或 热点响应

预留位: 每周1个slot空出，用于热点响应
        (如突发签证政策变更/航班取消/自然灾害)
```

### 当前缺口
- 无排期管理工具
- 无发布效果回收（阅读/点赞/收藏/评论/私信转化）
- 无内容类型轮换策略

## L10. 数据反馈层

| 维度 | 说明 |
|---|---|
| **作用** | 闭环：收集用户行为/满意度/退款原因，反哺L1-L9各层的优化决策 |
| **核心数据** | 需新增表：`user_feedback`（评价/NPS）、`delivery_events`（浏览/下载/分享行为）、`refund_reasons`（退款原因分类）。+ `plan_artifacts.download_count`（已有） |
| **已有实现** | ⚠️ 极少。`plan_artifacts`有`download_count`，但无用户满意度、无行为追踪 |
| **更新频率** | 实时：用户行为事件实时采集。满意度调查=交付后24h触发 |
| **长期保存** | ✅ 永久保存（脱敏后）。是产品优化的核心数据资产 |
| **是否过期** | 不过期。历史数据用于趋势分析 |
| **人工审核** | ❌ 数据采集不需审核。但分析结论需人工解读 |
| **适合模板化** | ✅ 反馈采集表单模板化（NPS 1-10 + 标签选择 + 可选文字） |
| **适合AI生成** | 部分适合：AI可分析反馈文本提取共性问题，生成改进建议报告 |

### 反馈闭环设计

```
用户行为采集:
  浏览行为 → 哪天最受关注（停留时间）
  下载行为 → PDF vs H5 偏好
  分享行为 → 裂变入口追踪

满意度采集(交付后24h):
  NPS评分(0-10)
  标签选择: 路线好/排版好/餐厅实用/酒店准确/交通清晰/信息过时/内容太少/...
  可选文字反馈

退款分析:
  退款原因标签: 行程不满意/交付太慢/重复购买/取消旅行/...
  自动关联到具体方案+模板，发现系统性问题

反哺路径:
  L1: 高差评实体 → 降低 data_tier 或 标记 avoid_warning
  L3: 高满意度路线 → 提升模板权重
  L4: "天数不够"反馈 → 调整裁剪策略
  L6: "图片不好看"反馈 → 触发图片更新
  L8: 高互动问题 → 转化为内容素材
```

### 当前缺口
- 无用户满意度采集机制
- 无用户行为事件追踪
- 无退款原因结构化分析
- 无反馈→产品改进的自动化闭环
