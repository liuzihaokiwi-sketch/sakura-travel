# 代码审计结论：约束真源、假绿根因与最小修复路径

## 一句话结论

这不是“再多加几个字段校验”就能解决的问题。  
**当前核心问题是：生产链路没有把统一约束真正接进 planner，导致回归链路和生产链路分叉；即使回归链路接通了约束，排序层仍然会被 S 级经典线优先级压回“安全经典线”。**

---

## 1. 我确认了什么

### 1.1 `constraint_compiler.py` 不是没做，而是“写好了但没接通”
`constraint_compiler.py` 里已经有：

- `PlanningConstraints`
- `ConstraintTraceItem`
- `constraint_trace`
- `record_consumption()`
- 编译后的约束字段（如 `blocked_clusters`、`avoid_cuisines`、`city_strict_day_types`、`preferred_tags_boost` 等）

这说明问题已经不是“缺编译器”，而是**生产 orchestrator 没有统一用它**。

### 1.2 回归链路和生产链路真的分叉了
回归脚本 `run_regression.py` 会：

- `compile_constraints(profile)`
- 把 `constraints` 传给 `rank_major_activities`
- 传给 `build_route_skeleton`
- 传给 `fill_meals`

而生产里的 `generate_trip.py` 当前审计结论是：

- 没有统一调用 `compile_constraints()`
- ranker / skeleton / filler 仍有调用不带 `constraints`
- 下游大量逻辑继续直接读 `profile.*`

**这意味着：回归 PASS，不代表生产真的按同一条约束链在跑。**

### 1.3 renderer 不是根因
`render_export.py` 和 `html_renderer.py` 的职责很明确：

- 读取 `plan_id`
- 从 DB 取 `ItineraryPlan / ItineraryDay / ItineraryItem`
- 渲染 HTML/PDF

它们**不会自己重新规划路线**。  
所以“PDF 看起来像经典线”并不是 renderer 另造数据，而是 planner 上游就已经把经典线写进 DB 了。

### 1.4 guardrails 现在主要查结构，不查“约束有没有被消费”
`run_guardrails.py` 会把 warning/error 写进 `plan_metadata`，再决定是否 enqueue `render_export`。  
但现状更像是：

- 它会拦结构问题
- 不会拦“硬约束没被消费”
- 所以假绿可以顺利通过

### 1.5 还有一层隐藏问题：同一语义被三处各自解释
我确认至少有这几套独立解释逻辑：

- `__main__.py`：`PARTY_TYPE_TAG_RULES`
- `constraint_compiler.py`：`_PARTY_RULES`
- `assembler.py`：`_PARTY_TO_SEGMENT`

这会导致：
- 同一个 `party_type`
- 在入库、编译、打分权重里
- 可能被解释成三套不同意思

---

## 2. 现在最像真相的根因排序

### P0：生产链路没有真正接通 compiler
这是第一优先级，也是最该先修的。

表现：
- 回归链路会编译约束
- 生产链路未必会
- 所以下游 `constraints` 支持再多，生产也等于没用

直接后果：
- `avoid_cuisines` 在生产可能失效
- `max_intensity` 在生产可能失效
- `arrival/departure` day 约束在生产可能失效
- `preferred_tags_boost` 在生产可能失效
- `constraint_trace` 也就无法真实落地

### P1：即使接通 compiler，排序仍会偏经典线
`major_activity_ranker` 的问题不是“完全没做个性化”，而是：

- S 级 + default_selected 的经典线优先级太强
- 它在排序里像一个“硬插队权”
- 结果是个性化 boost 常常只是在经典线下面轻微浮动
- 最终还是伏见、岚山、东山这种首刷线排到前面

所以如果只修 P0，不修 P1：
- 系统会更守禁忌
- 会更守 day_type
- 但仍然可能“不够像这个人”

### P2：文案源和行程源没有完全同源
这也是你一直觉得“摘要懂用户，但 PDF 不像”的核心来源之一：

- 文案生成更容易直接读 `profile`
- 实际日程来自 planner 选中的 majors / skeleton / meals
- 当 planner 没吃进去约束，文案还是会继续“说对的话”

结果就是：
- 摘要像定制
- 结果像模板

---

## 3. 所以，这是不是系统设计问题？

**是。**

但不是那种“架构全推翻”的设计问题，而是一个很具体的 **Single Source of Truth + Consumption Trace** 没收口的问题。

更准确地说，问题是三层：

1. **约束来源不统一**
   - profile 能表达用户意图
   - compiler 也能表达用户意图
   - 但生产链路没有强制只走 compiler

2. **约束消费不统一**
   - 有的模块读 profile
   - 有的模块读 compiler
   - 有的模块有签名但调用方没传
   - 有的约束编译了但没人消费

3. **约束消费不可观测**
   - 现在很难一眼回答：
     - 哪条约束被谁用了
     - 哪条没用
     - 为什么没用
   - 于是你只能继续看 PDF 猜

---

## 4. 现在是不是在做“加越来越多字段校验”？

我的判断：**不该这么做。**

应该把现在的工作定义成：

- **先统一约束来源**
- **再统一消费**
- **最后补 trace 和 guardrail 可观测性**

而不是：

- 再加更多零散字段 validator
- 再加更多基于最终结果的补丁判断
- 再靠肉眼看 PDF 兜底

原因很简单：

- validator 只能保证“输入合法”
- 它不能保证“同一约束在 ranker / skeleton / filler / report / render 中被一致理解”

---

## 5. 接下来最值得做的事（按顺序）

### Step 1：先修 production pipeline 的断路
最小目标：

- 在 `generate_trip.py` 中统一创建 `constraints = compile_constraints(profile)`
- 然后把它透传给：
  - `major_activity_ranker`
  - `route_skeleton_builder`
  - `meal_flex_filler`

这是当前 ROI 最高的一步。  
因为很多下游签名和 compiler 其实已经准备好了，缺的是入口没接。

### Step 2：让 `constraint_trace` 真正落库
不是只在内存里有，而是至少进：

- `plan_metadata["constraint_trace"]`
  或
- `PlannerRun.run_log`

这样你以后看一条失败 case，不需要再先翻 PDF，可以直接看：

- 编译了哪些约束
- 谁消费了
- 谁没消费
- 忽略原因是什么

### Step 3：再修 ranker 的“经典线硬插队”
这个阶段才该动排序逻辑：

- 降低 `is_must_go` 的二元优先级
- 把它改成“加分项”，而不是“天然排前”
- 去掉或收紧 S 级默认线对 MMR 的豁免
- 让个性化偏好至少有真正赢过经典线的机会

### Step 4：最后再碰文案同源
在上游 execution 结果稳定以后，再让：

- summary
- overview
- day pages
- final PDF

都更多基于 **actual selected majors / actual skeleton / actual meals** 来写，而不是继续大量从 `profile` 的理想愿望出发。

---

## 6. 这轮我不建议先做什么

下面这些都不是当前最优先：

- 不先继续大批量人工审 PDF
- 不先碰 renderer / UI
- 不先扩大量 grader 断言
- 不先重构 scorer
- 不先做大规模 party rules 合并
- 不先试图一次性解决“所有字段都同源”

原因：
- 现在最值钱的是先把主链路接通
- 否则你后面做的很多检查，都只是在给断路系统加更漂亮的报警灯

---

## 7. 我给你的最终判断

### 结论 1
**你们前面反复看 PDF 不是白费。**  
它已经帮你们找到真实翻车模式。

### 结论 2
**但现在继续主要靠 PDF 审核来收敛，收益已经很低。**  
因为主问题已经不再是“有没有发现错误”，而是“系统为什么不能自己暴露错误”。

### 结论 3
**下一步最对的方向不是加更多字段校验，而是做：**
- compiler 接通生产链路
- constraint_trace 落库
- guardrails 开始感知“硬约束未消费”
- 再处理 ranker 的经典线碾压

---

## 8. 最短决策版（可直接发团队）

> 当前核心问题不是字段校验不足，而是统一约束真源未接入生产主链路，且约束消费不可观测。  
> 第一优先级应为：在 `generate_trip.py` 接通 `compile_constraints()` 并向 ranker / skeleton / filler 统一透传，同时把 `constraint_trace` 写入 `plan_metadata` 或 `PlannerRun`。  
> 第二优先级才是修复 ranker 中 S 级经典线对个性化排序的碾压。  
> renderer 不是根因，guardrails 目前主要是结构检查，也应补充对“硬约束未消费”的感知。  
> 在这两步完成前，继续靠反复审 PDF 收敛系统，边际收益已经很低。
