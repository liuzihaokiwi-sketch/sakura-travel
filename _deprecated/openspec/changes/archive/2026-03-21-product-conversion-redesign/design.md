## Context

当前状态：已完成 Next.js 14 技术栈搭建（Tailwind + shadcn/ui + Framer Motion），有首页、排行榜页、定制服务页三个"技术展示型"页面。但前台没有转化漏斗、没有支付流程、没有交付闭环。

核心约束：
- 前台弱化 AI，强调"专业判断 + 数据驱动 + 个性化"
- 文案以"用户痛点 → 我们给你的结果"为核心，不讲功能列表
- 不造假：不写假平台合作、假官方背书、假实时数据来源
- 最省人力：AI 做 80%，人工只做纠偏审核
- 初期支付可手动确认（微信转账），后期接入支付 API

## Goals / Non-Goals

**Goals:**
- ✅ 设计完整 9 步转化漏斗（引流→问卷→预览→付费→交付→修改→升级）
- ✅ 输出全套痛点导向的前台文案（引流页/价格页/交付页/品牌规范）
- ✅ 设计三档定价（免费 / 248首发 / 888锚点）的包装和对比方式
- ✅ 设计结构化问卷（轻问卷+正式问卷），让用户感觉在"定制旅行"而非"填表"
- ✅ 设计用户端结构化修改 + 内部审核工作台
- ✅ 输出品牌语气规范，作为所有页面文案的写作标准

**Non-Goals:**
- ❌ 不做支付接口对接（初期微信手动收款）
- ❌ 不做用户账号系统（初期通过微信 + 订单号识别）
- ❌ 不做 AI 生成引擎开发（本变更只做前台 + 工作台，后台生成用现有能力）
- ❌ 不做多语言
- ❌ 不做 APP

## Decisions

### D1: 转化漏斗 — 9 步路径

```
引流页(/) → 轻问卷(/quiz) → 免费预览(/preview/[id])
    ↓
价格页(/pricing) → 付款确认(/checkout)
    ↓
正式问卷(/questionnaire/[id]) → 生成中(/generating/[id])
    ↓
交付页(/plan/[id]) → 修改页(/plan/[id]/edit)
    ↓
升级页(/plan/[id]/upgrade)
```

**理由**：用户先免费体验一天行程的质量，建立信任后再付费。免费预览是核心转化杠杆——HubSpot 数据显示"先试后买"可提升 30%+ 转化率。

### D2: 三档定价策略

| 档位 | 命名 | 价格 | 定位 |
|---|---|---|---|
| 免费 | 一日体验版 | ¥0 | 引流：展示质量，建立信任 |
| 主推 | 行程优化版 | ¥248（首发）/ 原价 ¥298 | 利润款：7日完整行程+餐厅酒店+避坑 |
| 锚点 | 深度管家版 | ¥888 | 锚点：1对1深度定制，衬托主推款超值 |

**命名逻辑**：不叫"定制版"（用户会觉得都是定制有什么区别），叫"行程优化版"——强调的是**结果**（你的行程被优化了），不是**过程**（我们定制了）。

**理由**：NN/g 强调"选项差异要明确"。三档的区别点是：**时间范围**（1天 vs 全程）、**深度**（景点推荐 vs 餐厅酒店避坑全包）、**服务方式**（自助 vs 有人跟进）。

### D3: 免费一日预览 — 包含/不包含

| 包含 | 不包含 |
|---|---|
| 1 天完整行程（时间轴+景点+交通） | 其余天数（模糊展示，不给详情） |
| 2-3 个景点的完整推荐理由 | 餐厅具体店名和评分 |
| 当天大致交通指引 | 酒店推荐 |
| 1 个"为什么这样安排"的专业解读 | 拍照攻略、避坑指南 |
| 行程质感展示（让用户看到交付物的品质） | 可打印/可分享版本 |

**理由**：免费版的目标是让用户产生两个念头："质量真的不错"+"但我只看到一天不够啊"。

### D4: 文案三大原则

1. **痛点先行**：每个模块先说"你在日本旅行最怕什么"，再说"我们怎么解决"
2. **结果导向**：不说"我们有12维评分系统"，说"帮你避开评分虚高的网红店"
3. **弱 AI 强专业**：不说"AI 生成"，说"数据驱动规划""专业判断框架""多源校验"

**允许的高级表达**：多源数据校验、动态更新、专业判断框架、个性化匹配、"不是泛泛而谈，而是更适合你的方案"
**禁止**：假平台合作、假官方背书、"与XX深度合作"、前台露出 AI/算法/引擎字眼

### D5: 问卷设计 — 对话感而非表单感

**轻问卷（4题，30秒完成）**：

| # | 问题 | 类型 | 真实用途 |
|---|---|---|---|
| 1 | 你大概什么时候去日本？ | 日期范围选择器 | 确定花期/季节 |
| 2 | 几个人一起？ | 图标单选（1人/2人/家庭/闺蜜团） | 确定场景 |
| 3 | 主要想去哪？ | 城市多选（东京/京都/大阪/其他） | 确定路线模板 |
| 4 | 你最在意什么？ | 3选多选（出片/美食/省钱/文化/亲子/放松） | 确定偏好权重 |

**正式问卷（付费后，6-8题，2分钟完成）**：
在轻问卷基础上追加：预算范围滑杆、住宿偏好、饮食禁忌/偏好、特殊需求文本框（限100字）。

**理由**：轻问卷用图标选择 + 大按钮，像在"选旅行风格"不像在"填表"。正式问卷才收敛到具体约束。

### D6: 修改机制 — 结构化为主

**用户端可改的（按钮/下拉）**：
- 某天行程节奏太紧/太松 → 滑杆调节
- 换掉某个景点 → 从备选列表选替代
- 换掉某家餐厅 → 从同区域备选选替代
- 调整某天的重点（购物多一点 / 文化多一点） → 标签切换

**用户端不能改的**：
- 不能自由输入 prompt
- 不能改整体路线骨架（那是重新生成）
- 不能无限修改（248档2次，888档5次）

**内部审核端**：
- 结构化切换为主（替换景点/餐厅/酒店 → 从数据库选）
- 提示词补充为辅（微调 AI 生成的推荐理由文案）
- 目标：5分钟内审核+发布一单

### D7: 页面架构总览

**用户前台**：
```
/                    → 引流首页（痛点+方案+预览入口+定价）
/quiz                → 轻问卷（4题图标选择）
/preview/[id]        → 免费一日预览（1天行程+升级CTA）
/pricing             → 价格页（三档对比）
/checkout/[id]       → 付款确认（微信扫码/手动）
/questionnaire/[id]  → 正式问卷（6-8题）
/generating/[id]     → 生成等待页（动画+预期时间）
/plan/[id]           → 交付页（完整行程杂志展示）
/plan/[id]/edit      → 修改页（结构化修改表单）
/plan/[id]/upgrade   → 升级页
```

**内部工作台**：
```
/admin               → 订单看板（待审核/已发布/待修改）
/admin/order/[id]    → 单订单审核页（预览+编辑+发布）
/admin/templates     → 模板管理
```

## Risks / Trade-offs

**[R1: 初期无支付接口]** → 用户在 `/checkout` 看到微信二维码，手动转账后客服确认。Mitigation: 页面设计成"专属顾问确认"的感觉，反而增加高端感。

**[R2: 免费预览可能被白嫖]** → 用户只要免费版不付费。Mitigation: 免费版只给 1 天且不给餐厅酒店，核心价值在完整版；加上限时提示增加紧迫感。

**[R3: 修改次数引发客诉]** → 用户觉得 2 次不够。Mitigation: 文案包装成"精调机会"而非"修改限制"，如"包含 2 次行程精调"。

**[R4: 文案风格不统一]** → 多个页面由不同人写。Mitigation: 先产出品牌语气规范文档，所有页面文案必须遵循。

## Open Questions

- Q2: 交付页是否需要密码保护？建议用随机长 ID 即可（如 `/plan/a3f8k2x9`）。
