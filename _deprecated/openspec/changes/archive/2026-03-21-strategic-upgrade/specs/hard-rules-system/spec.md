## C. 第一性原理结论

### C1. 用户为什么会为日本攻略付费？

不是为"信息"付费——信息在小红书/Google/ChatGPT上都能免费获得。用户付费买的是三件事：

1. **决策省心**：从1000个选项中帮我选出最适合我的30个，并告诉我为什么。用户付费的本质是"把决策外包给比我更懂的人"。
2. **执行安心**：不只告诉我去哪，还帮我排好顺序、算好时间、标好交通、提醒预约，拿着就能走。
3. **避错保险**：选错酒店多花2小时通勤、去了发现关门、排队2小时踩雷——这些"选错一次"的隐性成本远超¥248。

**硬推论**：如果免费预览的 Day 1 不能同时展示"决策判断力"+"执行细节力"+"避错能力"，用户就不会付费。这三者缺一不可。

### C2. 用户看到免费一天后，决定付费的核心触发因素是什么？

**触发公式**：付费意愿 = 感知专业度 × 感知个性化 × 剩余天数好奇心 / 自己做的成本

具体触发因素（按权重排序）：
1. **"这比我自己做的好"** — 推荐理由的专业判断力、时间精确到半小时、交通衔接细节
2. **"这是为我做的"** — 看到同行人类型、偏好、时间被考虑进去了
3. **"后面还有6天我想看"** — Day 2-7标题勾起好奇心但看不到内容
4. **"有人帮我兜底"** — 避坑提醒、PlanB的存在感（提到但不展开）
5. **"这个价格不贵"** — ¥248 < 选错一次酒店/餐厅的成本

### C3. 用户为什么会喜欢一份攻略、愿意推荐、甚至下次还来买？

1. **旅行中验证了推荐的准确性** — 去了发现确实好，这是最强的口碑驱动力
2. **体验到了"替我想到了"的周全感** — 避坑提醒命中、PlanB用上了、预约提醒有用
3. **方案本身有分享价值** — 长图漂亮、截图发朋友圈有面子
4. **微调体验好** — 自助切换几个酒店/餐厅没有摩擦感，方案变得更贴合自己

**硬推论**：复购和转介绍的核心不是"服务态度好"，而是"推荐命中率高+方案确实好用"。这意味着数据系统的质量 > 客服的温度。

### C4. 怎样的系统能在少人工介入下，仍稳定输出让用户满意的攻略？

答案不是"一个更强的模型"，而是"一条多角色校验的流水线 + 一套有深度的城市上下文数据系统"：
- **数据系统**决定推荐的"对不对"（事实层面）
- **多模型评审**决定方案的"好不好"（质量层面）
- **自助微调**让用户自己解决80%的"合不合我意"问题
- **人工**只处理剩下的20%高风险/高价值/低置信度订单

### C5. 为什么"自助微调层"会提升体验，而不是削弱专业感？

类比：高级西装定制店不会让你自己裁布，但会让你在面料样本册里挑选喜欢的布料、在预设的版型里选合身的款式。**可控范围内的选择权 = 掌控感 + 专业感的叠加，而非矛盾**。

关键约束：
- 用户只能在"系统预选的候选池"里切换，不能自由发挥
- 切换后系统自动校验整体合理性，不合理的自动拒绝
- 交互设计必须传达"这些都是我们筛选过的好选项"而非"你自己挑吧"
- 自助微调不暴露技术细节，UI语言是"想换个风格？看看这几个也不错"

### C6. 顶级攻略的本质是什么？

**顶级攻略 = 正确的选择 × 合理的编排 × 精准的上下文 × 用户的掌控感**

- 正确的选择：推荐的景点/餐厅/酒店是"在这个时间、这个客群、这个预算下的最优解"
- 合理的编排：地理动线高效、体力曲线合理、时间利用充分但不赶
- 精准的上下文：知道周三下午人少、知道这家店周末休息、知道樱花第二周最好看
- 用户的掌控感：用户觉得"这是我的方案，我做了选择"，而不是"被动接受别人的安排"

### D2. 首发客群裁决

**第一优先客群：闺蜜/朋友结伴（2-4人小团）**

理由：
1. **转化杀伤力最强**：多人出行 = 多人看到方案 = 多人分摊决策心理 = 一人付费全组受益 = 全组传播
2. **AI交付最稳定**：需求模式化（美食+购物+拍照+网红打卡），候选实体池匹配度高
3. **分享裂变最强**：闺蜜出行是小红书最高频内容类型，用户天然愿意发"出行前准备"和"旅行Vlog"
4. **复购最强**：年轻女性群体是日本旅行最高频客群（JNTO数据：20-30岁女性赴日比例最高），闺蜜旅行可能每年一次
5. **自助微调需求最健康**：闺蜜们会讨论"想换这家餐厅""想多逛购物"，正好是自助微调的最佳场景

**第二优先客群：情侣/蜜月**

理由：
1. **客单价潜力高**：蜜月/纪念日价格敏感度最低，888升级率可能最高
2. **方案感知价值高**：浪漫晚餐+出片点位+酒店氛围，方案本身就是"惊喜的一部分"
3. **AI交付稳定**：情侣需求同样模式化（浪漫+出片+美食+酒店好）
4. **分享动力强**：情侣旅行照/Vlog是社交平台主力内容

**暂缓的客群：亲子、带父母**

原因：
- 亲子和带父母的需求复杂度远超情侣/闺蜜，AI稳定交付难度大
- 人工兜底压力高（安全/适龄/无障碍/医疗等特殊需求多）
- 当前实体库缺乏亲子/适老标签深度
- 建议在标准版月成交 > 50 单后，再启动亲子版专项

## F. 自助微调层设计

### F1. 设计原则

自助微调层是一个**正式产品模块**，不是临时补充功能。它的目标是：
- 让用户在不消耗正式修改次数的前提下，在可控范围内个性化自己的方案
- 减少80%的人工修改请求
- 提升用户"这是我的方案"的掌控感
- 系统自动保障微调后方案仍满足整体质量要求

### F2. 可自助微调的模块（开放列表）

| 模块 | 微调方式 | 候选数 | 候选来源 | 约束条件 |
|---|---|---|---|---|
| **餐厅替换** | 点击餐厅→弹出候选卡片→选一个 | 3-5 个 | 同区域、同价位带、同餐饮类型 | 步行距离 ≤ 前后景点15分钟内 |
| **景点替换** | 点击景点→弹出候选卡片→选一个 | 3-5 个 | 同区域、同类型（文化/购物/自然/出片） | 停留时间差 ≤ 30分钟，交通衔接不打破 |
| **酒店区域替换** | 点击住宿区→弹出候选区域→选一个 | 2-3 个 | 同城市、同价位带 | 到核心景点通勤 ≤ 40分钟 |
| **当天节奏调整** | 滑杆：宽松↔紧凑 | 3 档 | 系统自动增减景点数 | 景点数在 2-5 个范围内 |
| **当天侧重调整** | 标签切换（购物/文化/出片/美食） | 4-5 个标签 | 当天景点优先级重排 | 不改变核心景点，只调优先级 |
| **出发时间调整** | 时间选择器（8:00-11:00，步进30分钟） | 7 个时间点 | 系统自动重排时间轴 | 末个景点 ≤ 21:00 |

### F3. 不可自助微调的模块（锁定列表）

| 模块 | 锁定原因 | 用户看到什么 |
|---|---|---|
| **城市/目的地** | 改了等于重做整个方案 | 不展示修改入口 |
| **旅行日期** | 改了影响全局（季节/天气/营业） | 不展示修改入口 |
| **同行人数/类型** | 改了影响全局推荐逻辑 | 不展示修改入口 |
| **总天数** | 改了影响路线骨架 | 不展示修改入口 |
| **整体路线骨架/城市顺序** | 改了等于重做 | 不展示修改入口 |
| **推荐理由文案** | 用户不需要改 | 不展示修改入口 |
| **交通方案** | 由景点组合自动推导 | 随景点微调自动更新 |
| **避坑提醒** | 系统自动生成 | 不展示修改入口 |
| **准备清单/安全须知** | 模板化内容 | 不展示修改入口 |

### F4. 自助微调边界硬规则

| 规则 | 描述 |
|---|---|
| **HR-TW-01** | 每次自助微调操作完成后，系统必须在 ≤ 5 秒内完成整体重排和重检 |
| **HR-TW-02** | 单日内景点替换 ≤ 该日景点总数的 60%（例如5个景点最多换3个） |
| **HR-TW-03** | 全程累计景点替换 ≤ 全程景点总数的 40%（超过此比例提示"需要正式精调"） |
| **HR-TW-04** | 替换后的新实体必须在数据库中存在且 quality_score ≥ B |
| **HR-TW-05** | 替换后相邻景点间通勤 ≤ 40 分钟（超过则拒绝替换并提示原因） |
| **HR-TW-06** | 替换后当日体力估算 ≤ 15000 步（超过则自动调整为宽松模式） |
| **HR-TW-07** | 餐厅替换后必须保持午餐(11:30-13:30)和晚餐(17:30-20:00)各至少1家 |
| **HR-TW-08** | 酒店区域替换后，所有天的首个景点通勤 ≤ 40 分钟 |
| **HR-TW-09** | 自助微调不消耗正式修改次数，但系统记录所有微调操作用于分析 |
| **HR-TW-10** | 微调次数无上限，但如果用户在 10 分钟内操作 > 15 次，弹出提示"需要更多调整？联系规划师帮你" |

### F5. 自助微调后的自动重排与重检机制

```
用户选择替换实体
    ↓
系统查询候选池（预计算，≤ 100ms）
    ↓
展示候选卡片（含图片+一句推荐理由+关键信息）
    ↓
用户选择一个候选
    ↓
系统执行约束检查（≤ 500ms）：
  ├── 通勤时间检查
  ├── 时间冲突检查
  ├── 体力曲线检查
  ├── 餐饮时段覆盖检查
  ├── 实体质量分检查
  └── 累计替换比例检查
    ↓
  ├── 全部通过 → 立即更新方案展示，交通/时间轴自动重排
  └── 任一不通过 → 拒绝替换，展示具体原因+建议（如"这家店距离太远，试试这个更近的？"）
```

### F6. 什么条件下才开始消耗正式修改次数

以下任一情况发生时，进入正式修改流程（消耗次数）：

| 条件 | 说明 |
|---|---|
| 用户在自助微调候选池中找不到满意选项 | 点击"以上都不满意，需要更多选择"按钮 |
| 用户需要修改锁定模块 | 如想换城市、换日期、换路线骨架 |
| 用户需要自由文本表达需求 | 如"第五天想安排求婚" |
| 单日替换比例超过 60% | 系统提示"这天改动比较大，建议提交正式精调" |
| 全程替换比例超过 40% | 系统提示"整体改动较多，建议提交正式精调" |

### F7. 自助微调的文案规范

| 场景 | 文案 | 禁止 |
|---|---|---|
| 微调入口 | "想换几个地方？看看其他选择" | ❌ "自助修改" "自助服务" |
| 展示候选时 | "这几个也很不错，都帮你筛过了" | ❌ "系统推荐" "算法推荐" |
| 替换成功 | "已更新！交通和时间也帮你调好了" | ❌ "系统已自动重排" |
| 替换被拒 | "这家距离有点远，试试这个更近的？" | ❌ "不满足约束条件" "违反规则" |
| 超出微调范围 | "这个改动比较大，让规划师帮你看看？" | ❌ "请提交正式修改" "次数将被扣减" |

### H2. AI自动维护机制

#### 事实抽取流水线
```
外部数据源（Tabelog/Google/官方旅游局/天气API）
    ↓
AI事实抽取Agent → 结构化数据（实体/评分/营业时间/价格）
    ↓
交叉审查Agent → 多源数据比对，标记冲突
    ↓
冲突解决Agent → 按数据源可信度排序取最可信值
    ↓
入库前校验 → 格式/范围/一致性自动检查
    ↓
  ├── 通过 → 直接入库（B/C级实体）
  └── 异常 → 进入人工审核队列（S/A级实体 或 有冲突）
```

#### 定期自动任务
| 任务 | 频率 | 内容 |
|---|---|---|
| 实体新鲜度检查 | 每日 | 标记 last_verified_at > 90天 的实体为 stale |
| 营业状态校验 | 每周 | AI检查高频实体是否还在营业 |
| 价格快照更新 | 每周（P0城市每3天） | 爬虫更新酒店/餐厅价格 |
| 候选池刷新 | 每周 | 重新计算 entity_alternatives 的相似度和排序 |
| 季节性更新 | 每月初 | 更新 city_monthly_context、seasonal_events |
| 全量质量审计 | 每月 | AI对所有活跃实体执行 entity_quality_audit |
| 用户反馈聚合 | 每周 | 将 user_entity_feedback 聚合回实体评分 |

## I. 多模型评审流水线

> 不让一个模型从头写到尾。拆分为多角色流水线，每个角色有明确输入/输出/通过标准。

### I1. 角色定义

| 角色 | 职责 | 模型建议 | 输入 | 输出 |
|---|---|---|---|---|
| **规划师模型** | 做主方案：候选召回→编排→文案 | GPT-4o / Claude 3.5 | 用户画像+城市数据+模板+候选池 | 完整方案JSON（days/slots/entities/copy） |
| **质检模型** | 挑事实错误和逻辑不合理 | GPT-4o-mini | 方案JSON+实体数据库 | 问题清单（severity: critical/warning/info） |
| **用户代理模型** | 从目标客群视角挑体验问题 | Claude 3.5 Haiku | 方案JSON+用户画像+客群画像 | 体验评分+改进建议清单 |
| **地接运营模型** | 看执行难度：排队/预约/交通/天气 | GPT-4o-mini | 方案JSON+餐厅时段表+交通矩阵+天气 | 执行风险清单（risk_level: high/medium/low） |
| **微调守门模型** | 判断哪些改动可放给用户自调 | GPT-4o-mini | 方案JSON+候选池+约束规则 | 每个slot的微调边界定义 |
| **总审模型** | 综合所有评审意见做最终裁决 | GPT-4o | 方案JSON+所有评审结果 | 最终裁决（pass/revise/reject）+修改指令 |

### I2. 流水线步骤

```
Step 1: 候选召回
  输入: 用户画像 + 城市数据 + 模板
  操作: 从数据库召回符合条件的实体候选集
  输出: 候选集（每个slot 10-20个候选）
  人工: 不需要

Step 2: 规划师模型生成主方案
  输入: 候选集 + 用户画像 + 模板骨架 + 城市上下文
  操作: 选择实体→编排时间轴→生成交通→写推荐理由
  输出: 方案v1 JSON
  人工: 不需要
  硬约束: 所有实体必须从候选集中选取（禁止幻觉）

Step 3: 质检模型审查
  输入: 方案v1 + 实体数据库 + 质量硬规则(QTY-01~10)
  操作: 逐条校验质量硬规则
  输出: 问题清单
  通过条件: critical=0 且 warning≤3
  不通过处理: 退回Step 2重新生成（附带修改指令），最多重试2次

Step 4: 用户代理模型评审
  输入: 方案v1 + 用户画像（客群/偏好/预算）
  操作: 模拟目标用户阅读方案，挑体验问题
  输出: 体验评分(0-10) + 改进建议
  通过条件: 体验评分≥7
  不通过处理: 将建议传回Step 2，触发定向修改

Step 5: 地接运营模型评审
  输入: 方案v1 + 餐厅时段表 + 交通矩阵 + 天气数据
  操作: 检查排队风险/预约需求/交通摩擦/天气影响
  输出: 执行风险清单
  通过条件: high_risk=0
  不通过处理: 标记高风险点位→替换为低风险候选

Step 6: 微调守门模型
  输入: 方案v1 + 候选池 + 约束规则
  操作: 为每个slot预计算可微调候选+边界条件
  输出: candidate_pool_cache 数据
  通过条件: 每个slot至少有2个可替换候选
  不通过处理: 扩大候选范围或标记为"不可微调slot"

Step 7: 总审模型
  输入: 方案v1 + 所有评审结果
  操作: 综合裁决
  输出: 最终状态 + 置信度分数(0-1)
  裁决逻辑:
    - 所有评审通过 + 置信度≥0.85 → 自动发布（¥248订单）
    - 任一评审不通过 + 重试2次后仍不通过 → 进入人工队列
    - 置信度<0.7 → 强制人工审核
    - ¥888订单 → 无论置信度都进人工审核
```

### I3. 投票与博弈机制

| 情况 | 处理 |
|---|---|
| 质检通过 + 用户代理不通过 | 以用户代理为准（用户体验优先于技术正确） |
| 地接运营标记高风险 + 其他都通过 | 替换高风险实体后重新跑Step 3-4 |
| 用户代理和地接运营都不通过 | 直接进入人工队列，不再自动重试 |
| 所有模型都通过但总审置信度<0.85 | ¥248可自动发布但标记为"建议人工抽检"；¥888必须人工 |

### I4. 重写触发条件

| 条件 | 重写范围 |
|---|---|
| 质检发现实体不存在 | 替换该实体，重排当天 |
| 交通超时（>40分钟） | 重排当天动线 |
| 餐厅定休日冲突 | 替换该餐厅 |
| 用户代理评分<5 | 全方案重新生成（换模板变体） |
| 连续2次重试仍不通过 | 进入人工队列 |

### I5. 人工介入时机

| 时机 | 人工做什么 | 预计耗时 |
|---|---|---|
| 总审置信度<0.7 | 查看评审报告→决定是否修改→发布 | 5-10分钟 |
| 2次重试仍不通过 | 手动调整方案→重跑质检→发布 | 10-20分钟 |
| ¥888订单 | 全面审核+个性化调整+1对1沟通 | 30-60分钟 |
| 用户投诉 | 排查问题→修复→重新交付 | 15-30分钟 |
