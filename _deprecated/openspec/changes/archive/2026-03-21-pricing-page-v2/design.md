## Context

当前价格页（`web/app/pricing/page.tsx`）已经从数据库拉取产品配置（ISR 5分钟缓存），但价格卡文案、差异化亮点、PDF 说明、FAQ 全部缺失。深度版（¥888）比标准版（¥248）贵 ¥640，但前端没有任何内容解释这 ¥640 买的是什么——「深度比价系统」（帮用户节省旅行开销 20-30%）是核心差异点，必须上页面。交付机制（PDF 主 + H5 辅、轻水印）也需要在付款前明确告知，降低交付后的摩擦。

## Goals / Non-Goals

**Goals:**
- 实现完整的价格卡组件（两张卡，各含价格/亮点/浮动说明/CTA）
- 实现深度版「节省 20-30% 旅行开销」比价系统展示模块
- 实现 FAQ 组件（8 条口径，手风琴展开）
- 在价格页/成功页/交付页三处加入 PDF 交付说明文案
- 所有文案直接内嵌为常量，不走 CMS，减少依赖

**Non-Goals:**
- 不修改后端定价 API 或数据库结构
- 不实现真实的比价数据计算（展示为固定文案，不是动态数字）
- 不实现 FAQ 搜索功能
- 不修改 PDF 生成/水印逻辑（水印文案在 PDF 渲染层另行处理，本 change 只做前端说明）

## Decisions

### D1：深度版差异化的展示位置

**决策**：在深度版价格卡内部用独立高亮 block 展示比价系统，而不是放在卡外的通用说明区。

**理由**：用户在比较两张卡时，差异化内容必须在对比视野内。放到卡外容易被跳过，放到卡内才能在用户决策瞬间发挥作用。

**备选方案**：在两张卡上方放一个 banner 说明深度版特色 → 被否，因为 banner 在移动端容易被当做广告忽略。

### D2：文案硬编码 vs 从 DB/CMS 拉取

**决策**：文案硬编码为 TypeScript 常量（`web/lib/content/pricing.ts`），不走 CMS。

**理由**：当前阶段文案变动频率低，CMS 引入成本高于收益。硬编码在 `content/` 目录下集中管理，产品经理可直接改文件，审查 PR 即可上线。

**备选方案**：Contentful/Sanity CMS → 被否，过度设计，当前团队规模不支持。

### D3：FAQ 组件实现方式

**决策**：使用原生 `<details>/<summary>` + Tailwind CSS，不引入新的手风琴库。

**理由**：无新依赖，无 hydration 问题，SEO 友好（内容在 HTML 中），动画用 CSS transition 实现足够。

### D4：PDF 说明文案的放置策略

**决策**：三处放置，各有侧重：
1. **价格页**：一行小字（最简洁，不打断决策流）
2. **提交成功页**：完整说明（用户已提交，此时有充足注意力阅读）
3. **交付页**：操作指引式说明（PDF 已在手，说明怎么用）

**理由**：在付款前放过多说明会增加摩擦；在付款后详细说明既完整又不影响转化。

### D5：「节省 20-30%」的表达方式

**决策**：用具体场景化金额举例（"7 天日本行平均花销 ¥8000，节省 20-30% 意味着省下 ¥1600-¥2400"），而不是只写百分比。

**理由**：百分比抽象，具体金额能让用户立刻算出"这个 ¥888 本身就能从节省的钱里回本"，大幅提升感知价值。

## Risks / Trade-offs

- **[风险] 文案硬编码导致非技术人员修改困难** → 缓解：把所有文案集中到 `web/lib/content/pricing.ts`，非常规的 TypeScript，产品经理参照格式即可修改，并在文件顶部注释使用说明。

- **[风险] 「节省 20-30%」的数字如果被用户较真** → 缓解：文案加注"（基于历史定制行程的餐厅/酒店/交通方案对比分析）"，并在 FAQ 中解释这是基于已有案例的统计区间，不是承诺。

- **[风险] 移动端两张价格卡并排显示空间不足** → 缓解：移动端改为纵向堆叠，深度版放在下方并加"推荐"标签，桌面端并排展示。

- **[Trade-off] 不引入 CMS 意味着文案上线需要走代码流程** → 可接受：当前阶段文案变动频率低，一个月不超过两次，工程成本低于 CMS 运维成本。
