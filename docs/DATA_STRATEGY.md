### 博主/固定站点（可配置）

你提供 URL 列表，系统按配置定期拉取：

```yaml
crawl_sources:
  - name: "小红书-关西美食"
    url: "..."
    frequency: weekly
    parser: xiaohongshu
    extract: [entity_mentions, ratings, tips, avoid_flags]

  - name: "某博主博客"
    url: "..."
    frequency: weekly
    parser: generic_blog
    extract: [entity_mentions, personal_tips]
```

提取的信息自动和现有库交叉匹配：已存在 → 更新信息；新发现 → 标记待审核。

### 两层采集策略

- **广度层（自动化）**：Tabelog/大众点评 + Google/高德 覆盖评分达标的全部实体，建走廊级候选池
- **深度层（人工+AI）**：从小红书/博主/百名店挖掘"评分不高但体验独特"的店，标记为 `discovery_gem: true`

## 二、质量档位（取代精细评分）

### 为什么不用精细分数

当前 `final_score = base_score(0-100) + editorial_boost(-8~+8)`，导致 82.3 永远排在 82.1 前面。0.2 分的差别是噪音不是信号，但系统当真——每次都选同一家。

### 质量档（5档，人工+规则判定）

| 档位 | 含义 | 进入条件 |
|------|------|---------|
| **5 极好** | 值得专程去、旅行记忆点 | 编辑认定 + 多源高分佐证 |
| **4 很好** | 推荐给大多数人没问题 | 评分+评论综合达标 |
| **3 不错** | 主流选择，稳定可靠 | 数据充分，无负面 |
| **2 凑合** | 能用但不会主动推荐 | 有瑕疵或数据不足 |
| **1 避开** | 有明确负面信号 | 风险标签命中 |

**核心规则**：
- 同档内完全平权，不排名
- 选择由上下文决定：菜式匹配、距离、轮转次数、用户画像
- 档位由"人工判定 + 规则辅助"，不是原始分数自动映射
- 默认只推荐 3档及以上

### 预算档（4档）

| 档位 | 日本餐厅 | 日本酒店/晚 | 国内餐厅 | 国内酒店/晚 |
|------|---------|------------|---------|------------|
| **低** | ~¥1000/人 | ~¥8000 | ~50元/人 | ~300元 |
| **中** | ¥1000-3000 | ¥8000-20000 | 50-150元 | 300-800元 |
| **高** | ¥3000-8000 | ¥20000-50000 | 150-500元 | 800-2000元 |
| **超高** | ¥8000+ | ¥50000+ | 500元+ | 2000元+ |

### 原始分数保留但不直接排名

```
entity:
  quality_tier: 4              # 粗粒度档位（决定是否进池）
  budget_tier: mid             # 粗粒度预算
  raw_scores:                  # 原始分数，仅供参考和档位判定依据
    tabelog: 3.58
    google: 4.3
    dianping: 4.5
    booking: 8.7
  discovery_gem: false         # 隐藏好店标记
```

## 三、避雷机制

**原则：宁可少推荐，不能推荐差的。一次糟糕体验毁掉整本手账的价值。**

### 三层过滤

```
第一层：硬黑名单（永不推荐）
  ├── 已关门
  ├── 多次被投诉/差评
  ├── 卫生问题
  ├── 坑游客/宰客
  └── 安全隐患

第二层：风险标签（默认过滤，用户可放开）
  ├── environment_basic     环境简陋（苍蝇小馆）
  ├── service_rough         服务态度差
  ├── queue_extreme         排队超长（>60min）
  ├── cash_only             仅收现金
  ├── no_chinese_menu       无中文菜单（日本）
  ├── no_english_menu       无英文菜单
  ├── hygiene_concern       卫生存疑
  └── noise_level_high      环境嘈杂

第三层：正常池（默认可推荐）
  └── 通过以上两层的实体
```

### 表单中的"我可以接受"选项

默认全部不勾（= 系统帮你避开）：

```
以下情况你可以接受吗？不勾选我们会帮你避开：
□ 环境比较简陋但味道好的小馆子
□ 服务态度不太好但出品稳定的店
□ 排队超过30分钟的热门店
□ 只收现金的店
```

勾选后对应风险标签解锁，相关实体可进入候选池。

### 避雷数据采集

| 来源 | 提取什么 |
|------|---------|
| 大众点评/Tabelog 低分评论 | AI 提取关键词："脏"、"态度差"、"坑"、"不新鲜"→ 映射到风险标签 |
| 小红书避雷帖 | 直接标记的避雷内容 |
| Google Maps 1-2星评论 | 负面关键词提取 |
| 用户旅行后反馈 | 直接标记"体验差" + 具体原因 |

AI 批量扫评论提取风险标签，人工抽检确认。

## 四、推荐多样性（反集中）

### 问题

1000个客户/月不能都去同一家店。当前 ranker 按分数排序选 Top 1，必然集中。

### 方案：档内平权 + 上下文选择 + 轮转

**第一步：质量档过滤**
- 只有 3档及以上进入候选池

**第二步：上下文匹配**（在同档内选择，不是排名）
- 菜式匹配用户画像（情侣→浪漫氛围，家庭→儿童友好，独行→一人食）
- 距离匹配当天路线（晚餐在酒店步行15分钟内）
- 菜式不重复（连续两餐不同菜式）
- 预算档匹配用户预算

**第三步：轮转机制**
- 每个实体记录近30天被推荐次数
- 同等条件下优先选推荐次数少的
- `discovery_gem` 标记的实体额外获得曝光机会

**第四步：软随机**
- 最终候选多于1个时，基于日期种子的确定性随机选择
- 同一客户重复生成结果一致，不同客户自然分散

### 用户不填喜好时的分散策略

不填 ≠ 没有偏好。从已知信息推断：

| 已知 | 推断 |
|------|------|
| 情侣 | 偏浪漫氛围、适合拍照 |
| 家庭带娃 | 有儿童餐、座位宽敞 |
| 独行 | 吧台座、一人食友好 |
| 预算=mid | 排除高端割烹和连锁快餐 |
| 夏天 | 偏冷面/刨冰/清淡 |
| 冬天 | 偏锅物/拉面/暖食 |

完全无信息时用基于订单日期的确定性随机，保证不同客户拿到不同组合。

## 五、整程体验编排

不只是单个实体好，要整个行程的餐饮+酒店搭配让人觉得丰富。

### 餐饮多样性规则

| 规则 | 说明 |
|------|------|
| 菜式不重复 | 连续两餐不推荐同一菜式（午餐拉面 → 晚餐不能还是拉面） |
| 体验梯度 | 第一天舒适熟悉 → 中间深度本地 → 高潮日 splurge → 最后一天轻松便捷 |
| 预算分配 | 不是每餐都 mid，而是"2顿奢华 + 4顿日常 + 2顿便宜但好吃"更有记忆点 |
| 距离合理 | 晚餐在酒店步行15分钟内，午餐在景点附近 |
| 不跨区吃 | 同一天内午餐和下午景点在同一区域 |

### 酒店联动

| 场景 | 规则 |
|------|------|
| 换酒店日 | 晚餐推荐新酒店附近（到新地方先吃好的） |
| 最后一晚 | 推荐酒店附近有仪式感的"告别餐" |
| 温泉酒店 | 标注含餐/不含餐，不含时推荐附近选择 |
| 到达日 | 推荐酒店附近简单舒适的店（飞了一天不想跑远） |

## 六、活动类型扩展

当前系统只有 POI（景点），缺两类重要内容：

### 季节性活动（每年固定时间）

```
seasonal_event:
  name: "祇园祭"
  type: seasonal_recurring
  recurrence: {month: 7, day_range: "1-31", peak_days: "14-17"}
  circle_id: kansai
  impact: high              # 对行程影响（交通管制、人流）
  worth_seeing: true        # 值得专门去看
  crowd_warning: true       # 人会非常多
  tips: "宵山（前夜祭）比山鉾巡行更适合拍照"
```

| 示例 | 时间 | 圈 |
|------|------|-----|
| 樱花季 | 3月下-4月中 | 关西/关东 |
| 红叶季 | 11月 | 关西/北海道 |
| 祇园祭 | 7月 | 关西 |
| 雪祭 | 2月 | 北海道 |
| 花火大会 | 7-8月 | 各地 |
| 广府庙会 | 春节 | 广府 |
| 薰衣草 | 6-7月 | 北海道 |
| 那拉提草原 | 6-8月 | 北疆 |

**决策链联动**：旅行日期撞上季节活动 → worth_seeing=true 时自动纳入行程 → 仅 crowd_warning 时提醒避开或调整时段。

### 限时活动（一次性）

```
limited_event:
  name: "teamLab 大阪特展"
  type: limited_time
  date_range: {start: "2026-04-01", end: "2026-09-30"}
  venue: "大阪中之岛美术馆"
  ticket_url: "https://..."
  sells_out: true
  booking_lead_days: 14
```

**采集**：定期从票务平台（猫眼/大麦/Ticket Pia/ぴあ）+ 场馆官网拉取。

**表单联动**：用户填了"已买XX门票"（booked_items）→ 系统围绕这个时间点安排前后行程。

## 七、数据更新机制

一次性爬不靠谱。餐厅关门/涨价/换菜单是常事。

### 滚动更新 + 事件驱动

| 策略 | 频率 | 做什么 |
|------|------|--------|
| **滚动复查** | 每天 5% 实体 | 按 `last_verified_at` 排序取最老的，20天完成一轮全量 |
| **关店监测** | 每天 | Google Maps "permanently_closed" / Tabelog "閉店" / 大众点评"已关闭" 自动标记 |
| **评分漂移** | 每周 | 评分变化超过 ±0.5 触发复查，可能需要调整质量档 |
| **季节切换** | 每季度 | 更新季节性营业时间、菜单、推荐理由 |
| **新店发现** | 每月 | 从新开业列表 + 小红书新帖抓新实体 → 待审核队列 |
| **用户反馈** | 实时 | 旅行后"这家店还开吗？体验如何？" → 回流修正 |
| **博主站点** | 按配置 | 你指定的 URL 按频率定期拉取 |

### 实现

不需要复杂系统。一个 cron job，每天：
1. 取 `last_verified_at` 最老的 N 个实体
2. 调 Google Maps / 大众点评 API 检查状态
3. 状态变化 → 更新 DB + 通知审核
4. 无变化 → 更新 `last_verified_at`

## 八、体验增强（超出基本推荐）

### 拍照时间优化

中国游客非常重视拍照。每个 S 级景点标注：

```
photo_tips:
  best_time: "6:00-7:30"          # 最佳光线
  best_spot: "从清水舞台西侧拍京都全景"
  avoid_time: "10:00-15:00"       # 人最多
  equipment: "广角"               # 建议设备
  seasonal_highlight: "11月红叶+舞台 = 最佳组合"
```

### 动线微优化

不是"去A、去B、去C"，是具体走法：
"去清水寺 → 沿二年坂走下来（5分钟）→ 看到八坂塔（拍照点）→ 花见小路 → 右转到午餐的店"

需要实体间的步行路线数据，不只是直线距离。

### 支付方式标注

```
payment_methods: [cash, card, paypay, wechat_pay, alipay]
```

连续安排现金店时系统自动提醒"建议提前在便利店ATM取现"。

### 排队预期管理

```
queue_estimate:
  weekday: "15min"
  weekend: "45min"
  holiday: "60min+"
  tips: "建议11:30前到，过了12点排队翻倍"
```

在攻略中直接写预计等待时间和建议到达时间。

### 人群日历

特定日期人暴多——日本公众假期（GW/盂兰盆/新年）、修学旅行季（5-6月/10-11月）、邮轮停靠日、国内黄金周。

生成时自动检查旅行日期是否撞上，调整景点时段或替换。

### 点菜建议

不只推荐去哪家，还说吃什么：
"招牌是炸猪排定食（¥1500），配味增汁，饭可以加。不吃炸物选刺身定食。"

比"一家不错的日料店"有用100倍。

## 九、实体 Schema 变更汇总

基于以上策略，实体需要新增/调整的字段：

```
entity_base 新增/调整:
  quality_tier: int(1-5)           # 取代 final_score 做选择
  budget_tier: enum(low/mid/high/ultra)  # 取代精确价格做匹配
  discovery_gem: bool              # 隐藏好店标记
  last_verified_at: datetime       # 上次验证时间
  recommendation_count_30d: int    # 近30天被推荐次数
  risk_flags: list[str]            # 避雷标签
  payment_methods: list[str]       # 支付方式

restaurant 新增:
  signature_dishes: list[str]      # 招牌菜 + 价格
  queue_estimate: dict             # 排队时长（工作日/周末/假日）
  booking_method: str              # 预约方式
  booking_url: str                 # 预约链接
  booking_lead_days: int           # 建议提前天数
  meal_style: str                  # 目的地餐/路线餐/快餐

poi 新增:
  photo_tips: dict                 # 拍照建议（时间/位置/设备）
  crowd_calendar: dict             # 人流量日历
  walking_routes_to: list[dict]    # 到相邻景点的步行路线

新表:
  seasonal_events                  # 季节性活动
  limited_events                   # 限时活动
  crawl_sources                    # 可配置爬虫源
  entity_recommendation_log        # 推荐计数（轮转用）
```
