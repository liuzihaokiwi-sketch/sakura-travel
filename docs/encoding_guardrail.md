# 编码防错守则

## 文档定位

本文档记录仓库内文本文件的编码守则、常见风险和最小检查动作。

目标不是解释所有编码理论，而是减少两类问题：

- 中文文档或注释被写成 mojibake
- 小改动因为整文件重写而放大成大面积乱码或换行噪音

## 当前判断

最近一次乱码问题，核心不是业务逻辑错误，而是文本读写链路失控。

主要风险通常来自：

1. 在 Windows / PowerShell 下整文件重写
2. 读写链路经过默认代码页而不是 UTF-8
3. 为了做小修改，使用了整文件覆盖式写回
4. 编辑器、终端、Git、脚本四者的编码设置不一致

## 仓库级默认规则

### 编码

- 所有文本文件默认使用 `UTF-8`
- 非必要不使用 `UTF-8 with BOM`
- Markdown、Python、TypeScript、JSON、YAML 都按 `UTF-8` 处理

### 换行

- 仓库默认使用 `LF`
- 不因本地系统是 Windows 就把仓库文本切回 `CRLF`

### 修改方式

- 优先做最小有效修改，不做无关整文件重写
- 对中文内容敏感的文件，避免使用会经过系统默认代码页的复制或重定向链路
- 修文档优先使用明确声明 UTF-8 的写法
- “最小修改”不等于继续在错误结构上打补丁

## 当前阶段禁止什么

当前阶段明确禁止补丁式治理：

- 不允许在旧前提上继续叠加 patch、补桥、临时胶水
- 不允许把 compatibility bridge 包装成主链完成
- 不允许靠 patch 掩盖 truth source、contract、source-of-truth object、main path 没修好的事实
- 遇到结构问题，优先修 truth source、contract、source-of-truth object、main path

在编码问题上，这条规则具体表现为：

- 禁止通过“先保留坏结构，再补一层”处理乱码
- 禁止通过管道 + 覆盖整文件的方式做恢复
- 必要时允许整份文档重写成正确 UTF-8 文本，但目标必须是恢复真实语义，而不是继续补丁式修补

## 已落地的仓库约束

当前仓库已增加：

- `/.editorconfig`
- `/.gitattributes`

它们分别负责：

- 统一编辑器默认编码与换行
- 统一 Git 工作树编码与文本换行

这两层不能百分之百消灭所有乱码来源，但已经能显著降低再次写坏的概率。

## 操作守则

### 1. 小改优先，但不在坏结构上续补

如果只是改几行：

- 优先局部修改
- 不要为了改几行把整份中文文档重新覆盖写回
- 但如果文件本身已经乱码或结构错误，应直接重写成正确文本，不保留错误层继续贴补丁

### 2. 明确使用 UTF-8

写文件时应显式指定 UTF-8。

在 PowerShell 中，优先使用带 `-Encoding UTF8` 的写法。

### 3. 避免高风险恢复方式

以下方式风险高，不应作为默认做法：

- `git show ... | Set-Content ...`
- 任何“管道输出 + 覆盖整个文件”的恢复动作
- 先在非 UTF-8 环境里打开，再原样保存中文文件

### 4. 中文改动后做最小检查

每次修改中文文档或注释后，至少做这三步检查：

1. 查看目标文件内容是否可正常显示中文
2. 查看 diff 是否出现与任务无关的大面积文本重写
3. 检查文件是否意外带入 BOM

## 建议检查命令

### 查看目标 diff

```powershell
git diff -- docs/architecture/10_l2_l3_closure_definition_package.md
```

### 检查 BOM

```powershell
python scripts/ci/check_text_encoding.py
```

## 需要立刻回退并重查的信号

出现以下任一情况，应立即停止继续修改并检查本次写回链路：

1. 目标只是小修，但 diff 出现整段中文大面积变化
2. 文件开头出现异常 BOM 痕迹或不可解释的前导字符
3. 文件能打开，但中文明显变成乱码
4. 本次改动没有业务价值，却引入了大量换行或编码噪音

## 最终规则

以后仓库内涉及中文文本的改动，默认遵守：

- UTF-8
- LF
- 最小有效修改
- 改后立即做编码巡检
- 不做补丁式修复

如果这些做不到，就不要继续扩大写入面。
