**A. 第二层当前链路组成**
- 主链是 worker 中的 city-circle 决策链：[generate_trip.py](d:/projects/projects/travel-ai/app/workers/jobs/generate_trip.py#L197)。当前真实顺序是 `TripProfile -> select_city_circle -> eligibility_gate -> precheck_gate -> compile_constraints -> rank_major_activities -> build_hotel_strategy -> build_route_skeleton -> secondary_fill -> meal_fill -> itinerary_fit -> build_itinerary_records -> generate_report_v2`，其中 Layer 2 的核心收敛点其实是 `constraints + day_frames + design_brief + generation_decisions + evidence_bundle`。
- 第二层的结构化决策能力已经存在于 [city_circle_selector.py](d:/projects/projects/travel-ai/app/domains/planning/city_circle_selector.py#L84)、[constraint_compiler.py](d:/projects/projects/travel-ai/app/domains/planning/constraint_compiler.py#L259)、[major_activity_ranker.py](d:/projects/projects/travel-ai/app/domains/planning/major_activity_ranker.py#L96)、[route_skeleton_builder.py](d:/projects/projects/travel-ai/app/domains/planning/route_skeleton_builder.py#L96)。
- 兼容层有两种。第一种是 staged fallback：[fallback_router.py](d:/projects/projects/travel-ai/app/domains/planning/fallback_router.py#L35) 定义了 `full_legacy / major_legacy / filler_legacy / section_adapter`，而 [generate_trip.py](d:/projects/projects/travel-ai/app/workers/jobs/generate_trip.py#L421) 在 `FULL_LEGACY` 和 `MAJOR_LEGACY` 时直接退回旧装配链。第二种是写入兼容：[itinerary_builder.py](d:/projects/projects/travel-ai/app/domains/planning/itinerary_builder.py#L35) 仍默认 `shadow`，说明新决策链在很多情况下还只是旁路写入，不是唯一主写入。
- 旧链仍然活着，而且不只是一条。`assemble_trip` 仍被 [generate_trip.py](d:/projects/projects/travel-ai/app/workers/jobs/generate_trip.py#L761)、[submissions.py](d:/projects/projects/travel-ai/app/api/submissions.py#L279)、[trips_generate.py](d:/projects/projects/travel-ai/app/api/trips_generate.py#L176) 直接调用；这意味着第二层现在不是“新链已接管”，而是“worker 主路径偏新、API 直调路径偏旧”。
- 旧输入假设驱动的链路主要有三处。第一处是 [quiz.py](d:/projects/projects/travel-ai/app/api/quiz.py#L164)，仍是 `quiz_v1` 的 `destination/duration_days/party_type/styles/travel_time`。第二处是 [workers/__main__.py](d:/projects/projects/travel-ai/app/workers/__main__.py#L281)，`normalize_trip_profile` 仍以 `cities/travel_start_date/interests/flight_info` 为核心输入，并把总天数算成 `sum(nights)+1`。第三处是 [fragment_pipeline.py](d:/projects/projects/travel-ai/app/domains/planning/fragment_pipeline.py#L55)，还保留从 `detail_forms` 失败后回退到 `quiz_submissions` 再默认 `tokyo` 的兼容逻辑。

**B. 新表单输入对第二层的影响**
- `profile / input normalization` 会被最先击中。当前 `TripProfile` 只有 `cities/travel_dates/party_type/budget_level/must_have_tags/avoid_tags/flight_info` 这类旧字段，[business.py](d:/projects/projects/travel-ai/app/db/models/business.py#L171)；而新表单需要一层新的 Layer 2 contract，至少要显式承载 `requested_city_circle`、`arrival_local_datetime`、`departure_local_datetime`、`companion_breakdown`、`budget_mode+budget_range`、`visited_places`、`do_not_go_places`、`booked_items`、`special_requirements`。
- `城市圈选择` 目前是“从 cities/airports/season/pace 推断最优圈”，不是“用户先选圈，系统再校验和细化”，见 [city_circle_selector.py](d:/projects/projects/travel-ai/app/domains/planning/city_circle_selector.py#L169)。新表单下，目标城市圈应成为强信号甚至首层约束，规划器的工作更像“圈内编排”和“圈内外冲突提示”，而不应继续把六个城市圈都当开放候选池。
- `skeleton / route planning` 会被准确落地/返程时间直接改写。当前 arrival/departure 语义一半来自 `flight_info`，一半来自 `arrival_shape` 和可选 `arrival_time`，[workers/__main__.py](d:/projects/projects/travel-ai/app/workers/__main__.py#L128)、[constraint_compiler.py](d:/projects/projects/travel-ai/app/domains/planning/constraint_compiler.py#L437)。新表单要求按落地后的在地时间计算，意味着 Day 1 / Last Day 不能再靠 `sum(nights)+1` 和模糊半天规则来推。
- `已预定内容约束` 当前几乎没有进入第二层主链。数据库 migration 曾经设计过 `booked_hotels / fixed_events / dont_want_places`，[20260322_010000_add_detail_forms.py](d:/projects/projects/travel-ai/app/db/migrations/versions/20260322_010000_add_detail_forms.py#L54)，但 ORM/API 现状并未真正接通这些字段，[detail_forms.py](d:/projects/projects/travel-ai/app/db/models/detail_forms.py#L73)、[detail_forms.py](d:/projects/projects/travel-ai/app/api/detail_forms.py#L40)。这意味着“已订酒店/餐厅/门票/城际交通”还没有成为 hotel strategy、day frame、slot locking 的正式输入。
- `去过 / 不想去的地方过滤` 当前只有弱版本。主链真正消费的是 `avoid_tags` 和回归脚本里的 `blocked_clusters / blocked_pois`，[constraint_compiler.py](d:/projects/projects/travel-ai/app/domains/planning/constraint_compiler.py#L269)、[constraint_compiler.py](d:/projects/projects/travel-ai/app/domains/planning/constraint_compiler.py#L281)。这足以支持“不要某类体验”，但还不等于“我去过清水寺，这次不要再安排”或“这次明确不去道顿堀”。
- `节奏 / 偏好 / 预算 / 同行结构` 当前已部分影响 ranker 和 skeleton，但粒度不够。现有输入只到 `party_type/budget_level/pace/has_children/has_elderly` 这一层，[major_activity_ranker.py](d:/projects/projects/travel-ai/app/domains/planning/major_activity_ranker.py#L96)、[route_skeleton_builder.py](d:/projects/projects/travel-ai/app/domains/planning/route_skeleton_builder.py#L96)。新表单里“成人/儿童/老人数量、早起接受度、夜间活动接受度、旅行感觉、预算方式/预算区间、换酒店接受度”都应进入约束编译或权重包，而不只是留在自由文本。
- `输出给第三层的 explain / evidence / decisions` 现在已经有骨架，但信息不完整。第二层会写 `generation_decisions` 和 `plan_metadata.evidence_bundle`，[decision_writer.py](d:/projects/projects/travel-ai/app/domains/planning/decision_writer.py#L31)、[generate_trip.py](d:/projects/projects/travel-ai/app/workers/jobs/generate_trip.py#L809)；`generate_report_v2` 也会读取这些信息来生成 `preference_fulfillment/skipped_options`，[report_generator.py](d:/projects/projects/travel-ai/app/domains/planning/report_generator.py#L1304)。但 [report_schema.py](d:/projects/projects/travel-ai/app/domains/planning/report_schema.py#L304) 里已有 `selection_evidence`，实际却没有被填充，这正是新 contract 进入第三层时最该补的一环。

- 验证链本身也过时。[detail_forms.py](d:/projects/projects/travel-ai/app/api/detail_forms.py#L404) 的 `validate_detail_form` 还在读 `pace_preference/arrival_date/arrival_time/departure_date` 这些字段，但当前 ORM 已经不是这套字段，[detail_forms.py](d:/projects/projects/travel-ai/app/db/models/detail_forms.py#L73)。这意味着“测试是否能 validate”本身都可能在验证一个漂移中的旧 contract。

**D. 第二层最先应该改的最小闭环**
- 最小闭环不应该是“从新表单一路打到 PDF”，而应该是“新表单真实输入 -> 第二层决策结果 -> 给第三层的稳定 handoff”。
- 我建议先锁定一个最小但真实的闭环：`目标城市圈 + 准确落地/返程时间 + 同行结构 + 预算边界 + 去过/不想去 + 已订酒店/固定事件`，进入第二层后至少要稳定产出 `normalized_layer2_input + selected_circle + compiled_constraints + hotel_strategy + day_frames + generation_decisions + evidence_bundle`。
- 这个闭环里必须先改的模块是 [detail_forms.py](d:/projects/projects/travel-ai/app/db/models/detail_forms.py#L43)、[detail_forms.py](d:/projects/projects/travel-ai/app/api/detail_forms.py#L40)、[business.py](d:/projects/projects/travel-ai/app/db/models/business.py#L171)、[workers/__main__.py](d:/projects/projects/travel-ai/app/workers/__main__.py#L281)、[city_circle_selector.py](d:/projects/projects/travel-ai/app/domains/planning/city_circle_selector.py#L84)、[constraint_compiler.py](d:/projects/projects/travel-ai/app/domains/planning/constraint_compiler.py#L259)、[hotel_base_builder.py](d:/projects/projects/travel-ai/app/domains/planning/hotel_base_builder.py)、[route_skeleton_builder.py](d:/projects/projects/travel-ai/app/domains/planning/route_skeleton_builder.py#L96)、[decision_writer.py](d:/projects/projects/travel-ai/app/domains/planning/decision_writer.py#L31)、[generate_trip.py](d:/projects/projects/travel-ai/app/workers/jobs/generate_trip.py#L197)。
- 这个闭环里先不要动的模块是 `secondary_filler / meal_flex_filler / itinerary_fit_scorer / report_generator_v2 / page pipeline`。原因很简单：这些模块吃的是第二层产出的 frame 和 decision，如果先把输入 contract、time semantics、locked booking、place exclusion 搞准，下游可以先继续复用。
- 这个最小闭环之所以最值先做，是因为它一旦打通，第二层就会第一次真正站在“六城市圈 + 表单优先 + 落地后规划 + 60页手册证据链”这条新边界上运转，而不是继续把旧 quiz/旧 detail form 的残留字段拼起来。

**E. 阶段建议**

**Phase 1：必须先做**
- 目标：冻结第二层输入 contract，并打通“输入 -> 决策 -> evidence”的最小闭环。
- 涉及文件 / 模块：[travel_form_multistep_frontend_v3.md](d:/projects/projects/travel-ai/docs/architecture/travel_form_multistep_frontend_v3.md#L71)、[detail_forms.py](d:/projects/projects/travel-ai/app/db/models/detail_forms.py#L43)、[detail_forms.py](d:/projects/projects/travel-ai/app/api/detail_forms.py#L40)、[business.py](d:/projects/projects/travel-ai/app/db/models/business.py#L171)、[workers/__main__.py](d:/projects/projects/travel-ai/app/workers/__main__.py#L281)、[city_circle_selector.py](d:/projects/projects/travel-ai/app/domains/planning/city_circle_selector.py#L84)、[constraint_compiler.py](d:/projects/projects/travel-ai/app/domains/planning/constraint_compiler.py#L259)、[route_skeleton_builder.py](d:/projects/projects/travel-ai/app/domains/planning/route_skeleton_builder.py#L96)、[generate_trip.py](d:/projects/projects/travel-ai/app/workers/jobs/generate_trip.py#L197)。
- 风险点：detail form migration、ORM、API 三套字段已经漂移；如果不先收敛字段命名，后面每改一个 planner 模块都会继续加临时映射。
- 为什么排这个顺序：第二层所有后续改造都依赖一个稳定输入 contract；没有它，后面的 ranker、filler、eval 只会继续围绕旧字段补洞。

**Phase 2：紧接着做**
- 目标：把新输入真正传到“选点、排程、解释”里，并替换过时的 mock/eval。
- 涉及文件 / 模块：[major_activity_ranker.py](d:/projects/projects/travel-ai/app/domains/planning/major_activity_ranker.py#L96)、[eligibility_gate.py](d:/projects/projects/travel-ai/app/domains/planning/eligibility_gate.py)、[secondary_filler.py](d:/projects/projects/travel-ai/app/domains/planning/secondary_filler.py)、[meal_flex_filler.py](d:/projects/projects/travel-ai/app/domains/planning/meal_flex_filler.py)、[report_generator.py](d:/projects/projects/travel-ai/app/domains/planning/report_generator.py#L1106)、[report_schema.py](d:/projects/projects/travel-ai/app/domains/planning/report_schema.py#L304)、[planning_grader.py](d:/projects/projects/travel-ai/app/evals/graders/planning_grader.py#L159)、[offline_eval.py](d:/projects/projects/travel-ai/app/domains/evaluation/offline_eval.py#L101)、[test_cases.py](d:/projects/projects/travel-ai/scripts/test_cases.py#L12)、[run_regression.py](d:/projects/projects/travel-ai/scripts/run_regression.py#L43)、[test_full_pipeline.py](d:/projects/projects/travel-ai/tests/e2e/test_full_pipeline.py#L52)。
- 风险点：如果 Phase 2 先于 Phase 1，会把 `visited/do_not_go/booked_items` 继续塞进 `avoid_tags/free_text` 这种临时渠道，后续更难清理。
- 为什么排这个顺序：只有 Phase 1 先收敛 contract，Phase 2 的测试和评测替换才不会一边换 case、一边换字段定义。

**Phase 3：可以后做**
- 目标：清理旧入口、统一兼容层、决定哪些旧链保留为长期 fallback。
- 涉及文件 / 模块：[submissions.py](d:/projects/projects/travel-ai/app/api/submissions.py#L161)、[trips_generate.py](d:/projects/projects/travel-ai/app/api/trips_generate.py#L176)、[quiz.py](d:/projects/projects/travel-ai/app/api/quiz.py#L164)、[fragment_pipeline.py](d:/projects/projects/travel-ai/app/domains/planning/fragment_pipeline.py#L55)、[fallback_router.py](d:/projects/projects/travel-ai/app/domains/planning/fallback_router.py#L35)、[itinerary_builder.py](d:/projects/projects/travel-ai/app/domains/planning/itinerary_builder.py#L35)、[10_e2e_testing_guide.md](d:/projects/projects/travel-ai/docs/10_e2e_testing_guide.md#L4)、[08_eval_cases.md](d:/projects/projects/travel-ai/docs/08_eval_cases.md#L135)。
- 风险点：这一阶段涉及入口、运营、旧模板链、测试指南，爆炸半径最大，不适合作为起手动作。
- 为什么排这个顺序：先证明新第二层 contract 能稳定产出，再决定哪些旧入口要迁、哪些兼容层要留，返工最少。

**最值得先动的 3 个点**
1. 先定义一个新的 Layer 2 输入 contract，把 `requested_city_circle`、`arrival/departure local datetime`、`visited/do_not_go`、`booked_items` 从“备注”升级成正式字段。
2. 先把 `normalize_trip_profile / city_circle_selector / constraint_compiler / hotel+skeleton` 接到这份新 contract 上，最早只要求稳定产出 `day_frames + decisions + evidence_bundle`。
3. 先重写 `scripts/test_cases.py`、`scripts/run_regression.py`、`tests/e2e/test_full_pipeline.py` 这一批输入样本；不先换测试输入，第二层改造会一直被旧字段假象拖着走。
