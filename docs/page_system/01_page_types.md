# 最终攻略页型总表与整本框架

## 1. 总目标
最终攻略不是一篇长文章，也不是一串组件自然流下去后再靠打印分页。
它应当是一本“按页构成的定制攻略”。

核心效果：
- 用户第一眼就知道这份攻略是认真设计过的
- 用户可以快速找到自己关心的页
- 重要内容不会挤在一起
- 长行程不会因为页数变多而显得重复
- Web 与 PDF 可以共用同一套页型语法

---

## 2. 页型系统总览

### A. 固定前置页
这些页几乎每本攻略都要有，用来建立“这份攻略有完整思路”。

1. 封面页
2. 目录页
3. 这次旅行如何设计 / 偏好兑现页
4. 主要活动总表页
5. 大路线总览页
6. 酒店策略总览页
7. 关键预订与时间窗页
8. 出发前指引页
9. 动态注意事项页

### B. 章节页
这些页用于把长行程拆成阶段，而不是直接 Day1 到 Day14 平铺。

1. 城市圈章节页
2. 阶段节奏页（可选）

### C. 每日核心页
这些页与天数强相关，是行程的主体。

1. 每日执行页

### D. 高价值专题页
这些页负责把真正值钱的内容独立展开，给用户明确感知价值。

1. 主要活动页
2. 酒店详情页
3. 餐厅详情页
4. 拍摄主题页
5. 交通页
6. 调整 / Plan B 页
7. 顺路补充页

---

## 3. 不同天数怎么组合

### 3–5 天
适合“展开型攻略”。

组合逻辑：
- 固定前置页完整保留
- 每天 1 页执行
- 主要活动大多单独展开
- 餐厅 / 拍摄页展开比例高
- 不太需要太多章节页

### 6–8 天
适合“平衡型攻略”。

组合逻辑：
- 固定前置页完整保留
- 每天 1 页执行
- 主要活动仍单独展开
- 次要餐厅 / 次要拍摄开始按章节合并
- 可能出现 1–2 个城市圈章节页

### 9–14 天
适合“章节型攻略”。

组合逻辑：
- 固定前置页完整保留
- 每天仍只有 1 页执行，不再天天机械加很多附页
- 每个城市圈 / 阶段先有章节页
- 高价值内容按章节策展
- 次要内容更多进入章节附页，而不是天天拆细页

---

## 4. 一页只承担一种阅读任务

### 允许的任务类型
- 决策理解
- 路线理解
- 执行落地
- 对象判断
- 视觉种草
- 风险提醒

### 禁止的混排
- 不把“总策略 + 酒店细节 + 时间轴”塞在同一页
- 不把“餐厅 + 拍摄 + 购物”塞在同一页
- 不把“执行信息 + 大段抒情文案”塞在同一页
- 不把“注意事项 + 亮点展开”塞在同一页

---

## 5. 页型与对象的对应规则

### 整页对象
以下对象默认值得独立成整页：
- 主要活动
- 主要酒店
- 主要餐厅
- 主要拍摄主题
- 复杂交通日
- 复杂 Plan B
- 城市圈章节开场

### 半页 / 合并对象
以下对象可以合并：
- 次要酒店
- 次要餐厅
- 次要拍摄点
- 顺路购物
- 可变活动
- 雨天替补
- 小型注意事项

### 强制整页条件
满足任意一条即整页：
- 它本身是用户付费感知的重要来源
- 它决定路线或酒店
- 它有明显时间窗
- 它有高决策价值
- 它容易出错，必须单独讲明白
- 它具有高情绪价值，值得“认真呈现”

---

## 6. 一整本攻略的推荐顺序

### 前置部分
1. 封面
2. 目录
3. 这次旅行如何设计 / 偏好兑现
4. 主要活动总表
5. 大路线总览
6. 酒店策略总览
7. 关键预订与时间窗
8. 出发前指引
9. 动态注意事项

### 主体部分
10. 城市圈 / 阶段章节页
11. 主要活动页
12. 酒店 / 餐厅 / 拍摄 / 交通等专题页
13. 每日执行页
14. 章节内次要补充页

### 收尾部分
15. 旅行中动态提醒或应变建议（如需要）
16. 必要的附录页（少量）

---

## 7. 页数不是目标，节奏才是目标

### 不要做的事
- 为了显得厚而机械重复
- 为了控制页数把内容硬塞进一页
- 为了统一结构，让所有天数都输出完全相同的页序

### 应该做的事
- 让短行程显得精致、完整、可看完
- 让长行程显得分段清晰、有层次、不疲劳
- 让重点内容自然“冒出来”

---

## 8. 组合优先级
生成 page plan 时，按这个优先级决定哪些页先保留：

1. 固定前置页
2. 每日执行页
3. 主要活动页
4. 酒店详情页
5. 主要餐厅页
6. 主要拍摄页
7. 交通页
8. 调整 / Plan B 页
9. 顺路补充页

当页数需要控制时，优先合并 7–9，不合并 1–6。
