## 文档第一性原理

这份文档回答"**怎么采集 + 整理真实可信的数据**"(餐厅/酒店/景点/店铺/事实层)。
- 不回答"数据字段长什么样"(看 `03_数据契约/SCHEMA.md`)
- 不回答"模板怎么写"(看 `04_操作SOP/模板写作.md`)
- 不回答"每个地区用哪些数据源"(看对应地区层,如 `japan/日本特有规则.md`)

读完应该能独立完成一条数据(景点/餐厅/酒店/店铺)从发现到入库的完整流程,且产出的数据可上生产。

### 🔗 横向联动(采集过程中顺手做的事)

采集数据时会遇到强传播性的冷知识/避坑点(某景点重建故事、某餐厅隐藏菜品、某个时段免费入场、某店铺搬迁但导航还没改)——**带小红书传播潜力的,顺手记**:

- 发现这类素材 → 顺手填 `marketing/{地区}/素材库.md`(主题:`本地冷知识` / `餐厅隐藏` / `避坑警示` 等)

**判断标准**:这条冷知识能让小红书读者"哇我之前不知道" = 值得记;只是常规信息(票价/地址) = 留 live_facts 即可,不需要丢素材库。

## 数据的精髓(用户视角,唯一判断锚)

数据最终通过模板到达旅行者。判断一条数据好不好,看两种感受——

> **① 用户照着这条数据去了现场,一切和本子上写的一样。**
>
> **② 本子上这条推荐,让用户觉得"这一家只有懂当地的人才会挑出来"。**

第一句是**下限**:数字、地址、开门时间、付款方式——错一个,整本手账本的信任就崩。
第二句是**上限**:选得平庸 = 和免费攻略一样 = 不值钱;选得有见地 = 感到被专家选中 = 值 298。

### 两句话拆解

**"现场和本子一样"**——
- 票价、开门时间、周几休、预约要求、付款方式全部对
- 路线走得通,店铺真在开,预约真能预约上
- 数字没有被 AI 推断的"看起来像真的"

**"只有懂当地的人才挑得出来"**——
- 不是"评分最高的" = 那是机器能做的
- 是"在符合筛选的里面,挑出最对的那一家"
- 中国游客口味、预算、场景、时段都被考虑过
- 负峰值被预判(服务差/价格陷阱/观光化严重的 → 不进池)

**两句合起来 = 用户拿到手账本后翻到餐厅页的那一刻**:他愿不愿意信这页的推荐,直接走进去点招牌。

### 用这两句话判断(灰色区域的判断标尺)

| 场景 | 问什么 |
|---|---|
| 这条数据要不要查 | 用户到现场会不会因为这个数字不对而崩?(不是"字段要不要填") |
| 要不要补这个来源 | 懂当地的人会不会看这个来源?(不是"来源越多越好") |
| Tabelog 3.8 vs 3.5 选哪个 | 中国游客去了哪个更不会后悔?(不是"分高的") |
| 要不要给这条写 editor_note | 用户看到这条,缺这句话会不会做错事?(不是"字段有没有填") |
| 小众景点要不要入池 | 朋友会带我去吗?还是"这个一般般"?(不是"score 够不够 3.0") |

**选让用户"信这一页"的那个答案**。

### 写完后的质量验证

用户**带着这本手账本走完一趟,回家后愿不愿意把这本推给下一个去的朋友**——这就是数据做到了:

- 推不出去(太多细节错) → 下限没守住
- 推出去但朋友说"这些网上都有" → 下限守了但上限没到
- 朋友翻了直接买一本 → 精髓做到了

### 所有采集铁律都是这两句话的展开

| 铁律 | 服务哪一面 |
|---|---|
| AI 不写事实(第一红线) | 防止"看起来像真的"摧毁"现场和本子一样" |
| verified 分级 | 让用户翻开时的信任分级对应真实可靠性 |
| 三轴判断(quality/traveler_fit/execution) | 让"懂当地的人挑的"不是口号,是实证 |
| 三层池(发现→入围→终选) | 终选才是"挑出来的那一家",前两层是"都看过了" |
| 允许稀疏 | 宁可空着不给,也不给"还行"的 → 保护"专家挑的"信任感 |
| 风险标签 | 替用户预判了他预判不到的坑 |
| N3 铁律(budget ≠ quality) | Tabelog 3.8 拉面和 3.5 怀石不是一档 → 挑错了就辜负用户 |
| editor_note 写永远成立的话 | 模板层讲行程,实体层讲永远真相——两层都不骗用户 |

**忘了哪条铁律不要紧,只要采集时带着"这条数据会不会让用户到现场被打脸""这条推荐够不够专家级"的姿态,多数铁律会自然遵守**。记住全部铁律但没这姿态 = 合规的烂数据。

## 一、第一红线:AI 不写事实,AI 只做判断

**违反此规则产生误导性数据,比没数据更危险**。

### AI 禁止做的

- 写 quality_evidence / traveler_fit_evidence / execution_evidence(事实性证据)
- 补写评分、价格、坐标、营业时间等可从真实来源获取的字段
- 用 Structured Outputs / JSON schema 生成"看起来像真实来源支持"的内容
  - (Structured Outputs 只保证格式,不保证事实是真的)

### AI 可以做的(判断,不是事实)

- one_line_editorial_note / why_selected / skip_if
- grade 判断 + selection_tags
- 负向信号识别(基于喂给它的真实证据)
- 只对最终入选的 S/A 级或关键边界项写编辑注释

### evidence 字段的正确产生方式

```
quality_evidence:
  → 规则化生成,直接引用真实信号:
    餐厅: "Tabelog {score},{city}×{cuisine}类前{pct}%"
    酒店: "MICHELIN {n} Keys / 楽天 ★{rating}({count}件)"
    景点: "japan-guide {level},{sub_type}类推荐"
  → 不让 AI 自由发挥

traveler_fit_evidence:
  → 从携程/小红书真实抓取后提炼
  → 只对 shortlist(selected+borderline)补抓,不做全量
  → 没有真实来源就填 null,不编

execution_evidence:
  → 从官网/Google Maps 抓取营业时间、预约、排队信息
  → 规则化提炼,不让 AI 编写
  → 没有真实来源就填 null
```

### 自检规则

每次写"事实"前停一秒问:"我这条是真搜过的,还是从上下文推断?"推断 → 去真搜 / 或标注"未核实"。**绝不把推断写成事实**。

**适用范围**:不只 JSON 数据字段,也包括文档里的元数据(数据源 coverage、WebFetch 可用性、更新节奏等)。

## 二、可信度分级

| 级别 | 定义 | 能否上生产 |
|---|---|---|
| `verified` | 2+ 个 P0/P1 源交叉验证 | ✅ 可 |
| `cross_checked` | 三轴中至少两轴有真实来源覆盖 | ✅ 可,标注 |
| `single_source` | 只有一个真实来源(如仅 japan-guide,无携程/小红书/Google) | ✅ 可,标注 |
| `ai_generated` | 无任何真实来源 | ❌ 不可 |

**关西教训**:`spots_ledger` 里大量条目只有 `source="japan-guide"`,却被标为 `cross_checked` → 错误。单源必须标 `single_source`,等补了 traveler/execution 真实来源才能升级。

**cross_checked 的正确定义**:**三轴中至少两轴有真实来源覆盖**,不是"两个来源"。

---

## 三、三轴判断模型

**终选不靠"几个 P0 源确认",靠三个判断轴**:

| 判断轴 | 回答什么 | 说明 |
|---|---|---|
| **quality** | 在该类中是否足够强 | 品质/专业性/口碑 |
| **traveler_fit** | 中国游客实际满意度 | 不是"好吃",是"中国游客去了不会后悔" |
| **execution** | 能不能去 | 营业状态/预约/排队/定休 |

**每个终选条目必须三轴都有实证**。只有 quality 没有 traveler_fit → 可能是"本地人爱但中国人不适应"(比如口味太重、排队文化不同)。

## 四、数据源优先级分级

```
P0(权威源): 可作为 quality 轴唯一来源
P1(辅助源): 需配合 P0,对中国游客极重要(traveler_fit 主信号)
P2(参考源): 交叉验证、独立站 indie_quotes
P3(兜底): AI 知识库 — 仅用于结构化分类、推荐语,不用于事实性数据
```

**通用数据源(不分地区)**:
- Google Maps API(execution 通用,$17/1000 请求,必用 field mask)

**地区专属数据源**:看各地区的"特有规则"文档(如 `japan/日本特有规则.md`)。

## 五、三层池架构

```
发现池(尽量不漏)
  - 五语搜索全开(简中+繁体台+繁体港+日文+英文) — 仅新城市圈首次采集/新类目时
  - 任何一个可信数据源提及即进入
  - 数量: 终选的 5-10 倍
       ↓
入围池(交叉验证)
  - 至少 2 个独立数据源确认
  - 核验语言收窄: 日文 + 官方 + 简中
  - 数量: 终选的 2-3 倍
       ↓
终选池(编辑精选)
  - 判断标准: "如果朋友第一次去,我会专门推荐这个吗?"
  - 品类内相对排名(样本 ≥ 15 取 top 10-20%,小样本用编辑判断)
  - 同一决策位最多 3 家(首推/稳妥替代/机动备选)
  - 允许稀疏(某区域某预算没好选项就空着,不补"还行"的)
  - 检查负向信号
```

### 5.1 发现池 → 入围池:归一化步骤(必做)

> **关西教训**:多源合并后 cuisine_type 75+ 种混乱、酒店 price_level 85% 被默认值覆盖、OTA 评分/JPY 价格等强信号在合并时被丢掉 → 无法直接跑选择模型。

**合并时必须保留的字段(不可丢弃)**:
- 餐厅: name_ja, city_code, area, cuisine_type, budget_tier, tabelog_score, michelin, source
- 酒店: name_ja, city_code, area, hotel_type, price_level, source + ranking_info(luxury 源) + brief_note(含 OTA 评分/JPY 价格) + key_features
- 景点: name_en, name_ja, city_code, area, japan_guide_level, main_type, sub_type

**从原始文件回溯提取的信号**:
- 酒店 mid/budget 源:从 brief_note 解析 OTA 星级、评论数、JPY 夜价
  - 正则:`★(\d\.\d)\((\d+)件\)` → ota_rating, ota_review_count
  - 正则:`1泊(\d+)円〜` → nightly_jpy_min
- 酒店 luxury 源:从 ranking_info 提取 Michelin Keys / Forbes stars
- 餐厅 各源:tabelog_score 和 michelin 必须带过来,不可丢

**归一化步骤(合并后,选择模型前)**:

| 步骤 | 操作 |
|---|---|
| N1 | cuisine_type 归一化 → 标准菜系码(见 SCHEMA),维护 `cuisine_mapping.json` |
| N2 | area → corridor 映射(用 `corridor_definitions.json` 文本匹配) |
| N3 | budget_tier 细化(加 premium 层) |
| N4 | 酒店 price_level 修正(用回溯的 JPY 价格重新分层,不用合并时的默认值) |
| N5 | 酒店 hotel_type 补全(从 name_ja 推断) |

### 5.2 N3 铁律:budget_tier ≠ quality_score

**错误**:Tabelog 3.8+ 且非快餐类 → premium
**正确**:按已知价格/菜系场景推断

**budget_tier 推断优先级**:
1. 已有 JPY 价格 → 直接按阈值分层
2. Michelin 非空 → luxury
3. 菜系场景推断(弱信号):
   - 怀石/kaiseki/割烹/铁板烧/河豚 → premium
   - 拉面/乌冬/咖喱/丼/takoyaki → mid/budget
4. 无法判断 → 保留原值,标 `tier_confidence: "inferred"`

**原因**:Tabelog 评分以 3.0 为基准、0.5 为刻度,反映**口碑强弱不反映消费层级**。3.8 的拉面店可能人均 1000 円(mid),3.5 的怀石可能人均 15000(premium)。

### 5.3 N4 酒店 price_level 修正

```
JPY >= 30000/晚 → luxury
JPY 15000-30000 → expensive
JPY 8000-15000 → moderate
JPY < 8000 → budget
无价格时: Michelin Keys → luxury; ryokan+onsen → expensive; business_hotel → moderate
```

## 六、效率规则(不降质量的前提下)

### 规则 1:模型做选品判断,脚本补齐结构字段

**模型负责(需要判断力)**:
- 候选列表筛选
- why_selected / skip_if / grade_reason(编辑判断)
- grade 判断 + selection_tags
- 负向信号识别(基于喂给它的真实证据摘要)

**脚本/真实来源补齐(机械性)**:
- 坐标(Google Places API)
- 价格(OTA API/爬虫)
- 营业时间(Google Places / 官网)
- 基础结构字段(id, city_code, corridor_tags 等)
- JSON 完整格式生成(种子 CSV → 完整 JSON)
- quality_evidence(规则模板 + 真实信号)
- traveler_fit_evidence(携程/小红书真实抓取后提炼)
- execution_evidence(官网/Google Maps 真实信息)

**这是最大的单点 token 节约**:模型产出种子 CSV(选品+判断),脚本扩展为完整 JSON。

### 规则 2:S/A 深审,B/C 轻审

| 等级 | 审核深度 | 审核内容 |
|---|---|---|
| S/A | 完整三轴审查 | quality + traveler_fit + execution 全验证,负向信号审查,替代品比较,编辑辩护 |
| B/C | 真实性 + 执行性 | 确认"是真的、还在开、能去"即可,不写长篇编辑辩护 |

**用户感知品牌力的是 S/A 级骨架推荐,不是每个 B/C 条目的推荐理由**。

### 规则 3:API 分层取数

不要一上来就查全量细节,按阶段逐层加深:

```
发现层(批量,低成本):
  - Google Places: 只要 name + rating + place_id + user_ratings_total
  - 楽天: Keyword/Area/Ranking API
  - 携程: 列表搜索

入围层(中等,只对入围候选):
  - Google Places: + address + opening_hours + price_level
  - 楽天: + 评分详情 + 价格带
  - 一休: 评分排名

终选层(完整,只对终选条目):
  - Google Places: Full Detail(field mask 指定需要的字段)
  - 楽天: Detail API + Vacant API
  - 一休: 完整设施信息 + 含餐信息
  - 官网: 预约方式、定休日确认
```

**Google Places 必须用 field mask**,跨 SKU 字段会按最高 SKU 计费。

### 规则 4:模型吃证据摘要,不吃原始网页

```
错误: 把整页 Tabelog 评论、整页攻略文章喂给模型
正确: 先脚本提取结构化摘要,再喂给模型
```

每个候选的 evidence block(8-12 行):
```
name_ja: 菊乃井本店
quality: Tabelog 4.52 / 百名店 2024 / 米其林三星
traveler_fit: 携程 4.8(1200 评论) / 小红书提及频率高
execution: 需预约(difficulty:hard) / 周一定休
negative: 价格极高 / 需提前 1 个月
editorial: 京都怀石最高峰,首次来京都的 foodie 必体验
```

模型只需要这个摘要就能做选品判断,不需要读原始网页。

### 规则 5:城市级公共事实抽成一次性资产

每个城市圈只做一次,后续每条数据引用不重复搜索:

```
城市级 research pack(一次性产出):
  - taxonomy.json: 分类体系、regions、画像加成
  - corridor_definitions.json: 走廊定义、连接关系、交通时间
  - indie_sites_evaluation.json: 独立站评估(10-15 个核心站)
  - seasonal_calendar.json: 旺季日历、节庆活动
  - city_cuisine_map: 城市特有菜系和代表店概览
  - city_hotel_landscape: 各区域酒店价位分布概览
```

每条数据只引用 pack 中的结论,不重新解释"京都是什么样的城市"。

### 规则 6:候选发现设停机规则

**精选不是学术穷举**。发现阶段的停机条件(满足任一即停):
- 某组连续新增 10 个候选,无 1 个能进入入围池
- 某组已有 3 个高质量终选 + 2 个备选,召回足够
- 新增来源只是重复提及已有候选,无增量信息
- 成熟类目(如京都怀石)已从主源获得充足候选

### 规则 7:搜索语言效率

**五语搜索(简中 + 繁体台 + 繁体港 + 日文 + 英文)只在以下场景全开**:
- 新城市圈首次采集
- 不熟悉的类目/菜系
- 小众目的地(资料稀缺)

成熟城市圈的成熟类目,默认**日文主源 + 中文 traveler 源**起步,够用就不开五语。

## 七、风险标签优先于推荐

**推荐一家差餐厅的伤害 >> 错过一家好餐厅的损失**。三层过滤:

- **硬黑名单**:永不推荐(食物安全/严重负面/已停业)
- **风险标签**:默认过滤(服务差/价格陷阱/观光化严重),用户可放开
- **正常池**:推荐候选

每条条目加 `risk_flags` 字段,有任何风险点都要标,**宁可少推荐**。

## 九、editor_note 写作标准(绑在实体上)

editor_note **写永远成立的话,不写某一天的特定安排**:
- ✅ "清水寺必须 8 点前进,8 点以后游客激增" — 永远成立,写在 editor_note
- ❌ "今天先去清水寺再去八坂神社" — 模板的事,不写在实体
- ✅ "子安塔背后可拍到舞台全景,游客通常不走这条路" — 景点自身属性,永远成立

**review 三段结构**(景点/餐厅):
```
亮点: 为什么值得去、核心体验是什么
怎么玩/必点必知: 具体玩法、进入方式、关键时间点、拍照机位
缺点: 人多 / 票价贵 / 季节限制 / 体验不稳定等
```

## 十、季节升级处理(景点 seasonal_notes)

如果一个景点平时是 A 级(score 3.x),樱花期变成"必去":
- 景点本身 score 保留平时的 A 级值
- 在 `seasonal_notes` 加 key(如 `"樱花期"`),value 写自然语言说明
- 实体只描述"这季节有什么看点",**不写具体日期**——日期由 L0 事实层(`events/`)维护
- 需要独立一天才值得的季节场景 → 走独立季节模板

```json
{
  "id": "kyo_kiyomizudera",
  "seasonal_notes": {
    "樱花期": "早开门 + 子安塔背后可拍舞台全景樱花,建议 7:30 前到",
    "红叶期": "夜间特别拝観,舞台从下方仰拍可见整片红叶包裹的清水寺"
  }
}
```

## 十一、小众内容入池判断流程

**非 S 级景点(score < 4.0)进入内容池前必须通过以下四步**,不通过则不入池:

**第一步:内容质量**
- 有没有不可替代的体验?能不能一句话说清楚"为什么值得去"?说不清楚不入池。

**第二步:顺路性和时间成本**
- 顺路且时间短(< 1.5h) → 候选加入常规行程的次要 slot
- 不顺路或时间长 → 候选作为"小众专题日"的组成部分

**第三步:与同区域其他次要景点比较**
- 加入常规行程前,和同一天同一区域的其他 B/C 级候选比较——**谁的体验更独特、更难在别处替代,谁才进 slot**
- 不是"这个也不错"就入,是"这个比那个更值"才入

**第四步:适合哪类人群**
- 明确标注适合的 party_type 或 vibe,不是所有用户都推
- 例:酒造参观 → 适合 default/couple,不适合 family_with_children
- 无明确人群的小众内容 → 默认不进常规行程,只作为 relaxed 节奏的备选

四步都通过才写入内容池,并在 editor_note 里注明"适合顺路加入 XX 主线"或"适合小众专题日"。

## 十三、score 评分规则(0-5 一档一分)

| 档位 | 分值 | 含义 |
|---|---|---|
| S 级 | 4.0-5.0 | 城市代表性,不去会遗憾 |
| A 级 | 3.0-3.9 | 重要搭配,值得专程,但次于 S |
| B 级 | 2.0-2.9 | 顺路可以,不必专程 |
| C 级 | 1.0-1.9 | 补充选项,只在时间富余考虑 |

**score 是编辑综合评分**,不是单一来源评分。综合多个 P0-P2 数据源判断后给出,可标来源(如 `"tabelog:3.72,michelin:bib"` 作为参考),但 score 字段本身是 0-5 浮点数。

## 十四、硬字段保留 5 条标准

满足任意一条才保留为硬字段,不满足的写进 editor_note 或 review:
1. 表单按它过滤
2. 用户看得到的展示分类
3. 客观事实
4. 硬约束机械判定需要
5. 系统运维需要

## 十五、数据源专属 SOP 放哪里

每个地区的具体数据源(Tabelog / 携程 / 小红书 / 楽天 / 一休 等的 URL 模式/访问方式/注意事项)放**地区层**:
- `japan/日本特有规则.md` — 日本专属数据源
- `china/xxx/` — 中国数据源(未来)
- `europe/xxx/` — 欧洲数据源

这样日本源(Tabelog/楽天)不和中国源(携程/小红书)混,跨地区查找清晰。
