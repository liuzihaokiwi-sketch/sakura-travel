## ⚠️ 2026-04-22 D36+D37 字段定稿覆盖（先读这段，下方旧细则按本段筛除）

### 新规则一：字段权威以 [SCHEMA.md](../03_数据契约/SCHEMA.md) 为准

本文档下面的"字段规则"段落是 2026-04-20 之前写的，部分字段已在 D36 砍掉或挪走。**任何字段定义冲突以 [SCHEMA.md](../03_数据契约/SCHEMA.md) 为准**。本文档只保留"怎么写好"的方法论。

### 新规则二：禁止随意扩展字段

写模板时想加新字段，先回答 SCHEMA §⚠️ 的 3 个问题：
1. 消费方是谁（渲染代码 / AI 装配 / 写作者）？
2. 能不能用现有 `note` / `description` / `curators_notes` 表达？
3. 是不是装配层的事（用户档位/天数/人群/A-B/酒店升档）？

**说不出消费方 = 不要加。能用 note 表达 = 不要加。装配层的事 = 去 assembly/ markdown 写**。

### 新规则三：模板字段终态（D36+D37，取代下方旧字段细则）

**顶层必填 7**：`template_id` / `label` / `variant_label` / `applicable_dates` / `note` / `description` / `slots`

**可选**：`hotel_area_note`（偏远才写） / `curators_notes`（有才写） / `contingencies`（5 子项有内容才写）

**已砍不要再写**：`template_kind`（文件夹位置表达）/ `selectable_tag`（装配 index.md 维护）/ `night_options`（同文件夹 `night.json` 绑定）/ `score` / `min_days` / `day_type` / `exclusive_with` / `core_entities`

**slots 结构（D37）**：按时间段分组的数组，每段 `main`（必做）+ `optional`（推荐）：

```json
"slots": [
  {
    "time": "08:00-10:00",
    "main": [
      {"type": "poi", "entity": "kyo_arashiyama_bamboo"},
      {"type": "poi", "entity": "kyo_tenryuji"}
    ],
    "optional": [
      {"type": "poi", "entity": "kyo_kameyama_park", "note": "体力好的用户加项"}
    ]
  }
]
```

**slot 元素**：极简 4 类，按 type 区分

| type | 字段 |
|---|---|
| `poi` | type + entity（+ note 见下方硬规） |
| `meal` | type + meal_type + meal_area（**永不写 entity**） |
| `hotel` | type + hotel_area（**永不写 entity**） |
| `transport` | type |
| `free_time` | type + theme + options_note（**不挂 entity**，用于自由漫游/留白时段） |

**slot 内不要 `priority` / `fun_fact` / `cuisine_hint` / `max_wait_min` / `stay_nights` / `time`**——time 在外层时段对象。

### 新规则三点七：slot.note 必填场景

**同一 entity 在同一天的 slots 里出现 ≥2 次时**（如渡月桥下午漫步 + 黄昏 peak-end），每次 slot 必须写 note 区分用途。否则装配/渲染分不清两次停留。

### 新规则三点八：main 不能为空

每个时段对象的 main 数组**必须 ≥1 个 slot**。如果某段没必做内容（自由漫游/留白），用 `free_time` slot 填 main，**不要把所有东西塞 optional 让 main 空着**。

### 新规则三点九：自由漫游段用 free_time slot

原来用长 note + poi 塞杂项（咖啡店/足湯）的做法**废止**。专门的 `free_time` slot：

```json
{
  "type": "free_time",
  "theme": "下午自由漫游",
  "options_note": "三方向任选：(A) 渡月桥南岸 % ARABICA 买咖啡河堤坐；(B) 嵐電嵐山站足湯泡 15 分钟；(C) 嵯峨鳥居本江户街道漫步"
}
```

### 新规则三点十：contingencies 砍 swap_candidates

不一天都下雨，用户雨时躲进 `indoor_backup` 推荐的本地室内景点/咖啡馆即可。不需要换整个模板。swap_candidates 字段废止。

### 新规则三点五：缓冲时间 SOP（D37）

`slot.time` 时长 = 景点最佳游玩时间 + 缓冲：

| 最佳游玩 | 缓冲 |
|---|---|
| < 1h | +30 分钟 |
| 1-2h | +30-45 分钟 |
| 2-3h | +45-60 分钟 |
| > 3h | +60 分钟 |

缓冲给用户"不一定掐点到、不一定准点结束" 的稳态。

### 新规则三点六：optional 推荐 SOP（D37）

`slot.optional` 数组写的判断：

- 推荐景点必须步行可达（≤15 分钟）或 1-2 站电车
- 主景点 + 推荐合并后总时间不超过 3 小时
- 推荐景点必须真值得（不是凑数）
- 装配按用户密度决定用不用：packed 全用 / balanced 用 1 个 / relaxed 不用
- **如果整个动线都是高强度**（如东山），单独做一个 `relaxed.json` 变体（写一套完整轻节奏 slots），不用靠 optional 来退档

### 新规则四：景点相关全部挪事实层

`photo_spots` / `fun_facts` / `nearby_shops` 不再写在模板里，**写在事实层 entity**。模板 slot 引用 entity_id，渲染时从事实层取。

事实层判断标准：
- `photo_spots`：这景点真有"游客通常不走的角度"才写
- `fun_facts`：朋友会忍不住讲的冷知识才写（不是百科摘要）
- `nearby_shops`：懂当地朋友会指给你看的小店才写

**有比较好的才写，不凑数**。

### 新规则四点一：四层职责硬规（D38）

设计宪法第二条"内容归属对象"的具体落地。写每一段话前先问：这句话的"事实主权"属于哪一层？错位了就挪到该去的地方。

| 层 | 职责 | 写什么 | 不写什么 |
|---|---|---|---|
| **entity** | 事实权威 | 票价 / 时长 / 地址 / 排队 / 设施 / 开放时间 / 官方特色 | 动线判断 / 装配逻辑 |
| **slot.main.note** | 动线即时判断 | 为什么这时间到这里 / 怎么接上下一段 / 即时人流提醒 | 原始事实（可引用不复述） |
| **slot.optional.note** | 取舍依据 | 为什么是 optional / 什么密度档位加 / 加了影响什么 | 景点本身描述（去 entity） |
| **整天 note**（模板正文） | 骨架逻辑（300-500 字） | 核心体验为什么成立 / 路线 / 节奏 / 情绪 / 旺季应对 / 硬限制 | 每个景点单独介绍 |
| **curators_notes** | 印刷级巧思（30-50 字/条） | 金句级神动线 / 冷知识 / 极其具体的小技巧 | 通用判断 / 大段说明 |
| **index.md** | 变体层 | 变体清单 / 互斥 / 前置 / 跨动线连接 / 装配说明 | 单模板内部逻辑 |
| **transport.md** | 动线交通 | 门到门表 / 延伸连接 / 停运窗口 | 内部步行段（slot 顺序即路线） |

**边界规则**：slot note **可以引用**事实（如"人流峰值到 15:00"）作为判断依据，但**不复述**原始数据（票价 / 地址 / 开放时间等 entity 已有字段）。判断依赖的事实简略提一句，完整数据让下游查 entity。

**同一信息只在一层存在**——entity 改了不回头改 50 个模板。这是 D36 宪法的核心落地。权威表格见 [SCHEMA.md](../03_数据契约/SCHEMA.md) §"四层职责硬规"。

### 新规则五：装配相关全部挪 assembly/ markdown

打分（core_experience/audience_bonus/execution_risk）/ min_days / day_type / exclusive_with / night_options / A-B 餐厅 / 酒店升档——**全在 [japan/kansai/assembly/](../../japan/kansai/assembly/) markdown**。

模板自己干净，只管动线。

### 新规则五点一：标准日模板时间表（D38）

**所有 `pace_type=adaptive` 的动线模板**按以下基准时间表写 slots：

- **基准 = 9 点出门档 + 春秋季**（peak-end 锚定 17:00-18:00）
- **slots time 写绝对时间**，装配按用户出门档（8/9/10）+ 当天季节自动调整

| 时段 | 活动 | 时长 |
|---|---|---|
| 9:00 | 出门 | — |
| 9:00-10:00 | 过渡到第一个景点（含步行+候车+车上+末段步行）| 1 小时 |
| 10:00-12:00 | 上午景点段（1-2 段）| 2 小时 |
| 12:00-12:30 | 过渡到午餐区 | 30 分钟 |
| 12:30-14:00 | 午餐（meal slot） | 90 分钟 |
| 14:00-14:30 | 过渡到下午景点 | 30 分钟 |
| 14:30-16:30 | 下午第一段景点 | 2 小时 |
| 16:30-17:30 | 弹性段（free_time / 小景点 / 咖啡）| 1 小时 |
| 17:30-18:30 | 黄昏 peak-end | 1 小时 |
| 18:30-19:00 | 过渡到晚餐 | 30 分钟 |
| 19:00-21:00 | 晚餐（meal slot）| 120 分钟 |
| 21:00 | 回酒店 / 接 night.json | — |

**3 档共用一套景点列表**——模板写 9 点档，装配按用户档调整：

| 出门档 | 装配如何调 |
|---|---|
| 8 点档 | 整天往前推 1 小时；上午段拉长（每个景点 +30 分钟慢游 / 加 1 段 free_time）；下午多 1 段景点 |
| 9 点档 | 模板原样 |
| 10 点档 | 整天往后推 1 小时；**不砍尾**（用户选 10 点档本身接受慢节奏晚玩到 21:00+）；真正挤不下时压上午段，不动弹性段和 peak-end |

**peak-end 跟季节走，不跟出门档走**：

| 季节 | 日落 | peak-end 实际时段 |
|---|---|---|
| 深冬（12-2 月）| 17:00 | 16:00-17:00（提前 1.5 小时，压上午段或午餐时长）|
| 早春/晚秋 | 17:30-18:00 | 16:30-17:30 |
| 春秋（樱花/红叶）| 18:00-18:30 | 17:00-18:00（基准）|
| 夏季 | 19:00-19:30 | 18:00-19:00（推后 0.5-1 小时）|

冬季模板自动比夏季紧凑——这本身符合产品直觉，写模板时不用纠结"冬季 10 点档行不行"，装配会自动处理。

### 新规则五点一点一：9 点档模板必自查 8/10 档不崩

写完 9 点档 slots 后，作者必须快速验算两档不崩再交付：

**8 点档自查**：
- 第一个景点 9:00 前开门吗？（天龙寺 8:30 开门 → 8 点档用户 9:00 到可以进 ✅；清水寺 6:00 开门 → 没问题 ✅；妙心寺 9:00 开门 → 8 点档要先走 citywalk/早餐再进 ⚠️）
- 多出来的 1 小时加到上午段哪里合理？加一段 free_time 还是延长某景点？

**10 点档自查**：
- 整体后移 1 小时后，深冬能撑到 peak-end 吗？（peak-end 深冬 16:00-17:00，10 点档上午从 11:00 起算，上午段+午餐+下午段必须压进 5 小时）
- 如果深冬 10 点档挤不下，压上午段或缩午餐，**不砍弹性段和 peak-end**
- 晚餐后移到 20:00-22:00 正常（10 点档用户本身接受晚玩）

自查不通过说明模板设计有问题，不是靠装配兜底——装配只做线性平移。

### 新规则五点一点二：hotel_area_note 只写动线特有信息

`hotel_area_note` 字段**不重复写全局规则**（9 点档基准、adaptive 默认等 D38 全局内容）。

只写**这条动线特有**的：
- 门到门时长（具体数字，如"京都站 40 分钟 / 四条 45 分钟 / 梅田 75 分钟"）
- 住本地加成（如"岚山本地旅馆步行 5-10 分钟省整个过渡段"）
- 动线特殊的酒店区筛选（如"城崎/有马温泉模板要求住本地温泉旅馆，不住市区"）

反例（冗余）：
- ❌ "9:00 出门是标准日基准"（D38 全局）
- ❌ "peak-end 跟季节走"（模板写作.md 已写）

### 新规则五点一点二点一：特殊时段模板出门档无效

某些模板的体验核心锚定**绝对时间**而非出门档偏移——例如:

- **傍晚/黄昏半日**(伏见稻荷傍晚上山看日落+夜景渐变):16:00 出发 → 日落 + 夜景是 peak,日落时间由季节决定,用户选 8/9/10 档对本模板无意义
- **夜樱/夜间点灯特别拝观**(清水寺春秋夜间拝观/高台寺夜樱):18:00 入园绝对时间受官方特别拝观时刻锁定
- **夜祭/祇园祭山鉾巡行**:祭典时刻写在事件数据里,用户出门档无法挪
- **早朝特别拝观**:寺院冬季早朝 7:00 开门本身就是卖点,属 fixed_early 范畴本期不做

这类模板在 `applicable_dates` 或 `label` 里明确标记时段,装配引擎按模板 slots 绝对时间装配,**忽略用户出门档**。写作时:

- slots 写绝对时间(如 16:00-22:00)
- note 里说明"本模板出门档无效,按绝对时间执行"
- 其他 D38 规则(过渡 1 小时/景点间 30 分钟/午晚餐基线/peak-end 季节锚定)**仍然适用**

### 新规则五点一点二点二：写模板前必查自有数据

写模板 note / slot note 前,**先查 `japan/kansai/entities/{kyoto,osaka,other}.json` 和 `assembly/restaurants/data/`、`assembly/hotels/data/` 是否已有相关数据**,优先引用/对齐,不凭空重写。

**先查清单(写每个动线前)**:
1. 涉及的每个 entity——读其 `notes` 子标题(## 排队 / ## 季节 / ## 避坑 / ## 拍照位置 / ## 顺路小店 / ## 衔接 / ## 票价指引 / ## 冷知识 / ## 当年临时)和 `live_facts`(closed_dates_2026 / temp_closures / sub_facility_availability / crowd_warning / sunset_reference_2026)
2. 餐厅区域——meal_area 对应的 `assembly/restaurants/data/` 城市 JSON 里有哪些具体店 / ab_role(A-B 角色)/ budget_tier / reservation_difficulty / risk_flags / notes
3. 酒店区域——若动线跨 hotel_area(温泉一泊/离岛两日),查 `assembly/hotels/data/` 对应 pool

**写作原则**:
- 模板 note / slot note 的焦点是**动线层**——怎么组合、顺序为什么这样、时段为什么这样选、A/B 怎么分。**景点细节(排队 / 避坑 / 冷知识 / 拍照)归 entity,不在模板重写**(D36 第二宪法)
- 若模板 note 想强调某个关键点,用"**见 entity_id 的 notes ## 排队**"这种**引用式表达**而非自己重写
- 若 entity 数据和你要写的动线有冲突(例如你想让用户 10 点档到景点但 entity notes 说"10 点后人流爆满连停步都难"),**要么调整动线(改 8 点档或下午档),要么在模板 note 明确承认"人流高峰"是用户接受的代价**
- 反例(凭空写):"樱花盛期 10 点档人墙" —— entity notes 已经有更精确的"10:00 后本堂舞台+仁王门一带挤到停步困难"
- 正例(引用):"上午段到清水寺,樱花盛期按 entity 排队说明接受 10 点档人流客观存在,packed 档可选 8 点档避峰"

**先查花费 2-3 分钟,避免凭空写和 entity 冲突+重复信息,后续阶段 3 深化时不返工**。

### 新规则五点一点三：时间线/景点时长冲突必须搜索

写模板时遇到**结构性冲突**——景点实际时长塞不下 D38 时间表、等位预估不准、季节特别窗口冲突——不能自己机械硬塞，必须搜真实方案：

| 场景 | 动作 |
|---|---|
| 上午段 2 小时塞不下 2 个景点 | opencli 搜爆款游记 3-5 条看真实游客排法 + WebSearch 查官方建议停留 |
| 餐厅等位时长不清楚 | opencli 搜该餐厅/区域 collects 高的笔记看等位描述 |
| 季节窗口冲突（开门时间/关门调整）| WebSearch 查官方公告 |
| 微小兼容调整（±15 分钟 slot 边界、次序小改）| 自己决定无需搜索 |

判断区别：结构性冲突必搜，微调不搜。搜完再定"延长 slot / 砍景点 / 换顺序 / 补过渡"，不是默认塞进去。

### 新规则五点二：pace_type 装配分类（D39·2026-04-24 重写）

> **本节替代 D38 的 fixed_early 早期写法**。详见 SCHEMA §1.3.1。

#### pace_type 三类（详见 [plans/写作规范.md §四](../../japan/kansai/plans/写作规范.md) + [assembly/engine.md §二](../../japan/kansai/assembly/engine.md)）

- **adaptive**（默认）：用户主导节奏·按出门档（8/9/10）平移·模板写 9 点档基准
- **fixed_early**：体验主导节奏·**四要素齐全**才贴·极度稀缺
- **deep_stay**：两日连住·子类 `onsen` / `deep_local`

#### fixed_early 四要素硬规（缺一不做）

1. **出片价值 + 早晚差异大**（值得出片+早到晚到体验明显不同；全年或季节都行）
2. **晨光 + 空景独家画面**
3. **景点实际开放**（早到时景点开门或 24h 公共空间·不做「门口冻 3 小时」的反人类模板）
4. **市区景点 + 交通短 + 能回酒店休息**（景点离酒店 15-30 分钟内·远郊如岚山/高雄/金阁/吉野永不合格）

**合格候选（参考）**：清水寺 / 伏见稻荷 / 八坂塔·三年坂 / 祇园一带（市区东山）。

**不合格（永远不做 fixed_early）**：岚山 / 高雄 / 吉野山 / 高野山 / 金阁寺 / 天龙寺 / 多数塔头寺院。

#### 判定树（写模板时用）

```
1. 必须住一晚才能做到核心体验吗？
   是 → deep_stay
   
2. 四要素齐全吗？
   是 → fixed_early
   
3. 其他全部 → adaptive
   （班次/抽签/祭典/夜樱等时间锚定·写在 note 和 slots·不单独成类）
```

#### 当前阶段默认写法

- 绝大多数动线按 **adaptive + 9 点档基准**写 slots
- fixed_early 极度克制·只做四要素都极致的市区景点
- deep_stay 只限「两日连住才成立」的模板（自带晨光/夜间/慢节奏片段）
- **时间锚定（班次/抽签/祭典/夜樱/夜间点灯/日出卡光）归 adaptive**·写在 note 和 slots 绝对时间·装配引擎按 ±20 分钟弹性匹配

#### 预订/班次/限期类 slot 的保险缓冲

当日有时间约束（小火车班次 / 瑠璃光院抽签 / 餐厅预订 / 夜间限期 / 日出机位 等）时：

- **从用户出门到约束时刻·保险缓冲约 1 小时**（交通+换乘+取票/候场+延误容忍）
- **不过度**：缓冲 2 小时以上用户车站干等违反体验
- 装配引擎层处理用户出门时间 ±20 分钟微调（SCHEMA §1.3.3）·模板 JSON 不写「±20 分钟可调」文字
- slot 按推荐班次/场次写绝对时间·note 补充「装配按用户档匹配最近保险时段」

**例：嵯峨野小火车班次（每小时 02 分发车·9:02/10:02/...）**
- 8 点档用户 → **9:02 班次**（8:00 出门·约 60 分钟缓冲）
- 9 点档 → **10:02 班次**
- 10 点档 → **11:02 班次**

### 新规则五点二点零：2026-04-24 标签回滚

2026-04-24 阶段 2.4 曾将 8 个模板贴 fixed_early（arashiyama 2/3/5/6/7/9/10 + okazaki/2）。按 D39 新规校准：

- **arashiyama 2/3/6/7 + okazaki/2 → adaptive**（岚山远郊+瑠璃光院八濑远郊·要素 D 不合格；7 与 okazaki/2 用 `time_sensitivity=soft` 表达班次/场次锚定）
- **arashiyama 9/10 → deep_stay**（子类 `deep_local`·住岚山本地才成立·D2 清晨早起作为「住本地推荐加分段」非强制）
- **arashiyama 5 保留 fixed_early（唯一例外）**·核心叙事是「6:30 渡月桥镜面 → 7:30 竹林光柱 → 8:30 曹源池倒影」的三机位晨光轮转，模板设计逻辑本身就是为「愿意为出片早起」的用户服务。虽然岚山交通 1 小时不满足要素 D 的「能回酒店休息」理想形态，但模板叙事不依赖「拍完回酒店睡」，下午自动进入 adaptive 松节奏即可。此例外仅适用于**以晨光轨迹编排的出片模板**，不扩展到其他远郊景点。
- 未来新的 fixed_early 候选（市区景点清水/伏见稻荷/八坂塔等）下轮按四要素挑

此回滚与 5 例外详见 DECISIONS D39。

### 新规则五点三：time_sensitivity 三档（D39·2026-04-24）

**每个模板顶层写 `time_sensitivity` 字段**，回答「当天时间约束有多硬」。默认 `flexible` 可省略。

| 档 | 判准 | 举例 |
|---|---|---|
| `flexible` | 早去晚去都无所谓 | 锦市场 / 商店街 / 鸭川 / 大部分神社 |
| `soft` · 光线/人流型 | 晚去光线差 / 部分关门 / 错过最佳但仍成立 | 金阁寺 17:00 关 / 伏见稻荷黄昏 / 岚山竹林 10 点后人多 / 日出日落机位 |
| `soft` · 定期班次型 | 一定周期一班/一场，各档位能对齐不同班次 | 嵯峨野小火车 1 小时一班 / 全天多场次抽签 / 半小时一班观光船 |
| `hard` | 固定时刻不可改，错过 = 产品当天崩 | 祇园祭巡行 9:00 / 五山送火 20:00 / USJ 快速券指定场次 / 一天一场预约餐厅 / 烟火大会 |

**硬规**：
- soft / hard 时必填 `time_sensitivity_note` 一句约束说明
- flexible 模板两个字段都省略
- **定期班次型怎么写**：模板 slot 按 9 点档基准写实际出门+班次时刻（如小火车写 `08:40-09:30` 出门 + `10:02-11:02` 班次），装配引擎按档位自动平移各档对应不同班次
- hard 模板装配层按约束时刻判用户档兼容性，不兼容直接换模板
- soft 模板装配引擎可用 ±20 分钟微调 start

### 新规则五点二点一：温泉酒店次日（D2）写法（D38）

温泉一泊 2 日模板（有马 / 城崎 / 高野山等）的 D2 不安排核心景点，按以下时间线模板写：

| 时段 | 活动 | 时长 |
|---|---|---|
| 09:00-10:30 | 旅馆朝食 + 温泉（朝食常含会席多道，需慢慢吃）| 90 分钟 |
| 10:30-12:00 | 自由活动（free_time slot，theme="温泉镇晨光漫步"，options_note 推荐 2-3 个步行可达的小景点 / 河岸 / 神社 / 街道）| 90 分钟 |
| 12:00-12:30 | 退房 + 过渡到午餐 | 30 分钟 |
| 12:30-14:00 | 午餐 | 90 分钟 |
| 14:00-... | 进入正常下午行程或返程 | — |

**关键约束**：
- D2 朝食是温泉旅馆的**核心体验**之一（会席式日式朝食 / 温泉粥），必须留 90 分钟
- 自由活动的周边推荐**必须步行可达**（≤15 分钟），不能再坐车去远景点
- **D2 午餐选店原则：避开重口/油腻类**——朝食可能已经吃得很饱，午餐推荐：
  - ✅ 蕎麦 / 乌冬 / 茶室点心 / 轻食三明治 / 海鲜清淡丼 / 抹茶套餐
  - ❌ 烤肉 / 拉面（重汤）/ 厚切炸物 / 怀石（量大）/ 寿喜烧
- 装配挑 D2 午餐餐厅时硬约束 `meal_role=light` 或 `cuisine_style=clean`

### 新规则五点三：用餐时长 + 餐厅位置基线（D38）

**meal slot 时长基线**：
- 午餐 90 分钟（含过渡 + 排队 ≤30 分钟 + 用餐 60 分钟）
- 晚餐 120 分钟

**meal slot 不写具体餐厅**（D36 已定），只写：
- `meal_period`: lunch / dinner
- `meal_area`: 跟当天动线绑定的区域

**餐厅位置约束**（写 meal slot 时遵守）：
- **午餐**：只能在上午景点附近 OR 下午景点附近 二选一（不允许跨区）
- **晚餐**：4 类位置（晚上景点商圈 / 酒店区 / 路径商圈；酒店附近无餐饮供给的不考虑）

详见 [餐厅装配.md](../../japan/kansai/assembly/restaurants/restaurants.md) A/B 规则。

### 新规则五点四：transport.md 必须写"门到门"时间（D38）

之前 transport.md 写"京都站 → 嵯峨岚山 JR 14 分钟"是**车上时间**——错了。真实门到门要算：
出酒店步行 5-10 分钟 + 候车 5-10 分钟 + 车上 14 分钟 + 末段步行（站到景点）15 分钟 = **40-60 分钟**。

写 transport.md 时**必须补"门到门"列**，模板 slots 的"过渡到第一个景点"按门到门时间排（默认 1 小时）。

## 模板的精髓(用户视角,唯一判断锚)

好模板让用户产生两种感受——

> **① 这不是一本攻略,是为我量身准备的一本日记。**
>
> **② 每翻一页,像有个懂当地的朋友在耳边跟我讲这段旅行。**

这两句都是**用户视角**,不是创作者视角。判断一个模板好不好,不问"我是不是按 SOP 做了",问"用户翻开会不会产生这两种感受"。

### 两句话拆解

**"不是攻略,是一本日记"**——
- "不是攻略" = 没有公共信息的疏远感,没有"网上都有"的平庸
- "量身准备" = 用户感到被针对:人群、季节、节奏都是为他挑的,不是 one-size-fits-all
- "一本日记" = 有情感、有温度、有叙事、值得保留,不是一次性清单

**"像朋友在耳边讲"**——
- 文案有温度,不是冷冰冰罗列
- 给线索不给结论——朋友不会强迫你,只会告诉你"你可以试试"
- 带本地视角——朋友知道你不知道的
- 有信任感——"他说这家值得就是值得"

**两句合起来 = 用户付 298 买的东西**。他不是买信息(信息免费),是买"被懂得被照顾的感受"。

### 用这两句话判断(灰色区域的判断标尺)

两个方案都合规,选哪个?回到这两句:

| 场景 | 问什么 |
|---|---|
| 选餐厅 A/B | 朋友会推哪家给我?(不是"评分更高的") |
| 排 slot 顺序 | 朋友会带我按什么顺序走?(不是"地理上最顺") |
| 写 note | 朋友会在这个时刻跟我说什么?(不是"字数够不够") |
| 加不加这个景点 | 朋友会带我去这个地方吗?还是"算了这个人多"?(不是"score 够不够") |
| fun_fact 写什么 | 朋友走到这里会忍不住告诉我什么?(不是"有字段就要填") |

**选用户能感到"这是朋友会做的事"的那个方案**。

### 写完后的质量验证

用户走完这一天,**能跟朋友讲出一个朋友不知道的具体瞬间**——就说明精髓做到了:

- 讲不出 → 没做到"一本日记"(没仪式感 / 没记忆点)
- 讲的是小红书都有的 → 没做到"朋友在耳边讲"(只是搬运)
- 讲完朋友惊讶 → 做到了,属于合格
- 讲完朋友说"我也要去" → 做出了"朋友的私人推荐"那份信任感,是伟大

### 22 条 SOP 都是这两句话的展开

| SOP | 服务哪一面 |
|---|---|
| Peak-end(S1) | 让日记有"值得记住的那一页" |
| 10% 惊喜(`01` 原则 2) | 让"朋友推荐"感超过"网上都有"的平庸 |
| 负峰值规避(S2) | 朋友不会让你踩坑 |
| B 要真惊(`01` 原则 3) | 朋友会推一个你没想到的方向 |
| 文案给线索(`01` 原则 9) | 朋友不会强推,只会告诉你"你可以试试" |
| 动线无折返(规则 11) | 朋友不会让你绕远路 |
| 事实核查(B1) | 朋友说的数不会错 |
| 人味(`01` 原则 6) | 朋友会带你去有温度的地方,不是空景 |
| 自由漫游段(S6) | 朋友知道"这里该留白",还会给你方向 |
| curators_notes(规则 5) | 朋友真正的小巧思,不是免责声明 |

**每一条都是这两句话在一个场景的落地**。记住 22 条但用户没那两种感受 = 合规但平庸;带着这两种感受的姿态写,多数 SOP 会自然遵守。

## Part 0:模板集合策展(动单个模板前必过)

### 0.1 为什么这一层最重要

单个模板做到 100 分,但集合里缺了"伏见稻荷""宇治"这类必看类型,整个季节产品就失败。反过来,集合齐全但每个模板 70 分,用户体验是"一趟完整的京都旅行,每天都还行"——**集合的完备性是下限,单个模板深度是上限。下限守不住,上限再高也救不回**。

### 0.2 模板三层架构(每个模板放哪层)

```
Layer 1 · full_day — 一整天一条独立动线,有 peak-end,6-9 个 slot
Layer 2 · half_day — 3-4 小时,可独立可拼装(morning / afternoon)
Layer 3 · night_module — 2-3 小时,固定挂在白天模板之后
```

| 场景 | 放哪层 | 理由 |
|---|---|---|
| 清水东山整天 | full_day | 一条完整动线,有 peak-end |
| 伏见稻荷登顶 2-3h | half_day | 撑不起一整天,但独立成立 |
| 宇治平等院+抹茶 | half_day | 3-4h 最合理,跨城留半天交通 |
| 二条城 2.5h + 御苑梅花 1.5h | **拆成 2 个 half_day** | 勉强凑 full_day 不如拆开灵活组合 |
| GEAR 夜秀 | night_module | 占 2h,白天不该被它绑架 |
| 到达/离开/换城日 | **不做独立模板** | 用 half_day × 2 + 交通拼装 |
| 恢复日 | **不做** | 产品未上线,用户真实行为未知 |

**核心洞察**:
- 功能性日(到达/离开/换城)不做独立模板——是 `half_day × 2 + 交通` 的装配结果
- half_day 是架构核心——既是独立体验又是拼装零件,做厚了全能应对
- full_day 只给真正"一整天才成立"的动线

### 0.3 季节架构(以关西为例,7 档)

**目录即装配逻辑**:
- `common/{city}/` = 全年基准,每条动线一份
- `{season}/{city}/` = 只在"季节玩法和公共版完全不同"时存在
- 装配:查季节目录 → 有用季节版 / 无回退 common

**关西 7 档**(2026-04 从 10 档收敛):
| 目录 | 窗口 | 主题 |
|---|---|---|
| `common` | 全年 | 伏见稻荷/宇治/金阁/东山/岚山等基准 |
| `early_spring` | 3/1-3/24 | 梅花主场 |
| `sakura` | 3/25-4/15 | 樱花 |
| `tsuyu` | 6/1-7/5 | 紫阳花梅雨 |
| `matsuri_peak_summer` | 7/10-7/24 | 祇园祭 |
| `summer_low` | 7/25-8/25 | 大文字送火锚定 |
| `koyo` | 11/5-12/5 | 红叶 |
| `deep_winter` | 12/20-2/28 | 雪景/初诣/灯饰 |

**判断规则**:
- 季节对这条动线玩法影响 < 30% → 公共层(加 season_notes 即可)
- 季节对动线玩法影响 > 30% → 季节层独立模板
- 季节独特但仅 1-3 天触发(大文字/宵山/巡行) → `template_kind: special_date`

**不建空目录**——某季节没有独立玩法就不建。等将来真出现 Tier B 以上特化再独立。

### 0.4 独立模板 vs 变种

同一个城市同季节,什么时候两个模板是"两条独立动线",什么时候是"一条动线的两种人群走法"?

**独立动线(独立模板)**:
- 地理不重叠 / 顺序不同 / peak-end 位置不同
- 例:东山标准(清水→二年坂→白川) vs 祇园慢线(花见小路→鸭川),地理相邻但动线完全不同

**变种(同模板加人群字段)**:
- 同动线,只某些 slot 具体内容不同(午餐升档 / 下午段替换)
- 60% 以上 slot 可复用

**现阶段策略**:
- 能独立就独立写,overlay 字段**先不启用**(装配层还没做)
- 独立动线集合稳定 + Opus 装配成熟后再做 overlay
- 相信 Opus——人群差异很多可由 Opus 读主模板 + 用户画像自行调整

### 0.5 集合完备性的真问题

不是"11 类体验每类打勾",是**"首次来关西 4-5 天的用户,能用这些模板装配出完整行程吗?"**

**必须能回答**:
- 用户到达关空/伊丹,下午在京都有什么可走?(→ 需要京都南部 half_day,如伏见稻荷)
- 最后一天中午飞机,上午在京都有什么可走?(→ 京都市中心 half_day,如锦市场)
- Day2 全天在京都,最想走哪条动线?(→ 东山 full_day)
- 喜欢传统文化想体验 2 小时工艺?(→ craft half_day)
- 想看夜景/夜秀?(→ night_module)

某类用户诉求**装不出完整行程** = 集合有洞。

**判断模板"要不要做"的 3 个问题**:
1. 去掉这个模板,集合能不能装配出典型用户行程?(能 → 不必要;不能 → 必要)
2. 这个模板在这季节有独特玩法吗?(没有 → 公共层;有 → 季节层)
3. 这个模板放哪层?(full / half / night_module / 不做)

### 0.6 Tier 分层(决定每个模板投入精力)

**Tier A 王牌(5-6 个)— 完整版**:用户高频选中,核心体验分 45+。所有字段 + 深度搜索 + photo_spots 4 + nearby_shops 5 + fun_fact 2-3。投入 30-40 分钟/个。

**Tier B 稳妥(8-10 个)— 精简版**:主线候补、主题/人群专属,分 35-45。所有必填 + photo_spots 2-3 + nearby_shops 3 + fun_fact 1-2。投入 15-20 分钟/个。

**Tier C 小众/补位(4-5 个)— 最小版**:特定偏好才触发,分 20-35。所有必填 + photo_spots 1-2(可略) + nearby_shops 略 + fun_fact 0-1。投入 10 分钟/个。

**意义**:集中精力在高频模板,避免"平均用力都做到 60 分"的全盘平庸。

### 0.7 产品边界(不走模板,走客服定制)

不做模板:
- 第二次来关西(要小众深度,模板是大众广谱)
- 奢华游(¥3000+/晚,走定制)
- 已订酒店(酒店反向约束动线)
- 预约失败场景(任天堂没抽中 — 客服处理)
- 恢复日(产品未上线,用户行为未知)

**原则**:服务 < 10% 特殊用户的功能默认留给客服定制线,主产品标准化。

- [ ] 该季节的模板清单,每个标注 **层级**(full/half/night_module) + **归属**(公共层/季节层) + **Tier**(A/B/C)
- [ ] 每个模板能用一句话回答:"承担什么用户诉求?去掉集合能不能装配完整行程?"
- [ ] 用"典型用户 4-5 天行程"跑一遍,验证能装出完整行程

**不能回答"为什么做这个模板"的,从集合里删掉,不要动手写。**

## Part 2:写 slot 的硬规

### 方法论:我们做的是信息整合 + 好的思考

- 不自己发明规则——参考市面上最好的方案(小红书爆款、japan-guide、Black Tomato、Tabelog 百名店、米其林、BRUTUS/&Premium 等杂志的策展范式)
- 规则的每一条都是从真实实践中沉淀的,不是"理论上应该有"
- 质量优先,简洁高效——冗余的规则会稀释真正重要的那几条

### 合并说明(2026-04-20)

原 22 条里有 6 条高度重叠,合并为 1 条("规则 2 文案密度与边界"):
- 原 2(不写 what-if) / 原 3(季节边界) / 原 4(人群边界) / 原 6(note 字数) / 原 9(hotel_area_note 字数) / 原 10(season_notes 精简)

**合并后结构**:Part 2 = 16 条字段/动线硬规 + Part 2 补充 6 条 peak-end 深化 + Part 2 扩充 3 条策展细节。

### 1. 时间 slot 用范围不用死点

- ✅ `11:30-13:30 午餐`(2h 窗口)
- ❌ `12:00-13:00 午餐` + note "晚 1h 也 OK"
- 用户知道自己几点饿,不需要模板给每种情况免责

### 2. 文案密度与边界(合并原 2/3/4/6/9/10)

一条规则统一处理 5 件事——写作时都用"朋友耳边讲"的口吻判断:

**(a) 不写 what-if 免责**(原 2):
- ❌ "推荐 8 点到,但不用硬起,晚些也能看"
- ❌ "如果体力不够可以跳过"
- ✅ "推荐 8-9 点到清水,人最少"(点到即止)
- 朋友不会给你每种情况免责。P2 优先级本身 = 弹性机制,不需要文字再解释

**(b) 季节/人群边界严守**(原 3/4):
- 早春模板**只写早春**,不写红叶季/樱花季方案
- 主线模板不给人群变种("情侣怎样/闺蜜怎样"),由专属人群模板承担
- 跨季节/跨人群的事,由对应目录的模板管

**(c) 字数密度**(原 6/9/10):
- slot 的 `note`: 50-100 字(超过砍,冷知识挪 fun_fact,应急挪 contingencies)
- `hotel_area_note`: 2-3 句推荐语气,不用"务必/必须"
- `season_notes`: 只留 seasonal_advantage + weather_tip,不冗余
- 没信息就不写(空字段比凑字数好)

**统一判断**:朋友在耳边讲这段,他会说这么多吗?多了 = 冗余,少了 = 信息不够。

### 5. curators_notes 规则

- **顶层字段**,给用户看的小巧思,不是内部工作总结
- **数组 1-4 条**,每条**一句话**(30-50 字)
- 必须**正向小巧思**(我们发现了什么/建议怎么玩),不是**负向免责**(不用硬起/做不到也 OK)
- 没有真巧思不硬凑

```json
"curators_notes": [
  "推荐 8-9 点到清水,人最少",
  "清水参道上的店贵 30%——午餐往下走到高台寺",
  "白川南通黄昏是 peak-end 不是晚餐,晚餐放松吃"
]
```

### 6. note 精简(50-100 字)

- 删"以官方为准""出发前查官网"的重复提醒
- 关键信息留 note,背景/冷知识挪 fun_fact,应急挪 contingencies

### 7. 餐厅处理

- 模板 slot 里**只写 `meal_area` 和 `cuisine_hint`**,不写具体餐厅 A/B
- 具体餐厅在独立的**区域餐厅池文件**,按 area 对齐
- 季节限定菜/川床/牡蛎期是**餐厅的季节属性**,写在餐厅那边

### 8. 不写具体年份活动

- 跨年复用的模板不写 "2026 年蜷川实花展"/"2026/3/14 伏见酒祭"
- 改写 "每年 3 月中旬有伏见酒祭,当年日期查官方"
- 近年每年有但每年不同的:"近年北野每年早春有梅苑艺术联名展,当年查官网"

### 9. hotel_area_note 简洁

- 几条到达方式 + 关键推荐时间,不超过 2-3 句
- 不写"务必""这是成立的关键",改推荐语气

### 10. season_notes 精简

- 保留:seasonal_advantage(为什么这季节来)+ weather_tip(温度)
- 删:跨季 note(夜樱/樱花季/红叶季)+ data_notes 重复(票价/官网重复)

### 11. 动线无折返(硬规)

**slot 顺序必须按真实地理位置串,不能来回折返**。扫 slot 顺序时用 Google Maps / 地图记忆把每个点的坐标拉出来看:A→B→C→D 应是**单向前进**(东南到西北 / 山上到山下 / 顺时针)。

**典型反复信号**:B 在 A 西北,下一站 C 又回到 A 东南 → 折返,重排。

**验证**:画地理直线或弧线,把 slot 顺序贴上去看是不是顺的。

**反例(清水东山)**:高台寺(南)→八坂之塔(同南)→八坂神社(北)→花见小路(南)→建仁寺(南)→白川(北) = **错**(南北折返 3 次)。
**正例**:高台寺→八坂之塔→建仁寺→花见小路→八坂神社→白川南通,一路从南到北。

**强度也一起控**:一天总步行 7-8 km 是"最高强度"边界,超 10 km 是贪心打卡。

### 12. 多来源验证动线(不只小红书)

- **小红书**:搜"景点名 + 一日游/路线/city walk"，看爆款顺序（地理顺序经过实践验证）
- **Japan Guide / Lonely Planet**:看欧美用户走法，时间分配往往更合理
- **不是直接抄**——爆款常贪心(12 个点塞一天),但**路线顺序可借鉴**
- 对照:我们 slot 顺序和爆款方向一致吗?如果他们 A→B→C,我们反过来 = 大概率错
- **就近原则**:优先串同一步行圈的点；跨区域跳跃需要有强理由（季节限定/唯一性）
- **模板帮用户做决定**:需要二选一的情况（如摩耶vs六甲），模板给出默认推荐+推荐理由，不留给用户自己判断

### 13. 三档强度 + 三级优先级(硬规)

**priority 三级**:
- **P1 核心**:模板存在的理由,任何密度都不砍。如 岚山竹林+渡月桥黄昏、清水寺+白川黄昏、peak-end、午餐、晚餐
- **P2 重要**:relaxed 档砍
- **P3 扩展**:balanced 档就砍

**三档密度验证**:
- **packed(特种兵)**:P1+P2+P3 全开,总步行 7-8 km,总 slot 6-9 个
- **balanced**:砍 P3,保留 P1+P2,总 slot 5-7 个
- **relaxed**:砍 P3+P2,只剩 P1 核心 slot。砍掉的时段给**轻松活动建议**,不是真空也不硬塞新 slot

**relaxed 档的正确实现**:
- ❌ 砍后让中间时段空 3 小时不管,用户站在路边不知道干什么
- ❌ 硬延长 P1 slot 时长(清水寺 2h → 3h),不自然
- ✅ 砍后模板明确告诉用户这段时间推荐做什么**轻松的事**:
  - 附近咖啡店坐 1 小时 / 逛小店杂货店 / 回酒店泡澡 / 找公园河边发呆 / 逛不知名老街小巷
  - 写在模板 `pace_variants.relaxed_tips` 字段

**能"坐很久不无聊"的景点有严格条件**:
1. 真有风景(鸭川河畔/渡月桥南岸/糺之森/天龙寺方丈前廊/白川南通石桥)
2. 有物理位置(石阶/长椅/缘侧/榻榻米/咖啡店座位)
3. 氛围安静(不是人挤人)

**大多数景点逛完没事做**,不能硬塞时长:清水寺舞台看完就走 / 金阁绕池 10 分钟 / 锦市场站着逛 / 小神社 5-10 分钟扫完。

**一天最多一两段慢节奏**——渡月桥南岸 1h + 咖啡店 1h 一天足够,不是整天发呆。

### 14. relaxed 档的用户画像

relaxed 档**不是 packed 的让步版**,是**对休闲游用户的量身设计**:

- **用户画像**:体力一般/偏弱、不追求"看尽景点"、愿意花 1-2h 发呆、**花钱买"不累"**
- **代表人群**:情侣度蜜月/纪念日、父母带小小孩、中老年、第二次来京都的深度游客
- **预算偏中高**(蜜月/带父母/周年纪念) → 午餐可升档怀石/町家割烹,酒店偏好体验型
- **peak-end 情绪价值被放大**:黄昏渡月桥坐 1h 对他们比走 8km 值

**文案要**:
- ✅ 主动推销"慢"是卖点——"下午 2-5 点留白给你找家町家咖啡慢慢坐,或回酒店泡澡"
- ❌ 不要说"砍掉景点""少看了什么"(packed 让步口吻)
- ✅ 突出"来京都不是打卡,你就想慢慢过一天"

**断链自查**:balanced/relaxed 砍点后,中间时段模板有没有给用户"可以做什么"建议——没有就是真空 bug,补。

**写模板时自查**:
- 每个 slot 都标 priority,核心标 P1(不砍)
- **P1 必须真是"模板存在理由"**:模板去掉这 slot 还能不能成立?不成立才是 P1
- **peak-end 默认 P1**,不可砍
- 午餐/晚餐默认 P1

**典型错误**:所有主线 slot 都标 P1,relaxed 砍不下去。正确:清水寺 P1、祇园散步 P2、高台寺 P3——真区分"核心/重要/扩展"。

### 15. slot 时间必须匹配景点开放时间(硬规)

写 slot 前**必须验证景点开门时间**。很多寺院/博物馆 9:00 或 10:00 才开,清晨 8 点 slot 不能放需要购票/进境内的景点。

**清晨 8 点 slot 只能用**:
- 神社境内(免费,大多通宵或清晨开)
- 街道/小路/河岸/公园
- 自由步行的免费寺院外围

**清晨 8 点不能放的**:
- 建仁寺(10:00)、高台寺(9:00)、龙安寺(8:00 但人早)、二条城(8:45) 等付费寺院
- 博物馆/美术馆(普遍 10:00)
- 神社内部特殊参观区(清水寺本堂 6:00 开是少数例外)

**典型错**:"08:00-10:00 散步建仁寺方丈前枯山水",但建仁寺 10:00 才开,用户去了进不去。

**自检**:任何 9:30 前或 17:00 后的 slot,搜"景点名 拝観時間"或"景点名 開門"。

### 16. 和服 tag 模板严禁进禁穿和服的寺院(硬规)

近年京都多家寺院**明确禁止穿和服(コスチューム)入内拍照**,违反会被请离或拒绝拝観。

**已知禁穿**:
- 建仁寺(2023 年起)
- 其他陆续增加,写和服日 slot 前必须 WebSearch 确认

**和服日 slot 安全清单**:
- ✅ 神社境内(八坂/八坂庚申堂/伏见稻荷)
- ✅ 街道/坡道(花见小路/二年坂三年坂/八坂通)
- ✅ 河岸/公园(白川南通/鸭川)
- ✅ 商店/咖啡馆内
- ⚠️ 寺院境内必查官方政策

**自检**:写有 `selectable_tag: "kimono"` 的模板时,每个 slot 涉及寺院都搜"寺院名 着物 撮影 禁止"。

## Part 2 补充:体验设计 6 条(peak-end 深化)

> 上面 16 条是"字段规则",下面 6 条是"体验设计",2025 学术研究证实。旅行记忆按"天/城市段"分 episode 分别打分,不是一条平均线。

### S1. 每一天都要有自己的 peak 和收尾画面

不只是整段行程 1-2 个大峰值——**每一天都要有自己的"这一天最强瞬间"**。不一定是 S 级景点,可以是日落啤酒、意外小巷、市场叫卖。

- 每一天要有自己的"收尾画面"——不是"回酒店睡觉",而是"今天以这画面结束"(花见小路艺伎背影/岚山竹林黄昏的光/道顿堀霓虹倒影)
- 每个城市段也要有"peak + end"——大阪最后一天不是"去京都前收拾",是"大阪告别仪式"

**判断**:翻一遍行程,能不能对每天说出"这天最强瞬间是 X""这天结束画面是 Y"。说不出 = 没立住。

### S2. 规避负峰值比制造正峰值更重要

2025 研究证实:**负峰值会永久覆盖当天所有好记忆**,权重甚至大于正峰值。一整天好设计可被一个 10 分钟负峰值毁掉。

**常见负峰值**:
- **文化纠察**(旺季京都严重):金阁寺合照 > 10 人被拦、神社台阶合照被念、小巷拍照被吼
- **体力崩溃**:伏见稻荷中午走到半山腿废又不能折返;东大寺→春日大社→若草山连走
- **长队**:USJ 闭园挤地铁、清水寺下山赶坂道高峰
- **语言障碍**:预约老店看不懂暗示、便利店取票机看不懂日文
- **预算踩坑**:居酒屋お通し座位费、嵯峨野小火车旺季被迫站票

**硬规**:模板里有任何"高风险负峰值"景点,flow detail 必须写"规避方式"。写不出 = 不合格。

### S3. 同类体验跨天去重(不只是菜系)

Memory interference:连续三天寺庙,第一天的会被后两天擦掉。需避免:
- Day 1 大阪城 + Day 2 姬路城 + Day 3 二条城(三天三个"日本城")
- Day 1 清水寺 + Day 2 东福寺 + Day 3 金阁寺(三天三个"京都寺庙")
- Day 1 海游馆 + Day 2 动物园 + Day 3 水族馆(三天"看动物")

**判断**:N 天主要景点按类型分组(寺庙/神社/城/博物馆/市场/自然/主题乐园/夜景台)。任一类型出现 3+ 次而没显著"进入方式/氛围/时段"差异 → 类型抹杀,调整。

**不是"不能连续两天寺庙"**——能说出"清水寺舞台俯瞰 vs 东福寺通天桥红叶"这种差异,两天寺庙没问题。说不出 = 类型抹杀。

### S4. 最后一天要有"上扬的小峰值"

研究显示结尾是记忆权重最重的位置。中性结尾("轻松+购物")把整段记忆拉平,**上扬结尾把整段记忆往上拉**。

离开日前半天应是**小上扬峰值 + 平静收束**,不是"购物 + 机场"。

**合格例**:
- 早晨黑门市场买能带走的食材 + 中午真值得的乌冬 + 下午机场
- 早晨早起才有的场景(鸭川晨雾/伏见稻荷无人千本鸟居) + 咖啡早餐 + 平静退房

**不合格**:"上午药妆 + 下午直接机场"(中性收尾,结尾权重浪费)

**硬规**:离开日模板 flow 必须有条目 detail 显式说明"这是这段旅行的告别画面"。

### S5. flow detail 精确到车站出口和长队时段

中国用户在日本最常遇到的摩擦不是大事,是:
- **走错车站出口**:Osaka-Umeda / Shinjuku / Kyoto 有几十个出口,选错走 500-1000 米
- **长队高峰**:酒店 check-in 15:00-16:30 全城连锁都排队 30 分钟
- **换乘时间**:超大站换乘 5 分钟不够,通常 10-15 分钟
- **取票复杂度**:便利店打印预约票、USJ 快速通关换纸票卡 15 分钟

flow detail 涉及时**必须精确**:
- ❌ "从 Osaka 站去道顿堀"
- ✅ "从 Namba 站 14 号出口出来,直行 200 米过第一个十字路口就是道顿堀入口"
- ❌ "下午 3 点到酒店 check-in"
- ✅ "14:30 前到酒店寄存行李(避开 15:00 集中 check-in 高峰),15:30 之后再回来办手续"

**用户付 300 块,期待的就是这种免摩擦**。

### S6. 有些天要有"自由漫游段"(仍是专家决策)

用户潜在渴望"有段时间自己走走看看",但如果真说"这半天你自己安排"就把决策权扔回用户了。

**折中**:设计"自由漫游段"——不是"什么都不排",而是**专家决定"在这里留白,给 2-3 个方向建议,用户自己选"**:

> "下午 15:00-17:00 先斗町自由漫游段:可沿鸭川往北走到四条大桥(河边茶屋)、往南进祇园花见小路(艺伎出没时段)、或向西钻进锦市场(小吃街)。三个方向都不会错,选一个顺感觉走。"

**这本身是专家决策**(我决定在这里留白 + 给三个不会错的方向),不是偷懒。

**该安排的场景**:
- 第 2+ 次来关西(有基础熟悉度)
- 悠闲节奏档
- 连续 2-3 天高密度后的恢复段
- 情侣/闺蜜档(需要"两人独处漫步")

**不该安排**:
- 第一次来 + 紧凑节奏(期待被排满)
- 亲子档(需要明确指引)
- 到达/离开日(已有特殊设计)

## Part 2 扩充:高端体验产品分层规范

高端体验类模板（舞妓座敷、高级料亭、特种住宿）必须按价格和体验性质分层描述，不能把不同层级混写：

| 层级 | 代表（舞妓座敷为例） | 价格/人 | 装配条件 |
|---|---|---|---|
| L1 高端 | 静水香等 | ¥11万–13万 | 高预算用户表单勾选 |
| L2 中档 | 灯花TOUKA | ¥2.4万–6万 | 中高预算 |
| L3 标准观光 | UTAGE | ¥2.5万–3.9万 | 默认推荐 |
| L4 入门体验 | 料亭プラン | ¥0.8万–2.2万 | 低预算 |
| L5 公演 | 都をどり | ¥0.4万–0.7万 | 文化兴趣 |

**规则**：模板里必须明确写当前推荐的是哪个层级，不同层级体验性质本质不同，不可混写。用户看完必须清楚"我花这个价格，得到的是什么级别的体验"。

## Part 2 扩充:策展细节 3 条(整合市面范式)

> 这 3 条来自对真实爆款攻略 + 日本本地杂志(BRUTUS / &Premium / Casa) + Black Tomato 等策展社范式的整合,补齐"字段要填但没讲清怎么填"的空白。

### X1. photo_spots 要写"游客通常不走的角度"

photo_spots 字段存在但常被填成 Instagram 大全景 — 那是所有攻略都有的。**好的 photo_spots 参考小红书"出片机位"帖和日本摄影杂志的视角**:

- 每条写清 **机位(站哪) + 时段(几点) + 拍什么**
- 优先"游客通常不走的角度",避免主舞台全景
- 反例:"清水寺舞台"(所有攻略都有)
- 正例:"清水寺子安塔背后的小径,7:30 光线斜打进舞台,人群还没上来"

Tier A 模板 4 条,Tier B 2-3 条,Tier C 1-2 条(可略)。

### X2. fun_fact 不是百科摘要

fun_fact 常被填成维基百科。**参考 BRUTUS / &Premium 等日本杂志的"本地编辑会脱口而出的冷知识"范式**:

- ❌ "金阁寺始建于 1397 年,由足利义满建造"(维基百科摘要)
- ✅ "金阁寺现在这座是 1955 年重建的——1950 年一个学僧纵火烧了原版,三岛由纪夫把这事写成了《金阁寺》"(朋友会讲的故事)

判断标准:朋友带你到现场,他会忍不住说的那段话。不是"有字段要填",是"这里有个值得讲的"。

Tier A 模板 2-3 条,Tier B 1-2 条,Tier C 0-1 条。

### X3. nearby_shops 是"朋友顺手带你进的门面",不是购物清单

nearby_shops 常被写成伴手礼店清单。**参考日本本地生活杂志 / 小红书"散步笔记"范式**:

- 每条写清 **位置(从主景点走多远) + 为什么值得进 + 不买什么也可以看什么**
- 优先"走完主景点朋友会指给你看的小店",不是"京都伴手礼 TOP 10"
- 反例:"京都八桥老铺 xxx"(谁都知道)
- 正例:"白川南通石桥旁的古书肆,门帘是老板太太手染的靛蓝色;不买书也可以在进门长椅坐一下看鸭川"

Tier A 5 条,Tier B 3 条,Tier C 可略。

## Part 2 扩充:主题一句话(在 Part 1 已提,这里强化)

Part 1.6 要求动手前想清楚"这一天的主题一句话"——**这句话必须是"情绪+场景",不是"动作"**,参考 Black Tomato Feelings Engine(问"你想感觉什么"而非"去哪"):

- ❌ "岚山一日游"(动作)
- ❌ "京都北区寺庙深度"(地理+类型)
- ✅ "早春岚山——樱花季前的空窗期,一年里独享竹林的机会"(情绪+场景+时机)
- ✅ "祇园黄昏慢线——放下任务感,看城市切换到夜晚的那 2 小时"(情绪+意图)

**主题一句话写不出 = 模板还没到位,不要开始写 Part 2**。

## Part B:交付前自检清单(每个模板写完强制过)

**Part A 是创作原则,创作模式下脑子记不住细节,特别是事实性错误**(开门时间/票价/寺院政策)——这些不是"风格"问题,是"对不对",必须靠流程截住。

**每项打勾才算交付,不打勾的不是"大概对",是"我不知道要去查"**。

### B1. 事实核查(必须真查)

- [ ] **每个付费景点开门时间 vs slot 时间段** — 任何 9:30 前或 17:00 后 slot 必搜"景点名 拝観時間"
- [ ] **每个票价数字** — 近 1 年内真实数据?查到写"约 ¥X,以官方为准";查不到只写"拝観料,以官方为准",**绝不编**(很多寺院 2024-2025 涨价,老攻略数字都不对)
- [ ] **每个 entity_id 真存在?** — Glob/Grep 搜,不存在先建 entity 或从 core_entities 删
- [ ] **周几休/年末休这种硬事实真搜了?** — 寺田屋月曜不定休、月桂冠盆休/年末年始休是典型,遗漏 = 用户撞门
- [ ] **和服 tag 模板里的寺院有无禁穿政策?**(规则 16)
- [ ] **付费方式**(现金/无现金/双轨) — 月桂冠 2026/7/1 起无现金、清水寺/金阁寺现金 only,写错用户付不了

### B2. 结构对账(机械检查)

- [ ] **core_entities 每个 entity 都在 slot 里作为 P1 出现?** — 不是"路过""氛围担当"
- [ ] **slot 顺序画地图单向前进?** — 按经纬度贴一遍
- [ ] **每个 slot 都标 priority?** — 漏标 = 装配系统砍不了
- [ ] **P1 slot 去掉任意一个模板还成立吗?** — 不成立才是真 P1
- [ ] **peak-end slot 是 P1?** — 默认必须
- [ ] **午餐/晚餐都是 P1?** — 默认必须
- [ ] **relaxed 砍 P2/P3 后时段有 pace_variants.relaxed_tips 接住?** — 不留 2h+ 真空
- [ ] **curators_notes 数组 1-4 条,每条 30-50 字?** — 不是段落、不是内部分析
- [ ] **每个 slot note 50-100 字?** — 超长砍
- [ ] **餐厅 slot 只写 meal_area + cuisine_hint?** — 没写具体店
- [ ] **hotel_area_note 2-3 句?** — 没有"务必"
- [ ] **season_notes 只 seasonal_advantage + weather_tip?** — 没有跨季

### B3. 关键词 grep(一条命令自检)

- [ ] `grep "2026|2027"` — 清理年份硬编码(规则 8)
- [ ] `grep "务必|必须|一定要"` — 硬要求改推荐语气(规则 9)
- [ ] `grep "不用硬|晚到也|如果.*不.*也"` — what-if 免责(规则 2)
- [ ] `grep "樱花|红叶|夏"` — 早春模板里出现 = 跨季 bug(规则 3)
- [ ] `grep "情侣|闺蜜|亲子"` — 主线模板出现人群变种(规则 4)

### B4. 体验设计检查

- [ ] **最后一个slot有情绪收束** — 不能是"回酒店"或"随便吃晚餐"，peak_end_line对应到真实体验节点
- [ ] **说法和排程一致** — 文案里说"下午光线最好"，slot就必须在下午；说法和时间安排矛盾是硬错误
- [ ] **无绝对化表述** — 禁止"独享""几乎无人""必然"，改成"相对最安静""人流明显少于高峰"
- [ ] **special_date模板有推荐离开时间** — 给2-3个时间节点，每个注明"到这里已拿到什么"，让用户自己决定
- [ ] **二选一已给默认推荐** — 需要用户选择的情况，模板给出默认方案+理由，不并列留白
- [ ] **跨区域跳跃有理由** — 如有跨区组合，说清楚为什么跨（季节限定/唯一性/顺路）
- [ ] **条件依赖型标注trigger_type** — required（没有条件不装配）或boost（条件满足时提升优先级）

### B5. 反向自问(5 秒直觉)

- [ ] 真实用户走完这一天,能记住某个具体时刻吗?(记不住 = 失败)
- [ ] 有没有哪条 note 是"看起来像攻略但其实废话"?(删)
- [ ] 这天体验,夏天模板能直接替代吗?(能 = 没抓住早春独有)
- [ ] 用户花 ¥298 买了这一天,钱值吗?(不值 = 没到位)

### 自检清单使用规则

- **每项打 ✅ 才算完成**,打 ❌ 或 ⚠️ 的必须解决后再交付
- **B1 事实核查是红线**:发现一个编造数字就返工该 slot,不允许"改成'以官方为准'遮掉"不去查
- **走清单比写模板不好玩,但质量在这里**:创作决定上限,自检决定下限,下限守不住上限无意义
- **发现清单漏了一条**(新 bug 类型) → 补进清单再批量应用

## 禁止反复修改同一个模板

每个模板审核 = **一次过完 SOP**。发现 SOP 漏了某条原则 → 补进 SOP 再批量应用,不是一个模板改 6 次试出来。

## 核心原则再强调

**模板是"一种推荐的走法",不是"兼容所有场景的万能方案"**。用户有判断力能调时间;极端情况客服和 minimum_viable 兜底。过度工程 = 不信任用户 + 把自己当系统 = 文案臃肿 + 用户看不清核心推荐。
