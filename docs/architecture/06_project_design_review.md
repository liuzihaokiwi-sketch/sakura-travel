# 06 项目整体设计审查

## 文档定位

本文件是基于当前真相源文档、关键旧文档与当前代码现状形成的一轮整体设计审查。

目标不是拆工程任务，也不是直接进入重构，而是判断：

- 哪些设计是优秀且应保留的
- 哪些设计只是平庸、可以修改
- 哪些设计可有可无、错误、或当前阶段应删除 / 冻结 / 降级

评审标准包括：

- 是否符合当前真实业务边界
- 是否贴近用户心智，而不是内部系统心智
- 是否强化“有趣、有设计感、像旅行共创”的方向
- 是否真正影响业务闭环
- 是否增加不必要复杂度
- 是否制造技术债或架构债
- 是否延续了旧前提或历史包袱

## 总体结论

这套项目设计的优点非常明确：

- 主产品思路是对的
- 决策与渲染分层方向是对的
- 手册化交付方向是对的
- “样片成交 -> 付费表单 -> 结构化生成 -> 交付 -> 回流”的主闭环也是对的

但它当前最突出的问题也很集中：

- 新旧链路并存导致表达与实现双重分裂
- 很多设计从“工程合理”出发，却还不够“用户感知合理”
- 一些旧阶段文档和旧产品包袱仍在持续制造架构债
- 品牌与前端体验表达还没有完全跟上“共创式旅行手册”这个更有趣的方向

因此，这个项目不是“方向错了”，而是“主方向不错，但中间夹了不少历史包袱与半迁移结构”。

---

## A. 优秀设计（保留）

### 1. 产品从“AI写攻略”转向“系统化旅行手册服务”

**为什么归到这一类**

这是当前项目最根本、也最正确的升级。它把产品从低壁垒的“内容生成工具”拉回到更有价值的“旅行决策与交付服务”。

**对用户体验的影响**

用户更容易理解自己买到的是什么：

- 不是一段文字
- 不是一个聊天机器人答复
- 而是一份可以出发、可以保存、可以回看、像为自己认真做过取舍的手册

**对业务闭环的影响**

这直接支撑了从样片成交、到付费表单、到交付、到回访的完整链路，比“生成一份攻略”更适合做收费产品。

**对技术债 / 架构债的影响**

它减少了“所有问题都交给大模型”的架构债，把系统拉向更稳定的结构化链路。

**建议动作**

保留，而且后续所有对外表达和对内规划都要继续围绕这个定义，不要回退到“AI生成攻略”的旧说法。

### 2. “低信息成本样片成交 + 付费后深采集”的两段式链路

**为什么归到这一类**

这是少数同时符合用户心智和业务转化逻辑的设计。用户先看到价值，再决定是否补更多信息，阻力明显低于一上来就填大表单。

**对用户体验的影响**

降低首次进入门槛，用户不会一开始就被复杂表单劝退。

**对业务闭环的影响**

样片承担成交，详细表单承担质量，前后分工清楚，是很强的闭环设计。

**对技术债 / 架构债的影响**

它避免把所有复杂性前置到首个页面，也避免客服人工成为前置瓶颈。

**建议动作**

保留。当前阶段只需要把这条链路的入口优先级从“独立站优先”进一步收敛成“抖音表单优先”。

### 3. Layer 2 从模板思维走向阶段化决策链

**为什么归到这一类**

`normalize -> gate -> select -> build -> fill -> rerank` 这种结构是成熟系统该有的形状，比“大 prompt 一次性生成”稳得多，也比纯模板拼装更有扩展性。

**对用户体验的影响**

更容易产出“像有人真的替我想过”的结果，因为系统是在做取舍，不是在堆内容。

**对业务闭环的影响**

这决定了交付质量上限，也决定后续能否支持不同城市圈、不同主题、不同预算偏向。

**对技术债 / 架构债的影响**

这是减少长期架构债的关键设计，因为它给未来的复盘、评测、替换、回库都留了结构位置。

**建议动作**

保留，并继续作为长期主链目标。当前阶段重点不是再抽象一轮，而是把文档与边界先收拢清楚。

### 4. Layer 3 走向 `chapter_plan -> page_plan -> page_view_model -> render adapters`

**为什么归到这一类**

对 60 页手册来说，这条链是必要且正确的。它把“页面系统”从 HTML 模板中解耦出来，是交付物升级后的正确路线。

**对用户体验的影响**

用户看到的会更像一本被设计过的手册，而不是拼接式报告。

**对业务闭环的影响**

它直接支撑：

- PDF / H5 同源
- 页型稳定
- 图片/插画位接入
- 后续分享、二次传播、版本衍生

**对技术债 / 架构债的影响**

这条链是在主动消化“长模板自然分页”的旧债。

**建议动作**

保留，作为长期渲染主链目标。

### 5. 60页手册 + 插画/图片素材位的交付升级

**为什么归到这一类**

这是目前最贴近“用户感知价值”的设计之一。它让产品从“有内容”升级成“有作品感、有保存价值”。

**对用户体验的影响**

用户会更容易感到：

- 这不是普通攻略
- 这份内容值得保留
- 这份产品有设计感和纪念价值

**对业务闭环的影响**

它有利于提高付费意愿、分享欲、复购与转介绍。

**对技术债 / 架构债的影响**

短期会增加渲染复杂度，但这是值得承担的复杂度，不属于无效架构债。

**建议动作**

保留。后续所有渲染、素材、页面系统讨论都要以这一定义为基准。

### 6. 输入校验、guardrails、review、回访回库的闭环意识

**为什么归到这一类**

虽然还没完全收口，但“输入前校验 + 生成后质量门 + 回访反馈回流”这个整体方向是非常成熟的系统思路。

**对用户体验的影响**

可以减少明显错误、降低翻车感，也能逐步让系统变得更稳。

**对业务闭环的影响**

这是从一次性交付走向可持续优化飞轮的基础。

**对技术债 / 架构债的影响**

如果坚持这条路线，长期是在减少债；问题只在于当前还比较分散。

**建议动作**

保留整体方向，但后续需要收束表达与边界。

## B. 平庸设计（可修改）

### 1. 产品定位里“专业规划感”很强，但“有趣、共创感”还不够强

**为什么归到这一类**

当前定位主要强调“专业取舍、少踩坑、少后悔”，这些都对，但整体仍偏理性、偏效率工具，不够像“有设计感、像一起做旅行”的产品。

**对用户体验的影响**

用户能感受到专业，但未必能感受到好玩、浪漫、被共同创作的乐趣。

**对业务闭环的影响**

会影响分享欲、记忆点和品牌区分度，尤其是在抖音入口场景下。

**对技术债 / 架构债的影响**

不会直接制造技术债，但会形成品牌表达债：系统越来越强，品牌感却还像效率工具。

**建议动作**

修改。保留“专业规划”主轴，但逐步强化“旅行共创、设计感、收藏感”的表达语言。

### 2. 获客链路里“独立站历史包袱”还比较重

**为什么归到这一类**

当前 architecture 已经明确“抖音表单优先”，但旧内容、旧页面、旧思考方式里仍能看出对独立站体系的惯性依赖。

**对用户体验的影响**

如果还按独立站逻辑先想，容易忽略抖音用户的轻决策、快进入、先看值感的心智。

**对业务闭环的影响**

会拖慢当前真正能收单的链路优化。

**对技术债 / 架构债的影响**

这会制造产品优先级债：团队时间花在当前不最重要的地方。

**建议动作**

修改。不是删独立站，而是把它从“当前主线”降为“后续承接层”。

### 3. Layer 2 设计已经很好，但仍偏内部系统心智表达

**为什么归到这一类**

城市圈选择、主活动、骨架、补槽这些设计都很强，但目前主要是在系统视角里清楚，在用户视角里的“为什么这样安排”还没有完全自然地显化。

**对用户体验的影响**

结果可能是合理的，但用户未必立刻感到“懂我”。

**对业务闭环的影响**

会影响用户对方案价值的感知，尤其是付费前后的说服力。

**对技术债 / 架构债的影响**

不是架构错，而是解释层没有完全跟上，会让好系统显得像冷系统。

**建议动作**

修改。继续保留阶段化决策链，但更强调解释对象、证据对象和“共创感文案”的落点。

### 4. Layer 3 目前目标正确，但“作品感”表达还没有完全压过“报告感”

**为什么归到这一类**

已经有 chapter/page/view model 的正确方向，但一些旧渲染思路和前端 plan 页仍然会把结果展示成比较传统的报告阅读页。

**对用户体验的影响**

交付值感会受限，看起来更像“内容页”，而不是“手册作品”。

**对业务闭环的影响**

影响转化、分享和复购场景下的第一感受。

**对技术债 / 架构债的影响**

继续维持这种中间状态，会延长新旧渲染链并存时间。

**建议动作**

修改。不是再发明新渲染概念，而是让“作品感”成为页面系统的明确目标。

### 5. Layer 4 方向正确，但“闭环”还更多存在于概念而非统一系统面

**为什么归到这一类**

现在有校验、有 guardrails、有 review、有反馈，但它们更像多个正确模块，而不是一个统一、清晰的运营闭环层。

**对用户体验的影响**

用户未必能持续感受到系统越来越好，很多优化还停留在内部。

**对业务闭环的影响**

会降低反馈回流的真实效率，也让运营控制显得分散。

**对技术债 / 架构债的影响**

会形成“能力存在但边界不清”的架构债。

**建议动作**

修改。当前先统一表达，不急着大改实现。

### 6. 品牌表达已经有温度，但还不够“好玩又高级”

**为什么归到这一类**

现有表达已经不算冷，但仍偏“功能型产品文案”，在“有趣、有设计感、像旅行共创”这条线上还没完全打透。

**对用户体验的影响**

用户会觉得靠谱，但未必觉得有记忆点。

**对业务闭环的影响**

会影响内容传播和品牌区分。

**对技术债 / 架构债的影响**

这不制造技术债，但会形成体验债和品牌债。

**建议动作**

修改。后续在不牺牲专业感的前提下，提高“共创感”和“作品感”。

## C. 可有可无 / 错误设计（建议删除、冻结、降级或重写）

### 1. 把独立站当作当前主线的设计惯性

**为什么归到这一类**

在当前边界下，这是错误优先级。不是说独立站没价值，而是它不是当前阶段最该优先做的事情。

**对用户体验的影响**

会把注意力从真正短期能收单的抖音表单链路上移开。

**对业务闭环的影响**

会直接拖慢当前商业验证速度。

**对技术债 / 架构债的影响**

会制造投入错位债，做了很多“好看但不当前关键”的东西。

**建议动作**

降级。冻结“独立站优先”的思路，只保留必要承接，不作为当前主线。

### 2. 继续延用“日本-only”为默认前提的设计

**为什么归到这一类**

在当前真相源下，这已经是过时前提。如果继续默认使用，会不断污染后续规划。

**对用户体验的影响**

短期用户未必直接感知，但会让未来范围扩展和内容表达越来越割裂。

**对业务闭环的影响**

会阻碍六城市圈边界的统一和后续产品复制。

**对技术债 / 架构债的影响**

这是明确的架构债来源，会把旧命名、旧规则、旧数据结构继续固化。

**建议动作**

删除默认前提。允许保留日本历史批次文档，但必须降级为历史输入，不再作为当前默认边界。

### 3. 继续把“长攻略/长报告”当默认交付物的设计

**为什么归到这一类**

在 60 页手册已经成为目标交付的前提下，这种默认思路属于错误惯性。

**对用户体验的影响**

会削弱产品的“收藏感、设计感、像共创作品”的差异。

**对业务闭环的影响**

会影响成交感知和复购传播。

**对技术债 / 架构债的影响**

会延长旧渲染链寿命，制造双系统长期并存。

**建议动作**

降级。旧报告链只保留为兼容层，不再作为长期主目标。

### 4. 旧 `product_sku` 心智继续主导产品表达

**为什么归到这一类**

代码里保留旧 SKU 是现实，但如果继续让它主导产品定义，就是错误的产品心智。

**对用户体验的影响**

会让前台产品看起来更像僵硬套餐，而不是围绕用户旅行目的组织的服务。

**对业务闭环的影响**

不利于按主题、预算偏向、时间层去灵活成交。

**对技术债 / 架构债的影响**

这是明显的历史包袱，会持续让产品层和代码层互相拖累。

**建议动作**

冻结其产品定义地位。允许它暂时作为兼容实现存在，但不再把它当真相源。

### 5. 继续让新旧决策链长期并列为“同等级主链”

**为什么归到这一类**

短期兼容是合理的，长期并列不是。长期并列意味着系统没有真正选择方向。

**对用户体验的影响**

会造成结果风格不稳定，用户很难形成对产品稳定性的预期。

**对业务闭环的影响**

会影响复盘、评测、解释、运营控制和复制能力。

**对技术债 / 架构债的影响**

这是重大架构债，会让每一次修改都要双链兼容。

**建议动作**

当前阶段先冻结“继续扩展旧链路”的动作。旧链只做兼容，不再继续增肥。

### 6. 继续让新旧渲染链长期并列为“同等级主交付方案”

**为什么归到这一类**

和决策层一样，短期兼容可以，长期并列不合理。

**对用户体验的影响**

会导致不同入口下交付体验不一致。

**对业务闭环的影响**

会让 PDF/H5/前端阅读页之间难以形成稳定统一产品。

**对技术债 / 架构债的影响**

这是典型渲染架构债，后续每改一处都可能要改两套。

**建议动作**

冻结“继续强化旧长模板链”的想法。旧链保留兼容用途，新页面系统明确为长期目标。

### 7. 一些旧文档仍把阶段性判断写成“当前唯一源头”

**为什么归到这一类**

这是当前最容易制造误导的文档问题之一。它不是功能 bug，但会持续制造决策 bug。

**对用户体验的影响**

不会直接影响用户，但会间接影响后续所有对外体验。

**对业务闭环的影响**

会导致团队在错误优先级上浪费时间。

**对技术债 / 架构债的影响**

这是典型文档架构债，会让系统不断在旧前提上修补。

**建议动作**

删除其“当前唯一源头”的地位。对旧文档继续做 superseded / align note，而不是继续默许其最高权威地位。

---

## 最后判断

这套项目最值得保留的，不是某个具体页面，也不是某个具体模块，而是三件事：

1. 从“AI写攻略”升级为“旅行决策与手册交付系统”
2. 从“大 prompt”升级为“阶段化决策链”
3. 从“长报告”升级为“60页手册页面系统”

而当前最该警惕的，也不是单个功能没做完，而是三类历史包袱：

1. 日本-only默认前提
2. 独立站优先惯性
3. 新旧链路长期并列不收口

如果后续能坚持：

- 保留优秀设计
- 修改平庸设计
- 冻结或降级错误设计

这套系统是有机会从“能生成攻略”真正走向“有作品感、有品牌感、能形成闭环的旅行共创产品”的。
