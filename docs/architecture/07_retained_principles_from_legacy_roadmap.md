# 07 从旧路线图抽取的仍有效原则

## 文件定位

本文件只沉淀从历史路线图文件中抽取出来、在当前边界下仍然有效的长期原则。

它不是旧路线图重写版，也不是新的执行计划。

本文件当前只保留三类原则：

- 私域 / 微信只补位，不替代系统主链路
- 用户状态分层 / 交付后状态分层
- 护城河不在 prompt，而在输入结构、方案树、规则体系、预览成交、自助微调闭环

---

## 一、私域 / 微信只补位，不替代系统主链路

### 保留原则

私域和微信的作用，是接住系统暂时接不住的部分，而不是替代系统主链路。

### 当前仍然成立的原因

当前真相源已经明确：

- 主链路是表单采集 -> 决策编排 -> 页面系统 -> PDF/H5交付 -> 质量回流
- 当前入口优先级是抖音表单收单，不是微信优先，也不是独立站优先

因此，私域 / 微信的正确位置仍然是：

- 承接高意向用户
- 处理异常情况
- 协助复杂取舍
- 接住系统暂时无法自动完成的部分

而不应该承担：

- 替代表单收集全部需求
- 替代系统完成普通替换与普通问答
- 成为主产品流程本身

### 当前使用方式

这条原则应作为“运营边界”长期保留。

---

## 二、用户状态分层 / 交付后状态分层

### 保留原则

用户不应被看成单一订单，而应按所处阶段进行状态分层。

### 当前仍然成立的原因

当前系统本身就是一条多阶段服务链，因此用户状态天然存在分层：

- 免费预览未付费
- 已付费待交付
- 已交付待精调
- 已完成可复购
- 高满意可转介绍

这类分层今天仍然有价值，因为它决定：

- 不同阶段的沟通目标
- 不同阶段的产品动作
- 不同阶段的运营承接方式

### 当前使用方式

这条原则应继续作为后续运营、CRM、交付后闭环设计的基础，而不是重新发明。

---

## 三、护城河不在 prompt，而在系统化能力

### 保留原则

项目真正的长期护城河，不在 prompt，也不在单次生成结果，而在系统化能力的组合。

### 当前仍然成立的原因

在当前边界下，真正沉淀价值的仍然是：

- 前台输入结构
- 后台方案树
- 规则体系
- 预览成交逻辑
- 自助微调闭环

这些能力决定了系统是否能：

- 稳定收单
- 稳定生成
- 稳定交付
- 稳定复用
- 稳定回流优化

相比之下，prompt 只是局部能力，不应被当作核心护城河。

### 当前使用方式

这条原则应长期保留，作为判断架构方向是否健康的重要标准。
