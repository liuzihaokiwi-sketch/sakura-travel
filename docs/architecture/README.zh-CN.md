# 当前架构人类可读版

这份文档给项目负责人看，用来在 5 分钟内理解：项目现在是什么、主链路是什么、当前最重要的问题是什么、为什么这一阶段先整理文件而不是先全面改代码。

## 一句话先说清楚

这个项目现在已经不是“让大模型直接写旅游攻略”的工具，而是一个完整的旅行手册系统：

- 前端或渠道负责收集用户需求
- 后端负责决策编排
- 渲染层把结果做成页面化手册
- 最终交付 PDF / H5
- 后面还要接评测、运营和反馈回流

## 项目现在是什么

现在的系统可以理解为一条已经成型、但新旧链路并存的主链：

1. 获客和需求采集
2. 表单与订单流转
3. 决策与行程编排
4. 页面化渲染
5. PDF / H5 交付
6. 校验、评审、反馈回流

代码并不是概念阶段，很多模块已经存在；但同时，旧方案和新方案还在并行。

## 当前主链路是什么

当前主链路的正确理解是：

**表单采集 -> 结构化输入 -> 决策编排 -> 页面系统 -> PDF/H5交付 -> 质量校验与反馈回流**

注意，这里的“表单采集”在当前阶段不是优先做完整独立站，而是优先让抖音来的用户把数据先传进后端。

也就是说，当前入口优先级是：

- 抖音表单采集
- 后端 intake / planning / delivery

而不是：

- 先把独立站页面体系全部做完

## 范围现在怎么定义

项目范围已经从“日本核心城市”扩展成 **六个城市圈**：

- 关西城市圈
- 关东城市圈
- 北海道城市圈
- 华南五市城市圈
- 北疆城市圈
- 广东城市圈

这意味着后续系统抽象不能再默认“日本-only”。

## 交付物现在怎么定义

交付物已经升级为 **60页旅行手册**，不是普通长攻略。

这份手册要同时考虑：

- 固定页型
- 章节结构
- PDF / H5 双交付
- 插画 / 图片素材位

所以渲染层的目标，不再是“把一段长内容拼成 HTML”，而是“把结构化结果做成真正的手册页面系统”。

## 新旧决策链分别是什么

### 旧决策链

旧决策链是：

- `assemble_trip`
- 模板装配器
- 基于旧模板 / 路线模板 / 填槽逻辑生成行程

它的特点是：

- 偏模板驱动
- 偏装配式
- 是当前系统里的历史主链

### 新决策链

新决策链是：

- 城市圈抽象
- 标准化画像
- 阶段化决策链

大致包括：

- profile normalization
- eligibility / precheck
- city circle selection
- major selection
- hotel strategy
- route skeleton
- secondary fill
- meal fill
- fit scoring

它的特点是：

- 更像真正的决策编排层
- 不再把“大模型直接生成全文”当主策略
- 已经有不少模块落到代码里

### 当前现实

当前现实不是“新链路已经完全替代旧链路”，而是：

- 旧决策链还在
- 新决策链也在
- 两条链路并存

这是当前最重要的架构事实之一。

## 新旧渲染链分别是什么

### 旧渲染链

旧渲染链是：

- 旧 `renderer`
- 长模板 / 长页面流
- 偏报告式输出

它更接近“把内容拼成一份长 HTML / PDF”。

### 新渲染链

新渲染链是：

- `chapter_plan`
- `page_plan`
- `page_view_model`
- `render adapters`

它的目标是：

- 先定义章节
- 再定义页面
- 再定义每页的数据模型
- 最后输出 PDF / H5

这条链更适合支撑 **60页手册 + 插画/图片素材位** 的交付形态。

### 当前现实

和决策层一样，渲染层当前也是新旧并存：

- 旧 renderer 还在
- 新页面系统链也已经开始出现

所以当前不是纯“旧系统”，也不是纯“新系统”，而是迁移中的混合状态。

## 当前最重要的缺口是什么

当前最重要的缺口不是“代码完全没写”，而是下面四个：

1. **新旧决策链并存**
   旧 `assemble_trip` 和新阶段化决策链同时存在，主链边界还没彻底收敛。

2. **新旧渲染链并存**
   旧 renderer / 长模板和新页面系统链同时存在，交付语义还没完全统一。

3. **真相源分散**
   最新边界、旧设计、当前实现、60页手册规则散落在多个文档里，容易混用旧前提。

4. **质量与运营层仍是分散接入**
   校验、guardrails、review、反馈回流都有，但整体表达和边界还没完全收口。

## 为什么这一阶段先整理文件与总规划

因为当前最大的风险不是“代码不够多”，而是“方向和表述不统一”。

如果现在直接全面改代码，很容易出现：

- 一部分还是按日本-only做
- 一部分已经按六城市圈做
- 一部分还是长攻略逻辑
- 一部分已经按60页手册做
- 一部分还在围绕独立站
- 一部分已经以抖音表单为入口

这样会把旧前提继续固化进代码里，后面返工更大。

所以这一阶段先做的事情是：

- 把边界写清楚
- 把旧前提作废项写清楚
- 把当前现状和目标架构写清楚
- 把仍然有效的缺口保留下来
- 先统一真相源，再进入执行

## 当前阶段先做什么

当前阶段先做：

- 整理仓库内规划文件
- 建立 `docs/architecture/` 作为最新真相源入口
- 明确六城市圈边界
- 明确 60 页手册 + 插画/图片素材位的交付定义
- 明确抖音表单优先、独立站暂不优先
- 统一“当前系统是什么 / 目标系统是什么 / 哪些缺口仍保留”

## 当前阶段不做什么

当前阶段不做：

- 不先全面改业务代码
- 不先删除旧文档
- 不先发起大规模链路迁移
- 不先做完整独立站体系
- 不先拆成详细开发排期

## 最后一句

当前这套系统的核心，不是“先把代码全推倒重来”，而是先把 **范围、交付物、入口优先级、旧新链路边界** 全部说清楚。

只有这些真相源稳定下来，后面的执行才不会继续在旧前提上打补丁。
