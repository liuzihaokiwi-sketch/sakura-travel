## A19. 用户可感知的交付结构

> 用户拿到完整攻略后，从第一页翻到最后一页的感知顺序

### 交付目录（10大板块）

| 序号 | 板块 | 页数(参考) | 用户心理 |
|---|---|---|---|
| ① | **开头：封面 + 行程概览** | 2-3页 | "哇，这排版也太好看了吧" |
| ② | **路线：城市串联地图** | 1-2页 | "路线逻辑很清晰，不走回头路" |
| ③ | **时间轴：每日行程详情** | 12-18页 | "几点去哪、玩多久、怎么走，全写好了" |
| ④ | **理由：为什么这样安排** | 1-2页 | "原来是这样考虑的，专业！" |
| ⑤ | **酒店：住宿推荐报告** | 2-4页 | "帮我选好了，不用自己翻携程" |
| ⑥ | **餐厅：美食推荐报告** | 2-4页 | "连吃什么都安排了，太省心" |
| ⑦ | **交通：卡券 + 省钱方案** | 1-2页 | "交通卡怎么买最划算，一目了然" |
| ⑧ | **Plan B：雨天/备选方案** | 1-2页 | "下雨也不慌，有备选" |
| ⑨ | **准备：行前准备清单** | 2-3页 | "签证、行李、APP，查漏补缺" |
| ⑩ | **安全：安全须知 + 紧急联系** | 1-2页 | "万一出事知道找谁，安心" |

### 各板块内容细节

#### ① 封面 + 行程概览
```
封面：
- 目的地城市名（大字）
- 出行日期 + 天数
- 同行人数/类型标签（如"2人·情侣·6天5晚"）
- 风格化封面图

概览页：
- 一句话行程摘要："东京3天 → 箱根1天 → 京都3天"
- 每日一句话亮点（Day1: 浅草寺→晴空塔→秋叶原）
- 总预算估算区间
- 最佳出行季节提示
```

#### ② 路线总览地图
```
- 城市间串联示意图（箭头连线）
- 每个城市停留天数标注
- 城市间交通方式 + 耗时
- 颜色分区：东京=蓝，关西=橙
```

#### ③ 每日行程时间轴（核心交付物）
```
每天一页或两页，包含：
- 日期 + 星期 + 天气预报参考
- 时间轴卡片（09:00-11:00 浅草寺）
  · 景点照片（1张主图）
  · 一句话推荐理由
  · 开放时间 / 门票 / 预约提示
  · 停留建议时长
- 景点间交通指引
  · 交通方式图标（🚶步行/🚇地铁/🚌巴士/🚄新干线）
  · 耗时 + 费用
  · 具体线路（"银座线 浅草→上野 3分钟"）
- 午餐/晚餐插入点（标注"附近推荐→见餐厅报告P25"）
- 当日小贴士（"今天走路多，穿舒服的鞋"）
```

#### ④ 为什么这样安排（路线设计理由）
```
- 路线编排逻辑说明
  · "把浅草和晴空塔放同一天，因为步行15分钟就到"
  · "箱根安排在东京和京都之间，顺路且换个节奏"
- 节奏设计说明
  · "第3天故意松一点，前两天走得多需要缓一缓"
- 季节/时间考量
  · "周末避开原宿竹下通，改去明治神宫更舒服"
```

#### ⑤ 酒店推荐报告
```
每个住宿区域：
- 区域简介（"新宿：交通枢纽，去哪都方便"）
- 推荐3-5家酒店，每家包含：
  · 酒店名（中+日文）
  · 价格区间 / 评分
  · 一句话推荐理由（朋友口吻）
  · 离车站步行距离
  · 避坑提示（"隔音一般，怕吵选高楼层"）
- 住宿区域对比表
```

#### ⑥ 餐厅推荐报告
```
按区域/天数组织：
- 每天午餐 + 晚餐各推荐2-3家
- 每家包含：
  · 餐厅名（中+日文）
  · Tabelog评分 / 人均
  · 推荐菜品（"必点炸猪排定食"）
  · 一句话推荐理由
  · 是否需要预约 + 排队预估
  · 营业时间 + 休息日
- 美食地图标注
```

#### ⑦ 交通攻略
```
- 交通卡购买指南（Suica vs PASMO vs ICOCA）
- JR Pass 是否值得买（按本行程计算）
- 机场→市区交通方案对比
- 城市间交通方案（新干线/巴士/飞机对比）
- 本次行程交通费用估算
```

#### ⑧ Plan B 备选方案
```
- 雨天替代方案（每天备选1-2个室内景点）
- 体力不够时的"偷懒版"路线
- 临时关闭/维修的备选
- 带小朋友/老人的节奏调整建议
```

#### ⑨ 行前准备清单
```
- 签证办理时间线
- 行李打包清单（必带/可带/不用带）
- 必装APP清单
- 货币与支付准备
- 保险购买建议
- 出发前3天/1天/当天 Checklist
```

#### ⑩ 安全须知
```
- 紧急电话（110报警/119急救/中国大使馆）
- 地震应对指南
- 台风季注意事项
- 就医流程
- 证件丢失处理
```

## A20. 内容分类：商品页明说 / 样张展示 / 交付后才出现

> 三层信息披露策略：让用户"先馋、再信、最后惊喜"

### 🟢 商品页文字明说（建立预期）

用户在购买决策页能直接看到的承诺：

| 内容 | 表述方式 |
|---|---|
| 30-40页杂志级排版 | "不是Word文档，是你愿意发朋友圈的攻略" |
| 每日时间轴精确到小时 | "几点出门、玩多久、怎么走，不用动脑" |
| 景点门票/开放时间 | "信息已核实，不用再查" |
| 不走回头路的串联路线 | "一天省2小时在路上的冤枉时间" |
| 酒店推荐+避坑（付费版） | "帮你排雷，不踩网红酒店的坑" |
| 餐厅推荐+预约提醒（付费版） | "Tabelog 3.5+的店，本地人也去" |
| 交通省钱方案 | "JR Pass买不买，帮你算好了" |
| 雨天Plan B | "下雨不慌，每天有备选" |
| 行前准备清单 | "签证、行李、APP，一页搞定" |

### 🟡 样张展示（建立信任）

在商品页通过截图/动图让用户"看到"品质：

| 样张内容 | 展示方式 | 目的 |
|---|---|---|
| 封面效果图 | 静态图 | 第一眼冲击："这比旅行社的好看" |
| 某一天的时间轴页 | 静态图（模糊后半段） | 核心卖点可视化 |
| 酒店推荐页截图 | 静态图（仅展示1家，其余遮挡） | 让用户知道推荐的深度 |
| 餐厅推荐页截图 | 静态图（仅展示1家，其余遮挡） | 同上 |
| 路线地图页 | 静态图 | 路线逻辑感 |
| 手机H5翻页效果 | 3秒动图/短视频 | "手机上也这么好看" |
| PDF整体翻页 | 5秒短视频 | 传达"30多页的厚度感" |

### 🔴 交付后才出现（制造惊喜）

商品页**不提**，用户收到攻略后发现的"超预期"内容：

| 内容 | 惊喜感来源 |
|---|---|
| "为什么这样安排"路线理由页 | "还解释了逻辑，不是随便排的" |
| 每日小贴士（穿鞋建议、充电提醒） | "连这种细节都想到了" |
| 景点间步行距离精确标注 | "650米，8分钟，这也太细了" |
| 当日备选景点（不只是雨天） | "体力不够也有轻松版" |
| 餐厅推荐菜品 + 日文点单提示 | "日文菜名都写好了，直接给店员看" |
| 紧急联系卡片页（可截图保存） | "这个截图存手机里真有用" |
| 封底彩蛋（旅行祝福语） | 情感加分 |

## A21. 推荐理由写法规范

> 核心原则：像一个去过10次日本的朋友在跟你说，不像算法在推荐

### ❌ 禁止写法

| 类型 | 反例 | 问题 |
|---|---|---|
| 评分腔 | "根据综合评分排名前3" | 冷冰冰，像机器 |
| 数据腔 | "该餐厅Google评分4.7，Tabelog 3.68" | 用户自己会查 |
| 广告腔 | "强烈推荐！不去后悔一辈子！" | 太假 |
| 百科腔 | "浅草寺建于645年，是东京都内最古老的寺院" | 没有帮用户做决策 |
| 万金油腔 | "性价比很高，值得一去" | 说了等于没说 |

### ✅ 正确写法：朋友口吻 + 具体理由

**公式：场景/人群定位 + 具体好在哪 + 一个小细节**

### 🏨 酒店推荐理由示例（3条）

**示例1：新宿格拉斯丽酒店**
> 选它就一个原因：楼下就是歌舞伎町入口，出门右转50米到一兰拉面，左转100米到唐吉诃德。晚上逛完直接上楼睡觉，不用赶末班电车。房间不大，但干净，日本酒店这个价位正常水平。

**示例2：京都三井花园酒店·京都站前**
> 拖着行李箱从京都站出来，走3分钟就到，这是最大的优势。第一天和最后一天不用折腾找路。旁边有个Porta地下街，晚上找吃的完全不愁。唯一注意：靠马路那侧的房间可能有点吵，订的时候备注"静かな部屋"（安静的房间）。

**示例3：大阪道顿堀某民宿**
> 我不推荐这家。虽然便宜，但离车站走15分钟，拖箱子走那段商业街每天人挤人。第一次去大阪，住难波站旁边才是正解——到道顿堀走5分钟，到心斋桥走8分钟，去机场坐南海电铁直达。多花100块住得近，省下来的时间值500。

### 🍜 餐厅推荐理由示例（3条）

**示例1：东京·一兰拉面（新宿店）**
> 第一次去东京可以打卡一次，毕竟名气在那。但如果你已经去过了，同样价格不如去旁边的富田拉面，本地人排队更多。一兰最大优点是一个人吃不尴尬——隔间设计，点完餐不用跟任何人说话。

**示例2：京都·锦市场附近·京极かねよ（鳗鱼饭）**
> 这家在锦市场走到头左转的巷子里，不太起眼。推荐它是因为"きんし丼"——蛋皮盖在鳗鱼饭上面，掀开那一下很有仪式感。价格比其他鳗鱼饭便宜30%左右，味道不输名店。中午去，晚上排队太长。

**示例3：大阪·黑门市场**
> 老实说，黑门市场已经很游客化了，价格比外面贵40%以上。如果只是想体验氛围可以逛逛，但不建议在这里"吃饱"。真要吃海鲜，黑门市场往南走10分钟有家"つるとんたん"（乌冬面），本地人去得更多，价格合理。

### 🎯 景点推荐理由示例（3条）

**示例1：东京·TeamLab Borderless（丰洲）**
> 你行程里第2天下午没什么安排，正好去。建议下午3点入场，人少一些，而且有些作品在光线变暗后效果更好。穿深色衣服拍照更出片——白色衣服会被投影"吃掉"。记得提前买票，现场经常售罄。

**示例2：京都·伏见稻荷大社**
> 千本鸟居就不多说了，必去。关键是**时间**：早上7点前到，可以拍到没人的鸟居通道；8点以后全是旅行团。如果你不想爬到山顶（来回2小时），走到"四ツ辻"就好，那里有个观景台能看到京都全景，性价比最高的折返点。

**示例3：箱根·大涌谷**
> 安排在箱根那天的上午。说实话风景一般，就是冒烟的火山口，但**黑�的温泉蛋**值得吃一个——据说能延寿7年（当然是传说）。关键是这里是箱根登山路线的一个中转点，你反正要坐缆车过来，下来逛30分钟就够了，不用专门为它安排太多时间。

### 写法检查清单

- [ ] 读起来像是跟朋友聊天，不像产品说明书？
- [ ] 有没有至少1个**具体细节**（距离/时间/价格/菜品名）？
- [ ] 有没有说清楚**适合谁/什么情况**去？
- [ ] 有没有诚实说**缺点或注意事项**？
- [ ] 删掉了所有"强烈推荐""不容错过""必去"这类空话？

## A22. 免费预览：包含什么 / 不包含什么

> 原则：给到"让人想要更多"的程度，不多不少

### ✅ 免费预览包含

| 内容 | 具体说明 | 目的 |
|---|---|---|
| **封面页** | 完整展示，含城市名/天数/人数标签 | 第一印象建立品质感 |
| **行程概览页** | 展示完整的"每日一句话亮点"列表 | 让用户看到全貌，产生"想看详情"的欲望 |
| **路线总览地图** | 完整展示城市串联和交通方式 | 路线逻辑清晰可见 |
| **Day 1 完整时间轴** | 全部内容不遮挡：景点/时间/交通/小贴士 | 核心卖点的"试吃装"，用品质说话 |
| **Day 2 前半天** | 展示到中午12:00，下午起模糊+锁定 | 制造"刚好到精彩处被打断"的感觉 |
| **基础交通概览** | "你的行程适合买Suica卡"一句话结论 | 有用但不够详细 |

### ❌ 免费预览不包含

| 内容 | 处理方式 | 设计理由 |
|---|---|---|
| **Day 2下午 ~ 最后一天** | 模糊处理 + 🔒图标 | 核心内容付费解锁 |
| **酒店推荐报告** | 仅显示板块标题 + 1句话tease | "你的行程建议住新宿附近，具体酒店→完整版" |
| **餐厅推荐报告** | 仅显示板块标题 + 1句话tease | "我们为你挑了14家餐厅→完整版" |
| **Plan B 备选方案** | 完全不展示 | 交付后惊喜 |
| **行前准备清单** | 仅显示目录（6大类标题） | 看到目录就知道很全面 |
| **安全须知** | 完全不展示 | 交付后惊喜 |
| **路线设计理由** | 完全不展示 | 交付后惊喜 |
| **避坑指南** | 完全不展示 | 付费核心价值 |
| **拍照攻略** | 完全不展示 | 深度版独有 |

### 免费预览页数对比

| 项目 | 免费预览 | 完整版 |
|---|---|---|
| 总页数 | 5-6页 | 30-40页 |
| 可读内容比 | ~15% | 100% |
| 品质感知 | 100%（排版一样好） | 100% |

## A27. 问卷体验规范

### 总体原则

| 原则 | 规则 | 理由 |
|---|---|---|
| **像聊天不像填表** | 每题一屏，大字引导语，无表头无编号 | 填表感 = 离开率 |
| **图标优先** | 除第7题可选输入外，全部图标/卡片选择 | 手指点两下比打字快10倍 |
| **速度感** | 轻问卷30秒（4题），正式问卷60秒（4题） | 超过90秒流失率翻倍 |
| **可跳过** | 正式问卷每题都有"跳过"选项 | 不强迫 = 好感度 |
| **不要手机号** | 全程不收集手机号/邮箱 | 微信支付后用OpenID触达 |
| **progressive disclosure** | 正式问卷一题一题滑出，不一次展示8题 | 减少认知负载 |

### 视觉规范

```
背景：#FAFAFA（暖白）
卡片：#FFFFFF + 1px #E8E8E8 边框 + 8px 圆角 + 微阴影
选中态：边框变 #FF6B6B + 背景 #FFF5F5 + 图标放大1.1x
引导语：18px #333 + 600粗细
选项文字：14px #555
进度条：4段圆点，当前=实心珊瑚红，未来=空心灰
过渡动画：卡片从右侧滑入（0.3s ease-out）
```

### 交互规范

| 行为 | 规则 |
|---|---|
| 选中后跳转 | 单选题：选中后0.3s自动滑到下一题 |
| 多选确认 | 多选题：底部出现"下一步"按钮 |
| 回退 | 左滑或点左上角"←"可回到上一题修改 |
| 跳过 | 每题右上角灰色"跳过"文字链 |
| 键盘 | 不弹出键盘（除第7题文字输入） |
| 震动反馈 | 选中时轻微震动（iOS haptic feedback） |
| 加载 | 任何操作响应时间 <100ms |

### 异常处理

| 场景 | 处理 |
|---|---|
