# 异常处理 SOP

> 最后更新: 2026-03-28
> 适用角色: 运营人员、技术值班

---

## 异常分级

| 级别 | 定义 | 响应时间 | 示例 |
|------|------|---------|------|
| P0 紧急 | 系统不可用，用户无法下单/无法收到手账 | 30分钟内 | 数据库挂了、任务队列全部积压 |
| P1 严重 | 功能受损，部分订单受影响 | 2小时内 | 生成链持续报错、PDF 渲染失败 |
| P2 一般 | 质量问题，不影响核心流程 | 当天处理 | 文案质量差、某实体数据错误 |
| P3 低 | 优化项 | 下个迭代 | UI 显示异常、日志过多 |

## 常见异常及处理流程

### 1. 生成链报错（订单卡在"生成中"）

**表现**: 订单状态长时间停在"assembling"，Trace 页面显示某步骤报错

**排查步骤**:
1. 进 Trace 页面，找到报错步骤
2. 查看错误信息

**常见原因与处理**:

| 错误 | 原因 | 处理 |
|------|------|------|
| `select_city_circle failed` | 目的地无对应城市圈 | 确认目的地在支持范围内；不在则转人工 |
| `eligibility_gate: 0 entities passed` | 该城市无足够实体数据 | 确认城市数据是否已导入；不够则转人工 |
| `rank_major_activities: empty` | 主活动排序后为空 | 检查实体 is_active 和 quality_tier |
| `hotel_strategy failed` | 无合适酒店 | 检查该城市酒店数据；可临时降低筛选条件 |
| `build_route_skeleton timeout` | 骨架构建超时 | 检查天数是否过长（>10天需拆册）；重试 |
| `AI client error` | OpenAI API 调用失败 | 检查 API Key 余额；等待恢复后重试 |

**操作**:
- 可修复的问题：修复后在订单详情点击"重新生成"
- 不可修复：手动推进状态到"review"，转人工处理

### 2. 质量门控失败（订单进入"待审核"）

**表现**: 质量门控报 error/warning，自动进入人工审核队列

**常见规则触发**:

| 规则 | 含义 | 处理 |
|------|------|------|
| `empty_day` | 某天无任何安排 | 审核 → 打回重生成 |
| `missing_entity` | 引用的实体不存在 | 检查实体是否被删除；修复后重生成 |
| `duplicate_entity` | 同一实体出现多次 | 审核 → 打回重生成 |
| `budget_mismatch` | 预算档与实际选择不匹配 | 检查实体 budget_tier 标注 |
| `too_many_slots` | 某天安排过多（>6） | 审核判断；合理则放行，不合理则打回 |

### 3. PDF 渲染失败

**表现**: 订单状态停在"reviewed"，render_export 任务报错

**排查**:
1. 检查 `ExportAsset` 表是否有记录
2. 查看 worker 日志中 `render_export` 的错误

**常见原因**:

| 错误 | 处理 |
|------|------|
| WeasyPrint 超时 | 检查页面数量；页数过多需拆册 |
| 图片 URL 无法访问 | 检查图片链接有效性；替换或使用 placeholder |
| CSS 解析错误 | 联系技术排查模板问题 |
| 内存不足 | 重启 worker；考虑减少并发 |

**操作**: 修复问题后，手动重新入队 render_export 任务

### 4. 多模型评审全部打回

**表现**: 评审结论 = REWRITE，重写后仍失败 → 转 HUMAN

**处理**:
1. 查看 4 个角色的评审意见
2. 判断是系统性问题（如数据不够）还是个例
3. 系统性问题 → 补数据/调配置后批量重生成
4. 个例 → 人工审核，必要时手动编辑后发布

### 5. 用户投诉内容不准确

**表现**: 用户反馈景点已关闭/价格不对/位置错误

**处理**:
1. 在内容库中找到对应实体
2. 核实信息（Google Maps / 官网 / Tabelog）
3. 更新实体数据
4. 如果是严重错误（如景点永久关闭）：
   - 立即下架实体（is_active = false）
   - 编辑加权 -8
   - 检查近期是否有其他订单使用了该实体
   - 受影响的未交付订单：重新生成

### 6. 任务队列积压

**表现**: Redis 队列中待处理任务持续增长

**排查**:
1. 检查 worker 是否存活
2. 检查 Redis 连接
3. 检查是否有任务持续失败导致重试积压

**处理**:
- Worker 挂了 → 重启 worker
- Redis 连接问题 → 检查 Redis 服务状态
- 重试积压 → 清理死信队列，修复根因后重新入队

### 7. 数据库连接异常

**表现**: API 返回 500、worker 报 connection error

**处理**:
1. 检查 PostgreSQL 服务状态
2. 检查连接池是否耗尽（`max_overflow` 配置）
3. 检查是否有长事务锁住了连接
4. 必要时重启 API 服务和 worker

## 升级路径

```
运营发现异常
  ├→ P2/P3: 运营自行处理（参照上述 SOP）
  ├→ P1: 通知技术值班
  └→ P0: 通知技术值班 + 暂停接新单

技术值班
  ├→ 能修复: 修复 → 验证 → 恢复
  └→ 不能修复: 升级到技术负责人
```

## 日常巡检清单

每天上班第一件事：

- [ ] 后台首页：有无积压在"生成中"超过30分钟的订单
- [ ] 后台首页：有无积压在"待审核"超过24小时的订单
- [ ] 评测结果：近期 overall 分数趋势是否下降
- [ ] Worker 状态：任务队列长度是否正常（< 10）
