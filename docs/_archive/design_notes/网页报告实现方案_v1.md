# 网页报告实现方案 v1

## 1. 结论

建议继续用 **HTML 实现网页报告**，但不是“拼接一整段 HTML 字符串”，而是：

- **Next.js App Router + React Server Components** 负责页面组织与数据装配
- **每一种类型页对应一个固定 React 页面模板（page-type template）**
- **模板下再拆 slot/block 组件**，而不是一页一个自由长文 section
- **Web 预览与 PDF 导出共用同一套 page plan / page view model**
- **Satori 只负责封面图、章节 opener 图、分享图等 hero image**
- **Playwright 负责从 HTML 页面导出 PDF**

不建议改成：

- Canvas / 图片拼整本报告
- 先产 PDF 再回推网页
- 每页都走 Satori 生成图片
- 纯 markdown -> 通用渲染器自动流式排版

原因：

1. 你们当前技术栈已经是 Next.js + Tailwind + shadcn/ui + Playwright，天然适合做 **HTML/React 页面系统**。Next.js App Router 本身就是基于 React 组件、Server/Client Components、layouts 和 route handlers 的页面架构。citeturn213651search0turn213651search13turn213651search7
2. Playwright 的 `page.pdf()` 本来就是从页面生成 PDF，并且默认消费 **print CSS**，所以最稳的路线是“网页是源，PDF 是导出产物”。citeturn213651search2turn346997search0
3. Satori / `ImageResponse` 更适合动态图片与 OG image，不适合承载整本正文报告。Next.js 的 `ImageResponse` 官方定位就是“动态生成图片”；Satori 本身输出 SVG，也更接近图像层而不是长文档正文层。citeturn213651search3turn213651search19turn346997search1
4. 你们当前样例 PDF 实际已经是 **HTML + CSS 文档**，里面直接有 `<!DOCTYPE html>`、大量 CSS class、timeline、report-block、prep-grid 等结构，说明现状本身就是 HTML 先行，只是还没有抽象成稳定的页型系统。fileciteturn28file8 fileciteturn28file2 fileciteturn28file14

## 2. 不建议的几种路线

### 2.1 不建议：一个大 HTML 模板自由流式输出

现有样例就是偏这条路：封面、时间轴、report-block、亮点、注意事项都在同一份长 HTML 里自然往下流。结果就是：

- 前面信息偏稀
- 后面日页信息偏密
- 同一页承担太多阅读任务
- PDF 分页被动依赖内容长度

这类问题已经在当前样例里暴露得很明显。fileciteturn28file5 fileciteturn28file12

### 2.2 不建议：每页导出成图片再拼 PDF

缺点：

- 文本不可选中，不利于搜索、复制、辅助功能
- 字体与打印质量控制更差
- 文件体积更大
- 页面迭代成本高
- 前端页面和 PDF 页面会分叉成两套系统

### 2.3 不建议：整本报告交给 Satori

Satori 很适合 hero image、章节图、社交分享图；不适合做几十页正文，尤其是存在复杂栅格、分页、长文本、组件复用、屏幕/打印双模式的时候。citeturn346997search1turn213651search3

## 3. 推荐实现架构

### 3.1 总体路线

采用：

**结构化数据 -> page plan -> page view model -> HTML/React page templates -> Web / PDF 双输出**

也就是：

1. 上游规划链路生成 `report_payload_vNext`
2. `page_planner` 根据页型规则产出 `page_plan[]`
3. `page_vm_builder` 为每一页组装对应 `PageViewModel`
4. Next.js 用固定页型组件渲染 HTML
5. Web 端直接浏览 HTML
6. Playwright 打开报告路由并导出 PDF

## 4. 每一种类型页是不是固定一个模板

答案是：**是，但不是“只有一个死模板”，而是“每一种页型有一个固定模板家族”。**

更准确地说：

- 每种 `page_type` 对应一个固定 React 模板
- 模板内部有固定的 slot 顺序
- 同类页允许 2~3 个 layout variants
- 所有页都遵守统一的 spacing / typography / hero / highlight 规则

### 4.1 推荐 page_type

```ts
export type PageType =
  | 'cover'
  | 'toc'
  | 'trip-strategy'
  | 'preference-fulfillment'
  | 'circle-opener'
  | 'major-activity-detail'
  | 'hotel-strategy'
  | 'hotel-detail'
  | 'day-execution'
  | 'restaurant-detail'
  | 'photo-guide'
  | 'shopping-flex'
  | 'reservation-key-window'
  | 'departure-checklist'
  | 'dynamic-caution'
  | 'appendix';
```

### 4.2 推荐 page template contract

```ts
interface ReportPageTemplate<TVM> {
  pageType: PageType
  variant?: string
  supportsPrint: boolean
  supportsWeb: boolean
  render: (vm: TVM) => React.ReactNode
}
```

### 4.3 为什么不是一个超级模板

因为你要的不是“把内容放进去就行”，而是：

- 不同页承担不同阅读任务
- 不同页有不同的情绪重点
- 不同页有不同的视觉重心
- 不同页有不同的数据需求
- 不同页有不同的 print break 规则

所以必须页型化，而不是一把梭。

## 5. Web 与 PDF 如何共用

### 5.1 共用什么

共用：

- `report_payload_vNext`
- `page_plan[]`
- `page_view_model[]`
- page type templates
- slot components
- design tokens

### 5.2 不共用什么

不强求完全共用：

- screen 与 print 的容器宽度
- 页眉页脚策略
- hover / 展开 / 锚点导航
- 目录交互
- 某些仅 Web 有意义的微交互

### 5.3 print 层建议

Playwright 导 PDF 时直接消费 print CSS。官方文档说明 `page.pdf()` 使用 print media；MDN 也建议用 `@media print`、`@page`、`break-before` / `break-inside` 等规则控制打印分页。citeturn213651search2turn213651search12turn213651search1turn213651search6turn213651search18turn213651search10

所以：

- Web 页面正常用 screen 样式
- PDF 导出时走 `@media print`
- 页型模板内部要显式声明 break 规则
- 不要依赖浏览器“自己分页”

## 6. 页面生成方式

### 6.1 推荐：服务端先组 page plan

在 Next.js App Router 下，推荐服务端先拿到报告数据并生成页面计划，然后页面组件只做渲染。

推荐流程：

```txt
Route / page.tsx
  -> fetch report payload
  -> build page_plan
  -> build page_view_models
  -> render <ReportShell pages={pages} />
```

这样做的好处：

- 结构稳定
- 适合导 PDF
- 适合缓存
- 易于测试
- 页型之间不互相污染

### 6.2 不推荐：前端组件边渲染边自己决定分页

因为：

- 内容长度波动大
- PDF 很难稳定
- 难做验收
- 会重新走回“自然流 section”老路

## 7. 具体实现建议

### 7.1 Page Plan 层

```ts
interface PlannedPage {
  pageId: string
  pageType: PageType
  variant?: string
  order: number
  sectionId?: string
  objectId?: string
  chapterId?: string
  printBehavior: {
    forceBreakBefore?: boolean
    avoidBreakInside?: boolean
    namedPage?: string
  }
}
```

职责：

- 决定有哪些页
- 决定页顺序
- 决定每页属于哪个主题对象
- 决定是否单独起页

### 7.2 View Model 层

```ts
interface BasePageVM {
  pageId: string
  pageType: PageType
  title: string
  subtitle?: string
  hero?: HeroAsset
  highlight?: HighlightBlock
  meta?: Record<string, string | number>
}
```

每一种页型再扩展自己的 VM，比如：

- `HotelDetailPageVM`
- `DayExecutionPageVM`
- `RestaurantDetailPageVM`
- `PhotoGuidePageVM`

### 7.3 Template 层

每一种页型一个组件：

- `CoverPage`
- `TOCPage`
- `TripStrategyPage`
- `HotelDetailPage`
- `DayExecutionPage`
- `RestaurantDetailPage`
- `PhotoGuidePage`

这些组件只消费 VM，不自己拉数据。

### 7.4 Slot / Block 层

复用小块：

- 标题区
- Hero 图
- 亮点卡
- 指标条
- 时间轴
- 选择理由
- 预约信息
- 替代方案
- Checklist

这样固定页型之下还能高复用。

## 8. 排版与模板策略

### 8.1 建议采用“固定模板 + 少量变体”

例如 `day-execution`：

- `variant = compact`
- `variant = standard`
- `variant = heavy`

什么时候变体：

- 5 天短行程：多用标准/展开型
- 14 天长行程：多用紧凑型 + 章节合并页

### 8.2 每页只承担一种任务

这条要写死：

- 执行页只讲怎么跑
- 酒店页只讲为什么住这里
- 餐厅页只讲为什么吃这家
- 拍摄页只讲何时何地如何拍

不要再把餐厅、酒店、拍摄、时间轴、Plan B 混成一个大页。当前 PDF 就存在这种问题。fileciteturn28file5 fileciteturn28file6

## 9. 设计感怎么落到实现上

网页报告不是靠“多写文案”给设计感，而是靠：

- 页型切分
- hero 区
- 标题层级
- 关键指标条
- 高亮块
- 留白
- 图片位置固定
- 一页一个视觉中心

### 9.1 建议用 Satori 的地方

只用在这些地方：

- 封面 hero
- 城市圈 opener hero
- 章节过渡图
- 分享图 / OG image

### 9.2 不建议用 Satori 的地方

- 时间轴
- 酒店详情正文
- 餐厅详情正文
- checklist
- 多段说明正文

## 10. 最终推荐方案

### 10.1 选型结论

**网页报告继续用 HTML/React 实现。**

准确说是：

- 用 **Next.js App Router 页面 + React 组件** 输出 HTML
- 用 **固定 page type template** 渲染每一类页
- 用 **Tailwind + shadcn/ui + 少量 print CSS** 管样式
- 用 **Satori / ImageResponse** 生成 hero image
- 用 **Playwright** 从 HTML 页面导出 PDF

### 10.2 最终架构一句话

**HTML 是主表达层，page type template 是主组织方式，Satori 是图片资产层，Playwright 是 PDF 输出层。**

### 10.3 落地优先顺序

1. 定 `page_type` 列表
2. 定 `page_plan` schema
3. 定每种页型的 `PageViewModel`
4. 先做 4 个核心模板
   - cover
   - trip-strategy
   - day-execution
   - hotel-detail
5. 加 print CSS
6. Playwright 联调 PDF
7. 再补餐厅页、拍摄页、动态注意事项页

---

## 11. 是否要固定一个模板

最终建议可以写成一句很硬的话：

**每一种类型页固定一个主模板；每个主模板允许少量布局变体；所有页型共享统一 slot 组件与设计 token。**

这比：

- 一份自由 HTML 长文
- 每天一个万能模板
- 页面导出成图片
- PDF 与 Web 各做一套

都更稳，也最符合你们现有技术栈。
