## 分配原则

### 模型选择标准

| 模型 | 适用场景 | 成本 |
|------|---------|------|
| **Opus** | 需要深度思考、跨步骤决策协调、用户体验判断、质量审核 | 高 |
| **Sonnet** | 明确的数据处理、规则应用、列表操作、结构化转换 | 中 |
| **Haiku** | 简单的分类、格式检查、轻量级校验 | 低 |
| **系统规则** | 硬约束检查、时间计算、定休日查表、资格过滤 | 无 |

### 思考强度定义

| 强度 | 定义 | 例子 |
|------|------|------|
| **深度** | 需要理解全局上下文、权衡多个目标、做战略性选择 | 选城市、规划每日强度、冲突处理 |
| **普通** | 给定明确的输入和规则，执行结构化操作 | 缩候选池、填时间线、选餐厅 |
| **轻量** | 直接调用系统接口或简单规则 | 定休日查表、计算travel time |

## 第一阶段：宏观方案（Step 1-4）

### Step 1：收集用户约束
- **执行方**：系统接口（不用AI）
- **思考强度**：无
- **说明**：从用户输入、表单、API 直接映射到结构化数据，没有判断
- **预期输入**：用户表单
- **预期输出**：UserConstraints JSON

```
用时预估：< 1 min（纯数据映射）
```

---

### Step 2：喂"区域级摘要"给 AI
- **执行方**：系统预处理（不用AI）
- **思考强度**：无
- **说明**：从候选库提取每个城市的高层摘要数据，不做任何判断
- **预期输入**：完整的 POI 候选库、餐厅库、酒店库
- **预期输出**：RegionSummary 数据结构

```typescript
{
  "region_summary": {
    "kyoto": {
      "style": "古都、传统、寺庙",
      "star3_count": 13,
      "star2_count": 56,
      "base_suitability": "high"
    }
  }
}
```

**技术选型**：
- 可以预计算（每周更新一次）
- 用 SQL 聚合（按 grade、city 分组，统计计数）

---

### Step 3：AI 输出城市组合候选
- **执行方**：**Opus**（深度决策）
- **思考强度**：**深度**
- **说明**：需要理解用户需求、季节特性、交通、是否能满足"必去景点"、时间搭配，输出多个方案给用户选
- **输入**：
  - UserConstraints
  - RegionSummary
  - 当前季节的开花期/祭典日期
  - 机场交通数据
- **输出**：CityPlanCandidate[] (Top 2-3)

**Prompt 框架**：
```
你是日本旅行专家。根据用户画像、预算、日期、已锁定项目，
为他生成 2-3 个城市组合方案。
每个方案需要说明：
  - 为什么这个城市组合适合他
  - 每城停留几天
  - 每天的主题建议（文化日/美食日/等）
  - 每天的强度建议（light/medium/heavy）

请同时输出"下一步需要确认的"（比如是否能接受长通勤）
```

**用时预估**：2-3 min（一次完成，无需迭代）
**成本**：1 个 Opus 调用
**token 预算**：~2-3K input, ~1-2K output

---

### Step 4：系统按主方案缩"玩的候选池"
- **执行方**：系统规则
- **思考强度**：无
- **说明**：给定城市后，从完整库里按规则切出一小包
- **规则**：
  ```
  - city IN (选定的城市列表)
  - grade IN (S, A, 精选B)
  - season_relevant (当前季节是否相关)
  - NOT (用户在约束里明确禁忌)
  ```
- **输出**：缩减后的候选 POI 列表（通常 50-100 个而不是 400+）

**技术实现**：
- 直接 SQL 查询
- 根据 best_season 字段过滤

**用时预估**：< 1 min
**成本**：无

## 回归测试 Checkpoint 1：宏观决策

**何时运行**：Step 4 完成后

**测试内容**：
1. ✅ 用户约束能否正确映射
2. ✅ 区域摘要数据是否准确（计数、分布）
3. ✅ Opus 生成的城市方案是否逻辑自洽
   - 日期和天数是否匹配
   - 强度分布是否合理（不能连续 heavy）
   - 是否有 light 日作缓冲
4. ✅ 缩减后的候选池是否符合规则
   - 城市是否正确
   - 等级是否符合
   - 季节是否过滤

**测试用例**：
- 用例 1：情侣，5天，首次关西，标准预算
  - 期望：京都3天+大阪2天 或 京都2天+大阪2天+奈良1天
- 用例 2：亲子，4天，有老人，经济预算
  - 期望：交通轻松的组合，强度偏 light

**Pass/Fail 标准**：
- ✅ 输出的方案数 = 2-3（不多不少）
- ✅ 每个方案的天数总和 = 用户日期数
- ✅ 候选池大小在 50-100 之间
- ✅ Opus 方案能被 Sonnet/Opus 再次验证为合理

## 第二阶段：锁主活动 + 住宿（Step 5-7.5）

### Step 5：AI 选主活动、锚点、每日片区
- **执行方**：**Opus**（深度决策）
- **思考强度**：**深度**
- **说明**：这是行程的骨架决策。需要：
  - 理解用户画像和主题搭配
  - 判断哪些是真正的"必看"，哪些是"可选"
  - 定位每日的地理中心（走廊）
  - 权衡多样性和连贯性
- **输入**：
  - CityPlanCandidate（从 Step 3 选出）
  - 缩减后的 POI 候选池
  - UserConstraints（用户偏好、禁忌）
  - 每日主题建议（来自 Step 3）
- **输出**：DailyPlan[] with mainActivities, corridors, anchors, optionalModules

**Prompt 框架**：
```
根据用户画像和每日主题，为这个城市组合分配主活动。

规则：
  - 每天只有 1-2 个"真正的"主活动（不能贪心）
  - 必须满足 Step 3 建议的强度（light/medium/heavy）
  - S/A 级景点优先
  - 同一片区的景点尽量集中（减少折返）

输出包括：
  - day1-5 各自的主活动
  - 可插拔的 1 星点
  - 每日的主片区（走廊）
  - 任何硬锚点（日落、演出等）
```

**用时预估**：3-5 min
**成本**：1 个 Opus 调用
**token 预算**：~3-4K input, ~2-3K output

---

### Step 5.5：主活动初筛校验
- **执行方**：系统规则 + Sonnet 轻量检查
- **思考强度**：普通（规则应用）
- **说明**：立刻检查 Step 5 选出的主活动是否在分配的那天营业
- **规则**：
  ```
  FOR EACH main_activity:
    IF activity.open_days NOT IN (当前日期的周几):
      MARK as "需改期或换方案"
  ```
- **如果有冲突**：
  - 系统输出"冲突列表"
  - Sonnet 快速给出 2-3 个同级替代方案
  - 不中断流程（标记为"待人工决策"）

**技术实现**：
- 系统规则：查表 when.open_days
- Sonnet 调用（只有冲突时）：快速替代方案生成

**用时预估**：< 1 min（规则检查），1-2 min（Sonnet 冲突时）
**成本**：可能的 1 个 Sonnet 调用（仅当冲突时）

---

### Step 6：系统缩住宿区域和酒店候选
- **执行方**：系统规则
- **思考强度**：无
- **说明**：根据 Step 5 选出的主片区，找最合适的 2-3 个住宿区域
- **规则**：
  ```
  1. 计算主片区的地理中心（加权平均）
  2. 找距离中心最近的 2-3 个潜在住宿区域
  3. 从这些区域里提取酒店候选（预算匹配、评分排序）
  ```
- **输出**：HotelOption[] with candidates

**技术实现**：
- 地理计算（纬经平均）
- SQL 距离排序

**用时预估**：< 1 min
**成本**：无

---

### Step 7：AI 选住宿区域与酒店
- **执行方**：**Sonnet**（普通决策）
- **思考强度**：普通
- **说明**：从候选里选最合适的住宿区域和具体酒店
- **输入**：HotelOption[] + UserConstraints + Step 3 的每日片区
- **输出**：
  ```json
  {
    "primary": {
      "region": "higashiyama",
      "hotel": "hotel_a",
      "check_in": "15:00",
      "check_out": "11:00"
    },
    "backup": {...}
  }
  ```

**Prompt 框架**：
```
选择最适合的住宿区域和酒店。考虑：
  - 用户预算
  - 到每日主片区的距离
  - 酒店的 experience_grade（如果有，优先选 S/A）
  - 入住出的时间便利性

输出：主方案 + 1-2 个备选
```

**用时预估**：1-2 min
**成本**：1 个 Sonnet 调用
**token 预算**：~2K input, ~1K output

---

### Step 7.5：住宿区与每日片区通勤检查
- **执行方**：系统 + Google Routes API
- **思考强度**：无
- **说明**：生成住宿区→各日主片区的通勤矩阵，检查是否合理
- **调用**：Google Routes API `computeRouteMatrix`
- **规则**：
  ```
  IF travel_time > 45 min:
    MARK as RED (建议换住宿区或改日程)
  IF travel_time 30-45 min:
    MARK as YELLOW (可接受但需注意)
  ```

**用时预估**：1-2 min（API 调用）
**成本**：Google Routes API cost

## 回归测试 Checkpoint 2：骨架锁定

**何时运行**：Step 7.5 完成后

**测试内容**：
1. ✅ Step 5 的主活动数是否合理（每天 1-2 个）
2. ✅ Step 5.5 的定休日检查是否有遗漏
3. ✅ Step 7 选出的住宿是否符合用户预算
4. ✅ Step 7.5 的通勤矩阵是否没有过多 RED

**测试用例**：
- 用例 1：京都3天，higashiyama 住宿
  - 每天通勤时间应该 < 20 min
- 用例 2：京都+大阪，中点住宿
  - 京都日的通勤应该 20-30 min
  - 大阪日的通勤应该 15-25 min

**Pass/Fail**：
- ✅ 主活动总数 = 城市数 × 2-3（不能太少或太多）
- ✅ 没有定休日冲突（或冲突已标记）
- ✅ 通勤矩阵没有 RED（或 RED 已说明理由）

## 第三阶段：每日时间线（Step 8-12）

### Step 8：系统整理每日约束包
- **执行方**：系统规则
- **思考强度**：无
- **说明**：编制每一天的所有硬约束
- **数据来源**：
  - 天文数据（日出日落）
  - 实体数据库（定休日、营业时间）
  - 交通数据（低频交通时刻）
  - Step 5 输出（锚点）
- **输出**：DailyConstraints[] with sunrise, sunset, closedEntities, anchors, etc.

**技术实现**：
- 预计算天文数据（季节性，可缓存）
- 查表定休日
- 硬编码或从数据库读交通时刻

**用时预估**：< 1 min
**成本**：无

---

### Step 9：AI 选每日最优顺序
- **执行方**：**Opus**（深度决策）
- **思考强度**：**深度**
- **说明**：最复杂的一步。需要：
  - 理解时间窗约束
  - 优化地理顺序
  - 平衡效率和体验
  - 预留合理的缓冲
- **输入**：
  - DailyPlan（主活动、片区、可选项）
  - DailyConstraints（时间窗、锚点、低频交通）
  - POI 间的距离矩阵
- **输出**：
  ```json
  {
    "daily_sequence": {...},  // 排好的顺序
    "remaining_slack_minutes": {...},  // 每天还剩多少时间
    "insertion_slots": {...},  // 哪里能插补充项
    "drop_order": [...]  // 超载时的删除顺序
  }
  ```

**Prompt 框架**：
```
为每一天设计最优的活动顺序。

约束：
  - 时间窗锚点（日落、演出等）不能动
  - 低频交通必须接上（只有几班）
  - 地理上要最优（不能无谓折返）
  - 每天必须预留缓冲（light 20%, medium 15%, heavy 10%）

输出：
  1. 按时间排列的活动顺序
  2. 每天还剩多少空闲时间
  3. 哪些时间段能插入可选项
  4. 如果超载，先删哪些可选项
```

**用时预估**：5-8 min（最复杂的一步）
**成本**：1 个 Opus 调用
**token 预算**：~4-5K input, ~3-4K output

---

### Step 10：系统做每日时间线可行性检查
- **执行方**：系统规则 + Haiku 轻量判断
- **思考强度**：轻量
- **说明**：用硬规则检查 Step 9 的输出是否可行
- **检查项**：
  ```
  1. 时间窗冲突？
  2. 交通时间是否赶得上？（用距离矩阵）
  3. 容量是否超载？（light ≤ 80%, medium ≤ 85%, heavy ≤ 90%）
  4. 动线是否反常？（往返超过 2 次？）
  ```
- **输出**：FeasibilityCheck with status (pass/fail) + violations

**技术实现**：
- 时间计算（简单减法）
- 距离查表（预计算的矩阵）
- 容量计算（加法）

**用时预估**：< 1 min
**成本**：无

---

### Step 11：冲突处理链
- **执行方**：系统规则（第 1-3 步）+ Opus 回退（第 4 步）
- **思考强度**：**深度**（仅 Opus 回退时）
- **说明**：这是系统和 AI 的分界线
  - 第 1-3 步：系统自动修复（删可选项、压时间、降强度）
  - 第 4 步：只有链条都走完，才回退 AI 重选

**处理链**：
```
Step 11.1: 系统自动删可选项
  IF 容量超载 AND 有可插拔项:
    FOR EACH item IN drop_order:
      DELETE item
      IF 现在满足容量:
        RETURN "处理成功"

Step 11.2: 系统自动压缩非核心停留
  IF 还是超载 AND 有 2 星点:
    FOR EACH poi IN 2_star_activities:
      IF poi.duration > 30 min:
        poi.duration -= 15 min
      IF 现在满足容量:
        RETURN "处理成功"

Step 11.3: 系统自动降强度
  IF 还是超载:
    intensity = heavy → medium
    recalculate 容量
    IF 现在满足容量:
      RETURN "处理成功"

Step 11.4: 回退 Opus
  IF 三步都走完还是不行:
    CALL Opus 说：
      "这个主活动组合在一天内物理上不可行。
       已删除的项：[...]
       已压缩的项：[...]
       已降级的强度：[...]
       请重新选择主活动或改变日期分配"
```

**用时预估**：
- Step 11.1-11.3：< 1 min（通常在这里解决）
- Step 11.4：2-3 min（仅在特殊情况）

**成本**：
- 99% 情况：无（系统规则）
- 1% 情况：1 个 Opus 调用（回退）

---

### Step 12：AI 输出最终每日时间线骨架
- **执行方**：**Sonnet**（普通数据处理）
- **思考强度**：普通
- **说明**：从 Step 9+11 的结果，生成精确到分钟的时间线
- **输入**：Step 11 处理后的可行方案
- **输出**：
  ```json
  {
    "day2": [
      {"time_start": "06:45", "time_end": "08:15", "activity": "清水寺", ...},
      {"time_start": "08:15", "time_end": "08:40", "type": "transit", ...},
      ...
    ]
  }
  ```

**Prompt 框架**：
```
根据已排序的活动，生成精确的时间线。

规则：
  - 每个活动的开始和结束时间都要明确
  - 考虑交通时间（从距离矩阵得出）
  - 餐厅位置作为 placeholder（下一步再填）
  - 标注固定锚点（不能改的）
```

**用时预估**：2-3 min
**成本**：1 个 Sonnet 调用
**token 预算**：~2K input, ~2K output

---

## 回归测试 Checkpoint 3：时间线可行性

**何时运行**：Step 12 完成后

**测试内容**：
1. ✅ 每日的时间线是否连贯（没有时间重叠）
2. ✅ 锚点是否被正确锁定（不能改的时间段）
3. ✅ 交通时间是否符合距离矩阵
4. ✅ 每天的容量是否满足设定的强度

**测试用例**：
- 用例 1：Day 2 有日落锚点 17:30
  - 倒推：最后一个活动必须在 16:30 前结束
  - 倒推：倒数第二个活动的交通必须在 15min 内
- 用例 2：Day 2 有低频缆车 14:00 和 16:00
  - 时间线必须对齐其中一班
  - 不能卡在中间时段

**Pass/Fail**：
- ✅ 每日时间线从早到晚连贯（没有空隙或重叠）
- ✅ 所有锚点都被尊重（时间不变）
- ✅ 每日占用时间符合 Step 3 建议的强度
- ✅ 容量利用率在 80-90% 之间（有缓冲，不过紧）

## 第四阶段：餐厅、预算、Plan B（Step 13-15）

### Step 13：系统按"每日路线"切餐厅候选
- **执行方**：系统规则
- **思考强度**：无
- **说明**：根据 Step 12 的时间线，知道每天何时在哪个片区，按此切餐厅候选
- **规则**：
  ```
  FOR EACH day:
    FOR EACH meal_time (noon, evening):
      IF day_schedule.time[noon] IN corridor_X:
        candidates = restaurant WHERE corridor_tags CONTAINS corridor_X
      candidates += restaurant WHERE best_pairing_corridors CONTAINS corridor_X
      limit to Top 20 by rating
  ```
- **输出**：RestaurantOption[] by day and meal time

**用时预估**：< 1 min
**成本**：无

---

### Step 13.5：AI 选餐厅与备选
- **执行方**：**Sonnet**（普通决策）
- **思考强度**：普通
- **说明**：从候选列表里选主餐厅和备选，考虑位置、菜系、价格、用户偏好
- **输入**：RestaurantOption[]  + UserConstraints + 前几天的餐厅历史
- **输出**：
  ```json
  {
    "meals": {
      "day2_lunch": {
        "restaurant": "rest_a",
        "time_window": "12:00-13:00",
        "reservation_required": false,
        "backup": "rest_b"
      }
    }
  }
  ```

**规则**：
- 菜系不连续重复（昨天吃了拉面，今天不能又吃拉面）
- 价格在用户预算内
- 预约失败直接切 backup（不回推改主骨架）

**用时预估**：2-3 min
**成本**：1 个 Sonnet 调用
**token 预算**：~2K input, ~1K output

---

### Step 14：预算核算
- **执行方**：系统规则
- **思考强度**：无
