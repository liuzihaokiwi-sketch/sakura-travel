# 关键决策记录

> 记录影响系统走向的重大决策。每条决策包含:背景、选择、理由、后果。
> 最后更新:2026-04-20(新增 D32 / D33,吸收 route_study 架构决策类)

## 文档第一性原理

这份文档回答"**过去为什么做了这些决策**"。
- 不回答"现在系统长什么样"(看 `03_数据契约/`)
- 不回答"怎么操作"(看 `04_操作SOP/`)
- 不回答"推迟了什么事"(看 `DEFERRED.md`)

**作用**:改架构前必读。避免重犯过去已经想清楚的选择。每条决策都应该**能解释"为什么不这样做"**。

## 精髓(唯一判断锚)

> **决策的价值在于"未来某个人想改这个时,他读完这条会知道为什么不能简单改"。**

好的决策记录:
- 不只写"做了什么",写"为什么"+"代价是什么"+"如果反悔会怎样"
- 写的时候假设 6 个月后的自己读不懂当时的背景 → 必须写清背景
- 被后续决策覆盖的保留,加链接指向新决策,不删

## D1: 从"AI写攻略"转向"结构化手账交付系统"

**日期**: 2026 Q1
**背景**: 最初定位是 AI 生成旅行攻略文本，输出长报告。
**选择**: 转为60页纸质手账本+贴纸DIY包的系统化交付。
**理由**: 长文本报告用户不读；纸质手账本有实体价值感，支撑298/348定价。
**后果**: 渲染层从"输出HTML"变为"60页页面系统"，需要 chapter/page/view-model 三层架构。

## D2: 从日本单国转向六城市圈

**日期**: 2026 Q1
**背景**: 最初只做日本（东京/大阪/京都）。
**选择**: 扩展为6个城市圈：关西、关东、北海道、华南五城、北疆、广东。
**理由**: 单一目的地天花板低，城市圈抽象可复用决策链。
**后果**: 决策链需要 policy bundle 机制（一条管线 + 多策略包），不是做6个 planner。

## D3: 抖音表单优先，不是独立站优先

**日期**: 2026 Q1
**背景**: 曾默认独立站是主入口。
**选择**: 抖音表单采集为主入口，独立站后置。
**理由**: 抖音有流量，独立站冷启动难；客服驱动的业务模型更适合表单采集。
**后果**: intake 层按表单结构设计，不按网站注册流程设计。

## D4: 消灭 Report-First 胶水层

**日期**: 2026-03-27
**背景**: 旧链路是 决策链 → report_generator(AI) → layer2_handoff → page_planner。中间多了"结构化→文本→再结构化"的绕路。
**选择**: 删除 report_generator.py 和 layer2_handoff.py，创建 planning_output.py 从 DB 直接构建页面数据。
**理由**: 决策链已经产出了结构化的 ItineraryPlan + DayFrame，不需要先转成"报告文本"再解析回来。
**后果**:
- 主链减少2个模块、约1900行代码
- AI 文案生成解耦为独立的 copy_enrichment.py（可选步骤）
- 页面结构先定，文案后填——结构不依赖 AI 成功

## D5: 老链不做回退，显式失败

**日期**: 2026-03-27
**背景**: 旧代码中城市圈管线失败会回退到 assemble_trip 老模板引擎。
**选择**: 删除 assembler.py，主链失败 = 显式失败 → 进人工队列。
**理由**: 只要老链能"悄悄成功"，新链的 bug 就永远发现不了。显式失败是驱动新链完善的唯一力量。
**后果**: fallback_router.py 中所有 use_legacy_* 返回 False。失败的请求进 human review queue。

## D6: 城市圈策略用"一条管线+多策略包"

**日期**: 2026 Q1
**背景**: 扩展到6个城市圈时，可以选择做6个独立 planner 或1个通用管线。
**选择**: 一条管线 + policy_resolver 按圈注入策略。
**理由**: 管线逻辑（选圈→排活动→酒店→骨架→填充）对所有圈通用，差异在策略参数不在流程。
**后果**: 新增 policy_resolver.py + config_resolver.py，7个正式策略模块。

## D7: 页面系统而非报告系统

**日期**: 2026 Q1
**背景**: 旧渲染是"一整篇HTML报告"。
**选择**: chapter_plan → page_plan → page_view_model → render adapter 的页面系统。
**理由**: 60页手账本需要精确到页的控制（页型、布局、DIY区域、贴纸位）。报告无法做到。
**后果**: 17种页型定义，页面级编辑覆盖，预算裁剪机制。

## D8: 多模型评审而非单模型

**日期**: 2026 Q1
**背景**: 单模型审核容易遗漏特定维度的问题。
**选择**: 6角色链式评审（QA/用户代理/运营代理/微调守卫/终审 + 多模型交叉）。
**理由**: 不同角色关注不同维度；Claude 擅长推理，GPT-4 擅长格式检查。
**后果**: review pipeline 成为独立模块，支持 publish/rewrite(max 2)/human 三路分流。

## D9: 不做航班服务，机票由用户自理

**日期**: 2026-03-28
**背景**: 曾建过 amadeus_client + price_monitor + notifier 的航班监控子系统。
**选择**: 删除整个 flights 模块。用户自行预订机票，表单中填写落地/返程时间。
**理由**: 系统定位是"落地后旅行规划"，不是机票代理。航班监控增加维护成本但不在核心价值链上。
**后果**: 删除 `domains/flights/`、`scan_flight_prices` job、amadeus 配置。TripProfile 中 arrival/departure 时间保留但来源是用户填写，不是航班查询。

## D10: 不做深度比价，仅做当日选择级性价比

**日期**: 2026-03-28
**背景**: 是否需要"整条路线方案级别的比价优化"——比如比较A方案住新宿 vs B方案住浅草哪个总价更低。
**选择**: 不做跨天/跨方案比价。仅在当天酒店选择和餐厅选择时，将性价比作为评分维度之一。
**理由**:
- 方案级比价需要预算全局优化，复杂度高但用户感知弱（用户不会看到"我帮你省了 2000 块"）
- 用户真正关心的是"这家酒店值不值"、"这餐厅性价比如何"——这是实体级判断
- 在 ranking/scorer 的 soft_rules 中，price_band 和 budget_level 匹配已经覆盖了"别推超预算的"
**后果**: 不需要全局预算分配算法。budget_strategy 是声明式的（"高价餐集中后半段"），不是优化式的。

## D11: 质量档取代精细评分

**日期**: 2026-03-28
**背景**: 现有评分体系 final_score(0-100) + editorial_boost(-8~+8)，导致 82.3 永远排在 82.1 前面，0.2 分差距无意义但系统当真。
**选择**: 改为5档质量分（1-5）+ 4档预算（低/中/高/超高），同档内完全平权，选择由上下文决定。
**理由**:
- 假精度比无精度更糟——系统总选同一家，导致推荐集中
- 同档平权迫使系统用菜式/距离/轮转等因素选择，自然产生多样性
- 编辑维护"从4降到3"比"从82.3调到78.6"直觉
**后果**: ranker 逻辑重构，实体表新增 quality_tier/budget_tier 字段。

## D12: 避雷优先于推荐

**日期**: 2026-03-28
**背景**: 推荐一家差餐厅对用户体验的伤害 >> 错过一家好餐厅的损失。
**选择**: 三层过滤——硬黑名单（永不推荐）→ 风险标签（默认过滤，用户可放开）→ 正常池。默认保守。
**理由**: 一次糟糕体验毁掉整本手账的价值。宁可少推荐，不能推差的。
**后果**: 实体新增 risk_flags 字段，表单新增"我可以接受"选项（默认不勾=帮你避开）。

## D13: 前置层大方展示，不模糊化

**日期**: 2026-03-28
**背景**: P13 预览应该模糊化防白嫖，还是大方展示炫技？
**选择**: 大方展示。具体酒店名、餐厅名、招牌菜全给。
**理由**:
- 能看到前置层的人都已付款（先付款后填表）
- 下单前的转化靠旧攻略样本/视频，不靠前置层
- 前置层的作用是让已付款用户确认方向 + 感受"钱花对了"
- 白嫖前置层得到的是"去东山区域"级别信息，真正值钱的每日执行细节在解锁层
**后果**: 前置层12-18页全量展示，退款在解锁前可全额退。

## D14: 定价锚定7天 ¥198

**日期**: 2026-03-28
**背景**: 7天是最大客群，定价应锚定在这个天数上。
**选择**: 7天国内 ¥198（前期优惠），公式 ¥198 + (天数-7) × ¥20，国外 +¥30，拆册 +¥29。
**理由**:
- ¥198 在 ¥200 心理关口以下，冲动购买最佳区间
- 每天 +¥20 用户心算简单
- 拆册 +¥29 感知是"工本费"不是"加价"
**后果**: 5天 ¥158（入门），7天 ¥198（主推），10天 ¥258，11天+ 拆册。后期恢复原价基础 +¥70。

> 2026-04 更新：产品定价已调整为国内 ¥298 / 国外 ¥348。

## D15: 事实层更新策略 — 后台定期扫，不在生成时检查

**日期**: 2026-04-11
**背景**: 票价、营业时间、预约政策等事实数据会变动，需要一套更新机制。讨论过"用户生成行程时触发检查"的方案。
**选择**:
- 默认 TTL 180 天（半年）
- 不在用户生成时检查，生成走缓存，不被更新卡住
- 运营定期主动扫超期数据并刷新
- 刷新失败用旧数据标记过期，不阻塞生成
**理由**:
- 生成时检查会拖慢响应，且加失败点
- 当前规模（模板级交付）事实变动频率低，半年粒度够用
- 运营主动扫比按需触发更可控，便于批量处理
**后果**: 实体/事实表新增 last_verified 字段；需要一个运维脚本定期扫超期项；短期内不做细粒度 TTL 和热点预刷，等实际跑起来有问题再补。

## D16: 用户纠偏回流系统 — 现阶段不做

**日期**: 2026-04-11
**背景**: 用户在预览页修改的每个动作都是模板质量的真实反馈信号——改了什么、改成什么，可以反向指导模板迭代。
**选择**: 现阶段不做记录和回流机制。
**理由**: 当前用户量极小，数据样本不足以形成有意义的质量信号，过早建系统是空转。
**后果**: 暂时接受"模板靠专家闭门造车"的现状；等产品有稳定用户后再启动，届时需要前端埋点 + 后端记录 + 运营看板三件套。

## D17: 数据可靠性就是"三源核对 + Plan B 兜底"，不建复杂分级

**日期**: 2026-04-11
**背景**: 常驻数据（景点/餐厅/酒店）和活动数据（樱花/红叶/祭典）的可靠性判断是否要统一一套规则。CLAUDE.md 原本有 verified/cross_checked/single_source/ai_generated 四级分级。
**选择**:
- 数据可靠性上限 = 三个权威源核对一致，到此为止，不再追求更高
- 剩余的现场风险由 Plan B 兜底（临时休业、排队、天气）
- 常驻和活动数据都走同一逻辑，不建两套标准，也不做系统化自动判级
**理由**:
- "数据是否真实"没有预言家，三源核对就是能做到的上限，用户自己去查也是这个信息密度
- Plan B 已经是系统现成的兜底机制，覆盖了"数据对了但现场变了"的情况
- 复杂分级规则最后还是靠人肉判断，系统化投入和回报不成正比
**后果**: CLAUDE.md 里的分级字段保留作为"数据录入时的自我提醒"，不作为自动过滤依据；不建专门的数据质量系统；维护时心里有数即可。

## D18: 实体停业 = 改开门时间字段，不加新机制

**日期**: 2026-04-11
**背景**: 实体（餐厅/景点）永久关闭时是否需要联动下线模板。
**选择**:
- 不加 closed_permanently 这类新字段
- 开门时间字段表达一切——营业写时间，关闭写空/关闭状态
- 维护定休时顺手改
- 餐厅关门：靠候补池自动替换
- 景点关门：手动改受影响模板的可用时间（核心景点没了就下线模板）
- 系统不做自动联动，全部运维流程
**理由**: 当前规模手工处理够用，加字段和联动机制是过度设计。
**后果**: 不新增字段，不新增检测逻辑，纳入日常运维流程。

## D19: 景点 note 与实体绑定，不写在模板里

**日期**: 2026-04-11
**背景**: 专家写模板时容易在每个用到同一景点的模板里重复写相同的提示（"清水寺早上人少"等），既费时也难维护。
**选择**: 景点级 note 放在实体表里，模板只引用 entity_id，运行时把 note 拼到当天的时段上。模板里只写"这一天特有"的东西，不写"这个景点永远成立"的话。
**理由**: 改一次所有模板都跟着更新；降低专家写模板的重复劳动；职责清晰——模板管编排，实体管自身属性。
**后果**: 实体表需要有 note/tips 字段；模板渲染时要合并两路 note。

## D20: 预览页推荐理由 — 暂缓设计，记录候选思路

**日期**: 2026-04-11
**背景**: 用户付费后看到行程会想"为什么推我这个"，需要展示匹配理由建立信任。
**选择**: 暂缓决定，不急着定方案。先记录候选思路。
**候选思路**:
- 想法 A：顶部一句话总结（扫一眼就懂）
- 想法 B：每天一句话（和当天内容绑在一起）
- 想法 C：关键景点后面加小字（只在需要解释处给理由）
- 想法 D：首页单独一页"为你选的原则"（整体设计理念）
**倾向组合**: D（首页原则） + B（每日理由），用户既看到整体逻辑也看到具体安排的理由。
**两条硬要求**:
- 理由必须是规则产出的，不让 AI 事后编——避免"理由美但和实际不符"
- 理由要人话不要字段名，需要字段到中文的翻译表
**后果**: 进入"想好了再做"队列；属于呈现层讨论的一部分。

## D21: 模板类型共 7 类

**日期**: 2026-04-11
**背景**: 讨论一个城市的模板要覆盖哪些场景。
**选择**: 7 类模板：
1. 主线经典版（围绕当前最高分景点生长）
2. 主线人群变体（仅默认版明显不合适时才做）
3. 主线季节版（仅季节高峰真改变体验才做）
4. 小众 / 文化深度
5. 季节独有景点补漏（平时排名一般但季节峰期为 S+）
6. 到达日 / 离开日（每个进出城市各一条，不安排核心景点）
7. 城市转场日（按城市对独立设计）

**关键判断**:
- day trip（不换住处的单向跨城，如住京都去奈良一天）= 主线日的子类，不是跨城日。判定标准："这一天要不要搬行李换酒店"
- 到达日 / 离开日 / 跨城日都不消耗核心景点
- 跨城日按城市对独立设计，核心景点是否进入由写模板的人当时决定，不由装配引擎运行时决定
**后果**: 造模板时明确知道自己在造哪一类，避免"所有日子都按主线日逻辑写"的混乱。

## D22: 模板从"第一景点生长"而非"分类后组装"

**日期**: 2026-04-11
**背景**: 讨论造一个城市的模板时的第一步。
**选择**: 挑当前城市综合评分最高的景点作为核心景点，围绕它生长出"完美一天"。第二条从剩余景点里再挑最高分，重复生长，直到剩下的景点无法独立成一天主题。小众/文化深度模板不走生长逻辑，单独写。
**理由**: 分类法（先把景点按类型分再拼日子）是从全局往下切，适合已经熟悉城市的情况，对 AI 是大包袱。从锚点生长每步都有具体判断对象，每条模板天然"有灵魂"，也和核心景点互斥机制天然吻合。
**后果**: 造模板流程的第一步从"定 S 级分类"改为"挑最高分景点作为核心景点"。

## D23: 经典版 = 大多数人的走法，不是最极致的走法

**日期**: 2026-04-11
**背景**: 讨论清水寺这种景点的默认版应该是"7:30 抢无人舞台"还是"9-10 点从容到场"。
**选择**: 经典版反映大多数人的走法，不是专家最推崇的极致方案。极致走法（早起版等）作为衍生版本单独写，仅在"骨架明显不同"时才独立写模板，否则在 note 里提示即可。
**理由**: 50%+ 的游客不会为了无人舞台 6:30 起床，强行把"早起版"设为默认是把策展变成说教。策展产品的本质是"最适合大多数人的那个"，不是"最极致的那个"。
**后果**: 清水寺东山日的默认版时间应该是 9-10 点到场；早起版仅在整天节奏灵魂不同时才独立成模板。

## D24: 滑动窗口——一条模板承载多种节奏

**日期**: 2026-04-11
**背景**: 讨论"早起/晚起/悠闲/紧凑"是否要写成不同模板。
**选择**: 一条模板用相对时间 + 锚点机制承载多种节奏，不写多条变体。
- 普通时段写"开场后 X 分钟"的相对时间
- 午餐时段硬约束落在 11:30-13:30（肚子不会随整体平移）
- 关键景点可标锚点时段（必须落在固定时段，超出则换模板）
- 上午时段吸收浮动差
**理由**: 模板数量爆炸是真实风险（一个核心景点 × 节奏 × 人群 × 季节 可达数十条）。用相对时间 + 锚点让一条模板通过"开场时间"参数适配多种节奏。
**后果**: 时段时间字段改为相对时间；模板新增锚点时段概念；装配引擎按用户节奏偏好决定开场时间，自动平移时段。

## D25: 早起是强制的，但不是天天

**日期**: 2026-04-11
**背景**: 讨论"用户不愿早起怎么办、要不要给早起/晚起两个版本"。
**选择**: 核心景点日强制早起，系统不给用户"要不要早起"的选项。
**规则**:
- 早起仅用于核心景点日（7 天行程 2-3 天，不是天天）
- 早起日的前一天自动"降温"——晚餐提前、不安排夜景酒吧、22:00 前就寝（装配引擎跨天联动）
- 早起时间是区间 6:30-7:30，不是单点
- 午餐硬约束 11:30-13:00，上午时段吸收浮动差
- 降级例外：团队有老人/小孩、到达次日
- 悠闲节奏 ≠ 晚起（悠闲 = 时段少 + 时长长 + 留白多，和早起独立）
**理由**: 策展产品的专家视角包含"替用户做他当下不愿做、但事后会感激的决定"。把早起做成选项是把策展拆散卖，退回定制工具。
**后果**: 表单不问"你愿意早起吗"；模板有早起日标记和早起时间窗口两个字段；前夜降温由装配引擎处理。

## D26: Plan B 分全局级和局部级

**日期**: 2026-04-11
**背景**: 讨论雨天/恶劣天气的应对机制。
**选择**: Plan B 分两层：
- **全局级**：模板加"天气敏感"标记。露天主导的模板（岚山竹林、哲学之道、大原）标 true，雨天装配引擎直接换一条室内模板
- **局部级**：非露天主导但某些时段是户外的模板，在对应时段描述里写雨天替代
**理由**: 整条都是户外的模板，雨天靠局部补丁救不了，只能整条换。区分两种 Plan B 避免硬塞"每个模板都有 Plan B"的无效工作。
**后果**: 模板新增天气敏感字段；装配引擎需要有"天气不好换模板"的逻辑。

## D27: 模板差异度看"灵魂"不看"区域"

**日期**: 2026-04-11
**背景**: 讨论一个城市写多条模板时怎么避免重复。
**选择**: 判断两条模板是否重复的标准是"灵魂"（体验主题/节奏/时段画面），不是地理区域。一个区域可以出几条完全不重复的模板（东山可以出清水寺早起日 + 八坂祇园夜生活日，灵魂完全不同）。
**配套**: 每个城市维护一份灵魂覆盖清单——列出已有模板的一句话灵魂，写新模板前查一眼。
**理由**: 地理重复不代表内容重复。反过来两条不同区域的模板都做"寺庙深度"才是真正重复。
**后果**: 新增每个城市一份灵魂覆盖清单文档（`templates_coverage_map.md`）。

## D28: 装配引擎用 Opus 驱动，不做规则引擎

**日期**: 2026-04-11
**背景**: 反复讨论"装配引擎到底是规则引擎 + 硬字段，还是 Opus + 自然语言"。这是今晚最核心的架构决策。
**选择**: 装配引擎就是 Opus。
**分工**:
- **系统做的机械工作**（薄薄一层）：模板有效性过滤（按季节/日期/人群等硬字段机械筛）、数据填充（按 Opus 产出的查询条件从实体池反查候选）、用户输入的确定性算术（算有效天数等）
- **Opus 做所有判断**：选模板、排序、跨天联动、选具体实体、写文案。硬约束作为 prompt 一部分，不用代码校验
**理由**:
- 规则引擎的死穴：专家判断力被拆碎后没地方整合，跨字段跨时段跨天的耦合关系无法处理
- Opus 能读自然语言，能处理整体感，能在全局中做联动判断
- 系统对 Opus 的每一步限制都会损失质量，限制越少发挥越好
- 手账本是批量产出不追求实时响应，Opus 1M 上下文完全装得下
**后果**: 大量原本为规则引擎预留的结构化字段可以砍掉或降级为自然语言；权威字典系统不再需要；模板从"结构化数据"变成"结构化文档"——必要的查询字段 + 给 Opus 读的自然语言。

## D29: 实体字段的两分法——客观事实 + 表单对应字段 + 自然语言

**日期**: 2026-04-11
**背景**: 在 D28（Opus 装配）基础上重新审视所有实体的字段设计。
**选择**: 结构化字段只有两类，其余全是自然语言：

**客观事实字段**（世界的属性，机器和人都要精确读）
- 景点：id / 区域 / 坐标 / 开门时间 / 门票 / 建议游览时长 / 是否需要预约 / 天气敏感
- 餐厅：id / 区域 / 菜系标签 / 适合餐型 / 开门时间 / 人均价格 / 预约难度
- 酒店：id / 区域 / 价格区间 / 入住退房时间
- 店铺：id / 区域 / 店铺类型标签 / 开门时间
- 所有实体：名称 / 评分（0-5）/ 风险标签 / 最后验证时间

**表单主观判断字段**（表单会问，实体要有落点让系统机械过滤）
- 共通：适合人群 / 预算档位 / 氛围标签 / 设施标签
- 酒店加：实用标签 / 质量偏好 / 体验偏好
- 餐厅加：AB 角色 / 餐次角色

**自然语言字段**：编辑一句话定位 + 三段专业点评（亮点/核心信息/缺点，中间段按实体类型不同）+ 景点额外有季节备注

**砍掉的旧字段**: 模板的排布位置/疲劳度/密度行为/前后偏好/出现次数控制；模板的餐厅引用字段；受控词表强制约束；餐厅的日期引用/替补去重字段；店铺的日期引用字段

**后果**: 实体条目大幅简化；写实体的人只需要想"表单会按什么过滤我" + "我是什么客观事实" + "用自然语言写清楚其他一切"。

## D30: 综合评分统一 0-5 浮点，档位内含

**日期**: 2026-04-11
**背景**: 讨论餐厅/酒店/景点/店铺的评分字段是用档位（S/A/B/C）还是数值还是两者并存。
**选择**: 所有实体统一用 0-5 浮点评分，档位由分数推导：4.0-5.0 = S，3.0-3.9 = A，2.0-2.9 = B，1.0-1.9 = C。
**理由**:
- 一档对应一分，直观易记，不需要额外档位字段
- 编辑综合多源信息（Tabelog / Google / 小红书 / 实地反馈）后产出一个数值
- D11 的假精度担忧在 Opus 时代不成立——规则引擎会因 0.1 分差距死板选同一家，Opus 能理解"4.1 和 4.3 基本同档"自己做多样性分配
**后果**: 造模板第一步"定 S 级"改成"从景点池挑评分 ≥ 4.0 的景点作为核心景点候选"；实体表不需要独立档位字段。

## D31: 分步装配——先选模板定酒店，再按天填餐厅

**日期**: 2026-04-11
**背景**: 讨论装配引擎 Opus 是一次搞全部还是分步。
**选择**: 分两步：
- **第 1 步**（一次 Opus 调用）：读用户画像 + 硬约束 + 有效模板池（系统过滤后约 30 条），产出每天用哪条模板 + 顺序。住处由每条模板写死的住宿区域决定，系统按区域 + 用户预算档从酒店池反查候选，Opus 或系统选定一家
- **第 2 步**（按天循环 N 次 Opus 调用）：每天一次调用，输入当天骨架 + 按区域/预算/人群/前几天已选菜系过滤后的餐厅候选，Opus 从中选 A/B 餐
**理由**: 一次性让 Opus 同时选模板 + 酒店 + 每餐任务太重注意力分散；分步后每步任务明确，错误率低；第 2 步每次上下文只约 15k，非常从容；总调用 1+N 次，批量产出完全承受得起。
**后果**: 装配引擎不是一次 API 调用，是一个按天循环的流程；每步的 prompt 模板需要分别设计。

## D32: 关西 v2 四层架构(知识 / 数据 / 装配 / 输出)

**日期**: 2026-04-17
**背景**: 单个模板同时在做 4 件事(手账本内容 + 装配指令 + 事实数据 + 策展思路),混一个 JSON 是所有混乱的根源。
**选择**: 四层分离:
- **Layer 1 真实数据层**(冷数据):`entities__{city}.json` + `live_facts__{city}.json` + 餐厅/酒店池。双角色:`display`(文案注入)+ `availability`(硬筛选)
- **Layer 2 策展知识层**(热知识):`templates/`——只写设计思路(为什么 8 点前去竹林 / 动线逻辑 / 峰值在哪 / 氛围),不写事实
- **Layer 3 装配层**:规则装配(按 `applies_when`/`excludes_when` 硬筛)+ Opus 判断(关键节点:选模板 / 餐厅 A/B / 酒店 / 夜间活动 / 最终审查)
- **Layer 4 手账本输出**:单一最终方案,不让用户选
**理由**:
- 分层后票价变了只改 live_facts 一处,不用改 21 个模板
- Opus 成本可控(只在判断节点,不在每 slot)
- 调试容易(错了知道在哪层)
- 质量稳定(规则不漂移)
**后果**: v1 的"模板+内容池"被拆成新四层;装配代码按 Layer 3 重写;entity 必须在 entities 层注册,不再有占位符。

## D36: 模板/事实层大瘦身 + 装配层 markdown 化 + 顶层模板花名册

**日期**: 2026-04-22
**背景**: 关西 98 个模板字段审计发现：模板字段从初始设计 30+ 演化到 50+，存在大量"以为有用其实没消费方"的字段；事实层 entity 也膨胀到 25+ 字段；装配规则散落在模板/事实层各处，且全部是 JSON——而装配实际由 Opus 读，JSON 嵌套 + 强 schema 反而让 AI 难读。
**选择**:
1. **模板大瘦身**：顶层从 30+ 砍到 7 必填 + 3 可选；slot 按 type 区分极简（poi 3 字段 / meal 4 字段 / hotel 2 字段 / transport 2 字段）
2. **slot 不再有 priority**：用 `slots: {packed, balanced, relaxed}` 三档**完整独立 slot 序列**取代 P1/P2/P3 砍法。写作者亲自排好三档，装配按用户密度档直接取用
3. **slot.note 取消**：整天思路写在模板顶层 `note`（不展示给用户的设计 note）；`description` 是给用户看的 note 提炼+钩子
4. **事实层瘦身到最多 15 字段**：必填 8 + 可选 7（且可选字段"有比较好的才写，不凑数"）
5. **景点附属内容（photo_spots / fun_facts / nearby_shops）从模板挪到事实层 entity**——跟着景点走，不跟着模板走
6. **装配层独立**：建 `japan/kansai/assembly/{模板装配,餐厅装配,酒店装配}.md` 三份 markdown，承载所有装配规则（打分、min_days、day_type、互斥、A/B 餐厅、酒店升档、关西 7 条化学反应等）
7. **顶层模板花名册**：所有模板的元属性（季节/人群偏好/打分/min_days/day_type/互斥/夜接/selectable_tag）集中在 `assembly/模板装配.md` 一份顶层 markdown 维护，**不分散到各个模板 JSON**
8. **events/ + live_facts__ 也迁 markdown**（方向已定，待执行）：消费方主要是 AI 装配
9. **字段扩展硬规**：禁止随意扩展字段。任何新字段先回答"消费方是谁 / 能否用现有字段表达 / 是不是装配层的事"

**理由**:
- 字段越多，写作者负担越重，写成 placeholder 概率越高（25/98 缺 tier 就是证据）
- "装配层的事"和"模板/事实层"混在一起，导致写作者要操心装配规则，越界
- AI 装配读 markdown 远比 JSON 自然，events.json 的 `_doc` 字段已自我吐槽"AI 读 description 综合判断"
- 顶层花名册让横向对比/批量调整成为可能（之前调一个打分要扫 98 个文件）
- "禁止随意扩展" 是过去 6 个月字段从 8 个膨胀到 50+ 的根本治理

**后果**:
- 模板字段从 30+ → 10 左右（必填 7 + 可选 3）
- 事实层字段从 25+ → 最多 15（必填 8 + 可选 7）
- 装配规则全部脱离模板/事实层，集中 3 份 markdown
- 98 个现有模板需要批量改造
- 现有 events/*.json 5 个文件待迁 markdown
- SCHEMA.md 重写，模板写作 SOP 顶部加覆盖段
**关联**: 本次窗口讨论沉淀（待生成 session memory）

## D37: 模板目录按动线重组 + slots 改单数组+时间段结构 + 节奏由 optional 表达

**日期**: 2026-04-22（跟 D36 同一窗口后半段）

**背景**: D36 定下"slots 三档 {packed, balanced, relaxed}" 和"模板按季节文件夹分" 后，再次讨论暴露两个问题：
1. 三档 slots 强迫写作者为每个模板写三套完整 slot 序列，**工作量翻倍**，且多数档位差异本质只是"加不加顺路景点"
2. 按季节分文件夹看不出"哪些模板属于同一条动线"——岚山 core / 岚山 koyo / 岚山 sakura 散在三个文件夹，无法感知动线全貌

**选择**:

1. **模板按城市/动线重组**：
   ```
   templates/
   ├── kyoto/
   │   ├── arashiyama/                  动线文件夹
   │   │   ├── index.md                 变体清单（标明每个编号是几日）
   │   │   ├── transport.md             该动线交通清单
   │   │   ├── 1.json 2.json 3.json ... 变体（数字编号，一日和两日统一编号不分子层）
   │   │   └── night.json               可选夜晚模块（一文件含多选项）
   │   ├── half_day/                    该城市半日池
   │   └── ...
   ├── osaka/
   ├── other/
   ├── arrivals/                        到达日
   ├── departures/                      离开日
   ├── special_dates/                   时间敏感型特殊日
   └── niche_spots/                     孤立小众景点
   ```
   
   **规则**：
   - **一日/两日统一编号，不分子层**。编号从 1 开始连续，index.md 标明"1=一日核心 / 2=两日温泉 / 3=两日红叶深度" 等
   - 同动线文件夹内变体**互斥**（用户只选一个）
   - 只有一日或只有两日的动线 → 平铺
   - 一日+两日并存的动线 → 也平铺，统一编号
   - **special_dates/**：时间敏感型（大文字 8/16、祇园祭 7/17、天神祭 7/24-25、琵琶湖花火、节分、通し矢、初诣、特别公开 等），装配按当年具体日期匹配才能装
   - **niche_spots/**：景点型孤立小众（铁道博物馆、醍醐寺樱花/红叶、三千院紫阳花、深冬雪景 等），全年或季节窗口可装
   - 这两个文件夹里每个文件 = 单一 JSON 独立模板，**不做变体互斥**

2. **同动线变体互斥**：一个动线文件夹内的所有 .json 变体**互相排斥**，装配每个动线文件夹最多选 1 个变体装入。

3. **一日 vs 两日分独立文件夹**：`arashiyama/`（一日）和 `arashiyama_2day/`（岚山温泉两日）动线不同，分文件夹。

4. **夜晚模板单文件多选项**：每个可配夜晚的动线文件夹含 `night.json`，**一个文件装多个夜晚选项（酒吧/夜パフェ/表演等）**，装配按用户人群/预算挑一个。岚山/USJ/有马/温泉一泊等默认无夜晚。**夜间参拜不做 night.json，并入对应红叶/樱花季白天模板**（本质是延时参拜不是独立夜晚活动）。

5. **half_day 每城一个池**：`templates/{city}/half_day/` 下装该城所有半日变体（伏见半日/宇治半日/岚山半日等），装配每段行程每个城市最多用一个半日模板。

6. **到达/离开/特殊日/小众景点 独立城市维度**：arrivals / departures / special_dates / niche_spots 各建独立文件夹，不挂任何城市。

7. **slots 结构改为"时间段 + main + optional"**（修订 D36 的三档 slots 设计）：
   ```json
   "slots": [
     {
       "time": "08:00-10:00",
       "main": [
         {"type": "poi", "entity": "kyo_arashiyama_bamboo"},
         {"type": "poi", "entity": "kyo_tenryuji"}
       ],
       "optional": [
         {"type": "poi", "entity": "kyo_kameyama_park", "note": "体力好/早到的用户加项"}
       ]
     }
   ]
   ```
   - **外层按时间段分组**（一天 5-8 个时间段）
   - **main** 必做（一段内可 1-N 个 slot）
   - **optional** 顺路推荐景点（不强制）
   - 装配按密度决定 optional 用不用：packed 全用 / balanced 用 1 个 / relaxed 不用
   - **默认节奏**：main + 0-1 个 optional。**独立 relaxed 版本**只在"高强度动线"（如东山）才单独做另一个变体

8. **缓冲时间 SOP**（写进 [模板写作.md](../04_操作SOP/模板写作.md)）：
   - 最佳游玩 < 1h 的景点 → +30 分钟缓冲
   - 最佳游玩 1-2h → +30-45 分钟
   - 最佳游玩 2-3h → +45-60 分钟
   - 最佳游玩 > 3h → +60 分钟
   - 缓冲是给用户"不一定掐点到、不一定准点结束"的稳态

9. **optional 推荐 SOP**：
   - 推荐景点步行可达（≤15 分钟）或 1-2 站电车
   - 主景点+推荐合并后总时间不超过 3 小时
   - 推荐景点必须真值得（不是"凑数"）

10. **文件命名**：
    - 动线变体：`1.json` / `2.json`（数字编号，前缀是父目录表达）
    - 动线夜晚模块：`night.json`
    - 动线文件夹：不带城市前缀（父目录已表达），如 `kyoto/arashiyama/` 不是 `kyoto/kyo_arashiyama/`

11. **全局装配导航**：`assembly/templates/index.md` —— 装配 AI 入口，含：
    - 所有动线/变体的筛选索引（动线 ID / 季节 / 人群适配 / 天数 / selectable_tag）
    - 跨动线互斥（关西 7 条化学反应，如宇治 vs 姬路/奈良不同日）
    - 装配打分公式
    - 到达/离开/半日模板的装配规则
    
    原 `assembly/templates/templates.md` 改名为 `assembly/templates/index.md`。

12. **模板顶层新增 `variant_label` 字段**：描述该变体是什么（如 "岚山红叶+情侣版"），给写作者和装配 AI 自描述，与文件名数字编号互为补充。

13. **模板顶层砍 selectable_tag / night_options**：
    - selectable_tag 挪全局 index.md 的变体元属性（D36 已砍）
    - night_options 不再需要——夜晚模块跟动线绑定（同文件夹有 night.json 才能挂）

**修订 D36 的具体项**:
- D36 §2 "slots 三档" → 改为 "slots 时间段 + main + optional 单数组"
- D36 §7 "模板花名册所有模板元属性集中一份" → 改为 "每个动线 index.md 自管 + 全局 index.md 只管跨动线规则与装配筛选索引"
- D36 未明确的"模板目录结构" → D37 明确按城市/动线重组

**D36 其他决策保留不变**：设计宪法两条、字段大瘦身、景点 entity 的 notes 自由文本子标题、装配层 markdown 化、assembly/ 三子系统结构。

**理由**:
- 三档 slots 工作量大，多数差异靠 optional 表达即可
- 按动线聚合比按季节聚合更贴近"产品单元"——岚山是一个产品，冬夏都是岚山
- 一个文件夹变体互斥是天然组织约束，装配逻辑清晰
- 夜晚模块跟白天动线绑定解决了"哪个白天能接哪个夜晚" 的配对问题

**后果**:
- 98 个模板需要**重新分类**到 22 个主流动线 + half_day × 3 + arrivals/departures/special_dates/niche_spots × 4 = **29 个顶层文件夹**
- slots 字段结构从三档对象改回单数组（格式升级）
- 写作者每个模板工作量降低（写一套 slots + optional 而非三套）
- 原 D36 落地工作计划作废，重写 D36_D37_落地工作计划.md
