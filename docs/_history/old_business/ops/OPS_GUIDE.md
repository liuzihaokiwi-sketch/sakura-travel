## 1. 订单管理

**入口**: 后台首页 `/admin`

### 看板视图

7列看板对应订单生命周期：

| 列 | 含义 | 操作 |
|----|------|------|
| 新建 | 用户刚提交 | 可创建详细表单 |
| 表单中 | 等用户填写详细表单 | 等待 |
| 待生成 | 表单已提交，等排队 | 点击"生成"触发 |
| 生成中 | AI 正在跑链 | 等待，可查 Trace |
| 待审核 | 生成完需人工审 | 进 Review 审核 |
| 已发布 | 审核通过 | 可查看/下载 PDF |
| 已交付 | 用户已收到 | 完成 |

### 订单详情

点击订单卡片进入详情页 `/admin/order/{id}`：

- **基本信息**: 用户名/目的地/天数/出发日/人数
- **状态推进**: 点击按钮推进到下一状态
- **创建表单**: 为该订单创建详细表单链接
- **计划预览**: 查看生成的行程概览
- **PDF 下载**: 在"已发布"状态后可用

### 关键操作

**触发生成**:
1. 确认详细表单已提交（状态 = "待生成"）
2. 点击"生成行程"按钮
3. 系统入队 `generate_trip` 任务
4. 在 Trace 页面追踪进度

**手动推进状态**:
- 仅在异常情况下使用（如系统卡住）
- 点击"推进状态" → 选择目标状态 → 确认

## 2. 实体内容库

**入口**: `/admin/catalog`

管理系统中所有的景点（POI）、餐厅、酒店数据。

### 搜索与筛选

| 筛选项 | 说明 |
|--------|------|
| 类型 | poi / hotel / restaurant |
| 城市 | osaka / kyoto / kobe / nara / ... |
| 数据层级 | S（深度完整）/ A（基本完整）/ B（骨架） |
| 关键词 | 中文名/英文名模糊搜索 |
| 状态 | 活跃/已下架 |

### 新建实体

1. 点击"新建"按钮
2. 选择类型（poi / hotel / restaurant）
3. 填写必填字段：中文名、城市代码
4. 填写类型专属字段（如 POI 的 poi_category、典型游览时长）
5. 保存

**字段说明**:

| 字段 | 说明 | 示例 |
|------|------|------|
| `name_zh` | 中文名（必填） | 清水寺 |
| `city_code` | 城市代码（必填） | kyoto |
| `area_name` | 区域名 | 东山 |
| `data_tier` | 数据层级 | S / A / B |
| `quality_tier` | 内容质量档 | S / A / B / C |
| `budget_tier` | 预算档次 | free / budget / mid / premium / luxury |
| `risk_flags` | 风险标签 | requires_reservation, weather_sensitive |
| `booking_method` | 预约方式 | walk_in / online_advance / phone |
| `best_time_of_day` | 最佳到访时段 | morning / afternoon / evening |
| `corridor_tags` | 走廊标签 | ["higashiyama", "gion"] |

### 编辑实体

- 点击实体行进入详情
- 修改任意字段后保存
- 修改 `data_tier` 会自动触发评分重算

### 下架实体

- 软删除：`is_active = false`，不再被决策链选中，但数据保留
- 硬删除：物理删除，仅在确认数据错误时使用

## 3. 编辑加权

**入口**: 实体详情页 → "编辑加权"

编辑加权让运营人员手动调整实体在推荐排序中的权重。

### 规则

| 参数 | 范围 | 说明 |
|------|------|------|
| boost_value | -8 ~ +8 | 正值提权、负值降权 |
| reason | 文本 | 调整原因（必填，方便回溯） |
| valid_until | 日期 | 过期时间，留空=永久 |

### 使用场景

| 场景 | 建议值 | 示例 |
|------|--------|------|
| 发现好店，想多推 | +3 ~ +5 | "本地人私藏拉面店，品质稳定" |
| 季节性推荐 | +2 ~ +4（设过期时间） | "樱花季岚山 +4，valid_until=4月15日" |
| 收到差评/投诉 | -3 ~ -6 | "近期多条差评反馈服务态度差" |
| 确认关闭/装修 | -8 | "暂停营业至6月" |
| 恢复正常 | 0 | 新增一条 boost=0 覆盖 |

### 查看历史

实体详情页 → "加权历史"，可查看所有 boost 记录及当前生效状态。

## 4. 配置管理

**入口**: `/admin/config`

### 配置层级

```
全局默认 → 城市圈级 → Segment 级（party_type 细分）
```

### 常用配置项

| 配置 | 说明 | 改动影响 |
|------|------|---------|
| 权重包 | 各评分维度权重 | 影响实体排序 |
| 阈值 | 最低评分/最大实体数等 | 影响选入行程的实体 |
| 功能开关 | enable_plan_b / enable_live_risk 等 | 影响生成流程 |

### 发布流程

1. 在配置编辑器中修改
2. 点击"保存草稿"
3. 在"预览"页面查看影响范围
4. 点击"发布" → 版本 +1，生效
5. 需要回滚：在历史版本中选择 → 点击"回滚到此版本"

## 5. Review 审核

**入口**: 后台首页看板 → "待审核"列，或 `/admin` 的审核 tab

### 审核流程

1. 系统生成行程后自动跑质量门控 + 多模型评审
2. 评审结论为 `HUMAN` 的单子进入人工审核队列
3. 点击进入审核页面，可查看：
   - 行程概览（天数/区域/主要景点）
   - AI 评审意见（4个角色的评分和建议）
   - 质量门控得分
   - 离线评测分数
4. 操作：
   - **发布**: 审核通过，进入渲染 → 交付
   - **打回**: 标记问题 → 重新生成
   - **编辑**: 微调后发布（调整个别实体/文案）

### 审核要点

- [ ] 天数是否完整（无空天）
- [ ] 景点是否适合用户画像（亲子/蜜月/独行）
- [ ] 节奏是否合理（不要一天排8个景点）
- [ ] 餐厅是否合理（价位/类型/预约难度）
- [ ] 酒店位置是否方便（交通/区域）
- [ ] 文案是否通顺（无 AI 味/模板感）
- [ ] Plan B 是否有（至少核心景点要有备选）

## 6. Trace 排查

**入口**: `/admin/trace` 或订单详情 → "查看 Trace"

Trace 页面展示某次生成的完整链路，用于排查问题。

### Trace 信息

| 步骤 | 展示内容 |
|------|---------|
| 城市圈选择 | 选中的圈、置信度 |
| 资格门控 | 通过/淘汰的实体数 |
| 主活动排序 | 选中的主活动列表 + 分数 |
| 酒店策略 | 住法（单点/双点）+ 选中酒店 |
| 骨架构建 | 每天框架（区域/时间/强度） |
| 次要活动 | 填充的次要景点 |
| 餐厅填充 | 每天的餐厅选择 |
| 质量门控 | 得分 + 触发的规则 |
| 评审结论 | 4个角色的评分 + 最终裁决 |

### 常见排查场景

| 问题 | 查看位置 |
|------|---------|
| 选了奇怪的景点 | 主活动排序 → 看评分明细 |
| 酒店位置不合理 | 酒店策略 → 看约束条件 |
| 天数不对 | 骨架构建 → 看日期框架 |
| 被质量门控挡下 | 质量门控 → 看触发的硬规则 |
| AI 评审打回 | 评审结论 → 看具体角色意见 |

## 7. 评测结果

**入口**: `/admin/evals`

查看离线评测对历史生成结果的打分，用于监控系统整体质量趋势。

| 维度 | 说明 | 合格线 |
|------|------|--------|
| completeness | 行程完整度 | > 0.8 |
| feasibility | 可行性（时间/交通） | > 0.7 |
| diversity | 多样性（类型/区域） | > 0.6 |
| overall | 综合 | > 0.7 |

## 8. 异常处理 SOP

> 适用角色: 运营人员、技术值班

### 异常分级

| 级别 | 定义 | 响应时间 | 示例 |
|------|------|---------|------|
| P0 紧急 | 系统不可用，用户无法下单/无法收到手账 | 30分钟内 | 数据库挂了、任务队列全部积压 |
| P1 严重 | 功能受损，部分订单受影响 | 2小时内 | 生成链持续报错、PDF 渲染失败 |
| P2 一般 | 质量问题，不影响核心流程 | 当天处理 | 文案质量差、某实体数据错误 |
| P3 低 | 优化项 | 下个迭代 | UI 显示异常、日志过多 |

### 常见异常及处理

#### 8.1 生成链报错（订单卡在"生成中"）

**表现**: 订单状态长时间停在"assembling"，Trace 页面显示某步骤报错

| 错误 | 原因 | 处理 |
|------|------|------|
| `select_city_circle failed` | 目的地无对应城市圈 | 确认目的地在支持范围内；不在则转人工 |
| `eligibility_gate: 0 entities passed` | 该城市无足够实体数据 | 确认城市数据是否已导入；不够则转人工 |
| `rank_major_activities: empty` | 主活动排序后为空 | 检查实体 is_active 和 quality_tier |
| `hotel_strategy failed` | 无合适酒店 | 检查该城市酒店数据；可临时降低筛选条件 |
| `build_route_skeleton timeout` | 骨架构建超时 | 检查天数是否过长（>10天需拆册）；重试 |
| `AI client error` | OpenAI API 调用失败 | 检查 API Key 余额；等待恢复后重试 |

**操作**: 可修复 → 修复后"重新生成"；不可修复 → 手动推进到"review"转人工

#### 8.2 质量门控失败

| 规则 | 含义 | 处理 |
|------|------|------|
| `empty_day` | 某天无任何安排 | 打回重生成 |
| `missing_entity` | 引用的实体不存在 | 检查实体是否被删除；修复后重生成 |
| `duplicate_entity` | 同一实体出现多次 | 打回重生成 |
| `budget_mismatch` | 预算档与实际选择不匹配 | 检查实体 budget_tier 标注 |
| `too_many_slots` | 某天安排过多（>6） | 合理则放行，不合理则打回 |

#### 8.3 PDF 渲染失败

| 错误 | 处理 |
|------|------|
| WeasyPrint 超时 | 检查页面数量；页数过多需拆册 |
| 图片 URL 无法访问 | 检查图片链接有效性；替换或使用 placeholder |
| CSS 解析错误 | 联系技术排查模板问题 |
| 内存不足 | 重启 worker；考虑减少并发 |

修复后手动重新入队 render_export 任务。

#### 8.4 多模型评审全部打回

1. 查看 4 个角色的评审意见
2. 判断是系统性问题（数据不够）还是个例
3. 系统性 → 补数据/调配置后批量重生成
4. 个例 → 人工审核，必要时手动编辑后发布

#### 8.5 用户投诉内容不准确

1. 在内容库中找到对应实体，核实信息
2. 更新实体数据
3. 严重错误（景点永久关闭）：下架 + boost -8 + 检查受影响订单

#### 8.6 任务队列积压 / 数据库连接异常

- Worker 挂了 → 重启 worker
- Redis 连接问题 → 检查 Redis 服务状态
- 重试积压 → 清理死信队列，修复根因后重新入队
- DB 连接耗尽 → 检查连接池 `max_overflow` 配置；必要时重启服务

### 升级路径

```
运营发现异常
  ├→ P2/P3: 运营自行处理（参照上述 SOP）
  ├→ P1: 通知技术值班
  └→ P0: 通知技术值班 + 暂停接新单
```

### 日常巡检清单

每天上班第一件事：

- [ ] 有无积压在"生成中"超过30分钟的订单
- [ ] 有无积压在"待审核"超过24小时的订单
- [ ] 评测 overall 分数趋势是否下降
- [ ] 任务队列长度是否正常（< 10）

| 角色 | 联系方式 | 职责 |
|------|---------|------|
| 运营 | （填入） | 日常订单管理、内容审核 |

### Docker 服务架构

```
docker-compose.yml 统一管理 7 个服务:

  postgres (pgvector:16)  ─── 数据库
  redis (7-alpine)        ─── 缓存/队列
  api (FastAPI)           ─── 后端 API
  worker (arq)            ─── 后台任务
  frontend (Next.js)      ─── 前端
  nginx                   ─── 反向代理 + HTTPS
  ai-ops                  ─── AI 运维助手
```

### 自动化运维

| 机制 | 说明 |
|------|------|
| **GitHub CD** | push main → CI 通过 → 自动 SSH 部署到 ECS |
| **Watchdog** | cron 每 5 分钟检查健康，异常自动重启 + 邮件通知 |
| **AI Agent** | 重启 3 次失败后调 AI 分析日志，每天上限 20 次被动调用 |
| **Sentry** | 后端 500 / worker 失败自动上报 |
| **邮件告警** | 部署成功/失败、服务异常都会发邮件到 QQ 邮箱 |
| **SSL 自动续期** | certbot cron 每 2 个月续期 |

### 常用运维命令

```bash
# 查看所有服务状态
docker compose ps

# 查看某服务日志
docker logs japan_ai_api --tail 50
docker logs japan_ai_worker --tail 50
docker logs travel-web --tail 50

# 重启某个服务
docker compose restart api
docker compose restart worker
docker compose restart frontend

# 重建镜像（代码更新后）
docker compose build api && docker compose up -d api worker

# 健康检查
curl http://localhost:8000/health

# 手动部署
bash deploy/deploy.sh
bash deploy/deploy.sh --backend
bash deploy/deploy.sh --frontend

# 同步本地数据库到 ECS（在本地执行）
bash scripts/sync-db-to-ecs.sh

# AI 运维助手
# 访问: https://kiwitrip.cn/ops-ai/
# 密码: admin123（可通过 .env 的 OPS_PASSWORD 修改）
```
