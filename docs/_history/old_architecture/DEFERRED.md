## 模块级推迟

### 1. 装配引擎（Prompt 工程和分步流程落地）
**是什么**: 装配引擎 = Opus 驱动，方向已定。推迟的是具体落地：给 Opus 的 prompt 怎么写、分几步（初筛 → 模板选定 → 餐厅酒店选定 → 跨天联动检查）、输出格式如何规范、错误重试机制。
**为什么推迟**: 模板内容池还没建完，没有真实数据就没法调 prompt。先把内容池做出来。
**什么时候启动**: 内容池有 20+ 模板、候选餐厅/酒店池建完后。
**相关决策**: D6、以及 Opus 装配的新方向决策（D21+）

### 2. 方案质检系统
**是什么**: 生成完的方案在交付前过一遍自动检查（硬约束/预算/节奏/去重/走廊闭合）。
**为什么推迟**: 先把模板和装配跑通，质检是兜底不是主链。
**什么时候启动**: 主链稳定后，上线前。
**相关决策**: D8（多模型评审）是对 AI 产出的评审，不是对模板方案的质检，两者不同。

### 3. 表单/展示层枚举标准化
**是什么**: 表单字段枚举（budget_tier / fit_audience / cuisine_tag 等）的统一词表，确保前端表单和内容池条目用同一套枚举值。
**为什么推迟**: 装配引擎改为 Opus 后，大量原来给规则引擎用的中间字典（meal_ref / hotel_area / assembly.position 等）已经不需要了。剩下的枚举（表单过滤和展示分类用的）等字段稳定后统一梳理。
**什么时候启动**: 内容池字段稳定、前端表单设计确定后，一次性对齐。
**相关决策**: 无专门决策。

### 4. 呈现层 / 预览页
**是什么**: 用户付款后看到的行程预览页的具体设计（含推荐理由展示方式）。
**为什么推迟**: 主链未完成，先保证结构正确再谈展示。
**什么时候启动**: 装配引擎跑通后。
**相关决策**: D20（预览页推荐理由候选思路，倾向 D+B 组合）

### 5. 事实层爬取系统
**是什么**: 票价 / 营业时间 / 预约政策的批量拉取和更新的统一入口。
**为什么推迟**: 会做成独立系统，和模板装配链解耦。
**什么时候启动**: 独立排期。
**相关决策**: D15（事实层后台定期扫，不在生成时检查）

### 6. 用户纠偏回流
**是什么**: 记录用户在预览页的修改动作，作为模板质量反馈信号。
**为什么推迟**: 当前用户量太小，样本不足以形成信号。
**什么时候启动**: 产品有稳定用户后。需要前端埋点 + 后端记录 + 运营看板三件套。
**相关决策**: D16

## 不做的事

- **机票服务**: 用户自理，系统只收落地/返程时间（D9）
- **跨方案比价**: 只做当日实体级性价比，不做全局预算优化（D10）
- **数据可靠性复杂分级**: 三源核对是上限，Plan B 兜底现场风险（D17）
- **实体停业联动**: 只改开门时间字段，不加新机制（D18）
