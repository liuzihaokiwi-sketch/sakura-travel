## 本文档定位

这份指引**给 AI 看**。AI 按这份指引生产 day 模板。人工专家也可以按此流程写，但主要读者是 AI。

模板是"专家写死的一日最优解"，是给 Opus 读的结构化文档，不是给规则引擎读的结构化数据。早上去哪、下午去哪、怎么衔接，全部由模板写死；餐厅和酒店的最终选定，由 Opus 读模板 flow 里的自然语言描述 + 内容池候选，自己决定。

## 产品定位（必读）

我们卖的是**策展产品**，不是定制工具。
用户付费买的是"专家帮你做好了决策，你照着走"，不是"给你一堆选项自己拼"。

核心判断标准：**一个真实的日本旅行顾问看到方案会不会认可。**

## 体验设计八条原则

这些是 AI 写模板时心里要有的标准。目的不是让每天都"很有设计感"，而是**防止景点堆砌——看起来很满但什么都没记住**。

> 原则 6、7、8 带主观词汇（人味、温度、降档体验），这些词需要对照样本才能稳定判断。每条后面都会附正反例，AI 判断时对着例子判断，不要对着形容词判断。

### 1. 一段行程 2-3 个情绪阶段，不是每天换主题

主题按**城市/阶段**自然切换，同一个城市内部不刻意制造情绪反差。

```
例：8 天关西
大阪（前段）：热闹、能量、城市感
京都（中段）：安静、深度、文化沉浸
神户/收尾（后段）：放松、味觉、余韵
```

真正的情绪转折发生在城市切换的时候——从大阪的喧嚣走进京都的安静，用户自然就感受到了。

### 2. 70% 保底，10% 才是"值了"

| 比例 | 内容 | 作用 |
|------|------|------|
| 70% | 核心 S 级景点 | 保证"不亏"，不为主题牺牲 |
| 20% | 次要搭配（咖啡/步行段/餐厅） | 立住当天的氛围 |
| 10% | 惊喜 moment | **用户真正记住的部分** |

用户回忆时记住的不是 70% 的热门景点（大家都去过），而是那 10%——摩天轮上日落的 15 分钟、小巷里的发现、师傅在面前切鱼的手法。10% 时间最短，选择要最用心。

### 3. B 选项要真的"惊"

B 不是"同类型另一家"，而是让用户眼前一亮。

- ✅ A 回转寿司 + B 路边关东煮 → "从没想过可以这样吃"
- ✅ A 串炸名店 + B 米其林焼鳥割烹 → "原来大阪还有这种水平"
- ❌ A 拉面 + B 另一家拉面 → 没有惊喜

判断样本：**用户看完想发给朋友说"你看这个"**，就是合格的 B。

### 4. 热门景点差异化靠"怎么玩"，不靠"去不去"

| 维度 | 说明 |
|------|------|
| 时间点 | 清晨/黄昏/夜晚感受完全不同 |
| 进入方式 | 从哪边切入，路上经过什么 |
| 搭配对象 | 同一核心景点+不同次要点=不同氛围 |
| 逗留方式 | 快进打卡 vs 慢慢散步 |
| 收尾方式 | 每天有个收束画面 |

### 5. 峰值要稀缺

不追求天天高潮。Peak-End Rule：人记住的是"最强烈的时刻"和"结尾"。

| 行程 | 大峰值 | 小峰值 |
|------|--------|--------|
| 6-8 天 | 1 个 | 1 个 |
| 9-13 天 | 2 个 | 1-2 个 |

峰值前要有铺垫（高光是一个序列不是一个点），峰值后要有余韵（不再塞同等级刺激）。

### 6. 人味——旅行不能只有空景

每 2-3 天至少有一个"有人在的 moment"。不需要每天安排，但整段行程不能全是"看建筑物"。

- ✅ 黑门市场早上的叫卖声 / 居酒屋街的笑声 / 寺庙钟声 / 祭典的人群
- ❌ 整段行程全是寺庙、花园、观景台——这叫"空景"，不及格

判断样本：翻一遍整段行程，数有多少个时段是"有真人正在生活/工作/热闹"的场景。每 2-3 天至少一个才算过关。

### 7. 降档也是体验

高档用户吃一顿路边关东煮不是省钱，是被设计的本地生活 moment。

- ✅ 中档行程里安排一顿站着吃的大阪烧 → "本地人每天的味道"
- ✅ 高档行程里安排一顿早晨的黑门市场刺身丼 → "市场工人早饭的级别"
- ❌ 降档没有理由，纯粹因为预算压力 → 用户会觉得被糊弄

判断样本：降档的那一餐必须能写出"为什么这家餐厅即便不贵也值得专门安排"的理由。写不出来就是纯省钱，不合格。

### 8. 文案有温度

每个推荐不是冷冰冰的信息罗列，而是让用户感受到"有个人在跟我说话"的文字。

- ❌ "大阪城，1583 年建，世界文化遗产"
- ✅ "你第一眼看到天守阁从樱花里冒出来的时候，会明白为什么大阪人管这儿叫'太阁さんのお城'"

- ❌ "清水寺，京都代表性景点之一"
- ✅ "清水寺必须 8 点前进，8 点以后游客激增，照片里会是人头；7:30 抵达子安塔背后可以拍到无人的舞台全景"

判断样本：一条 note 读完，用户能想象到那个画面/感觉/动作细节吗？能 = 有温度；只是罗列年代/类型/介绍 = 没温度。

## 体验设计补充六条(2026-04-12 基于外部研究补充)

> 上面八条原则从产品哲学推出来,下面六条来自对 Peak-End rule 研究、国外顶级定制社(Black Tomato 等)、中日旅行者反思贴的系统梳理。**都是"我们原本没考虑到但会严重影响体验"的盲区**,不是锦上添花。

### 9. 每一天都要有自己的 peak 和收尾画面

原则 5 讲了整段行程 1-2 个大峰值,但研究显示**旅行记忆按"天/城市段"分成多个 episode 分别打分**,不是一条平均线。意思是:
- **每一天都要有自己的"这一天的最强瞬间"**——不一定是 S 级景点,可以是一杯日落时的啤酒、一条意料外的小巷、一个市场的叫卖
- **每一天都要有自己的"收尾画面"**——不是"回酒店睡觉",而是"今天以这个画面结束"。花见小路的艺伎背影、岚山竹林黄昏的光、道顿堀的霓虹倒影都算
- **每个城市段也要有自己的"peak + end"**——大阪段最后一天不是"去京都前收拾",是"大阪的告别仪式"

判断样本:翻一遍整段行程,能不能对每一天说出"这天最强的瞬间是 X"和"这天结束时的画面是 Y"。说不出 = 这天没立住。

### 10. 规避负峰值比制造正峰值更重要

研究:一个被纠察员呵斥的合照、一次中午走不回去的体力崩溃、一次长队后的空手而归——**负峰值会永久覆盖当天所有好记忆**。一整天的好设计可以被一个 10 分钟的负峰值毁掉。

所以每条模板的 story 或 flow detail **必须显式标注"今天这里最容易出事的点 + 规避法"**。常见负峰值点:
- **文化纠察**(旺季京都特别严重):金阁寺合照人数 > 10 会被拦、神社台阶合照会被念、小巷拍照会被吼
- **体力崩溃**:伏见稻荷中午走到半山发现腿废了又不能折返;东大寺 → 春日大社 → 若草山连走
- **长队**:USJ 闭园挤地铁、清水寺下山赶坂道高峰
- **语言障碍**:预约的老店看不懂暗示、便利店取票机看不懂日文
- **预算踩坑**:居酒屋的お通し座位费、嵯峨野小火车旺季被迫站票

**硬规则**:模板里如果有任何"高风险负峰值"景点,flow detail 里必须写一行"规避方式"。写不出 = 模板不合格。

### 11. 同类体验要跨天去重,不只是菜系

研究里的 memory interference 效应:连续吃薯片,后吃的会抹掉前吃的记忆。同样,**连续三天寺庙,第一天的寺庙会被后两天擦掉**。

现有的"跨天菜系去重"已经做了餐厅维度,但**景点类型**维度没有。需要避免:
- Day 1 大阪城 + Day 2 姬路城 + Day 3 二条城(三天三个"日本城")
- Day 1 清水寺 + Day 2 东福寺 + Day 3 金阁寺(三天三个"京都寺庙")
- Day 1 海游馆 + Day 2 动物园 + Day 3 主题水族馆(三天三个"看动物")

判断样本:把 N 天的主要景点按类型分组(寺庙/神社/城/博物馆/市场/自然/主题乐园/夜景观景台...)。如果任一类型出现 3 次以上,而它们之间又没有显著的"进入方式/氛围/时段"差异 → 有类型抹杀风险,需要调整。

**不是说"不能连续两天寺庙"**——原则 4 讲过差异化靠"怎么玩"而不是"去不去"。能说出"清水寺的舞台俯瞰 vs 东福寺的通天桥红叶"这种差异,两天寺庙没问题。说不出 = 类型抹杀。

### 12. 最后一天要有"上扬的小峰值",不是中性的"轻松"

Step 2 里说"最后半天安排轻松",这一条要升级:**"轻松"是中性的,但研究显示结尾是记忆权重最重的位置**。中性结尾会把整段旅行的记忆拉平,上扬结尾会把整段记忆往上拉。

所以到达/离开日模板里,离开日的前半天应该是**一个小上扬峰值 + 平静收束**,不是"购物 + 去机场"。合格的例子:
- 早晨去黑门市场买能带走的食材当"告别仪式" + 中午一碗真的值得的乌冬 + 下午才去机场
- 早晨去一次早起才有的场景(鸭川边的晨雾 / 伏见稻荷无人的千本鸟居) + 咖啡早餐 + 平静退房
- **不合格**:"上午购物药妆 + 下午直接去机场"(这是中性收尾,等于把结尾权重浪费了)

**硬规则**:离开日模板的 flow 里必须有一个 flow 条目的 detail 显式说明"这是这段旅行的告别画面"。没有 = 不合格。

### 13. flow detail 要精确到车站出口和长队时段

原则 4 讲了"进入方式",现在把它升级成硬规则。中国用户在日本最常遇到的"摩擦点"不是大事,是:
- **走错车站出口**:Osaka-Umeda / Shinjuku / Kyoto 这种站有几十个出口,选错走 500-1000 米
- **长队高峰**:酒店 check-in 15:00-16:30 全城连锁商务酒店都排队 30 分钟
- **站内换乘时间**:超大站换乘 5 分钟绝对不够,通常 10-15 分钟
- **取票复杂度**:便利店打印预约票、USJ 快速通关换纸票,不熟的用户卡 15 分钟

所以 flow detail 里涉及到以上场景时**必须精确**:
- ❌ "从 Osaka 站去道顿堀"
- ✅ "从 Namba 站 14 号出口出来,直行 200 米过第一个十字路口就是道顿堀入口"

- ❌ "下午 3 点到酒店 check-in"
- ✅ "14:30 前到酒店寄存行李(避开 15:00 集中 check-in 高峰),15:30 之后再回来办手续"

这些细节单条看起来琐碎,但它们是原则 "基础面的无摩擦是护城河" 的具体落地。**用户付了 300 块,期待的就是这种免摩擦**。

### 14. 有些天要有"自由漫游段",但仍然是专家决策的结果

这是和"专家替你决策"哲学的一个微妙张力——用户潜在的渴望是"有一段时间我自己走走看看",但如果我们真的说"这半天你自己安排",就把决策权扔回给了用户。

**折中方案**:模板里可以设计"自由漫游段"——不是"什么都不排",而是**专家决定"在这里留白,给 2-3 个方向建议,用户自己选"**。比如:
- "下午 15:00-17:00 的先斗町自由漫游段:可以沿鸭川往北走到四条大桥(河边茶屋方向)、往南走进祇园花见小路(艺伎出没时段)、或向西钻进锦市场(小吃街)。三个方向都不会错,选一个顺着感觉走就行。"

这样的设计**本身是专家决策**(我决定在这里留白 + 我给你三个都不会错的方向),不是偷懒,不是把决策扔给用户。

**什么时候该安排自由漫游段**:
- 第 2 次以上来关西的用户(有基础熟悉度)
- 悠闲节奏档(表单第 3 屏)
- 连续走了 2-3 天高密度后的恢复段
- 情侣/闺蜜档,他们需要"两人独处的漫步时间"

**什么时候不安排**:
- 第一次来 + 紧凑节奏(他们期待的是被排满)
- 亲子档(有孩子反而需要明确指引)
- 到达/离开日(那两天已经有特殊设计)

### 15. 文案密度:手账本不要把自己的 prediction 定得太高

原则 8 讲了"文案要有温度",这一条是给它加边界。期望值研究显示:**如果预览页的文案把某个景点吹得过于精彩,用户到现场发现"和文案不完全一致"时会产生 prediction error,实际峰值感受会被削弱**。

所以两种文案要有不同的密度:
- **预览页文案(方案确认页,纠偏前)**:可以稍微往上拉,因为这是转化的关键页。"傍晚走进道顿堀的霓虹和人潮" 这种画面感是 OK 的
- **手账本文案(用户真正带去旅行的那本)**:应该**精确 + 有温度,但不夸大**。重点不是"让用户期待",而是"让用户到现场时感到'果然如此'"

判断样本:假设用户已经在现场,读一条手账本文案时,他会觉得"和文字写的一样"(合格)还是"没想象中那么……"(不合格,文案过度)?

反例:
- ❌ "你会在清水寺的舞台上看到一生难忘的京都全景" (过度渲染,万一当天雾大就翻车)
- ✅ "清水寺舞台朝西,下午 3-4 点光线最好,能看到整个京都盆地往岚山方向的层次" (精确 + 有期待空间但不夸大)

这一条不是让模板文案变平淡,是让"专家密度"取代"营销密度"。专家密度的文字本身就有温度,不需要靠形容词堆砌。

## 模板类型共 8 类

写模板前先判断是哪一类，不同类型的写法侧重不同：

| 类型 | 说明 | 典型例子 |
|------|------|---------|
| **常规 default** | 全年适用、大多数人都会走的一天 | 东山灵魂日、岚山深度日 |
| **人群替代** | 特定人群的专属版本，常规版不适合 | 铁道博物馆亲子日、东山情侣版 |
| **季节池** | 特定季节限定，平时不触发 | 醍醐寺樱花日、东福寺红叶日 |
| **日期锁定** | 固定日期的祭典/花火/特别公开 | 祇园祭山鉾巡行（7/17）、五山送り火（8/16）|
| **小众行程** | 清单顺位靠后，给第 3-5 次来的深度旅行者 | 西阵老街深度、北山杉贵船奥宫 |
| **文化深度** | 单一主题沉浸，非主线 addable | 手工艺体验日、伏见酒蔵巡礼 |
| **现代/当代体验** | 非传统景点的现代吸引力 | 任天堂博物馆、teamLab京都、漫画博物馆 |
| **两日版** | `duration_days: 2`，必须过夜才值得 | 岚山温泉两日、鞍马贵船料理旅馆 |

## 造模板的流程

### Step 0：体验类型盘点（规划新城市时做一次）

造模板的起点不是"有哪些景点"，而是"这个城市/区域有哪些值得花一天的体验类型"。

1. 列出该区域所有可能的体验类型（见下表）
2. 逐类检查：这个体验类型在当前城市有没有值得做的内容？
3. 有 → 判断是否需要独立模板（值得花一天的 → 独立模板；1-2h 的 → 嵌入现有模板的 slot）
4. 产出缺口清单，按优先级排序后逐个做

**关键思维转变**：不是"还有哪些景点没做模板"，而是"还有哪些体验维度缺失"。景点是视觉型体验的载体，但触觉（手工艺）、身体参与（温泉）、留白（恢复日）、章节切换（换城日）同样需要模板承载。

**体验类型参考清单**（不需要每类都做，判断标准是"这个体验类型是否值得专门花一天"）：

| 体验类型 | 说明 | 判断标准 |
|---|---|---|
| 核心景点日 | 经典必去，不去会遗憾 | S 级景点是否都有模板承载 |
| 人群专属日 | 特定人群的专属版本 | 主要人群（情侣/亲子/闺蜜）是否有专属体验 |
| 季节限定 | 特定季节才有的体验 | 樱花/红叶/夏/冬各有代表模板 |
| 节日/活动限定 | 固定日期的祭典/花火 | 该城市标志性节日是否覆盖 |
| 温泉住宿体验 | 一泊二食，住宿本身是体验 | 区域内值得住的温泉地/旅馆是否有模板 |
| 恢复/留白日 | 让前后天更精彩的"负空间" | 每个主要城市是否有恢复日模板 |
| 换城衔接日 | 章节切换，不是"多塞一个景点" | 常见换城路线是否有专门设计 |
| 文化深度体验 | 身体参与型（手工艺/料理/坐禅） | 触觉/身体参与的体验是否有独立模板 |
| 夜间限定 | 夜间特别公开/点灯/夜景 | 不是嵌在白天模板里的晚间 slot，是独立的夜间体验 |
| 户外/自然深度 | 需要半天以上的自然体验 | 该区域的自然体验是否覆盖 |
| 现代/当代体验 | 博物馆、沉浸式艺术、现代建筑 | 非传统景点的现代吸引力是否覆盖 |

**搜索方法**：盘点体验类型时需要大量搜索学习。不只搜景点数据，更要搜"旅行设计怎么做"、"去过5次的人怎么安排"、"顶级定制社怎么设计"。遇到有深度的文章要认真学习消化，用 OpenCLI 扩大搜索范围（日文/英文独立站、旅行杂志、定制社案例）。好的模板来自对目的地的深度理解，不是对景点列表的机械排列。

### Step 1：选 core_entity（从景点池挑 score ≥ 4.0 的景点）

1. 从景点池列出当前城市所有 `score ≥ 4.0`（S 级）景点
2. 判断每个 S 级景点适合哪种模板类型（见上表）：
   - 全年顺路型 → 写进常规 day 模板
   - 半天决策型（3h+，会挤掉其他内容）→ 单独建一个 day 模板
   - 季节升级型 → 走季节模板
   - 日期锁定型 → 走日期锁定模板
3. 搜索每个 S 级景点的**真实游览时间**——必须搜，不猜
4. 确定 `weather_sensitive`：全天主景点中有户外且怕下雨的 → true，雨天需全局 Plan B

**hardcode 禁止**：entity_id 必须是景点池真实存在的条目，不能凭 AI 知识编

### Step 2：大量搜索 + 组装骨架

1. 按**地理分区**组织一天：同一区域的景点排在同一天，不来回跑
2. 从**四个维度交叉搜索**，组装候选（见"搜索方法论"一节）
3. 考虑**餐厅位置约束**：午餐餐厅应在上午活动区域附近，不要专门绕路去吃
4. 确定到达日和离开日的固定安排

**硬规则**：
- 到达日不排正式景点（用户需要休息）
- 最后半天安排轻松（购物 + 出发）
- 操作性经验要写进 flow detail（如：最后一晚住泉佐野赶飞机、USJ 火木最空）

### Step 3：写 slots（时间槽）

每个 slot 需要的字段见 [SCHEMA.md §1.3](../SCHEMA.md#13-slot-时间槽硬字段)。

**slot 类型**：
- 景点 → `type: "poi"`，用 `entity_hint` 写景点名（日文），`area` 对齐城市区域枚举
- 餐 → `type: "meal"` 或 `"snack"`，`area` 写在哪个区域吃，`note` 描述对餐厅的要求
- 交通 → `type: "transport"`，`note` 写具体路线和用时
- 休息 → `type: "rest"`
- 散步 → `type: "walk"`
- 可选 → `type: "optional_poi"`，`options` 列出备选

**slot 排序原则**：
1. **核心景点（P1）优先拿最佳观赏时段**：东福寺红叶季8:30开门到、瑠璃光院上午东光、岚山竹林7:00前无人——核心景点的最佳时段是排序第一优先级
2. **次要景点配合路线流畅性**：P2/P3 景点在路线顺畅的前提下，能给好时段就给，不行也没关系
3. **不为时段牺牲路线**：不能为了"永观堂傍晚最美"让用户中午从西边跑到东边再跑回去
4. **地理连续性是底线**：同一区域的景点排在一起，不来回跳

**季节模板的 60% 规则**：樱花/红叶观赏体验占一天的比例不超过 60%，除非该景点本身有丰富的非季节体验维度（如东福寺有枯山水+禅文化，高山寺有鸟兽戏画）。防止一天变成"看完这片花去看那片花"的单调循环。

### Step 3.5：优先级、节奏与夜间活动

#### 一、核心原则

模板按**紧凑节奏**设计（slot 最多、时间最早）。Opus 装配时根据用户的节奏偏好做**减法**。减比加容易。

#### 二、优先级框架（只有 P1/P2/P3）

每个节奏档 = 上一档的"必去" + 本档的"可选"：

| 优先级 | 角色 | 悠闲 | 标准 | 紧凑 |
|--------|------|------|------|------|
| P1 | 灵魂景点 | 必去 | 必去 | 必去 |
| P2 | 顺路推荐 | **可选** | 必去 | 必去 |
| P3 | 顺路加项 | 跳过 | **可选** | 必去 |

**P1 分配标准**：不去就不是这一天的灵魂景点。

**P2 分配标准**（悠闲用户的可选项）：
- 在主路线上，不需要绕路
- 时间弹性大（15min-1h 都可以）
- 和 P1 体验类型不重复

**P3 分配标准**（标准用户的可选项）：
- 在主路线顺路范围内
- 固定耗时 30-60min
- 让紧凑节奏的一天饱满

#### 三、时间预算

| 优先级 | 总活动时间 | 总路程时间 |
|--------|-----------|-----------|
| P1 | ~4h | ~45min |
| P2 | ~3h | ~30min |
| P3 | ~1.5h | ~15min |
| 两餐 | ~2h | — |
| **全天合计** | **~10.5h** | **~1.5h** |

以上是总量参考，不限定单个 slot 的时长——如果有 10 个 P1 每个 20 分钟加起来 ~4h，那也可以。

#### 四、三种节奏

| | 走什么 | 出发时间 | 总时长 | 大概结束 |
|---|-------|---------|--------|---------|
| **紧凑** | P1+P2+P3+餐 | 08:00-08:30 | ~12h | ~20:00 |
| **标准** | P1+P2+餐，P3 可选 | 09:00-09:30 | ~10.5h | ~19:30 |
| **悠闲** | P1+餐，P2 可选 | 10:00-10:30 | ~9h | ~18:30 |

模板的 `time_range` 按紧凑节奏写。Opus 装配时根据节奏档把整条时间线往后推。

#### 五、人群标签

每个 slot 可带 `fit_audience` 标签（默认 `"all"`），Opus 根据用户人群过滤：

- `"all"` — 所有人群都推荐
- `"no-kids"` — 不适合带小孩（酒吧、深夜活动等）
- `"no-elderly"` — 不适合老人（爬山、长距离步行等）
- `"couple"` / `"friends"` — 特定人群才推荐

Slot 级别的人群标签用于微调，当核心体验根本不同时应做独立模板（见"主角判断"）。

#### 六、夜间活动

夜间活动用**统一框架**处理——每个夜间 slot 带人群标签 + 时长，Opus 根据这两个属性自动决定怎么调白天：

| 时长 | Opus 怎么调白天 | 是否单独模板 |
|------|----------------|------------|
| **≤1.5h** | 不调白天，晚餐后去 | 否，写在基础模板的晚间 slot 或晚餐 note 里 |
| **1.5-2.5h** | 从后往前砍 1 个 P3 或晚段 P2，晚餐提前 | 否，基础模板里加一个夜间 slot |
| **≥3h（夜间是主角）** | 白天大幅缩减，下午留休息 | **是，做独立模板**，白天 slot 降为 P2/P3，夜间标 P1 |

**主角判断**（适用于所有活动，不只是夜间）：

> 一个活动如果**重要且耗时 2h 以上**，它就是这一天的主角，应该为它单独设计模板、整天围绕它排。不是主角的活动（短/可选/锦上添花）在基础模板上叠加即可。

例：
- 东福寺红叶 2.5h + 改变路线起点 → 主角 → 独立模板（东山红叶日）
- 宵山 4h + 是全天核心 → 主角 → 独立模板
- 永观堂夜间红叶 1.5h → 不是主角 → 冈崎红叶日加一个夜间 slot
- 高台寺夜间点灯 1h → 不是主角 → 东山日晚餐 note 里提一句
- 歌舞伎町深度 3-4h + 特定人群 → 主角 → 独立模板（带人群标签 no-kids/no-elderly）

**人群夜间方向参考**（短时夜间，写在晚餐 note 里）：
- 情侣：日本酒吧/威士忌吧（安静+氛围）、夜景展望台
- 闺蜜/朋友：居酒屋街（热闹+发现感）、夜景展望台
- 亲子：夜间散步（轻松收尾）、不安排夜间活动
- 默认：先斗町/木屋町漫步（不需要消费也有氛围）

#### 七、季节提示（嵌入基础模板）

基础模板的 slot note 里可以写季节性信息，这**不是条件开关**，只是给 Opus 的参考素材：

- ✅ "红叶季（11月）：北野天满宫的御土居红叶苑限定公开（¥1,200含茶果子），值得加 30 分钟"
- ✅ "樱花季（3月底-4月初）：二条城夜间灯光 art 投影，另收 ¥2,000，需预约"
- ❌ `if season == "sakura"` 这种条件逻辑

当季节体验是**主角**（重要+耗时+改变路线）时，做独立的季节模板；当季节体验只是**加分项**时，在基础模板 note 里提一句。

---

**声明 `core_entities`**：模板必须声明 1-2 个"没去这个就不是这一天"的灵魂景点。顺路经过的不填。

**声明 `tags`**：用于 skip_tags 过滤，写这一天的体验类型标签。

### Step 4：补 slot note（专家素材）

每个 slot 的 note 是给 Opus 的素材，要有专家判断，不是"去这里参观"。每个景点 slot note 至少涵盖：

- **时间管理**：几点前到最好、什么时段人少/光线好
- **怎么玩最好**：从哪边进、看什么重点、停留多久
- **坑点**：游客陷阱、不要在哪里吃、哪里会被坑
- **交通衔接**：从上一个点怎么到这里

### Step 5：补灵魂细节（惊喜 moment）

10% 惊喜 moment：在两个大景点中间，加一个"游客不知道但本地人会去"的内容。可以是 nearby_options、includes_snack、或单独的轻量 slot。

### Step 6：体力预演

- 用 Google Maps 查相邻景点间步行时间
- 步行 >20 分钟的段落要加 transport slot
- 全天步行超过 12km 要在中途加休息 slot
- 体力不顺就砍 slot 或调顺序，**宁少不多**

**时间窗口要有弹性**：关键景点定锚点时间（如"清水寺 8 点前"），其他相对推算。午餐 12:00-13:30 是硬锚点。

### Step 7：检查清单

模板写完后自检：

- [ ] 硬字段齐全：template_id / tags / core_entities / fit_audience / weather_sensitive / assembly
- [ ] 每个 slot 有 note，且 note 有专家判断（不是路线描述）
- [ ] slot 时间段不重叠，交通时间算进去了
- [ ] 跨天检查：相邻两天的 core_entities 不重复
- [ ] 地理检查：午餐 slot 的 area 在上午活动区域附近
- [ ] 有没有连续两天暴走
- [ ] weather_sensitive: true 的模板有没有室内备选提示

## 从第一景点生长的思路

好的模板是"生长"出来的，不是"堆砌"出来的：

1. **先定一个最强的 entity**（score 最高的景点，或季节性最强的触发点）
2. **问：去了它之后，顺着走最自然的下一个是什么？** 不是"还有哪些 S 级还没去"，而是"走完清水寺，从石阶下来，顺着感觉会走去哪"
3. **地理连续性 > 评级最高化**：一天内不应该有大跨越（东边走完跑去西边，再跑回来），宁可放弃一个 A 级景点也要保持地理连贯
4. **收尾画面先定**：一天的最后一个画面是什么？花见小路黄昏、岚山竹林傍晚、道顿堀夜景——先想好结尾，再往前推骨架
5. **餐厅不是景点**：餐厅跟着景点走，不要为了一家餐厅绕路。午餐 slot 的 note 里写"附近找……级别的"，Opus 装配时会从内容池选

## 经典版 = 大多数人的走法，不是最极致的走法

"经典版"（常规 default 类型）的定位是：

> **第一次来的人，不需要特别功课，跟着走就不亏的一天**

这意味着：
- **不是最深度的走法**——深度走法留给小众行程类型
- **不是最极致的走法**——极致体验留给文化深度类型
- **是绝大多数人会满意的走法**——评分最高的景点 + 合理的节奏 + 不踩坑

写常规 default 时，如果你发现自己写了一个"需要有一定了解才能欣赏"或"只有第二次来才会这样走"的方案，那它应该是小众行程类型，不是 default。

经典版写完后问自己：**一个第一次来日本的人，按这个走，会不会留下遗憾？** 如果有明显遗漏的名场面，补进去；如果没有，就够了。

## 搜索方法论（四维度交叉）

造模板时需要搜索真实数据（S 级景点游览时间、路线验证、攻略交叉检查）。任何品类的搜索必须从**至少 4 个维度交叉**，不能只从一个方向搜：

| 维度 | 搜什么 | 能发现什么 |
|------|--------|-----------|
| 按区域 | 各主要区域逐个搜 | 常规选项 |
| 按排名/榜单 | top 榜、米其林、Conde Nast、一休排行 | 头部名牌 |
| 按体验类型 | 温泉/川床/宿坊/町家/设计等 | 特色体验型 |
| 按地理扫描 | 把城市分成东/西/南/北/中心/近郊逐一搜 | 不在主流区域但值得去的 |

搜完后做**交叉检查**：别的来源反复提到但我的清单里没有的，一定是漏了。

**常见漏洞**：
- 只搜了"最方便的区域"，漏了"值得专程去住的近郊"
- 只搜了"常规酒店"，漏了"体验型住宿（寺庙/山里/河边）"
- 只看了中文来源，漏了日文/英文独立站的推荐

## 实战补充规则（2026-04-13 基于大阪京都模板实战补充）

> 以下规则来自实际编写 27 个模板过程中发现的问题和最佳实践。

### 16. 价格必须标注币种和验证年份

门票价格写日元（¥）不写人民币——用户到现场看到的是日元价格。如果价格是动态的（如海游馆）标注范围而不是固定值。2024 年后日本多个景点涨价（大阪城 ¥600→¥1,200），不要用旧价格。

### 17. 季节模板必须通过 core_entities 实现与常规版互斥

不要依赖 condition 条件判断——硬筛引擎看 core_entities 去重。`maruyama_yozakura_day` 和 `higashiyama_day` 共享 `kyo_kiyomizudera` + `kyo_yasaka_shrine`，装配时自动互斥。如果季节版和常规版没有共享 core_entities，需要手动确保不会同时被选中。

### 18. 人群变体拆独立模板时 core_entities 重叠是 OK 的

`audience_couple_day` 和 `audience_family_day` 都含 `osa_kaiyukan`，但因为 `fit_audience` 不同不会同时进入候选池。这个重叠防止的是 Opus 在同一 trip 里给情侣用户既选了 couple_day 又选了 default_day（两天都去海游馆）。

### 19. 餐厅 slot 的 note 要写"不要在哪里吃"

"不要在清水寺参道吃正餐——贵30%且全是游客店"这类负面指引和正面推荐同样重要。每个 meal slot 的 note 至少包含一个"避开"方向。

### 20. 祭典/花火模板要写散场策略

130 万人同时散场的天神祭、50 万人/晚的宵山——不写散场路线用户会被困 1h+。每个大型活动模板的最后一个 slot note 必须包含"回程避开哪个站/走哪条路"。

### 21. slot 的 area 必须在城市 _areas 枚举中

新增景点如果不在现有 area 里，先在 `days/_meta.json` 的 `_areas` 字典里添加定义。不要用未定义的 area，否则硬筛代码无法按区域匹配餐厅池。

## 硬红线（绝对不能破）

1. **不编数据**：entity_id、游览时间、营业时间、评分——必须来自景点池或真实搜索验证。搜不到宁可 null，不编
2. **不把内容池数据塞进模板**：模板只写引用（entity_id）和自然语言描述（flow detail）。具体餐厅/酒店/店铺的描述、评分、点评全部在内容池里，见 [CONTENT_POOL_WRITING_GUIDE.md](CONTENT_POOL_WRITING_GUIDE.md)
3. **不在模板里写跨天联动逻辑**：跨天菜系去重、跨天体力平衡这些是 Opus 装配时做的事，模板只写自己这一天
4. **不在常规模板里加季节条件开关**：不写 `if season == "sakura"` 这种条件逻辑。当季节体验是主角（重要+耗时+改变路线）时做独立季节模板 + `core_entities` 互斥（见 SEASONAL_SYSTEM_DESIGN.md）。季节加分项可以写在 slot note 里作为 Opus 参考素材（见 Step 3.5 §七）

## 不在本流程范围（已挪走）

以下内容在 v1 指引里属于"造模板"，但 v2 架构下挪到了别处，**造模板时不处理**：

| 内容 | 现在去哪 |
|------|----------|
| 具体餐厅是哪家 / A/B 选哪个 / 菜系去重 | Opus 装配时做，内容池条目的字段（ab_role / meal_role / cuisine_tag）提供过滤依据 |
| 餐厅 / 酒店条目编写规则 | [CONTENT_POOL_WRITING_GUIDE.md](CONTENT_POOL_WRITING_GUIDE.md) |
| 店铺推荐矩阵 / 人群兴趣替换 | [CONTENT_POOL_WRITING_GUIDE.md](CONTENT_POOL_WRITING_GUIDE.md) — 店铺条目编写 |
| 多维评级 | 内容池条目的字段 |
| 专业点评 | 内容池条目的字段 |
| 拍照点 / 咖啡厅 | 内容池（当 entity 处理），模板按需引用 entity_id |

模板只负责骨架 + 引用 + 专家视角文案。餐厅/酒店的具体选定是 Opus 的事，模板告诉 Opus"这一餐在哪个区域、什么类型、什么感觉"就够了。
