## 日模板内容完整性清单

> 硬字段定义见 [SCHEMA.md §1](../SCHEMA.md#1-日模板候选池day_templates)。本节管的是自然语言内容——模板里应该包含哪些内容板块，用于检查完整性。

日模板有两类读者：**Opus 装配引擎**（选模板、排序、配餐配酒店）和**最终用户**（通过手账本看到的内容）。Opus 读硬字段 + 编辑 note 做决策，最终用户看到的故事/Plan B/交通攻略由 Opus 根据模板素材现场生成。

### 每个日模板必须包含的内容

**1. description（一段话概述）**
这一天在做什么、为什么这样安排、整体节奏是什么。Opus 读这段判断模板是否适合当前用户。

**2. 每个 slot 的 note（编辑专家备注）**
slot 里的 note 是编辑写给 Opus 的素材，不是直接给用户看的。应该包含：
- 为什么安排在这个时段（比如"8:30前竹林几乎没人"）
- 怎么玩最好（比如"不建议爬完整圈，前段已经极美"）
- 交通衔接（从上一个点怎么到这里）
- 本地人才知道的信息（比如"花见小路16:30后不要拍行人"）

**3. hotel_area_note（住宿区域对交通的影响）**
不同住宿区域会影响当天的出发方式和用时，写清楚。

**4. 季节变化提示**
如果某个 slot 在特定季节有明显不同（樱花/红叶/雪景/夏日祭），在 slot 的 note 或 seasonal_note 里写明。不写具体日期，日期由 L0 事件层管。

### 以下内容由 Opus 最终生成，模板里不需要预写

- 完整的故事叙述（story）
- Plan B 方案（雨天/人多/临时关闭的替代）
- 餐厅具体推荐（由装配第二步配餐）
- 酒店具体推荐（由装配根据酒店池选）
- 个性化的专家点评语气

### 内容质量标准

- 给线索，不给结论（见 [brief.md §3.9](../../content/kansai/brief.md)）
- note 里写的每个事实必须有真实来源，不许 AI 推断当事实
- 时间安排要合理：交通时间算进去，不要无缝衔接
- 到达时段：关西到达一般是中午或傍晚，不需要处理早上到达的情况

## 字段两分法原则（设计哲学）

> 完整字段定义见 [../SCHEMA.md](../SCHEMA.md)。本节只讲设计意图。

所有内容池条目的字段分两类：

**硬字段**（客观事实 + 表单对应字段）：程序能精确匹配的，用于过滤、排序、展示分类。

**自然语言字段**（editor_note + review）：专家叙述、场景描述、理由说明。其他一律用自然语言，不要为了"方便程序处理"把自然语言硬拆成枚举。

**硬字段保留 5 条标准**（D29）：满足任意一条才保留为硬字段——

1. 表单按它过滤
2. 用户看得到的展示分类
3. 客观事实
4. 硬约束机械判定需要
5. 系统运维需要

不满足以上 5 条的 → 写进 editor_note 或 review 里。

### score 评分规则（0-5 一档一分）

| 档位 | 分值 | 含义 |
|------|------|------|
| S 级 | 4.0-5.0 | 城市代表性景点/餐厅，不去会遗憾 |
| A 级 | 3.0-3.9 | 重要搭配，值得专程，但次于 S 级 |
| B 级 | 2.0-2.9 | 顺路可以，不必专程 |
| C 级 | 1.0-1.9 | 补充选项，只在时间富余时考虑 |

score 是编辑综合评分，不是单一来源的评分。综合多个 P0-P2 数据源判断后给出，标记来源（如 `"tabelog:3.72,michelin:bib"` 可以作为参考，但 score 字段本身是 0-5 浮点数）。

## 小众内容入池判断流程

非S级景点（score < 4.0）进入内容池前必须通过以下四步判断，不通过则不入池：

**第一步：内容质量**
这个地方有没有不可替代的体验？能不能用一句话说清楚"为什么值得去"？说不清楚的不入池。

**第二步：顺路性和时间成本**
- 顺路且时间短（<1.5小时）→ 候选加入常规行程的次要slot
- 不顺路或时间长 → 候选作为"小众专题日"的组成部分，与其他小众体验打包

**第三步：与同区域其他次要景点比较**
加入常规行程前，和同一天同一区域的其他B/C级候选比较——谁的体验更独特、更难在别处替代，谁才进slot。不是"这个也不错"就入，是"这个比那个更值"才入。

**第四步：适合哪类人群**
明确标注适合的party_type或vibe，不是所有用户都推。例如：
- 酒造参观 → 适合default/couple，不适合family_with_children
- 古着街区漫游 → 适合photogenic/younger用户
- 无明确人群的小众内容 → 默认不进常规行程，只作为relaxed节奏的备选

四步都通过才写入内容池，并在editor_note里注明"适合顺路加入XX主线"或"适合小众专题日"。

## 硬红线（绝对不能破）

### 1. 数据真实性
所有事实性字段（评分、价格、坐标、营业时间）**必须来自真实数据源验证**。搜不到填 null，绝不编造。

- ❌ 给餐厅填 `rating: 4.3` 而没真去 Tabelog 搜过
- ❌ 给酒店填坐标而没真去 Google Maps 查过
- ❌ 写"这家店每天排队 30 分钟"而没真查过评论
- ✅ 搜不到评分 → 填 null + 标记 `data_confidence: "unverified"`

**自检规则**：每次要写一条"事实"之前，停一秒问自己"我这条是真搜过的，还是从上下文推断出来的？"。如果是后者，要么去真搜，要么明确标注待核实，**绝不把推断写成事实**。

### 2. 三源核对
数据可靠性的上限是**三个权威源核对一致**。三源核对通过的条目可以直接用，不再追求更高。

数据源优先级：
```
P0（权威源）：Tabelog / Michelin / 一休 / 楽天 / Google Maps API
P1（中国用户源）：小红书 / 携程 / 大众点评 / 马蜂窝
P2（参考源）：japan-guide / JNTO / 旅行博客
P3（兜底）：AI 知识库 — 仅用于结构化分类、推荐语，不用于事实性数据
```

事实性字段必须来自 P0-P2 验证。P3 数据标记 `data_confidence: "ai_generated"`，**不可直接上生产**。

### 3. 风险标签优先于推荐
推荐一家差餐厅的伤害 >> 错过一家好餐厅的损失。三层过滤：

- **硬黑名单**：永不推荐（食物安全问题/严重负面评价/已停业）
- **风险标签**：默认过滤（服务差/价格陷阱/观光化严重），用户可放开
- **正常池**：推荐候选

每条条目加 `risk_flags` 字段，有任何风险点都要标，宁可少推荐。

## 景点条目

景点是最基础的实体，被模板 flow 的 `entity_id` 引用，也被其他内容池间接引用。

### 字段定义

景点共 14 个硬字段 + 自然语言字段（editor_note / review / seasonal_notes）。

**字段定义见 [../SCHEMA.md §2.1](../SCHEMA.md#21-景点-attractions14-字段)**。本节只讲怎么填好这些字段。

**review 三段结构**：
```json
{
  "亮点": "为什么值得去、核心体验是什么",
  "怎么玩": "具体玩法、进入方式、关键时间点、拍照机位",
  "缺点": "人多 / 票价贵 / 季节限制 / 体验不稳定等"
}
```

### editor_note 写作标准

editor_note 绑定在实体上，**写永远成立的话，不写某一天的特定安排**：

- ✅ "清水寺必须 8 点前进，8 点以后游客激增" → 永远成立，写在 editor_note 里
- ❌ "今天先去清水寺再去八坂神社" → 这是模板的事，不写在实体里
- ✅ "子安塔背后可拍到舞台全景，游客通常不走这条路" → 景点自身属性，永远成立

### 季节升级处理

如果一个景点平时是 A 级（score 3.x），在樱花期变成"必去"：
- 景点本身 score 保留平时的 A 级值
- 在 `seasonal_notes` 对象里加一个 key（如 `"樱花期"`），value 写自然语言说明
- 实体只描述"这个季节有什么看点"，**不写具体日期**——日期由 L0 事实层（`data/events/`）维护，装配引擎自己取用户行程日期 ∩ L0 当年窗口判断是否触发
- 需要独立一天才值得的季节场景 → 走独立的季节模板（见 [SEASONAL_SYSTEM_DESIGN.md 第 2 节](SEASONAL_SYSTEM_DESIGN.md#2-机制一独立完整模板)）

```json
{
  "id": "kyo_kiyomizudera",
  "seasonal_notes": {
    "樱花期": "早开门 + 子安塔背后可拍舞台全景樱花，建议 7:30 前到",
    "红叶期": "夜间特别拝観，舞台从下方仰拍可见整片红叶包裹的清水寺"
  }
}
```

## 餐厅条目

餐厅是内容池里条目最多、规则最复杂的一类。Opus 装配时根据模板 flow detail 里的描述（位置/类型/感觉要求）+ 内容池结构化字段（area / cuisine_tag / budget_tier / ab_role / meal_role）来选定具体餐厅。

### 字段定义

餐厅共 17 个硬字段 + 自然语言字段（editor_note / review）。

**字段定义见 [../SCHEMA.md §2.2](../SCHEMA.md#22-餐厅-restaurants17-字段)**。`meal_role` 取值见 [SCHEMA §2.2.1](../SCHEMA.md#221-meal_role-取值)。本节只讲怎么填好这些字段。

review 三段结构：亮点 / 必点必知 / 缺点。

### 预算档分布

核心原则：**不是每顿都吃同档，而是普通餐压低、高光餐拉高**。

**笼统版(写内容池时按这个思路判断每家餐厅的定位)**:

| 用户档位 | 8 顿正餐的分布 |
|----------|----------------|
| 经济 | 5-6 顿经济 + 2 顿中档 + 0-1 顿中高档 |
| 中档 | 2 顿经济/低中档 + 4 顿中档 + 1-2 顿中高档 + 0-1 顿高档 |
| 高档 | 1 顿经济/低中档 + 2-3 顿中档 + 3-4 顿高档 + 0-1 顿顶级 |

日本大量"低价但非常值得"的餐位天然适合降档——到达恢复餐、园区恢复餐、本地生活体验餐。

**详细版(BudgetCalculator 按天数切分的混搭比例,迁移自 form_design.md)**:

用户档位命名对齐前端:街头地道 / 本地好店 / 好好享受 / 顶级体验。

| 用户选的档位 | 5-6天(10-12顿) | 7-8天(14-16顿) | 9-10天(18-20顿) | 11-13天(22-26顿) |
|------------|-----------------|-----------------|------------------|-------------------|
| 街头地道 | 8顿本档+2顿升中档 | 10顿本档+2顿升中档+2顿平价 | 13顿本档+3顿升中档+2顿平价 | 16顿本档+4顿升中档+2顿平价 |
| 本地好店 | 7顿本档+1顿升享受+2顿平价 | 10顿本档+2顿升享受+2顿平价 | 13顿本档+2顿升享受+3顿平价 | 17顿本档+3顿升享受+3顿平价 |
| 好好享受 | 7顿本档+1顿升顶级+2顿中档 | 10顿本档+1顿升顶级+3顿中档 | 13顿本档+2顿升顶级+3顿中档 | 17顿本档+2顿升顶级+4顿中档 |
| 顶级体验 | 7顿本档+1顿showcase+2顿享受 | 9顿本档+1顿showcase+2顿享受+2顿中档 | 12顿本档+2顿showcase+3顿享受+3顿中档 | 15顿本档+2顿showcase+4顿享受+4顿中档 |

> showcase = 全程最高光餐(每城最多1顿,全程最多2顿)。

**住宿升级比例(迁移自 form_design.md)**:

| 用户选的档位 | 5-6晚 | 7-8晚 | 9-10晚 | 11-13晚 |
|------------|-------|-------|--------|---------|
| 干净方便 | 全部本档 | 全部本档 | 全部本档 | 全部本档 |
| 品质舒适 | 全部本档 | 1晚升享受型 | 1晚升享受型 | 1-2晚升享受型 |
| 享受型 | 1晚升顶级或特色旅馆 | 1晚升顶级或特色旅馆 | 1-2晚升特色旅馆 | 2晚升特色旅馆 |
| 顶级 | 1晚最顶级体验 | 1晚最顶级体验 | 1-2晚最顶级体验 | 2晚最顶级体验 |

> 干净方便档不做升级(用户选便宜就尊重选择)。
> 品质舒适以上自动安排特色住宿,升级费用已算入总价。

> **注(2026-04-12 迁移)**:笼统版的 3 档命名(经济/中档/高档)和详细版的 4 档命名(街头地道/本地好店/好好享受/顶级体验)需要统一一次。前者是写内容池时的思考框架,后者是前端真实档位。两者都要保留但应显式映射(街头地道=经济、本地好店=中档、好好享受=高档、顶级体验=新增的最高档),待 BudgetCalculator 实现时一次对齐。

### 日类型自动升降

| 日类型 | 调整 |
|--------|------|
| 到达日午餐 | -1（刚到不折腾） |
| 主题公园/暴走日晚餐 | -1（累了吃什么不重要） |
| 人群日午餐 | +1（开始升温） |
| 人群日晚餐 | +1~+2（全程最高点） |

### 人群修饰

| 人群 | 修饰 | 体验重点 |
|------|------|----------|
| 情侣 | 人群日晚餐 +1，可触发顶级 | 氛围 + 好看 |
| 闺蜜 | 人群日晚餐 +1，可触发顶级 | 颜值 + 拍照 |
| 亲子 | 整体 -0.5~-1 档，上限中高档 | 互动 + 友好 |
| 默认 | 人群日 +1 | 稳定 |

### 全程约束

- 经济档：最多 1 顿中高档
- 中档：最多 1 顿高档，至少 1 顿经济体验
- 高档：最多 1 顿顶级，至少 2 顿中档或以下
- 亲子：不触发顶级
- **showcase meal 上限**：每城最多 1 顿，全程最多 2 顿。防止城城都顶到最满拉坏预算

### A/B 结构

| | A（安全牌） | B（惊喜牌） |
|---|------------|-------------|
| 定位 | 系统默认推荐 | 用户可能不知道但值得试 |
| 知名度 | 高，大众接受 | 低，小众特色 |
| 体验角色 | 主流体验 | 与 A 体验角色不同 |
| 呈现 | 主推 | "想试试不一样的？" |

**核心理念**：
- 推荐 ≠ 让用户选。A 是主推，B 是加分项
- A/B 是**隐性个性化**——不可能把用户所有偏好都问一遍，A/B 给两个不同方向
- A 和 B 必须**体验角色不同，优先不同菜系**——氛围、形式、价格带、在地感差异够大，才算合格 B

判断样本（B 是否合格）：**用户看完想发给朋友说"你看这个"**。

### 升档归属原则

**升档花的钱要花在"这个城市才能吃到"的东西上**。不能在 A 城市升档吃 B 城市的代表菜系。

- ✅ 神户吃和牛升档
- ✅ 京都吃怀石升档
- ❌ 大阪吃和牛升档（和牛归神户）
- ❌ 神户吃怀石升档（怀石归京都）

### 口味搭配规则

- 午餐和晚餐菜系不重复
- 一天内不吃两顿同类主食（粉物/拉面/乌冬/大阪烧不连吃）
- 连续两天同一菜系不出现两次
- 口味要多样：不能天天本地传统菜，要穿插西餐/其他风格
- 不推荐用户本国菜系（中国用户不推中华料理）
- 小吃 slot 不参与菜系去重

### 跨城市菜系去重

**只对 A（主推）轨道去重，不对全部候选去重**。B 是备选，用户可能不选，不应该因为 B 的存在就误判为"已消费"。

去重逻辑：
1. 先拼装全程所有城市的 A 轨道
2. 扫描 A 轨道有没有同菜系重复
3. 重复了 → 优先城市保留，其他城市触发 fallback
4. 再检查 B 是否与同天 A 完全重合（同菜系 + 同体验角色），重合则换 B

菜系冲突处理：
- 城市独占菜系不参与去重（如只有 A 城市有的名物）
- 有优先城市的菜系（和牛归神户），其他城市同菜系候选仍然可以进池，但 Opus 装配时会参考模板 story 里的升档归属原则
- 通用菜系（拉面/天妇罗/居酒屋/烧鸟等）不硬排他，全程上限 1-2 次，**第二次必须承担不同 meal_role**

### 预约难度过滤

`reservation_difficulty` 字段用以下枚举：

| 值 | 说明 |
|----|------|
| `walk_in` | 随时去，无需预约 |
| `1week` | 提前 1 周内可预约 |
| `1month` | 提前 1 个月 |
| `2month+` | 提前 2 个月以上，或需介绍/会员制 |

`2month+` 的餐厅原则上不进推荐池。

**review 写作规则**：
- 必须来自真实来源（Tabelog / Google / 一休 / 小红书 / 大众点评）
- 负面信息圆滑表达（让用户有心理准备但不受伤）
- 中文为主，名称中日双语

## 酒店条目

Opus 装配时根据模板 `hotel_area` 字段（自然语言描述）+ 酒店池结构化字段（area / budget_tier / vibe_tags / facility_tags）来选定具体酒店。

### 字段定义

酒店共 14 个硬字段 + 自然语言字段（editor_note / review）。

**字段定义见 [../SCHEMA.md §2.3](../SCHEMA.md#23-酒店-hotels14-字段)**。本节只讲怎么填好这些字段。

review 三段结构：亮点 / 位置房间 / 缺点。

> **注**：旧的 `quality_bias` / `experience_bias` 已砍掉（2026-04-12）。理由：不是表单过滤维度、不是客观事实，可完全用 editor_note 表达。原本想表达的"性价比偏向 / 体验型酒店"等语义,统一写进 editor_note。

### 搜索方法论（四维度交叉）

酒店搜索必须从至少 4 个维度交叉：

| 维度 | 搜什么 | 能发现什么 |
|------|--------|-----------|
| 按区域 | 各主要区域逐个搜 | 常规选项 |
| 按排名/榜单 | Booking top / 一休排行 / Conde Nast | 头部名牌 |
| 按体验类型 | 温泉/町家/设计/老牌/度假 | 特色体验型 |
| 按地理扫描 | 把城市分东西南北中心近郊逐一搜 | 不在主流区域但值得去的 |

**常见漏洞**：
- 只搜了"最方便的区域"，漏了"值得专程去住的近郊温泉"
- 只搜了"常规酒店"，漏了"体验型住宿（寺庙/山里/河边）"
- 只看了中文来源，漏了日文独立站的老牌

### 推荐方式

- Opus 直接推最匹配的一家，不给用户 3 家让他选
- 手账本上：推荐 + 一句话理由 + 备选 1 家（小字）
- 备选要与推荐差异化
- 不推荐极难预约的酒店

## 店铺条目

店铺的定位不是时间线上的主要景点，而是**"附近逛一逛"信息附录**。店铺不做多段 review，只用 `editor_note` 一段话，因为用户看店铺信息的时间很短，一段话够了。

### 字段定义

店铺共 10 个硬字段 + 自然语言字段（editor_note）。店铺不做多段 review,因为用户看店铺信息的时间很短,一段话够了。

**字段定义见 [../SCHEMA.md §2.4](../SCHEMA.md#24-店铺-shops10-字段)**。本节只讲怎么填好这些字段。

editor_note 写作要求：一段话 2-4 句,说清楚：这家卖什么、为什么值得专程来、什么人会喜欢。

### 推荐数量规则

| 当天自由时间 | 推荐数 |
|--------------|--------|
| 半天以上 | 5 家 |
| 1-2h | 3 家 |
| <1h | 1-2 家 |
| 无 | 0 |

### 替换矩阵

- 默认 5 家差异化（不同类型）
- 人群替换（情侣/闺蜜/亲子各替换部分）
- 兴趣替换（二次元/购物/古着等）
- **替换优先级**：兴趣 > 人群 > 默认

### 跨天联动

- 跨天不推荐同区域/同类型的店铺
- 每家打评级，时间不够从低评级砍

## 通用原则：score + fit_audience

所有实体的候选排序依靠 `score`（编辑综合评分）+ `fit_audience`（人群过滤）。Opus 装配时读这两个字段做初筛，再结合结构化字段和 review 做最终判断。不做多维度×人群的矩阵打分（那是给规则引擎的，Opus 不需要）。

## 数据真实性自检清单

每次添加或修改条目后自检：

- [ ] 所有事实性字段（评分、价格、坐标、营业时间）都有真实来源
- [ ] 来源符合 P0-P2 要求，没有用 P3 AI 知识填事实
- [ ] 搜不到的字段填 null，没有编造
- [ ] `last_verified` 日期填了
- [ ] `risk_flags` 检查过（价格陷阱/服务差/观光化/食安问题）
- [ ] review 来自真实评论，不是 AI 编的
- [ ] 中日双语名称拼写正确（日文汉字不要写成简体中文）
