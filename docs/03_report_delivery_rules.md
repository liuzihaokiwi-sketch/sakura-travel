## 17. 攻略正文结构（最终统一版）

### 第 1 层：总纲
必须有：
1. 为什么这份攻略这样设计
2. 一眼看懂总览
3. 关键预订提醒
4. 出发前准备 / 总注意事项

### 第 2 层：每天固定骨架
每天固定 4 页：
1. 今日执行总览
2. 为什么今天这样排
3. 今日亮点展开
4. 今日注意事项 / Plan B

### 第 3 层：条件页
按需触发：
- 酒店页
- 餐厅页
- 交通页
- 出片页
- 顺路补充页
- 预算感知页

### 原则
不要再用“每天很多页”来堆厚度。  
要用：
- 总纲前置
- 固定骨架
- 条件页触发
- 静态块复用
- AI 少量解释  
来堆“值感”。

## 19. PDF / H5 交付统一规则

### PDF
- 正式交付主形态
- 方便保存 / 离线看 / 发给同行人
- 带轻度个性化水印
- 仅供本人非商业使用

### H5 / 在线版
- 预览
- 自助微调
- 交互查看
- CTA / 微信承接
- 埋点

### 水印要求
- 用户昵称 / 订单标识
- “仅供本人使用”
- 不影响阅读

## 20. 自助微调与正式修改统一规则

### 自助微调
优先开放：
- 酒店替换
- 餐厅替换
- 局部体验替换
- 节奏轻重调整
- 夜间方案微调

### 自助微调后必须自动重检
- 顺路性
- 时间可执行性
- 预算偏移
- 相邻安排冲突
- 是否破坏当天主线

### 正式修改次数
只有用户完成系统允许的自助微调后仍不满意，  
才进入正式修改次数。

## 21. 状态机（已统一实现）

> ⚠️ 已实现版本见 `app/api/orders.py` VALID_TRANSITIONS，以代码为准。

统一 11 状态（附加 cancelled / refunded）：

```
new → sample_viewed → paid → detail_filling → detail_submitted
→ validating → needs_fix → validated → generating → done → delivered
```

- new — 用户提交 quiz
- sample_viewed — 查看了样片页
- paid — 已付款
- detail_filling — 正在填写详细表单
- detail_submitted — 详细表单已提交
- validating — 红黄绿校验中
- needs_fix — 校验不通过，需补填
- validated — 校验通过
- generating — 正在生成完整攻略
- done — 生成完成，待审核/交付
- delivered — 已交付用户
- cancelled — 已取消（任意状态可转）
- refunded — 已退款

## 22. 反馈回库与片段沉淀

### 回访节奏
- T+2~3：第一轮反馈
- T+6~7：第二轮补采
- T+7~14：高满意用户触发转介绍 / 复购邀请

### 回访要做什么
- 验证实体体验
- 验证节奏
- 验证餐厅和酒店推荐
- 记录满意点 / 后悔点
- 更新片段质量分
- 判断是否值得沉淀成新片段

### 每次订单完成后都要判断
- 有没有新的路线片段可回库
- 有没有新的决策片段可回库
- 有没有值得标准化的 Plan B
- 有没有新的高命中组合值得保留

- 交付必须分 PDF 与 H5

- 付费后才发详细表单
- 后台优先展示红黄问题摘要
- 一键复制追问话术
- 通过状态清晰可见

### 23.4 数据层
必须补：
- 片段主表
- 片段实体关系表
- 片段 embedding
- 片段 compatibility
- 片段 usage stats

- profile_version / template_version / score_version / render_version 记录
