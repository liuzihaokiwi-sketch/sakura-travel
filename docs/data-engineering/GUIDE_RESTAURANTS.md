# 餐厅采集指南

> 版本: 3.0
> 更新: 2026-04-02
> 上位文档: MASTER_GUIDE.md
> 适用范围: 日本圈（关西/关东/北海道等）

---

## 一、采集目标

为用户提供最优质的餐厅选择。不做餐厅分类系统，只回答一个问题：**"来这里吃什么"**。

### 核心原则

- **Tabelog 为唯一质量主源**，按菜系×价格带内排名取 top，不设固定分数门槛
- **圈级排名**，标清城市和区域（区域直接用 Tabelog 的エリア标签）
- **餐厅采集和景点解耦** — 不按景区配餐厅，纯按品质采集，位置只是标签
- **"食物本身是体验"的（锦市场、道顿堀）归入景点采集，不算餐厅**

## 五、采集流程

### Phase 0：菜系地图（每个圈 1 次）

调研该圈的菜系全景，产出两个配置文件：

**cuisine_map.json** — 菜系组与 Tabelog 子分类的映射：
```json
{
  "kaiseki": {
    "label_zh": "怀石/割烹",
    "tabelog_categories": ["懐石・会席料理", "割烹・小料理", "京料理"],
    "price_tiers": ["~5000", "5000~15000", "15000~"]
  }
}
```

**specialty.json** — 每个城市的特色菜清单：
```json
{
  "kyoto": {
    "must_try": ["京料理", "抹茶甜品", "湯豆腐"],
    "worth_trying": ["おばんざい", "京都ラーメン"]
  },
  "osaka": {
    "must_try": ["たこ焼き", "串カツ"],
    "worth_trying": ["お好み焼き", "箱寿司"]
  },
  "kobe": {
    "must_try": ["神戸牛"],
    "worth_trying": ["洋食"]
  },
  "nara": {
    "must_try": [],
    "worth_trying": ["柿の葉寿司", "茶粥"]
  }
}
```

Phase 0 的信息来源：
- Tabelog 地区菜系分类页（看有哪些菜系、各菜系数量）
- 各城市観光協会、"ご当地グルメ"榜单（确定特色菜）
- Tabelog 搜索结果侧边栏的价格筛选器（确定价格分布）

### Phase 1：Tabelog 批量搜索（第一轮，采基础信息）

```
对 cuisine_map 中每个菜系组:
  对 circle 中每个城市:
    搜索 Tabelog "{城市} {菜系}" 排名页
    取前 2-3 页（40-60 家）
    从列表页提取:
      - 店名 (name_ja)
      - 评分 (tabelog_score)
      - 价格带 (price_band) — Tabelog 标注的 ランチ/ディナー 价格
      - 区域 (area) — Tabelog 的エリア标签
      - 百名店/award 标记
    
    按价格带分组，每个价格带取 top 10 进入池子
```

产出：~240 家，每家只有基础字段。

### Phase 1.5：百名店 + 米其林补漏

```
获取关西百名店完整名单（按年度，各品类）
获取关西米其林完整名单（星级 + Bib Gourmand）

对照 Phase 1 的 240 家:
  不在池中的 → 补入，标记来源
```

产出：+30-50 家。

### Phase 1.6：独立站 + 小吃/体验型补充

```
从 5-6 个精选独立站提取推荐餐厅:
  - 街头小吃 / 食べ歩き（Tabelog 不覆盖的品类）
  - 体验型餐饮（寺院料理、料理教室等）
  - 博主隐藏推荐（Tabelog 排名不高但体验好的）

去掉与 Phase 1/1.5 重复的
```

产出：+40-50 家。

### Phase 1 总产出

~310-340 家，每家有基础字段：

```
name_ja          — 店名
tabelog_score     — Tabelog 评分（独立站来源的可能无此字段）
price_band       — 价格带
city             — 城市
area             — 区域（Tabelog エリア）
cuisine          — 菜系组
source           — 来源标识（tabelog / hyakumeiten / michelin / indie_site）
source_url       — 来源页面链接
local_specialty  — 特色菜标注（null 或菜名，对照 specialty.json 自动匹配）
```

### Phase 2：详情补全（只对推荐池）

Phase 1 产出经筛选/排序后，确定进入推荐池的餐厅（预计 150-200 家），逐家补全：

```
补全字段:
  name_zh          — 中文名
  address          — 地址
  open_hours       — 营业时间
  closed_days      — 定休日
  reservation      — 预约方式（walk_in / easy / hard / impossible）
  meal_type        — lunch / dinner / both
  tabelog_url      — Tabelog 详情页
```

S/A 级（约 50-80 家）额外补全：
```
  signature_dish   — 招牌菜
  queue_risk       — 排队风险
  language_friendly — 语言友好度
  payment_method   — 支付方式
  editorial_note   — 推荐理由（一句话）
```

## 六、特色菜标注

### 规则

特色菜**不影响排名和评分**，只是标签，供行程规划使用。

每个城市的特色菜分两档：
- **must_try** — 来这个城市一定要吃的
- **worth_trying** — 值得尝试的当地特色

### 标注方式

**菜系级特色**（京料理、神戸牛等）：
- 餐厅的 cuisine 匹配 specialty.json 中该城市的特色 → 自动标注 `local_specialty`

**菜品级特色**（たこ焼き、串カツ等，没有独立 Tabelog 菜系分类的）：
- Phase 1.6 独立站搜索时专项搜索 → 结果自动标注 `local_specialty`

### 下游使用

行程规划（Step 13.5 选餐）时：
- 检查用户在某城市期间，must_try 是否已安排
- 未安排的 must_try → 优先插入餐食安排
- worth_trying → 有空位时推荐

## 七、排名规则

### 不设固定分数门槛

不用 "Tabelog >= 3.5" 之类的硬性门槛。每个菜系×价格带内按 Tabelog 评分从高到低排，取 top 10。

原因：
- 3.5 在怀石里是中等，在拉面里已经很强
- 小城市（奈良）好店可能只有 3.2-3.4
- 同一个分数在不同品类含义完全不同

### 圈级排名，标清位置

排名范围是整个圈（如关西），不是单个城市。但每家餐厅必须标清 city + area，供行程规划按位置筛选。

## 八、数据源

| 用途 | 主源 | 说明 |
|------|------|------|
| 发现 + 质量排名 | **Tabelog** 排名列表页 | 唯一质量主源 |
| 权威补充 | 百名店、米其林 | 补漏 Tabelog 排名可能遗漏的名店 |
| 小吃 + 体验型 | 独立站 + 网络搜索 | Tabelog 覆盖弱的品类 |
| 街头小吃补盲 | Google Maps | 评分 4.0+ 且评论 200+ |
| 用户适配验证（Phase 2） | 携程 + 小红书 | 中国游客实际满意度 |
| 执行性验证（Phase 2） | Google Maps + 官网 | 营业状态、定休日、预约方式 |

## 九、负向信号（Phase 2 补全时检查）

以下情况即使评分不低也考虑降级或不推荐：

| 信号 | 处理 |
|------|------|
| 排队 >90 分钟且同区有替代 | 降级或标注排队风险 |
| 网红效应（小红书爆火但 Tabelog 低）| 谨慎推荐，grade 不高于 B |
| 价格与体验不匹配 | 标注性价比问题 |
| 体验波动大（好评差评极端分化）| 标注风险 |
| 已大众化/连锁化失去特色 | 考虑不收 |
| 游客陷阱（本地人公认）| 不收 |
