# 精选哲学: 从"覆盖系统"到"编辑精选系统"

> 版本: 1.0
> 更新: 2026-04-01
> 地位: 所有品类指南的上位文档。如果品类指南与本文冲突，以本文为准。

---

## 一、我们在做什么

做一本纸质手账本(298/348元)，让中国游客照着走就行。

用户付费买的核心价值:
- 确定性 -- 照着走不会扑空
- 信息准确 -- 时间、价格、地址都对
- 惊喜感 -- 有一些"攻略上找不到"的本地推荐
- 节奏感 -- 时间分配合理

用户最怕:
- 到了发现店关门了(最致命)
- 推荐的店不好吃/不好住(信任崩塌)
- 和免费攻略一模一样(不值298元)

**因此，手账本上印的每一条推荐，都必须是"真的值得写"，而不是"有总比没有好"。**

## 二、三层池架构

所有数据经过三层筛选:

```
发现池 (Discovery Pool)
  -- 目标: 尽量不漏，广泛搜索
  -- 数量: 多(可能是终选的5-10倍)
  -- 准入: 任何一个可信数据源提及即可进入
  -- 五语搜索全开
     |
     v
入围池 (Shortlist Pool)
  -- 目标: 交叉验证，确认真的好
  -- 数量: 中(终选的2-3倍)
  -- 准入: 至少2个独立数据源交叉确认
  -- 标记各数据源原始评分
     |
     v
终选池 (Final Selection)
  -- 目标: 狠准地选，允许稀疏
  -- 数量: 只留真正值得写进手账本的
  -- 准入: 编辑判断 -- "如果我自己去，会不会因为这条推荐而感谢这本手账本？"
  -- 某区域某预算没有好选项就空着，不补"还行"的
```

**关键区别: 发现池追求召回率，终选池追求精确率。两者不能用同一套规则。**

### 各层的数据操作

| 操作 | 发现池 | 入围池 | 终选池 |
|------|--------|--------|--------|
| 搜索语言 | 五语全开 | 日文+官方+目标用户语言 | 不搜索，做判断 |
| 独立站 | 20-30个广泛搜 | 核心5-10个深度看 | 不增加，做取舍 |
| AI角色 | 不参与 | 辅助标记 | 写文案+适配判断 |
| 数据源要求 | 任一来源 | >=2个交叉 | 必须有P0来源 |
| 覆盖逻辑 | 越多越好 | 按品类分组 | 允许稀疏 |

## 三、编辑取舍的核心原则

### 3.1 "为什么选它" -- 正向编辑标准

不同品类有不同的"值得选"逻辑，但共通的判断是:

**问自己: "如果一个朋友第一次去这个城市，我会专门告诉他这个地方吗？"**

- 如果是 "当然会，不去太可惜了" -- 入选
- 如果是 "去了不错，不去也行" -- 看有没有更好的替代
- 如果是 "凑数的" -- 不选

### 3.2 "为什么不选它" -- 负向编辑规则

**精选库的价值，恰恰体现在"明明很多人写，我却不收"的判断。**

以下情况即使平台分数不低也应考虑不收录:

| 负向信号 | 说明 | 适用品类 |
|----------|------|---------|
| 排队成本远超体验 | 排3小时吃一碗拉面，体验不值得 | 餐厅 |
| 只靠单一爆品支撑 | 除了那一道菜/那个打卡点，其他平平 | 餐厅/景点 |
| 体验波动大 | 好评差评极端分化，去了赌运气 | 餐厅/酒店 |
| 营业不稳定 | 经常临时休业/缩短营业 | 餐厅/店铺 |
| 高价但缺乏记忆点 | 贵但吃完/住完说不出特别的 | 餐厅/酒店 |
| 和周边走廊搭配差 | 位置孤立，去一趟浪费半天交通 | 所有品类 |
| 对第一次来的人收益低 | 需要深度背景知识才能欣赏 | 景点 |
| 过度商业化 | 游客陷阱，本地人不去 | 餐厅/店铺/景点 |
| 替代品更优 | 同类型同区域有明显更好的选择 | 所有品类 |

**负向判断的记录方式:**

入围池中被负向淘汰的条目，保留记录并注明原因:
```json
"editorial_exclusion": {
  "excluded": true,
  "reason": "排队平均2小时，同区域有3家同品质免排队替代",
  "better_alternatives": ["xxx", "yyy"]
}
```

这些信息将来可以用在 "skip_if" 字段或Plan B推荐中。

### 3.3 稀疏好过注水

**某个区域某个预算层没有真正好的选项，就空着。**

不好的做法: "难波budget层只有2家好的，再补3家凑到5家"
好的做法: "难波budget层精选2家，附近的天王寺budget层有更多好选项"

终选池允许不均匀。用户需要的是"每条都可信"，不是"每格都有"。

## 四、三条标签线

每个入选条目标记其"入选逻辑"，而不是一个笼统的grade:

| 标签 | 含义 | 判断来源 | 示例 |
|------|------|---------|------|
| **city_icon** | 城市身份/文化名片 | 编辑判断+权威来源(JNTO/japan-guide/Tabelog百名店) | 京都怀石(菊乃井)、神户牛(Mouriya)、金阁寺 |
| **traveler_hot** | 游客高热度 | 携程/小红书/Google评论量 | 道顿堀章鱼烧、伏见稻荷千本鸟居 |
| **local_benchmark** | 本地口碑标杆 | Tabelog高分/Retty/日文博客 | 本地人排队的拉面店、地元居酒屋 |

这三者经常重叠(如菊乃井三条都满足)，但不应混为一类。

**关键区别:**
- city_icon: "去京都就应该体验这个" -- 不一定排队最长或评分最高
- traveler_hot: "中国游客都在讨论这个" -- 不一定是本地最好的
- local_benchmark: "日本本地人认为最好的" -- 不一定对中国游客友好

**一个条目可以同时有多个标签。** 菊乃井 = city_icon + local_benchmark。
道顿堀章鱼烧 = traveler_hot，但不一定是 local_benchmark。

### 手账本上的使用

- city_icon: 行程骨架，"来XX必须体验的"
- traveler_hot: "大家都在去的热门"(标注排队风险)
- local_benchmark: "本地人推荐"(手账本的差异化价值，区别于免费攻略)

## 五、品类内相对排名，不用固定阈值

### 问题

固定阈值(如Tabelog >= 3.5)适合粗筛发现池，不适合精选终选池。

- 寿司3.6分和拉面3.6分含义完全不同(寿司竞争更激烈，3.6已经很高)
- 京都怀石3.8和大阪串炸3.8不是同一回事
- budget层3.3可能是顶尖，luxury层3.3可能是平庸

### 正确做法: 品类内相对位置

```
终选标准不是"Tabelog >= 3.5"，而是:
"在这个城市、这个菜系/类型、这个价位段中，是否属于前10-20%？"

具体操作:
1. 先按 city x cuisine x budget_tier 分组
2. 每组内按Tabelog评分(或该品类主评分源)排序
3. 取该组前10-20%作为终选候选
4. 再用编辑判断做最终取舍
```

这意味着:
- 京都怀石top 5家 -> 可能都是3.8+，选2-3家
- 大阪串炸top 5家 -> 可能3.5-3.7就是顶尖了
- 城崎蟹料理 -> 可能只有3家值得写，就选3家

### 评分归一化(从SYSTEM_DESIGN_V2继承)

```
Tabelog 3.0-5.0 -> 0-100: score = (tabelog - 2.5) * 40
Google  1.0-5.0 -> 0-100: score = (google - 1.0) * 25
一休    1.0-5.0 -> 0-100: score = (ikyu - 1.0) * 25
携程    1.0-5.0 -> 0-100: score = (trip - 1.0) * 25
Booking 1.0-10  -> 0-100: score = (booking - 1.0) * 11

归一化后在组内比较，不跨组比较。
```

## 六、品类特有的精选逻辑

### 6.1 餐厅

精选三轴: **品类内地位 x 城市代表性 x 游客适配度**

- 品类内地位: 该城市该菜系中的相对排名(Tabelog为主)
- 城市代表性: 是否代表该城市的饮食文化(city_icon标签)
- 游客适配度: 中国游客是否能愉快体验(语言/排队/支付/口味)

Tabelog百名店 = 品类内地位的权威信号(按品类评top100，天然是相对排名)
米其林 = 品质标杆，但偏高端
携程/小红书高频 = 游客适配度信号

### 6.2 酒店

分轨比较: **hotel_type x district x price_level 内比较**

不把商务酒店和顶级旅馆放一起比。而是:
- 京都祇园luxury_ryokan中，哪家最值得？
- 大阪难波business_hotel中，哪家性价比最高？
- 有马温泉ryokan中，哪3家最值得推荐？

MICHELIN Keys体系(不是旧的"米其林推荐酒店"):
- Three Keys = 住宿体验的最高荣誉
- Two Keys = 杰出住宿
- One Key = 值得特别关注的住宿
- 这是S/A级体验酒店的权威信号

### 6.3 景点

双S制: **heritage_s** 和 **popular_s**

| 类型 | 含义 | 判断来源 | 示例 |
|------|------|---------|------|
| heritage_s | 文化/历史/自然名片 | JNTO/japan-guide/世界遗产 | 金阁寺、姬路城、高野山 |
| popular_s | 大众热门/体验名片 | Google评论量/携程/小红书 | 伏见稻荷、USJ、道顿堀 |

JNTO和japan-guide是景点S/A的主骨架，不是给Google热度背书。
一个景点可以同时是heritage_s和popular_s(如伏见稻荷)。

### 6.4 店铺

按类型拆源:

| 店铺类型 | 精选来源 | 不只靠小红书 |
|----------|---------|-------------|
| 古着/中古 | 本地时尚媒体(F-STREET/装苑)、店铺协会 | 小红书辅助 |
| 工艺/特产 | 协会、博物馆店、百货买手清单 | 独立站辅助 |
| 商店街 | 官方商店街名录、地元媒体 | 攻略站辅助 |
| 杂货/书店 | 设计媒体、本地推荐 | 小红书/攻略站辅助 |

## 七、负向编辑规则详细版

### 7.1 "明明很多人写，我们不收"的情况

每个不收录的决定都要有理由，记录在入围池中:

**餐厅负向规则:**
- 排队>90分钟且无预约选项且同区域有同品质替代 -> 不收或降级+标注
- 只有一道菜值得吃，整体体验一般 -> 降级
- 网红效应明显(小红书爆火但Tabelog < 3.3) -> 标注traveler_hot但不给高grade
- 只收现金+无图片菜单+不会日语完全无法沟通 -> 标注风险，不影响grade但影响推荐优先级
- 价格与体验不匹配(高价但评价"还行") -> 不收或标注 "value_perception: below"

**酒店负向规则:**
- 隔音差的评论反复出现(3+人提及) -> 标注，轻度降级
- 设施老旧多人提及但价格不低 -> 标注 "value_perception: below"
- 位置标称"步行5分钟"但实际要爬坡/过天桥/转弯多 -> 调整access_friction

**景点负向规则:**
- 严重过度商业化(本地人公认的游客陷阱) -> 降级或不收
- 需要特定背景知识才能欣赏(对first_visit用户收益低) -> 标注，不降级但不推荐给首次游客
- 体验高度依赖天气/季节但未标注 -> 不是负向，但必须标清

**店铺负向规则:**
- 营业不稳定(Google显示"营业时间可能不准确") -> 标注 "建议出发前确认"
- 已经大众化/不再有"宝藏感"(如3COINS是连锁) -> 不收录在"宝藏"分类

### 7.2 grade不是由分数机械决定

grade是编辑综合判断，分数只是输入之一:

```
grade = f(
  品类内相对位置,      -- Tabelog/一休排名(主)
  城市代表性,          -- 是否是城市名片(加分)
  游客适配度,          -- 中国游客体验(加分/减分)
  独立站推荐共识,      -- 多个原创推荐(加分)
  负向信号,            -- 排队/波动/不稳定(减分)
  不可替代性           -- 这个体验只有这里有(大加分)
)
```

**同样Tabelog 3.7的两家店，一家是城市唯一的川床料理(不可替代)，一家是普通居酒屋(有很多替代)，grade应该不同。**

## 八、数量不是目标

再强调一次:

**终选池的数量由"有多少真正值得写的"决定，不由"我们计划收录多少"决定。**

- 如果京都怀石只有5家真值得写 -> 就写5家
- 如果某区域budget层没有好选项 -> 就空着，在临近区域补充提示
- 如果某个冷门城市只有10家餐厅值得推荐 -> 就10家

**100家verified的精选 > 400家ai_generated的注水。**
