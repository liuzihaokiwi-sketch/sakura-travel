# 数据采集指南（真实数据源驱动）

> 版本: 2.0
> 更新: 2026-04-01
> 适用: 所有城市圈数据采集

**铁律：真实数据源是唯一可信来源。AI只做辅助判断和文案。没有真实数据验证的条目标记 `ai_generated`，不可用于生产。**

---

## 一、完整采集流程

```
Phase 1: 准备阶段
  ├── 1.1 确定城市圈范围
  ├── 1.2 区域调研（真实数量分布）
  └── 1.3 建立分类体系 + 走廊定义

Phase 2: 真实数据源采集（核心阶段）
  ├── 2.1 景点：Google Maps + japan-guide + JNTO
  ├── 2.2 餐厅：Tabelog百名店 + 米其林 + Google Maps
  ├── 2.3 酒店：一休 + 楽天 + 携程Trip + Booking
  └── 2.4 交叉验证：小红书 + 马蜂窝

Phase 3: AI增强（辅助阶段）
  ├── 3.1 基于真实数据做评级（S/A/B/C）
  ├── 3.2 写攻略文案（why_selected / skip_if / insider_tip）
  ├── 3.3 判断适配人群（best_for / not_suitable_for）
  ├── 3.4 补充覆盖空白（标记ai_generated）
  └── 3.5 性价比评估（基于真实价格）

Phase 4: 结构化 + 校验
  ├── 4.1 脚本生成完整JSON
  ├── 4.2 自动校验（validate_data.py）
  └── 4.3 人工终审S/A级

Phase 5: 导入
  └── 5.1 导入DB + 端到端测试
```

## 1.5 Phase 1 补充：区域调研的具体搜索方法

### 每一步判断都必须基于搜索结果，不能凭"我觉得"

区域调研不是让AI凭记忆列表，是真的搜索。

#### 酒店区域分布调研

```
搜索关键词（逐个搜索，记录结果）：

日文搜索（获取实际数据）：
  → "{城市} ホテル 数 統計" — 官方统计数据
  → "{城市} ホテル エリア別" — 区域分布
  → "{区域名} ホテル 一覧" — 具体区域酒店列表
  → "booking.com {city} hotels" — Booking上的酒店数量

中文搜索（获取中国游客视角）：
  → "{城市}住宿攻略 区域推荐" — 区域选择指南
  → "{城市}住哪里方便" — 交通便利性
  → "小红书 {城市}酒店推荐" — 热门酒店
  → "携程 {城市}酒店" — 携程上的区域分布

实际验证（关键步骤）：
  → 打开 booking.com，搜索{城市}，看各区域筛选后的酒店数量
  → 打开 一休.com，搜索{城市}，看各区域酒店数量和价位
  → 打开 携程Trip.com，搜索{城市}酒店，看区域筛选数量
  → 这些数字就是真实的区域分布

产出：
  → 每个区域的实际酒店数量（来自OTA搜索结果，不是AI估算）
  → 每个区域的价位分布（来自OTA价格排序）
  → 按真实比例计算采集目标数量
```

#### 餐厅区域分布调研

```
搜索关键词：

日文搜索：
  → "tabelog {城市} ランキング" — Tabelog城市总排名
  → "tabelog {城市} {菜系} ランキング" — 分菜系排名
  → "{城市} 百名店 一覧" — 百名店名单
  → "{城市} ミシュラン 一覧" — 米其林名单

中文搜索：
  → "{城市}必吃美食 排名" — 中文攻略
  → "小红书 {城市}美食" — 热门餐厅
  → "携程 {城市}美食" — 携程美食排名
  → "马蜂窝 {城市}美食攻略" — 马蜂窝视角

实际验证：
  → 打开 tabelog.com，搜索{城市}各区域，记录餐厅数量
  → 打开 Google Maps，搜索"{区域} restaurant"，记录数量
  → 打开 携程美食频道，看{城市}排名前50家
```

#### 景点分布调研

```
搜索关键词：

权威数据源：
  → "japan-guide.com {city}" — 英文权威指南
  → "jnto {城市} 観光スポット" — 日本旅游局推荐
  → "tripadvisor {城市} things to do" — 排名

中文数据源：
  → "{城市}景点排名" — 综合排名
  → "小红书 {城市}必去景点" — 中国游客热门
  → "携程 {城市}景点" — 携程景点排名
  → "马蜂窝 {城市}景点" — 马蜂窝排名

Google Maps验证：
  → 搜索 "{区域} tourist attraction"
  → 记录评分>4.0且评论>500的景点
```

## 二、Phase 2 详细流程：真实数据源采集

### 这是整个流程中最重要、最耗时的阶段。不可跳过、不可用AI替代。

---

### 2.1 景点数据采集

#### 数据源及获取方式

| 顺序 | 数据源 | 获取方式 | 得到什么 |
|------|--------|---------|---------|
| ① | **Google Maps Places API** | API调用 | 名称、坐标、评分、评论数、营业时间、地址、照片 |
| ② | **japan-guide.com** | WebFetch各城市页 | 景点列表、推荐等级（top/recommended）、描述 |
| ③ | **JNTO官网** | WebFetch | 官方推荐景点列表 |
| ④ | **TripAdvisor** | API或WebFetch | 排名、评分、评论数 |
| ⑤ | **小红书** | OpenCLI+Chrome | 中国游客高频景点、实际体验反馈 |
| ⑥ | **携程Trip.com** | WebFetch/API | 中国游客评分、门票价格 |

#### 执行步骤

```
Step 1: Google Maps API 批量搜索
  → 按走廊区域搜索 "tourist attraction" / "shrine" / "temple" / "museum" 等
  → 每个走廊取评分>3.5且评论>100的结果
  → 产出：候选景点列表（名称+坐标+评分+评论数+营业时间）

Step 2: japan-guide 交叉
  → WebFetch 该城市的 japan-guide 页面
  → 提取其推荐等级
  → 与Step1结果匹配，未匹配的补入候选池

Step 3: JNTO + TripAdvisor 补充
  → 补充遗漏景点
  → 补充排名信息

Step 4: 小红书/携程 中国游客视角
  → 搜索"{城市}景点" "{城市}必去"
  → 记录高频出现的景点
  → 与候选池交叉，确认覆盖度
  → 补充候选池中未有的热门景点

Step 5: 合并候选池，标记数据来源
  → 每个景点标记来自哪些数据源
  → 记录各数据源的原始评分
```

#### 产出格式（候选池CSV）

```csv
name_ja,name_en,name_zh,google_rating,google_reviews,jg_level,tripadvisor_rank,xiaohongshu_freq,trip_rating,coord_lat,coord_lon,open_hours,admission_jpy,sources
伏見稲荷大社,Fushimi Inari Taisha,伏见稻荷大社,4.6,85000,top,1,very_high,4.7,34.9671,135.7727,00:00-24:00,0,"google,japan_guide,tripadvisor,xiaohongshu,trip"
```

### 2.2 餐厅数据采集

#### 数据源及获取方式

| 顺序 | 数据源 | 获取方式 | 得到什么 |
|------|--------|---------|---------|
| ① | **Tabelog 百名店年度名单** | WebFetch年度名单页 | 各品类top餐厅、评分 |
| ② | **米其林指南** | WebFetch官网 | 星级餐厅+Bib Gourmand名单 |
| ③ | **Tabelog 区域搜索** | WebFetch按区域+菜系 | 评分、价格带、地址、营业时间 |
| ④ | **Google Maps** | API | 评分、评论数、营业时间、坐标 |
| ⑤ | **小红书** | OpenCLI | "{城市}美食"高频店 |
| ⑥ | **携程Trip.com 美食频道** | WebFetch | 中国游客评价、人均价格 |
| ⑦ | **马蜂窝/大众点评（海外版）** | WebFetch | 中国游客评价 |

#### 执行步骤

```
Step 1: 权威名单采集（S/A级骨架）
  → Tabelog百名店（按城市筛选）→ 入围grade≥A
  → 米其林星级+Bib Gourmand → 入围grade≥A
  → 产出：权威推荐列表（约50-80家/城市）

Step 2: Tabelog 全量评分采集（不只是百名店！）
  → 按区域×菜系搜索 Tabelog
  → 评分≥3.5 → 全部收录入候选池
  → 评分3.2-3.5 → 标记"待交叉验证"，Step4中如果携程/小红书也推荐则入围
  → 每个区域每个菜系都要搜，不能只看头部
  → 产出：各区域各菜系的评分3.2+全量名单

  具体搜索方式：
  → tabelog.com → 选择地域（京都/大阪/神戸等）
  → 选择菜系（和食/ラーメン/焼肉/洋食 等）
  → 按评分排序
  → 记录3.2以上全部店铺
  → 注意：budget/street层（~1000日元以下）在Tabelog上评分体系不同，
    便宜的店评分3.3就算很好了，不要用3.5标准卡死

Step 3: Google Maps 交叉验证（全量）
  → 用Step1+2的店名搜索Google Maps
  → 记录Google评分和评论数
  → 确认营业状态（排除已关店）
  → 获取精确坐标
  → 补充Tabelog未覆盖的店（street/小吃类Tabelog不收录，Google Maps有）

Step 4: 中国游客数据源交叉（携程+小红书+攻略网站）
  → 携程Trip美食频道该城市排名
  → 小红书搜索"{城市}美食" "{城市}必吃" "{城市}餐厅"
  → 美食攻略网站：
     - MATCHA (matcha-jp.com) — 多语种日本旅游攻略
     - 乐吃购！日本 (letsgojp.cn) — 台湾/中文深度美食攻略
     - allabout-japan.com — 日本美食专题
     - 马蜂窝美食攻略 — 各城市美食排行
     - Retty — 日本本地美食社区（比Tabelog更偏大众口味）
     - 食べログまとめ — Tabelog用户整理的菜系专题
     - Time Out Tokyo/Osaka — 英文美食推荐
     - Google Maps "best {菜系} in {city}" — 英文视角
  → 记录高频推荐但Step1-2未覆盖的店
  → Step2中3.2-3.5的店，如果携程/小红书也推荐 → 升级入围
  → 这些数据源发现的新店补入候选池，标记来源

Step 5: 合并去重，构建候选池
  → 日文名为去重主键
  → 每家记录所有数据源评分
  → 标记菜系、价格带、所属走廊
  → 标记每家来自哪些数据源（sources字段）
```

#### 不同预算层的采集策略差异

```
luxury/premium（人均3000+日元）:
  → 主力数据源：Tabelog 3.5+ / 米其林 / 一休レストラン
  → 交叉：携程高端餐厅排名 / 小红书"XX高级料理"
  → 这个层级Tabelog覆盖最全，以Tabelog为主

mid（1000-3000日元）:
  → 主力数据源：Tabelog 3.2+ / Google Maps 4.0+
  → 交叉：携程 / 小红书 / Retty
  → 这个层级数量最大，需要区域×菜系逐个搜

budget（500-1000日元）:
  → 主力数据源：Google Maps 4.0+ / Retty / 携程
  → 交叉：小红书 / 马蜂窝
  → Tabelog对这个层级评分偏低（3.0-3.3就算好的），不要用3.5卡死
  → 注意连锁店（松屋/吉野家不收录，但地方特色连锁可以）

street（<500日元）:
  → 主力数据源：Google Maps / 小红书 / 携程
  → Tabelog基本不覆盖这个层级
  → 重点看小红书实际推荐（"XX市场必吃小吃"）
  → 重点看携程美食排名中的小吃类
  → 攻略网站的"街头美食"专题
```

#### 产出格式（候选池CSV）

```csv
name_ja,name_zh,cuisine,budget_tier,tabelog_score,tabelog_award,michelin,google_rating,google_reviews,xiaohongshu_freq,trip_rating,corridor,lunch_jpy,dinner_jpy,address_ja,coord_lat,coord_lon,open_hours,closed_day,sources
菊乃井 本店,菊乃井本店,nihon_ryori,luxury,4.52,百名店2024,3,4.6,3200,high,4.8,higashiyama,8800-22000,22000-55000,京都市東山区下河原町459,34.9992,135.7810,12:00-13:00;17:00-19:30,月曜,"tabelog,michelin,google,xiaohongshu,trip"
```

#### 入围/淘汰规则

```
直接入围（grade ≥ A）:
  → Tabelog百名店
  → 米其林星级/Bib Gourmand
  → Tabelog ≥ 3.7 且 Google ≥ 4.3

入围（grade B）:
  → Tabelog ≥ 3.5 且 Google ≥ 4.0
  → 小红书高频(≥30篇) 且 Google ≥ 4.0

候选（需确认）:
  → 仅一个数据源高分
  → AI补充的覆盖空白

淘汰阈值（按数据类型区分）:
  → 餐厅：Google < 3.5 或 Tabelog < 3.0
  → 景点：Google < 3.0（注意：很多神社/公园评分3.8-4.0但是必去）
  → 酒店：Google < 3.3
  → 已永久关闭 → 淘汰
  → 小红书差评率 > 40% → 标记风险

特别注意：
  → Tabelog百名店但小红书搜不到 → 仍入围，但标记"中国游客认知度低"
  → 小红书很火但Tabelog平平 → 入围为B级，标记"网红热度高，本地口碑一般"
  → 目标用户是中国20-40岁游客，小红书/携程热度比Tabelog更贴近真实需求
```

### 2.3 酒店数据采集

#### 数据源及获取方式

| 顺序 | 数据源 | 获取方式 | 得到什么 |
|------|--------|---------|---------|
| ① | **一休.com** | WebFetch区域搜索 | 评分排名、价格、设施、含餐信息 |
| ② | **楽天トラベル** | Travel API | 评分、价格、设施、评论 |
| ③ | **じゃらん** | WebFetch | 评分、价格、口碑 |
| ④ | **携程Trip.com** | WebFetch/API | 中国游客评分、实际价格、中文评论 |
| ⑤ | **Booking.com** | Affiliate API | 国际评分、价格、设施 |
| ⑥ | **Agoda** | WebFetch | 补充价格对比 |
| ⑦ | **Google Maps** | API | 评分、坐标、营业状态 |
| ⑧ | **小红书** | OpenCLI | 中国游客住宿体验、实拍 |

#### 执行步骤

```
Step 1: 区域调研（已在Phase 1完成）
  → 已知每个区域有多少酒店、什么价位分布

Step 2: OTA批量采集
  → 一休.com：按区域搜索，按评分排序，每区域取前20-30
  → 楽天トラベル：同上
  → 携程Trip：同上（特别重要——中国游客真实价格和评价）
  → Booking.com：同上

  每家酒店记录：
  - 各平台评分
  - 各平台价格（淡旺季）
  - 含餐情况
  - 房型信息
  - check-in/check-out时间

Step 3: 合并去重
  → 日文名为主键
  → 合并各平台评分和价格
  → 计算综合评分（多平台加权平均）

Step 4: 按区域×价位筛选
  → 每区域每价位取综合评分top N:
     luxury: top 3-5
     expensive: top 5-8
     moderate: top 8-12
     budget: top 5-8
     backpacker: top 3-5

Step 5: 中国游客视角补充
  → 小红书搜索"{城市}住宿" "{区域}酒店"
  → 携程Trip评论分析
  → 确认高频推荐酒店已收录
  → 补充中国游客特别偏爱但OTA排名一般的酒店

Step 6: 独立攻略站交叉（酒店同样适用2.5节的流程）
  → 先从已验证核心站获取该城市酒店推荐
  → 搜索补充到20-30个独立站
  → 筛选5-10个核心站
  → 提取高频推荐酒店
  → 提取入住体验维度（设施/位置/早餐/隔音等）

Step 7: Google Maps验证
  → 用name_ja + Google Places text search获取正确坐标
  → 不能只查坐标——要用店名搜索拿到正确坐标再对比
  → 确认营业状态
  → 评分交叉
```

#### 产出格式（候选池CSV）

```csv
name_ja,name_zh,hotel_type,district,ikyu_score,rakuten_score,trip_score,booking_score,google_rating,google_reviews,off_jpy_min,off_jpy_max,reg_jpy_min,reg_jpy_max,peak_jpy_min,peak_jpy_max,meals,check_in,check_out,coord_lat,coord_lon,address_ja,sources
俵屋旅館,俵屋旅館,luxury_ryokan,祇园东山,4.8,4.7,4.9,9.5,4.7,1200,80000,150000,100000,200000,150000,300000,早晚餐,16:00,11:00,35.0071,135.7681,京都市中京区麩屋町通姉小路上る中白山町278,"ikyu,rakuten,trip,booking,google"
```

#### 价格采集要点

```
酒店价格必须来自OTA真实数据，不可AI估算。

采集方法：
  → 一休/楽天/携程各搜索3个时段：
     淡季（1月平日）
     平季（5月平日）
     旺季（11月周末/樱花季）
  → 记录最低和最高价格
  → 标注是否含餐、含税

注意事项：
  → 同一酒店不同平台价格可能差30%以上
  → 记录多个平台价格，取中位数作为参考价
  → 旺季价格波动极大，标注"仅供参考"
  → 温泉旅馆通常含早晚餐，城市酒店通常不含
```

### 2.4 交叉验证

#### 小红书验证（所有品类）

```
目的：确保手账本推荐的店/景点/酒店，中国游客真的喜欢

方法：
  → 搜索 "{城市}+{品类}" 关键词
  → 统计高频出现的名字
  → 我们的候选池中有 → 标记 xiaohongshu_verified
  → 我们没有但小红书很火的 → 补入候选池
  → 我们有但小红书差评多的 → 标记风险

具体搜索词：
  景点："{城市}景点" "{城市}必去" "{城市}小众"
```

### 2.5.1 独立攻略站数据的使用规则

### 核心问题：多站推荐 ≠ 多票投票

独立攻略站之间可能存在**内容同源**——一个博主原创，其他博主转载/引用/改写。
如果盲目把"5个站都推荐"当5票，实际可能只是1个原创信号。

### 去重判断方法

```
对于同一家餐厅/酒店被多个攻略站提到：

Step 1: 检查是否有原创内容
  → 有原创照片（实拍，非官方图）= 原创
  → 有具体体验描述（"我点了XX，味道如何"）= 原创
  → 只是列名单+官方描述 = 可能转载

Step 2: 比对内容相似度
  → 多个站的推荐语/菜品描述/照片高度相似 → 视为同源，只算1票
  → 各站描述角度不同、照片不同 → 视为独立证据，分别计票

Step 3: 判定投票权重
  → 有原创体验的独立推荐 = 1票（满权重）
  → 内容疑似转载但无法确认 = 0.3票
  → 明确转载/引用同一来源 = 0票（不计）
  → 带商业推广标记（PR/AD/sponsored）= 0.2票（降权）
```

### 独立站对评级的影响

```
独立站数据的价值：
  → 不能独立决定评级（必须有P0/P1数据源）
  → 但可以影响评级升降

升级条件（最多升1级）：
  → ≥3个独立原创推荐 + 评价真实可靠 → 可升1级（如B→A）
  → 但升级后的等级不超过P0数据源支撑的上限
  → 例如：Tabelog 3.3 + 5个独立站强推 → 最多升到B，不到A

降级条件（标记风险）：
  → P0数据源高分（如百名店）但多个独立站/小红书差评多
  → 不直接降级，而是标记 risk_flag: "高评分但用户体验差评多"
  → 在descriptions.skip_if中说明差评原因

具体信号对评级的影响：
  → 3+个独立原创好评 → grade_boost: +0.5
  → 5+个独立原创好评 → grade_boost: +1.0（最大）
  → 3+个独立差评 → grade_penalty: -0.5
  → 但最终grade由人工确认，boost/penalty只是建议
```

### 独立站数据的采集流程

```
Step 1: 按城市搜索攻略文章
  → 搜索 "{城市}美食推荐" / "{城市}必吃" / "{城市} food guide"
  → 记录前20篇文章的来源URL和推荐的店铺列表

Step 2: 建立"提及频率表"
  → 统计每家店被多少篇文章提到
  → 记录每篇文章是否有原创内容

Step 3: 去重计票
  → 按上述规则去重
  → 得到每家店的"独立推荐票数"

Step 4: 与P0数据源交叉
  → 独立站高频推荐 + Tabelog高分 → 高置信入围
  → 独立站高频推荐 + Tabelog无数据 → 入围但标记"待P0验证"
  → 独立站零提及 + Tabelog高分 → 正常入围，标记"中国游客认知度低"
```

## 三、Phase 3: AI增强

### AI的角色边界

```
 AI可以做：
  → 基于真实评分数据做 S/A/B/C 评级判断
  → 写攻略文案（why_selected, skip_if, insider_tip）
  → 判断 best_for / not_suitable_for
  → 判断 experience.grade（酒店体验等级）
  → 判断 access_friction（交通摩擦）
  → 补充候选池覆盖空白（但必须标记 ai_generated）
  → 性价比评估（基于真实价格数据）

 AI不可以做：
  → 编造评分（Tabelog分、Google分等）
  → 估算价格（必须来自OTA）
  → 作为唯一数据来源
  → 生成的数据直接标记为 verified
```

```
输入：候选池CSV（含各平台真实评分+价格）
```

## 3.5 关键原则：每一步判断都必须先搜索

```
 错误示范：
  AI: "菊乃井本店的Tabelog评分大概是4.5左右"
  → 这是AI根据"米其林三星"推测的，不是真的

 正确示范：
  搜索 tabelog.com "菊乃井 本店" → 看到实际评分 4.52
  搜索 Google Maps "菊乃井 本店 京都" → 看到 4.6分 3200评论
  搜索 trip.com "菊乃井" → 看到 4.8分 人均¥1500-5000
  → 综合三个真实数据源做判断
```

这个原则适用于所有环节：

| 环节 | 必须搜索验证 | 不能凭AI记忆 |
|------|-------------|-------------|
| 区域调研 | 搜索OTA各区域酒店数量 | 不能AI说"大概有200家" |
| 评分 | 搜索Tabelog/Google/携程真实评分 | 不能AI说"大概4.5分" |
| 价格 | 搜索OTA真实价格 | 不能AI说"大概2万日元" |
| 营业状态 | 搜索Google Maps确认在营 | 不能AI说"应该还在" |
| 坐标 | 搜索Google Maps获取 | 不能AI凭记忆给坐标 |
| 营业时间 | 搜索官网或Google Maps | 不能AI说"大概9点到5点" |
| 排名 | 搜索各平台实际排名 | 不能AI说"应该排前几" |

**如果搜索不到某个数据，标记为缺失（null），不要用AI填充假数据。**
**标记缺失远比填充假数据好。**

```python
def compute_confidence(item):
    sources = item.get('sources', [])
    p0_sources = {'tabelog', 'michelin', 'ikyu', 'rakuten', 'google', 'japan_guide'}
    p1_sources = {'booking', 'trip_com', 'jalan', 'tripadvisor', 'jnto'}
    
    p0_count = len(set(sources) & p0_sources)
    p1_count = len(set(sources) & p1_sources)
    
    if p0_count >= 2:
        return 'verified'
    elif p0_count >= 1 or (p0_count + p1_count) >= 2:
        return 'cross_checked'
    elif len(sources) >= 1:
        return 'single_source'
    else:
        return 'ai_generated'  # ← 不可用于生产
```

```
  → 人民币价格直接显示（用户最直观）
  → 中文评论可直接引用作为攻略素材
  → 排名反映中国游客实际偏好（而非日本本地口味）

获取方式:

  方案A: WebFetch（推荐，无需API）
  → 酒店: https://www.trip.com/hotels/{city}/hotel-list/
  → 景点: https://www.trip.com/travel-guide/{city}/tourist-attractions/
  → 餐厅: https://www.trip.com/travel-guide/{city}/restaurants/
  → WebFetch提取排名、评分、评论数、价格

  方案B: Trip.com Open Platform API（如果需要批量）
  → 官网: https://www.trip.com/openplatform/
  → 需申请合作伙伴资格
  → 适合大量数据批量获取

具体操作步骤:

  酒店:
  → 搜索"{城市}"，按"好评优先"排序
  → 记录前50家：名称、评分、评论数、价格区间
  → 切换"价格最低"排序，记录budget层选项
  → 按区域筛选，了解各区域分布
  → 重点关注：中文评论中反复提及的优缺点

  景点:
  → 搜索"{城市}景点"
  → 记录排名前30：名称、评分、评论数、门票价格
  → 注意携程独有的"必去景点"标签

  餐厅:
  → 搜索"{城市}美食"
  → 记录排名前30：名称、评分、人均消费
  → 携程的人均消费是人民币，直接可用

数据对比价值:
  → 一家酒店在一休4.5分但携程3.8分 → 说明日本人喜欢但中国客不满意
  → 一家餐厅Tabelog 3.5但携程4.8 → 说明中国游客特别喜欢（可能是网红效应）
  → 这种差异本身就是攻略的素材（"日本人评价一般但中国游客普遍好评"）
```

### 5.5 小红书（第二优先级辅助源 — 贴近目标用户）

```
用途: 中国游客真实体验、热门度、实际痛点
地位: P2但应提前使用（目标用户画像最匹配的数据源）

为什么要提前到第二优先级:
  → 目标用户是中国20-40岁游客，小红书是他们旅行决策的第一信息源
  → Tabelog 4.0但小红书搜不到的店，对目标用户意义不大
  → 小红书的差评比好评更有价值（"排队3小时"、"实际很坑"）
  → 小红书能发现OTA覆盖不到的信息（如"和服体验哪家好"）

获取方式:
  → OpenCLI + Chrome扩展 + 登录（项目已安装）
  → 手动搜索 + AI辅助提取

具体搜索策略:

  基础搜索（每个城市必做）:
  → "{城市}旅游攻略" — 整体认知
  → "{城市}必去" — 高频景点
  → "{城市}美食" "{城市}必吃" — 高频餐厅
  → "{城市}住宿" "{城市}酒店推荐" — 热门酒店
  → "{城市}踩雷" "{城市}避坑" — 差评信息（极有价值）

  深度搜索（按品类）:
  → "{城市}拉面" "{城市}和牛" — 分菜系搜索
  → "{区域}住哪里" — 区域选择
  → "{酒店名}" — 具体酒店评价
  → "{餐厅名} 排队" — 排队/预约信息

  提取方法:
  → 搜索后看前50篇笔记
  → 记录高频出现的店名/景点名（出现≥5次=高频）
  → 记录反复提到的优点和缺点
  → 记录实际花费（比AI估算更真实）

  输出格式:
  → 高频景点列表（名称+出现次数+主要评价）
  → 高频餐厅列表（名称+出现次数+人均+排队情况）
  → 高频酒店列表（名称+出现次数+价格+评价）
  → 避坑列表（名称+被吐槽的原因）

偏差修正:
  → 网红偏差：出片好≠真好，需与Tabelog/Google交叉
  → 时效偏差：3个月前的攻略信息可能已过时
  → 样本偏差：年轻女性用户占比高，需考虑其他画像
  → 不能作为唯一来源，但差评信息可信度高
```

## 六、Agent使用规范

### 6.1 并发控制

**绝对不超过3个并发agent。** 超过会触发API rate limit。

### 6.2 每批数量

| 任务类型 | 每agent最大条数 | 说明 |
|----------|----------------|------|
| 数据源WebFetch | 不限 | 按URL逐个抓 |
| AI评级+文案 | 30-35条 | 每条需要判断力 |
| JSON结构化 | 不用agent | 脚本处理 |

### 6.3 模型选择

| 任务 | 模型 | 理由 |
|------|------|------|
| 数据源抓取+解析 | sonnet | WebFetch+提取 |
| 评级判断 | opus | 需要深度判断力 |
| 攻略文案 | opus | 需要文笔和洞察 |
| 适配判断 | sonnet | 规则化判断 |
| 覆盖空白补充 | sonnet | 结构化工作 |
| S/A级终审 | opus | 最终质量把关 |

### 6.4 Token优化

| 方式 | 每条token | 说明 |
|------|----------|------|
| 完整JSON内联prompt | ~500 |  浪费 |
| Schema引用+精简prompt | ~200 |  可用 |
| 候选池CSV→AI增强→脚本合并 | ~100 |  推荐 |

## 七、关西经验教训

### 7.1 犯过的错（绝不再犯）

| 错误 | 后果 | 正确做法 |
|------|------|---------|
| 让AI凭空生成380家餐厅 | 评分/价格全不可信 | 先Tabelog/Google采集真实数据 |
| 让AI估算酒店价格 | 与OTA实际价格可能差50% | 一休/楽天/携程真实价格 |
| 没用携程Trip数据 | 缺失目标用户最常用平台 | 携程必须作为核心数据源 |
| 追求数量(380家)忽视质量 | 大量数据不可信 | 宁少勿假，100家verified > 400家ai_generated |
| 跳过数据源验证直接生成 | 生产环境不可用 | Phase 2不可跳过 |

### 7.2 数量参考（不是目标）

数量由真实数据源决定，不要预设目标再凑数。

| 品类 | 合理范围 | 决定因素 |
|------|---------|---------|
| 景点 | 100-200/城市圈 | Google Maps + japan-guide实际数量 |
| 餐厅 | 200-400/城市圈 | Tabelog百名店 + 区域×预算覆盖度 |
| 酒店 | 150-300/城市圈 | OTA区域×价位top N |

**如果真实数据源只支撑200家餐厅，就做200家。不要为了凑400家用AI编剩下200家。**
