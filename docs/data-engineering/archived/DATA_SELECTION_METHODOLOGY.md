# 数据选品方法论：数据源驱动 + AI增强

> 版本: 1.0
> 更新: 2026-04-01
> 核心原则: **真实数据决定选什么，AI只做判断和文案**

---

## 一、问题

### 当前做法的缺陷

```
AI凭训练数据生成推荐 → 可能过时、幻觉、遗漏
```

- AI推荐的"名店"可能已经倒闭或搬迁
- AI不知道2025年新开的米其林店
- AI的"性价比判断"没有真实价格锚定
- AI推荐偏向知名度高的，遗漏真正的本地好店

### 应该的做法

```
多数据源采集排名 → 交叉验证 → AI做判断+文案 → 人工终审
```

**AI的角色从"推荐者"变成"编辑"——数据源告诉我们有什么，AI判断值不值得写进手账本。**

## 二、三类数据的选品流程

### 2.1 餐厅选品

#### 数据源优先级

| 优先级 | 数据源 | 权重 | 获取方式 | 说明 |
|--------|--------|------|---------|------|
| P0 | Tabelog 百名店 | ★★★★★ | 年度名单公开 | 日本最权威美食评选 |
| P0 | 米其林指南 | ★★★★★ | 年度名单公开 | 国际权威，偏高端 |
| P1 | Tabelog 3.5+ 评分 | ★★★★ | API/爬取 | 日本本地口碑 |
| P1 | Google Maps 4.3+且>500评论 | ★★★★ | API | 大样本跨国口碑 |
| P2 | 小红书 高频出现（>50篇提及） | ★★★ | 爬取/手动 | 中国游客真实偏好 |
| P2 | 马蜂窝/携程 高分 | ★★★ | 爬取 | 中国游客量大 |
| P3 | AI知识库 | ★★ | 模型输出 | 补充覆盖，兜底 |

#### 选品规则

```
入围条件（满足任一）：
  - Tabelog百名店 → 直接入围，grade ≥ A
  - 米其林星/Bib → 直接入围，grade ≥ A
  - Tabelog ≥ 3.7 且 Google ≥ 4.3 → 入围，grade ≥ B
  - 小红书 ≥ 50篇提及 且 Google ≥ 4.0 → 入围，grade B
  - 以上都不满足但AI认为有特色 → 候选，需人工确认

淘汰条件（满足任一）：
  - Google < 3.5 → 淘汰
  - Tabelog < 3.0 → 淘汰
  - 小红书差评率 > 30% → 标记风险
  - 已永久关闭 → 淘汰
```

#### 城市必吃（is_city_must_eat）认定标准

必须满足以下**至少两条**：
1. Tabelog百名店 或 米其林星
2. Google评论 > 1000 且 评分 > 4.3
3. 小红书搜索"{城市}美食"前50篇中出现 > 5次
4. 代表该城市独特菜系（如京都汤豆腐、大阪章鱼烧）

#### 执行步骤

```
Step 1: 抓取数据源名单
  → Tabelog百名店（按城市筛选）
  → 米其林指南（按城市筛选）
  → Google Maps 按走廊搜索 "restaurant" 取评分>4.0
  → 小红书 "{城市}美食" 高频店名提取

Step 2: 合并去重，建立候选池
  → 按店名去重（日文名为准）
  → 标记每家在各数据源的排名/评分

Step 3: AI评级（opus）
  → 输入：候选池（名字+各数据源评分+菜系+价位）
  → 输出：grade、profile_boosts、why_selected、skip_if
  → AI不决定选什么，只决定怎么评价

Step 4: 按走廊×预算层检查覆盖度
  → 每个走廊每个预算层至少2家
  → 空白处由AI补充推荐，但标记"AI推荐，待验证"

Step 5: 人工终审S/A级
```

### 2.2 酒店选品

#### 数据源优先级

| 优先级 | 数据源 | 权重 | 获取方式 | 说明 |
|--------|--------|------|---------|------|
| P0 | 一休.com 区域评分排名 | ★★★★★ | 页面/API | 日本高端住宿权威 |
| P0 | 楽天トラベル 评分排名 | ★★★★★ | API | 日本最大OTA |
| P1 | Booking.com 评分>8.5 | ★★★★ | API | 国际样本 |
| P1 | Google Maps 4.3+且>200评论 | ★★★★ | API | 跨平台验证 |
| P2 | 小红书 住宿推荐高频 | ★★★ | 爬取 | 中国游客偏好 |
| P2 | じゃらん 口コミ高評価 | ★★★ | 页面 | 日本家庭客偏好 |
| P3 | AI知识库 | ★★ | 模型输出 | 补充兜底 |

#### 选品规则

```
每个区域×价位的选品逻辑：

Step 1: 确定区域实际酒店总数（调研阶段已完成）
Step 2: 按价位分层，每层取OTA评分top N
  - luxury: 该区域评分前3
  - expensive: 前5
  - moderate: 前8
  - budget: 前5
  - backpacker: 前3

Step 3: 交叉验证
  - 一休 + Booking 都高分 → 可信
  - 只在一个平台高分 → 需查看原因（可能是该平台独占）
  - 小红书高频推荐 → 中国客体验好的信号

Step 4: AI做体验判断
  - experience.grade: 看实际设施（温泉/景观/建筑）
  - access_friction: 查实际交通路线
  - best_for/not_suitable_for: 综合判断
```

#### 性价比评估

酒店的性价比不是"便宜"，而是"同价位中体验最好"：

```json
"value_assessment": {
  "score": 9,
  "reasoning": "同价位（12000日元）唯一含天然温泉大浴场+免费夜宵",
  "compared_to": "同区域同价位酒店只有淋浴，无温泉",
  "best_deal_tip": "平日比周末便宜30%，淡季可能低至8000"
}
```

评估维度：
1. **设施性价比**: 同价位提供更多（温泉/早餐/景观）
2. **位置性价比**: 核心区域但价格不贵
3. **体验性价比**: 花同样的钱但体验升级
4. **时机性价比**: 淡季/午餐/平日的价格差

### 2.3 景点选品

#### 数据源优先级

| 优先级 | 数据源 | 权重 | 获取方式 | 说明 |
|--------|--------|------|---------|------|
| P0 | Google Maps 评分×评论数 | ★★★★★ | API | 最大样本 |
| P0 | japan-guide.com 评级 | ★★★★★ | 爬取 | 最权威英文旅游指南 |
| P1 | JNTO 推荐 | ★★★★ | 官网 | 日本官方旅游局 |
| P1 | TripAdvisor 排名 | ★★★★ | API | 国际游客排名 |
| P2 | 小红书/马蜂窝 高频 | ★★★ | 爬取 | 中国游客偏好 |
| P2 | 台湾旅游博主推荐 | ★★★ | 爬取 | 中文深度攻略 |
| P3 | AI知识库 | ★★ | 模型输出 | 补充兜底 |

#### S级认定标准（城市圈名片）

必须满足以下**全部**：
1. Google评论 > 10000 且 评分 > 4.3
2. japan-guide 或 JNTO 重点推荐
3. 小红书搜索该城市时前10篇中出现
4. 具有不可替代性（这个体验只有这里有）

#### 评级量化公式（参考，非机械执行）

```
raw_score = 
  google_rating × 2 +
  (log10(google_review_count) - 2) × 1.5 +   // 评论数对数化
  japan_guide_stars × 3 +                       // 0-3星
  jnto_featured × 2 +                          // 0或1
  xiaohongshu_frequency × 0.5 +                // 0-10
  uniqueness × 3                                // AI判断 1-5

S: raw_score > 25
A: raw_score 18-25
B: raw_score 12-18
C: raw_score 8-12
不收录: < 8
```

**注意：这个公式只是参考，最终评级由AI+人工综合判断。**
特殊加分项（AI判断）：
- 世界遗产 +5
- 国宝建筑 +3
- 季节限定体验（如樱花/红叶名所）+3
- 独一无二的体验（如宿坊、川床）+3

## 三、性价比评估体系

### 3.1 什么是"性价比"

性价比 ≠ 便宜。性价比 = **花出去的钱带来的体验值不值**。

三个层面：
1. **绝对性价比**: 500日元的章鱼烧 vs 50000日元的怀石 — 不比较
2. **同层性价比**: 同是luxury层，虹夕诺雅 vs 俵屋哪个更值（价格×体验）
3. **跨层惊喜**: budget层但体验超预期（如Dormy Inn温泉）

### 3.2 性价比字段

在三类数据中统一加入：

```json
"value_signal": {
  "tier_value": "high",          // high/medium/low — 同预算层内的性价比
  "surprise_factor": "string",   // 超预期的体验（如果有）
  "best_deal_tip": "string",     // 怎么获得最佳性价比
  "avoid_trap": "string"         // 性价比陷阱提醒
}
```

示例：

**餐厅：**
```json
"value_signal": {
  "tier_value": "high",
  "surprise_factor": "午餐8800日元体验米其林三星",
  "best_deal_tip": "午餐比晚餐便宜60%，菜品同级别",
  "avoid_trap": null
}
```

**酒店：**
```json
"value_signal": {
  "tier_value": "high",
  "surprise_factor": "12000日元含天然温泉+免费夜宵拉面",
  "best_deal_tip": "官网早割（28天前预订）省15%",
  "avoid_trap": "旺季价格翻3倍，不如换区域"
}
```

**景点：**
```json
"value_signal": {
  "tier_value": "high",
  "surprise_factor": "免费入场但体验顶级",
  "best_deal_tip": "清晨6点去无人，体验完全不同",
  "avoid_trap": "白天人多体验大打折扣"
}
```

## 五、改进后的完整采集流程

### V2流程（数据源驱动）

```
Phase 1: 数据源采集
  ├── 抓取 Tabelog百名店/米其林（餐厅P0）
  ├── 抓取 一休/楽天 排名（酒店P0）
  ├── 抓取 Google Maps 评分top（景点P0）
  ├── 抓取 japan-guide 推荐（景点P0）
  └── 抓取 小红书高频（P2交叉验证）

Phase 2: 候选池构建
  ├── 合并去重（按日文名）
  ├── 标记各数据源评分
  ├── 按选品规则筛选入围
  └── 按区域×价位检查覆盖度

Phase 3: AI增强（opus）
  ├── 评级: S/A/B/C（基于数据源排名 + AI判断）
  ├── 体验标签: experience.grade/types
  ├── 适配: best_for / not_suitable_for
  ├── 文案: why_selected / skip_if / insider_tip
  ├── 性价比: value_signal
  └── 补充: 数据源未覆盖的特色小店

Phase 4: 结构化填充（脚本/haiku）
  ├── 坐标、地址、营业时间 → Google Maps API
  ├── 价格换算 → 脚本
  ├── JSON格式化 → 脚本
  └── 校验 → validate_data.py

Phase 5: 终审
  ├── S/A级逐条人工审核
  ├── 价格抽查对比OTA
  ├── 坐标地图验证
  └── 覆盖度检查
```

### V1流程（当前关西用的，回顾对比）

```
AI凭记忆生成 → 生成完整JSON → 人工抽查
```

**V2相比V1的优势：**
- 选品有数据支撑，不是AI拍脑袋
- 性价比有真实价格锚定
- 不会遗漏真正的好店
- 不会推荐已关闭的店
- 评级有量化依据

**V2的代价：**
- Phase 1需要爬取/API调用（额外开发）
- 流程更长
- 但质量显著提升

## 七、关西数据事故复盘与处理方案

### 7.1 事故描述

2026-04-01，在关西城市圈数据采集中，使用AI（Claude Sonnet）凭空生成了：
- 380家餐厅推荐数据
- ~250家酒店推荐数据
- 消耗约150万token + 数小时时间

**这些数据没有使用任何真实数据源验证。** 评分、价格、坐标、营业时间全部是AI估算。
这是一次严重的流程错误。

### 7.2 根本原因

1. **流程设计错误**: 让AI直接输出"推荐列表"，而不是先采集真实数据
2. **没有数据源验证环节**: 生成完就直接当成果，没有交叉验证
3. **追求数量忽视质量**: 为了凑够380家/300家目标，批量生成不把关
4. **混淆了AI能力边界**: AI可以做判断和文案，但不能作为数据来源

### 7.3 处理方案

#### 原则：保留骨架，全量标记存疑，真实数据回填后才能使用

#### Step 1: 全量标记（脚本，立即执行）

对所有现有数据加 `data_confidence` 字段：

```python
# 所有现有关西数据
for item in all_items:
    item['data_confidence'] = {
        'level': 'ai_generated',
        'sources': [],
        'verified_fields': [],
        'unverified_fields': ['grade', 'pricing', 'review_signals', 'coord', 'when'],
        'last_verified_at': None,
        'notes': '2026-04 AI生成，未经真实数据源验证，不可用于生产'
    }
```

#### Step 2: 骨架价值评估

**可保留的部分（AI能做好的）：**
- 店铺/酒店名单（大部分是真实存在的知名店铺）
- 分类归属（cuisine码、hotel_type、corridor_tags）
- 结构化设计（字段定义、枚举值）
- best_for / not_suitable_for 的逻辑判断
- why_selected / skip_if 的攻略文案（需真实数据验证后微调）

**必须替换的部分（AI不可信的）：**
- 评分（Tabelog分、Google分 → 全部是AI编的）
- 价格（餐厅价格、酒店三季价格 → 全部是AI估算）
- 坐标（部分已确认反转，其余未验证）
- 营业时间（格式混乱，内容可能不对）
- 评级（S/A/B/C没有真实数据支撑 → 验证后可能大量变动）

#### Step 3: 真实数据回填（按优先级）

```
P0 — 立即做（阻塞生产）
  ├── Google Maps API: 验证全量坐标+营业状态+评分（景点+餐厅+酒店）
  ├── 淘汰已关闭店铺
  └── 修复坐标反转

P1 — 尽快做（影响推荐质量）
  ├── Tabelog: 验证S/A级餐厅评分（~200条）
  ├── 一休/楽天/携程Trip: 验证S/A级酒店评分+实际价格（~60条）
  └── 基于真实评分重新评级（S/A可能降级，B可能升级）

P2 — 陆续做（提升覆盖度）
  ├── 小红书: 验证热门店覆盖度，补充遗漏的网红店
  ├── Booking.com: 补充国际评分
  └── 全量价格回填（OTA实际价格替换AI估算）

P3 — 最后做（锦上添花）
  ├── value_signal 性价比评估（基于真实价格）
  └── 体验描述去模板化（S级人工重写）
```

#### Step 4: 升级可信度

```
验证通过后：
  data_confidence.level: 'ai_generated' → 'cross_checked' 或 'verified'
  data_confidence.sources: 添加实际验证的数据源
  data_confidence.verified_fields: 添加已验证的字段
  data_confidence.last_verified_at: 更新时间

验证不通过：
  Google < 3.5 → 淘汰
  Tabelog < 3.0 → 淘汰
  已关闭 → 淘汰
  评分差距 > 1.0 → 降级 + 标记
```

### 7.4 禁止事项（铁律）

以下行为在任何城市圈数据采集中**绝对禁止**：

```
 让AI凭空生成推荐列表并当成数据使用
 AI生成的评分/价格直接用于生产
 跳过真实数据源验证直接发布
 为了凑数量降低验证标准
 把 ai_generated 的数据标记为 verified
```

以下行为是**正确做法**：

```
 先从真实数据源采集候选池
 AI只做辅助判断（评级、适配、文案）
 所有数据标记可信度等级
 存疑数据不进生产环境
 定期用真实数据源刷新验证
```
