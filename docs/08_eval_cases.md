# 完整攻略生成评测飞轮与首批用例系统（统一版）

## 0. 文档目标

这份文档的目标不是再讨论“要不要做评测”，而是把完整攻略生成的评测体系一次性定成可以执行的系统。

本文档覆盖：
- 为什么必须建立评测飞轮
- 评测对象到底测什么
- code AI、API AI、你、我四方如何分工
- 首批 20 个用例模板
- 3 个 grader rubric 模板
- 数据结构、运行流程、回归机制、发布门槛
- 失败归因与优化优先级
- 你需要亲自参与的环节

本文档默认采用你后续已经定下来的新口径，而不是旧文档中的旧产品定义：
- 免费阶段 = 1 天完整样片 + 后续高光钩子
- 付费后再补详细表单
- 完整攻略生成优先复用片段攻略，而不是每单从零生成全文
- 后端主链 = 画像标准化 → 片段召回 → 硬规则过滤 → 软规则重排 → 装配 → AI 少量解释 → 质量门控 → 渲染交付

---

## 1. 为什么必须做评测飞轮

完整攻略生成不是普通的“生成一段文案”。
它同时包含：
- 输入理解
- 路线规划
- 实体选择
- 预算取舍
- 节奏控制
- 交付结构
- 用户感知价值

只靠主观感觉去改 prompt、改规则、改模板，很快会出现 4 个问题：
1. 修好了一个问题，又把另一个问题弄坏
2. 你不知道问题来自输入理解、片段命中、规则过滤、软排序、装配还是 AI 文案
3. 代码 AI 和 API AI 各自忙，但没有共同成功标准
4. 线上翻车后，无法稳定复现、无法快速修正

所以必须把完整攻略生成做成一套：
**有用例、有 rubric、有回归、有失败归因、有版本记录、有准入门槛** 的飞轮系统。

---

## 2. 评测对象：到底测什么

不要只测“最终 PDF 好不好看”。
完整攻略生成至少拆成 4 层来测。

### 2.1 输入理解层
看用户输入是否被正确理解和标准化。

重点测：
- 目的地是否标准化正确
- 同行人识别是否正确
- 第一次去 / 去过几次 是否正确
- 多城顺玩 / 一地深玩 是否正确
- 预算偏向是否正确
- 固定时间约束是否被正确抽取

### 2.2 规划层
看系统是否选对了片段、路线和候选。

重点测：
- 是否优先命中合适的片段攻略
- 硬规则是否正确过滤掉不该进来的片段
- 软规则是否把更适合的片段排到前面
- 路线骨架是否顺
- 首日/末日是否和到离时间匹配

### 2.3 交付层
看生成的完整攻略是否符合产品定义。

重点测：
- 是否有总纲
- 每天是否有固定骨架
- 条件页是否触发正确
- 免费 / 标准 / 尊享的内容边界有没有串掉
- PDF/H5 渲染结构是否完整

### 2.4 体验层
看用户拿到后是否会觉得“值、顺、懂我”。

重点测：
- 是否像为这个人写的
- 是否少踩坑
- 是否更顺
- 是否值得付费
- 是否容易引发大改
- 是否有记忆点 / 分享感

## 3. 四方分工：code AI / API AI / 你 / 我

### 3.1 code AI 负责什么
- 写和改后端逻辑
- 写用例运行器
- 写 grader 执行脚本
- 接评测流水线
- 打日志、埋 trace、保版本
- 改数据结构、状态机、回归脚本

### 3.2 API AI 负责什么
- 跑生成
- 跑 grader
- 产出分数和评语
- 做 A/B 对比
- 按 rubrics 给失败样本分类

### 3.3 你负责什么
你不要参与所有细节。
你只参与 4 个最值钱的环节：
1. **定义成功标准**：什么才算“像人做的、值钱、值得买”
2. **挑首批 20 个用例方向**：哪些用例最像你真正想卖的单
3. **看失败样本**：每轮只看少量高价值失败样本，判断到底是哪里不对
4. **拍板 trade-off**：系统优先保什么，不优先保什么

### 3.4 我负责什么
- 帮你定义 eval 结构
- 帮你写 rubrics
- 帮你归纳失败模式
- 帮你判断应该改规则、改模板、改片段库还是改 prompt
- 帮你补回归用例和阶段性策略

## 4. 评测飞轮的全流程

### Step 1：建立首批核心用例集
先建 20 个，不求多，但要覆盖典型、高价值、复杂、边界、回归。

### Step 2：给每个用例写成功标准
不是只写输入输出，而是明确：
- 期望命中的片段
- 期望触发的硬规则
- 期望触发的软规则偏好
- 输出必须具备的结构
- 绝不能犯的错误

### Step 3：跑基线版本
用当前后端版本，先全量跑 20 个用例。
输出：
- 每个用例的生成结果
- grader 分数
- trace
- 命中片段
- 失败点

### Step 4：做失败归因
每个失败必须归因到某一层：
- 输入理解错
- 片段召回错
- 硬规则漏挡
- 软规则排错
- 装配错
- AI 文案解释错
- 渲染或交付错

### Step 5：每次只改一个变量
一轮只改：
- 一个规则
- 一个权重包
- 一个片段选择逻辑
- 一个 prompt
- 一个模板

### Step 6：回归运行
改完后，重新跑：
- 首批 20 个用例
- 所有 regression 用例

### Step 7：把线上翻车单沉淀成新用例
任何真实线上事故、用户强不满意、大改单，必须抽象成新 case 加进回归集。

## 5. 用例文件标准结构（统一格式）

每个 case 建议使用 YAML 或 JSON，至少包含以下字段：

```yaml
case_id: string
case_group: standard | high_value | constrained | edge | regression
title: string
summary: string
user_input:
  destination: []
  days: number
  group_type: string
  familiarity: string
  trip_style: string
  vibe: string
  budget_bias: string
  fixed_constraints: []
expected_profile:
  normalized_destination: []
  normalized_group: string
  normalized_style: string
  normalized_budget_bias: string
expected_hits:
  should_hit_fragments: []
  should_avoid_fragments: []
  should_use_theme_family: []
expected_hard_rules:
  must_trigger: []
  must_not_trigger: []
expected_output:
  required_sections: []
  required_days_shape: []
  must_have_features: []
  must_not_have_features: []
grader_targets:
  structure_min_score: number
  planning_min_score: number
  value_min_score: number
known_risks:
  - string
notes:
  - string
```
