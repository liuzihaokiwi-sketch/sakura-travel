# 城市圈构建总流程

> 项目从研究到上线的工作节奏。每步引用对应 SOP·不重复细节。
> 跨地区通用·关西样板间已跑通·其他圈按这个流程复用。

---

## 第一性原理

**先做成品·再倒推系统**——用户付 ¥298 买的不是数据库·是"专家替我想过的成品行程"。

所以流程不是"先建数据池再装配"·而是：

```
研究透这地区 → 做 1-2 个完整 demo → 拆 demo 反推数据需求 → 一城一城建数据 → 扩矩阵 → 上线 → 反馈循环
```

每一步都为"成品质量"服务。

---

## 上线前·9 步

### 1. 圈级研究（让 AI 先变这个地区的旅行专家）

**没研究透·后面所有工作都是 AI 想象**。

- 通读最佳攻略 / 顶级策展社方案（Black Tomato / Condé Nast）/ 长居作者博客
- 输出：`{圈}/这是什么.md`（含定位 / 7 化学反应等装配硬规 / 区域基因 / 餐位需求分级 / 区域类型分类）
- 输出：`{圈}/research/insights.md`（城市洞察 / 文化锚点 / 痛点预判 / 季节骨架）
- **采集前必读**——4 份数据规范都引用这份当采集前提

详见 [研究方法.md](上线前/研究方法.md)。

### 2. 跟 AI 一起做 1-2 个先行 demo 行程

**先做"完美的 1 个 7 日行程"**·把研究的结论实战检验。

- 选最主流 archetype（如关西"7 日京都深度·情侣"）
- 写完整行程：每天每段去哪 / 吃什么 / 住哪 / 什么时间 / 为什么这样
- 不是模板·是**成品体验**·作为后续所有工作的锚点

输出：`{圈}/research/demo_行程.md`（或类似命名）。

### 3. 拆 demo（手工·人 + AI 一起）

把 prose 翻译成 schema 需求·这是产品护城河。

- demo 里哪些可拆成可复用动线（→ 模板需求）
- 涉及哪些景点（→ entity 需求清单）
- 涉及哪些餐厅（→ 餐厅需求清单·按区域+档位）
- 涉及哪些酒店（→ 酒店需求清单·按区域+档位）
- 一次性作者判断 vs 可复用规则
- 时间/节奏规则（contract）

这一步不 SOP 化·是手工拆。产出的是 Step 4-5 的输入清单。

### 4. 先做 1 个城市完整数据

按拆 demo 反推的需求·先把**1 个城市**做完整。不要一开始铺所有城市。

按规范顺序：

1. **entity 库**（基础设施·所有件套都挂这）→ [entity规范.md](上线前/数据池构建/entity规范.md)
2. **餐厅** → [餐厅规范.md](上线前/数据池构建/餐厅规范.md)
3. **停留点** → [stops规范.md](上线前/数据池构建/stops规范.md)
4. **酒店** → [酒店规范.md](上线前/数据池构建/酒店规范.md)

4 件可以并行（餐厅/停留/酒店 都依赖 entity·entity 必先）。

**校验前提**：每份数据用对应 validator 校验·不通过不算完。

`scripts/validate_*.py` 五件套（hotels / restaurants / restaurants_pool / entity / template）都是**任意区圈通用**——只要新区圈根目录放好 `area_registry.json`·脚本会从输入路径自动向上找。开新城市圈不用改脚本。

### 5. 模板写作（这个城市的动线骨架）

把 demo 里这个城市部分拆成可复用的单日动线模板。

详见 [模板写作.md](上线前/模板写作.md)。

### 6. 装配联调（这个城市跑通）

用 demo 当 golden case·测装配引擎能不能用模板+数据装出原 demo 行程。

技术 gate：
- 装配跑通·没崩
- 字段齐·没缺
- 装出的行程跟 demo 8 成一致（剩 2 成是"装配优于人写"或"人写优于装配"的灰度）

详见 [装配设计.md](上线前/装配设计.md) + [联调测试.md](上线后/pilot反馈.md)。

### 7. 扩到下一个城市（重复 4-6）

第一个城市跑通后·第二个城市的工作量明显减少（规范熟了·SOP 熟了·常见坑都遇过）。

按"主场密集 → 日归延伸 → 温泉镇 → 单日景点"顺序扩。

### 8. 行程矩阵扩展

所有城市数据齐了·把单 demo 扩到完整覆盖。

**不是简单复制 N 份**·是**受控泛化**：
- archetype 维度（情侣/闺蜜/家庭/默认/特种兵）
- 季节维度（樱花/红叶/春秋/夏祭/冬蟹）
- 天数维度（3/5/7/9 日）

详见 [装配设计.md §二 行程矩阵覆盖](上线前/装配设计.md)。

### 9. 上线前最终验收

两个 gate 都过才能上线：

**技术 gate**：
- 装配跑通整圈·所有 archetype × 季节 × 天数能装出
- 字段齐·跑 validator 全 PASS
- 跨日去重 / 跨城互斥 / 节奏调整等装配硬规执行正确

**内容 gate**：
- 抽样审：随机抽 10 套装配出的行程·真专家点头率 ≥ 80%
- Alpha 自走：作者自己照行程走 1-2 段·抓"翻页时手感顺不顺"
- 孤立 entity 检测：哪些 entity 建了但没模板用·真孤立还是数据漏

详见 [联调测试.md](上线后/pilot反馈.md)。

## 上线后·3 步

### 10. Pilot 种子用户（10-20 本）

种子用户真实出行·**diary 跟踪**（每天 1 条小红书式反馈）。

**不只问"喜欢吗"**·结构化采集：
- 哪页被改写（用户自己改了什么）
- 哪页被忽略（跳过没看的）
- 哪个推荐踩雷（餐厅没去成 / 酒店不达预期）
- 哪段惊喜（超出预期）

详见 [上线后运营.md](上线后/内容运营客服.md)。

### 11. 内容运营 + 客服 FAQ

基于 pilot 真实反馈·不是凭空写：
- 小红书种草 SOP（基于真实用户照片+反馈）
- 客服 FAQ（基于 pilot 真实问题）
- 素材库沉淀（爆款洞察/本地冷知识）

### 12. 正式上线 + 反馈回流循环

定期回流：
- 用户反馈"哪页被改写最多" → 修整体行程或换默认推荐
- 关店/搬迁/新开店 → 数据池更新
- 季节性内容更新（樱花预测/红叶预测每年变）
- 每季度 audit 一次终选池

## 核心节奏：迭代·不是瀑布

```
Step 1-3 串行（研究 → demo → 拆解）
Step 4-6 一城一闭环（建数据 → 模板 → 联调）
Step 7 复制扩展
Step 8-9 整体收口
Step 10-12 上线后循环
```

**不是 Phase 0 → 1 → 2 → ... 全做完再下一个**·而是每个城市跑一个完整闭环·用第一个城市验证流程·后面城市批量复用。
