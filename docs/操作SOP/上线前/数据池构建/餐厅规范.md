## 一、7 条铁律（违反任何一条 = 全部不通过）

### 1. 字段不准膨胀

字段定稿见 §二。**不准私自加任何字段**：
- `walk_min_estimated` ❌（已禁用·历史漂移）
- `drive_only` / `tags` / `vibe` 等 AI 自创 ❌
- 发现需要新字段 → **停下问用户**，不要先加再说

校验脚本对字段做严格白名单，未知字段 = FAIL。

### 2. 数据不准造假

**查不到 ≠ 写假的**。三种处理：
- 模糊化（"小店"替"18 席"）
- 标"待核实"或写"营业时段以官网为准"
- 字段留空或 null

**禁止行为**：
- 全部餐厅写"通常 11:00-21:00"凑营业 ❌
- 评分/星级/年份猜 ❌
- 步行分钟全标 estimated ❌
- 跨店比较句"全 X 城最好的 Y" ❌

详见 [数据采集.md §一 第一红线](../数据采集.md)。

### 3. area 必须从白名单选

`area` 字段是全局事实（跨餐厅/酒店/模板/装配），**禁止自由命名**。

写 area 前必 grep `{圈}/area_registry.json`：
- 在白名单里 → 用 registry 标准名
- 不在 → **停下报告**，由用户决定加 registry 还是改用现有 area

校验脚本对 area 做严格白名单，不在 registry = FAIL。

### 4. entity 不为挂店服务

entity 必须**自己有意义**才加：
- 用户必去 / 模板动线锚点 / 文化地标
- 不是为了"店挂不上"而凑

**店挂不上就挂最近合理 entity，walk_min 大一点没关系**。挂不上任何合理 entity 的店 → 降 skeleton 或不进 full，**不为它加新 entity**。

### 5. walk_min 弹性

- **正餐**：用户为吃顿好的愿意走，15-20 分都接受
- **不为了"挂得近"伪造小数字**

### 6. depth 必须真实

- 地址查不到 / 营业全靠类目均值 = 自动降 skeleton
- full 必须三段式完整（系统字段 + note 全填 + 准 entity 挂载）
- 不强凑 full 撑数量

### 7. 主次分明（避免反复纠结）

- **A 类**（装配跑不动 / 字段缺失） → 停下报告
- **B 类**（用户体验硬伤：地址错 / 档位错 / 价格离谱） → 停下报告
- **小事**（评分缺 / 营业不全 / 文笔粗糙） → 标 verified 或留空，**继续做不停**

## 二、字段定稿（全球统一）

```json
{
  "id": "kyo_gion_hyotei",
  "area": "北区",
  "near_attractions": [{"entity_id": "nanzenji_temple", "walk_min": 3}],
  "tier": "showcase",
  "cuisine": ["怀石"],
  "recommended_meals": [{"meal": "lunch", "price_cny": [800, 1500]}],
  "reservation_difficulty": "must",
  "queue_level": "none",
  "closed_days": [3],
  "season_months": null,
  "depth": "full",
  "note": {
    "店名": "瓢亭 本店（ひょうてい / Hyotei）",
    "简介": "...",
    "亮点": ["米其林三星", "庭院茶室", "朝粥名物"],
    "招牌菜": ["朝粥（限早场 8:00-10:00）", "松花堂弁当（午）"],
    "地址": "京都市左京区南禅寺草川町 35（地铁东西线·蹴上站步行 8 分钟）",
    "营业": "11:00-14:00 / 17:00-21:00；周三定休",
    "预约": "电话预约（075-...），3 个月前 1 日 10:00 开放受理",
    "到店提醒": "建议 14:00 后到避高峰；红叶季全天无空位务必预约",
    "评分": 4.3
  }
}
```

### 系统字段（11 个）

| 字段 | 类型 | 必填 | 枚举值 / 格式 |
|---|---|---|---|
| `id` | string | ✅ | `{城市拼音简写}_{区域简写}_{店名拼音}` |
| `area` | string | ✅ | 动线级粒度（与 area_registry 对齐） |
| `near_attractions` | array | ✅（≥1） | `[{entity_id, walk_min}]`。entity 必须真实存在 |
| `tier` | string | ✅ | `showcase` / `high` / `mid` / `economy` |
| `cuisine` | array | ✅（≥1） | 字符串数组，主菜系第一。多菜系老铺如 `["稻荷寿司", "鳗鱼", "乌冬"]` |
| `recommended_meals` | array | ✅（≥1） | `[{meal, price_cny}]`。meal: `breakfast/lunch/dinner/cafe/bar`。**4 档单顿统一标准·午晚通用**：见 §三 |
| `reservation_difficulty` | string | 选填，缺省 `none` | `none` / `recommended`（不预约要排队≥1h）/ `must`（不预约进不去） |
| `queue_level` | string | 选填，缺省 `mild` | `none` / `mild` / `high` |
| `closed_days` | array \| null | ✅ | ISO 8601 数字数组：1=周一... 7=周日。三态：`[]`=已查实无定休 / `[3]`=周三定休 / `null`=未查到 |
| `season_months` | array/null | 选填 | `null` = 常年；`[5,6,7,8,9]` = 季节限定 |
| `depth` | string | 选填，缺省 `skeleton` | `skeleton` / `verified` / `full` |

### note 字段（9 个）

| key | 必填 | 说明 |
|---|---|---|
| `店名` | ✅ | 中文（本地语言假名 / 英文）。如 `瓢亭 本店（ひょうてい / Hyotei）` |
| `简介` | full 必填 | 80-150 字，"懂当地的朋友讲口吻" |
| `亮点` | full 必填 | 3-5 个短词标签 |
| `招牌菜` | full 必填 | 列表，含限定提示如"朝粥（限早场）" |
| `地址` | full 必填 | 含最近交通+步行分钟 |
| `营业` | full 必填 | 自然语言一句：开门时段+定休 |
| `预约` | full 必填 | 渠道+开放窗口。`reservation_difficulty=none` 写"无需预约" |
| `到店提醒` | 选填 | 错峰 / 旺季 / 不规则定休 / 外国客提示 |
| `评分` | 选填 | 数字（如本地权威评分平台分数）。查不到不写。地区特化平台见各地区"特化"文档 |

## 三、分类标准

### 3.1 跟停留池的边界

判断进 restaurants 还是 stops：
- **能解决一顿正餐** → restaurants（即使便宜也算）
- **纯咖啡甜品/购物/手作** → stops

### 3.2 tier 4 档（全球统一·按单顿人民币价格）

| tier | 中文 | 单顿 RMB | 典型场景 |
|---|---|---|---|
| `economy` | 街头地道 | 0-180 | 街食/快食/平民咖啡店简餐 |
| `mid` | 本地好店 | 150-500 | 本地中端日常餐厅 |
| `high` | 好好享受 | 450-1200 | 高端正餐/必比登/米其林一星午餐 |
| `showcase` | 顶级体验 | 1100-5000+ | 顶级餐厅/米其林二三星 |

**4 档区间是单顿统一标准·午晚通用**——不是"午餐档"和"晚餐档"两套尺子。一家店午餐落 mid、晚餐落 high 是常态（按实际单顿价填）。

价格区间精度：取整到 50 或 100 RMB·禁精确到个位/十位（如 `[2327, 2327]` ❌·`[2000, 2500]` ✅）。

### 3.3 cuisine 多菜系处理

老铺常多菜系。`cuisine` 是数组，**主菜系放第一**：

```json
"cuisine": ["稻荷寿司", "鳗鱼", "乌冬"]
```

装配按 `cuisine.includes("乌冬")` 跨菜系命中。

cuisine 标准枚举见 [字段权威.md](../../字段权威.md)。开新地区时新菜系先扩 SCHEMA 再写数据。

### 3.4 采集目标比例（采集时按这个铺池子）

按真实消费分布（中国游客赴日餐饮调研）·全城各 tier full 池目标比例：

| tier | 全城比例 | 装配视角 |
|---|---|---|
| `showcase` | 不进比例 | 每城 ≤ 1·全程 ≤ 2·装配硬顶·池子 5-8 家够用 |
| `high` | **25%** | 装配主力（用户表单"好好享受"） |
| `mid` | **50%** | 装配主力（用户表单"本地好店"·主流） |
| `economy` | **25%** | 装配主力（用户表单"街头地道"） |

**比例 = 采集目标·不是装配运行时硬规**。装配按字段查·比例只决定"采集时该铺多少各档"。

### 3.5 区域基因可变形（采集硬规）

**3.4 比例是基准·不是铁律**。区域类型决定基因偏向：

- **中转快食区**（如车站周边）：可偏 economy 60%·high 1-2 家足够
- **高端夜食区**（如商务区夜店街）：可偏 high+·showcase 1·mid 0·economy 0
- **温泉宿坊镇**：旅馆怀石主场·外食薄·mid+economy 主力·无 showcase
- **景区单菜系区**（如岚山豆腐/吉野柿叶寿司）：high/mid 两极·no mid

**判断逻辑**：先看该区域的"用户在这吃几顿"+"本地餐饮密度"，再决定偏向。
具体地区的区域归类 → 见各地区"特化"文档（如 `japan/kansai/这是什么.md` §餐厅基因）。

### 3.6 餐位需求分级（4 类区域·决定 full 池目标量）

不机械套"每区 high≥2/mid≥4/economy≥2"·按区域类型定 full 池目标：

| 区域类型 | 用户行为 | 单次行程餐位数 | full 池目标 |
|---|---|---|---|
| **主场密集区** | 住附近·多次返回吃饭 | 4-8 顿 | high 4-6 / mid 8-12 / economy 4-6 |
| **日归动线区** | 当天 1-2 顿 | 1-2 顿 | high 1-2 / mid 3-4 / economy 2-3 |
| **温泉宿坊镇** | 旅馆怀石主场·外食很少 | 0-1 顿 | mid 1-2 / economy 1 |
| **景点单日** | 园内/园外 1 顿 | 1 顿 | mid 2 / economy 2 |

**主场密集 vs 日归动线的判断**：用户是否会在这住 ≥1 晚？住 → 主场密集；只白天去 → 日归动线。

**采集硬规**：温泉宿坊镇 / 景点单日不强求 high·主场密集才需要 full pool 厚度。

### 3.7 努力成本（X 轴·硬比例）

> **枚举权威 → [字段权威.md §2.5 努力档枚举](../../项目核心/字段权威.md#25-努力档枚举餐厅酒店并列两类不互通d46)**·跨规范统一改名先改字段权威·餐厅 4 档 / 酒店 3 档不互通。

> **本段是 D46 立的核心规则**：除 tier × cuisine 之外·**每家店都要在「努力成本」这第二维上落位**·控制数据池整体可达性。

#### 4 档努力档定义

| 努力档 | 定义 | 字段判定（系统自动算） |
|---|---|---|
| `effortless` 零努力 | walk-in 即进·无排队·无预约 | reservation=`none` AND queue=`none/mild` |
| `low_effort` 低努力 | 当日/前 1 天可订·或排队 ≤30 分 | reservation=`recommended` OR queue=`mild/medium` |
| `medium_effort` 中努力 | 提前 1-2 周·或排队 30-60 分 | reservation=`must` AND 非常客制 |
| `hard_effort` 高努力 | 月初打电话·常客优先·正装·提前 1-3 个月 | reservation=`must` AND 备注含"常客优先/月初"等 |

**说明**：`努力档`不需要在 JSON 中加新字段·从现有 reservation_difficulty + queue_level + note.预约 自动推算。校验脚本聚合统计·不入库。

#### 数据池配比硬规（**最重要**）

| 努力档 | 占比目标 | 含义 |
|---|---|---|
| `effortless` 零努力 | **25-30%** | 性价比+招牌+本地国民食为主 |
| `low_effort` 低努力 | **50%（核心）** | 「准顶级但好订」型·数据池 50% 主力 |
| `medium_effort` 中努力 | **15-20%** | 高分名店·愿意提前订的客 |
| `hard_effort` 高努力 | **5-10%** | 顶级精品·showcase 冲顶用·**绝不是主力** |

**违反这条的处理**：
- `low_effort` < 40% → 数据池失衡·补池阶段必须**优先补 low_effort**·而不是补 hard_effort 顶级店凑评分
- `hard_effort` > 12% → **必须剔除**部分难订高分店·宁可数据池总数变少

#### 为什么这样设计（第一性原理）

> 用户付费买的是「**有人替我想过 + 敢对结果负责**」的确定感。
> 数据池里 50% 顶级难订店·实际推用户也用不上·**不是替用户想过·是把负担推给用户**。

50% 主力定位「准顶级但好订」是这个产品的本质：
- 评分顶级店是少数精品·让用户主动选择 8-10% 即可
- 50% 的店应该是「平日好订/排队 ≤30 分/价位明确·但味道离顶级一步之遥」
- 这才是日本本地人真正吃饭的主场·也才是「关西本地朋友会带客的店」

## 四、数据准入

> 理论锚点详见 [数据采集.md §三 三轴模型 / §五 三层池](../数据采集.md)。本段是 D46 大改后的版本：把「值的入口」（5 入口·Y 轴）和「努力成本」（X 轴·见 §3.7）正交·准入按矩阵命中。

### 4.1 「值」的 5 入口（Y 轴·OR 关系·任一命中即准入）

> **唯一总闸门**：「这家店能不能让用户在手账里翻到那一页时·至少有一个『值』的瞬间？」

| 入口 | 描述 | 客观信号 | 例 |
|---|---|---|---|
| **A. 味道 top** | 食物本身做到地区头部 | Tabelog ≥3.6 / 米其林 / 本地百名店奖项 | 鮨さえ喜·瓢亭·和久傳 |
| **B. 体验仪式** | 餐就是一场表演/出片/记忆点 | SNS 爆款 / 主题演出 / 重箱旅のしおり / 古民家改造 | ryu no hige·MITATE·川床料理 |
| **C. 本地独家** | 只有当地人/资深旅日才知道·游客难自找 | 本地 blog 反复 / 无中文攻略 / 隐藏入口 / 师承 | 大原松門·伏见大河乌冬·北区隐家 |
| **D. 招牌代表** | 「这道菜不在这家吃就白来此地」 | 名物発祥店 / 老铺 80+ 年 / 当地 DNA 食物 | 千とせ肉吸い·会津屋章鱼烧·551 蓬莱·かに道楽（蟹打卡） |
| **E. 景观/季节** | 风景或季节本身就是菜单 | 川床 / 红叶料亭 / 望景餐厅 | 貴船ひろや·廣川鳗鱼·大阪城望景 |

**判断逻辑**：5 入口 OR 关系·**任一强命中**即过 Y 轴。命中弱时再看是否有第二条入口辅助命中。

### 4.2 Tabelog 评分用法（**关键澄清**）

**Tabelog 评分不是入口·是辅助决策**。三种用法：

1. **A 入口的客观信号**：≥3.6 是 A 强命中·3.5-3.6 是 A 弱命中
2. **托底验证**：当 BCDE 命中但用户怀疑「这家真好吗」时·评分 ≥3.4 给托底
3. **价格反差放大**：当一家店命中 5 入口之一、且价位明显低于该入口的常态价位时·提升优先级（如朝食独立池瓢亭朝粥 ¥6,600 是 A+E 入口本来就强·但同体验在京都通常 ¥15,000+ → 性价比形成额外加分）

**红线**：
- 5 入口都不命中、Tabelog 再高也剔（避免「评分高但去了无趣」的店挤名额）
- 5 入口强命中、Tabelog 3.3-3.4 也收（避免「把活通道掐死」的盲点）

### 4.3 准入剔除（任一命中即剔）

| 剔除条件 | 原因 |
|---|---|
| **N1. 5 入口全不命中** | 没有任何「值」的支点·留着浪费数据池权重 |
| **N2. 信息不真实** | 数据来自推断而非真搜·违反第一红线（[数据采集.md §一](../数据采集.md)） |
| **N3. 区域不在动线** | 装配引擎用不到·占数据池权重 |
| **N4. 同店同 area 同特征重复** | 比如都在心斋桥的 5 家差不多 economy 拉面·留 1-2 家代表 |
| **N5. 关店/搬迁/装修中** | 立即剔除或标注待复访 |

### 4.4 三轴可信度（数据来源验证）

每个 Full 必须**三轴中至少两轴有真实证据**：

| 轴 | 餐厅信号源（通用） |
|---|---|
| **quality** | 本地权威评分平台 / 米其林 / 百名店类奖项（具体平台见各地区"特化"文档） |
| **traveler_fit** | 中国游客 OTA 评分（携程/大众点评） + 小红书 collects > likes / 中文深度博客 |
| **execution** | 营业稳定 / 预约难度可承受 / 排队 ≤90 分钟 / 定休清晰 |

**三轴规则**（4 档可信度，详见数据采集.md §二）：
- 三轴全过 → `verified` 入 full
- 两轴过 → `cross_checked` 入 full
- 一轴过 → `single_source`·标注后入 full
- 无任何真实证据 → `ai_generated`·**禁止上生产**

### 4.5 准入决策树（实操）

收一家店时按这个顺序问：

```
Step 1：5 入口 Y 轴 → 任一命中？
  ❌ 全不命中 → 剔（N1）
  ✅ ≥1 命中 → 进 Step 2

Step 2：努力成本 X 轴落位
  → 算 effortless / low_effort / medium_effort / hard_effort
  → 检查当前数据池该 cuisine 中此努力档是否已饱和（hard_effort 上限 12%）
  ❌ 已饱和 → 剔（除非比现有更优）
  ✅ 未饱和 → 进 Step 3

Step 3：数据真实性三轴
  → quality + traveler_fit + execution 至少 2 轴有真凭据
  ❌ 不到 2 轴 → 标 single_source 或不收
  ✅ ≥2 轴 → 收·depth=full
```

### 4.8 数据收集 5 通道（按池子配比目标·D46 改）

旧 D45 收集只有「Tabelog 高分自动挑」一条主路·导致演出体验店（B 入口）/ 准顶级好订店（A × low_effort）漏掉。新版本 5 通道按目标配比并行：

#### 通道 1：50% 主力·「准顶级但好订」（核心通道）
- **目标**：A × low_effort 主力·占数据池 50%
- **搜法**：
  - 日文「コスパ最強」「平日空いてる」「予約取りやすい」专题
  - 小红书「平价 omakase」「不排队」「本地人推荐」专题
  - Tabelog 3.5-3.7 + 营业时间核实平日好订
- **过滤**：排除「常客优先」「月初抢」「正装必需」类
- **例**：いわ志（福岛 ¥2,750/20 贯·平日好订）

#### 通道 2：零努力·性价比国民食（次主力）
- **目标**：D × effortless·占 25-30%
- **搜法**：
  - 食べログ 100 名店中无预约 / 排队 ≤30 分
  - 关西おでかけ手帖「庶民派」专题
- **例**：千とせ 肉吸い ¥850·551 蓬莱 ¥210/個

#### 通道 3：D 入口·关西名物代表（招牌覆盖）
- **目标**：每个名物 ≥1 家发祥店 + 1 家性价比代表
- **覆盖范围**：章鱼烧/大阪烧/拉面/うどん/串炸/红豆包/抹茶/松叶蟹/和牛/汤豆腐
- **例**：会津屋（章鱼烧发祥）·美津の（大阪烧老铺）·てんま 中村（鳗鱼老铺）

#### 通道 4：B/E 入口·演出+景观（D45 漏的盲区）
- **目标**：B 体验仪式 + E 景观季节·占 8-12%
- **搜法**：
  - 小红书「出片」「氛围感」「网红餐厅」收藏 ≥ 1000
  - 川床 / 古民家 / 主题演出餐厅
- **例**：ryu no hige（旅のしおり演出）·MITATE 集团·貴船 ひろや（夏川床）

#### 通道 5：精品冲顶·8-12% 上限严控
- **目标**：A × hard_effort showcase·占数据池 ≤ 12%
- **搜法**：米其林 ≥1 星·Tabelog ≥3.8·常客制
- **比例硬规**：超过 12% 必剔
- **例**：瓢亭 本店·鮨さえ喜·北新地 のぐち

### 4.9 诚实标注定位（D46 加）

**不假装**：打卡店写「打卡·味道一般但氛围在」·性价比店写「性价比·不要期待顶级」·顶级店写「顶级·难订/正装」。

例：
- かに道楽 道顿堀本店：标 D 入口·simple 标注「关西=蟹打卡地标·味道一般但 1962 年大蟹招牌是 DNA」
- いわ志 本店：标 A × low_effort·标注「Tabelog 3.58·¥2,750 / 20 贯·平日好订·性价比寿司天花板」
- 鮨さえ喜：标 A × hard_effort·标注「3.85+·常客优先·新客 3-6 个月预约·熟人介绍最稳」

**禁忌**：把「打卡店」吹成「顶级」让用户去了失望——这正是「敢对结果负责」的反面。

### 4.12 漏店反验（每次扩池后必跑·**核心命题**）

> **核心命题**：「我能不能搜到那家漏的店？」如果答否·数据池就是漏的。

扩池完成后**反向自验**：

1. 用户最常问的 N 类菜（地区特化文档定 N=5-8）各跑一次「**<城市> <不排队词> <性价比词> <菜> 推荐**」
2. 看 top 5 结果是否在数据池里
3. 不在 → 是真漏·按通道反推补；在 → OK

#### 严禁的反模式（真实犯过的错·见 §六.8）

- ❌ 「这家店本地人不知道·不需要收」——游客想吃但本地人不去的招牌打卡店有 D 入口意义·收并诚实标注「打卡为主」
- ❌ 「评分不到阈值·不收」——演出 fusion 评分常在中高区段但 SNS 爆款·B 入口强命中应收
- ❌ 「连锁店不收」——好品牌的旗舰店味道是真好·按 5 入口判断·不一刀切
- ❌ 「单通道收集」——只用单一权威评分自动挑·B/C/E 入口全漏

## 六、历史教训（通用·关西作 case）

### 6.1 area 命名漂移（已解决）

D43 前 area 自由字符串，导致同一动线区有多个名字（关西"三宮"/"旧居留地"/"神户三宮-旧居留地"3 个桶散在多处）。
**对策**：area_registry.json 白名单 + validator 强制校验。**新地区开圈时第一步建 area_registry**。

### 6.2 单一评分门槛漂移（已解决）

之前仅按本地权威评分卡线，导致"某区某档没合格"——其实有 traveler_fit 强的本地店被卡掉。
**对策**：准入改三轴模型，本地评分是 quality 轴硬指标但不是唯一通道。

### 6.3 价格区间漂移（已解决）

之前把外币换算成精确人民币（如 [2327, 2327]），档位无意义。
**对策**：4 档单顿统一区间·精度取整到 50 或 100·禁 min=max。

### 6.4 walk_min_estimated 字段膨胀（已禁用）

之前为标"步行分钟估算"加了 `walk_min_estimated: bool`，结果 AI 全部标 true 等于没标。
**对策**：字段删除，walk_min 一律按真实查得填。

### 6.5 营业类目均值漂移（已解决）

之前 AI 把不同店都写"通常 11:00-21:00"凑营业。
**对策**：营业查不到必须写"营业时段以官网为准"或具体真实时段，禁用类目均值。

### 6.6 entity 挂载粒度太粗（已解决）

之前关西大阪 13 家全挂"难波站"一个站名 entity，装配按景点 filter 时漂移。
**对策**：必须挂具体地标级 entity（如美国村/心斋桥筋/堀江/日本桥），不够 → 停下补 entity 不将就。

### 6.7 tier 错档（注意点）

容易把"高分但价格平价"的店硬挂 high/showcase（如关西京都站京林泉 ¥150 lunch 标 showcase）。
**对策**：tier 必须真实反映 price 档位，不为"显得高级"挂错档。

### 6.8 D46：单通道收集 + 二元剔除盲点（已修复）

D45 收集只用「Tabelog 高分自动挑」一条通道 + 「连锁化无特色一刀切剔除」二元规则·导致：
- 「演出体验店」（如 ryu no hige·MITATE 集团）漏掉——非传统主轴
- 「准顶级但好订」店缺失——`hard_effort` 顶级店挤占主力位置（数据池 hard_effort 一度 > 30%·`low_effort` < 30%）
- 优秀连锁高端店（叙々苑游玄亭等）被「连锁=不收」一刀切误剔

**对策**：
1. **5 入口（Y 轴）+ 4 努力档（X 轴）二维准入**——见 §4.1 / §3.7
2. **5 通道收集**（替代单通道）——见 §4.8
3. **删除「连锁化无特色」二元剔除**·改为「5 入口都不命中」实质判断（见 §4.3 N1）
4. **类目最低覆盖**——避免「关西吃 X 没推荐」空洞（见 §4.6）

**第一性原理**（CLAUDE.md）：
> 用户付费买的是「**有人替我想过 + 敢对结果负责**」的确定感。
> 数据池 50% 顶级难订·实际推用户也用不上·**不是替用户想过·是把负担推给用户**。
