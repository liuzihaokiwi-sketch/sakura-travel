> ⚠️ **采集前提**：本地区"圈级研究"已完成（含主流停留点类型/区域分布），输出在 `{圈}/这是什么.md`。研究没做完不能开始采集。

> 全球通用·跨地区复用。停留池 = 动线插件（顺路歇脚 / 顺路逛），不强制·走 near_attractions 就近匹配。
> 跟餐厅 / entity / 酒店三类规范并列。
> 地区特化 → `{地区}/docs/stops特化.md`

## 一、6 条铁律（违反任何一条 = 全部不通过）

### 1. 字段不准膨胀

字段定稿见 §二。**不准私自加任何字段**：
- `walk_min_estimated` ❌（已禁用·历史漂移）
- `drive_only` / `tags` / `vibe` 等 AI 自创 ❌
- 发现需要新字段 → **停下问用户**

校验脚本对字段做严格白名单·未知字段 = FAIL。

### 2. 数据不准造假

**查不到 ≠ 写假的**：
- 模糊化（"小店"替"30 平米"）
- 标"营业时段以官网为准"
- 字段留空或 null

**禁止行为**：
- 全部停留点写"通常 11:00-21:00"凑营业 ❌
- 步行分钟全标 estimated ❌
- 跨店比较句 ❌

详见 [数据采集.md §一 第一红线](../数据采集.md)。

### 3. area 必须从白名单选

跟餐厅/entity/酒店共用同一 `area_registry.json`·禁止自由命名。
不在 registry → 停下报告。

### 4. entity 不为挂店服务

挂不上合理 entity → 降 skeleton 或不收·不为它加新 entity。

### 5. depth 必须真实

- 地址查不到 / 营业类目均值 = 自动降 skeleton
- full 必须三段式完整（系统字段 + note 全填 + 准 entity 挂载）

### 6. 主次分明

- **A 类**（装配跑不动 / 字段缺失） → 停下报告
- **B 类**（用户体验硬伤：地址错 / 类型错 / 与餐厅混淆） → 停下报告
- **小事**（营业不全 / 文笔粗糙 / 招牌品类细节） → 标 verified 或留空·**继续做不停**

## 二、字段定稿

```json
{
  "id": "kyo_arashiyama_arabica",
  "area": "岚山",
  "near_attractions": [{"entity_id": "togetsukyo_bridge", "walk_min": 2}],
  "type": "咖啡",
  "depth": "full",
  "note": {
    "店名": "% Arabica 京都 嵐山",
    "简介": "渡月桥北詰的爆款拍照咖啡店·世界各地分店里中国游客密度最高的一家。京都拿铁是招牌——浓缩奶泡上点抹茶粉·杯子上是樱花季限定花纹。20 平米小店·外带为主·店外渡月桥背景天然出片。",
    "亮点": ["京都拿铁", "渡月桥拍照机位"],
    "地址": "京都市右京区嵯峨天龙寺芒之马场町 3-47（渡月桥北詰即到）",
    "营业": "9:00-18:00；无定休；周末高峰排队 30-40 分钟"
  }
}
```

### 系统字段（5 个）

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | string | ✅ | `{城市拼音简写}_{区域简写}_{店名拼音}` |
| `area` | string | ✅ | 动线级粒度·跟 area_registry 对齐 |
| `near_attractions` | array | ✅（≥1） | `[{entity_id, walk_min}]`·entity 必须真实存在 |
| `type` | string | ✅ | 开放枚举（见 §三） |
| `depth` | string | 选填，缺省 `skeleton` | `skeleton` / `verified` / `full` |

### note 字段（5 个）

| key | 必填 | 说明 |
|---|---|---|
| `店名` | ✅ | 中文（本地语言/英文） |
| `简介` | full 必填 | 60-120 字·"懂当地的朋友讲口吻"·讲招牌+氛围+独特点 |
| `亮点` | full 必填 | 2-4 个短词标签 |
| `地址` | full 必填 | 含最近交通+步行分钟 |
| `营业` | full 必填 | 一句话含开门+定休+排队/季节/特殊提醒·所有时间敏感信息都在这一句 |

## 三、分类标准

### 3.1 跟餐厅的边界

- **能解决一顿正餐** → restaurants（即使便宜也算）
- **纯咖啡甜品/购物/手作** → stops

边界 case：
- 仅咖啡/品酒/小吃·无正餐 → stops
- 酒馆居酒屋·能正餐 → restaurants
- 书+咖啡+杂货复合空间 → stops（type=书店咖啡）
- 咖啡馆有 2F 正餐午市·能正餐 → restaurants

### 3.2 type 开放枚举

**吃喝类**：咖啡 / 甜品 / 抹茶 / 和菓子 / 茶寮 / 喫茶店 / 刨冰 / 日本酒
**购物类**：古着 / 古书 / 书店 / 书店咖啡 / 杂货 / 设计杂货 / 文具 / 当地土特产
**体验类**：古道具 / 工艺品 / 手工艺 / 唱片 / 御宅店 / 道具屋

**不强求枚举**——AI 写到合适词就行（如新品类"文房具"/"陶器"），但同一品类必须用同一词避免漂移。

### 3.3 采集原则：覆盖动线即可·不堆量

stops 跟餐厅/酒店不一样——**不强求比例·不卡数量**。原则：
- 主要动线区有 stops 选项即可（每动线 3-5 家）
- 区域特色 type 优先（关西西陣有古着店·堺有刀具店）
- 不为补 type 凑数（伏见没有古道具店就不收）

## 四、数据准入

### 4.1 三轴模型应用（停留点）

| 轴 | 信号源 |
|---|---|
| **quality** | 媒体推荐 / 行业奖 / 老铺背书 / 当地朋友会专门带去 |
| **traveler_fit** | 携程评分+评论 / 小红书 collects > likes |
| **execution** | 营业稳定 / 排队 ≤90 分钟 / 不需预约 |

详见 [数据采集.md §三 三轴模型](../数据采集.md)。

### 4.2 数据源

参考各地区"特化"文档·按地区找对应权威平台。

通用思路：
- **P0**：Google Maps + 官网（execution）
- **P1**：小红书 collects > likes（traveler_fit 主信号）/ 携程 / 大众点评
- **P2**：品类专题站（古着/书店/咖啡 各自的专门媒体）

### 4.3 负向信号（任一触发即降 skeleton 或不收）

| 信号 | 处理 |
|---|---|
| 公认游客陷阱（本地人避开） | 不收 |
| 连锁化无特色（如全国连锁咖啡店） | 不收 |
| 排队 >90 分钟且同区有替代 | 降 skeleton |
| 小红书爆火但 Google <3.8 / 评价两极 | 降 skeleton |
| 价格与体验不匹配 | 降 skeleton |
| 关店/搬迁/装修中 | 立即降 skeleton 标注 |

## 五、校验

校验脚本：`scripts/validate_restaurants.py`（同时校验 restaurants + stops）

校验规则：
- 系统字段白名单（5 个）+ note 字段白名单（5 个）→ 未知字段 FAIL
- area 字段必须在 `{圈}/area_registry.json` → 不在 FAIL
- near_attractions.entity_id 必须在 `{圈}/entities/*.json` → 不在 FAIL
- 必填字段缺 → FAIL
- depth=full 但 note 字段缺 → FAIL（自动建议降 skeleton）

跑法：`python scripts/validate_restaurants.py`·粘**完整输出**到交付消息。

## 六、历史教训（通用·关西作 case）

### 6.1 area 命名漂移（已解决）

D43 前关西 area 自由字符串·导致同一动线区有多个名字。
**对策**：area_registry.json 白名单 + validator 强制校验。

### 6.2 walk_min_estimated 字段膨胀（已禁用）

之前为标"步行分钟估算"加了 `walk_min_estimated: bool`·结果 AI 全部标 true 等于没标。
**对策**：字段删除·walk_min 一律按真实查得填·查不到 → 写真实估算值。

### 6.3 营业类目均值漂移（已解决）

之前 AI 把不同店都写"通常 11:00-21:00"凑营业。
**对策**：营业查不到必须写"营业时段以官网为准"或具体真实时段·禁用类目均值。

### 6.4 跟餐厅混淆（注意点）

边界 case 如"咖啡+正餐"复合店容易塞错池。
**对策**：判断按"能否解决一顿正餐"·能 → restaurants·不能 → stops。
