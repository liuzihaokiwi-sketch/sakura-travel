# 搜索方法（全球通用·跨数据类型）

> 跨数据类型通用·适用于酒店 / 餐厅 / entity / stops 4 类数据池构建。
> 跟 4 类规范的关系：**搜索方法 = 起点（怎么拿到候选名字）·4 类规范 = 终点（候选过准入）**·两边解耦。
> 地区特化关键词词典 → `{地区}/docs/{类型}特化.md`（如 `japan/docs/餐厅特化.md` §关键词词典）。

---

## 一、第一性原理

### 1.1 我们要找什么

**宝藏候选**——满足以下任一：
- **专家会专程推荐**的店/景（不是榜单常客·是「懂行人才知道」的）
- **懂行的本地朋友会带客**的地方（本地 DNA·非游客陷阱）
- **路径 B 精准价值**：对明确人群是必去级·对其他人跳过即可

反例（**不是**宝藏）：评分高但去了无趣 / 小红书爆款但灵魂空 / 大家都推但没差异化。

### 1.2 为什么搜索引擎找不到（4 个盲点）

搜索引擎/AI 自由搜「[城市] X 推荐」时·结构性偏向：

| 盲点 | 表现 |
|---|---|
| **爆款偏向** | 出来都是高曝光物业·小众老铺/隐家被淹没 |
| **中文优先** | 中文圈覆盖少的（日本宿坊/老舗旅館/本地老店）漏掉 |
| **标准化偏向** | 国际连锁/大集团反复出现·非标本/独立物业排不上 |
| **类目空洞** | 某些类目（宿坊/手作工坊/独立咖啡）整体覆盖薄 |

**反例案例**——智积院会馆（京都宿坊）：
- 长谷川等伯国宝障壁画 + 利休好庭园 + 平价 ¥600/晚（一泊二食）
- Yahoo 评分 4.50 / 13 评论·携程几乎无覆盖
- 自由搜「京都宿坊」首先出**高野山**（宿坊代名词）·智积院被淹没
- 「专家知道、用户搜不到、AI 也搜不到」——按搜索路径**永远收不到**

### 1.3 我们的解法

> **数据池以「专家清单穷举」为骨架·搜索结果只做补漏。**
> 反过来 = 做了一份小红书/爆款汇总·违反「专家替你想过」第一性原理。

3 阶段流程：
1. **阶段 1：穷举种子库**（不靠搜索·靠 6 类种子穷举所有候选名字）
2. **阶段 2：搜索补漏**（增量补阶段 1 漏掉的本地爆款）
3. **阶段 3：负向交叉**（看不到的反而珍贵·主动找漏）

## 二、宝藏 6 类种子（跨类型通用框架）

> 6 类种子是跨数据类型的**通用骨架**——酒店/餐厅/entity/stops 各自落实方式不同·框架相同。
> 各类数据的具体落地映射 → §四。

### 2.1 大集团/品牌目录穷举

**做法**：从大集团官网下载该地区物业完整目录·穷举·**不靠搜索**。

| 适用类型 | 例 |
|---|---|
| 酒店 | 万豪/凯悦/IHG/Hilton/Mitsui/Hoshino/Prince 关西全物业·含副牌（OMO/Aloft/Courtyard） |
| 餐厅 | 米其林集团旗下品牌（Joël Robuchon 系）/ 老铺连锁（叙々苑/つきじ喜代村）/ MITATE 集团 |
| entity | 大型寺社联营（古都京都 17 处世界遗产）/ 博物馆运营机构（MIHO/京都国立系列） |
| stops | 连锁咖啡集团（% Arabica/Blue Bottle 关西全店）/ 出版社直营书店（蔦屋/丸善） |

**为什么这是地基**：大集团官方目录 = 完整 + 权威 + 标准化体验·首次客的 worry_free 主力。

### 2.2 权威认证清单

**做法**：从认证机构官网下载该地区**完整名单**·全部进种子。

| 适用类型 | 来源 |
|---|---|
| 酒店 | MICHELIN Keys / Forbes Travel Guide 五星 / 一休 100 选 / プロが選ぶ日本のホテル・旅館 100 選 |
| 餐厅 | 米其林星付（含必比登）/ 食べログ 100 名店 / WEST 100 名店 / Asia 50 Best |
| entity | UNESCO 世界遗产 / 国宝/重要文化财 / 米其林绿星景点 / 国家公园 |
| stops | 老铺 100 选 / 全国茶屋指南 / 古书店组合名簿 |

**关键**：认证清单**全部入种子库**·进准入决策时再 filter（不是搜索时 filter）。

### 2.3 协会/组合名簿

**做法**：当地行业协会（旅館組合/老舗会/伝統工芸協会）的会员名簿·**穷举**。

| 适用类型 | 例 |
|---|---|
| 酒店 | 有马温泉旅館協同組合 / 城崎温泉旅館協同組合 / 高野山宿坊組合 / 京都老舗旅館会 |
| 餐厅 | 京都料理組合 / 老舗のれん会 / 京都中央卸売市場関連 / 大阪寿司商組合 |
| entity | 京都仏教会加盟寺院 / 神社本庁加盟 / 国営公園協議会 |
| stops | 京都伝統工芸会 / 全国和菓子組合 / 古本協同組合 |

**为什么重要**：协会名簿 = 当地人公认的「正经在做这件事的人」·**淘汰非正规店家**·游客陷阱大多不在协会内。

### 2.4 类型专题穷举

**做法**：按**类型**穷举该类型在该地区的**所有可能候选**·不限于评分。

| 适用类型 | 例 |
|---|---|
| 酒店 | 京都市内宿坊穷举（智积院/妙心寺/智恩院/南禅寺等所有可住宿寺院·10-15 家） / 京都町家公司化运营商目录（Nazuna/葵 KYOTO STAY/Hotel Kanra/庵/京小宿 × 各物业·30-50 家） |
| 餐厅 | 京都怀石老铺全清单（菊乃井系/瓢亭/未在/和久傳·穷举） / 大阪鳗鱼老铺穷举（江户风+关西风） / 川床全店清单（夏季限定·鸭川/贵船/嵐山） |
| entity | 京都市内非热门寺院穷举（大徳寺塔頭/妙心寺塔頭等小院） / 隐藏庭园穷举（重森三玲作品全清单） |
| stops | 京都古书店穷举（三月書房/萩书房等 50+ 家） / 京都喫茶老铺穷举（イノダ/前田/六曜社等） |

**为什么这是宝藏入口**：搜索引擎按「人气」排序·类型专题穷举按「类型完整度」排序——**两个维度不同**·能挖出搜索看不到的小众。

### 2.5 锚点反向（entity 周边 / 区域内）

**做法**：从已有 entity（景点）反向挂酒店/餐厅/stops·按**地理范围**穷举·而不是按搜索词。

| 适用类型 | 做法 |
|---|---|
| 酒店 | 每个 S/A 级 entity 周边 1-3 km 查现存酒店·防漏（如清水寺周边 / 锦市场周边） |
| 餐厅 | entity 步行 5 分内餐厅穷举·按 area 而不是按菜系搜 |
| stops | entity 出口 100m 内顺路店穷举（茶屋/伴手礼/咖啡） |
| entity | 已有 S 级 entity 步行 15 分内的「次级 entity」穷举（防主景点附近漏宝藏） |

**为什么补漏强**：地理穷举绕过搜索词偏向·能挖出「在动线上但不出名」的店。

### 2.6 用户/客服主动推荐

**做法**：用户/客服反馈进种子池·**直接信任专家锚点**·不靠机械门槛筛。

例：
- 用户「智积院不错」→ 直接进种子（你说的就算）
- 客服「上次客户回来夸 X 店」→ 进种子
- 你/团队成员/真去过的朋友推荐 → 进种子

**关键**：种子库要支持「**人推荐就进**」这个入口·机械门槛只用于阶段 2 的搜索补漏·不用于阶段 1 的种子构建。

## 三、3 阶段流程

### 3.1 阶段 1：穷举种子库（**先做完才能进阶段 2**）

按 6 类种子穷举·**只列名字·不查详情**·一次性产出种子表。

种子表格式：
```
| 候选名 | 来源种子类型 | 备注 |
|---|---|---|
| 智积院会馆 | 2.4 类型专题（京都市内宿坊穷举） | 国宝障壁画 + 平价 |
| Hyatt Regency Kyoto | 2.1 大集团（凯悦关西） | 七条 |
| Aman 京都 | 2.1 大集团 + 2.2 权威（MICHELIN Keys） | 顶奢档 |
```

**实操**：
1. 列**所有 6 类种子**（按数据类型对应 §四 映射表）
2. 每类种子穷举完候选名字·不漏
3. 去重合并（同一物业可能多源命中·正常）
4. 阶段 1 完成 = **种子表 ready**·进阶段 2

**判定**：阶段 1 没穷举完不进阶段 2——种子是地基·搜索是补丁。

### 3.2 阶段 2：搜索补漏

种子表 ready 之后·搜索的角色变成「**补漏**」而不是「主路径」：

| 通道 | 用途 |
|---|---|
| 主源 OTA 评分排序 | 补「专家清单漏掉的本地爆款」 |
| 小红书 collects 排序 | 补「中国客真实热度但不在协会/认证清单」 |
| 关键词搜索 | 补「新开业不到 1 年」「季节限定」「类目空洞」 |
| 实测博客 / 长文 | 补「深度信息但不在权威清单」 |

**关键词词典见地区特化文档**（如 `japan/docs/餐厅特化.md` §一关键词词典）。

**比例信号**：阶段 2 命中候选 / 阶段 1 候选 = 健康比例约 **10-20%**。
- 比例 > 30% → 阶段 1 种子库不够穷举·补 §二某类种子
- 比例 < 5% → 可能阶段 2 标准过严或地区已饱和

### 3.3 阶段 3：负向交叉（看不到的反而珍贵）

主动找漏：

| 检查 | 做法 |
|---|---|
| **类目空洞** | 「[地区]吃 X 推哪」「[地区]住 X 推哪」回不出 ≥3 家不同档位 → 该类目漏 → 重穷举 §2.4 |
| **区域空洞** | 某主住宿区某档位 0 家 → 漏 → 重穷举 §2.5 |
| **季节空洞** | 樱花季/红叶季无对应物业 → 漏 → 季节专题补 |
| **人群空洞** | 路径 B 某精准人群（艺术爱好者/亲子/特种兵）无对应 → 漏 |

**为什么 3 阶段必做**：阶段 1 + 2 完成后·主动反问「**我可能漏了什么**」·才能挖出真正的宝藏盲区。

## 五、宝藏判断（不靠搜索结果·靠人的判断）

种子表上的候选过准入时·除了 4 类规范的客观信号·**还要问 3 个主观判断锚**：

| 判断锚 | 含义 |
|---|---|
| **专家会专程推荐吗** | 真正的旅行顾问/本地通会主动带客来吗·不只是「评分高所以推」 |
| **本地朋友会带客吗** | 本地人请远方朋友吃饭/住宿会选这家吗·过滤纯游客陷阱 |
| **N 个独立信号呼应吗** | 至少 2 个**独立来源**（如协会名簿 + 实测博客）说同一件好事·避免单一来源放大 |

**禁忌**：
- 把「评分高」当「值得去」（评分高 ≠ 专家会推）
- 把「小红书爆款」当「本地认可」（爆款 ≠ 本地 DNA）
- 把「单一权威信号」当全部（米其林/Tabelog 也会漏宝藏）

## 六、案例库

### 智积院会馆（酒店 / 宿坊 / 路径 B 精准）

- **怎么找到**：§2.4 类型专题（京都市内宿坊穷举）+ §2.6 用户主动推荐
- **搜索引擎为啥找不到**：搜「京都宿坊」首先出高野山·智积院被淹没；中文圈覆盖少
- **宝藏特征**：长谷川等伯国宝障壁画 + 利休好庭园 + 平价 ¥600/晚 + 名勝庭園貸切案内
- **路径 B 一句话**：「佛教文化+国宝艺术爱好者·京都唯一可住进等伯国宝障壁画寺院的宿坊」

### ryu no hige（餐厅 / B 入口·体验仪式）

- **怎么找到**：§2.6 用户推荐 + 小红书 collects 排序补漏
- **搜索引擎为啥找不到**：「演出体验店」非传统美食轴·Tabelog 不强·标准搜索路径漏
- **宝藏特征**：旅のしおり 演出 + 重箱呈现 + 出片
- **B 入口判定**：体验仪式强命中 = 餐就是表演

### 法隆寺（entity / §2.5 锚点反向修复）

- **历史问题**：D40 时期套用 nara_park area·实际在斑鸠町·12 分钟车程外·属奈良西线动线
- **怎么修复**：§2.5 锚点反向——按地理穷举奈良西线 entity·法隆寺/唐招提/药师独立成动线
- **宝藏教训**：寺名带某区不代表就属于该区·area 是物理位置不是就近联想
