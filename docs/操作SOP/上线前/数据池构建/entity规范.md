## 一、6 条铁律（违反任何一条 = 全部不通过）

### 1. entity 不为挂店服务（核心铁律）

entity 必须**自己有意义**才加：
- 用户必去 / 模板动线锚点 / 文化地标
- 不是为了"店挂不上"而凑

**店挂不上就挂最近合理 entity·walk_min 大一点没关系**。挂不上任何合理 entity 的店 → 降 skeleton 或不收·**不为它加新 entity**。

历史教训：D40 时期为了让店家"挂得近"凑了 4 个大阪 entity（堀江/日本桥/美国村/心斋桥筋）·每个用户用到都正好是动线锚点 → 这种该加；但若是为了挂某家路边咖啡而加 → 拒绝。

### 2. 字段不准膨胀

字段定稿见 §二。**不准私自加任何字段**。

D43 已禁字段：name_zh/name_ja/name_en（合并 note.店名）/ coordinates / official_url / short_desc / admission / opening_hours / closed_days / access / photo_ok / tripod_allowed / reservation_required / seasonal_notes / notes / is_public_path / unesco / address_ja / phone / reservation_url / _alias / 8 个一次性字段。

发现需要新字段 → **停下问用户**。

### 3. 数据不准造假

- 推断 ≠ 事实（米其林星级年份不猜·营业时段不编）
- 查不到 → 模糊化 / 标"待核实" / 字段留空
- 不写跨景点比较句（"全 X 城最美"）
- 数据来源必须真访问过·不是 AI 推断

### 4. area 必须从白名单选

跟餐厅/stops/酒店共用同一 `area_registry.json`。entity 的 area 是**该景点物理所在区域**（动线粒度）·不是城市级粗粒度。

**核心规则**：area 不在 registry → **停下问用户**·绝不自己加 registry·绝不就近代用现有 area。

「就近代用」是 D44 实战中反复出现的问题——法隆寺套 `nara_park`（实际在斑鸠町·12 分钟车程外）·唐招提/药师套 `nara_park`（实际在西之京）·大原三千院套 `北区`（实际在洛北大原·跟北区不同动线）。**寺名带某区不代表就属于该区**·area 是物理位置不是就近联想。

**命名风格**：写新 entity 前先扫 `area_registry.json` 看相邻区域怎么命名·沿用周围风格（kyoto 多数英文 snake_case 如 `higashiyama` / `arashiyama`·部分中文如「北区」「京都站」混存）。**风格混存不是你引入新风格的理由**·选周围最常见的那个。

校验脚本对 area 不在 registry 的会 FAIL（D44 修复·此前不查）。

### 5. depth 必须真实

- skeleton：基础信息（id / city / area / category / note 简介）
- verified：note 全填 + 元数据完整
- full：verified + templates_meta 至少 3 个 key（拍照位置/冷知识/避坑 优先）

**templates_meta 全空也能 verified·但不能 full**。

### 6. 主次分明

- **A 类**（装配跑不动 / 字段缺失） → 停下报告
- **B 类**（用户体验硬伤：地址错 / 季节标错 / 模板用不到） → 停下报告
- **小事**（templates_meta 不全 / 文笔粗糙 / 数据来源单一） → 标 verified 或留空·**继续做不停**

### 系统字段（7 个）

| 字段 | 必填 | 说明 |
|---|---|---|
| `entity_id` | ✅ | `{城市拼音简写}_{景点名}` |
| `city` | ✅ | 中文城市名 |
| `area` | ✅ | 动线粒度·与 area_registry 对齐 |
| `category` | ✅ | 类型枚举·12 类中文（详见 [字段权威.md §2.1.5](../../项目核心/字段权威.md)） |
| `depth` | 选填，缺省 `skeleton` | `skeleton` / `verified` / `full` |
| `season_months` | 选填 | 季节独家月份数字数组（红叶=[11,12]）·常年=null 或省略 |
| `reservation` | 选填，缺省 `none` | `none`（无需预约）/ `recommended`（旺季建议）/ `required`（必预约·普通难度·苔寺/任天堂博物馆）/ `hard`（必预约·名额秒抢·桂离宫/修学院/仙洞）。装配读此字段生成出发前预约提醒；人话说明（怎么预约·开放窗口）写到 `note.营业` 句末 |

### note 块（5 个 key·给用户最终看）

| key | 必填 | 说明 |
|---|---|---|
| `店名` | ✅ | 中文（本地语言/英文） |
| `简介` | full 必填 | 80-200 字·"懂当地的朋友讲口吻"·吸收原 short_desc + 拍照规则 + 预约要求 + 季节亮点 + 避坑 tip |
| `票价` | 选填（免费景点省略） | 自然语言。"成人 ¥500"或"免费" |
| `营业` | full 必填 | 自然语言一句·24h 通行型写"24h"·商店街类写"各店不同" |
| `怎么去` | full 必填 | 含最近交通+步行分钟 |

### templates_meta 块（7 个 key·模板/AI 写文案时用·全选填）

| key | 说明 |
|---|---|
| `亮点` | 3-5 个短词标签·模板挂载时拉来 |
| `拍照位置` | 结构化机位（位置/角度/时段对象数组）·或 markdown 列表自由文本 |
| `冷知识` | curators_notes 渲染时引用的 fun_fact 素材 |
| `衔接` | 跟其他 entity 的动线衔接（"步行 N 分到 X"） |
| `季节看点` | 按季节 key 分（樱花/红叶/夏夜/雪）·装配按季节挑 entity 时配套消费 |
| `顺路小店` | 自由漫游段引用 |
| `避坑` | 模板/AI 写文案时引用 |

整块可空。但 depth=full 必须至少 3 个 key（**拍照位置/冷知识/避坑** 优先）。

### 元数据块（3 个）

| 字段 | 必填 | 说明 |
|---|---|---|
| `可信度` | ✅ | `verified` / `cross_checked` / `single_source` / `ai_generated`（不可上生产） |
| `数据来源` | ✅（≥1 URL） | 数组 |
| `最后核实` | ✅ | YYYY-MM-DD |

## 三、分类标准

### 3.1 category 枚举（12 类中文）

D44 瘦身：33 类英文 → 12 类中文（寺社/城/庭园/自然/展望/博物馆/水族动物/主题乐园/体验/街区/温泉/交通）。完整定义见 [字段权威.md §2.1.5](../../项目核心/字段权威.md)。

细类型信息（米其林绿星 / UNESCO / 建筑师 / 主题）走 `note.简介` + `templates_meta.亮点` 自由文本·不再加字段。

不在 12 类的 → 停下问用户。

### 3.2 季节独家 vs 常年

`season_months` 字段决定：
- 常年（多数 entity）→ null 或省略
- 季节独家（红叶名所/樱花夜枫/川床）→ `[11, 12]` 等月份数组

装配按用户行程日期 filter 季节独家 entity·常年 entity 不受季节影响。

### 3.3 采集原则：按"动线必备"+"模板已引用"挑·不堆量

entity 采集**不强求比例**·原则：
- 模板已引用的 entity → 必建（动线必备）
- 用户专程去的 S 级地标 → 必建
- A/B/C 级按"朋友会专程推荐吗"判断
- 不为补 type 凑·不为挂店凑（铁律 1）

## 四、数据准入

> 核心：**这条 entity 该不该收·不靠机械门槛·靠编辑判断**。3 套递进——价值 → 证据 → 角色。
> 理论锚点详见 [数据采集.md §三 三轴 / §五 三层池 / §十一 小众入池 / §六 MTE](../数据采集.md)。

### 4.1 价值判断（要不要这条 entity）

**两条入池路径·OR 关系·任一满足即可**：

#### 路径 A：普世价值

对绝大多数游客都值得去。判断锚——MTE 七维度（享乐 / 刷新 / 本地文化 / 意义感 / 知识 / 参与 / 新奇）：

- **单维度做到关西/该城同类前列** = 收（如清水寺本地文化×意义×新奇都强 / 海游馆关西最大水族馆 / 伏见稻荷新奇极强）
- **多维度都打中**（每个不必头部·但 2-3 维度同时过线） = 收

#### 路径 B：精准价值

对**明确人群**是必去级·对其他人跳过即可。判断锚——能一句话说清「**谁应该来 + 为什么这里不可替代**」：

- 例：日本桥（电子/宅向爱好者·关西唯一秋叶原级电器+宅文化街）
- 例：任天堂博物馆（任系粉·全球唯一）
- 例：西阵织会馆（手作派·京都老铺织物体验）

**写法**：把「谁应该来 + 为什么」自然写进 `note.简介`·让用户在路径 B 类景点之间自己选。**不加 `适合人群` 字段**——简介说清就够·让用户判断。

#### 不收的情况

- 单维度普通过线 + 没明确精准人群 → 不收
- MTE 七维度一个都打不中 → 不收

### 4.2 证据判断（怎么标 `可信度`）

**不是入池门槛·是写完后回标**。三轴（quality / traveler_fit / execution）作为判断辅助·不机械化等级：

| 轴 | entity 信号源 |
|---|---|
| **quality** | 官方观光局 / 本地权威指南 / 米其林绿星 / 学术/媒体长文 |
| **traveler_fit** | 携程评分+评论 / 小红书 collects > likes / 知乎长文 / 马蜂窝 |
| **execution** | 营业稳定 / 票价清晰 / 交通可达 / 人流可控 |

写之前自问：
- 我说的事实·**真搜过还是推断**？推断的标「未核实」或留空。
- 数据来源 ≥1 个**真访问过的 URL**（铁律 3）。

`可信度` 字段是写完回看的标签：
- `verified`——主源 + 至少一个交叉源核实过
- `cross_checked`——主源核实过·有其他参考
- `single_source`——单源·标注后入·后续要升级
- `ai_generated`——AI 推断兜底·**不可上生产**（铁律 3）

### 4.3 角色判断（怎么用 / Tier 分层）

#### Tier 决定投入精力

| Tier | 定位 | 投入 | 字段完整度 |
|---|---|---|---|
| **S 级**（城市代表·不去会遗憾） | 必去 | 30-40 分钟/个 | depth=full + templates_meta 全填 |
| **A 级**（重要搭配·值得专程） | 主线 | 15-20 分钟/个 | depth=full + 至少 3 key |
| **B 级**（顺路可以·不必专程） | 补位 | 10 分钟/个 | depth=verified + 拍照位置 1-2 |
| **C 级**（补充选项·时间富余） | 兜底 | 5 分钟/个 | depth=skeleton 即可 |

#### B/C 级小众入池·编辑判断 4 问（任一答不上 = 不入）

1. **能不能一句话说清「为什么值得去」**？说不清不入
2. **顺路成本能接受**？< 1.5h 顺路 → 进次要 slot；不顺路但精准人群强 → 候选「小众专题日」
3. **同区有没有更适合的替代**？或者**精准服务不同人群**（路径 B）也算独特
4. **note.简介里写清「谁可能喜欢」了吗**？路径 B 必写·让用户自己选

### 4.4 数据源

参考各地区"特化"文档·按地区找对应权威平台。

通用思路：
- **P0**：官方观光局 / 本地权威观光指南 / Google Maps
- **P1**：中国游客 OTA + 小红书 + 知乎长文 + 马蜂窝
- **P2**：英文区域专题站（Inside X / Japan-guide / MATCHA）+ 季节专题

### 4.5 负向信号（任一触发即降 skeleton 或不收）

| 信号 | 处理 |
|---|---|
| 公认游客陷阱 | 不收 |
| **商业化打卡墙 AND 体验本身不好玩**（双门·只满足一个不算） | 归 stops 不归 entity |
| 装修中 / 永久闭馆 | 立即降 skeleton 标注·不删 |
| 小红书爆火但无官方/Google 背书 | 进 skeleton 观察 1-2 个月 |

**双门说明**：白色恋人公园（巧克力工艺主题）/ 任天堂博物馆（互动展品）/ 心斋桥固力果板（道顿堀地标）—— 商业化但有体验深度·**不归 stops**·照常进 entity。无主题深度的纯打卡墙（如 UNIQLO 旗舰店外墙）才归 stops。

### 4.6 使用优先级（实操）

1. 现有 entity 库（`entities/{city}.json`）— 首选·不重复建
2. 模板已引用但 entity 不存在 → 优先补
3. P0 官方观光局 → 列发现池
4. P1 小红书 opencli + 知乎 → 补 traveler_fit
5. P2 季节专题 → 补 templates_meta.季节看点

**不自己编**。

## 五、校验

校验脚本：`scripts/validate_entity.py`

校验规则：
- 系统字段白名单（7 个）+ note 字段白名单（5 个）+ templates_meta 白名单（7 个）+ 元数据白名单（3 个） → 未知字段 FAIL
- reservation 不在 `none/recommended/required/hard` 枚举 → FAIL
- area 字段必须在 `{圈}/area_registry.json` → 不在 FAIL
- category 必须在枚举内 → 不在 FAIL
- 必填字段缺 → FAIL
- depth=full 但 templates_meta < 3 个 key → 自动建议降 verified
- 可信度=ai_generated → FAIL（不可上生产）

跑法：`python scripts/validate_entity.py`·粘**完整输出**到交付消息。

## 六、历史教训（通用·关西作 case）

### 6.1 字段大瘦身（D43·已解决）

D40 时期 entity 字段达 36 个·包括 unesco/phone/address_ja/bridge_length_m 等单一 case 字段·SCHEMA 跟数据双向脱节（107 errors）。
**对策**：D43 重写为三块结构（系统 6 + note 5 + templates_meta 7 + 元数据 3 = 21 字段）·单一 case 字段合并到 templates_meta 自由文本。

### 6.2 entity 为挂店凑数（已禁）

D40 时期为让店家"挂得近"加 entity（如某餐厅没有合理景点 → 加一个新 entity 仅给它挂）。
**对策**：铁律 1——entity 必须自己有意义·店挂不上就降 skeleton 不补 entity。

### 6.3 ID 重复混用（已合并）

`kyo_nijo_castle` vs `kyo_nijojo` / `nar_kasuga_taisha` vs `nar_kasugataisha` 等并存。
**对策**：ID 选规范命名（英文优先·snake_case·城市前缀统一）·D43 时旧 ID 全部合并清理。

### 6.4 category 枚举跟现实脱节（已解决）

D40 SCHEMA 枚举写漏 13 项（aquarium/observation_deck/amusement_park/shopping_street/boulevard/ferris_wheel/experience/unesco_temple 等）。
**对策**：D43 扩枚举·新类型不进枚举的停下问用户。

### 6.5 opening_hours 24h 通行型空缺（已解决）

街道/河岸/桥这种 24h 通行型 opening_hours 大量空缺。
**对策**：合并到 note.营业 自然语言一句话·24h 通行写"24h"·商店街写"各店不同"。

### 6.6 templates_meta 全空但标 full（注意点）

depth=full 但 templates_meta 7 个 key 全没填·无法支撑模板拉来用。
**对策**：full 必须至少 3 个 key（拍照位置/冷知识/避坑 优先），否则降 verified。

### 6.7 D44 准入二次定型（2026-04-25）

D43 完成字段瘦身·但准入规则、category 粒度、预约表达、area 约束四件事还是 D40 机械门槛思路。D44 一起定型——详见 [历史决策.md D44](../../../项目核心/历史决策.md)。

**核心改动**：
- 准入：MTE ≥2 机械门槛 → 路径 A 普世 / 路径 B 精准 OR
- category：33 英文 → 12 中文（细类型走自由文本）
