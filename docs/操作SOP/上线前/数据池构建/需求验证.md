# 数据池需求验证 SOP（全球通用）

> **写完一类数据池后必跑一次**。验证你筛的池子能不能满足真实用户的真实需求·不是你拍脑袋想的需求。
> 适用：酒店 / 餐厅 / stops / 景点 4 类 × 任意城市圈。

## 一、为什么要做

数据池建好不等于有用。常见盲点：

1. **价格带漏档**：池里都是中端·真实用户里 30% 是预算控
2. **场景漏类**：池里都是热门旅游 spot·没满足"亲子/带老人/小红书打卡/独处"等小众场景
3. **品牌漏头**：池里都是高端独立·万豪/IHG/Hilton 经济线（Four Points Flex / Holiday Inn Express / Hampton）漏了——这些恰是会员积分用户最关注的
4. **新开业漏接**：2024-2026 新开业的酒店在小红书爆但池里没

| 时机 | 触发 |

## 三、流程（4 步）

### Step 1 — 制定查询词矩阵

按 3 个维度交叉·覆盖真实用户搜索的话术：

| 维度 | 示例（酒店） | 示例（餐厅） |
|---|---|---|
| **预算带** | 200/300/500/1000/1500 元 一晚 | 500 円 / 千円 / 3000 / 一万円 一顿 |
| **场景** | 亲子 / 闺蜜 / 独行 / 带老人 / 蜜月 | 排队 / 不踩雷 / 一人食 / 必吃 / 隐藏 |
| **位置** | 京都站 / 河原町 / 祇园 / 清水寺 | 难波 / 心斋桥 / 锦市场 |

每个城市圈最少 8-12 条查询·覆盖低中高 3 个预算带 × 3-4 个核心场景。

### Step 2 — 用 opencli 搜小红书 + WebSearch 搜公开榜单

**小红书**（中国用户真实需求）：

```bash
cd D:/projects/projects/travel-ai/opencli-main
node dist/main.js xiaohongshu search "京都酒店 500元" --limit 8 --format md
```

判断爆款：**collects > likes 是真爆款**（用户收藏率 > 点赞率·说明实用性强·不只是好看）。
读爆款用 `xiaohongshu note <URL>` 看正文+评论里被提到的具体店名。

**WebSearch**（覆盖小红书外）：trip.com/booking/jalan/楽天 榜单·tabelog 评分 / 一休 100 选 / Forbes / MICHELIN / SLH·补 OTA + 权威认证视角。

### Step 3 — 对照池子

把搜出来的真实店名跟数据池 grep·分 3 类：

| 状态 | 处置 |
|---|---|
| ✅ 池里已有 | 不动 |
| ⚠️ 池里没有但价值高（爆款/连锁/认证）| 加入·走标准采集流程 |
| ❌ 搜出来但价值低（一次性网红/已闭店/位置不对） | 不收·写一行理由 |

### Step 4 — 输出缺口报告

每次跑完写一份 Markdown 报告·**留痕到** `_archive/{D 编号}_{池子名}_需求验证/`·内容：

```markdown
# {池子名} 需求验证 - {日期}

## 查询矩阵
- 预算带：[列表]
- 场景：[列表]
- 位置：[列表]

## 池现状
- 总量：X 家
- {维度 1} 分布：...

## 真实需求 vs 池缺口

### 缺口 1：低价带（200-500 元）漏
- 真实需求案例：xx 家小红书爆款（collects 800+）
- 池里现状：最低档 ¥550 起·空白
- 建议补：Four Points Flex / Holiday Inn Express / Smile Hotel 等

### 缺口 2：xxx
...

## 行动项
1. [ ] 补 X 家·predicate 完成日期
2. [ ] 删 Y 家（已闭店/数据污染）
3. [ ] 跟产品确认：要不要扩档（涉及定价表）
```

## 四、4 类数据池的差异

| 池 | 主要查询维度 | 关键判断指标 |
|---|---|---|
| **酒店** | 预算 + 位置 + 场景 | 价格带覆盖 / 连锁集团覆盖 / 新开业接续 |
| **餐厅** | 预算 + 菜系 + 位置 + 排队 | 菜系覆盖 / A-B 角色平衡 / Tabelog 评分阈值 |
| **stops（小店）** | 主题 + 位置 + 季节 | 是否真本地人在去 / 网红打卡 vs 实用 / 季节限定 |
| **景点** | 类型 + 位置 + 季节 + 时段 | 必去 / 隐藏 / 季节限定 / 早起独享 |

**关键原则**：景点是动线锚点（要必去）·餐厅/stops/酒店是配套（要满足实用需求）·两类的"价值"判断标准不同。

## 五、AI 复用提示

任何 AI 在新城市圈做这个验证·按以下指引：

1. **先读** `docs/操作SOP/上线前/数据池构建/{x}规范.md` 确认池子的设计意图
2. **拿** `japan/kansai` 的需求验证报告作为模板（看 `_archive/d49_*` 子目录）
3. **小红书查询**用 opencli·**注意 collects > likes 才是真爆款**·只看 likes 容易被假数据骗
4. **不许编造缺口**·每条缺口必须有具体真实店名+爆款 URL+访问数据作证
5. **不许擅自补池**·缺口列出后由项目主人决定补哪几家·AI 只汇报+建议
6. 报告写完留痕到 `_archive/`·便于追溯
