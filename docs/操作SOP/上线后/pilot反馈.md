# pilot 反馈

> Phase 8·上线后启动。
> 用户拿到本子去旅行回来·结构化采集 diary 反馈·循环回模板/数据池迭代。

---

## 第一性原理

**真实用户的旅行体验是模板质量的唯一裁判**——专家闭门策展能保证下限·但「真实地铁站换乘走错没」「推荐餐厅周三关门没标」「黄昏 peak-end 是不是真的美」只有用户走过才知道。

pilot 反馈是把「闭门策展」转向「**用户验证策展**」的关键链路。

**判断锚**:每条反馈处理完·我们的模板/数据池**变得更值得用户付费了吗**?

---

## 启动条件

[D16](../../项目核心/历史决策.md) 当前定的是「用户纠偏暂不做·样本太小」。pilot 反馈在以下任一条件触发后启动:

- 单一城市圈累计交付 ≥ 30 单·且至少 10 单已回访
- 出现具体事件性反馈值得记录（餐厅关门 / 路线踩雷等）·先单条记录不建系统
- 进入正式运营期·不再是「样板间打磨」

---

## 三层反馈结构

### Layer 1:出发前预约确认（自动·已在业务流）

用户在解锁全本后·系统自动追踪预约项的完成状态·这一层不是真正的「反馈」是「执行确认」。详见 [业务流.md §阶段四](../../项目核心/业务流.md)。

### Layer 2:旅行中即时反馈（轻量·客服微信）

用户在旅行过程中遇到问题随时发客服:

- 推荐餐厅临时关门 / 装修
- 实地路线跟手账描述对不上
- 等位时间远超预期
- 季节限定景点没赶上

**客服处理**:即时给备选 + 记录 issue 到反馈池·不阻塞用户当天行程。详见 [客服应对.md §四](客服应对.md)。

### Layer 3:回来后 diary 采集（结构化·本文重点）

用户回来后 3-7 天·客服发起结构化反馈采集。这是 pilot 反馈的**主战场**。

## diary 采集模板

```
亲~ 您从关西回来啦。这本手账如果有任何不顺手的地方·我们都很想知道——
不是为了回访客气·是真的会改下一版本。

请您花 5 分钟回答几个问题:

【1. 三问】
- 整本手账最满意的一个安排是什么?
- 整本手账最失望的一个安排是什么?
- 你会推荐这本给打算去关西的朋友吗?如果会·会怎么形容它?

【2. 每天打分】
请用 1-5 星打分·每天:
- D1 关西机场→大阪难波夜    ⭐⭐⭐⭐⭐ + 一句话原因
- D2 大阪城+心斋桥           ...
- D3 京都岚山一日             ...
- ...

【3. 实地修正】
有没有哪天的安排「跟手账写的实际情况不一样」?比如:
- 餐厅关门 / 装修 / 改菜单 / 涨价
- 景点路线在现场发现更近的走法
- 时间估算明显偏离（建议 1.5 小时实际 30 分钟就够了 / 建议 30 分钟实际不够）
- 季节限定的东西没看到（红叶没红 / 樱花没开 / 祭典停办）

【4. 加分项】
有没有发现手账没写但你觉得超棒的地方?(我们考虑加进下版)

【5. 体验吐槽】
有什么不太涉及具体安排但「整体感觉不对」的吐槽吗?(任何方面)
```

## 反馈分类与处理 SOP

每条反馈进来·客服按 4 类归档:

| 类别 | 处理人 | 处理动作 | 反馈池字段 |
|---|---|---|---|
| **数据错误** | 数据团队 | 改 entity / restaurant / hotel JSON | category=data_error |
| **节奏问题** | 模板团队 | 改对应模板 slots / pace_type / time_sensitivity | category=template_pace |
| **推荐踩雷** | 餐厅/酒店团队 | 调整入池 score / 加 risk_flags / 降 skeleton | category=pool_quality |
| **服务问题** | 客服 / 运营 | 客服话术升级 / 流程优化 / 印刷质量改善 | category=service |

**反馈池路径**:`feedback_pool/{城市圈}/{YYYY-MM}/{order_id}.json`（待建·当前阶段先用客服 issue 单文档记录）

## 反馈循环回模板/数据池的链路

```
用户 diary
  ↓
客服归档 4 类
  ↓
每月汇总 → 数据/模板/池/服务团队
  ↓
团队评估:
  - 单点错误 → 直接改 JSON
  - 系统性问题（≥3 单同类反馈）→ 进入决策讨论 → 可能升级为新决策（D44+）
  ↓
改完跑 validate_template.py
  ↓
影响 ≥ 3 个模板时·标 last_updated 重新过审核
```

**关键原则**:
- **绝不静默修复**——每条用户反馈处理完都要回信告诉用户「您当时反馈的 X 我们怎么处理了」·这是「敢对结果负责」的具体兑现
- **反馈不是工单**——反馈池要留得长·6 个月后回看才能看出系统性问题
- **样本量警戒线**:单一反馈不动模板（可能是用户偶发体验）·≥3 单同类才考虑改

## 反馈循环跟运营的关系

好的反馈是**运营素材**:

- 「最满意的安排」直接做小红书脚本（用户视角验证·比博主背书强）
- 「会怎么形容这本」是测试品牌定位的真实语言（用户怎么说我们 = 我们的品牌）
- 实地修正中的本地知识（小路 / 巷子 / 老铺）→ 沉淀到 [marketing/{圈}/素材库.md](../../../marketing/)

详见 [内容运营客服.md](内容运营客服.md)。
