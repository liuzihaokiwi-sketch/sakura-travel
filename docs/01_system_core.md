# Travel AI 统一系统化方案（从“免费一日极简收集”开始的完整整合版）

> 本文档只整合以下阶段确定的内容：  
> 从“免费一日只收极简信息”开始，到后续的样片交付、客服承接、付费后详细表单、自动校验、片段攻略复用、硬规则/软规则接入、后端生成链路、PDF/H5 交付、反馈回库。  
> 如与更早的旧讨论冲突，以本文档为准。

---

## 0. 一句话总纲

系统主策略不再是“用户一上来填很多信息，然后 AI 从零生成完整攻略”，而是：

**免费阶段：极简输入 → 自动给高质量一日样片 → 样片页成交**  
**付费阶段：详细表单 → 自动校验 + 客服补齐 → 片段复用优先 → 规则装配 → AI 少量解释 → 质量门控 → PDF/H5 交付**

这套结构的目标是同时实现：

1. 免费阶段转化阻力最低
2. 正式生成前尽量暴露关键错误
3. 后端尽量复用已验证片段，减少 token 和人工复验
4. 让交付结果更稳定、更像“专业顾问做的取舍”

## 1. 当前单一真相源（本阶段相关）

### 1.1 免费一日样片
免费一日不再承担“真正定制的前半段”角色。  
它的定位是：

**低信息成本的高质量样片**

#### 免费阶段只收 2 个字段
- 目的地
- 旅行偏好 / 风格

#### 免费样片的本质
- 不是完全从零定制
- 是从预先准备好的样片模板池 / route family 中选最合适的一天
- 再做轻量个性化装配
- 给用户一个完整的一天样片

#### 免费样片必须包含
- 当天去哪
- 路线怎么走
- 当天交通衔接
- 当天餐厅建议
- 当天重点提醒
- 当天 Plan B
- 少量“为什么这样排”的专业判断

#### 免费样片不包含
- 全部天数完整展开
- 全旅程安全事项
- 全旅程准备清单
- 全程预约总表
- 多天酒店完整方案
- 正式修改权益
- 尊享服务权益

#### 免费样片必须包含的成交结构
1. 一天完整样片
2. 后续高光钩子
3. 解锁完整攻略 CTA
4. 次级微信 / 客服承接

## 2. 免费阶段前端流程（极简引流路径）

### 2.1 免费阶段表单
表单只出现两个字段：

#### 字段 1：目的地
- 东京
- 京都
- 大阪
- 关西
- 北海道
- 其它（如要开放，可做搜索下拉）

#### 字段 2：旅行偏好 / 风格
当前推荐先用统一一题版本，MVP 可选：

- 经典不踩坑
- 性价比更高
- 奢华体验
- 小众深玩
- 情侣氛围
- 亲子轻松

> 注意：前台可以先只问一题，但后台不要把这 6 个选项当成同一个维度。
> - 经典 / 小众 → 主映射到主题/玩法
> - 性价比 / 奢华 → 主映射到预算偏向
> - 情侣 / 亲子 → 主映射到同行人模板

### 2.2 免费阶段表单要求
- 手机端优先
- 两个字段都尽量做成选择题
- 目的地若支持更多城市，走自动补全/下拉
- 不问天数
- 不问机票
- 不问酒店
- 不问固定行程
- 不问预算

### 2.3 表单提交后主路径
**主路径：自动进入在线样片页**

不要把“联系客服领取样片”作为主路径。  
原因：
- 用户刚提交时热度最高
- 等客服发会增加流失
- 客服会变成样片发放瓶颈
- 免费阶段不值得让客服人工确认基本输入

### 2.4 免费阶段的次路径
在线样片页中再放：
- 下载 PDF
- 解锁完整攻略
- 加微信咨询

## 3. 免费样片页如何设计

### 3.1 主路径：网页在线预览
样片页主路径应为在线页，不是先给 PDF。  
原因：
- 更适合做首屏成交
- 更适合插钩子
- 更适合埋点
- 更适合加客服微信
- 更适合引导解锁付费

### 3.2 PDF 的角色
PDF 作为补充下载存在：
- 方便保存
- 方便离线看
- 方便发给同行人
- 海外网络不稳时可直接查看

### 3.3 样片页必须出现的模块
按顺序建议：

1. **完整的一天样片**
   - 时间轴
   - 区域
   - 餐厅建议
   - 路线安排
   - Plan B

2. **为什么这一天这样排**
   - 1–3 个专业判断点

3. **后续高光钩子**
   - 第 3 天的高光体验
   - 某一晚的夜景 / 温泉 / 晚餐高光
   - 整趟路线脉络（锁细节）

4. **主 CTA**
   - 解锁完整攻略

5. **次 CTA**
   - 加客服微信聊这趟更适合你的版本

6. **PDF 下载**
   - 作为补充入口

### 3.4 样片页文案目标
不是只证明“我们有内容”，  
而是要让用户产生这两种感受：
- 这份东西真的值钱
- 后面几天还有更值得买的部分

## 4. 客服在免费阶段的角色

### 4.1 客服不负责
- 发放基础样片
- 手动确认免费阶段基础信息
- 逐单审核免费样片

### 4.2 客服负责
- 接住样片看完后犹豫的用户
- 回答价格和付费问题
- 处理特殊需求
- 付费后发详细表单
- 付费后协助补齐红黄问题

### 4.3 微信的最佳位置
微信不要拦在最前面。  
应放在：
- 样片页中部
- 后续钩子下方
- 页面底部固定次按钮

---

## 5. 付费后的详细表单：目标与原则

### 5.1 目标
付费后详细表单不是为了“多问”，而是为了：

**锁死真正影响完整攻略质量的关键约束**

### 5.2 核心原则
1. 分步填写，不做一页超长表单
2. 选择题优先，少开放输入
3. 地点全部标准化，不允许用户乱写
4. 条件字段默认隐藏
5. 行内校验，不等最后报错
6. 提交后不直接生成，必须先过后台校验

---

## 6. 详细表单完整结构（6 步）

### Step 1：基础路线信息
目标：先把骨架定住

#### 必填
- 去几天
- 目的地 1
- 是否第一次去日本
- 这次更像：
  - 多城顺玩
  - 一地深玩
  - 还没想好

#### 建议收
- 目的地 2（可选）
- 目的地 3（可选）
- 每个目的地预计停留天数（可选）

#### 设计要求
- 目的地必须用搜索 + 下拉结果选择
- 选中后保存标准化 `place_id / location_id`
- destination_2 / destination_3 默认隐藏，用户点击“+ 添加目的地”后才出现

---

### Step 2：交通与固定时间
目标：锁死不能动的约束

#### 必填
- 是否已买机票 / 车票
- 到达日期
- 到达时间
- 到达地点（机场 / 车站 / 城市）
- 离开日期
- 离开时间
- 离开地点（机场 / 车站 / 城市）

#### 条件触发
如果用户有固定安排，再展开：
- 固定安排类型（演出 / 预约 / 见朋友 / 餐厅 / 其它）
- 固定日期
- 固定时间
- 固定地点
- 备注

---

### Step 3：住宿情况
目标：确认可调整空间

#### 必填
- 是否已订酒店
  - 全没订
  - 已订一部分
  - 已订全部

- 是否接受更换酒店区域
  - 可以
  - 尽量不要
  - 不接受

- 是否希望尽量少换酒店

#### 条件触发
若已订酒店，则展开：
- 酒店所在城市 / 区域
- 入住日期
- 退房日期
- 酒店名（可选）
- 备注（可选）

---

### Step 4：同行人与风格
目标：让系统知道这趟该怎么排

#### 必填
- 和谁去
  - 情侣
  - 闺蜜好友 / 朋友
  - 带父母
  - 带孩子
  - 独自出行

- 这趟更想要什么感觉
  - 经典不踩坑
  - 更有审美 / 更容易出片
  - 吃得更好
  - 更轻松
  - 更小众一点
  - 温泉疗愈
  - 购物为主

- 节奏偏好
  - 轻松
  - 均衡
  - 能接受高密度

#### 条件触发
- 带父母 → 额外收步行/换乘承受度
- 带孩子 → 年龄段 / 推车需求 / 午休需求
- 情侣 → 是否纪念日 / 氛围感是否优先

---

### Step 5：预算与取舍
目标：别让用户自己选模块，让系统懂预算怎么花

#### 必填
- 预算范围
- 更想把钱花在哪里
  - 住得更好
  - 吃得更好
  - 体验更特别
  - 更均衡
  - 更看重整体性价比

#### 建议收
- 是否想安排温泉旅馆
- 是否想安排纪念日晚餐
- 是否想安排购物
- 是否想安排文化体验
- 是否优先出片

---

### Step 6：硬约束与避坑
目标：减少返工

#### 必填
- 必去点 / 区域
- 不想去点 / 区域
- 饮食禁忌 / 过敏
- 体力情况（低 / 中 / 高）

#### 建议收
- 不想早起
- 不想太晚回
- 是否需要无障碍 / 特殊照顾
- 其它特殊备注

---

## 7. 后台校验：红黄绿系统

付费表单提交后，不直接生成。  
先进入自动校验。

### 7.1 红色：必须补充 / 必须改，不能生成
示例：
- 到达/离开时间缺失
- 已买机票但没填到离地点
- 主要目的地为空
- 固定安排缺时间或地点
- 已订酒店但酒店信息不完整
- 必去和不想去里有冲突地点
- 离开日期早于到达日期
- 行程天数与到离时间明显不一致

### 7.2 黄色：建议确认，可先不拦截
示例：
- 选择一地深玩，但添加了多个跨区域目的地
- 预算偏性价比，却选择多个高成本项目
- 体力偏低，但想走高密度路线
- 到达太晚 / 离开太早
- 已锁死酒店，但路线跨度很大
- 带父母 / 带孩子，却同时想多城高密度

### 7.3 绿色：可进入生成
- 主要字段完整
- 无红色冲突
- 黄色项可被客服确认或默认保守处理

---

## 8. 客服在付费后的正确角色

### 8.1 客服要做的事
- 发详细表单链接
- 查看后台红黄绿结果
- 只追问红色和黄色问题
- 确认后触发生成
- 在必要时接管复杂情况

### 8.2 客服不该做的事
- 人工通读整个表单
- 人工推理整个路线
- 人工清洗所有自由文本
- 人工发基础样片

### 8.3 后台给客服看的界面
客服后台不该展示完整原始表单为主，  
而应展示：
- 红色问题摘要
- 黄色问题摘要
- 一键复制给用户的话术
- 当前通过状态

---

## 9. 后端正式生成：完整主链路

付费订单生成完整攻略，不再靠一个大 prompt 一次性写完。  
正确主链如下：

1. 创建订单与 trip 草稿
2. 等待详细表单
3. 自动校验表单
4. 客服补齐红黄项
5. 用户画像标准化
6. 优先复用片段库
7. 硬规则过滤
8. 软规则重排
9. 路线骨架装配
10. 动态快照补槽
11. AI 少量解释
12. 质量门控
13. 多模型评审 / 人工复核
14. 渲染 PDF / H5
15. 交付
16. 自助微调
17. 正式修改
18. 回访与反馈回库

---

## 10. 用户画像标准化（生成前必做）

详细表单通过后，不直接进 planner。  
先把用户数据整理成内部统一画像：

- 目的地序列
- 天数
- 同行人类型
- 日本熟悉度
- 多城 / 深玩
- 主题偏好
- 预算偏向
- 固定时间点
- 酒店限制
- 必去 / 不想去
- 饮食禁忌
- 节奏偏好

输出：
- `planning_profile`
- `profile_version`
