# 前置与总纲页型详细说明

> 每一种页型都按同一结构说明：
> 1. 主要是干什么的
> 2. 设计思路
> 3. 放哪些数据
> 4. 排版顺序
> 5. 哪里要单独有设计感
> 6. 工程注意点

---

# 1. 封面页

## 主要是干什么的
让用户一眼感觉：这不是普通攻略截图，不是临时拼贴 PDF，而是一份认真设计的正式交付物。

## 设计思路
封面不承担信息密度，只承担“第一眼质感”和“本次旅程身份感”。

## 放哪些数据
- trip_title
- travel_dates
- duration_days
- city_circle_name
- travel_keywords（3–5 个）
- traveler_label（如情侣 / 闺蜜 / 带父母）
- hero_image_asset
- subtle_brand_mark
- watermarks_for_pdf

## 排版顺序
1. 大标题
2. 日期与城市圈
3. 旅行关键词
4. 用户标签 / 旅程气质副标题
5. 封面图或图形层
6. 品牌与版本信息（很轻）

## 哪里要单独有设计感
- Hero 区必须有明显呼吸感
- 标题层级要像“书名”
- 背景图不能只是大图平铺，要有裁切与叠层
- 关键词标签应有精致而不过度的点缀感

## 工程注意点
- 封面优先使用稳定素材或 Satori 生成封面图
- 标题长度必须可控，最多两行
- 背景图需要 screen / print 双适配

# 2. 目录页

## 主要是干什么的
帮助用户先看懂整本攻略结构，也方便快速跳转。

## 设计思路
目录不只是“页码列表”，而是一本攻略的结构预览。要让用户看一眼就知道：
- 前面讲什么
- 中间怎么玩
- 后面有哪些专项页

## 放哪些数据
- toc_sections[]
- toc_days[]
- toc_special_pages[]
- page_numbers
- anchor_ids（Web 用）
- chapter_colors_or_icons（可选）

## 排版顺序
### 目录第一页
1. 前置页目录
2. 章节目录
3. 主要活动目录

### 目录第二页
1. Day 列表
2. 酒店 / 餐厅 / 拍摄 / 交通等专题页列表
3. 结尾说明

## 哪里要单独有设计感
- 目录标题与页码要非常清晰
- 层级缩进要明显
- 当前章节类目可加轻微视觉引导
- 不要做成死板表格

## 工程注意点
- Web 目录需要可点击
- PDF 目录要能稳定对应页码
- 章节增减时目录必须自动重排

# 3. 偏好兑现页（这次旅行如何设计）

## 主要是干什么的
这是整本攻略“最像定制”的一页。
它负责回答：为什么这份攻略是你的版本，而不是别人也能拿去用的模板。

## 设计思路
不要长篇抒情，要做成“偏好 → 决策”的映射。
让用户看到系统做过取舍，而且这些取舍和自己有关。

## 放哪些数据
- user_preferences_summary[]
- resolved_tradeoffs[]
- selected_city_circle_reason
- selected_main_activities[]
- intentionally_skipped[]
- planner_summary_sentence

## 排版顺序
1. 页标题：这次旅行如何为你设计
2. 1 句总判断
3. 4–6 条“你的偏好如何被落实”
4. “这次没有放进来的内容”短区块
5. 这次最核心的 3–5 个巧思

## 哪里要单独有设计感
- “偏好兑现”必须像被高亮的证据卡，不是普通段落
- 每条映射应有强对照：你的偏好 / 我们的安排
- 可使用细线、角标、对勾式设计，但不要像 PPT

## 工程注意点
- 每条映射都应来自结构化数据，不靠纯 AI 发挥
- 必须限制条数，避免失去重点
- 这页内容要高度可复用，但语气可轻量个性化

# 4. 主要活动总表页

## 主要是干什么的
先把整趟旅行最重要的 3–7 个活动摆出来，让用户先理解骨架。

## 设计思路
这是“全局骨架页”，不是展开页。
它要解决：这一趟真正值得期待的是什么。

## 放哪些数据
- major_activities[]
  - title
  - circle_or_area
  - day_hint
  - type
  - reservation_level
  - emotional_goal
  - page_ref
- activity_count
- travel_pacing_summary

## 排版顺序
1. 页标题
2. 主要活动总览说明
3. 活动卡片按顺序排布
4. 轻量注释：哪些活动后面有单独详页

## 哪里要单独有设计感
- 活动卡片要有明显主次
- 每个活动至少有一个“让人想去”的短亮点
- 不要做成数据表，要像展陈清单

## 工程注意点
- 活动数量多时切成两栏或双页
- card 信息必须控字数
- page_ref 要与后续详页联动

# 5. 大路线总览页

## 主要是干什么的
帮助用户快速理解整体走法，而不是一页页去拼出路线。

## 设计思路
这页只讲“大逻辑”：
- 怎么分阶段
- 哪天轻哪天重
- 哪几天在哪个区域
- 哪天转场 / 日归

## 放哪些数据
- route_spine[]
- city_circle_segments[]
- day_area_map[]
- intensity_map[]
- transfer_days[]
- travel_flow_notes

## 排版顺序
1. 页标题
2. 一句路线总判断
3. 线路图 / 阶段带状图
4. 每天所属区域与强度
5. 特别说明：转场 / 日归 / 节奏变化

## 哪里要单独有设计感
- 这页的路线图必须做成视觉主角
- 强度变化要可见，不要全靠文字
- 城市圈 / 区域之间的关系要有明显流动感

## 工程注意点
- 不依赖真实地图 API 也能渲染
- 优先做抽象路线图，不做复杂地理地图
- PDF 中要避免过细线条导致失真

# 6. 酒店策略总览页

## 主要是干什么的
解释为什么住这些地方、换不换酒店、最后一晚为什么这么安排。

## 设计思路
这页不讲每家酒店细节，只讲酒店作为路线锚点的作用。

## 放哪些数据
- hotel_bases[]
- nights_per_base[]
- served_days[]
- switch_reasons[]
- last_night_strategy
- hotel_strategy_summary

## 排版顺序
1. 页标题
2. 住法总判断
3. 酒店基点时间轴 / 分段图
4. 每个基点服务哪些活动
5. 为什么换 / 不换
6. 最后一晚说明

## 哪里要单独有设计感
- “酒店基点”不是普通列表，要像路线锚点
- 最后一晚策略可以有单独强调块
- 住法切换的逻辑要看起来很稳，不仓促

## 工程注意点
- 这页与酒店详情页要有关联但不重复
- 适合用横向 timeline 或 stacked cards
- 需要兼容 1 基点 / 多基点两种情况

# 7. 关键预订与时间窗页

## 主要是干什么的
把执行上最重要、最容易错过的预订和时间窗集中说明。

## 设计思路
这是“执行价值页”，不讲种草，只讲行动优先级。

## 放哪些数据
- booking_items[]
  - item_name
  - level（必须订 / 建议订）
  - deadline_hint
  - consequence_if_missed
  - official_source_hint
- time_sensitive_items[]
- precheck_flags[]

## 排版顺序
1. 页标题
2. 必须处理的事项
3. 建议尽早处理的事项
4. 关键时间窗
5. 极简行动建议

## 哪里要单独有设计感
- 必须订与建议订要有强层级差
- deadline 与 consequence 必须一眼看到
- 这页更偏功能，但仍应保持整洁和可信感

## 工程注意点
- 内容必须尽量来自规则和快照
- 不要让 AI 自由编造订票细节
- Web 版可挂官方链接，PDF 版保留文本说明

---

# 8. 出发前指引页

## 主要是干什么的
把出发前真正需要准备的事情一次讲清，降低焦虑感。

## 设计思路
不是百科页，而是“这次出发前你需要做的事”。

## 放哪些数据
- prep_checklist[]
- apps_needed[]
- payment_connectivity_notes
- luggage_notes
- arrival_day_reminders
- destination_specific_prep[]

## 排版顺序
1. 页标题
2. 出发前 checklist
3. 到达当天准备
4. App / 支付 / 通讯
5. 行李与穿着
6. 极简提醒

## 哪里要单独有设计感
- checklist 要非常好扫
- 不要做成长段说明
- 到达当天可以做成独立提醒卡

## 工程注意点
- 大部分内容应来自静态块
- 目的地 / 季节差异通过动态插槽补齐
- 控制字数，避免变成百科附录

---

# 9. 动态注意事项页

## 主要是干什么的
告诉用户：临近出发或旅行中，哪些情况最可能影响执行。

## 设计思路
这页只放“动态风险”，不重复前置可规避风险。

## 放哪些数据
- live_disruption_risks[]
- weather_sensitive_items[]
- transport_sensitivity[]
- crowd_risk_notes[]
- monitoring_sources[]
- fallback_refs[]

## 排版顺序
1. 页标题
2. 这次最需要关注的实时扰动
3. 哪些天 / 哪些活动受影响最大
4. 对应调整方向
5. 查看时机建议

## 哪里要单独有设计感
- 风险等级视觉要清晰但不过度吓人
- 最重要的是“如果发生，怎么改”
- 不要做成红色告警墙

## 工程注意点
- 这页强依赖动态数据，允许临近生成
- 必须区分 precheck 与 live risk
- PDF 中应写明“以临近更新为准”
