# 初版预览 → 用户微调 → 最终版：可执行方案 v1

## 1. 目标

在不显著增加系统复杂度的前提下，为网站增加一层轻量的“预览确认”流程：

- 用户先看 **初版骨架预览**，不是直接看完整攻略
- 用户只修改少数 **高价值约束**
- 系统基于修改后的约束 **重跑 Layer 2 决策与编排**
- 最后再补齐剩余数据与页面，生成 **最终版攻略**

这层能力的产品目标：

1. 降低一次性生成完整攻略的返工率
2. 提高“这是按我喜好定制的”感知
3. 把用户修改控制在少量高价值变量，避免变成复杂编辑器
4. 让生成成本向“先轻后重”迁移，减少无效计算和低价值文案产出

---

## 2. 结论

### 2.1 要不要做

要做，而且更适合放在 **网站前台**，不适合放在后台人工流程里，也不适合放在聊天里。

### 2.2 要不要先完成四层再做

**不要等“四层所有高级能力都做完”再做。**

但也 **不要在四层主链还没跑通时就先做这个功能。**

正确顺序是：

### 先完成“最小可运行四层闭环”

也就是：

- Layer 1：数据侧最小可用
- Layer 2：城市圈 → 主活动 → 酒店 → 日骨架 → 次要活动/餐厅 能稳定跑通
- Layer 3：能输出“骨架预览页”和“最终报告页”两种模式
- Layer 4：至少有前置校验、基本质量门、基础 fallback、基础 trace

### 然后做“预览确认层”

因为这层本质上不是替代主链，而是插在：

`第一次轻生成` → `用户确认` → `第二次重生成`

之间。

如果主链本身还不稳定，这层只会放大混乱。

---

## 3. MVP 范围（一定要控制）

不要做开放式改稿器，只做“轻确认层”。

### 3.1 首期只开放 5 类可修改项

1. **必去 / 不去**
   - 一定要去某活动 / 某区域
   - 明确不要某类活动

2. **住宿偏好**
   - 想住某家酒店
   - 想住某个区域
   - 想少换酒店
   - 最后一晚靠近机场 / 车站

3. **节奏偏好**
   - 更松弛 / 更丰富
   - 接受 / 不接受远距离日归
   - 接受 / 不接受早起

4. **预算倾斜**
   - 更重酒店 / 更重餐厅 / 更重拍摄 / 更重整体省钱

5. **一条补充说明**
   - 纪念日
   - 长辈行动不便
   - 想看夜景
   - 想住温泉酒店

### 3.2 首期不开放

- 每天逐点拖拽修改
- 每个餐厅逐条替换
- 每个景点任意替换
- 评分权重滑杆
- 报告排版样式修改
- 无限次重生成

### 3.3 首期只允许 1 次结构修改轮

流程：

- 初版骨架预览
- 用户修改一次
- 系统重跑
- 进入最终补全

不做无限来回修改。

## 4. 产品流程

## 4.1 主流程

### Step 1 基础表单提交
输出：`normalized_trip_profile`

### Step 2 轻生成（Preview Build）
只跑到：

- 选城市圈
- 选主要活动
- 定酒店基点
- 排日骨架
- 选预览日 / 预览片段

不补全所有餐厅、拍摄、专题页、完整文案。

输出：`preview_payload`

### Step 3 网站预览确认页
用户看到：

- 这次选了哪个城市圈
- 为什么这么选
- 3–5 个主要活动
- 酒店策略
- 每天大致节奏
- 没选的重点内容及原因
- 可修改项入口

### Step 4 用户微调
写入：`user_revision_pack`

### Step 5 二次生成（Final Build）
生成逻辑读取：

- 原始表单
- 规范化画像
- 用户 revision pack
- 第一轮 decision snapshot（可选）

再跑正式 Layer 2 + Layer 3，补齐：

- 次要活动
- 餐厅
- 酒店详页
- 拍摄页
- 条件页
- 完整 explain
- 最终 page_plan

### Step 6 最终版交付
输出：

- Web 阅读页
- PDF
- 后台可交付版本

## 5. 网站页面设计（MVP）

## 5.1 页面 1：初版预览页

### 页面目标
让用户确认“方向对不对”，不是让他审整本报告。

### 页面模块

1. 顶部状态条
   - 已提交
   - 初版已生成
   - 等待你确认
   - 最终版生成中 / 已完成

2. 本次方案摘要
   - 城市圈
   - 天数
   - 同行人
   - 节奏标签
   - 预算标签

3. 为什么这样安排
   - 城市圈选择理由
   - 主要活动选择理由
   - 酒店策略理由

4. 主要活动卡片
   - 3–5 个 major activities
   - 每个活动一条 why_selected

5. 酒店策略卡片
   - 住几个基点
   - 最后一晚逻辑
   - 是否换酒店

6. 每日节奏速览
   - Day 1 ~ Day N
   - 每天一个主线标题
   - 强度标签

7. 这次没放进去的内容
   - skipped options + reasons

8. 修改入口区
   - 改必去项
   - 改住宿偏好
   - 改节奏
   - 改预算倾向
   - 补充说明

9. CTA
   - “确认这版方向，继续生成完整版”
   - “我想修改几个关键项”

## 5.2 页面 2：修改偏好页（多步）

不要一个大表单，做成 5 个小步骤或 5 个抽屉：

- 修改必去 / 不去
- 修改住宿偏好
- 修改节奏
- 修改预算倾斜
- 补充说明

每次只改一组。

## 5.3 页面 3：确认并生成最终版页

展示：

- 你做了哪些修改
- 系统会据此调整什么
- 点击后开始补全最终版

## 6. 数据结构建议

## 6.1 新增：preview_payload

用于轻生成预览页。

建议字段：

```ts
interface PreviewPayload {
  planId: string
  schemaVersion: string
  circleSelection: {
    selectedCircleId: string
    reasons: string[]
    rejectedOptions?: Array<{ id: string; reason: string }>
  }
  selectedMajors: Array<{
    majorId: string
    title: string
    whySelected: string
    timeWindow?: string
    bookingLevel?: string
  }>
  hotelStrategy: {
    bases: Array<{ baseId: string; nights: number; whyHere: string }>
    lastNightSafe: boolean
    switchReason?: string[]
  }
  dailyFrames: Array<{
    dayIndex: number
    title: string
    intensity: 'light' | 'balanced' | 'heavy'
    primaryCorridor: string
    mainDriver: string
  }>
  skippedOptions?: Array<{
    title: string
    reason: string
  }>
  previewDay?: {
    dayIndex: number
    rationale: string
  }
}
```

## 6.2 新增：user_revision_pack

```ts
interface UserRevisionPack {
  planId: string
  revisionRound: number
  hardConstraints: {
    mustGoIds?: string[]
    mustAvoidIds?: string[]
    requiredHotelIds?: string[]
    requiredAreaIds?: string[]
    lastNightNearTransit?: boolean
  }
  softPreferences: {
    paceMode?: 'relaxed' | 'balanced' | 'packed'

    hotelFocus?: boolean
    foodFocus?: boolean
    photoFocus?: boolean
    budgetLean?: 'save' | 'balanced' | 'upgrade_one_thing'

  hotelPreferences: {
    avoidHotelSwitch?: boolean
    wantOnsenHotel?: boolean
    preferredAreas?: string[]
    avoidedFeatures?: string[]
  }

}
```

```ts
interface PlanGenerationStage {
```

## 7. 后端实现方案

## 7.1 新增两种生成模式

### A. preview mode

调用链：

`normalize → validation/precheck → select_circle → rank_major → build_hotel → build_skeleton → preview_engine(optional) → build_preview_payload`

### B. final mode

调用链：

`normalize + revision_pack → select_circle / lock circle(optional) → rank_major → build_hotel → build_skeleton → fill_secondary → fill_meals → explain → report payload → page plan → render`

## 7.2 revise 时不必全盘重算的部分

### 可以锁定的
- 基础 profile normalize 结果
- 可复用的实体候选池
- route matrix cache
- 首轮 circle selection（若用户未改核心约束）

### 应该重算的
- selected major activities
- hotel strategy
- skeleton
- secondary / meals
- explain
- final report payload

## 7.3 约束写入逻辑

所有用户修改，不直接改数据库里的实体权重。

而是：

- 写到 `user_revision_pack`
- 在 Layer 2 读取时转成 runtime constraints

例如：

- `requiredHotelIds` → hotel_base_builder 的 hard constraint
- `mustGoIds` → major_activity_ranker 的 boost + lock
- `mustAvoidIds` → eligibility / filter
- `avoidHotelSwitch` → hotel strategy penalty 提高

## 9. 如何支持“想住某个酒店”

这是首期最值得支持的用户诉求之一。

## 9.1 交互层

在“住宿偏好”页里开放：

- 指定酒店名称（搜索 / 推荐）
- 指定区域
- 不想换酒店
- 想住温泉酒店
- 最后一晚靠近机场/车站

## 9.2 数据层

拆成：

### 硬约束
- 必住某酒店
- 必住某区域
- 最后一晚必须靠近交通枢纽

### 软偏好
- 更偏好新宿 / 梅田 / 札幌站等区域
- 偏好大浴场 / 温泉 / 景观
- 希望连住

### 拒绝项
- 不要小房型
- 不要共享卫浴
- 不要胶囊

## 9.3 规划层

酒店策略读取这些约束时：

- 硬约束优先
- 软偏好只加分，不强制
- 拒绝项参与 filter

这样实现最简单，不需要给用户开放酒店权重调节器。

## 10. 与四层架构的关系

## 10.1 这层功能依赖哪些“必须先完成”的能力

### 必须先有

#### Layer 1
- 城市圈 / 活动簇 / 酒店预设 / 基础实体能查

#### Layer 2
- 选圈、排主活动、定酒店、排 skeleton 稳定
- 基础 explain 字段可拿到

#### Layer 3
- 能生成 preview payload
- 能生成 final payload
- Web 能读两种模式

#### Layer 4
- 表单前置校验
- 基础 quality gate
- 基础 fallback
- trace / decisions 至少有一套

## 10.2 不必等全部做完的能力

以下能力可以后补，不是预览确认层的阻塞项：

- live_risk_monitor
- 完整 grader 系统
- 完整 feedback flywheel
- page_blueprint 模板库全部完成
- 全量 chapter system
- 高级 operator overrides

也就是说：

### 不需要“所有四层高级能力全完成”
### 但需要“最小可运行四层闭环先完成”

## 12. Feature Flag / 灰度建议

必须加 feature flag，不要一次性全开。

建议：

- `preview_flow_enabled`
- `preview_revision_enabled`
- `preview_lock_circle_enabled`

灰度顺序：

1. 内部订单可见
2. 少量真实用户
3. 指定城市圈
4. 全量放开

并且必须支持快速回退到：

- 无预览，直接生成最终版

---

## 13. 验收标准

### 功能验收
- 用户能看到骨架预览
- 用户能修改 5 类关键项
- 用户修改后能生成最终版
- 用户指定酒店/区域能被系统消费

### 工程验收
- preview mode 和 final mode 分离
- revision pack 可审计
- 不直接改基础权重表
- 有 feature flag 和 fallback

### 体验验收
- 用户不需要读完整攻略也能确认方向
- 修改入口清楚
- 只需一次修改轮即可完成大多数 case

---

## 14. 最终建议

一句话版：

**先完成四层的“最小可运行闭环”，再上“初版预览 → 用户微调 → 最终版”功能；不要等所有高级能力做完，也不要在主链未稳时提前做。**

产品上，要做成一个轻量确认层，而不是开放式改稿器。
