# 管理后台：权重与运营配置方案 v1

## 1. 目标

这套后台不是普通 CRUD 后台，而是旅行规划系统的“运营控制台”。

它要解决 4 件事：

1. 不发版即可调整活动、酒店、餐厅、主题、城市圈的权重与规则。
2. 在发布前先预览新配置对真实订单/测试用例的影响。
3. 支持灰度、审批、回滚，避免一次改坏全站。
4. 留下完整审计记录，知道是谁、为什么、何时改了什么。

---

## 2. 管理后台新增的 4 个中枢

### 2.1 配置中心 Config Center
管理“会影响生成结果”的配置，不直接碰代码。

### 2.2 预览对比 Preview Lab
在发布前对比：旧配置 vs 新配置。

### 2.3 发布中心 Release Center
管理草稿、审批、灰度、正式发布、回滚。

### 2.4 审计历史 Audit Center
记录每次改动、版本、影响范围、回滚记录。

## 4. 运营配置中心：应该管理什么

### 4.1 基础权重
适合给运营可调，但必须限制范围。

#### 活动相关
- major_activity_base_weight
- secondary_activity_base_weight
- theme_family_bias
- season_bias
- photo_bias
- food_bias
- shopping_bias
- recovery_bias

#### 酒店相关
- hotel_quality_weight
- hotel_location_weight
- hotel_commute_weight
- hotel_experience_weight
- hotel_last_night_safe_weight
- hotel_switch_penalty_weight

#### 餐厅相关
- dining_quality_weight
- dining_queue_penalty_weight
- dining_detour_penalty_weight
- destination_dining_bonus
- backup_meal_bonus

#### 节奏相关
- dense_day_penalty
- backtrack_penalty
- long_transfer_penalty
- same_corridor_bonus
- day_trip_penalty
- late_arrival_penalty
- early_departure_penalty

---

### 4.2 阈值与上限
这些比权重更适合做运营约束。

- max_major_per_trip_by_day_count
- max_secondary_per_day
- max_cross_city_days
- max_hotel_switches
- max_transfer_minutes_per_day
- max_walk_minutes_per_day
- strong_risk_redline
- low_match_review_threshold

---

### 4.3 规则开关
用于快速启停某种策略。

- enable_city_circle_pipeline
- enable_fragment_first
- enable_hotel_base_strategy
- enable_conditional_pages
- enable_shadow_write
- enable_live_risk_monitor
- enable_operator_override
- enable_review_required_for_low_hit

---

### 4.4 硬规则
不建议开放成随便填数字，而是做成规则条目管理。

例如：
- <7 天不允许 2 个完整城市圈
- 带父母默认禁高密度多城
- 晚到达首日不排重活动
- 已订酒店禁止跨区域高折返
- 高风险活动需要 precheck pass 才能进 major

每条规则应有：
- rule_code
- title
- description
- priority
- enabled
- scope
- reason_template

## 5. 配置作用域设计

同一配置需要支持 4 级作用域：

1. global
2. circle
3. segment
4. plan_override

优先级：

plan_override > segment > circle > global

### 5.1 global
全站默认值。

### 5.2 circle
按城市圈覆盖，例如：
- tokyo_extended_circle_v1
- kansai_classic_circle_v1

### 5.3 segment
按客群覆盖，例如：
- couple_aesthetic
- family_child
- first_time_fit
- repeat_fit
- premium

### 5.4 plan_override
对某一单、某次交付临时覆盖。

---

## 6. 哪些不应该让运营自由调整

以下配置不建议直接开放：

1. eligibility gate 规则主体
2. precheck gate 规则主体
3. DB schema 相关字段
4. explain prompt 主体
5. renderer 结构规则
6. page_type 模板定义
7. 核心状态机

这些更适合：
- 代码配置
- 版本文件
- 管理端只允许 on/off 或有限参数

## 7. 内容库和配置中心的边界

### 内容库负责
- entity_base
- city circles
- activity clusters
- hotel presets
- circle_entity_roles
- 评分原始数据
- why_selected / what_to_expect 等解释资产

### 配置中心负责
- score weight packs
- segment packs
- page trigger thresholds
- review thresholds
- fallback switches
- rollout scopes

不要把“内容数据”和“运营配置”混在一个页面里。

## 8. 必须新增的 3 个核心页面

### 8.1 配置详情页
用途：编辑一个配置包。

页面结构建议：
1. 基本信息
2. 作用域
3. 权重组
4. 阈值组
5. 开关组
6. 影响预估
7. 保存草稿
8. 送审

### 8.2 预览对比页
用途：看新配置会把结果改成什么样。

必须支持：
- 选 1 个订单
- 选 1 个 case
- 旧配置 / 新配置 对比
- 主要活动变化
- 酒店策略变化
- 每日骨架变化
- 分数变化
- explain 变化
- 风险变化

### 8.3 发布中心
用途：正式发配置。

状态流转：
- draft
- pending_review
- approved
- canary
- active
- rolled_back
- archived

## 9. 运营最常用的 6 个配置场景

### 场景 1：情侣审美路线想更强调拍照
调：
- photo_bias
- shareability_bias
- night_view_bonus
- aesthetic_hotel_bonus

### 场景 2：父母单最近被投诉太赶
调：
- family/parents segment 的 dense_day_penalty
- long_transfer_penalty
- max_secondary_per_day

### 场景 3：东京最近更想推温泉酒店升级
调：
- premium segment 下 hotel_experience_weight
- onsen_hotel_bonus
- destination_dining_bonus

### 场景 4：某圈路由太跳
调：
- same_corridor_bonus
- backtrack_penalty
- cross_city_penalty

### 场景 5：某类餐厅排队翻车率高
调：
- dining_queue_penalty_weight
- queue_risk_redline
- backup_meal_bonus

### 场景 6：节假日希望更保守
调：
- review_required_for_low_hit
- live_risk_conservatism
- long_transfer_penalty
- fallback_threshold

## 10. 发布机制

### 必须有的流程
草稿 → 预览 → 审批 → 灰度 → 全量 → 观察 → 回滚（如需）

### 10.1 草稿
任何配置改动先进入 draft。

### 10.2 预览
至少过：
- 3 个真实订单
- 3 个核心 eval case

### 10.3 审批
至少 1 个非编辑本人审批。

### 10.4 灰度
先放给：
- 某一个城市圈
- 某一个 segment
- 某 5% 流量

### 10.5 回滚
回滚必须一键恢复到上一个 active 版本。

## 12. 数据表建议

### 12.1 config_packs
存配置包主信息。

### 12.2 config_pack_versions

### 12.3 config_scopes
存作用域绑定。

### 12.4 config_preview_runs

### 12.5 config_release_records

### 12.6 operator_overrides
存人工对具体订单/实体/活动的临时覆盖。

## 13. 页面级交互建议

### 13.1 输入框不要直接裸露所有数字
更适合：
- 分组折叠
- 默认值 + 可调范围
- 滑杆 / stepper
- 影响说明

### 13.2 每个配置都要显示
- 默认值
- 当前值
- 建议范围
- 上次改动时间
- 最近影响订单数

### 13.3 危险配置要单独标记
例如：
- 会影响所有城市圈
- 会影响酒店切换逻辑
- 会影响 fallback 触发

## 15. 实施优先级

### P0
1. 配置中心（只做基础权重 + 作用域）
2. 预览对比页
3. 发布中心（draft/active/rollback）

### P1
1. operator_overrides
2. segment packs
3. 灰度发布
4. 审计历史

### P2
1. 与 eval / review / feedback 全打通
2. 配置版本对订单/转化影响分析
3. 自动推荐调参建议

---

## 16. 最终原则

管理后台不是“把所有参数都开放”，而是：

1. 让运营能调真正有价值的行为
2. 让配置变更可预览、可灰度、可回滚
3. 让内容数据、规则逻辑、发布流程各自归位
4. 让系统越来越像“可运营的产品”，而不是“只能靠开发改代码的引擎”
