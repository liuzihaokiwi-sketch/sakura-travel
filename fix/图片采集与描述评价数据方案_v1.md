# 图片采集与描述评价数据方案 v1

## 1. 结论

推荐走 **混合方案**：

- **机器采集与规则抽取** 负责 70%~80%
  - 官方站 / 官方社媒 / Google Places Photo / 自有拍摄素材 / 已获授权图库
  - 结构化字段抽取：名称、地址、菜系、房型、设施、营业时间、价格带、评分、评论量、官方描述、菜单/设施标签
  - 首轮图片分类与去重：外观、内景、客房、菜品、活动主场景等
- **人工审核与少量补写** 负责 20%~30%
  - 确认首图是否真正代表该对象
  - 修正错误标签、剔除低质图、剔除不适合展示的图
  - 补“为什么选它”“期待管理”“适合谁”这类高价值文案

不建议两种极端：

- **纯人工填写**：质量高，但太慢，后期不可扩张
- **纯 AI / 纯爬虫自动采集**：速度快，但图和描述很容易失真，版权与来源风险也更高

最稳的是：

**事实和素材由机器做底座，首图与高价值描述由人工把关。**

## 2. 先定采集原则

### 2.1 图片来源原则

优先级建议：

1. **官方来源**
   - 酒店官网
   - 餐厅官网 / 官方菜单页
   - 景点 / 活动官方页面
   - 官方社媒（前提是可用并保留来源）
2. **合规 API / 已授权来源**
   - Google Places Photos（要遵守归属与展示要求）
3. **自有素材 / 合作摄影 / 已购图库**
4. **其他公开网页的元数据图片**
   - 仅作为候选，不建议默认入库主图

如果使用 Google Places Photos，官方要求在展示时保留必要归属，并根据 `authorAttributions` 字段显示要求的作者归属；Google Maps Platform 内容也必须清晰与其他内容区分，并满足归属要求。citeturn665644view0turn665644view1

### 2.2 描述与评价原则

- **描述可以采集与生成，但要分层**
  - 官方描述：原始来源，不直接长段照搬给用户
  - 结构化事实：设施、菜系、预约、价格带、景观、交通、时长
  - 面向用户的短描述：由规则/AI 生成，但基于事实字段
- **评价不建议大量保存平台原文评论**
  - 更适合保存：`aggregate_rating`、`review_count`、`rating_source`
  - 以及从评论中抽象出的 **标签**：景观好、房间大、早餐强、排队久、适合纪念日等
- **用户端展示尽量以总结为主，不以长段摘抄评论为主**

Schema.org 本身支持 `image`、`description`、`aggregateRating`、`review` 等字段；酒店还支持 `starRating`、`amenityFeature`，餐厅支持 `servesCuisine`、`hasMenu`、`acceptsReservations`、`priceRange` 等字段，所以你完全可以把“图、描述、评价、设施”组织成标准化对象。citeturn305047search8turn665644view2turn665644view3turn665644view4turn665644view5

## 3. 三类对象分别采什么图

## 3.1 酒店图片规则

### 必采类型

1. **外观图**
   - 建筑外立面
   - 夜景外观（如有）
2. **大堂 / 公共空间**
   - 大堂
   - 休息区 / 酒廊
3. **代表房型**
   - 标准房型
   - 更高价值房型（景观房 / 套房 / 日式房）
4. **浴室 / 温泉 / 大浴场**
   - 特别重要，尤其日本酒店
5. **早餐 / 餐饮空间**
   - 早餐厅
   - 代表性早餐图
6. **核心特色设施**
   - 温泉
   - 景观露台
   - 泳池
   - 健身房
   - 儿童设施
7. **周边 / 景观 / 交通锚点**
   - 窗外景观
   - 离车站的步行场景

### 推荐采集数量

- 候选池：8~15 张
- 最终精选：4~7 张

### 首图优先级

- 体验型酒店：景观/温泉/代表房型 > 外观
- 过渡型酒店：房型/交通便利感 > 外观
- 机场酒店：房型/接驳信息感 > 设计感

### 不建议作为主图

- 纯 logo 图
- 低清宣传海报
- 只有会议室
- 重复角度的客房图
- 与用户真实入住体验偏差过大的“效果图”

---

## 3.2 餐厅图片规则

### 必采类型

1. **招牌菜 / 代表菜**
2. **完整一餐呈现**
   - 套餐整体
   - Omakase 多贯集合
3. **店内氛围**
   - 吧台 / 餐桌 / 包厢
4. **门头 / 入口识别图**
   - 方便现场找店
5. **菜单 / 价位感图**（可选）
   - 套餐牌
   - 午市/晚市菜单
6. **特殊体验图**
   - 夜景
   - 景观位
   - 铁板/炉端/现烤/茶点演示

### 推荐采集数量

- 候选池：6~12 张
- 最终精选：3~5 张

### 首图优先级

- 目的地型餐厅：招牌菜 / 整套餐呈现
- 纪念日餐厅：氛围 + 主菜 / 景观位
- 顺路补位餐厅：最稳定最有识别度的一道菜

### 不建议作为主图

- 纯环境图但看不出吃什么
- 菜品过度摆拍、与实际差距过大
- 用户拍摄质量过差的模糊图
- 仅饮料图（除非是酒吧/甜品店主打）

---

## 3.3 活动 / 景点图片规则

### 必采类型

1. **主场景英雄图**
   - 一眼代表这个活动的画面
2. **空间上下文图**
   - 它在什么环境里：街区、山路、海边、寺社园林、乐园区域
3. **时间窗图**
   - 白天 / 日落 / 夜景 / 灯光 / 花期 / 雪景
4. **执行辅助图**
   - 入口、交通、缆车、步道、观景台
5. **体验动作图**
   - 喂鹿、泡汤、观景、和服散步、乘船、徒步等

### 推荐采集数量

- 候选池：8~16 张
- 最终精选：4~8 张

### 首图优先级

- 季节型活动：季节限定主场景
- 复杂活动：主场景 + 执行识别图
- 城市地标：最稳定的经典视角

### 不建议作为主图

- 纯导览图 / 地图截图
- 只见人不见场景的自拍图
- 与实际季节严重不符的旧图

## 6. 采集流程建议

## 6.1 机器流程

### 第一步：抓事实
- 官方站
- 官方 structured data
- Open Graph
- 合规 API

Open Graph 本身就把 `og:image`、`og:title`、`og:description`、`og:url` 定义成网页分享对象的基础元数据，很适合做图片候选和描述候选的补充来源。citeturn665644view6

### 第二步：抓图片候选
- 官方主图
- 官方图库
- Places Photos
- 已授权第三方图片

### 第三步：自动分类
- 酒店：外观/客房/浴场/早餐/景观/交通
- 餐厅：招牌菜/整套餐/内景/门头/菜单
- 活动：主场景/日夜/入口/交通/动作体验

### 第四步：自动去重与初筛
- 相似图去重
- 模糊图剔除
- 小图剔除
- 水印太重剔除
- 与 role 不一致的图剔除

### 第五步：自动生成描述候选
- 基于事实字段生成 1 句短描述
- 基于评论标签生成“期待管理”
- 基于对象角色生成“为什么选它”候选

---

## 6.2 人工审核流程

只审核高价值对象：

- 主要酒店
- 主要餐厅
- 主要活动
- 封面页 / 章节页 / Hero 图

人工只需要做 4 件事：

1. 选首图
2. 确认标签
3. 改 1 句高价值描述
4. 标记不适合展示的图

也就是：

**不要让人工从零填所有字段，而是让人工做“最后 20% 的质量把关”。**

---

## 7. 推荐的执行边界

## 7.1 哪些机器做
- 名称、地址、地理信息
- 菜系、菜单链接、营业时间、设施标签
- 评分、评论数
- 图片候选抓取
- 图片初步分类
- 评论标签抽取
- 生成短描述候选

## 7.2 哪些人工做
- 主图确认
- 封面图 / 章节图确认
- 代表房型是否准确
- 招牌菜是否真的代表这家店
- “为什么选它”是否够像你们品牌口吻
- 有争议字段的纠正

---

## 8. 最终建议

### 现在就可以开始采图片与描述评价数据

但要按规则采，不要无规则地把网上图片和评论一把抓进来。

### 最推荐的方式

**机器采集底座 + 人工精修关键对象**

更具体地说：

- 所有对象先自动建候选池
- 主要酒店 / 主要餐厅 / 主要活动做人工审核
- 次要对象只保留机器筛选结果
- 面向用户展示时优先展示“结构化总结 + 短描述”，少展示长评论摘抄

### 为什么不是纯人工

因为后面城市圈、活动簇、酒店、餐厅会越来越多，纯人工会成为瓶颈。

### 为什么不是纯 AI

因为首图、版权、来源、真实代表性、季节一致性，这些地方纯 AI 最容易翻车。

一句话收口：

**可以开始采；规则必须先定；实现上走“机器采集，人工选主图与修高价值描述”的混合模式最稳。**
