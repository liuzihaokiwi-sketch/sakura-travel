# 四层逻辑补充接入审计 v1

## 结论

这份补充思路整体是对的，而且不是“可有可无的小优化”，里面至少有 **6 个能力值得正式并入四层逻辑**。其中：

- **应立刻并入**：`route_matrix`、`validation/engine`、`offline_eval`、`review_ops/pipeline`、`feedback/distillation`
- **应复用而不是重写**：`swap_engine`、`swap_safety`
- **应保留为种子数据/缓存机制**：`region_router`、`fragment_pipeline`
- **应并入 Layer 3**：`preview_engine`

核心原因不是“项目里已经有这些代码”，而是它们分别补齐了四层模型当前最明显的 4 个缺口：

1. **真实通勤时间缺口**（目前 itinerary-fit 仍偏估算）
2. **输入表单门控缺口**（缺表单级 precheck）
3. **后置质量门缺口**（缺自动评测与回归检测）
4. **反馈飞轮缺口**（只有前向链路，没有后向回流）

## 一、你的这份补充里，哪些判断我认同

### 1. `route_matrix.py` 应该进 Layer 2，而且优先级高

这条判断是对的。你文件里指出，`itinerary_fit_scorer` 的 `sequence_fit` 和 `backtrack_penalty` 现在还是基于 `estimated_transit_min`，没有接 `route_matrix` 真正拿通勤时间。这个缺口会直接影响日骨架和日内适配评分的可信度。fileciteturn35file2

Google Routes API 官方文档本身就把 `Compute Route Matrix` 定义为用于多起点/多终点的距离与时长计算接口；`computeRoutes` 也明确返回 route duration，而且在 traffic-aware 模式下会把交通条件计入时长。也就是说，把通勤时间从“估算字段”升级到“真实 route duration”是有明确技术收益的。citeturn158150search0turn158150search2turn158150search18

### 2. `swap_engine.py` 不该重写，应该吃进 E3

这条也对。你文件里已经指出，现有 `swap_engine` 有 4 维候选排序和跌幅安全检查，本质上就是 E3 想做的 same-corridor / backup 替换能力的成熟版本。fileciteturn35file2

我的建议是：
- 不新写 `suggest_swap()` 的核心逻辑
- 只补一层 adapter，把 `corridor signal`、`meal type`、`locked/tunable` 喂给现有 swap engine
- 再用 `swap_safety` 做 post-check

这样比从零重写更稳。

### 3. `validation/engine.py` 应该放到 Layer 4 的前置门控前面

这条判断非常值得保留。你文件里已经说得很准确：现在 Layer 4 有 `eligibility_gate` 和 `precheck_gate`，但它们都是“实体级”和“风险级”门控，缺的是**表单输入级**门控。fileciteturn35file2turn35file6

也就是说，用户如果提交了结构上不完整、冲突或缺关键信息的表单，系统现在仍可能直接进入 Layer 2。这个会白白浪费生成与评估资源。

### 4. `offline_eval.py` 应该进 Layer 4 的 quality gate

这条判断也对，而且和你现在 E6a 的 shadow write 路线很配。你文件里把它定位成“生成后的自动化质量评估”和“回归检测”，这个位置是对的。fileciteturn35file6

这类离线评测非常适合做：
- 新旧链路对比
- feature flag 切换前验收
- E6a shadow 结果回归检测

### 5. `review_ops/pipeline.py` 和 `feedback/distillation.py` 确实应该并入四层模型

这个补充是这份文档里最重要的部分。你指出当前四层模型只有前置门控、trace 和 freshness，没有后置评审，也没有反馈回流；因此数据从 Layer 1 → Layer 2 → Layer 3 单向流动，没有回路。这个判断我认同。fileciteturn35file1turn35file5turn35file7turn35file8

从工程上说，这意味着系统有生成链路，但还没有真正的“质量飞轮”。

## 二、我会调整的地方

### 1. `swap_safety.py` 更适合放在 Layer 4，而不是 Layer 2 内部

你文件里给了两个位置：Layer 4 或 Layer 2 填充后的 post-check。fileciteturn35file6

我更倾向于：
- **Layer 2** 负责“排得出来”
- **Layer 4** 负责“排出来之后能不能发布/替换/重写”

所以 `swap_safety` 更适合做：
- `secondary_filler` 之后的 post-check
- `swap_engine` 替换结果的 guardrail
- review pipeline 之前的快速结构护栏

也就是把它当成 **post-plan validator**，而不是 filler 自己的一部分。

### 2. `region_router.py` 不建议真的接回主链

你文件里已经倾向于“不接入链路，只迁移矩阵数据到 `CityCircle.fit_profiles`”。这个判断我基本同意。fileciteturn35file6

我只补一句：
- **保留数据，不保留主调用路径**
- 把它当 `city_circle_selector.profile_fit` 的 seed source
- 不要形成两个并行的选圈器

否则 Layer 2 又会重新出现双路决策。

### 3. `fragment_pipeline` 不该继续作为内容复用系统，而应升级成 decision cache

这条你已经提到了，我同意。fileciteturn35file0

但我会再收紧一点：
- 不要缓存整段文案
- 不要缓存最终 HTML 片段
- 只缓存 **同画像 + 同城市圈 + 同主要活动集合** 的骨架级结果

也就是缓存：
- `selected_circle`
- `selected_major_activities`
- `hotel_strategy`
- `day_frame`
- `page_plan`

这比旧的 fragment 复用更适合现在的新链路。

## 三、我建议正式纳入四层逻辑的方式

## Layer 1：数据与知识层

### 建议纳入
- `feedback/distillation.py` 的回写侧
- `region_router` 的 profile 矩阵数据
- `fragment_pipeline` 重构后的 decision cache

### 作用
- 把用户反馈、评审回写、历史高分 skeleton 变成数据层资产
- 不只是存实体真相源，也存“系统学到了什么”

---

## Layer 2：决策与编排层

### 建议纳入
- `route_matrix.py`
- `swap_engine.py`
- `CorridorResolver` + `swap_safety` 的轻接口

### 具体接法

#### 2.1 `route_matrix`
接入：
- `route_skeleton_builder`：校准 `transfer_budget_minutes`
- `itinerary_fit_scorer`：替代 `estimated_transit_min`

这样 `sequence_fit` / `backtrack_penalty` 就从估计值变成真实 route duration。fileciteturn35file2 citeturn158150search0turn158150search18

#### 2.2 `swap_engine`
接入：
- `itinerary_fit_scorer.suggest_swap()`
- `meal_flex_filler.backup_meal_replace()`

不要重写核心排序逻辑。

#### 2.3 `swap_safety`
接法：
- 不是 filler 的核心逻辑
- 而是 filler / swap 后的安全护栏

## Layer 4：验证、评测与运营层

这是这次最该扩充的一层。

### 建议纳入
- `validation/engine.py`
- `offline_eval.py`
- `review_ops/pipeline.py`
- `swap_safety.py`
- QA / Ops 结果回写机制

### 具体接法

#### 4.1 表单级前置门控
`validation/engine` 放在 Layer 2 之前：
- 红灯：不允许进生成链路
- 黄灯：允许进，但挂 warning
- 绿灯：正常继续

#### 4.2 生成后质量门
`offline_eval` 放在：
- E6a shadow write 后
- 新旧结果对比前
- feature flag 放量前

#### 4.3 后置评审
`review_ops/pipeline` 不应停留在“旧 assembler 的 plan_json 评审”。你文件里建议改成读取 `ReportPayloadV2` 或 `skeleton+items`，并把 issues 写回 `generation_decisions`；这个方向我同意。fileciteturn35file3

## 四、我建议的最终优先级

### 第一批：立刻接入
1. `route_matrix` → Layer 2
2. `validation/engine` → Layer 4 前置
3. `offline_eval` → Layer 4 quality gate
4. `review_ops/pipeline` → Layer 4 后置评审

这 4 个投入小、收益大，而且能直接提升“能跑、跑稳、可评”。fileciteturn35file1turn35file2turn35file6

### 第二批：复用已有高级能力
5. `swap_engine` → E3 复用
6. `swap_safety` → post-plan guardrail
7. `preview_engine` → Layer 3 preview 选天

### 第三批：飞轮与沉淀
8. `feedback/distillation` 扩到 entity / cluster / circle
9. `fragment_pipeline` 重构为 decision cache
10. `region_router` 仅迁移 profile 矩阵，不保留主链调用

## 五、最后的客观结论

你的这份补充不是“还能再加一点东西”，而是在纠正四层模型的一个真实短板：

- Layer 2 还缺真实路线时间和成熟替换能力
- Layer 3 还缺 preview 选天与 explain 反哺
- Layer 4 还缺输入门控、后置评审、自动评测、反馈回流
- Layer 1 还没有把这些回流沉淀成真正的数据资产

所以我的最终判断是：

**这批实现里，至少有 6 个应该正式纳入四层逻辑；其中最值得优先做的是 `route_matrix`、`validation/engine`、`offline_eval`、`review_ops/pipeline`、`feedback/distillation`，而 `swap_engine` 应该直接复用，不要重写。** fileciteturn35file0
