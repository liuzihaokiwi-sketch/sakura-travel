# 01｜城市圈到完整攻略生成流程表

面向：程序 AI / 后端 / 排程器 / 前端配置层  
原则：先定圈，再定主活动，再定住法，再定日程，再补餐饮与可变项。  
输出目标：稳定生成 `report_payload_vNext`，允许前端改主要活动、次要活动、酒店策略后局部重排。

---

## 0. 核心对象

| 对象 | 作用 | 最少字段 |
|---|---|---|
| `trip_profile` | 表单标准化结果 | `days` `arrival_airport` `departure_airport` `season` `budget_band` `pace` `must_go[]` `must_avoid[]` `hotel_switch_tolerance` `daytrip_tolerance` |
| `city_circle` | 旅行主决策单元 | `circle_id` `name` `bases[]` `major_activity_pool[]` `secondary_pool[]` `recommended_day_range` `expansion_nodes[]` |
| `major_activity` | 会决定半天/一天、酒店或关键时间窗的活动 | `activity_id` `cluster_id` `duration_band` `base_candidates[]` `reservation_need` `seasonality` `precheck_risk` `live_disruption_risk` |
| `secondary_activity` | 顺路增强，不应推翻主线 | `activity_id` `parent_corridor` `detour_budget_minutes` `can_cut_first` `meal_pairing` `photo_value` |
| `hotel_base_strategy` | 住宿基点决策 | `base_city` `nights` `supports_major_activities[]` `switch_cost_score` |
| `day_frame` | 每日骨架 | `day_index` `sleep_base` `primary_corridor` `main_driver` `secondary_slots[]` `cut_order[]` |
| `fact_snapshot` | 高时效事实快照 | `entity_id` `field` `value` `source_type` `fetched_at` `expires_at` `confidence` |
| `risk_packet` | 活动/当日风险结果 | `precheck_blockers[]` `live_disruption_flags[]` `execution_friction[]` `user_condition_flags[]` |

## 1. 总流程表

| 阶段 | 模块 | 输入 | 输出 | 硬规则 |
|---|---|---|---|---|
| 1 | `normalize_trip_profile` | 表单、历史偏好、机酒约束 | `trip_profile` | 缺关键字段先补默认，不直接排日程 |
| 2 | `select_city_circle` | `trip_profile`、城市圈池 | `selected_circle` `rejected_circles[]` | `<7` 天默认只允许 1 个完整城市圈；跨圈必须解释 |
| 3 | `rank_major_activities` | `trip_profile` `selected_circle.major_activity_pool` | `selected_major_activities[]` | 先定容量，再定具体活动；到达/离开日容量按 0.5 计 |
| 4 | `build_hotel_base_strategy` | `selected_major_activities[]` 机酒约束 | `hotel_base_strategy[]` | 非必要不换酒店；换酒店必须显著降低总通勤或酒店本身是主活动 |
| 5 | `build_route_skeleton` | 圈、主活动、住法 | `day_frame[]` | 每天先有主驱动，再允许补槽 |
| 6 | `fill_secondary_activities` | `day_frame[]` `secondary_pool[]` | 更新后的 `day_frame[]` | 次要活动不能推翻主活动，也不能制造跨圈跳跃 |
| 7 | `fill_meals_shops_flex` | `day_frame[]` 餐厅/商店池 | `day_frame[]` | 餐厅分级填充；备胎优先 |
| 8 | `run_fact_gate` | 选中实体清单 | `fact_snapshot[]` `precheck_blockers[]` | 价格/营业时间/预约规则不写死在 prose |
| 9 | `run_risk_pass` | 行程骨架、事实快照、天气/交通信号 | `risk_packet[]` | 先处理可预判风险，再处理临近/当日扰动 |
| 10 | `assemble_report_payload` | 全部结构化对象 | `report_payload_vNext` | 文案只能解释结构，不能创造新事实 |
| 11 | `render_web_or_pdf` | `report_payload_vNext` | Web/PDF | 共用 schema，不共用表现层 |

## 2. 城市圈选择规则

| 规则 ID | 规则 |
|---|---|
| `CIRCLE_001` | 先按机场、总天数、换酒店意愿、日归容忍度筛圈，再看“想去哪些点” |
| `CIRCLE_002` | 少于 7 天默认只选 1 个完整城市圈 |
| `CIRCLE_003` | 7–9 天允许 `1 主圈 + 0~1 轻扩展圈`，扩展圈不得吃掉主圈核心活动 |
| `CIRCLE_004` | 10 天以上才默认开放 2 个完整城市圈 |
| `CIRCLE_005` | 若用户 `must_go` 已指向多个圈，必须计算“跨圈损失”并显式提示 |
| `CIRCLE_006` | 城市圈评分优先级：`must_go fit > airport fit > season fit > pace fit > transfer cost > hotel stability` |

### 推荐评分字段

```ts
interface CircleScore {
  must_go_fit: number;
  airport_fit: number;
  season_fit: number;
  pace_fit: number;
  transfer_penalty: number;
  hotel_instability_penalty: number;
  final_score: number;
}
```

## 3. 主要活动选择规则

### 3.1 主要活动定义

满足下列条件中的 2 项及以上，才可升级为 `major_activity`：

| 条件 | 判定 |
|---|---|
| `dedicated_trip_value` | 值得专门去一个区域、半天或一天 |
| `time_window_dependency` | 依赖清晨 / 日落 / 夜场 / 花期 / 季节窗口 |
| `hotel_or_route_impact` | 会改变住法、跨区顺序或当天主线 |
| `reservation_dependency` | 需要预约、购票、排队策略 |
| `experience_uniqueness` | 属于强识别度体验，适合作为这趟旅行“核心记忆点” |

### 3.2 数量容量

| 行程长度 | 平衡型 | 紧凑型 | 上限建议 |
|---|---:|---:|---:|
| 3–4 天 | 2–3 | 3–4 | 4 |
| 5–6 天 | 3–4 | 4–5 | 5 |
| 7–8 天 | 4–5 | 5–6 | 6 |
| 9–10 天 | 5–6 | 6–7 | 7 |

### 3.3 主要活动最少字段

```ts
interface MajorActivity {
  activity_id: string;
  title: string;
  cluster_id: string;
  circle_id: string;
  city_node: string;
  duration_band: "half_day" | "full_day" | "overnight";
  best_time_window: string[];
  reservation_need: "none" | "recommended" | "required";
  seasonality: "all_year" | "spring" | "summer" | "autumn" | "winter" | "event_based";
  base_candidates: string[];
  paired_secondary_tags: string[];
  precheck_risk: RiskLevel;
  live_disruption_risk: RiskLevel;
  execution_friction: RiskLevel;
  user_condition_risk: RiskLevel;
}
```

## 4. 酒店基点策略

| 规则 ID | 规则 |
|---|---|
| `HOTEL_001` | 主要活动确定后，立刻决定酒店基点；不要排完天再回头找酒店 |
| `HOTEL_002` | 同一城市圈内，默认一个基点支撑 3–4 晚 |
| `HOTEL_003` | 只有“连续两天合计节省 ≥90 分钟通勤”或“酒店本身就是主要活动”才允许换酒店 |
| `HOTEL_004` | 最后一晚可为赶飞机/新干线友好而单独优化 |
| `HOTEL_005` | 酒店必须至少服务 2 天；否则判为低价值换住 |

### 住宿对象

```ts
interface HotelBaseStrategy {
  base_city: string;
  nights: number;
  hotel_role: "core_base" | "airport_buffer" | "experience_hotel";
  supports_major_activities: string[];
  avg_transfer_load: number;
  switch_cost_score: number;
  justification: string[];
}
```

## 5. 次要活动与补槽规则

| 规则 ID | 规则 |
|---|---|
| `SECONDARY_001` | 次要活动只允许在已确定的 `primary_corridor` 内补槽 |
| `SECONDARY_002` | 平衡型一天默认 `1 主活动 + 1~2 次要活动` |
| `SECONDARY_003` | 轻松型一天默认 `1 主活动 + 0~1 次要活动` |
| `SECONDARY_004` | 紧凑型一天允许 `1 主活动 + 2 次要活动`；仅偶发允许 `2 个轻量主活动` |
| `SECONDARY_005` | 每个次要活动都必须标记 `can_cut_first=true/false` |
| `SECONDARY_006` | 次要活动优先补齐体验差异：吃 / 看 / 拍 / 逛 / 休息，避免同质堆叠 |

### 次要活动字段

## 6. 餐厅 / 商店 / 可变活动分层

| 层级 | 定义 | 对排程影响 |
|---|---|---|
| `destination_meal` | 值得为它调整时间或预约的餐厅 | 可占主/强次要槽 |
| `route_meal` | 顺路正餐 | 不得推翻主线 |

| `flex_shop` | 顺路购物点 | 默认尾插 |

## 7. 风险模型（关键）

### 7.1 风险分类

> 先把“可提前规避的风险”和“真正的突发风险”分开。  
> 像迪士尼闭园、USJ 已知维护、展馆计划休馆、季节活动结束、门票售罄，这些都不算突发风险；它们属于 **预检型风险**，必须在生成前或锁线前规避。

| 风险层 | 含义 | 例子 | 处理时机 |
|---|---|---|---|
| `precheck_blocker` | 可以通过官方日历、营业时间、维护公告、预约规则提前知道的阻断 | 已公布闭园、计划休馆、季节结束、预约售罄、营业日不匹配 | 生成前 / 锁线前 |
| `live_disruption_risk` | 接近出发或当天才可能触发的扰动 | 暴雨警报、列车停运、索道停运、临时交通中断、现场临时限流 | 出发前 72h / 24h / 当天 |
| `execution_friction` | 不会让活动彻底失效，但会显著伤害体验 | 极长排队、长坡、步行过量、晚归、换乘复杂 | 排程阶段 |
| `user_condition_risk` | 与同行人群或身体条件相关 | 婴儿车困难、老人爬坡、孕期、晕车、儿童夜场疲劳 | 画像阶段 |

### 7.2 风险字段

```ts
type RiskLevel = "none" | "low" | "medium" | "high";

interface RiskStatus {
  precheck_risk: RiskLevel;
  live_disruption_risk: RiskLevel;
  execution_friction: RiskLevel;
  user_condition_risk: RiskLevel;
  has_precheck_blocker: boolean;
  needs_same_day_backup: boolean;
  backup_strategy_ids: string[];
}
```

### 7.3 风险处理顺序

| 顺序 | 模块 | 目标 |
|---|---|---|
| 1 | `precheck_gate` | 清掉“已知不能去”的活动 |
| 2 | `skeleton_repair` | 用同类活动修补主线，不动整圈结构 |
| 3 | `live_monitor_attach` | 给高波动活动加 `T-72h/T-24h/day-of` 检查点 |
| 4 | `same_day_backup_fill` | 为高实时扰动活动绑定同走廊替补 |
| 5 | `user_condition_adjust` | 根据体力/年龄/儿童/孕期降密度 |

### 7.4 前端可编辑字段

| 字段 | 用途 |
|---|---|
| `is_major_locked` | 锁定主活动，不允许自动替换 |
| `is_secondary_locked` | 锁定次要活动 |
| `manual_remove_reason` | 人工删掉活动的原因 |
| `manual_promote_to_major` | 将次要活动提升为主要活动 |
| `manual_demote_to_secondary` | 将主要活动降为次要活动 |
| `risk_override_note` | 运营/编辑手工覆盖风险说明 |
| `show_risk_badge` | 是否在前端显示风险标签 |
| `show_backup_option` | 是否展示替代活动 |

## 8. 日级骨架

```ts
interface DayFrame {
  day_index: number;
  sleep_base: string;
  primary_corridor: string;
  secondary_corridor?: string;
  main_driver: string;
  secondary_slots: string[];
  meal_slots: string[];
  flex_slots: string[];
  cut_order: string[];
  return_anchor: string;
  risk_packet: RiskStatus[];

```

### 日级校验

| 校验 ID | 规则 |
|---|---|
| `DAY_001` | 同一天不得出现重复实体 |
| `DAY_002` | 标题区域必须覆盖当日 70% 以上有效时长 |
| `DAY_003` | 到达日 / 离开日不得塞满高依赖预约点 |
| `DAY_004` | 每天必须有 `cut_order` |
| `DAY_005` | 主要活动之间必须保留合理缓冲，不允许连续高波动叠加 |
| `DAY_006` | 高 `live_disruption_risk` 活动必须绑定同走廊替补 |

## 9. 输出到前端的最小协议

```ts
interface FrontendEditablePlan {
  selected_circle_id: string;
  selected_major_activity_ids: string[];
  selected_secondary_activity_ids: string[];
  hotel_base_strategy: HotelBaseStrategy[];
  day_frames: DayFrame[];
  facts_registry: FactSnapshot[];
  plan_edit_state: {
    user_locked_fields: string[];
    operator_notes: string[];
    last_manual_edit_at?: string;
  };
}
```

### 前端动作

| 动作 | 后端响应 |
|---|---|
| 替换主要活动 | 只重跑 `hotel_base_strategy + route_skeleton + downstream fill` |
| 增删次要活动 | 只重跑单日补槽与餐饮补位 |
| 切换酒店策略 | 重跑骨架，不重算城市圈 |
| 锁定某天结构 | 下游只允许局部微调 |
| 隐藏风险 badge | 只影响展示，不影响结构 |

## 10. 推荐实现顺序

1. 先落 `city_circle` / `major_activity` / `secondary_activity` / `hotel_base_strategy` 四类 schema。  
2. 再写 `select_city_circle`、`rank_major_activities`、`build_hotel_base_strategy`。  
3. 再写 `build_route_skeleton` 和 `fill_secondary_activities`。  
4. 再接 `precheck_gate`、`facts_registry`、`live_monitor_attach`。  
5. 最后才接文案装配与前端编辑器。  

---

## 11. 风险语义基线（供程序使用）

- 官方运营日历 / 园区时间表 / 维护公告属于 **预检型信号**。  
- 天气预警、铁路运行信息、索道或山地设施即时停运、临时限流属于 **实时扰动信号**。  
- 预检型信号用于“提前不排/提前改排”；实时扰动信号用于“保留 fallback / 临近检查 / 当天替换”。
