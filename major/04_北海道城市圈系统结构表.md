
# 04｜北海道城市圈系统结构表

面向：程序 AI / 城市圈建模 / 前端配置层  
目标：把“北海道核心城市圈”固化成可排程、可编辑、可扩展的数据结构。  
默认版本：`hokkaido_core_circle_v1`

---

## 1. 圈定义

```ts
interface CityCircle {
  circle_id: string;
  name: string;
  region: string;
  core_bases: BaseNode[];
  daytrip_nodes: DaytripNode[];
  major_activity_clusters: MajorActivityCluster[];
  secondary_activity_clusters: SecondaryActivityCluster[];
  default_hotel_strategies: HotelStrategyPreset[];
  recommended_day_range: [number, number];
  expansion_rules: ExpansionRule[];
}
```

### 1.1 建议圈边界

| 字段 | 值 |
|---|---|
| `circle_id` | `hokkaido_core_circle_v1` |
| `name` | 北海道核心城市圈 |
| `region` | Hokkaido |
| `core_bases` | 札幌 / 登别温泉(条件型) / 函馆(扩展型) |
| `daytrip_nodes` | 小樽 / 札幌近郊 / 富良野美瑛(条件型) |
| `optional_expansion_nodes` | 洞爷湖 / 二世谷 / 旭川 / 函馆 |
| `recommended_day_range` | `5~9` 天 |
| `hard_cap_without_expansion` | `8` 天 |

### 1.2 默认策略

| 天数 | 默认住法 | 默认核心 |
|---|---|---|
| 4–5 天 | 单基点（札幌）+ 1 个体验节点 | 札幌 + 小樽 + 登别/洞爷择一 |
| 6–7 天 | 双基点（札幌 + 温泉/函馆） | 札幌 + 小樽 + 登别 + 富良野美瑛择优 |
| 8–9 天 | 双基点 / 三基点条件开放 | 可加入函馆或富良野美瑛的更重版本 |

## 5. 北海道圈活动兼容矩阵

| 主活动 | 最适配次要活动 | 不建议强拼 |
|---|---|---|

## 6. 风险标签系统（给活动簇）

### 6.1 字段

```ts
interface ActivityRiskTags {
  precheck_risk: RiskLevel;
  live_disruption_risk: RiskLevel;
  execution_friction: RiskLevel;
  user_condition_risk: RiskLevel;
  known_schedule_dependency: boolean;
  day_of_disruption_possible: boolean;
  backup_same_corridor_required: boolean;
}
```

### 6.2 分类原则

- 雪场营业、缆车季节、花期主题、旅馆售罄，属于前置可规避风险。  
- 暴雪、低能见度、道路结冰、强风导致的体验变化，属于实时扰动风险。  
- 北海道圈的 `winter_snow_resort / nature_hiking_green / furano_biei_scenic` 默认需要 fallback。

## 8. 北海道圈默认生成模板

### 4–5 天

- 必选主活动：`札幌都市美食 / 小樽港町`
- 画像命中后再加入：`登别 or 洞爷`
- 不默认加入：`函馆 + 富良野美瑛双扩展`

### 6–7 天

- 默认 4–5 个主活动簇
- 札幌必须承接 2–3 天
- `登别 / 富良野美瑛 / 函馆` 三者择二，不建议全塞

### 8–9 天

- 可开放 `札幌 + 登别 + 函馆` 三基点
- 夏季可以替换一个扩展为 `富良野美瑛`
- 冬季雪场只在明确画像下开放

interface HokkaidoCircleRegistry {

## 10. 来源锚点（供程序维护）

- JNTO 常见北海道线以札幌、小樽、登别、函馆、富良野美瑛为核心骨架。
- Hokkaido Rail Pass 与官方经典线路支持把札幌作为主基点，向温泉、港城、花田方向扩展。
