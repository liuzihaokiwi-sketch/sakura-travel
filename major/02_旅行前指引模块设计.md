# 02. 旅行前指引模块设计

> 面向：AI 程序员 / 内容装配 / QA / 运营  
> 目标：把“旅行前指引”从零散文案升级为**可结构化生成、可按目的地补槽、可按用户画像触发**的正式模块。  
> 定位：回答“出发前要准备什么”。

---

## 1. 模块目标

“旅行前指引”不等于一堆提醒。

它应该帮助用户在出发前完成下面三件事：

1. **准备齐**：证件、支付、通讯、药品、保险、预订、行李。
2. **查清楚**：入境要求、健康要求、活动要求、当地法律与限制。
3. **不临门翻车**：避免因为证件有效期、处方药、保险不覆盖、网络没准备、预约没确认而影响出发。

因此，旅行前指引必须满足四个属性：

- **可执行**：用户看完知道要做什么
- **可排序**：知道先做哪件、后做哪件
- **可动态补槽**：不同国家/活动/季节显示不同内容
- **可门控**：高风险、高成本、高时效事项必须被前置

---

## 2. 内容边界

### 属于旅行前指引的内容

- 证件与入境要求
- 签证 / ETA / eGate / 海关限制
- 保险与医疗准备
- 处方药与常备药
- 通讯 / 网络 / 离线地图 / 常用 App
- 支付、现金、银行卡、取现
- 交通准备
- 预约确认
- 行李准备
- 紧急联系人 / 行程共享
- 临出发检查清单

### 不属于旅行前指引的内容

- 今日行程 Plan B
- 当天临场避坑
- 活动期间的现场安全守则
- 已在目的地之后才会触发的注意事项

这些应归到“旅行注意事项 / 动态风险模块”。

---

## 3. 输出形式

建议 `pretrip_guide` 采用**分层 + 分时点**输出，不要再写成一页散文。

### 3.1 用户侧阅读顺序

1. **必须先完成**（高优先级）
2. **建议提前 7–14 天准备**
3. **建议出发前 48 小时确认**
4. **出发当天再检查一次**

### 3.2 程序侧数据结构

```ts
interface PreTripGuideSection {
  summary: PreTripSummary;
  must_finish: GuideTask[];
  prepare_7d: GuideTask[];
  confirm_48h: GuideTask[];
  departure_day: GuideTask[];
  optional_modules: GuideModule[];
  source_snapshot_ids: string[];
}

interface GuideTask {
  task_id: string;
  title: string;
  body: string;
  priority: "critical" | "high" | "medium" | "low";
  time_bucket: "must_finish" | "prepare_7d" | "confirm_48h" | "departure_day";
  category: string;
  check_type: "static" | "dynamic" | "conditional";
  completion_hint?: string;
  dependency_ids?: string[];
  source_snapshot_ids?: string[];
}
```

## 4. 必须包含的 10 个基础模块

### 4.1 证件与入境

固定提醒：

- 护照/旅行证件有效期是否满足入境要求
- 是否需要签证、电子签、ETA、入境卡、出入境申报
- 是否需要返程/后续行程证明
- 是否需要酒店地址、联系人、保险证明、资金证明
- 是否存在转机国家的额外要求

程序规则：

- 这是最高优先级模块。
- 任何跨境行程都必须出现。
- 如存在多国、多段转机，应拆成多条任务，不要合并成一句话。

### 4.2 旅行保险

固定提醒：

- 是否已购买旅行保险
- 是否覆盖目的地、时长、计划活动、既往病史、财物损失、取消改签
- 是否包含境外医疗与医疗转运 / medical evacuation
- 是否包含自驾、滑雪、徒步、潜水等活动附加险

程序规则：

- 跨境行程默认显示。
- 一旦行程存在高风险活动，自动升级到高优先级。

### 4.3 药品与健康准备

固定提醒：

- 处方药带原包装和处方复印件
- 尽量同时携带通用名 / generic name
- 某些国家对精神类、麻醉类、含特定成分药物有携带限制
- 如有慢病、过敏、孕期、儿童同行，要补定制建议
- 根据目的地和活动查看疫苗、预防药、健康通知

程序规则：

- 所有行程默认显示基础提醒。
- 如用户填报长期用药、孕期、儿童、老人同行，提升优先级。

### 4.4 网络与通讯

固定提醒：

- eSIM / SIM / Pocket Wi‑Fi 方案
- 离线地图、翻译、交通、航班、酒店 App
- 出发前把重要地址和预订号离线保存
- 手机电量、充电器、转换插头、移动电源

程序规则：

- 出境游默认显示。
- 境内游只保留离线地图、酒店地址、移动电源等简版提示。

### 4.5 支付与现金

固定提醒：

- 主支付方式、备用支付方式、少量现金
- 是否支持银行卡 / 数字钱包 / 当地交通卡
- 是否需要提前开通境外刷卡、ATM、免密、小额支付
- 高消费活动是否需信用卡担保

程序规则：

- 任何有跨境或乡村/山区/小店场景的行程都需要。
- 如国家/地区现金依赖高，则提升排序。

### 4.6 交通准备

固定提醒：

- 机场交通和首晚到达方案
- 是否建议提前开通交通卡 / 下载交通 App
- 是否有自驾、铁路票、长途巴士、船票等提前准备事项
- 是否需要国际驾照翻译件 / IDP / 当地驾照认证

程序规则：

- 有落地晚到、跨城、自驾、离岛、山区交通时必须升级。

### 4.7 预订确认

固定提醒：

- 机票 / 酒店 / 重点餐厅 / 门票 / 接送机 / 一日团 / 活动预约二次确认
- 关键预约的取消规则、迟到规则、集合地点
- 需要实名、护照、信用卡、穿着要求的项目

程序规则：

- 任何 `booking_level = must_book / should_book` 的项目都应自动进入这一模块。

### 4.8 行李准备

固定提醒：

- 天气与温差
- 鞋子与外套
- 雨具、防晒、保暖
- 个人卫生用品
- 小药包
- 备用收纳袋
- 活动专用装备

程序规则：

- 行李模块永远显示。
- 具体项目由季节、天气、活动类型补槽。

### 4.9 行程共享与紧急联系人

固定提醒：

- 把航班、酒店、主要交通、护照首页、保险单号发给同行或家人
- 记录当地报警、急救、使领馆、保险救援电话
- 设定至少一位国内紧急联系人

程序规则：

- 出境游默认显示。
- 独行、夜间抵达、偏远地区行程自动升优先级。

### 4.10 临出发检查

固定提醒：

- 证件是否带齐
- 设备是否充满电
- 重要票券是否已下载
- 药物是否随身放置
- 现金 / 卡 / 备用支付是否准备
- 接送与集合时间是否确认

程序规则：

- 固定为出发当天 bucket。

## 5. 建议的时间分桶

### A. 必须先完成

适合展示：

- 证件与签证
- 保险
- 疫苗 / 处方药限制
- 高风险活动附加准备
- 自驾资格

### B. 建议提前 7–14 天准备

适合展示：

- eSIM / 网络方案
- 交通 App / 离线地图
- 外币 / 支付方式
- 预约确认
- 行李与装备
- 紧急联系人同步

### C. 建议出发前 48 小时确认

适合展示：

- 航班值机
- 酒店与接送
- 天气与交通中断风险
- 重点预约时段
- 身份证件、护照、保险、处方复印件

### D. 出发当天再检查一次

适合展示：

- 证件 / 票券 / 手机 / 充电 / 药物 / 钱包 / 地址 / 联络方式

## 6. 动态补槽规则

旅行前指引必须支持四类动态补槽。

### 6.1 目的地补槽

根据国家/地区补：

- 入境要求
- 签证 / ETA / 健康申报
- 药品携带限制
- 现金与支付习惯
- 插头、电压、网络、实名要求
- 特定法律 / 海关规则

### 6.2 活动补槽

根据行程活动补：

- 徒步 / 爬山：鞋、头灯、补水、路线、离线地图
- 高海拔：缓升、高反药、保暖、防晒
- 滑雪：雪镜、手套、头盔、保险附加险
- 潜水：潜水资质、禁飞时间、装备确认
- 自驾：驾照、IDP、保险、取还车规则
- 海边 / 浮潜：防晒、拖鞋、防水袋、救生设施

### 6.3 用户画像补槽

根据用户属性补：

- 儿童同行：药物、推车、辅食、儿童安全座椅
- 老人同行：慢病药物、休息节奏、医疗可及性
- 孕期：医疗、保险、飞行限制
- 独行：夜间安全、共享定位、离线备份
- 长住 / 数字游民：备用支付、网络冗余、文件备份

### 6.4 时效风险补槽

根据临近出发的动态事实补：

- 旅行警示 / 健康通知
- 大规模天气 / 自然灾害
- 交通罢工 / 交通中断
- 景点停业 / 预约变更

## 8. 文案策略

### 8.1 一律写成“可执行动作”

不写：

- 注意签证
- 注意保险
- 注意带药

要写：

- 检查护照有效期是否满足本次行程要求，并确认转机地是否有额外入境或过境材料要求。
- 确认旅行保险覆盖目的地、旅行时长、已预订活动和医疗转运。
- 长期服用的药物请保留原包装，并准备处方复印件与英文通用名。

### 8.2 能前置的不要后置

凡是会导致**无法出发、无法入境、无法进行高价预约活动**的事项，必须前置到 `must_finish`。

### 8.3 不要把国家特例写成通用结论

例如：

- 不是所有国家都要求 6 个月有效期
- 不是所有国家都必须打印纸质材料
- 不是所有地方都大量使用现金
- 不是所有活动都需要额外保险

因此所有国家特例都必须由动态补槽给出。

## 9. 质量门控

### 必须通过的规则

1. 跨境行程必须出现“证件与入境”“保险”“药品与健康”“行程共享与紧急联系人”。
2. 有高风险活动时，必须出现活动专用准备项。
3. 有预约依赖时，必须出现“预订确认”任务。
4. 所有动态事实必须有来源快照。
5. 如无法确认事实，应降级为中性文案：
   - “请以官方入境与承运方要求为准”
   - “请在出发前再次确认”

## 10. 建议对接的数据源类型

### 一级优先：官方 / 主事实源

- 目的地官方入境与签证信息
- 使领馆 / 外交部门旅行建议
- CDC / WHO 旅行健康信息
- 官方交通、航空、景点、活动预约页

### 二级优先：平台 / 商业事实源

- 航空公司 App
- 酒店预订平台订单页
- 活动预订平台
- 地图与交通平台

### 三级优先：内部事实层

- 已缓存快照
- 订单信息
- 用户上传的确认信息
- 内部质检标记

## 11. 推荐默认内容清单（可直接做静态底座）

### must_finish 默认底座

- 检查证件与入境要求
- 确认旅行保险覆盖范围
- 检查处方药与健康要求

### prepare_7d 默认底座

- 准备网络与常用 App
- 准备支付与少量现金
- 确认交通与关键预订
- 整理行李与活动装备
- 共享行程与紧急联系人

### confirm_48h 默认底座

- 复查天气、交通、预约状态
- 下载离线票券与地址
- 再核对证件、保险、药物

### departure_day 默认底座

- 带好证件、票券、药物、手机、钱包、充电设备

---

## 12. 一句话实现原则

> **旅行前指引不是“多写一页准备事项”，而是把所有会影响能否顺利出发和顺利落地的关键准备动作，按优先级和时点结构化地交给用户。**
