# 最终报告数据缺口与阶段模板存储建议 v1

## 1. 目标

回答两个问题：
1. 为了稳定生成新的最终攻略，当前数据层还缺哪些关键数据。
2. 生成链路中哪些阶段性结果值得存数据库做复用，哪些不值得长期存，哪些更适合缓存或对象存储。

本文默认基于当前已确定方向：
- 从“模板驱动”改为“城市圈驱动”
- 生成链路改为：画像标准化 → 城市圈选择 → 主要活动 → 酒店基点 → 日骨架 → 次要活动/餐厅填充 → 报告 payload → 页型渲染
- 最终攻略采用固定页型系统，而不是长 section 自然流分页

---

## 2. 先给结论

不要走两个极端：
- 不是“所有中间结果都不存，现场每次重算”
- 也不是“每一步都永久落库，最后数据库里全是难维护的派生垃圾”

**推荐策略：混合式存储。**

### 应长期存数据库的
这些是未来会复用、回溯、重生成、评测、人工修改都要依赖的：
- 标准化后的 trip profile
- 城市圈 registry / 活动簇 / 酒店策略 preset / 角色映射
- 选择结果（圈、主要活动、酒店策略、日骨架）
- 最终 itinerary_days / itinerary_items
- report_payload_vNext（结构化真相源）
- page_plan（页序与页型真相源）
- 版本号与 trace
- 反馈与回库信息

### 只建议短期缓存或保留最近版本的
这些会频繁变化、可重算、但重算成本不低：
- candidate ranking 结果
- page_view_model
- Satori hero 图
- 渲染前聚合数据
- 调试用 explain trace 详细矩阵

### 不建议长期落库的
这些要么噪音太大，要么强依赖实时数据，要么长期价值低：
- 全量原始候选池逐项分数明细（每次生成全量永久保留）
- live risk 的每次轮询原始结果
- 每页最终 HTML 的每次中间版本
- 短命 render cache

### 不要把“整份最终攻略全文”当默认复用模板
真正应复用的是：
- 片段
- 活动簇
- 酒店策略 preset
- page blueprint
- report payload / page plan 的结构层

而不是把一份完整旧攻略整包拿去下次继续套。

## 3. 当前为了生成最终报告，最缺的不是文案，而是数据

### 3.1 画像标准化层缺口（P0）
这一层决定后面城市圈、酒店、页型是否站得住。

当前最该补的字段：
- normalized_origin / normalized_destination
- arrival_node / departure_node
- arrival_day_shape（到达日是否可玩半天）
- departure_day_shape（离开日是否只能轻安排）
- trip_nights / trip_days
- budget_level + budget_flex
- pace_preference（慢 / 平衡 / 特种兵）
- hotel_switch_tolerance
- day_trip_tolerance
- queue_tolerance
- weather_risk_tolerance
- food_priority / photo_priority / shopping_priority / hotel_priority
- must_go_entities[]
- must_avoid_entities[]
- hard_constraints[]
- celebration_flags（纪念日、生日、求婚等）
- mobility_notes（老人、小孩、孕期、拖箱）

### 为什么它是 P0
因为城市圈选择、主要活动数量、酒店基点策略、最后一晚安排、主要餐厅是否值得独立页，全都依赖这一层。

## 4. 城市圈与活动系统缺口（P0）

### 4.1 当前缺少的不是“更多景点”，而是“城市圈真相源”

- circle_name
- hub_nodes[]
- day_trip_nodes[]

- airport_fit_rules

- transfer_penalty_rules
- style_tags[]

#### corridors / subregions
- corridor_id
- circle_id
- corridor_name
- parent_hub
- traversal_cost_hint
- typical_visit_slots

#### activity_clusters
比单景点更重要，建议独立建模。
- activity_cluster_id
- circle_id
- cluster_type（major / secondary / mixed）
- title
- anchor_entities[]
- required_time_units
- ideal_time_window
- booking_intensity
- seasonality
- weather_sensitivity

#### circle_entity_roles
不要把实体永远写死成 major / secondary。
应该存“实体在某个城市圈、某条玩法下的角色”。

- entity_id
- role_type（major / secondary / meal / photo / hotel_anchor / fallback）
- corridor_id
- priority_band
- replaceability

### 为什么它是 P0
因为你要生成的是“城市圈驱动”的报告。没有这层，后面 report_payload 和页型系统都只能继续吃旧模板。

## 6. 风险与可执行性层缺口（P0）

### 6.1 现在必须明确拆两层风险

#### precheck_risk（前置可规避）
生成前就该消费，不该拖到最终报告里才发现。
- 已知闭园 / 固定休馆
- 已知季节不对
- 已知售罄
- 已知维护 / 停运
- 已知不在营业窗口
- 已知超过用户预算 / 时间承受边界

#### live_disruption_risk（临近/当天扰动）
挂监控和 fallback，不作为长久主数据真相源。
- 天气预警
- 强风 / 海况
- 山路封闭
- 大面积交通延误
- 低能见度

### 6.2 这层还缺的数据
- precheck_status
- precheck_checked_at
- precheck_reason_codes[]
- live_monitor_required
- live_monitor_checkpoints[]
- fallback_corridor_id
- fallback_cluster_id
- cut_first_if_disrupted

## 10. trace / 版本 / 评测层缺口（P0-P1）

这一层不直接给用户看，但不做，后续无法复盘与复用。

### 必须记录的版本
- profile_version
- fragment_catalog_version
- circle_registry_version
- hard_rules_version
- scorer_version
- planner_version
- renderer_version
- grader_version

### 必须保留的 trace
- normalized_profile_trace
- circle_selection_trace
- major_ranking_trace
- hotel_strategy_trace
- day_frame_trace
- secondary_fill_trace
- meal_fill_trace
- ai_delta_trace
- quality_gate_trace

### 为什么它是 P0
因为你后面一定会问：
- 为什么这次圈选错了？
- 为什么这天太挤？
- 为什么这个酒店会被选中？
- 为什么用户改了偏好后没有明显变化？

没有 trace，排查会非常慢。

## 11. 哪些阶段性模板适合存数据库

下面按阶段给建议。

---

### Stage A：标准化画像结果
#### 产物
`normalized_trip_profile`

#### 建议
**长期存数据库，版本化。**

#### 原因
- 后续所有选择都依赖它
- 用户正式修改、微调、客服补充时都要复用
- 评测也要复盘这一步

#### 最低字段
- trip_request_id
- normalized_payload
- source_profile_id
- profile_version
- generated_at

---

### Stage B：城市圈选择结果
#### 产物
`circle_selection_result`

#### 建议
**长期存数据库。**

#### 原因
- 决策解释页会直接用到
- “没选哪些圈以及原因”能形成强定制感
- 后续用户改天数或机场时，可以只重跑这一步起步

#### 最低字段
- selected_circle_ids[]
- rejected_circle_ids[]
- reason_codes_by_circle
- circle_registry_version
- planner_version

---

### Stage C：主要活动选择结果
#### 产物
`major_activity_plan`

#### 建议
**长期存数据库。**

#### 原因
- 它决定酒店、页数、重点专题页
- 是最值得复用的“高价值中间层”
- 后续用户改风格时，经常只需从这层回溯

#### 附加建议
不要只存选中的；最好同时存：
- selected_major_ids[]
- near_miss_major_ids[]
- reject_reason_codes

---

### Stage D：酒店策略结果
#### 产物
`hotel_base_strategy`

#### 建议
**长期存数据库。**

#### 原因
- 是大路线锚点
- 用户后续微调酒店时，不必整单重做
- 酒店策略本身很适合做 preset 回库

#### 进一步可复用模板
可以抽出：
- `hotel_strategy_presets`
- `circle_base_patterns`

---

### Stage E：日骨架结果
#### 产物
`day_frame[]`

#### 建议
**长期存数据库。**

#### 原因
- 这是“结构真相源”
- 每日执行页、餐厅页、拍摄页都依赖它
- 自助微调本质上就是改骨架再重填

#### 注意
这里不要只当临时对象；它应该成为 itinerary_days 的上游语义层。

---

### Stage F：次要活动 / 餐厅填充结果
#### 产物
`itinerary_days + itinerary_items`

#### 建议
**长期存数据库。**

#### 原因
- 这是目前系统已成熟的下游格式
- 后续 report_generator、微调、QA、客服都能直接消费

#### 是否还需要单独存 filler trace
需要，但不一定永久保留全量。建议：
- 结构化结果长期存
- 候选排序明细短期缓存 / 最近版本存

---

### Stage G：report_payload_vNext
#### 产物
结构化最终报告 payload

#### 建议
**长期存数据库，且应作为“最终报告结构真相源”。**

#### 原因
- Web 和 PDF 都要消费它
- 后期改页型或重渲染时无需重跑规划链路
- 人工修改也更安全

#### 备注
比起只存渲染后的 HTML，这层更值得存。

---

### Stage H：page_plan
#### 产物
`page_plan[]`

#### 建议
**长期存数据库。**

#### 原因
- 它是“有哪些页、以什么顺序、哪些对象独立占页”的真相源
- 页型系统要稳定，必须把它当成结构资产，而不是渲染时临时算

#### 适合回库成模板的内容
- page blueprints by trip shape
- chapter/page composition presets
- 7天/10天/14天不同页数策略

---

### Stage I：page_view_model
#### 产物
一页一个 view model

#### 建议
**不建议全部长期永久存。建议：缓存或只保留当前激活版本。**

#### 原因
- 它过于贴近前端模板
- 一旦页组件改版，很容易失效
- 可以从 report_payload + page_plan 再生成

#### 什么时候值得存
- 导出前稳定版本
- 用户正在看的当前版本
- 人工审稿过程中的冻结版

---

### Stage J：最终 HTML / PDF / 图片产物
#### 产物
- html snapshot
- pdf
- satori hero / share image

#### 建议
**产物要存，但更适合对象存储；数据库只存 metadata。**

#### 数据库保留字段
- asset_id
- report_version_id
- asset_type
- object_storage_key
- checksum
- created_at

#### 原因
- 正式交付需要可回溯
- PDF 导出成本不算低
- 但二进制大文件不适合塞业务主表

---

## 12. 哪些更适合做“模板回库”

不是每个阶段都适合当模板。

### 最适合回库复用的
#### 1. 活动簇模板
例如：
- 京都东山半日线
- 奈良公园古都线
- 箱根温泉 + 湖景线

#### 2. 酒店策略 preset
例如：
- 东京 5 天单基点
- 关西 7 天京都大阪双基点
- 最后一晚机场友好策略

#### 3. 页面蓝图模板
例如：
- 5 天轻奢情侣版页序
- 7 天东京单圈平衡版页序
- 12 天双城市圈章节版页序

#### 4. 决策理由模板
例如：
- 为什么不换酒店
- 为什么把高级餐放收尾日
- 为什么某景点不进主线

#### 5. detail page block 模板
例如：
- 酒店页 block skeleton
- 餐厅页 block skeleton
- 拍摄页 block skeleton

### 不建议直接回库当模板的
- 整份完整旧攻略
- 每页最终 prose 全文
- 全量候选排序明细

---

## 13. 哪些不存更方便

### 13.1 不建议长期存的典型对象
- live risk monitor 每次轮询原始结果
- every candidate every score 的全量明细永久表
- 每次微调后所有 page_view_model 历史版本
- render 过程中的中间 HTML 片段
- 调试过程的临时 explain 文本

### 13.2 原因
- 量大
- 高度可重算
- 真实复用率低
- 会污染数据库，增加维护成本

## 15. 是否比“不存储更方便”更优

### 我的结论
**不是全不存更方便。真正更方便的是：只把“能复用、能解释、能重渲染、能回溯”的层存下来。**

### 最稳的边界
#### 必存
- normalized_trip_profile
- circle_selection_result
- major_activity_plan
- hotel_base_strategy
- day_frame
- itinerary_days / itinerary_items
- report_payload_vNext
- page_plan
- export asset metadata
- version + trace summary

#### 缓存即可
- page_view_model
- 候选排序矩阵
- hero image render 结果
- live risk 轮询结果

#### 不建议长期存
- 渲染中间态
- 全量无用 debug 文本
- 临时 HTML 片段

---

## 16. 推荐落地顺序

### 第一批（立刻补，直接影响最终报告能不能成立）
1. normalized_trip_profile 扩字段
2. city_circles / activity_clusters / circle_entity_roles
3. hotel_strategy_presets
4. day_frame 扩字段
5. report_payload_vNext 扩字段
6. page_plan 落库

### 第二批（让“定制感”和“设计感”真的成立）
1. preference_fulfillment_items[]
2. skipped_options_with_reasons[]
3. chapter_goal / emotional_goal
4. page hero asset registry
5. why_selected / why_here_not_elsewhere

### 第三批（让系统变成可复用飞轮）
1. fragment distillation queue
2. page blueprint templates
3. hotel strategy presets 回库
4. activity cluster usage stats
5. trace / eval / feedback 闭环

---

## 17. 一句话决策

**存“结构与决策”，缓存“视图与重算结果”，回库“片段与蓝图”，不要把整份旧攻略全文当默认模板。**
