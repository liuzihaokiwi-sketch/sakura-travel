# Travel AI 统一系统化方案（从“免费一日极简收集”开始的完整整合版）

> 本文档只整合以下阶段确定的内容：  
> 从“免费一日只收极简信息”开始，到后续的样片交付、客服承接、付费后详细表单、自动校验、片段攻略复用、硬规则/软规则接入、后端生成链路、PDF/H5 交付、反馈回库。  
> 如与更早的旧讨论冲突，以本文档为准。

---

## 0. 一句话总纲

系统主策略不再是“用户一上来填很多信息，然后 AI 从零生成完整攻略”，而是：

**免费阶段：极简输入 → 自动给高质量一日样片 → 样片页成交**  
**付费阶段：详细表单 → 自动校验 + 客服补齐 → 片段复用优先 → 规则装配 → AI 少量解释 → 质量门控 → PDF/H5 交付**

这套结构的目标是同时实现：

1. 免费阶段转化阻力最低
2. 正式生成前尽量暴露关键错误
3. 后端尽量复用已验证片段，减少 token 和人工复验
4. 让交付结果更稳定、更像“专业顾问做的取舍”

---

## 1. 当前单一真相源（本阶段相关）

### 1.1 免费一日样片
免费一日不再承担“真正定制的前半段”角色。  
它的定位是：

**低信息成本的高质量样片**

#### 免费阶段只收 2 个字段
- 目的地
- 旅行偏好 / 风格

#### 免费样片的本质
- 不是完全从零定制
- 是从预先准备好的样片模板池 / route family 中选最合适的一天
- 再做轻量个性化装配
- 给用户一个完整的一天样片

#### 免费样片必须包含
- 当天去哪
- 路线怎么走
- 当天交通衔接
- 当天餐厅建议
- 当天重点提醒
- 当天 Plan B
- 少量“为什么这样排”的专业判断

#### 免费样片不包含
- 全部天数完整展开
- 全旅程安全事项
- 全旅程准备清单
- 全程预约总表
- 多天酒店完整方案
- 正式修改权益
- 尊享服务权益

#### 免费样片必须包含的成交结构
1. 一天完整样片
2. 后续高光钩子
3. 解锁完整攻略 CTA
4. 次级微信 / 客服承接

## 2. 免费阶段前端流程（极简引流路径）

### 2.1 免费阶段表单
表单只出现两个字段：

#### 字段 1：目的地
- 东京
- 京都
- 大阪
- 关西
- 北海道
- 其它（如要开放，可做搜索下拉）

#### 字段 2：旅行偏好 / 风格
当前推荐先用统一一题版本，MVP 可选：

- 经典不踩坑
- 性价比更高
- 奢华体验
- 小众深玩
- 情侣氛围
- 亲子轻松

> 注意：前台可以先只问一题，但后台不要把这 6 个选项当成同一个维度。
> - 经典 / 小众 → 主映射到主题/玩法
> - 性价比 / 奢华 → 主映射到预算偏向
> - 情侣 / 亲子 → 主映射到同行人模板

### 2.2 免费阶段表单要求
- 手机端优先
- 两个字段都尽量做成选择题
- 目的地若支持更多城市，走自动补全/下拉
- 不问天数
- 不问机票
- 不问酒店
- 不问固定行程
- 不问预算

### 2.3 表单提交后主路径
**主路径：自动进入在线样片页**

不要把“联系客服领取样片”作为主路径。  
原因：
- 用户刚提交时热度最高
- 等客服发会增加流失
- 客服会变成样片发放瓶颈
- 免费阶段不值得让客服人工确认基本输入

### 2.4 免费阶段的次路径
在线样片页中再放：
- 下载 PDF
- 解锁完整攻略
- 加微信咨询

## 3. 免费样片页如何设计

### 3.1 主路径：网页在线预览
样片页主路径应为在线页，不是先给 PDF。  
原因：
- 更适合做首屏成交
- 更适合插钩子
- 更适合埋点
- 更适合加客服微信
- 更适合引导解锁付费

### 3.2 PDF 的角色
PDF 作为补充下载存在：
- 方便保存
- 方便离线看
- 方便发给同行人
- 海外网络不稳时可直接查看

### 3.3 样片页必须出现的模块
按顺序建议：

1. **完整的一天样片**
   - 时间轴
   - 区域
   - 餐厅建议
   - 路线安排
   - Plan B

2. **为什么这一天这样排**
   - 1–3 个专业判断点

3. **后续高光钩子**
   - 第 3 天的高光体验
   - 某一晚的夜景 / 温泉 / 晚餐高光
   - 整趟路线脉络（锁细节）

4. **主 CTA**
   - 解锁完整攻略

5. **次 CTA**
   - 加客服微信聊这趟更适合你的版本

6. **PDF 下载**
   - 作为补充入口

### 3.4 样片页文案目标
不是只证明“我们有内容”，  
而是要让用户产生这两种感受：
- 这份东西真的值钱
- 后面几天还有更值得买的部分

## 4. 客服在免费阶段的角色

### 4.1 客服不负责
- 发放基础样片
- 手动确认免费阶段基础信息
- 逐单审核免费样片

### 4.2 客服负责
- 接住样片看完后犹豫的用户
- 回答价格和付费问题
- 处理特殊需求
- 付费后发详细表单
- 付费后协助补齐红黄问题

### 4.3 微信的最佳位置
微信不要拦在最前面。  
应放在：
- 样片页中部
- 后续钩子下方
- 页面底部固定次按钮

---

## 5. 付费后的详细表单：目标与原则

### 5.1 目标
付费后详细表单不是为了“多问”，而是为了：

**锁死真正影响完整攻略质量的关键约束**

### 5.2 核心原则
1. 分步填写，不做一页超长表单
2. 选择题优先，少开放输入
3. 地点全部标准化，不允许用户乱写
4. 条件字段默认隐藏
5. 行内校验，不等最后报错
6. 提交后不直接生成，必须先过后台校验

---

## 6. 详细表单完整结构（6 步）

### Step 1：基础路线信息
目标：先把骨架定住

#### 必填
- 去几天
- 目的地 1
- 是否第一次去日本
- 这次更像：
  - 多城顺玩
  - 一地深玩
  - 还没想好

#### 建议收
- 目的地 2（可选）
- 目的地 3（可选）
- 每个目的地预计停留天数（可选）

#### 设计要求
- 目的地必须用搜索 + 下拉结果选择
- 选中后保存标准化 `place_id / location_id`
- destination_2 / destination_3 默认隐藏，用户点击“+ 添加目的地”后才出现

---

### Step 2：交通与固定时间
目标：锁死不能动的约束

#### 必填
- 是否已买机票 / 车票
- 到达日期
- 到达时间
- 到达地点（机场 / 车站 / 城市）
- 离开日期
- 离开时间
- 离开地点（机场 / 车站 / 城市）

#### 条件触发
如果用户有固定安排，再展开：
- 固定安排类型（演出 / 预约 / 见朋友 / 餐厅 / 其它）
- 固定日期
- 固定时间
- 固定地点
- 备注

---

### Step 3：住宿情况
目标：确认可调整空间

#### 必填
- 是否已订酒店
  - 全没订
  - 已订一部分
  - 已订全部

- 是否接受更换酒店区域
  - 可以
  - 尽量不要
  - 不接受

- 是否希望尽量少换酒店

#### 条件触发
若已订酒店，则展开：
- 酒店所在城市 / 区域
- 入住日期
- 退房日期
- 酒店名（可选）
- 备注（可选）

---

### Step 4：同行人与风格
目标：让系统知道这趟该怎么排

#### 必填
- 和谁去
  - 情侣
  - 闺蜜好友 / 朋友
  - 带父母
  - 带孩子
  - 独自出行

- 这趟更想要什么感觉
  - 经典不踩坑
  - 更有审美 / 更容易出片
  - 吃得更好
  - 更轻松
  - 更小众一点
  - 温泉疗愈
  - 购物为主

- 节奏偏好
  - 轻松
  - 均衡
  - 能接受高密度

#### 条件触发
- 带父母 → 额外收步行/换乘承受度
- 带孩子 → 年龄段 / 推车需求 / 午休需求
- 情侣 → 是否纪念日 / 氛围感是否优先

---

### Step 5：预算与取舍
目标：别让用户自己选模块，让系统懂预算怎么花

#### 必填
- 预算范围
- 更想把钱花在哪里
  - 住得更好
  - 吃得更好
  - 体验更特别
  - 更均衡
  - 更看重整体性价比

#### 建议收
- 是否想安排温泉旅馆
- 是否想安排纪念日晚餐
- 是否想安排购物
- 是否想安排文化体验
- 是否优先出片

---

### Step 6：硬约束与避坑
目标：减少返工

#### 必填
- 必去点 / 区域
- 不想去点 / 区域
- 饮食禁忌 / 过敏
- 体力情况（低 / 中 / 高）

#### 建议收
- 不想早起
- 不想太晚回
- 是否需要无障碍 / 特殊照顾
- 其它特殊备注

---

## 7. 后台校验：红黄绿系统

付费表单提交后，不直接生成。  
先进入自动校验。

### 7.1 红色：必须补充 / 必须改，不能生成
示例：
- 到达/离开时间缺失
- 已买机票但没填到离地点
- 主要目的地为空
- 固定安排缺时间或地点
- 已订酒店但酒店信息不完整
- 必去和不想去里有冲突地点
- 离开日期早于到达日期
- 行程天数与到离时间明显不一致

### 7.2 黄色：建议确认，可先不拦截
示例：
- 选择一地深玩，但添加了多个跨区域目的地
- 预算偏性价比，却选择多个高成本项目
- 体力偏低，但想走高密度路线
- 到达太晚 / 离开太早
- 已锁死酒店，但路线跨度很大
- 带父母 / 带孩子，却同时想多城高密度

### 7.3 绿色：可进入生成
- 主要字段完整
- 无红色冲突
- 黄色项可被客服确认或默认保守处理

---

## 8. 客服在付费后的正确角色

### 8.1 客服要做的事
- 发详细表单链接
- 查看后台红黄绿结果
- 只追问红色和黄色问题
- 确认后触发生成
- 在必要时接管复杂情况

### 8.2 客服不该做的事
- 人工通读整个表单
- 人工推理整个路线
- 人工清洗所有自由文本
- 人工发基础样片

### 8.3 后台给客服看的界面
客服后台不该展示完整原始表单为主，  
而应展示：
- 红色问题摘要
- 黄色问题摘要
- 一键复制给用户的话术
- 当前通过状态

## 9. 后端正式生成：完整主链路

付费订单生成完整攻略，不再靠一个大 prompt 一次性写完。  
正确主链如下：

1. 创建订单与 trip 草稿
2. 等待详细表单
3. 自动校验表单
4. 客服补齐红黄项
5. 用户画像标准化
6. 优先复用片段库
7. 硬规则过滤
8. 软规则重排
9. 路线骨架装配
10. 动态快照补槽
11. AI 少量解释
12. 质量门控
13. 多模型评审 / 人工复核
14. 渲染 PDF / H5
15. 交付
16. 自助微调
17. 正式修改
18. 回访与反馈回库

## 10. 用户画像标准化（生成前必做）

详细表单通过后，不直接进 planner。  
先把用户数据整理成内部统一画像：

- 目的地序列
- 天数
- 同行人类型
- 日本熟悉度
- 多城 / 深玩
- 主题偏好
- 预算偏向
- 固定时间点
- 酒店限制
- 必去 / 不想去
- 饮食禁忌
- 节奏偏好

输出：
- `planning_profile`
- `profile_version`

## 11. 片段攻略库：必须成为默认生成策略

### 11.1 核心思想
不要每单从零生成整份攻略。  
应该优先复用数据库里已验证的“片段攻略”。

### 11.2 片段不是整份旧攻略
片段库里存的是：
- 路线片段
- 决策片段
- 体验片段
- 附录片段

并且存的不是最终大段成文，而是：
- 核心骨架
- 适用条件
- 关联实体
- 风险提示
- 可复用理由
- Plan B 模式

### 11.3 为什么必须这样做
- 更 grounded，减少幻觉
- 更省 token
- 更少人工复验
- 更容易把历史成功经验沉淀下来

## 12. 片段攻略库 Schema（最小可用版）

### 12.1 主表：guide_fragments
每条记录代表一个可复用攻略片段。

> ⚠️ 已实现版本见 `app/db/models/fragments.py`，以代码为准。

实际字段（对齐实现）：
- fragment_id (UUID)
- fragment_type（route / decision / experience / logistics / dining / tips）
- title, summary
- city_code, area_code
- theme_families (JSONB 数组，支持多值匹配)
- party_types (JSONB 数组)
- budget_levels (JSONB 数组)
- season_tags (JSONB 数组)
- day_index_hint (建议放在第几天)
- duration_slot (morning / afternoon / evening / full_day / half_day)
- body_skeleton (JSONB 骨架结构)
- body_prose (润色文案)
- body_html (渲染 HTML)
- quality_score (0-10)
- source_type (manual / ai_generated / distilled / imported)
- source_trip_id
- author, version
- status (draft / active / deprecated / archived)
- is_active
- created_at, updated_at, last_used_at

### 12.2 关联表：fragment_entities
存片段和实体之间的关系。

> ⚠️ 已实现版本见 `app/db/models/fragments.py`，以代码为准。

实际字段（对齐实现）：
- fragment_id
- entity_id (FK → entity_base)
- entity_role（primary / secondary / alternative / nearby）
- slot_order (排列顺序)
- is_replaceable (是否可被同类替换)

### 12.3 向量表：fragment_embeddings
用于做语义召回。

建议字段：
- fragment_id
- embedding_model
- embedding_vector
- embedding_text
- updated_at

### 12.4 兼容表：fragment_compatibility
用于判断片段和片段能不能拼。

建议字段：
- left_fragment_id
- right_fragment_id
- compatibility_score
- reason_codes
- hard_block

### 12.5 统计表：fragment_usage_stats
记录片段复用效果。

建议字段：
- fragment_id
- hit_count
- accepted_count
- edited_count
- rejected_count
- avg_user_rating
- preview_unlock_rate
- post_edit_rate
- refund_risk_score
- updated_at

### 12.6 回库队列：fragment_distillation_queue
用于从已交付攻略反向沉淀新片段。

建议字段：
- queue_id
- source_plan_id
- source_day_id
- candidate_fragment_type
- extract_status
- extracted_payload
- reviewer_note
- created_at
- updated_at

## 13. 片段库里到底存什么

### 应该存
- 路线骨架
- 核心高光
- 为什么这样排
- 适用人群
- 适用节奏
- 适用预算偏向
- 关键风险
- Plan B 方向

### 不应该存太死
- 精确到分钟的时间轴
- 实时价格
- 实时营业状态
- 某次订单特定航班
- 某次订单特定酒店房态

这些应该交给动态快照层补，不该固化在片段里。

## 14. 片段复用流程（正式版）

### Step 1：标准化画像
输入：
- 目的地
- 天数
- 同行人
- 偏好
- 预算偏向
- 熟悉度
- 多城 / 深玩
- 固定约束

输出：
- normalized_trip_profile

### Step 2：先做 metadata filter
先按结构化字段过滤：
- city
- theme_family
- trip_style
- suitable_for
- season
- familiarity
- budget_bias
- verified_status
- is_active

### Step 3：再做语义召回
在过滤后的集合里做 embedding / semantic retrieval，找更接近当前画像的片段。

### Step 4：硬规则过滤
把命中的片段丢进硬规则闸门：
- 时间冲突
- 固定事件冲突
- 同行体力不合
- 酒店依赖冲突
- 预算越界
- 季节性不匹配

### Step 5：软规则重排
通过硬过滤后，用软规则表 + weight packs 重新排序。

### Step 6：片段装配
按顺序拼：
1. 总纲片段
2. 每日主路线片段
3. 决策片段
4. 体验片段
5. 条件页片段
6. 附录片段

### Step 7：动态快照补槽
只补少量高时效信息：
- 酒店/餐厅营业状态
- 花期 / 天气 / 节庆窗口
- 重点体验可预约性

### Step 8：AI 少量解释
AI 只负责：
- 总设计思路
- 今日为什么这样排
- 个性化亮点解释
- 少量语气润色

### Step 9：质量门控
检查：
- 总纲完整
- 每天固定骨架完整
- 条件页触发合理
- 事实没过期
- 是否需要人工复核

## 15. 命中 / 不命中策略（必须分 4 档）

> ⚠️ 已实现版本见 `app/domains/planning/fragment_reuse.py` HitTier，以代码为准。

### A 档：强命中（final_score ≥ 0.7 且 quality ≥ 7）
条件：
- metadata + embedding + 硬规则 + 软规则全通过
- 质量分 ≥ 7

策略：
- 直接复用骨架
- 只做轻量替换和 AI 润色
- 默认不需要人工复核

### B 档：普通命中（final_score ≥ 0.4）
条件：
- 通过硬规则，综合评分中等

策略：
- 复用骨架
- 替换部分实体
- 重写少量解释
- 重新过硬规则与软规则

### C 档：弱命中（final_score < 0.4 但通过硬规则）
条件：
- 通过硬规则但综合评分低
- 无更好替代时仍可采用

策略：
- 复用骨架但大幅补充
- AI 补衔接说明
- 默认黄灯审核

### D 档：不命中（未通过硬规则）
条件：
- 硬规则不通过
- 或无任何候选片段

策略：
- gap 标记为 ai_generate
- 从实体池重新装配
- 生成后进入 review
- 好的结果后续蒸馏回片段库

## 16. 硬规则表 / 软规则表在这里怎么接

### 16.1 硬规则表
硬规则表不负责“谁更讨喜”，只负责：
- 过滤
- 拦截
- 质量门控

它在系统中的位置：
1. 片段召回后的第一道 gate
2. 装配完成后的 quality gate

### 16.2 软规则表
软规则表负责：
- 片段 rerank
- 免费样片更容易成交
- 完整攻略更少后悔
- 自助微调候选更容易被接受

软规则表建议至少维护这些维度：
- emotional_value
- shareability
- taste_signal
- localness
- uniqueness
- smoothness
- pacing_fit
- food_confidence
- photo_potential
- recovery_friendliness
- premium_feel
- preview_conversion_power
- confidence_of_choice
- redo_resistance

### 16.3 客群权重包
最小先覆盖：
- couple
- besties
- friends_small_group
- parents
- family_child
- first_time_fit
- repeat_fit
- deep_stay

### 16.4 阶段权重包
至少覆盖：
- preview_day1
- standard
- premium
- self_serve_tuning

## 17. 攻略正文结构（最终统一版）

### 第 1 层：总纲
必须有：
1. 为什么这份攻略这样设计
2. 一眼看懂总览
3. 关键预订提醒
4. 出发前准备 / 总注意事项

### 第 2 层：每天固定骨架
每天固定 4 页：
1. 今日执行总览
2. 为什么今天这样排
3. 今日亮点展开
4. 今日注意事项 / Plan B

### 第 3 层：条件页
按需触发：
- 酒店页
- 餐厅页
- 交通页
- 出片页
- 顺路补充页
- 预算感知页

### 原则
不要再用“每天很多页”来堆厚度。  
要用：
- 总纲前置
- 固定骨架
- 条件页触发
- 静态块复用
- AI 少量解释  
来堆“值感”。

## 21. 状态机（已统一实现）

> ⚠️ 已实现版本见 `app/api/orders.py` VALID_TRANSITIONS，以代码为准。

统一 11 状态（附加 cancelled / refunded）：

```
new → sample_viewed → paid → detail_filling → detail_submitted
→ validating → needs_fix → validated → generating → done → delivered
```

- new — 用户提交 quiz
- sample_viewed — 查看了样片页
- paid — 已付款
- detail_filling — 正在填写详细表单
- detail_submitted — 详细表单已提交
- validating — 红黄绿校验中
- needs_fix — 校验不通过，需补填
- validated — 校验通过
- generating — 正在生成完整攻略
- done — 生成完成，待审核/交付
- delivered — 已交付用户
- cancelled — 已取消（任意状态可转）
- refunded — 已退款

## 22. 反馈回库与片段沉淀

### 回访节奏
- T+2~3：第一轮反馈
- T+6~7：第二轮补采
- T+7~14：高满意用户触发转介绍 / 复购邀请

### 回访要做什么
- 验证实体体验
- 验证节奏
- 验证餐厅和酒店推荐
- 记录满意点 / 后悔点
- 更新片段质量分
- 判断是否值得沉淀成新片段

### 每次订单完成后都要判断
- 有没有新的路线片段可回库
- 有没有新的决策片段可回库
- 有没有值得标准化的 Plan B
- 有没有新的高命中组合值得保留
